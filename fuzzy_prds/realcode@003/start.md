## Product Requirement Document

We have expensive connections to external services — databases, caches, message brokers — that we cannot afford to open fresh on every request. We need a shared pool that holds a fixed number of them and lends them out on demand. The pool should work as a block-scoped resource: a caller borrows a connection, does its work, and the connection is automatically returned when the block finishes, even if an error is raised. When all connections are busy a caller should wait up to a configurable timeout and then fail cleanly with a normalized error — never hang forever, and never leak runtime exception details. Sequential requests with no contention should reuse the same connection instance rather than create new ones each time. We also need a transparent proxy variant: drop-in replacements for a pooled resource that forward ordinary method calls straight through without the caller needing to know about the pool. The proxy should also answer capability questions — whether it can handle a given call — the same way as the connection itself (including virtual capabilities), plus it should always report that it handles the block-borrow operation. The design must follow the same re-entrancy pattern as the existing database connection wrapper: borrowing inside an already-held borrow on the same thread reuses the live connection rather than trying to acquire a second one.
