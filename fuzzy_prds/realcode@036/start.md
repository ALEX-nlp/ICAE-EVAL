## Product Requirement Document

We need a server-side toolkit that connects a web framework to the output of an external front-end bundler. The bundler compiles JavaScript and stylesheets into a public directory and emits a manifest file — a JSON map from each logical asset name to the digest-fingerprinted URL it compiled to. Templates should reference stable logical names and have those resolved at runtime to the correct cache-busted URLs. The toolkit needs four things: first, resolve which named build environment is currently active from a configuration document and two ordered candidates (an explicit override and the host framework environment), falling back to a default when neither matches; second, derive all the project filesystem paths — source directory, entry subdirectory, output directory under public, manifest file path, build config directory — from a small configuration document relative to the project root, with defaults for any missing keys; third, look up a logical asset name in the compiled manifest and return the public URL and disk path, or fail cleanly with a category error when the name is absent; fourth, render view helpers that produce a public URL, a script tag, or a stylesheet link tag for a given asset name. Handle the manifest-not-found error the same way the auth module handles missing tokens — surface a category error with the offending name, not a stack trace.
