## Product Requirement Document

We need a lightweight, in-process search library that our application can drop in to let users find items from a modest catalog without standing up any external service. The library should handle the whole pipeline from raw text to ranked results: normalize the text (case folding, stripping punctuation, splitting on spaces), reduce words to a common root form so that plural and singular forms match each other, then store a reverse index keyed by those normalized terms. Documents to be indexed are plain in-memory objects; each document must carry a single integer identifier field and one or more text fields. Some fields should be explicitly excluded from indexing — words appearing only in those excluded fields must never be searchable. The search API should support an optional field-restriction so callers can say 'look only in the title field'. Searching with an empty restriction list should return nothing. The system should also return prefix-based suggestions so the UI can show autocomplete hints as the user types — suggestions come back ranked by relevance. Adding the same document twice should be ignored (no duplicates). Removing a non-existent document should be a safe no-op. Invalid documents — missing identifier, wrong identifier type, or more than one identifier field — should produce a clear, machine-readable error category rather than a crash. Make the text-processing steps individually testable and composable as a pipeline so we can reuse the same normalization logic for both indexing and querying, similar to how the ranking helpers are shared internally.
