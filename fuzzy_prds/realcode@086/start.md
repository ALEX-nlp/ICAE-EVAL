## Product Requirement Document

We're building a service that lets people track their code-hosting notifications in one place. We need the core domain logic for managing accounts and pulling the notification feed. Specifically: (1) when the UI needs to show a flash banner it should ask for the right visual style class by passing a category name — success, error, warning, info — and unknown categories should pass through unchanged; (2) an account can have a default access credential and optionally a personal one, but we only use the personal one when our instance has that feature turned on AND the user has actually provided one; (3) when someone signs in via the external provider, their provider user ID, their display name, and their session credential should map into the account's stored fields — look at how the authentication callback handler projects these from the provider payload; (4) when we pull the notification feed for an account, the 'since' timestamp should be one week back for returning users but one month back for a first-time sync; (5) feed items should be stored with a canonical link to the subject — for repository-invitation items the subject URL comes from a different source than for ordinary items; (6) before storing an account, validate that required fields are present and that uniqueness constraints hold, reporting each failure separately; (7) a brand-new account should automatically trigger an initial feed pull immediately on creation; and (8) before trusting a personal access credential, verify it.
