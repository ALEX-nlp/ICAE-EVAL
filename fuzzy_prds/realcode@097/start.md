## Product Requirement Document

We need a request quota enforcement engine for our API gateway that protects backend services from traffic overload. The system should determine, for any incoming HTTP request, whether a usage cap applies to it based on which endpoint it targets. Operators will configure a set of named routes (each identified by a path prefix) and attach quota policies to specific routes. A policy defines how many requests are allowed within a time window and which dimensions (caller IP, URL path, authenticated user identity, etc.) are used to partition callers into independent buckets so they don't share each other's allowance.

When a request arrives, the engine should first resolve which declared route it belongs to by checking path prefixes in order. If the matched route has an associated policy and the feature is globally switched on, the request is subject to quota accounting. Each accounted request should decrement the caller's remaining allowance and report back the current limit and remaining count via standard response metadata headers (look for how the filter sets response headers in the core filter implementation). Once a caller's allowance hits zero, subsequent requests must be rejected with the standard HTTP "too many requests" status code and the response should flag the request as having exceeded the quota.

The system should be cleanly designed so the accounting store can be swapped out (in-memory for development, external stores for production). A test adapter that reads a JSON command describing the configuration, the incoming request, and a replay count from standard input, then drives the engine and writes observable signals to standard output, is required for automated testing. The quota bucket key construction logic — which combines a configurable application prefix with route and caller identity segments — should be inspectable through the test harness.
