## Product Requirement Document

We need the data modeling layer for a file cache that stores downloaded remote files on device. The cache needs to remember everything about each cached file: where it came from, what it's stored as on disk, when it expires, how large it is, and a lookup key. Three things need to work: first, the cache record model itself — building records from a URL, loading them back from stored flat maps, saving them to flat maps for persistence, and making modified copies of existing records; second, parsing the HTTP response metadata to figure out the file extension, revalidation tag, content size, and how long the content stays fresh; third, calculating download progress as a fraction. Key design question: when you create a record with just a URL and no explicit lookup key, the key defaults to the URL itself. The persistence format needs a 'touched' timestamp for tracking last access, using the current time at the moment of saving. For HTTP freshness: if the response has a max-age directive use that, otherwise fall back to a one-week window — handle this the same way our image-loading module decides cache duration. File extension mapping: common media types get their conventional extension, unknown types fall back to using the subtype as the extension. When a copy operation overrides a field, the lookup key is always preserved from the original.
