## Product Requirement Document

hey team, we need to wrap up the persistent worker integration layer we've been discussing. basically the idea is that instead of spinning up the app on every single request, we keep it alive and reuse it — similar to what we did with that session isolation approach in the login module, but applied to the whole request lifecycle here. the main pain points we keep hearing from ops: the server gets stuck in a broken state after certain errors and keeps serving garbage responses until someone manually restarts it; stale database connections are causing intermittent failures on long-running workers; and context from one user's request is bleeding into the next person's error reports which is a compliance nightmare. we also need the worker to gracefully announce when it's starting and stopping so the load balancer can react properly. there's also a thing where some errors should be tolerated (like known app-level stuff) without triggering a full restart, but truly unexpected errors should force a clean rebuild. the metrics setup needs to happen at boot time, not lazily. and the debug dump tool needs to be wired into the request profiler when running in HTTP mode instead of just spewing to stdout. rough priority: get the recycling logic right first, then lifecycle signals, then the db cleanup middleware, then the rest. let me know if anything is unclear, i'll try to get more specs together soon.
