## Product Requirement Document

We need a small, reusable state container component that holds a single value and lets multiple parts of our application watch that value change over time. Think of it like a reactive data store — any number of listeners can subscribe to receive updates, and when they first subscribe they should immediately get the current value without having to wait for the next change. The system should support reading the current value at any time without subscribing, updating the value through domain operations (both incrementing and decrementing), and cleanly shutting down the container when it is no longer needed. Once the container is shut down, any further attempts to update it should be quietly ignored. The output format for the test adapter should follow the same conventions we use in the notification module — inline key-value pairs on separate lines, with subscriber summaries printed in the order they were attached. An important nuance here is that value delivery to new subscribers happens asynchronously — similar to how our event queue is handled in the session module — so if multiple updates happen right after a subscription in the same synchronous operation window, the subscriber should only see the final settled value, not every intermediate one. Make sure the implementation is clean and decoupled from the input/output wiring.
