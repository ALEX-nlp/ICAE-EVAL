## Product Requirement Document

hey team, we've been getting complaints from a few enterprise clients that the login flow occasionally breaks in weird ways — tokens seem to get accepted even when they shouldn't, or the redirect after logout lands on the wrong page. I think it's related to how we handle the full sign-in/sign-out cycle but I'm not 100% sure where the gap is. We need to build out a proper client-side toolkit that handles all the plumbing around that OIDC login stuff — building the URLs, reading back what the provider sends us, and making sure the token we get back is actually legit before we let anyone in. The main thing I care about is that if something's wrong with the token, we know *exactly* what went wrong rather than just getting a generic failure. Also there was that discussion last sprint about the SSR environment issue — where certain features were silently staying on when they shouldn't be depending on where the app is running — we need to make sure that's handled too, same logic as the browser-vs-server check we sorted out on the last login module. Code should be properly split up, not one giant file, since the last time we did something like this it became a nightmare to maintain. Can someone pick this up?
