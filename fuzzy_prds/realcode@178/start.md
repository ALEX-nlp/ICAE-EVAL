## Product Requirement Document

We need a web request handling toolkit that lets developers build servers by composing small, reusable processing units rather than writing low-level plumbing. Both incoming requests and outgoing replies are treated as immutable data objects you can inspect and transform at any point. Each object carries header information accessible in two ways: a convenient single-value view and a full multi-value view, both sorted alphabetically when output. The system automatically determines body size from actual content, not from any manually set header. When a developer specifies a character set for a body, the content-type field is automatically rewritten to match. Bodies can only be consumed once — a second attempt triggers a well-defined error signal rather than failing silently. Incoming addresses are split into the portion already handled versus the portion yet to be routed, enabling clean nested routing without path arithmetic. Outgoing status codes come with convenience constructors for the most common outcomes — success, redirect, not-found, server error, and cache-unchanged — each populating the right default body and location information automatically. Date-related fields like freshness expiry and last-modification time are surfaced as structured timestamps. Handlers can be arranged in a fallback chain where the platform tries each one in order and skips unacceptable results. Cross-cutting concerns are layered via a wrapping mechanism where outer layers see the request first and the response last. Chunked framing is applied automatically to streaming responses that lack a declared size.
