## Product Requirement Document

We need a unified SMS dispatch library that works across multiple provider back-ends without callers having to know each provider's particular request format. The library should handle: (1) normalizing a recipient phone number — the caller provides a local number and optionally an international calling code (which might have a leading plus or leading zeros); the library should expose the raw number, the parsed country code, a plus-prefixed universal form, a double-zero-prefixed form, and a JSON representation; (2) reading and mutating a nested credential configuration using dot-notation keys, including descending into nested objects and numbered list entries; (3) modeling a message as either a simple string or a richer object with content, template ID, template variables, and a preferred list of delivery providers; (4) resolving a named delivery provider — short names should be matched case-insensitively and ignoring hyphens and underscores, with a configurable default — and failing fast with a normalized error when the name is unknown or no default has been set; (5) determining the effective request timeout for a provider, with a precedence hierarchy: per-instance override wins, then config-supplied value, then a built-in fallback; and (6) assembling the exact outbound HTTP request for each supported provider and normalizing the provider's reply into either an accepted outcome or a consistent error with the provider's own code and message. Follow the same signature-prefix convention that the message-validation layer uses for the first listed provider.
