## Product Requirement Document

We need a framework layer that gives a web controller all seven standard request handlers — listing a collection, showing one record, rendering a blank creation form, rendering an edit form, creating a record from submitted fields, updating an existing record, and deleting one — without hand-writing those handlers for every resource. Each handler should do the conventional thing: load the right data from the backing store, expose it under a predictable name, and either render a view or redirect with a flash message. On success, writes redirect to the record's own page; on failure, they re-render the form. The redirect on destroy should always go to the collection page, carrying success or failure feedback. We also need format negotiation: the default HTML behavior stays the same, but when a JavaScript representation is requested a successful write should render a view instead of redirecting. Unsupported formats must produce a clean, neutral error line — no stack traces. Configurable overrides are needed: teams should be able to change what name the loaded record is exposed under (think of it like how our auth module aliases the current user), choose a custom field to look records up by instead of the default identifier, and disable specific actions so the redirect logic falls back gracefully. Deliver a clean library plus a test adapter reading from stdin.
