## Product Requirement Document

We need a fluent guard-clause library for validating method inputs. The idea is simple: at the top of any method, you write one readable expression that either lets the value through unchanged or throws a well-described exception. Right now everyone hand-writes their own if-then-throw patterns, leading to different exception types, inconsistent messages, and missing the parameter name in the error. The library should cover: null rejection, string checks (empty, whitespace, length limits, equality, substrings, prefix/suffix), numeric ordering and range checks, boolean state checks, collection emptiness and size constraints, timestamp kind validation, enum membership, URI scheme and kind checks, and a custom condition guard for domain-specific rules. On rejection, the output always follows the same pattern: first a line naming the outcome, then the error category (we have four: null argument, out-of-range, general argument, and generic), then the parameter name, then a message explaining the problem. Range rejections add one more line echoing the actual offending value — follow the same convention our payment-range validator uses. On pass-through, return the original value rendered deterministically. Boolean values render with an initial capital. Collections render as bracketed comma-separated lists. Custom exception support: users should be able to raise their own exception type with a custom message by providing a condition and a factory.
