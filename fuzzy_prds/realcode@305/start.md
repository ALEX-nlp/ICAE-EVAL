## Product Requirement Document

Hey team, we need to get this GrAMPS-style GraphQL composition layer wrapped up. The idea is basically what we scoped out in Q3 — one HTTP endpoint that works in both dev and prod, with the dev mode returning placeholder data and prod hitting real resolvers. We also need that interactive docs page (you know, the browser-based query explorer thing) available during development. There's also the middleware piece where we attach context and data sources per request — similar to how we handled the auth tenant stuff in the old login module, so check that for reference on how context keys get merged.

On the schema side, sources should contribute their own query fields and the built-in root ops (ping and version) should always be present. Dev sources loaded from external paths should be able to swap out local ones when the context key matches, and we should log something when that happens. For outbound calls, connectors need consistent request config (JSON mode, full response, etc.), and GETs should play nicely with Redis caching — serve cached when available, refresh in background, skip cache on errors. Mutations go out as POST or PUT with JSON body. Errors need to be normalized so clients never see raw stack traces or internal exception names, and the format should differ slightly between dev and prod (some fields hidden in prod). Make sure the model base classes blow up properly if someone tries to instantiate them directly.
