## Product Requirement Document

Hey team, we need to build out that auth handoff flow we talked about in the last sprint — the one where a third-party app can get a short-lived code on behalf of a user without us ever handing them the user's password. Basically it's the redirect-based handshake thing. The tricky part (and where we got burned last time on the login module) is knowing WHEN to throw an error directly vs. when to bounce the user back to the app with the error in the URL — we need to get that split exactly right or clients will have no idea what went wrong. There are also some config toggles the ops team needs to be able to flip — things like whether we force that anti-forgery value, whether we're strict about the callback URL matching, and optionally enabling that identity assertion feature for our SSO partners. Scope handling needs to work too — if someone sets up the deployment without a fallback scope, the request should fail gracefully. We also need the issued codes to be one-time-use and redeemable at a separate token endpoint. The output format for the runner needs to be totally deterministic — no raw secrets in logs ever. Please separate the concerns properly, don't just dump everything into one file like we did on the billing service. Refer to how we structured the redirect reconstruction piece for the URL component handling.
