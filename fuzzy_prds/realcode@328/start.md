## Product Requirement Document

hey team, we need to wrap up the state estimation toolkit for the navigation crew. basically they want a single entry point that can handle a bunch of different operations — running the two flavors of recursive filter (you know, the 'extended' and the 'sigma-point' one), checking some shared bookkeeping stuff like how measurements are ordered and what filter params look like after a round-trip, doing the coordinate projection thing we talked about (lat/lon ↔ grid), and querying that ring-buffer history thingy. the driver reads JSON from stdin, routes to the right handler, and prints results to stdout. the tricky part is the output formatting — there was some agreement about how numbers get printed (check how we handled the covariance diagonal in that other nav module) and the filters need identical external contracts so they're swappable. the first measurement in a filter session is special — it doesn't predict, it just seeds the state. also the geo conversion should round-trip to a fixed number of decimal places that the QA team signed off on somewhere. the buffer needs to handle capacity overflow by dropping the oldest entry, and queries outside the stored range should label themselves differently from interpolations. make sure errors are typed properly and the core logic stays totally decoupled from stdin/stdout plumbing.
