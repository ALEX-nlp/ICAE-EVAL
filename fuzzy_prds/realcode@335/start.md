## Product Requirement Document

We need a reusable toolkit for managing independently deployable software modules — the kind of self-contained extensions that get dropped into a running application without rebuilding everything from scratch. Each module declares its own version and lists the other modules it depends on, with an optional minimum/maximum version requirement per dependency. The toolkit needs to answer a few concrete questions: does a given version string satisfy a declared requirement? which of two versions is newer? can we parse a dependency coordinate string into its parts? given a whole set of modules, can we figure out a safe load order — and if not, why not? The load-order check should handle cycles, missing dependencies, and dependencies whose present version doesn't match what was asked for. We also need a general-purpose directed-graph utility that can topologically sort vertices and report how many edges point in and out of each one. Finally, we need a way to parse a module's key-value metadata file into a normalized descriptor object. Handle it similarly to how the version-comparison logic works — return a well-typed result, don't throw raw exceptions, surface error categories clearly.
