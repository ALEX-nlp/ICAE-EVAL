## Product Requirement Document

We need a key-derivation toolkit that covers the full chain of operations a self-custody wallet must perform at account setup and recovery time. Starting from raw random bytes, the library should encode them into a human-memorizable word phrase (using a fixed 2048-word English dictionary with a checksum appended before encoding); then it should be able to validate that a phrase a user typed conforms to the allowed word counts and uses only recognized words; then it should stretch a valid phrase into a long binary seed using a standard key-stretching function with a fixed salt string and a fixed iteration count; and finally it should walk a standard hierarchical path through a deterministic key tree derived from that seed to produce both a payment address (in the standard checksum-encoded address format) and the corresponding compressed public key, as well as serialized extended keys for watch-only wallet export. All outputs are stable strings — no randomness after the initial entropy input — so two correct implementations of the same standard must produce identical outputs. Error cases (empty entropy, wrong word count, unrecognized word, non-numeric path component) must come back as predictable machine-readable tags rather than raw exceptions. The derivation path format should accept an optional leading prefix, treat an empty or root-only path as the master key, and support hardened child indices marked with the standard apostrophe notation, consistent with how the extended-key export is handled elsewhere in the wallet module.
