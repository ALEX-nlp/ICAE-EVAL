## Product Requirement Document

We're building the operational transformation core for a real-time collaborative document editing backend. Multiple clients can edit the same document at the same time and we need a library that keeps everything consistent. The library needs to handle the full lifecycle of a collaborative document: checking whether an incoming change is even valid, looking up which editing strategy a document uses by either its short name or its full identifier, applying changes to a versioned document snapshot, and transforming a queued-up change so it still makes sense after someone else's change was already applied. Documents can be created from scratch, have edits applied to their content, or be fully deleted — each of these needs to behave correctly and return precise status information. When something goes wrong, we need stable, named error categories rather than free-form messages, because clients rely on these to branch their error-handling logic. We also need to handle cursor-like presence markers: when a change is applied to the document, any collaborator's cursor position should shift correctly — similar to how the version-advance logic works for the main document. Finally, we need a batch-apply path that processes a sequence of changes one after another, stopping cleanly at the first problem. The whole system should be built so the core rules are completely separate from how commands come in over the wire.
