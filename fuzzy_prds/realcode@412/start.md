## Product Requirement Document

Hey team, we've been getting complaints from a few partner integrations that our URL handling is inconsistent — basically they're building things on top of our parsing output and keep hitting weird edge cases that break their pipelines. The core ask is: we need a proper URL parsing library (Python) that actually follows the living web standard the browser folks use, not just a regex slapped on a string. It should handle the full lifecycle: parse a string (maybe against a base URL), read individual pieces back, mutate those pieces, and re-serialize cleanly. Think of it like what we did for the config validation module — same idea where setting an invalid value should never corrupt the existing state, it just silently rejects. Also needs a lightweight 'would this even parse' probe so callers don't have to do a full parse just to validate. Query string manipulation is a big one too — partners are encoding things wrong constantly, the space/plus encoding thing keeps coming up. The multi-file structure thing is non-negotiable per the platform team's last architecture review. Oh and the I/O adapter layer needs to stay completely separate from the actual parsing logic — we got burned on that coupling before. Should read commands from stdin as JSON and write results to stdout. Reach out if you need the old browser compat reference doc.
