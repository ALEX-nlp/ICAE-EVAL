## Product Requirement Document

We need a rule-checking engine that lets development teams express and enforce structural guardrails over a compiled code base. The tool should accept a JSON description of a check — identifying which code elements to inspect and what constraint they must satisfy — and produce a human-readable report of any violations found. When everything is fine, the output should be a single confirmation line. When something goes wrong, the output must show the exact rule that failed, followed by an indented list of violations, each naming the offending element and the target element it should not have referenced or called. Two main scenarios need to be supported: first, checking that specific classes do not introduce dependencies on other named classes (handle these two flavors the same way as the auth module handles both direct and predicate-based lookups), and second, checking that the methods declared inside certain classes do not call into methods in other named classes. Method-level violations must name both the calling method and the called method using their full runtime-style signatures including return type, declaring type, method name, and parameter types. When the input is malformed, names an unknown element, or uses an unsupported operation kind, the system should emit a structured error line rather than crashing. The check must be evaluated against a fixed reference codebase containing a handful of interconnected service, domain, and infrastructure types with known dependency and call relations. All output must go to standard output with no trailing metadata.
