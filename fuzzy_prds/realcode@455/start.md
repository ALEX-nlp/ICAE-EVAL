## Product Requirement Document

Our vulnerability-tracking platform ingests raw security-feed documents and needs to derive several clean, query-friendly data structures from them without re-implementing the same parsing logic in every consumer. We need four utility functions: (1) given a raw vulnerability configuration document (the deeply-nested kind we receive from the national feed), extract which vendors and products are affected — deduplicate the pairs and return a mapping from each vendor to its distinct products, sorted; (2) given that vendor-product map, flatten it into a single ordered token list that each subscription filter can match against — the format must use a fixed separator string between vendor and product tokens, look at how the subscription-matching module constructs these tokens and follow the same convention; (3) given the weakness classification entries attached to a vulnerability, return the distinct set of identifier strings sorted in ascending order; and (4) return the fixed alphabet of bucket labels used by the vendor directory page to group vendors — this is a constant sequence that should never change. All four functions are pure transformations and should produce canonical output: sorted keys, sorted arrays, and a consistent separator for the combined tokens.
