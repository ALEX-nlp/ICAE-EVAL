## Product Requirement Document

# Stylesheet Module Declaration Engine - Black-Box Contract for Class Exports and Declarations

## Project Goal

Build a stylesheet module processing tool that allows developers to convert module stylesheet files into class export maps and declaration text without manually maintaining duplicated type declarations or class-name mappings.

---

## Background & Problem

Without this library/tool, developers are forced to inspect stylesheet files, track generated class names, and hand-write declaration files whenever classes are added, renamed, imported, or preprocessed. This leads to repetitive code, stale declarations, broken editor feedback, and fragile integration between stylesheets and application code.

With this library/tool, stylesheet module files can be processed into deterministic class export maps and declaration text, while path recognition, external resolution, and preprocessing options let projects keep their existing file naming and stylesheet organization.

---

## Architecture & Engineering Constraints

To ensure this project is delivered as a maintainable software artifact, the following architectural and non-functional requirements (NFRs) MUST be strictly observed:

1. **Scale-Driven Code Organization:** The physical structure of the codebase MUST perfectly match the complexity of the domain.
   - **For micro-utilities/simple scripts:** A well-organized, single-file solution is perfectly acceptable, provided it maintains clean logical separation.
   - **For complex systems:** If the project involves multiple distinct responsibilities (e.g., I/O routing, business rules, formatters), it MUST NOT be a single "god file". You must output a clear, multi-file directory tree (`src/`, `tests/`, etc.) that reflects a production-grade repository.
   Do not over-engineer simple problems, but strictly avoid monolithic files for complex domains.

2. **Strict Separation of Concerns (Anti-Overfitting):**
   The JSON input/output test cases provided in the "Core Features" section represent a **black-box testing contract** for the execution adapter, NOT the internal data model of the core system. The core business logic must remain completely decoupled from standard I/O (stdin/stdout) and JSON parsing. The execution adapter is solely responsible for translating JSON commands into idiomatic method calls to the core domain.

3. **Adherence to SOLID Design Principles:**
   The architectural design must follow SOLID principles to ensure maintainability and scalability (scaled appropriately to the project's size):
   - **Single Responsibility Principle (SRP):** Separate parsing, routing, validation, core execution, and output formatting into distinct logical units.
   - **Open/Closed Principle (OCP):** The core engine must be open for extension but closed for modification.
   - **Liskov Substitution Principle (LSP):** Derived types must be perfectly substitutable for their base types.
   - **Interface Segregation Principle (ISP):** Keep interfaces/protocols small and highly cohesive.
   - **Dependency Inversion Principle (DIP):** High-level modules should depend on abstractions, not low-level I/O implementation details.

4. **Robustness & Interface Design:**
   - **Idiomatic Usage:** The public interface of the core system must be elegant and idiomatic to the target programming language, hiding internal complexity.
   - **Resilience:** The system must handle edge cases gracefully. Errors should be modeled properly (e.g., specific Exception types or Result/Monad patterns) rather than relying on generic faults.

## Core Features


### Feature 1: Class Name Alias Generation

**As a developer**, I want to generate all externally usable property names for stylesheet classes, so I can consume class names in code regardless of whether the source stylesheet uses dashes, underscores, mixed case, or a requested naming convention.

**Expected Behavior / Usage:**

The input contains `behavior`, an optional `class_name_style`, and a `class_names` list. The output is a JSON array (two-space indented) with one inner array per input class name, preserving input order. Each inner array lists the property names that should be exposed for that class. When no style is supplied, or the style requests names "as is", the original class name is exposed unchanged. The camel-case adding style keeps the original name and additionally exposes a camel-cased alias when it differs. The camel-case only style exposes just the camel-cased name. The dash-to-camel adding style keeps the original name and additionally exposes a dash-segment-camelized alias when it differs. The dash-to-camel only style exposes just the dash-segment-camelized name.

**Test Cases:** `rcb_tests/public_test_cases/feature1_class_name_aliases.json`

```json
{
    "description": "Transforms requested stylesheet class names into the externally usable property-name aliases for each supported naming mode.",
    "cases": [
        {
            "input": {
                "behavior": "class-name aliases",
                "class_names": [
                    "class-name-a",
                    "classNameB",
                    "class-Name-C",
                    "__class_nAmeD--"
                ]
            },
            "expected_output": "[\n  [\n    \"class-name-a\"\n  ],\n  [\n    \"classNameB\"\n  ],\n  [\n    \"class-Name-C\"\n  ],\n  [\n    \"__class_nAmeD--\"\n  ]\n]\n"
        },
        {
            "input": {
                "behavior": "class-name aliases",
                "class_name_style": "asIs",
                "class_names": [
                    "class-name-a",
                    "classNameB",
                    "class-Name-C",
                    "__class_nAmeD--"
                ]
            },
            "expected_output": "[\n  [\n    \"class-name-a\"\n  ],\n  [\n    \"classNameB\"\n  ],\n  [\n    \"class-Name-C\"\n  ],\n  [\n    \"__class_nAmeD--\"\n  ]\n]\n"
        }
    ]
}
```

### Feature 2: Stylesheet Import Path Recognition

**As a developer**, I want to identify whether import-like paths refer to module stylesheets, so I can decide when stylesheet module processing should be applied and whether the import is local to the project.

**Expected Behavior / Usage:**

*2.1 Default Path Recognition — classifies paths using the built-in module stylesheet naming rules.*

The input contains `behavior` and a `paths` list. The output emits three lines per path: `path=<path>`, `module_stylesheet=<yes|no>` for whether the filename is recognized as a module stylesheet, and `relative_module_stylesheet=<yes|no>` for whether it is both recognized and written as a relative path. By default a recognized module stylesheet filename ends with `.module.css`, `.module.scss`, `.module.sass`, `.module.less`, or `.module.styl`. A path is relative when it begins with `./` or `../`; a recognized filename without a leading relative segment is reported as a module stylesheet but not relative.

**Test Cases:** `rcb_tests/public_test_cases/feature2_1_default_path_recognition.json`

```json
{
    "description": "Classifies import-like paths using the default module stylesheet filename rules and separately reports whether each matching path is relative.",
    "cases": [
        {
            "input": {
                "behavior": "stylesheet path recognition",
                "paths": [
                    "./myfile.module.scss",
                    "./myfile.module.sass",
                    "./myfile.module.less",
                    "./myfile.module.styl",
                    "./myfile.module.css",
                    "./myfile.module.s",
                    "./myfile.scss",
                    "./myfile.sass",
                    "./myfile.less",
                    "./myfile.styl",
                    "./myfile.css",
                    "../folder/myfile.module.css",
                    "myfile.module.css"
                ]
            },
            "expected_output": "path=./myfile.module.scss\nmodule_stylesheet=yes\nrelative_module_stylesheet=yes\npath=./myfile.module.sass\nmodule_stylesheet=yes\nrelative_module_stylesheet=yes\npath=./myfile.module.less\nmodule_stylesheet=yes\nrelative_module_stylesheet=yes\npath=./myfile.module.styl\nmodule_stylesheet=yes\nrelative_module_stylesheet=yes\npath=./myfile.module.css\nmodule_stylesheet=yes\nrelative_module_stylesheet=yes\npath=./myfile.module.s\nmodule_stylesheet=no\nrelative_module_stylesheet=no\npath=./myfile.scss\nmodule_stylesheet=no\nrelative_module_stylesheet=no\npath=./myfile.sass\nmodule_stylesheet=no\nrelative_module_stylesheet=no\npath=./myfile.less\nmodule_stylesheet=no\nrelative_module_stylesheet=no\npath=./myfile.styl\nmodule_stylesheet=no\nrelative_module_stylesheet=no\npath=./myfile.css\nmodule_stylesheet=no\nrelative_module_stylesheet=no\npath=../folder/myfile.module.css\nmodule_stylesheet=yes\nrelative_module_stylesheet=yes\npath=myfile.module.css\nmodule_stylesheet=yes\nrelative_module_stylesheet=no\n"
        }
    ]
}
```

*2.2 Custom Path Pattern Recognition — overrides recognition with a caller-supplied filename pattern, with safe fallback.*

The input contains `behavior`, a `pattern` (a regular-expression string), and a `paths` list. The output uses the same three-line-per-path shape as the default mode. When the supplied pattern is valid, it decides whether each path is a module stylesheet, while relative reporting still depends on the leading path segment. When the supplied pattern is invalid, recognition silently falls back to the default module stylesheet rules and a single trailing line `pattern_error=ignored` is appended after all path results.

**Test Cases:** `rcb_tests/public_test_cases/feature2_2_custom_path_pattern.json`

```json

                "pattern": "\\.css$",
                "paths": [
                    "./myfile.css",

                    "../folders/myfile.scss"
                ]
            },
            "expected_output": "path=./myfile.css\nmodule_stylesheet=yes\nrelative_module_stylesheet=yes\npath=./myfile.m.css\nmodule_stylesheet=yes\nrelative_module_stylesheet=yes\npath=./myfile.scss\nmodule_stylesheet=no\nrelative_module_stylesheet=no\npath=../folder/myfile.css\nmodule_stylesheet=yes\nrelative_module_stylesheet=yes\npath=../folder/myfile.m.css\nmodule_stylesheet=yes\nrelative_module_stylesheet=yes\npath=../folders/myfile.scss\nmodule_stylesheet=no\nrelative_module_stylesheet=no\n"
        }
    ]
}
```

### Feature 3: Stylesheet Class Export Extraction

**As a developer**, I want to process module stylesheet content and extract the externally visible class export map, so I can use generated class names consistently across supported stylesheet syntaxes, imports, external resolution strategies, and custom preprocessing.

**Expected Behavior / Usage:**

Every leaf below takes a `behavior` of stylesheet export extraction plus a `stylesheet` selector and produces formatted (two-space indented) JSON containing a `classes` object whose keys are the source class names discovered from local selectors and whose values are the generated, scoped class strings. Keys are emitted in sorted order. Empty stylesheets produce an empty `classes` object.

*3.1 Single-Syntax Class Extraction — extracts the class map from a standalone stylesheet in each supported syntax.*

The `stylesheet` selector names a stylesheet sample. The contract covers plain CSS, Less, Stylus, SCSS, and Sass inputs, as well as empty stylesheets of each syntax, all of which were exercised by the original tests. Each non-empty sample yields a `classes` object mapping every locally declared class name to its generated scoped value; empty samples yield `{}`.

The `stylesheet` selector names a sample that imports additional stylesheet content. The output `classes` object contains the class exports of the composed result, including both locally declared selectors and selectors pulled in through the import. This applies across the syntaxes that support imports.

The input adds an optional `resolution` mode alongside the `stylesheet` selector. Resolution modes cover package-style imports, additional include-path directories for indentation-based and brace-based preprocessors, and compiler-style path aliases that remap import specifiers to files outside the current directory. ### Feature 4: Declaration Text Generation

**As a developer**, I want to generate declaration text for the classes discovered in a stylesheet, so I can consume stylesheet modules from typed source code with default and named exports, optional templating, and optional source alignment.

**Expected Behavior / Usage:**

Every leaf below takes a `behavior` of declaration generation plus a `stylesheet` selector and emits raw declaration text (not JSON) to stdout.

*4.1 Default and Named Declaration Output — emits a default class object plus safe named exports.*

The output defines a default `classes` object literal containing every discovered class name as a string-typed property, followed by an exported default binding. It additionally emits one named export statement per class name that is a safe source-level identifier. Class names that contain characters not valid in an identifier, or that collide with reserved words, remain available as properties on the default object but are not emitted as named exports.

### Feature 5: Tilde Import Resolution

**As a developer**, I want to resolve tilde-prefixed stylesheet imports against the installed module tree, so I can reference shared stylesheet packages with a short tilde specifier instead of long relative paths.

**Expected Behavior / Usage:**

The input contains `behavior`, an `import_path`, a `source` file path (the importing stylesheet), and an `installed_files` list describing the files present in the installed module tree (relative paths, where installed packages live under a `node_modules` directory). The resolver walks from the source file's directory up through parent directories looking for the first installed module file that matches the import. The output is a single line `resolved=<relative-path>` with the matched file path, or `resolved=none` when the import does not apply or no file is found.

Resolution rules: an import that does not begin with a tilde, or that begins with a tilde followed by a slash, is never resolved and yields `none`. A tilde import whose specifier already ends with a stylesheet or css extension is matched directly. A specifier without an extension is matched by trying brace-based and indentation-based stylesheet extensions and then a css extension. Index files are supported by looking for an `_index` stylesheet inside the specifier directory. Sass partials are supported by also matching files whose base name is prefixed with an underscore, so both explicit-partial and implicit names resolve to the underscored file. When no matching file exists in the installed tree, the result is `none`.

## Deliverables

1. **The Core System:** A cleanly structured codebase implementing the features described above. Its physical structure (single-file vs. multi-file repository) MUST strictly align with the "Scale-Driven Code Organization" constraint, ensuring high maintainability without over-engineering.

2. **The Execution/Test Adapter:** A runnable program (CLI script or entry point) that acts as a client to your core system. It reads a JSON command from stdin, invokes the appropriate core logic, and prints the result to stdout, strictly matching the per-leaf-feature contracts above. This adapter must be logically (and ideally physically) separated from the core domain.

3. **Automated test harness.** The cases embedded in this PRD live under `rcb_tests/public_test_cases/`. A single entry point `bash rcb_tests/test.sh` reads every `*.json` case file from a case directory and runs the full suite; it accepts `--cases-dir <subdir>` to point at a directory of case files (default `test_cases`). For each case it writes one file to `rcb_tests/stdout/<cases-dir>/{filename.stem}@{case_index.zfill(3)}.txt` (e.g. the first case in `feature1_class_name_aliases.json` run with `--cases-dir public_test_cases` -> `rcb_tests/stdout/public_test_cases/feature1_class_name_aliases@000.txt`). Output is namespaced by `<cases-dir>` so running different case directories never overwrites each other. Each `.txt` file contains **only** the raw stdout from the program under test (no PASS/FAIL summaries or metadata) so it can be compared directly against `expected_output`.
