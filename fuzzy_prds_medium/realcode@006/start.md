## Product Requirement Document

We need a small plugin-style hook for an existing Markdown rendering engine already in use by our editor. Some documents are regular notes, but some should render as slide decks. The key challenge is that the engine is shared, so we cannot swap it out globally — the hook must leave normal documents completely untouched and only activate the slide-deck behavior when a document explicitly requests it through a configuration flag at the very top of the file.

When slide mode is on, the output HTML must include a uniquely identifiable container element and an injected stylesheet so the preview layer can detect whether it is looking at a slide or a regular document. The slide renderer should also handle emoji displayed as images rather than Unicode characters, lay out math formulas with a special fitting treatment, and syntax-highlight code blocks using the slide engine's own highlighter. For diagram blocks that the highlighter does not understand, the content should be wrapped for a downstream diagramming tool. If the language tag is entirely unknown, the code block should still preserve that label so consumers can identify it.

Existing third-party plugins for emoji and math that are already registered on the engine must continue to work correctly in non-slide mode. In slide mode, the hook should silently take over those rendering paths without the caller having to deregister the original plugins.

Finally, the library must expose a stable entry point — referenced in the VS Code extension manifest — so that the host can obtain the hook through a single standard call rather than importing internals directly.

One extra detail from the questions that came back: this is decided per render by looking only at the document’s leading front-matter block, not by any global editor state. The front-matter has to match `/^---\s*([^]*)?(?:-{3}|\.{3})\s*/`, and inside that body the line `marp: true` has to be present as its own line, matched by `/^marp\s*:\s*true\s*$/m`. If either part is missing, we just render as plain Markdown and do not inject any slide markers. So this is specifically a YAML front-matter block at the start of the Markdown source containing the line `marp: true`, detected by regex `/^---\s*([^]*)?(?:-{3}|\.{3})\s*/` combined with `/^marp\s*:\s*true\s*$/m`.

Also, the actual Marp instance is configured as `container: { tag: 'div', id: 'marp-vscode' }`, so the wrapper element in slide output is `<div id="marp-vscode">`. The stylesheet marker the preview should look for is `<style id="marp-vscode-style">...</style>`, and that gets prepended into the token stream by the custom core rule `marp_vscode_style`. Same point in shorter form since people asked for the exact strings: `<div id="marp-vscode">` (the Marp container configured with `{ tag: 'div', id: 'marp-vscode' }`) and `<style id="marp-vscode-style">` (prepended by the `marp_vscode_style` core rule).

On the emoji side, if the engine already has a third-party emoji renderer rule at `renderer.rules.emoji`, we do not remove it up front. We wrap it in a new function instead. That wrapper checks the per-instance `marpVscodeEnabled` symbol flag. If that flag is true, it forwards to `renderer.rules.marp_emoji`, which is the slide engine path and emits `data-marp-twemoji`. If the flag is false, it calls the original emoji rule. That override is installed lazily in the `marp_vscode_postprocess` core rule and only applied once.

For math, the toggle behavior is explicit so we do not fight with existing plugins like `@neilsustc/markdown-it-katex`. In `marp_vscode_toggle`, when `marp: true` is detected we call `md.block.ruler.disable('math_block', true)` and `md.inline.ruler.disable('math_inline', true)`. When marp is off, both are enabled again with `true` in silent mode. The intent there is just to let slide mode own math when active, and hand parsing back to the external plugin when slide mode is inactive.
