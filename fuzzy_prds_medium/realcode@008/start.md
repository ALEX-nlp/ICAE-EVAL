## Product Requirement Document

We need a lightweight server toolkit that lets teams expose shell scripts over HTTP without writing any server-side glue code. The system reads a declarative configuration file listing endpoints, validates each entry up front so broken configs fail immediately rather than silently at runtime, then maps each entry to the right HTTP verbs. When a single entry declares multiple verbs, each verb gets its own route. Before any command is executed, the toolkit translates the raw command string into the exact executable and argument list required by the underlying operating system, handling both Windows-family and Unix-like hosts transparently. For concurrent execution, the system assigns visually distinct log stream colors by cycling through an internal palette so operators can tell outputs apart at a glance. The validation layer enforces strict rules on the three required parts of every endpoint definition: what HTTP action to listen on, what URL path to serve, and what shell instruction to run. The color cycling logic lives in a dedicated logging subsystem that wraps the core generator. The end goal is a contract-driven core that any HTTP layer can drive directly, with all error conditions surfaced as clean, neutral status strings rather than raw runtime exceptions.

One other pass on the validation details since the team asked for the exact edges here. For the HTTP action, we only accept exactly four verbs: GET, POST, PUT, DELETE. The comparison is case-insensitive, so 'get', 'Post', 'PUT' all pass. That field can come through as a single string or a non-empty array of such strings. An empty array, empty string, number, null, or plain object should fail validation.

For the path, the rule is /^\/[a-zA-Z0-9\-\/]*$/. So it has to start with a forward slash, and after that it can only use ASCII letters, digits, hyphens, and additional forward slashes. A trailing slash is fine. A dot, question mark, dollar sign, or any other character makes it invalid, and non-string values plus the empty string are invalid too. For the shell instruction, we’re keeping that simple and strict: it must be a non-empty string (joi.string().required()). The empty string, a number, null, an object, or an array should fail, and any non-empty string value passes.

Also, on the command normalization piece, the behavior needs to stay exact. In src/server/process/normalizeArgs.js, the normalizeArgs(command) function branches on process.platform === 'win32'. On win32 platform the result is: { file: process.env.comspec || 'cmd.exe', args: ['/s', '/c', '"<command>"'], options: { windowsVerbatimArguments: true } }. The command is wrapped in double quotes verbatim. If a comspec value is supplied via the request it replaces cmd.exe. The output JSON key order is file, args, options. On any platform other than win32 the result is: { file: '/bin/sh', args: ['-c', '<command>'], options: {} }. The options map is an empty object. The output JSON key order is file, args, options.

And for the logging/color bit, the files are split up pretty cleanly: src/server/logPrinter/colorGenerator.js contains the *cycleColor generator; src/server/logPrinter/create.js wraps it and exposes log/logError/logStart/logExit functions; src/server/logPrinter/colors.js defines the default palette [chalk.yellow, chalk.cyan, chalk.green, chalk.magenta].
