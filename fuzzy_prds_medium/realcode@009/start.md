## Product Requirement Document

Hey team, we need a small to-do list core library built out — basically the same pattern we used for that notes feature a while back, where we kept the data layer totally separate from how it gets called. The idea is developers should be able to create individual to-do items, manage a whole list of them, and save/load that list without writing the same glue code everywhere. Items need a label and some kind of done/not-done flag. We also need to be able to flip that flag back and forth, pull out just the finished ones, and persist the whole thing so it survives app restarts. The entry point should read a command in and print results out — but critically the business logic itself shouldn't care about that I/O stuff at all, keep those concerns apart like we always do. One thing I keep seeing go wrong on other projects is when the storage piece is hardcoded in — make sure that's swappable. Also the default state for a new item when you don't say whether it's done or not should be obvious but let's make sure it's explicit. Output format for each operation needs to match what the test harness expects exactly. Malformed inputs shouldn't blow up the whole thing either. Should be compact, not over-engineered. Reach out if anything's unclear!

One quick follow-up since a couple details came up. For a task item, if it's constructed without the completion flag, it needs to default to false. If that flag is provided at construction time, use it exactly as given, so true stays true and false stays false. In construct mode, the output needs to be exactly two lines and only two lines: first 'text=<the task text>', then 'completed=<true or false>'. No extra whitespace and no extra blank lines.

On serialization, this needs to stay symmetric. If we decode a JSON map into a task item and then encode it again, it should come back as a value-equal map with exactly the keys 'text' and 'completed' in that order, where 'text' is a string and 'completed' is a boolean. No extra keys and nothing missing. For serialize mode specifically, print exactly three lines: 'text=<value>', 'completed=<true|false>', then 'json={"text":"<value>","completed":<true|false>}'. That JSON needs the keys in the order text first, then completed, with no extra spaces.

Also for toggle mode, print the starting state as 'initial=<true|false>', then one line per toggle as 'after_<N>=<true|false>' with N starting at 1 and increasing from there. How many times to toggle comes from the 'times' field in the command input.

And just to restate the structure piece a little more plainly, the pattern I was referring to is Separation of Concerns / SOLID: the execution adapter is the main entry point that reads JSON commands from stdin, and that should stay physically and logically separate from the core domain models (TaskItem, TaskCollection) and the persistence abstraction (StorageInterface). Last thing, if you're running the suite directly, it's via 'bash rcb_tests/test.sh'. It accepts '--cases-dir <subdir>' and defaults to test_cases. For each case it writes the raw stdout of the program to 'rcb_tests/stdout/<cases-dir>/<filename_stem>@<zero_padded_index>.txt', and those outputs are namespaced by cases-dir so runs in different dirs don't overwrite each other.
