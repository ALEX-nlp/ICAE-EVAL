## Product Requirement Document

Hey team, we need a tool that can take a bunch of SQL files and bundle them into a deployable database package. Think of it like a compiler for database projects. Developers are constantly complaining that they have to manually track which SQL files are 'real' schema vs one-off deployment scripts, and it's causing broken deployments. We also need it to handle the case where one database project depends on another — similar to how we handled cross-module dependencies in that shared-library linking feature we shipped last quarter, same general idea.

The tool should produce some kind of build result that tells you whether the build succeeded, what tables and views ended up in the package, whether pre/post deployment scripts are attached, and any errors. If something references a database object that doesn't exist, it should fail loudly with a stable error code (not just a stack trace) so CI pipelines can parse it.

We also need a way to explicitly say 'this file is in the folder but don't compile it into the schema' — useful for utility scripts that live alongside the project. The output format needs to be consistent so our test harness can do exact string comparisons. The whole thing should be runnable from the command line with JSON piped to stdin, and there should be an automated test suite that runs all the cases and saves raw stdout to files for diffing.

A couple more specifics from the questions that came in. The build output needs to be totally rigid because the harness is doing exact comparisons. Per build it is exactly seven lines, in this order: exit_code, package_exists, tables, views, pre_deploy_script, post_deploy_script, diagnostics, and each line is `key=value` with a trailing newline. For tables and views, each one gets its own line as a comma-separated list of sorted (alphabetically) fully-qualified names, for example `tables=[dbo].[Table1],[dbo].[Table2]`. If there aren’t any, leave the value empty, like `tables=`.

Also on failures, if a schema object references something missing, the diagnostics line must be exactly `diagnostics=error_code=SQL71501;object=<referencing-object>;referenced=<missing-object>`. The error code is always SQL71501. No stack traces, exception types, file paths, or extra prose mixed in there. If the build is clean, just emit `diagnostics=none`.

On excluding files, there are two knobs and they mean slightly different things. `remove_build` accepts a list of relative paths or glob patterns like `ReferenceProj/**/*.*` and removes matching files from the build set. `none` accepts a list of explicit file paths and classifies those files as non-schema content. In both cases, any objects from those files should be absent from the compiled package. And just to be explicit, anything listed in `pre_deploy` or `post_deploy` is also excluded from schema compilation, but still attached to the package as payload.

For project references, there are two flavors. `project_references` is a list of objects with a `path` and `sql_files` map, where the `path` is absolute and outside the current project root. Those referenced objects are available for dependency resolution, but they do NOT show up in the current project’s package output. `nested_project_references` works the same way except the referenced project lives under the current project root. In that case you also need `project_reference_paths` listing the `.sqlproj` paths, and you should use `remove_build` with a glob like `ReferenceProj/**/*.*` so those files don’t get compiled into the current project’s own schema. This is the same scope for Feature 4 (Project Reference Resolution): the `project_references` and `nested_project_references` input fields, which supply schema objects from external `.sqlproj` projects to satisfy dependency resolution for the current project build without including those external objects in the current package output.
