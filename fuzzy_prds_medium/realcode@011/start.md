## Product Requirement Document

Hey team, we need to build out this HTML assertion utility we've been talking about. Basically it's a tool that lets devs write checks against rendered markup output without having to do all the messy DOM traversal stuff manually every time. The core idea is you feed it some HTML and tell it what you expect to find, and it either confirms the match or tells you what failed in a consistent, readable way.

We need it to handle the usual stuff — checking that certain text shows up, that specific element types exist, that links point to the right places, that CSS classes are on the right elements, that attribute values are exactly right. There's also a scoping thing — like the validation flow we used in that last auth module — where you can narrow down to a specific part of the document before running your checks. And we should support picking a specific element from a list by position, not just "find any match."

There's also that reusable charset check that keeps coming up in onboarding issues — we should make that a first-class thing so people stop reinventing it.

The output format needs to be totally deterministic and machine-parseable so CI pipelines can consume it without custom parsing logic. When something matches, say so cleanly. When it fails, tell us what condition failed without leaking internal error details.

Input should come in via stdin as structured data, one check per invocation.

One follow-up from the questions that came in: the adapter reads a single JSON object from stdin, and that object always includes a 'document' key with the raw HTML string. After that, the keys vary based on what kind of check is being asked for, using only 'text', 'selector', 'scope_selector', 'selection_selector', 'selection_mode', 'position', 'attribute', 'value', 'class', 'href', or 'charset'. The feature is inferred from the combination of keys that shows up in that JSON object, so there is no explicit 'feature' or 'command' key.

For output, keep it very strict. On success, it should be key=value lines printed to stdout. A matched result always starts with 'result=matched' on the first line, followed by the relevant context fields in the exact order for that feature. For example, a text presence check outputs: 'result=matched\ntext=<fragment>\n'. All output lines end with a newline, and the entire output ends with a trailing newline. On failure, the output must start with 'error=assertion_failed' on the first line, then 'condition=<condition_name>' on the second line, then the relevant selector or text fields. The condition names are fixed per feature: 'text_present' for text checks, 'selector_present' for selector checks, 'text_present_in_scope' for scoped text checks, 'attribute_value_matches' for attribute checks, 'class_present' for class checks, 'link_target_present' for href checks, and 'text_present_in_selection' for ordered selection text checks. Also just to be explicit, no internal exception details should appear in output.

On the scoping behavior, 'scope_selector' is what narrows the document to the subtree(s) matching that CSS selector before the main assertion runs. For selector presence in feature 2, the presence check runs within that scoped subtree. For text presence in feature 3, the text fragment is searched within the rendered content of the scoped elements. Same general behavior applies to the scope_selector mechanism described in features 2, 3, 4, 5, 6 — it narrows the document to a CSS-selected subtree before running the primary assertion, and its value must be echoed in the output as 'scope_selector=<value>'. When that field is present in output, it goes before the 'selector=' or 'text=' line.

And on picking one element out of a set, 'selection_mode' controls how the chosen element is taken from the results of 'selection_selector'. It accepts exactly 'first', 'last', and 'index'. 'first' picks the first matched element, 'last' picks the last matched element, and 'index' picks the zero-based element at the integer index given by the 'position' field. When these are present, the selection_mode and selection_selector values should be echoed in the output too, and 'position' is only required and only output when selection_mode is 'index'.
