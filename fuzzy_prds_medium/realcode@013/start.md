## Product Requirement Document

Hey team, we need a small utility built for managing a single piece of state that gets updated through a series of async steps. The core idea is that we have one value at a time, and we want to push a bunch of update instructions at it — things like setting it to a fixed number or doing some math on it — and have those run strictly one after another, never overlapping. The tricky part is we want to observe each transition as it happens, but only when the value actually changes. Also, and this is the part that burned us last time on the login module flow — if one of the update steps blows up at runtime, the whole pipeline shouldn't die. It should log the problem in some neutral, non-implementation-specific way and keep going with whatever the last good value was. At the end we always want to know the final settled value. We'll need a test runner script that can point at different folders of case files without the outputs stomping on each other. Output should go somewhere predictable so we can diff it. The implementation language is Dart. Keep the I/O wiring separate from the actual queue logic — don't want a god file situation like we ended up with on that other project.

One small follow-up from the team questions: for division, this needs to be truncating integer division only, so use the integer quotient with any fractional part discarded, equivalent to Dart's `~/` operator, and there shouldn't be any floating-point intermediate values involved. If we hit divide-by-zero, the emitted error line needs to be exactly `error=division_by_zero`. The category string is `division_by_zero` and nothing else. It's fine if the core layer throws an idiomatic Dart exception, but the adapter layer is the part that maps that to this neutral category string.

Also, on the state output behavior, a `state=<value>` line should only be emitted when the newly produced value actually differs from the current state. If an action lands on the same value as the current state, then that action does not emit a `state=` line. After all actions are processed, we always print exactly one `final=<value>` line showing the last successfully produced state value, and that line always appears even if there were errors along the way.

For the runner/output path, each test case result file should go to `rcb_tests/stdout/<cases-dir>/{filename.stem}@{case_index.zfill(3)}.txt`, where `cases-dir` is the name of the case directory and defaults to `public_test_cases`, `filename.stem` is the JSON file name without extension, and `case_index` is the zero-padded three-digit index of the case within that file. The important bit there is that outputs are namespaced by `<cases-dir>` so runs against different folders never overwrite each other.

And just to restate the failure behavior in the same spirit as the original ask, we do want the error-containment and recovery pattern described in Feature 2 of start.md: catch runtime exceptions per action, emit a neutral `error=<category>` line, preserve the last good state, and continue the queue, basically mirroring the prior flow where failures were isolated instead of halting the whole pipeline.
