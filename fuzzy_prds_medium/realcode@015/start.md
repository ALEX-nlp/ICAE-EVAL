## Product Requirement Document

Hey team, we need to build out the qBittorrent metrics exporter adapter thing we talked about last sprint. Basically it needs to talk to the qBittorrent web UI and push stats into Prometheus. There are a bunch of operations it needs to handle — stuff like figuring out the right URL to hit based on deployment config, building metric objects with reasonable defaults so we don't have to specify everything every time, and converting those into the Prometheus family format. It also needs to safely pull the category list and torrent inventory without blowing up if the remote side is unavailable. We need filtering too — by lifecycle state and by category — and the category one has that edge case we handled similarly in the login module, where blank values should fall through to a special bucket. There's also a count metric builder that should tag things with state, category, and server info. Finally we need the full server health snapshot — connectivity, firewall status, DHT, byte counters, the works — output as an ordered list of metrics. The output format for each operation should match what the test harness expects. Make sure the server label logic is consistent everywhere and counters vs gauges are used correctly. Should be clean, separated by concern, not one big file.

One small follow-up after the questions that came in: for the endpoint URL, use `https` when the `ssl` config flag is explicitly set to true OR when the port is exactly `443`. In all other cases it uses `http`. That scheme choice is independent of hostname. Also, the server label should be built as `host:port` when no `url_base` is provided. When a `url_base` is present and non-empty, the server label becomes `host:port/url_base` like `localhost:8080/qbt/`. That exact same server label string should be reused everywhere we emit a `server` label so it stays consistent across all the metric outputs.

A couple format/details things too: if we create a metric record without explicit metadata, apply the defaults quietly as `labels` = `{}`, `help` = `""`, and `metric_type` = `"gauge"`. For metric family samples, a `gauge` keeps the sample name exactly the same as the metric name, but a `counter` uses the metric name with `_total` appended, so `qbittorrent_dl_info_data` becomes `qbittorrent_dl_info_data_total`, while the family name itself still stays `qbittorrent_dl_info_data`. And anywhere we serialize label dictionaries, make that compact JSON with no spaces after separators and keys sorted alphabetically, including `sample_labels` in family output. The example to match is `{"category":"movies","server":"localhost:8080/qbt/","status":"downloading"}`.

Also on the health snapshot side, `connected` is `True` when `connection_status` equals `"connected"`, and `firewalled` is `True` when `connection_status` equals `"firewalled"`. For any other value, both should be `False`. Those should come through as boolean gauge values rendered in Python style as `True`/`False` in the output.

And for the category filtering edge case, the blank/empty string torrent category behavior should match the earlier pattern we discussed: when `Uncategorized` is requested, blank/empty string category values should be treated as equivalent to `Uncategorized`, not left out separately.
