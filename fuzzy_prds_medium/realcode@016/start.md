## Product Requirement Document

hey team, we need a PNG image comparison tool — basically something devs can drop into their screenshot testing pipeline so they stop writing all that manual pixel-diffing boilerplate every time. the core idea is: give it two images, get back whether they look the same or not, and optionally spit out a diff image that highlights what changed. we've had a bunch of regressions slip through lately because people are copy-pasting their own ad-hoc comparison scripts and the tolerance logic is all over the place.

for the comparison side, there should be a few modes — something strict for exact matching, something more lenient for when pixel values are close enough that a human wouldn't notice, and also a special case for that cursor/caret flicker issue we keep seeing in the editor screenshot tests (you know the one, same kind of ignore logic we used in that rendering QA spike last quarter). images can come in either from disk or from memory, similar to how we handled the two transport modes in the login asset loader.

for the diff output, matching pixels should be preserved from the reference, changed areas get painted with a configurable highlight color, and you can get the result back as a file or as raw bytes. if someone tries to combine modes that don't make sense together, we need a clean error back — not a crash. output should be structured and machine-readable so the CI pipeline can parse it. keep the core logic separate from the CLI wiring please.

one extra note from the questions that came in: by default, if you call this with no options, we are not doing exact RGB matching. the default is a perceptual/human-visibility model, so tiny pixel differences that a person wouldn’t actually notice should still be treated as equal and can return look_same. that’s separate from color_tolerance, it’s just the built-in default behavior. only when strict_pixel_match=true is set do we switch to the truly exact path where any RGB difference counts, even 1 channel off by 1. that’s why different-unnoticable.png comes back look_same with no options, but different with strict_pixel_match=true.

also, if somebody passes both strict_pixel_match=true and color_tolerance at the same time, we should fail fast before doing anything with the images at all. no loading, no comparison, no diff generation. the response needs to be exactly two lines: 'error=conflicting_comparison_options\n' followed by 'options=strict_pixel_match,color_tolerance\n'. keep the option names in that fixed order.

for the caret handling, the ignore_single_vertical_caret option is only meant to suppress the editor text-cursor blink artifact, and it should be pretty literal. after the normal comparison, we only return look_same if every differing pixel is confined to exactly one contiguous vertical column: all diffs share the same single x-coordinate, and that column runs the full image height with no gaps. if diffs show up on two or more x-coordinates, or the column is broken/non-contiguous, the result is different. same idea as the cases the team was asking about with the caret/no-caret/two-caret/broken-caret PNGs.

last thing on diff output sizing: if the two inputs have different dimensions, the generated diff image should use the maximum width and maximum height of the two inputs, so max(ref_width, current_width) x max(ref_height, current_height). anything that exists in the larger image but not the smaller one should just be painted with the highlight color as a differing region, and then inside the overlapping area we compare pixels normally.
