## Product Requirement Document

Hey team, we need to build out that ripple/wave click feedback thing for our component library — you know, like the material design effect where you tap a button and a circle spreads out from your finger. We talked about this in the last sprint retro. The main pain points we've been hearing from devs are: (1) they keep rewriting the same geometry math every time they need this effect on a new component, (2) the thing breaks in older app shells — I think Jess mentioned something about it not working in the legacy plugin system, same kind of fix we did for that login directive back in Q2, you can check that compatibility logic for reference — and (3) there's no consistent way to scope which elements actually respond to taps vs. which ones should be ignored (like disabled buttons). The library needs a clean adapter so we can test it from the command line by piping in JSON and reading the output. Default behavior should just work out of the box with no config needed. Visual timing, colors, and whether clicks bubble up should all be tunable. We also need to handle the case where someone wants to name their activation zones so multiple independent ripple regions don't step on each other. Please make sure the test harness writes output files per case so QA can diff them directly.

One extra pass on the details the team asked for: if nothing is overridden, the CLI output needs to report the defaults exactly as directive=wave, color=currentColor, initialOpacity=0.2, finalOpacity=0.1, duration=0.4, dissolveDuration=0.15, easing=ease-out, cancellationPeriod=75, trigger=auto, tagName=div, disabled=false, respectDisabledAttribute=true, stopPropagation=false. Also on trigger handling, the string 'auto' should be classified as kind=automatic with requiresMatchingNamedActivator=false. Boolean true or false should be classified as kind=boolean with requiresMatchingNamedActivator=false. Any other string should be classified as kind=identifier with requiresMatchingNamedActivator=true, which means there has to be a matching named activator.

Also clarifying the boundary behavior since that came up: when the trigger is 'auto', the boundary attribute is the boolean true, so it prints as boundary=true. When the trigger is a named string identifier and not 'auto', the boundary attribute should hold that exact string value, so for example trigger='stringId' produces boundary=stringId.

For the geometry bit, relative coordinates come from subtracting the element box s top and left offsets from the pointer's y and x respectively, exactly as relativeX = pointer.x - elementBox.left, relativeY = pointer.y - elementBox.top. Coordinates are NOT clamped, so if the pointer lands outside the element boundary the result can be negative. For the ripple size, compute the Euclidean distance from the origin point to all four corners of the element rectangle at (0,0), (width,0), (0,height), (width,height), then take the maximum of those four distances as the radius. The diameter is just radius multiplied by 2. Any floating-point output should be printed as raw JavaScript-precision numeric strings, for example radius=106.06601717798213.

And one more compatibility note tied to the older app shell question from the original ticket: we do need the host runtime lifecycle compatibility detection that maps 'modern' and 'legacy' runtime labels to their mount/update lifecycle phase hook names, specifically mounted/updated vs inserted/componentUpdated.
