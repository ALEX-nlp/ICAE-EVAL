## Product Requirement Document

Hey team, we need to build out that virtual DOM rendering engine we discussed last sprint. The basic idea is developers write a description of what they want the UI to look like (as a data structure), and the engine figures out the minimal set of real DOM changes needed — no more throwing away and rebuilding everything on every state change. We've had complaints from users about UI flicker and lost focus states, so this is a real pain point we need to fix.

The engine should support composing reusable view pieces inside each other, kind of like how we handled the nested widget logic in that dashboard module. References to live nodes need to work too — devs want to grab a handle to a specific rendered element after the fact.

One tricky bit: when children get reordered (think drag-and-drop lists), we want to move existing nodes rather than rebuild them. The scheduler piece also needs to handle both the 'batch it up for the next frame' mode and the 'do it right now' mode, including when updates trigger more updates mid-flush.

The whole thing needs a clean JSON-in / text-out interface for testing purposes. Error cases should come back as a simple category string, not a stack trace or anything runtime-specific. Refer to the serialization format we agreed on — stable node identity across renders is the key observable thing here.

One extra detail the team asked about on the test output side: every real DOM element node gets a stable identity token the first time it is serialized. The format is #n<integer>, with a monotonically incrementing counter, so you’ll see values like #n1, #n2, #n3. After a node gets one of those, we keep reusing that exact token for that same node at every later observation point. That’s the thing that lets us tell the difference between something that was patched in place and something that got replaced. The element serialization format stays the one from start.md, so think <tag#token attr="v">children</tag>, and more explicitly <tag#token attr1="v1" attr2="v2">children</tag>. Attributes should always come out alphabetically sorted by attribute name no matter what order they showed up in the input, just so the output is deterministic and not dependent on input ordering. Also keep the full probe block format as-is, including the conditional refs section.

A couple reconciliation specifics too. For child elements or components with a 'key' field, matching is by key identity rather than by position. So if the same key moves to a different spot after a re-render, we move the existing node instead of rebuilding it, which means it keeps the same identity token, and refs attached to keyed components should still resolve to the same instance after reordering. On the component side, comp nodes in render specs should follow the same composition rules we talked through before. A component definition with 'update: true' means if the same component definition by name shows up again in the same position, we patch that existing instance in place and reuse its DOM element, so same identity token. If update is false or omitted, then we don’t patch in place — we throw away the old instance and build a fresh one, which means the replaced element gets a new identity token.

Last thing, root swaps are a hard no. If an already-rendered component would need its root element to become a different tag, we reject that update completely and return the error category string 'unsupported_root_node_change'. In that case the whole block stream should be replaced by exactly one line: error=unsupported_root_node_change.
