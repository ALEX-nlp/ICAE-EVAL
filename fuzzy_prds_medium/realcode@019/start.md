## Product Requirement Document

Hey team, we need a small utility layer for our PHP libraries so they can announce deprecated code paths without stepping on the host application's toes. The core idea is that a library just declares 'this thing is deprecated' with a stable URL and a message, and the host app decides what to do with it — or nothing at all if it hasn't opted in. Right now our libraries are either throwing errors directly or spamming logs in ways consumers can't control, and we keep getting complaints about noise in test suites and production logs.

The layer should support a few different modes of operation — counting only, routing to the runtime error channel, or sending to a structured logger — and the host should be able to flip between these either in code or via some kind of environment config (think the same pattern we used in that deployment-selector mechanism from the auth service refactor).

There should also be a way to silence specific deprecations or whole vendor packages without affecting whether they get counted. And we need the repeated-trigger noise problem handled by default, with an escape hatch to turn that off.

One tricky bit: some deprecations should only fire when a consumer outside the owning package calls into it — internal calls within the same package should stay quiet. Also need a reset/disable capability so test suites can clean up state between runs. Please keep the internals cleanly separated — routing, counting, and suppression logic should not be tangled together.

One extra detail that came up: if nobody has selected an activation mode yet, either in code or through the environment selector, the manager should be a full no-op. In that state, triggering a deprecation should neither record a counter nor notify any backend, and the summary should come out as unique_count=0 followed by tracked=(none). Related to activation, the enable field should be able to take a list of modes so we can turn on more than one at the same time, like 'track' plus 'trigger' or 'track' plus 'psr'. Also, the 'env' field in the JSON scenario input accepts a string value ('track' or 'trigger') to activate a mode without programmatic calls — see feature6_env_activation.json and the env activation logic in the execution adapter. One important behavioral point here: once any mode is enabled, counting is on as well.

On the tracking side, when 'track' mode is enabled, every trigger should increment a per-identifier counter keyed by the deprecation's link (URL identifier). The unique_count in the summary is the sum of ALL per-identifier counters, not the number of distinct identifiers, so repeated hits to the same identifier keep growing that identifier's count independently. The identifiers themselves should still be listed once each, in first-seen order. Deduplication should stay ON by default, which means the backend only gets notified on the FIRST occurrence of each identifier within a run. After that, later occurrences still increment the counter but do not produce another notification block. If we want a notification every single time, deduplication has to be explicitly set to false.

Also worth spelling out the message behavior a little more clearly: the message field is a printf-style template, and any trailing args values are substituted before the message reaches a backend. %s consumes a string argument, %d consumes an integer argument, and multiple placeholders consume multiple arguments in order. If there are no placeholders, the template is emitted verbatim. Whatever that formatted result is, that's what should show up in the notice block's message field.

For reset behavior, disabling the manager should both turn tracking off and reset all per-identifier counters to zero. The key nuance is that identifiers already seen during the run should still remain listed in the summary, still in first-seen order, just with count=0 each, and unique_count should drop back to zero. That's the cleanup behavior tests were asking for. The 'disable_after' boolean field in the scenario input should trigger that disable+reset behavior described in feature8_disable_reset.json: counters are zeroed but tracked identifiers remain listed with count=0.

Last formatting note on the output: the summary always comes after all notice blocks. It starts with a line exactly like 'unique_count=<N>'. If any identifiers were tracked, then we print one 'count[<identifier>]=<n>' line per tracked identifier in first-seen order. If nothing was tracked, or everything got suppressed via ignore_links, then instead of count lines we print a single line 'tracked=(none)'. The one exception is after a disable/reset: in that case, previously tracked identifiers should still be shown with count=0, so tracked=(none) is NOT used there.
