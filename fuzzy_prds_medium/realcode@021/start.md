## Product Requirement Document

hey team, we need a small utility for computing workspace navigation on multi-monitor setups — basically the same kind of thing we did for the session switching logic a while back, same general idea. the core ask is: given where the user currently is on their focused monitor, figure out which workspace they should land on when they hit 'go forward' or 'go back'. there's a wrapping/cycling mode some users want, and also some users have complained about accidentally spawning new empty workspaces they didn't mean to open, so we need toggles for that. the tricky part is around the edges — like what happens when you're at the very end of the id range, or when other monitors are 'in the way' numerically but shouldn't actually block navigation on your monitor. the tool should accept a JSON blob on stdin and spit out a single line with the result. errors should come out clean, not as stack traces or anything noisy. keep the core math totally separate from the input/output stuff — we may want to reuse it later. refer to the boundary handling we agreed on in the workspace id spec for min/max values. should be fully testable with a shell script that runs all the case files. let me know if anything is unclear.

one quick follow-up after the team questions: workspace ids are bounded pretty simply. the lowest valid workspace id is 1, and workspace ids must be positive integers. so if we're searching for a new empty workspace in the 'previous' direction, that search must never go below 1. if the active workspace id is already 1 and no lower free id exists, the active id is returned unchanged. on the top end, the highest valid workspace id is 2147483647, the largest representable signed 32-bit integer. when searching for a new empty workspace in the 'next' direction, the search stops at this ceiling. if no free id exists at or below this maximum, the active id is returned unchanged.

also, on the behavior side, the intended pattern is the GNOME-style workspace switching pattern described in the PRD's Background section: navigating along a focused monitor's sorted workspace id list, with boundary behavior (empty creation or cycling) when falling off either end. related to that, the order of ids in the monitor_ids input does not matter. the library treats monitor_ids as a sorted set — it always sorts them internally before performing any navigation logic. navigation steps to the immediately next or previous element in the sorted order.

for the option interaction, cycle takes precedence over both no_empty_before and no_empty_after. when cycle is true and the user is at the boundary of the focused monitor's list, navigation wraps within the monitor regardless of whether no_empty_before or no_empty_after is also true. and if cycling is enabled and the focused monitor has only a single workspace, both 'next' and 'previous' wrap onto that same workspace — the active id is returned unchanged, basically wraps to itself.

last piece is the test runner detail since that came up: the entry point is 'bash rcb_tests/test.sh'. it reads every *.json case file from a case directory (default: public_test_cases) and accepts --cases-dir <subdir> to specify an alternative. for each case it pipes the 'input' object to the adapter and writes the adapter's stdout to 'rcb_tests/stdout/<cases-dir-basename>/{filename.stem}@{case_index.zfill(3)}.txt'. outputs are namespaced by the case-directory basename so different runs never overwrite each other.
