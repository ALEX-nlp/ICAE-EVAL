## Product Requirement Document

Hey team, we need to build that controller helper we've been talking about for a while — the one that automatically wires up action method arguments from the incoming request instead of making every action body do it by hand. Right now devs are writing the same lookup-and-default boilerplate in every single action and it's becoming a real maintenance headache. The idea is that the method signature itself becomes the contract — if a param is there, it gets filled; if it's not and it's optional, it falls back gracefully; if it's not and it's mandatory, we reject cleanly with a useful error rather than blowing up silently.

One thing to keep in mind: we had a similar positional-vs-named resolution situation in the old login module flow, follow the same kind of ordering logic we used there for how positional slots get filled when some are missing in the middle.

Also, the output/adapter layer should stay totally separate from the core logic — same philosophy we've applied on other projects. The test harness should be runnable with a single bash command and write raw output files we can diff directly. Make sure the file layout reflects the complexity here; don't just dump everything in one file. We want this thing maintainable long-term.

One extra detail the team asked about: if any required or required_keyword param is missing from the request bag, the whole resolution should fail right there and we should not emit any partial bindings at all. In that case the error output is exactly two lines: `error=missing_required` on the first line, then `missing=<names>` on the second line, with the absent required param names joined by commas and sorted alphabetically.

Also, when we do print resolved bindings, they should always come out sorted alphabetically by parameter name, no matter how the signature was declared. For rendered values, string values should show up as `value="<the string>"` with the double quotes, while an explicit empty placeholder should be `value=<nil>` with no quotes. The source field should be `source=injected` when the resolver supplied the value from the bag or when it had to inject that empty placeholder, and `source=default` when we actually used the declared default.

On the positional behavior, the important nuance is that an absent optional does not always get to keep its default. If an absent optional sits positionally before a later parameter that is present, whether that later one is another optional or a required, we need to inject an explicit empty placeholder instead, using `value=<nil>` and `source=injected`. It cannot just fall back to its declared default in that case. Only a trailing absent optional, where there are no supplied parameters after it, should keep its declared default with `source=default`. Same point as the positional slot-filling algorithm from Features 5 and 6: absent optional parameters that sit positionally before a later present/required parameter are injected as explicit <nil> placeholders rather than keeping their defaults; only trailing absent optionals beyond the last supplied position keep their declared defaults.

And just to remove any ambiguity on keywords: required_keyword and optional_keyword are always matched strictly by name from the bag and never participate in the positional slot-filling behavior. A present optional_keyword is injected, an absent optional_keyword keeps its declared default, a present required_keyword is injected, and an absent required_keyword triggers that same `error=missing_required` flow described above.
