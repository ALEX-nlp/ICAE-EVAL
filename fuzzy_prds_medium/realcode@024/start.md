## Product Requirement Document

Hey team, we need a library that can read and write those Apple-style config files (the XML ones you see in iPhoto libraries, cookie exports, that sort of thing). The parsing side should handle all the usual stuff — nested objects, lists, numbers, yes/no flags, timestamps, raw binary blobs, and make sure text with special characters comes out clean. We had a similar entity-decoding requirement in the old login module, so just follow that same pattern. On the write side, it should spit out well-formed XML fragments or a full wrapped document depending on what the caller asks for. Collections need proper indentation (tabs, not spaces, like we always do), and dictionaries should come out in a consistent order so diffs are readable. Binary content should be line-wrapped the same way we handled it in the export pipeline. The whole thing should be callable from a small adapter that reads a JSON command off stdin and writes plain text to stdout — nothing fancy, just enough to drive it from tests. One thing I keep forgetting to mention: empty plists and comment-only documents should not blow up, they should just report back gracefully. Also the date handling needs to be solid — second-level precision, always UTC. Basically I want developers to stop hand-rolling XML walkers every time someone drops a .plist file on us.

One quick follow-up from the questions that came in. On the parse report side, boolean values should come through as the literal strings 'true' and 'false', so examples like first_album_master=true and second_album_slideshow_use_titles=false are exactly what we want to see. And when we write them back out, boolean true should emit a self-closing <true/> element and boolean false should emit a self-closing <false/> element.

Also wanted to pin down the date formatting a little more tightly. All date/datetime/timestamp values need to serialize as <date>YYYY-MM-DDTHH:MM:SSZ</date>, always second precision, always with the trailing Z for UTC. If we get a bare date string like '2020-01-02', pad it to '2020-01-02T00:00:00Z'. On the parsing side, date elements should also be reported back in that same ISO-8601 UTC form with second precision.

For parsed data elements in the data_elements report, the adapter should output three lines per key: '{key}_type=data', '{key}_bytes={N}', and '{key}_sha256={hex}', where N is the decoded byte count and hex is the lowercase SHA-256 digest of the decoded bytes. If the input is an empty <data/> element, that still counts cleanly as 0 bytes and the SHA-256 of an empty byte string.

For the album_library_overview report, please emit 'keys=' followed by the top-level dictionary keys joined with the pipe character '|', sorted in lexicographic order, all on one line. The example to follow is 'keys=Application Version|Archive Path|List of Albums|List of Keywords|List of Rolls|Major Version|Master Image List|Minor Version'. Same sorting rule applies any time we serialize a dictionary too: all keys need to be sorted lexicographically by their text value before emission so the output stays deterministic and diff-friendly no matter what order they came in.

And just to make the formatting piece unambiguous, indentation is a single tab character (\t) per nesting level. Array and dict children each sit one additional tab in from their parent opening tag, and the closing tag comes back to the parent’s indentation level.

Last bit is string/key text handling. During parsing, the XML entity decoding should follow the same behavior as before for '&amp;' → '&', and when serializing we need the reverse escaping of '<', '>', '&' in text values and key text.
