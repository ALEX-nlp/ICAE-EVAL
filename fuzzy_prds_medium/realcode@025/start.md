## Product Requirement Document

Hey team, we need a native interop utility library in Dart (the ffi package). Basically devs keep complaining that every time they need to talk to native code they end up writing the same allocation/cleanup boilerplate over and over, and then someone forgets to free memory on an error path and we get a leak. We want a clean solution for two main pain points: (1) string handling — converting text back and forth between Dart strings and native memory buffers, both for the UTF-8 stuff and the wide-char stuff Windows likes. Make sure edge cases like funky characters and embedded nulls work properly, similar to how we handled encoding boundaries in that auth module a while back. (2) resource lifetime management — we need scoped cleanup that works for sync code, async code, and when things blow up with an exception. There's also a reusable variant that some internal tooling relies on where the release callback might itself allocate things, so that needs to be handled cleanly. Oh and errors should come back in a portable, runtime-neutral format rather than whatever Dart happens to throw. The test harness reads JSON from stdin and writes plain-text key=value lines to stdout, so the I/O adapter needs to stay decoupled from the actual logic. Check the existing test files to understand the exact expected output shapes.

One extra pass on the details the team asked about: for UTF-8 text_to_native_units, the stdout shape needs to be three lines exactly as encoding=utf8, native_units=[...], and terminator=0. In that native_units list, include every encoded byte INCLUDING the trailing zero byte, so the zero terminator lives inside the array itself, like [72,101,...,10,0]. On the flip side, for UTF-8 native_units_to_text, if there is no 'length' field, decoding stops before the first zero byte, and that zero byte is NOT included in text_code_units or in the returned text. That output should be encoding=utf8, text_json="...", and text_code_units=[...]. If a 'length' field is supplied, then decode exactly that many bytes regardless of zero bytes inside them, and embedded zero bytes are preserved in the text so we can round-trip binary data with null bytes.

Also, for bad UTF-8 input, please normalize it to the portable error form, not a Dart exception name. The output in that case is exactly two lines: error=invalid_text_encoding and then encoding=utf8. Same normalization rule applies across the UTF-8 and UTF-16 cases we called out earlier: malformed byte sequences map to error=invalid_text_encoding, while boundary situations like negative length or null pointer should come back as their own portable error category fields rather than Dart-native exception types.

On resource scopes, in an asynchronous explicit scope, nothing should be released while the async body is still running, so released_during_scope=[] during execution, and only after the Future/async result completes should those resources show up in released_after_scope. The scope line there should be scope=asynchronous. For the ambient versions, use scope=ambient_synchronous for normal sync completion, scope=ambient_asynchronous for async completion, and scope=ambient_error when it exits via an error. The released_during_scope and released_after_scope behavior should match the explicit scopes. Calling this out because there was some confusion from the earlier wording around scope=synchronous, but for ambient sync the token is scope=ambient_synchronous.

And for the reusable scope behavior, yes, we do need the variant where the release callback can itself allocate and still get cleaned up before release() returns. The expected output there is four lines exactly: allocated_before_release=1, freed_before_release=0, allocated_after_release=2, freed_after_release=2. The callback allocation is tracked as part of the same release flow, so after_release includes both the original allocation and the one created by the callback. That’s the same reusable_scope behavior tied to Feature 12 and the NativeResourceScope variant.

Last thing, the adapter split stays important: read one JSON object from stdin, look at fields like 'encoding', 'direction', 'resource_scope', and 'completion', route to the right core domain method in NativeTextConverter or NativeResourceScope, and then format the result back out as plain key=value lines on stdout. The core domain itself should not know anything about dart:io, JSON parsing, or stdout.
