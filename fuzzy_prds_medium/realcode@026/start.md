## Product Requirement Document

We need a small scheduling library that lets developers rate-limit noisy event handlers without writing their own timer bookkeeping. The core idea: wrap an action or a value, declare a quiet window in milliseconds, and get back a version that only fires or settles after activity has paused for that long. There are two primitives — one for functions/actions that take arguments and run side effects, and one for values that track a frequently-changing source. Both should support leading-edge execution (fire immediately on the first event), trailing-edge (the default, fire after the quiet window), or both together. When both edges are enabled, a burst with only one trigger fires just once; more than one trigger fires on both edges. There must also be a hard upper bound option: if the source keeps firing continuously, execution is forced no later than that bound measured from the first pending trigger. Both primitives need explicit cancel (discard pending) and flush (run immediately) controls. After teardown, any pending action is suppressed — handle disposal the same way the value primitive handles its equality check, where cleanup happens predictably. For the value primitive, a custom equality function can suppress updates when the new value is considered the same as the current one. The library must use a deterministic virtual clock for testing rather than real wall-clock time.

One small thing the team asked about on the edge behavior: when both leading and trailing are turned on, a burst containing only ONE trigger fires exactly ONCE on the leading edge, with no extra trailing execution afterward. If that same burst has MORE THAN ONE trigger, then it fires twice total: once on the leading edge right away, and once on the trailing edge after the quiet window.

Also, just to make the default behavior explicit, trailing defaults to true. So if someone uses the debounced action without specifying options, it should behave as trailing-edge-only: the action fires once after the quiet window fully elapses with no further trigger, and it should use the most recent arguments.

And on maxWait, the reference point is the first pending trigger in a burst, not each later trigger. So even if the source keeps firing and keeps resetting the normal quiet window, execution still gets forced at maxWait ms after that first trigger, and when that happens it should run with the most recent arguments available at that moment.
