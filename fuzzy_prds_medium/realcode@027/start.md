## Product Requirement Document

Hey team, we need to wire up the OpenID Connect auth flow for our platform. Basically we want something similar to what the identity team built for the SSO login module a while back — you know the one that handles the provider redirects and session stuff. The core ask is: when a user hits sign-in, we redirect them to the provider correctly; when they come back from the provider, we validate everything before letting them through; and when they log out, we either kick them to the provider's logout page or just continue normally if that's not available. We also need to expose profile info (name, email, that kind of thing) and token details in a consistent way so downstream code doesn't have to care about provider-specific claim names. Oh and there's the key selection thing for verifying tokens — it needs to handle the different flavors we use. The failure cases are really important here, bad state or missing tokens should produce a predictable redirect so the frontend knows what to do. One thing I keep hearing from ops is that the defaults should just work out of the box without anyone having to configure every single endpoint manually. Can someone pick this up and make sure the output format matches what the test harness expects? Check how we did it before for the exact field names.

One extra pass on this since a few folks asked for specifics: for the out-of-the-box behavior, the client defaults should be scheme=https, port=443, authorization_endpoint=/authorize, token_endpoint=/token, userinfo_endpoint=/userinfo, jwks_uri=/jwk, and when the flow is client_defaults those need to come out as plain key=value lines. Same general idea for the rest of this work too: keep the stdout format as key=value lines across the feature coverage, and keep the same claim-to-field mappings the tests expect, including 'picture' -> 'image', 'access_token' -> 'token', and 'phone_number' -> 'phone'.

On the sign-in redirect, the output order matters. It should include status=302 first, then authorization_url built from https plus the issuer host plus /authorize, then client_id=1234, then response_type with the default being 'code', then scope=openid. After that, include login_hint, ui_locales, claims_locales, and response_mode only when they were actually supplied. State should be reported as 'generated_32_hex' after normalization, and nonce_length=32. Please keep that sequence exactly as expected.

For logout, if discovery gives us an end_session_endpoint, return status=302 and logout_url=<endpoint>, and include post_logout_redirect_uri=<uri> when that’s configured. If there isn’t an end-session endpoint available, including cases where discovery is off or not present, then it should just be status=200 and route=application.

Also, the failure behavior on callback needs to stay very predictable. These always return status=302, plus failure_url in the form /auth/failure?message=<msg>&strategy=openid_connect, and an error= field. Invalid state should be error=invalid_callback and detail=invalid_state with message=invalid_credentials. If response_type=code and the code is missing, use error=missing_authorization_code with message=missing_code. If response_type=id_token and the id_token is missing, use error=missing_identity_token with message=missing_id_token. Don’t leak exception class names or runtime messages into the output.

And just to close the loop on the callback path, the callback validation logic in the callback_request flow handler is the bit that needs to enforce the state/nonce matching and the response_type-based checks for code vs id_token presence. When response_type=id_token and state matches, a successful identity-token callback should output status=200, provider=openid_connect, uid=<sub claim>, name=<name claim>, email=<email claim>, with those values sourced from decoded_id_token.
