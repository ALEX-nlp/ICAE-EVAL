## Product Requirement Document

Hey team, we need to build out that auth strategy wrapper we've been talking about. Basically, devs are tired of hand-rolling all the boilerplate every time they wire up a new identity provider — they keep getting the URLs wrong, forgetting to validate tokens properly, and the user profile shapes coming back are inconsistent depending on which flow they used. It's causing a bunch of support tickets.

The library should be configurable once with the provider details and then just handle everything: figuring out the right endpoints automatically (but still letting you override them if needed), building the login redirect params based on what hints are available, doing the full token checking sequence, and spitting out a consistent user object regardless of whether it came from the legacy format or the newer OIDC style.

For the token checks, refer to the same claim-by-claim validation ordering we settled on during that login module refactor — same priority sequence, same error categories. The profile normalization should handle the 'pipe-delimited ID' edge cases the same way we discussed.

One thing that burned us before: the config setup must not touch the caller's original options object. That caused a really subtle bug last time in staging.

Please separate concerns properly — don't dump everything in one file. Each major responsibility should live in its own module. The adapter layer that handles the JSON in/out should be totally decoupled from the core logic.

A couple follow-ups from the questions that came in. On the derived provider URLs, there are five of them and they all come off the provider domain by default: expectedIssuer is https://<domain>/, authorizationURL is https://<domain>/authorize, tokenURL is https://<domain>/oauth/token, userInfoURL is https://<domain>/userinfo, and apiUrl is https://<domain>/api. All five can be individually overridden by passing explicit values in the config object, so the defaults should be helpful but not boxed-in.

Also want to restate the config behavior because this was one of the painful parts last time: the constructor must NOT mutate the caller's original options object. The strategy must copy configuration onto a separate internal object. The caller's original object must never gain derived endpoint fields (like authorizationURL, tokenURL, etc.) as a side effect of construction. The test verifies both that the strategy's config is a distinct object and that the caller's object is free of derived fields. Related to defaults, the anti-forgery state flag defaults to true when not explicitly provided. When the caller explicitly passes state: false or state: true, that value is preserved exactly as supplied.

For the auth params, the output order needs to stay fixed as connection, connection_scope, audience, prompt, login_hint, acr_values, max_age, nonce. Each is only included if it passes a type check (must be a string for text fields). connection_scope is only forwarded when connection is also present. max_age is only forwarded when numerically configured on the strategy itself (not per-request). nonce is only forwarded when previously recorded on the strategy's internal auth params (not per-request). If nothing is forwarded, output a single (none) line.

On token validation, please keep the exact sequencing we aligned on and don’t reshuffle it, because the error surfaced should come from the first failing check in that order. The sequence is: (1) token presence — error token_missing; (2) issuer present — error issuer_missing; (3) issuer match — error issuer_mismatch with expected= and actual= context; (4) subject present — error subject_missing; (5) algorithm allowed (RS256 or HS256 only) — error unsupported_algorithm with alg= context; (6) audience valid (string or array of strings) — error audience_invalid; (7) audience match — error audience_mismatch with expected= context; (8) authorized party when audience is multi-value: present — error azp_missing, matching expected audience — error azp_mismatch with expected= and actual=; (9) nonce if expected: present — error nonce_missing, matching — error nonce_mismatch with expected= and actual=; (10) exp is a number — error exp_invalid; (11) token not expired — error token_expired; (12) iat is a number — error iat_invalid; (13) if maxAge configured: auth_time present and numeric — error auth_time_invalid, not too old — error auth_time_expired. Same thing in shorthand from the PRD if that’s easier to follow while implementing: token_missing → issuer_missing → issuer_mismatch → subject_missing → unsupported_algorithm → audience_invalid → audience_mismatch → azp_missing/azp_mismatch → nonce_missing/nonce_mismatch → exp_invalid → token_expired → iat_invalid → auth_time_invalid → auth_time_expired.

Last bit on profile normalization: when no identities array is present, split the user_id or sub on the first '|' character to extract the provider prefix; if no '|' exists, provider is null/absent. That should match the legacy edge-case behavior we already talked through.
