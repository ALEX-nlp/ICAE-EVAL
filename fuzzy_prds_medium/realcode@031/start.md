## Product Requirement Document

Hey team, we need a utility that handles editable list fields inside our form system — basically the same pattern we figured out for the login module's field-registration logic, but applied to arrays. The core problem is that our devs keep writing tons of repetitive boilerplate every time they need a form with a variable-length list: they manually name each row, lose track of the indexes when rows are added or removed, and things go silently wrong in ways that are really hard to debug in production. We want something that wraps all that up cleanly.

It needs to support multiple ways to plug in the row UI (we have teams that prefer different coding styles), and it should shout loudly when someone wires it up wrong instead of just doing nothing. There's also a recurring complaint that devs can't tell how many rows are in the list unless they subscribe to the whole form value, which causes unnecessary re-renders — that needs to be fixed.

The list itself needs to support walking over items, transforming them, attaching a rule that checks the whole collection at once, and growing/shrinking the list. Oh, and if someone changes which list the field is pointing at, it should automatically follow the new one. Please keep the code organized sensibly — last time someone dumped everything in one file and it was a nightmare to maintain.

One small follow-up from the questions that came in: for Feature 1, the subscription_diff piece is specifically the logic that compares two optional subscription maps over a chosen key set to decide whether re-registration with the form engine is needed. It takes two optional maps, a and b, where each side can be null/absent or a key-to-flag object, plus the list of keys we actually care about. If exactly one side is present and the other is absent, they differ. If both are absent, they do not differ. If both are present, they only differ when at least one of the specified keys has a non-identical value between the two maps, and anything outside that compare list is ignored completely. It also needs to report whether each side existed via a_present and b_present.

Also confirming the row rendering part: we only support exactly three interchangeable render styles here, which are a component prop, a render-function prop, and a function passed as children. They all get the exact same field bundle, and for a static initial array each one is invoked exactly once. The reported item count lives on that bundle as length, and it needs to match the number of items in the initial array every time, including 0 for an empty array. That same length should always be there no matter how narrow the subscription is, so even if somebody only subscribes to something unrelated like 'dirty', they still get the correct item count from the initial array, and a pristine untouched array reports dirty=false.

For iteration, the helper should call the callback once per item, in order, with two args: the item's field name and the zero-based numeric index. The item s field name (the array's name followed by a bracketed zero-based index, e.g. 'foo[0]') and the numeric zero-based index are the two things passed through each time. The field name format is strictly: parentName + '[' + index + ']'. The helper itself returns undefined.

And on the warning behavior, there are still three and only three misuse cases we care about, and each one should emit exactly one diagnostic, not a pile of duplicates. Those are missing_form_context when there’s no enclosing form, missing_render_strategy when we’re inside a form but nobody supplied component/render/children, and missing_array_mutators when the form config doesn’t have the mutation helpers. The output should report the total count followed by each category, and the stable category names to use are 'missing_form_context', 'missing_render_strategy', and 'missing_array_mutators'.
