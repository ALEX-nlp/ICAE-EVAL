## Product Requirement Document

Hey team, we've been getting complaints from devs that writing tests for async streams is a total nightmare — people keep shipping bugs because they forget to check the last value, or tests pass even when the stream throws an error nobody read. We talked about this in the Q3 planning thing and agreed we need a small testing helper library for this.

Basically devs should be able to 'open a session' over a stream, pull out events one at a time, and have the library yell at them if they missed something. Similar to how we handled the observable wrapper in the notification service refactor — that same idea of a scoped block that auto-validates at the end.

A few things I know we need: some way to just grab whatever event comes next without caring what type it is, a way to say 'I'm done, throw away the rest', and a way to verify the stream hasn't produced anything yet. We also need it to not hang forever if the stream is slow — there was a whole incident last sprint where CI was blocked for 40 minutes because of that.

One thing I'm not sure about: how exactly the clock/timing enforcement works when someone passes zero as the wait window — does that disable it or error immediately? And what happens with leftover events vs explicitly draining them — are those treated the same? Please make sure the end-of-block cleanup behavior is clearly defined for all the exit paths (exception, explicit stop, etc.).

Quick follow-up after the team questions: for the timing bit, when timeoutMs is set to 0, the deadline is disabled entirely, so the wait stays pending indefinitely no matter how much virtual clock time advances. It does NOT trigger an immediate timeout. That’s the 'zero' mode. The only time we should abort a wait for timing is a nonzero timeoutMs with 'enforced' mode, and in that case it should report error=timeout with waited_ms=<window>. This is the same virtual clock setup we already talked about around timeoutMs, mode ('enforced' vs 'zero'), and virtual clock advances advance1Ms/advance2Ms so tests can be deterministic without real sleeping.

Also, to make the scoped behavior more explicit: the observation block pattern is observe(stream, block). It opens a scoped context, buffers all stream events, runs the caller’s assertions, and then does the unconsumed-events check when the block exits. If the block ends normally and there are still received events sitting in the buffer, that should fail as unconsumed_events, listing leftovers in arrival order as 'unconsumed=item:<value>', 'unconsumed=complete', or 'unconsumed=error:<category>'.

On the stop variants, these are not all the same. Plain 'stop' halts further collection but keeps whatever was already buffered, and those retained events still count for the end-of-block unconsumed check. So if nobody consumes them, the block fails with error=unconsumed_events listing each leftover. 'stop_and_drain' also halts collection, but it returns all remaining buffered events as 'remaining=...' lines and clears the buffer, so the block can end cleanly. 'stop_and_discard' halts collection and silently drops everything, emitting a single 'ignored' line. The unconsumed check is bypassed if stop_and_drain or stop_and_discard was used, because those clear the buffer, but it is NOT bypassed by plain stop.

One more detail on event reads: take_event should consume the next buffered event of ANY kind without asserting type, and yield it tagged as 'event=item:<value>', 'event=complete', or 'event=error:<category>'. It should never fail just because the kind was different than expected. By contrast, take_item, take_completion, and take_error are the strict ones, and if the next event is the wrong kind they fail with error=unexpected_event plus expected=<kind> and found=<actual>.

And for exceptions inside the caller’s code, those should not get swallowed. If the caller throws inside the observation block, that exception propagates out of the observation call, and the adapter renders it as 'error=block_exception'. Collection is torn down in that path too, so if anything checks liveness afterward with reportAfter=true, it should report collecting=false.
