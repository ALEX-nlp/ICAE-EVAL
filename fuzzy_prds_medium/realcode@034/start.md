## Product Requirement Document

Hey team, we need to get this route protection wrapper thing finished up. Basically the core problem is that right now anyone can just navigate directly to protected pages without being logged in, and we're getting complaints that users sometimes see a flash of protected content before being kicked out, or worse they just stay on the page in a broken state. We want something reusable so devs don't have to copy-paste the same auth check into every single screen component.

The wrapper should handle the usual stuff — send people to login if they're not authenticated, and make sure the URL they were trying to get to is preserved somewhere in the redirect so we can send them back after they log in. Similar to what we did with the compatibility logic on the login module last time, there's also a pending/loading state to think about — we don't want to redirect someone mid-authentication.

Also we need it to support more advanced access rules beyond just 'is logged in', like checking profile fields. And it needs to work for nested routes too, not just top-level ones. There's also a case where we explicitly do NOT want to record the attempted URL — some routes are sensitive and we don't want that leaking into query params. The wrapper should also pass through any props or metadata the underlying component needs. Let's make sure this doesn't fire duplicate redirects after mount either, that was a problem before.

One extra pass on the redirect details since a few people asked: when we do send someone away, the attempted URL needs to be URL-encoded into a query param named `redirect` on the failure destination. So if someone tries `/auth?test=foo` while unauthenticated, they should land on `/login?redirect=%2Fauth%3Ftest%3Dfoo`. That encoded value includes both the path and the original query string. We also do have the option to suppress that completely. When that setting is on, a denied navigation just goes to the failure destination with an empty search string and no `redirect` query parameter at all, matching the `no_redirect_back` policy. Also, the failure destination does not have to be a hardcoded string. It can be a function that gets current state and route params and returns something dynamic, like `/login/1` or `/login/0`, and in that case we still append the denied URL as the `redirect` query parameter.

Also worth calling out that this is not only the component wrapper version. There’s a route-entry guard version too, like an `onEnter` hook, and it should behave the same way on redirects, including the same return-location encoding. For that case, an attempted `/onEnter` should become `?redirect=%2FonEnter` on the redirect target. And if no explicit dispatch redirect action is wired up, redirects can still happen through router context directly. That path handling and search encoding should stay exactly the same there too, including `context_router_redirect`.

On the pending auth side, if authentication is in progress we should NOT redirect at all. The router stays on the attempted protected route with an empty search string. If there’s a custom loading component configured, render that. If there isn’t one, render an empty placeholder component, and that one receives no props, which shows up as `placeholder_props={}`. For authorization, this is broader than just checking whether a session exists. A predicate can inspect whatever fields it needs from the selected session profile, and it has to return true to allow access. If it returns false, we treat that as unauthorized and redirect, with the denied URL recorded the same way as above.

Last thing, this should stay aligned with the same session selector and auth state shape we already use in the other auth flows, especially how `authSelector` pulls session data and how `authenticatingSelector` indicates pending state. That matters for keeping behavior consistent around `start_authentication`, `login`, and `logout`, and for the harness policies like `requires_session` and `pending_custom_loading`.
