## Product Requirement Document

Hey team, we need to wrap up the i18n routing utility for the static export project. The core idea is that we have a translation catalog (a nested dictionary keyed by language code) and we need a small runtime library that handles all the typical locale stuff — picking the right language, building navigation links, rendering translated strings, that kind of thing. We talked about this back when we did the browser-detection piece, so carry that pattern forward here.

The tricky parts I keep hearing about from the frontend folks: when someone's browser reports a regional variant like 'en-US', we should gracefully handle that (I think we strip the region part? check how we did it before). Also there are some edge cases around what happens when the URL has a language code we don't recognize — we want a sensible fallback rather than a broken page.

There's also the language switcher component — it needs to be accessible (screen readers etc.) and correctly wire up the router navigation, including preserving any other query params already in the URL. Oh and template strings in translations should support some kind of variable interpolation.

The outputs need to match exactly what the test harness expects — I know there are some subtle formatting differences between the 'resolved via browser' path vs the 'normal config' path so please double check those. Reach out if anything is unclear.

One quick follow-up since a few details came back from the team. On the browser-default path, when use_browser_default=true and browser_language has a regional bit like 'foo-US', we should split on '-' and only use the primary subtag 'foo'. If that primary subtag is actually present as a key in the translations dictionary, that becomes the resolved default_language. If it is not in the catalog, we keep the configured default_language and use the non-resolved-via-browser output instead of the resolved one.

Also, the output shape really does change depending on whether that browser resolution actually worked. If use_browser_default is true and the browser language after stripping the region matches a catalog key, use the resolved format exactly as 'config_default_language=<lang>\navailable_languages=<sorted>\ntranslation.<lang>.<key>=<val>\n...' — note 'config_default_language' (not 'default_language') and all translation key-value pairs are listed in sorted order. If browser detection is turned on but the browser language does not match any catalog key, then use the unresolved format exactly as 'default_language=<lang>\nbrowser_language=<raw>\nbrowser_default_enabled=true\navailable_languages=<sorted>\n'. And when use_browser_default is false, use that same unresolved format with browser_default_enabled=false.

There are also three separate config error cases and they should be checked in order, since they are mutually exclusive. If the translations dictionary is empty, the output is exactly 'error=missing_translations\n' and nothing else. If default_language is an empty string, the output is exactly 'error=missing_default_language\n'. If default_language is non-empty but is not a key in the translations dictionary, the output is 'error=invalid_default_language\ndefault_language=<value>\n'.

For selected language resolution, please include a source field alongside selected_language. If router_query has a lang key and that value exists in the translations dictionary, output 'selected_language=<lang>\nsource=query\n'. If router_query is empty, null, or has a lang value that is not in the translations dictionary, output 'selected_language=<defaultLang>\nsource=default\n'. And for router_query_sequence, treat the first entry as initial_selected_language, the second as updated_selected_language, and if the second query had a lang that got ignored because it was not in the catalog, report that as ignored_query_lang.

On query composition, the precedence is strict: forced_language if present and non-empty wins and becomes the lang value first. If that is not set, use selected_language if non-empty. If selected_language is empty, then keep the existing router_query lang value. All other keys from router_query stay unchanged. If router_query is null and selected_language is empty and there is no forced_language, return {}. The output key here is query_json, and the value should be compact JSON with keys in insertion order.

Last thing, just repeating the core bit for feature1_configuration_resolution because this was where the confusion was: when use_browser_default=true, split browser_language on '-', take the primary subtag, check whether it exists in the translations dictionary keys, and only then override the resolved default_language. If it does not exist, keep the configured default and stick with the unresolved output format.
