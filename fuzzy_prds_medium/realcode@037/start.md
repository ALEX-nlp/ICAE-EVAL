## Product Requirement Document

We need a command-line tool that helps developers manage multiple versions of an external SDK side by side. The tool should accept version strings in whatever natural form developers typically type them and automatically normalize each one to the exact identifier used by the upstream release catalog — handle this the same way as the version resolution module does when it tries an exact match first, then tries a slightly modified form. The tool must be able to query the upstream catalog and report, for each queried identifier, whether it is published there or not; results should be emitted in the same order the references were provided. For locally cached versions, the tool needs to verify integrity: a cached version is only healthy if two specific sub-folders are both present; if one or both are missing, the tool should print a human-readable cleanup notice (naming the problematic version) before reporting it as not correctly set up. When fetching a new version from the upstream repository and the reference doesn't exist, the tool must report a structured, language-neutral error without exposing any internal exception details. Installation must reject a missing version reference upfront with a distinct error code rather than attempting a download. Finally, the tool should report its own software version number on demand. All output lines must follow a consistent key=value format terminated by a newline, and error output must always include a second line echoing back the rejected reference — handle error formatting consistently with the fetch failure module.

One extra bit on version lookup since folks asked: bare numeric refs like '1.8.0' should be treated the same way inferFlutterVersion() in golden/lib/utils/helpers.dart does it. It first checks if the version string (lowercased) appears verbatim in the upstream SDK list returned by flutterListAllSdks(). If that is not found, it prepends 'v' and checks again, so something like '1.8.0' can end up resolving to 'v1.8.0'. If the published thing is already a dev or pre-release tag without the prefix, like '1.17.0-dev.3.1', it stays exactly as-is. Channel names 'master', 'stable', 'dev', 'beta' short-circuit early via isFlutterChannel() and always resolve to themselves. Resolution is case-insensitive, and if neither form matches it throws ExceptionNotValidVersion.

Also, for local cache verification, if a version directory exists but is missing either the '.github' folder or the 'bin' folder, or both, the tool should print exactly '<version> exists but was not setup correctly. Doing cleanup...' followed by a newline, then remove the directory, then emit 'installed_correctly=false\n'. That cleanup notice should use the exact version string provided as input. If the directory does not exist at all, there is no cleanup notice in that case, only 'installed_correctly=false\n'.

On the fetch failure path, when fetch_version fails because the reference does not exist upstream, the output is exactly two lines: first 'error=could_not_clone\n', then 'version=<the requested reference>\n'. The requested reference needs to be echoed back verbatim, including exact case, and we still do not include any exception details or host-runtime info.

And for the tool’s own version output, the tool_version action prints exactly '0.8.2' followed by a newline, with no other output. That value comes from the packageVersion constant defined in the generated version source file.
