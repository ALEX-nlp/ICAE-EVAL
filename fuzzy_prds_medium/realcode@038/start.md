## Product Requirement Document

hey team, so we got some complaints from the frontend folks that CORS is still being a pain in certain edge cases. basically we need to build out that cross-origin middleware thing we talked about last sprint. the core idea is: devs should be able to define reusable policies and attach them to URL patterns, and the middleware should just handle all the preflight/response header stuff automatically without them needing to touch every endpoint.

the tricky parts i keep hearing about are: (1) sometimes the browser sends a method name in a weird casing and the current approach just rejects it silently, which is super confusing — we need to handle that gracefully somehow, similar to how we handled the case-sensitivity stuff in the auth header module last quarter; (2) there's also this ask from the platform team to support a 'hard override' for the allow-origin header value for certain deployments where they can't use the reflected origin pattern; (3) policy selection needs to be smart — different URL areas should get different rules, and some rules should only kick in for specific hostnames, not all traffic; (4) when multiple policy sources are active at the same time, later ones should win on a field-by-field basis, not replace the whole thing. priority ordering should also be configurable. we don't want to break anything that's already working for same-origin traffic. someone check how the existing request phase handling works before touching that part.

one quick follow-up since a few details came up after the first writeup. on the method casing issue, if the client sends Access-Control-Request-Method with the same letters but different casing than what we configured, we need to preserve both forms in the preflight allow-methods output instead of normalizing it away. so if configured methods are ['LINK', 'PUT'] and the client requests 'Link', the Access-Control-Allow-Methods value needs to be 'LINK, PUT, Link'. that same case-mismatch behavior is the one called out around feature1_preflight_success.json, specifically the example where Access-Control-Request-Method is 'Link' and configured is 'LINK'.

also, the hard override piece is an actual field named 'override_allow_origin'. when that is set, Access-Control-Allow-Origin should always use that exact value no matter what came in on Origin. that applies across the board: successful preflight, rejected preflight, accepted actual requests, and even requests without an Origin header. the value is emitted verbatim, so it might be '*' or something like 'http://example.com http://other.com'.

for vary behavior, preflight responses handled in the policy layer should include vary=Origin. for actual non-preflight cross-origin responses, vary must be empty, so vary=. that still holds when override_allow_origin is set — if it was a preflight OPTIONS exchange, vary=Origin still shows up.

on actual non-preflight cross-origin responses that get handled in the response phase, we should not be filling in the preflight-only stuff. Access-Control-Allow-Methods and Access-Control-Allow-Headers need to be empty on actual requests. really the only thing populated there should be Access-Control-Allow-Origin, plus credentials or expose headers if configured. and again, Vary stays empty for actual requests, no 'Origin' entry.

one more important pass-through detail: if the request Origin matches the request host, meaning same-origin, the request phase output should be 'response_created=no' and we should not generate any synthetic HTTP response. same thing if the origin doesn’t match any allowed origin in the policy and it is not a preflight OPTIONS request — still 'response_created=no'. in both of those situations, the output only includes phase, url, and method lines. that’s the same cors_exchange behavior called out for exchange=request in feature4_non_cors_and_unmatched_origin.json.

and for picking which policy applies by path, path_policies are checked in insertion order and we return the FIRST regex pattern that matches the request path. earlier entries win over later ones. if nothing matches, we fall back to the default policy values. so for example '^/test/regex' should match before '^/test/' if it was listed first and the path matches both.
