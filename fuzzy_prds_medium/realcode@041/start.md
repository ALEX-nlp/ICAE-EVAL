## Product Requirement Document

Hey team, we need to build out the session token auth service we've been discussing. Basically it should handle the full login/logout lifecycle for our API — when someone logs in with valid credentials they get back a token they can use on protected endpoints, and we need the usual stuff like being able to log out just the current session or nuke all sessions at once. The token should have an expiry (we talked about this in the standup, same approach as that credential refresh logic from the old auth module) and there should be a way to auto-extend sessions when someone is actively using the API. We also need to make sure stale or tampered tokens get rejected cleanly through the HTTP layer. Oh and one thing that's been a pain — users sometimes get stuck in a weird state when they have too many active sessions, so we need some kind of cap on that. The stored tokens need some lookup metadata that lines up with what was actually submitted. Also there's some config around token prefixes that needs validation before it blows up at runtime. Broadly speaking the HTTP responses need to be consistent and predictable across all these flows. Let me know if anything is unclear — I know I'm probably missing some details around the exact formats and limits but just check what we've done before.

One other pass on the details the team asked about: the issued session token needs to be exactly 64 characters long. For expiry, the default session lifetime is 10 hours, so a token issued at 2020-01-01T00:00:00 will expire at 2020-01-01T10:00:00Z. On the prefix config, the maximum allowed issued-token prefix length is 10 characters. If someone configures a prefix longer than 10 characters, we should reject it with error=token_prefix_too_long, and the response also needs max_prefix_length=10 and the actual prefix_length.

On the protected endpoint behavior, the route is /api/ and the URL in the response also needs to be /api/. If a token is invalid, expired, too-short, or tampered, that should always come back as HTTP 401 with the detail message 'Invalid token.'. If the request is actually authenticated successfully against /api/, it should return HTTP 200 with the body text 'User is authenticated.'. Separately, if no valid credentials are provided to a protected endpoint, like the wrong scheme situation, the HTTP 401 detail should be 'Authentication credentials were not provided.'.

Also confirming the session storage behavior: each successful login call creates an independent persisted token, so after N successful logins, token_count must equal N and stored_token_keys_present must be true. And when an expired token gets used and rejected, its stored record should be removed as part of that validation path through the protected endpoint, so stored_expiry becomes null when queried.

Last piece is the auto extension toggle in feature6_session_expiry_refresh. When auto_extend_sessions is enabled, every successful authenticated request resets stored expiry to current_time + session_lifetime (10 hours by default). When it’s disabled, the stored expiry is never updated after issuance.
