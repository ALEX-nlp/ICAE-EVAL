## Product Requirement Document

Hey team, we need to ship that HTML compression thing we talked about last sprint. The basic idea is we want to automatically shrink the HTML we send to browsers so pages load faster, but without breaking anything — especially that 'protected content' behavior we figured out with the template team a while back. Devs need to be able to drop this in globally (like a middleware layer that just handles every response automatically) but also opt individual views in or out if needed. There are some edge cases around comments — some teams want them stripped, others want to keep certain ones, and there are some special browser-compatibility comments that should ALWAYS survive no matter what the config says. Also there's a debug mode situation where minification shouldn't fire when you're developing locally. The byte-handling fallback logic should work similarly to what we did for that internationalization encoding module — try the preferred encoding, fall back gracefully through alternatives. Path-based exclusions should work by matching against the URL without the leading slash. Global on/off toggle needs to play nicely with debug mode. Please keep this modular — no giant single files, the core logic shouldn't know anything about HTTP or JSON. The middleware should only touch responses that actually went through the request phase properly.

One quick follow-up from the questions that came in: the special browser comments we talked about are the downlevel-revealed conditional comments in the form <!--[if ...]> ... <![endif]-->, and those always stay in the output no matter whether comment stripping is on or off. Everything else in HTML comments gets stripped by default unless the ignore_comments flag is set to false.

Also, for protected content, the protected regions are pre, script, and textarea. The inner content of those tags needs to stay byte-for-byte exactly as it came in, so whitespace, indentation, and newlines inside them are never touched. That said, the opening tags around that content still go through normal parse/serialize cleanup, so attribute serialization is normalized there too. More generally, attributes on all tags, including protected tags like script and textarea, are sorted alphabetically in the serialized output. So if a script tag comes in with type, src, charset, it should come back out as charset, src, type.

On the document wrapper side, if the source has a DOCTYPE declaration, we keep it, but collapse internal whitespace to single spaces, so a multi-line DOCTYPE with public and system identifiers split across lines gets flattened to one line. If there was no DOCTYPE in the source, we do not add one.

A couple behavior details to lock down too: when minify_enabled is explicitly set to false, minification is off no matter what anything else says. When minify_enabled is null/unset, then debug decides it — debug=true means no minification, and debug is false or unset means minification is enabled. So null minify_enabled + debug=true means no minification.

For the middleware/filtering part, only responses with a 2xx success status code AND an HTML content type should be touched. Anything else should pass through unchanged, including 301 redirects and any 4xx or 5xx response, even if the content type is text/html. And same as before, keep the structure modular: a pure minification core that only works on HTML strings and knows nothing about JSON, HTTP, or stdin/stdout, a byte-decoding utility, an HTTP integration/response-filter layer that depends only on abstract request/response shapes, and an execution adapter that is the only piece reading JSON from stdin and writing to stdout. No single god-file.

Last thing, the byte decoding should follow the same fallback pattern we already discussed: use decode_bytes behavior where you try the preferred encoding first, default UTF-8, then fall back through alternatives like latin-1/ISO-8859-1, and still let the caller override via the encoding field.
