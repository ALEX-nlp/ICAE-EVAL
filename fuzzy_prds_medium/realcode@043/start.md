## Product Requirement Document

Hey team, we need to ship a coordinate utility wrapper for our maps SDK integration. Basically we keep copy-pasting the same spherical math and polyline wire-format stuff across every project and it's getting messy. The idea is a clean adapter layer that devs can drop in and call without worrying about the underlying geometry plumbing.

We need it to handle the usual suspects: encoding/decoding those compact path strings, figuring out if a shape is closed, trimming down paths to fewer points when the detail isn't needed, all the bearing/distance/offset math, interpolating between two points, area calculations (including the signed version for winding order checks), point-in-polygon queries, edge proximity checks, and the map object wrappers for lines and polygons. Oh and don't forget the heat-map weight conversion thing and the flat point component accessor — Marcus mentioned those got dropped in the last sprint.

Coordinate precision should be consistent everywhere, same as what we agreed on for the location service outputs. For the simplification feature, the tolerance is distance-based, not angle-based, just like we handled it in the route-matching module before.

The adapter needs to run as a standalone black-box process: read a JSON command from stdin, write the result to stdout. Output format rules should follow the same conventions we nailed down for the geo-search results endpoint. Please keep the core logic fully decoupled from the I/O layer so we can unit test it independently.

One quick follow-up from the questions that came in: anytime we decode or compute a latitude/longitude, it needs to be formatted to exactly six decimal places, everywhere, no exceptions. So think values like 1.000000 and -90.000000, and when we echo a query point back it should be in the form lat,lng with those same six decimal places. That applies across the board and should stay aligned with the feature test cases in rcb_tests/public_test_cases/ and the usual Google Maps API coordinate precision conventions.

A couple output specifics too. For the polyline encoding piece in Feature 1, print the encoded polyline string to stdout and make sure it has a trailing newline. For a single point like lat=1.0, lng=2.0, the output is the Google Encoded Polyline Algorithm string for that coordinate sequence, printed to stdout with a trailing newline. For path length in Feature 10, print total meters as a floating-point value with at least one decimal place, and if the coordinate list is empty or only has one coordinate, the output should be '0.0'. Same general idea for area: Feature 11 prints the unsigned area in square meters as a floating-point number with at least one decimal place, and Feature 12 prints the signed area for the given traversal on the first line and the signed area for the reversed traversal on the second line, with opposite signs.

For the polygon and edge-style checks, the text output should stay super literal. Feature 13 prints 'inside' or 'outside' on the first line, then the query coordinate as 'lat,lng' with six decimal places on the second line. Feature 14 and Feature 15 print 'on_edge' if the query point lies on or within the default tolerance of the edge/path, or 'not_on_edge' otherwise, and then the query coordinate as 'lat,lng' with six decimal places on the next line. The map wrappers should mirror that behavior: Feature 16 prints 'on_edge' or 'not_on_edge' then the query coordinate, Feature 17 prints the total length in meters with at least one decimal place and should print '0.0' for an empty line, Feature 18 prints 'inside' or 'outside' then the query coordinate, and Feature 19 prints 'on_edge' or 'not_on_edge' then the query coordinate.

Last couple details: Feature 20 should print the stroke width as a float, the stroke color as an integer, and the count of vertices, each on a separate line. Example values are '1.0', '-16777216', '1'. Feature 21 should print the x component followed by the y component, each on a separate line, formatted as floating-point numbers like '1.0' and '2.0'. And for simplification in Feature 4, the tolerance is definitely tolerance_meters and it should use a spherical-distance-based Ramer–Douglas–Peucker algorithm, not an angular or planar tolerance.
