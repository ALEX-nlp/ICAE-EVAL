## Product Requirement Document

Hey team, we need a command-line tool for working with JWTs. Developers keep complaining they're spending too much time hand-rolling token stuff in bash scripts and it's causing bugs in staging. Basically the tool should let you create tokens and also read them back / check if they're legit. A few things I know we need: it should handle the usual token fields like who issued it, who it's for, expiry, etc., plus let you throw in your own custom fields including numbers, booleans, and nested JSON objects (not just strings). For the time stuff, people don't want to calculate Unix timestamps by hand — they want to say something like 'expires in 10 minutes' and have it just work. Also we need it to work with key files on disk, not just inline secrets, because some of our infra uses RSA keys. Oh and it should behave like that validation approach we settled on for the auth service — where bad inputs give back a clean normalized error code instead of a raw exception dump. The test runner should output files that map to each case so we can diff them. I don't have all the exact flag names sorted out, refer to what the team already standardized for the encoding/decoding options. Should be well-structured code, not a single giant file. Let me know if you have questions.

A couple follow-ups from the questions that came in. For custom claims, we do want people passing them with the -P flag in the format key=value. The value should get parsed as JSON if it looks like a number, boolean, array, or object, so examples here are exactly -P number=10, -P array=[1, 2, 3], -P object={"foo": "bar"}. If it doesn’t look like one of those, it stays a string. Also, if the value does not contain exactly one = separator, return error=invalid_payload_item.

On the time side, relative durations should be accepted as strings like '+10 min -30 sec' or '+5 min'. We do want to support multiple additive/subtractive terms, and the offset is calculated from the automatically generated iat (issue time). If the string can’t be parsed as either an integer Unix timestamp or a valid relative duration, return error=invalid_timestamp_or_duration. Also confirming the encoder adds an iat (issued-at) claim by default, and you can suppress that with --no-iat. One subtle thing there: expiration via --exp without an explicit timestamp should still be set relative to the current time even when --no-iat is used, and validation should still succeed if the exp claim is present.

For keys on disk, the way to do it is prefixing the -S argument value with @, for example -S @./tests/private_rsa_key.der. That file is read as raw bytes. Same behavior for HMAC secrets and asymmetric keys (RSA, ECDSA). For RSA specifically, sign with the private key file and verify with the public key file, both in DER format.

One more decode behavior detail: if a token is decoded with a mismatching secret, the output should be exactly status=invalid\nerror=invalid_signature\n followed by the decoded header and claim fields that were requested in the report. The payload fields still get decoded and rendered even when the signature is invalid. If no secret is provided at all, treat it as unverified and return status=valid.

And just to restate the validation piece in the same spirit as the original ask, feature 6 input validation errors are invalid_payload_item for malformed -P values and invalid_timestamp_or_duration for unrecognized -e/-n values, and those should come back as plain error=<code> lines with no exception stack traces.
