## Product Requirement Document

Hey team, we need to build that reactive state container thing we've been talking about. The basic idea is devs should be able to just write normal JS like `store.user.name = 'ada'` and everything else happens automatically — no dispatch calls, no action creators, none of that Redux ceremony. When something changes, subscribers get notified with the path of what changed, and the store should do that smart memory thing we did in the immutable updater module (you know the one — like how the login module handles nested clones). Also the store object itself should never get swapped out, even if you replace everything inside it — people hold references to it.

We also need to handle when someone tries to write into a path that doesn't exist yet (like a missing parent), that should fail gracefully. Arrays should just work naturally too. And there's a replace operation that wipes everything and installs new stuff — old keys get reported as gone first, then new ones as added.

The output format for the test harness is very specific — reads, then changes, then identity probes, then snapshot — and keys in snapshots need to be sorted. Can someone also make sure the codebase isn't just one giant file? We've had complaints about that before. Ping me if the path rendering format isn't clear.

One extra thing the team asked about on the harness output: the order is not "whatever happened first" at runtime, it’s fixed. We always print first all `get` lines in the order the get ops were requested, then exactly one `changes=[...]` line, then all `identity` probe lines in the order specified, then the `snapshot=` line. That order is mandatory regardless of op ordering. Also, if we hit a bad write/read path because we have to walk through a non-existent or non-navigable intermediate node, the whole request should return exactly one line: `error=invalid_path` and nothing else — no changes, no snapshot, no gets.

On the smart memory behavior, this should stay the same idea we already talked about from the immutable updater. Structural sharing here means when a nested property changes, we rebuild only the objects on the path from root to the changed node, and everything off that path keeps the same reference. So only the direct path from the root to the changed property is rebuilt, while sibling sub-trees and other off-path objects are preserved. The identity probe should grab references before mutation and compare after so it can report `preserved` or `replaced`. This should live in the immutable update module pattern we already use, like `src/updater.js` or equivalent.

For `replace`, the sequence matters a lot. It should first delete all currently existing top-level properties, and those removals get reported in their original key order, then it installs all new properties from the replacement object, reported in the order they appear. Even with that full wipe-and-install flow, the store object itself still cannot change identity — it has to be the same object reference before and after replacement.

Last detail on rendering: all object keys in `snapshot=` output, and also in `get` output when the value is an object, need to be sorted in lexicographic alphabetical order. Arrays still render in index order. That applies recursively for nested objects too.
