## Product Requirement Document

Hey team, we need to build out that stylesheet module processing tool we've been talking about. The basic idea is that devs are constantly having to hand-maintain type declaration files and class name mappings whenever they touch a CSS module file — it's a huge pain and things go stale constantly. We need something that can take a stylesheet file and spit out the right class export map and declaration text automatically.

The tool needs to handle all the usual stylesheet flavors people use (you know, the ones we support in the main app build pipeline). It also needs to do the smart naming thing — like when someone has a dash-separated class name and they want to use it as a JS property, we should be able to expose both versions or just one depending on how they've configured things. Similar to how we handled identifiers in that config parser module a while back, but for class names specifically.

For path handling, it should recognize which imports are actually stylesheet modules vs regular stylesheets, and also tell us if they're local relative imports or coming from somewhere else. Custom pattern overrides should be supported too, with a graceful fallback if someone gives us a bad regex.

Declaration output needs to be flexible — support for custom templates, and also that line-aligned mode for source maps. The scoped class name generation should be deterministic and consistent. Please check what the existing test fixtures expect for the exact output formats before assuming anything.

A couple extra specifics from the questions that came in: on the stylesheet side, this only treats a file as a module stylesheet when the filename ends with one of these exact suffixes: .module.css, .module.less, .module.styl, .module.scss, and .module.sass. So yes, we support plain CSS (.module.css), Less (.module.less), Stylus (.module.styl), SCSS (.module.scss), and Sass (.module.sass), but something like .module.s alone is not recognized. That same exact set is the default path recognition behavior we should stick to.

For the class export naming behavior, there are five modes and they should behave pretty literally. We need no style / 'asIs' where we expose the original name unchanged, camelCase adding where we keep the original AND add a camelCased alias when it differs, camelCase only where we expose only the camelCased form, dash-to-camel adding where we keep the original AND add a dash-segment-camelized alias when it differs, and dash-to-camel only where we expose only the dash-segment-camelized form. Each input class name produces an inner array of one or two property name strings. This is the same class-name alias generation logic in Feature 1, specifically the five naming styles (asIs, camelCaseAdding, camelCaseOnly, dashToCamelAdding, dashToCamelOnly) described in feature1_class_name_aliases.json and the corresponding core domain logic for transforming dash/underscore/mixed-case class names into JavaScript-safe property name aliases.

On path reporting, a path only counts as relative when it starts with './' or '../'. If it’s a valid module stylesheet filename but doesn’t start with either of those, then it should come back as module_stylesheet=yes but relative_module_stylesheet=no. The output format for this part is always three lines per path: 'path=<path>', 'module_stylesheet=<yes|no>', 'relative_module_stylesheet=<yes|no>'. If the caller-supplied pattern is invalid (i.e., it cannot be compiled as a regular expression), we silently fall back to the default module stylesheet rules (.module.css, .module.scss, etc.), and then append one extra line after all the per-path result lines: 'pattern_error=ignored'.

Also, for scoped names, the standard class format is '<module-name>__<className>---<hash>' and the module name comes from the stylesheet filename with dots replaced by dashes and underscores used for certain characters. Animation/keyframe classes are different on purpose: '__<module_name_underscored>__<className>' with no trailing hash segment. The hash component is a 5-character base-62-like string.
