## Product Requirement Document

Hey team, we've been getting complaints from a few backend devs that their Django apps are hammering the database way too hard on repeated reads — stuff that should obviously be the same query is hitting the DB every single time. We talked about doing something similar to what we did with the login session layer (you know the one, where we wrapped the read path and tracked origin state), and the consensus is we should build something proper here rather than everyone writing their own ad-hoc solution.

Basically we need a caching layer that sits on top of Django's ORM so that repeated reads just work without devs having to think about keys or invalidation. The big pain points are: stale data showing up after a record gets updated (especially when a related child or parent record changes), pages not refreshing after a save even when the user can clearly see the data changed in the admin, and template fragments being re-rendered on every request even when nothing changed.

There's also a reported issue where switching languages mid-session causes some users to see wrong cached content — apparently the cache doesn't know about locale context. And some teams are using multiple database connections and seeing data bleed between them.

We also want devs to be able to opt out of caching per-query when needed, and handle edge cases like empty result sets and methods on model instances. Should support Jinja templates too. Needs proper test coverage. Refer to how the parameterized query identity stuff was handled in the raw query path for the key uniqueness approach.

One extra thing the team asked for: every returned record or result set needs to carry a `from_cache` boolean field so it’s obvious what happened on read. The first evaluation of any lookup must report `from_cache=false`, and then a repeated identical lookup within the same application context must report `from_cache=true`. That applies across single-record lookups, filtered list lookups, sliced list lookups, raw table lookups, and parameterized raw lookups. Same idea after writes too: if a save blows away the relevant cached entries, the next read goes back to `from_cache=false`, and then later repeats of that same query can land at `from_cache=true` again once it’s been re-cached.

On invalidation, this needs to be broader than just “the one object you saved.” Saving a record must invalidate all cache entries associated with that record, including both single-record cache entries and list-query cache entries that contain that record. That propagation also needs to travel across relationships: saving a child record must invalidate cache entries for related parent/referenced records, and updating a related record must invalidate cached parent records that include that relation. Also worth calling out the reverse-relation empty-state problem: if we cached `[]` for a parent’s children, that empty result can’t stick around after new matching child records are created. After the first child is created, the reverse relation lookup must return a list containing that child's value. After a second child is created, both values must appear in insertion order. The initial cached empty state `[]` must not persist after new members are added.

For the locale issue, the cache key needs to include locale/language context, so entries created under the default locale and entries created under an alternate locale like 'fr' are separate and don’t clobber each other. But invalidation should still cut across locales: saving a record while an alternate locale is active must invalidate cache entries that were originally populated under the default locale, so we’re not only clearing whatever locale happens to be active at save time.

Also, for raw SQL-style parameterized lookups, parameter values have to be part of the cache key identity. Two raw lookups with the same query text but different parameter values like id=1 and id=2 must produce separate cache entries and return their own corresponding record, not accidentally reuse a cached result from the earlier parameter value. Each distinct parameter combination starts with `from_cache=false`.

And on the template side, we do want caching for rendered template fragments in the Jinja2 engine, keyed either by a single record or by a record list/queryset. A fragment should render once, get reused on repeat renders, and then automatically invalidate when a member record is saved with changed data. Multiple cache blocks in the same template should stay independent from each other. Fragments may also include an extra key value to create distinct cached entries per extra-key value, and fragment caching should support optional timeout configuration alongside those extra key values.

Last bit, just to be explicit since a few folks asked: the `from_cache` boolean field attached to each returned record/queryset result is the thing tracking whether the result came from the cache backend or straight from database storage, and it should follow the same contract already laid out in start.md plus the execution adapter pattern the architecture notes describe.
