## Product Requirement Document

Hey team, we need to build out that bulk insert helper we've been talking about. Basically the idea is devs can queue up a bunch of rows and flush them to the database in one go instead of doing one insert at a time — that's been killing perf on the import flows. It should work kind of like how we handled the batching logic in the old data pipeline tool, where you configure a ceiling and it auto-flushes when you hit it. Rows can come in either as ordered value lists or as named field maps, both should work. There needs to be some way to get a SQL preview for different database backends without actually hitting the db — important for the QA team. Also need lifecycle hooks so teams can filter or modify rows before they land. Oh and the timestamp stuff should just work automatically, we've had too many bugs where people forget to set created_at. The writable columns list should exclude the auto-generated primary key. One thing I'm not totally sure about is the exact conflict resolution behavior per backend — I think MySQL, Postgres, and SQLite each do it differently so we'll need to handle all three. The block-style entry point should also be supported for teams that prefer that pattern. Default batch ceiling should match what we agreed on in the architecture review.

Quick follow-up from the questions that came in: the default batch ceiling we talked about is 500. The behavior at the edge matters a little here — when the queue reaches exactly this number of rows and a new row is added, the existing queued rows are flushed first and the new row remains pending. Same deal for the configurable version too: the batch_limit feature works so that when the queue size equals the configured batch_limit and a new row is added, the existing queue is flushed first and the new row stays pending, which is the behavior tied to the BulkInsert batch_limit option.

On columns, the default writable list should be greeting, age, happy, created_at, updated_at, color in that order, and id is not part of that list. For row objects, if age is left out it should persist as null, and if happy is left out it should default to false. That’s separate from color, where omitting the field should use the database default value 'chartreuse', but if color is explicitly set to null then it should persist as null.

For the block-style path, the block/lambda gets a writer instance, and rows added inside the block are saved automatically when the block exits. It’s basically equivalent to immediate_insert, just using a scoped writer pattern.

A couple more specifics on preview output since QA asked: table and column names are wrapped in double quotes, like "testings" and "greeting". Boolean false renders as 0 and boolean true renders as 1. Strings are wrapped in single quotes, and NULL renders as unquoted NULL. Also for the per-engine SQL rendering in the sql_preview flow, we do need to cover MySQL/Mysql2 INSERT IGNORE + ON DUPLICATE KEY UPDATE, PostgreSQL ON CONFLICT DO NOTHING / ON CONFLICT (...) DO UPDATE SET, and SQLite INSERT OR IGNORE.

Last thing on timestamps: if created_at and updated_at are omitted, they should be filled automatically during save. All rows in the same batch share one generated timestamp value for both created_at and updated_at, and within that batch the created_at and updated_at values are equal to each other.
