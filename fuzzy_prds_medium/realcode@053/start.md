## Product Requirement Document

Hey team, we need a little utility that auto-generates those module dependency diagrams and slaps them into the right spot in our markdown docs. Right now devs are hand-drawing these things and they go stale immediately whenever someone adds or removes a dependency — it's a real pain and the docs are always lying to us. The idea is: you feed it a description of which modules depend on which, pick some visual options, and it spits out a ready-to-commit diagram. Similar vibe to what we did with the config-injection approach in the reporting pipeline last quarter, if you remember that. It should handle different colour schemes including a fully customisable one where you can pass in your own colour values. There should also be a way to zoom in on just a subset of modules — like if I only care about one module I can filter to just the stuff touching it, with those nodes visually called out somehow. If I give a filter that doesn't match anything that should fail gracefully, not blow up with a stack trace. Docs might not exist yet, or might already have a section from a previous run — both cases need to work cleanly without nuking unrelated content. Needs to be well-structured code, not one big file. Target language is Kotlin or Python, whatever the team thinks fits best.

A couple extra specifics from the questions that came in: the supported theme values are 'dark', 'default', 'forest', 'neutral' and 'base', with 'neutral' as the default when none is specified. 'base' is the customisable one, and if someone passes a 'variables' map, that should be theme-variable names to colour/size values and get serialised as a compact JSON object inside the init block. Orientation is also fixed to LR, RL, TB, and BT, with LR as the default, and it needs to show up right in the graph declaration line like 'graph LR'.

On output shape, the Mermaid block always starts with the init section in this exact format: '%%{\n  init: {\n    \'theme\': \'<themeName>\'\n  }\n}%%'. Then there’s a blank line, then 'graph <ORIENTATION>'. In short-name mode with subgraphs, there’s another blank line before the subgraph blocks. The fenced block should open with triple backtick + 'mermaid'. Also worth being explicit that self-edges are always dropped. Duplicate edges are collapsed when source and target are the same and, when linkText is 'configuration', scope is also the same. If linkText is 'none', then two edges between the same pair of nodes under different scopes still collapse into one unlabelled arrow.

For naming/grouping, in short-name mode, meaning showFullPath is false or absent, modules that share the same parent folder segment should be gathered into a labelled 'subgraph <folder>' block, and each subgraph lists the short names, which is the last non-empty colon-separated segment. If showFullPath is true, then every node is printed with its full colon-separated path like ':groupFolder:example2', subgraphs are NOT emitted at all, and edges should come out as '  <source path> --> <target path>'. One tiny formatting gotcha there: the blank line before edges that exists in short-name mode between the last subgraph end and the first edge is also absent in full-path mode.

And on structure, yes, we do want the same separation pattern as the reporting pipeline: a core rendering engine in a src/ directory that only works on in-memory data structures and knows nothing about stdin/stdout or JSON, plus a separate execution adapter that is the only piece parsing JSON and printing text, plus a tests/ directory. The core should be callable through normal in-memory APIs, and the execution adapter is basically a thin I/O wrapper that parses JSON commands from stdin and delegates to that pure rendering core.
