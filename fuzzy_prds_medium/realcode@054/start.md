## Product Requirement Document

Hey team, we need a terminal command safety guard tool — basically something that sits in front of shell commands and flags dangerous ones before they actually run. Think of it like that risk-filter layer we discussed last quarter but never shipped. The core idea: you define a bunch of rules (some patterns, some more complex), load them from a config file, and when a command comes in it gets checked against whatever rules are active. We also need to be able to manage those rule sets — add groups, remove groups, swap out the confirmation style, and reset everything back to defaults when things get messy. The config stuff is YAML-based, similar to what the ops team is already used to. One thing that kept coming up in the retro: the 'challenge' mechanism needs to be configurable per-deployment, not hardcoded. Also, when you reset to defaults, the system should support optionally keeping a copy of the old config — legal asked about audit trails last week. The default groups after a reset should match what we standardized on in the base rollout (you know the three buckets we settled on). Output formatting needs to be consistent and parseable so the test harness can validate it automatically. Please keep the matching engine decoupled from the config I/O so we can swap pieces independently later.

One extra pass of detail based on the questions that came in: for matching, we only support exactly three methods, 'contains', 'starts_with', and 'regex'. 'contains' is a literal substring match anywhere in the command, 'starts_with' means the command has to begin with the pattern, and 'regex' is a full regular-expression match against the command text. If a check comes through using anything else, treat it as unsupported.

On the config output side, each check line needs to be rendered in one consistent format so it stays machine-parseable: check=<group>|<method>|<enabled>|<challenge_override>|<description>\n. For example, 'check=default-check|Regex|true|Default|'. The method should be title-case in output, so 'Regex' not 'regex', and when there is no per-check override the challenge override should be 'Default'. Also, the description can be blank, but that trailing pipe still needs to be there. The top-level 'challenge' setting in the YAML config is the thing that controls the confirmation prompt style, like 'Math' or 'Yes', and whenever we print config state it should show up as 'challenge=<value>'.

Also clarifying reset behavior a bit more, since that came up a few times. When reset happens, the included groups go back to exactly three defaults, in this order: 'base', 'fs', 'git'. Those are the standardized buckets from the base rollout, and the includes line should come out as 'includes=base,fs,git'. Old checks do not carry forward on reset, so anything from the previous config is dropped. That means the resulting check_count after reset will be 0 if none of the available_checks provided belong to those three default groups. And for the reset output ordering, it needs to start with 'backup_created=yes' if backup_existing is true, or 'backup_created=no' if it is false. That line comes before the challenge, includes, and check lines.

For command evaluation output, the block for each command starts with 'command[<index>]=<command_text>\n' and then 'match_count=<n>\n'. After that, for each matched rule, in order, print 'match_group=<group>\n' and 'match_description=<description>\n'. If there are multiple commands, put a separator line '---\n' after each one except the last. And if a command has no matches, then it only prints the command line and 'match_count=0\n'.
