## Product Requirement Document

Hey team, we need to wrap up that cross-platform desktop notification routing library we've been planning. The basic idea is that app developers should only have to write one options object and the library handles all the messy platform differences under the hood — no more hand-rolling shell commands or worrying about which binary to call on what machine.

A few things I want to make sure we nail: the library should know about all the notification senders we support and be able to tell you which ones exist plus confirm a default one is ready to go. Each sender has its own quirky naming conventions for things like icons and urgency, so we need that normalization layer that maps our friendly option names into whatever that specific tool actually expects. One of the senders (the network-push one) is a bit special because it needs to handle icon data differently from the others — remember how we handled binary assets in that earlier media pipeline work, same kind of idea.

Also really important: the command arguments each tool gets need to be constructed safely — users can put weird characters in their notification text and we can't let that cause problems. Windows has that extra wrinkle where you need to pick the right binary depending on the machine architecture. And the quiet/sound behavior on Windows needs to be toggleable. Finally, any request missing a message should be rejected cleanly before anything gets dispatched. Keep the code organized — no giant single files please.

One follow-up from the questions that came in: the library exposes exactly five named backends, and that list is fixed no matter what machine you’re on. When someone asks what’s available, return them in alphabetically sorted order, one per line as 'backend=<Name>', for Growl, NotificationCenter, NotifySend, WindowsBalloon, and WindowsToaster, and then add a 'default_notify=available' line so it’s explicit the default sender is ready.

A couple normalization details to lock down too. For the Linux notify-send path, both the generic 'appIcon' key and the single-letter 'i' shorthand collapse to the canonical key 'icon'. The full shorthand expansion there is u→urgency, e→expire-time, c→category, i→icon, h→hint. Title and message pass through unchanged. On the macOS terminal-notifier side it flips the other way: the generic 'icon' key and the 'i' shorthand both collapse to the canonical key 'appIcon', and again title and message pass through unchanged. So Linux uses 'icon' as canonical, macOS uses 'appIcon' as canonical.

Also, for the backends that wrap values in double quotes, which are notify-send and terminal-notifier, make sure these characters are backslash-escaped before wrapping: double quotes ("), backticks (`), dollar signs ($), and backslashes (\). Single quotes do NOT need escaping. The notifu (Windows) backend is the exception here and passes text through literally with no escaping or quote-wrapping.

For Growl/GNTP, 'host' and 'port' are constructor-level configuration, not per-message options, and they get forwarded to the underlying push transport when sending. If they aren’t supplied, both stay unset and should show up as '<unset>' in test output. The application name defaults to 'Node' regardless of whether host/port are provided.

And on that special icon handling for the network-push backend, the important bit is that it needs raw binary bytes, not a file path string. The placeholder '@app-icon' points at a bundled 16-byte image asset included with the library. Whether the caller provides it under 'icon' or 'appIcon', we should read it into a byte buffer and expose it under the key 'icon'. The rendered output format for the test adapter is 'bytes:<length>', for example 'bytes:16'. Same deal specifically in feature2_4: '@app-icon' must be resolved to that bundled 16-byte binary asset and loaded into a byte buffer, not kept as a path string.
