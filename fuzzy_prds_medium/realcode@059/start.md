## Product Requirement Document

Hey team, we need to build that headless routing core we've been talking about. Basically it's a small engine that figures out which parts of the app are 'active' based on the current URL, and lets us drive navigation through plain store actions. Think of it like the matching logic we wired up in the auth flow last quarter but generalized — same idea of registering patterns and querying results, but now it's the single source of truth for the whole app.

The tricky bits are: URLs can have extra junk at the end (filters, anchors, etc.) that shouldn't break matching, some route segments are optional so they need to still show up even when nothing's there, and we definitely don't want the engine mangling any encoded characters in the URL values — that caused a bug last time.

On the action side, one of the three command types has a side effect baked in — it should talk to the browser history as part of being created, not separately. And the reducer needs to handle the case where nobody passes in a starting state gracefully.

For querying state, components need to ask things like 'is this specific route active', 'are ANY routes active right now', and 'give me the params for this route'. The whole thing should be split into sensible modules, not one big file. Can someone pick this up? We need test coverage against the JSON contract format we agreed on.

One extra pass on the matching behavior since a few folks asked: before we even try to compare a location to a pattern, strip off any query string starting at '?' and any hash fragment starting at '#'. Matching only uses the clean path part. So '/product/1?asdf=123' becomes '/product/1' before comparison, and '/product/1#section' also becomes '/product/1'. Also, matches need to be exhaustive, not partial. A pattern only counts if it covers the entire location path, so '/product/:id' does NOT match '/product/1/edit' because ':id' only covers one segment and 'edit' is still left over.

On the optional segment piece, a named param with a '?' suffix like ':id?' is optional, but it still needs to be present in the output even when the URL doesn’t supply anything for it. In that case the match still succeeds and we report it as 'param.id='. Same deal on captured values: take them exactly as they appear in the URL and do NOT decode them. If a segment contains '%2F' or anything else percent-encoded, keep that exact form in the result, like 'param.id=before%2Fafter'.

Also confirming the action behavior: the NAVIGATE command factory should do the browser history write immediately when the command object is created, not later in reduction. So when 'action kind=navigate path=/about' is processed, the output includes 'type=NAVIGATE', 'path=/about', AND 'history_pushState=/about'. And for reducer startup, if state is missing or undefined, initialize from the default where activeRoute is '/' and the routes table is empty. Reducing with no action from that starting point should yield 'activeRoute=/'. The matcher module is where the segment-by-segment comparison, optional param handling, full-path coverage behavior, and the stripping of '?...' and '#...' from the active_route location string should live.
