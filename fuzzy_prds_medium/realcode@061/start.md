## Product Requirement Document

Hey team, we need to wrap up the MongoDB adapter library that lets devs skip all the usual boilerplate when talking to Mongo. You know how it is — everyone ends up writing the same callback-to-promise conversion code over and over, and it's a mess. The idea is basically the same pattern we used in that older driver shim project (you can dig around the repo to find how we handled the connection shorthand stuff back then).

The library needs to handle the full lifecycle: connecting with short-form addresses, grabbing collections by name (even names that clash with built-in JS properties), running queries, inserts, updates, deletions, and the atomic modify-and-return thing. Cursors are a big deal here — they need to support the whole range of iteration styles including streaming and piping, and they should be rewindable. Bulk write support is also required, including the big-batch splitting behavior and the plan-inspection feature before you actually execute. Collection admin stuff (stats, indexes, renaming, dropping, capped detection) needs to work too.

One thing that kept coming up in user feedback is that when you do a removal or update, the count reporting needs to match exactly what the server says — people kept getting confused about whether 'matched' vs 'modified' counts were right. Same deal with the find-and-modify — returning old vs new doc depending on a flag. Make sure error handling for duplicate users throws the right domain error type with the correct code. Details on exact field shapes for bulk plan inspection are in the test files.

One extra pass on the details the team asked about: the client needs to accept both 'localhost/dbname' and bare 'dbname' forms, and 'localhost/test' and 'test' must both resolve to the same local database and return identical query results. This is the same shorthand normalization we want in the mongoist() entry point when it expands those forms into a full mongodb:// URI, so the connection behavior stays consistent no matter which version devs pass in.

Also, collection access through the db facade has to keep working even when the name collides with normal JS property names. In particular, 'options', 'client', 'connection', 'domain', and 'features' all need to come back as usable collection handles, and each one should be independently countable with zero documents initially. On the catalog side, listCollections() should only show user-created collections and leave out internal ones like system.users and system.indexes, so after seeding one collection and creating two more, the result should be exactly 3 entries and their names should sort alphabetically.

For collection admin, rename(newName) should move all documents over to the new name, drop() should remove the collection entirely, and stats() should return an object with a count field. Index handling should behave the usual way: ensureIndex(spec) adds an index so there are 2 total including the default _id index, dropAllIndexes() brings that back down to 1, createIndex(spec) adds one back, dropIndex(name) removes a named index, and reIndex() should succeed without adding indexes. isCapped() should return true for something created with {capped:true, size:N} and false for a regular collection.

A few bulk specifics too: if a batch goes over 1000 documents it should split automatically and still report ok=1 with the correct total document count. We also need bulk replace behavior that matches by query and replaces the full document, plus mixed ordered ops where insert, update using $set and a level field, remove, and insert-if-not-exists can all live in the same run. If the bulk op is empty, that should still come back as a successful no-op with ok=1. Before execution, the plan has to be inspectable through both toPrettyJSON() and toString(), plus toObject(), and all of those need to agree on nInsertOps, nUpdateOps, nRemoveOps, and nBatches. For the example plan with 2 inserts, 1 update, and 1 removeOne, nBatches should be 3 because the inserts batch together while update and remove are separate.

On the database-level behavior, stats(scale) needs to return an object with ok=1 plus collections and indexes fields, and runCommand({dbStats:1, scale:N}) should also return ok=1. getLastError() should return null or an object with an 'err' field that is null when no error has happened. The client should emit a 'connect' event the first time a query actually causes the underlying connection to open, and after dropDatabase(), a fresh connection using the same database name should see zero documents in collections that were populated before.

Last thing, on user management, the first addUser() should succeed with ok=1, and if the same username is added again the library needs to reject with the domain-specific DuplicateKeyError and a 'code' property equal to 11000. The error should be obviously recognizable as duplicate, whether that’s by class name or name/message content, and after that failure dropUser() still needs to work and return ok=1.
