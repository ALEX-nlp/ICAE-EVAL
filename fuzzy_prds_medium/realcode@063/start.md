## Product Requirement Document

Hey team, we need to build out that release automation thing we've been talking about forever. Basically the idea is: dev runs one command, tool looks at git history, figures out what version bump is needed, updates the version everywhere it needs to go, spits out a changelog, does the commit and tag. Pretty much like what the payments team built for their pipeline last quarter — same kind of conventional commit parsing logic.

A few things that came up in the last retro that we definitely need covered: people want to be able to override what version gets cut if they disagree with the auto recommendation. Also there's this whole pre-1.0 project situation where a breaking change accidentally pushed someone to 1.x when they weren't ready — need to guard against that. Pre-release builds (like alpha/beta/dev suffixes) should work too, and they should chain properly across multiple runs without resetting weirdly.

Changelogs need to stack correctly when you run releases repeatedly, and old-format headers from legacy projects shouldn't blow everything up. There should also be some escape hatches — dry run, skip certain stages, lifecycle hooks if someone needs to inject custom logic. Oh and make sure bad config surfaces as something readable rather than a raw crash. File formatting should be preserved when we write version bumps back. That's the gist, let me know what's unclear.

One extra pass on the version logic since a couple people asked how literal we want to be there: for conventional commits, 'fix:' = patch, 'feat:' = minor, '<type>!:' = major, implemented in the version recommendation module using commit subject prefix and exclamation-mark detection. So a breaking change is the exclamation mark form like 'feat!:' or 'fix!:', and that means a major bump with minor and patch reset to zero. Plain 'feat:' bumps minor, plain 'fix:' bumps patch, and anything else in the subject has no release weight at all and shouldn’t affect the recommendation.

Also on the pre-1.0.0 guardrail, that only applies when we’re computing the increment automatically. If the current version is below 1.0.0 and we see a breaking-change commit, we downgrade that by one level so it bumps minor and resets patch instead of jumping the project to 1.x. But if someone explicitly says '-r major' or '--release-as major', we should honor that and promote straight to 1.0.0.

For pre-releases, the chaining behavior should be consistent across reruns. When a pre-release identifier is active, repeated runs increment the trailing numeric counter (.0, .1, .2 …) as long as the recommended underlying increment does not exceed the level the active pre-release already represents. If a higher-level increment is requested or recommended, then we raise the base version and restart the counter at zero. And if there isn’t an explicit identifier, the suffix should just be numeric-only as '-0'.

A small but important thing on the commit/tag message templating too: every occurrence of '{{currentTag}}' gets replaced with the version string, so zero occurrences means leave the template as-is, and if it shows up multiple times then all of them get replaced. Same idea for the legacy '--message' option except the placeholder is '%s', and every occurrence of '%s' should become the new version. If nobody customizes anything, the default release commit and tag message format is 'chore(release): <version>'.

On the command we print after a release, it should always include 'git push --follow-tags origin master'. Then we only append a publish step as '&& npm publish' when a publishable primary manifest was bumped and the package is not marked private. For a pre-release, that publish part needs the dist-tag too, so use '--tag <identifier>' based on the pre-release label, like '--tag dev', and if the identifier is empty then use '--tag prerelease'.

And for changelog stacking, the legacy case is specifically when the existing file starts with free-form legacy header text ending in an old-style release anchor marker like '<a name=1.0.0>'. In that situation, strip the legacy header text before that anchor, keep everything from the anchor onward, and prepend the newly generated release section above that retained block. If there’s already a normal release section at the top, just prepend the new one above it and make sure the header line only shows up once. Same basic rule for legacy changelog seed text ending in an HTML anchor tag '<a name=X.Y.Z>' — the text before the anchor is stripped and the anchor block onward is preserved; see feature6_3_changelog_legacy.json.
