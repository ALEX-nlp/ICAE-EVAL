## Product Requirement Document

Hey team, we need to build out the coverage tracking system we've been talking about. Basically the idea is that we can record which lines of code actually ran during a real session, save that somewhere, and then pull it back later. We want it to work with at least two different storage options (like the one we used in the caching layer before, plus a simple file-based one). The tricky part is that if we run the same app on multiple servers or across deploys, the file paths are going to be totally different — we need some way to reconcile those so the reports make sense. Also we need merging: if the same file ran twice across two sessions, we don't want to overwrite the old data, we want to add it up. There should be a filtering step too so we can exclude vendor stuff and generated files. Oh and similar to the exclusion logic we wired up in that other collector module, there are a couple of built-in files that should always be excluded automatically on top of whatever the user specifies. The whole thing should be cleanly split into separate files/modules, not one big blob. We'll need a test runner that goes through a folder of JSON test cases and checks outputs. Let me know if anything is unclear but I think the shape of it is pretty obvious from context.

One small follow-up after the questions that came in: for storage, we do specifically need two interchangeable options, a file-backed store using 'file' and a networked key-value store backed by Redis using 'redis'. The important part is that they’re perfectly substitutable, so the accumulation/reporting side should only know about the shared storage abstraction and not care which concrete backend is underneath. And yes, that includes the Redis-backed networked key-value store, which is one of the two required pluggable storage backends alongside the file-backed store.

Also wanted to be more explicit about how the line arrays should behave when we merge them. A position marked null (no-data) means the line is not executable or was never instrumented. When merging two arrays, if both sides are null, the result is null; if one side is null and the other has a count, the result is that count. In other words, the null side is treated as zero only when the other side has an actual integer count, not null. Same rule applies whether we’re accumulating across snapshots or combining arrays from two hosts.

On the path reconciliation piece, normalization takes an ordered list of root-path prefixes called 'roots'. The last entry in the list is the local/canonical root. For each prefix, in order, if the file key starts with a match for that prefix, treated as a regular expression anchored at the start, we strip the matching prefix portion and replace it with the canonical root. That gives us the canonical path. If a key is already under the canonical root, it should come back effectively unchanged.

And on exclusions, the built-ins should always get appended after whatever the user provided, not mixed in ahead of time. The two built-ins are 'internal:prelude' and 'schema.rb', in that order, so the effective list is user rules first and then those two.
