## Product Requirement Document

Hey team, we need a small CLI utility for managing hosted script projects — think of it as a lightweight wrapper around the usual developer workflows. Basically devs are constantly complaining that they have to manually piece together editor links, remember which file type labels map to what, and clean up auth files from multiple spots when things go sideways. It's causing a lot of support noise.

The tool should handle a handful of things: showing helpful usage text for subcommands (similar to how we did it for the login flow in the auth module), converting project IDs into the right browser URLs, dealing with file type label translation in both directions (local ↔ remote), saving project config to disk, listing available APIs in a readable format, and gracefully handling mutation commands that aren't ready yet. Oh and logout needs to actually clean up creds from both places — we've had incidents where stale tokens caused weird behavior in CI.

For the API management piece, some subcommands are real, some are placeholders, and unknown ones should fail loudly rather than silently succeed. The output format should be consistent across all commands so our test harness can parse it easily — just flat key=value lines, nothing fancy.

Architecture-wise, keep concerns separated — don't just dump everything in one file. We want this maintainable long-term. Should be runnable as a single entry point that takes JSON on stdin.

A couple extra specifics from the questions that came in. The entry point should read a JSON object from stdin. For the command-style stuff, the input comes in an 'input' array of CLI argument strings, which is what we expect for Features 1, 6, 7, 8. For the string-based pieces in 2, 3, 4, 5, 'input' is just a plain string. For logout, 'input' is an object with 'local_credentials' and 'home_credentials' string fields. Also reiterating the separation we wanted: keep the core logic in src/, keep the adapter/entry-point separate, and keep rcb_tests/ for the harness and case files. The business logic needs to stay fully decoupled from stdin/stdout and JSON parsing, and the adapter is the only thing that should touch I/O.

For the help text, this needs to stay very literal because the parser is picky. The run command help must output exactly five lines in this order: exit_code=0, usage=Usage: run [options] <functionName>, description=Run a function in your Apps Scripts project, option=-h, --help  output usage information. Each field is a key=value line terminated by a newline. Same idea for logs, but with the full set and ordering preserved: exit_code=0, usage=Usage: logs [options], description=Shows the StackDriver logs, option=--json      Show logs in JSON form, option=--open      Open the StackDriver logs in browser, option=-h, --help  output usage information. The --json and --open options need to come before --help, and the spacing alignment there matters. More generally, use that same key=value help rendering pattern for the 'run' and 'logs' subcommands, specifically for exit_code, usage, description, and option fields as defined in Feature 1 test cases.

A few exact output/value rules too. The editor URL should be built as https://script.google.com/d/{scriptId}/edit, with script.google.com as the host and /d/<identifier>/edit as the path shape, and the identifier inserted verbatim. The output line for that is script_url=<url>. For file type mapping, SERVER_JS maps to local extension 'js'. All other remote labels are lowercased as-is, for example GS → gs and JS → js. The output line format there is local_type=<extension>.

On the harness side, the suite runs with 'bash rcb_tests/test.sh'. It accepts --cases-dir <subdir> and defaults to public_test_cases. For each case it writes raw stdout to rcb_tests/stdout/<cases-dir>/<filename_stem>@<zero_padded_index>.txt, for example feature1_1_run_help@000.txt. Those output files should contain only raw stdout and no PASS/FAIL metadata. Using different --cases-dir values should create separate namespaced output directories so runs never overwrite each other.

And one last thing on logout since that came up more than once: it really does need to delete both the project-local credential file and the user-home credential file. Those are two separate filesystem paths and both must be removed.
