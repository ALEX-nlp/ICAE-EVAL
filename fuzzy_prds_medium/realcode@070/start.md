## Product Requirement Document

We need a library that gives our periodic task schedules a portable, language-neutral representation so they can be saved to a key-value store and reloaded on any machine or process restart. We have several kinds of schedules: simple fixed-interval ones (run every N seconds), calendar-based ones (specific minutes/hours/days), and full recurrence rules with start times, end times, occurrence limits, and day-of-week constraints. When we persist these to storage and load them back, all the timing information including timezone offsets must survive intact — handle timezone preservation the same way the existing entry serialization logic does. The library should also expose two operational functions for recurrence-rule schedules: given the last time a job ran and the current time, compute how many seconds until the next firing (or report that there are no more firings); and decide whether the schedule is currently due to fire, also reporting when the next firing after this one is expected. All inputs to these computations must be explicit (no relying on the system clock) so results are deterministic and testable. Datetime values need a clean round-trip path through epoch seconds — convert a timezone-aware moment to a whole-second timestamp and back, preserving the UTC offset. The serialized format uses a type discriminator field and stable key ordering.

One small follow-up after the team questions: for timezone storage in the JSON, the 'timezone' field should mirror exactly what kind of timezone we started with. If it’s a datetime with a fixed numeric UTC offset rather than a named IANA zone, store it as a number for the total seconds of the UTC offset, like +8 hours = 28800.0. If it’s a named IANA zone, store the zone name string. If it’s UTC or there’s no timezone, use the literal string 'UTC'.

Also, for the rrule payload, the JSON object key order needs to stay fixed as __type__, freq, interval, wkst, count, bysetpos, bymonth, bymonthday, byyearday, byeaster, byweekno, byweekday, byhour, byminute, bysecond, and then dtstart/dtstart_tz and until/until_tz when those anchors are present. From wkst through bysecond, those keys should always be there even when we’re not using them, and in that case they should be null. And for freq, please use the numeric codes exactly as YEARLY=0, MONTHLY=1, WEEKLY=2, DAILY=3, HOURLY=4, MINUTELY=5, SECONDLY=6.
