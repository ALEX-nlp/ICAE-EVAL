## Product Requirement Document

Hey team, we need to wire up the new image redirection logic for our cluster admission setup. Basically the idea is that when a pod comes in, we want to automatically rewrite its container images so they pull from our internal registry instead of wherever they were originally pointing. This is the same kind of 'transparent swap' pattern we talked about in the infra sync last quarter.

A few things I know we need: there should be some way to configure which pods/containers actually get redirected (not everything should be redirected blindly), and we need to handle the auth side so the mirroring process can actually pull from source registries. Also the system should be resilient — if someone passes in a bad config keyword it shouldn't just blow up, it should degrade gracefully.

One thing I'm not totally sure about is how image names get normalized when they're rewritten — like what happens with images that have both a tag and a digest, or bare names like 'nginx' with no registry prefix. I think there was some prior art in how we handled image references in the old registry-compat module but I don't remember the exact rules.

The output format for the CLI adapter needs to match what the test harness expects, so please check the existing test cases carefully. Let me know if anything is unclear — I'll try to get back to you same day.

A couple follow-ups from the questions that came in. For the fallback behavior, if someone passes an unrecognized swap-mode keyword, we should not hard-fail it. We fall back to the safe default `exists`, and the output needs to be three lines in this exact order: `policy=exists`, then `error=unknown_swap_policy`, then `input=<the offending value>`. Same idea for copy timing, except the fallback there is `delayed`, not `immediate` or `force`. In that case the output should be `policy=delayed`, then `error=unknown_copy_policy`, then `input=<the offending value>`, again in that exact order. The recognized keywords there are `delayed`, `immediate`, and `force`.

On the selection side, the filters live under `source.filters` in the YAML config document, and each one is just an object with a `jmespath` key. The adapter output should start with `filters=<count>`, then print one `filter[<index>].jmespath=<expr>` line per filter, keeping declaration order exactly as written. If the YAML document is empty or blank, that should just come out as `filters=0` and nothing else after it. For evaluation, the expression context has two top-level keys: `obj` and `container`. `obj.metadata.namespace` reflects the pod's namespace, and `container.name` plus `container.image` are available for the current container. If the expression returns boolean `true`, that's a match, and boolean `false` is a non-match. If it returns something non-boolean, report `reason=non_boolean_result`. If it can't be evaluated at all, report `reason=evaluation_error`. In both of those cases we still keep going.

Also wanted to pin down the secret handling since that came up. We retain every named document verbatim and build one merged JSON object from all top-level keys. The output is `secret_count=<n>`, then one `secret[<name>]=<verbatim-content>` line per secret in sorted name order, alphabetical, and then `aggregate=<merged-json>`. The merged JSON should also have keys in sorted order, and if the same top-level key shows up more than once, later documents win. And just to anchor this back to the PRD mapping: Feature 6 (`swap_images`) is the admission-time image rewriting engine that processes AdmissionReview objects and emits JSON-patch replace operations for every container and init-container image. Features 5.1 and 5.2 are secret aggregation (`aggregate_secrets` op) and pull-credential resolution (`resolve_pull_secrets` op) with the kubernetes and noop resolver modes.
