## Product Requirement Document

Hey team, we need to build out a fluent builder toolkit for our Places SDK wrapper. Basically devs keep complaining that constructing place objects and service request objects is super tedious and error-prone — they have to remember which fields are required vs optional every single time, and the boilerplate is everywhere. We want a single clean entry point per object type where you pass in the required stuff upfront and then configure anything optional in one trailing block.

There are two main groups of things to build: the data model side (photos, plus codes, address stuff, places, opening hours, predictions) and the service request side (fetch place, fetch photo, find current place, autocomplete, text search). The request objects also need to support cancellation — similar to how we handled it in that token-lifecycle pattern from the last SDK module — where you can prove the request is actually holding a live handle and not just a snapshot.

Also important: the codebase needs to be structured cleanly with the JSON adapter stuff kept totally separate from the core builder logic. No leaking of output/formatting concerns into the domain layer. We want this to be extensible so adding new object types later doesn't require touching existing code.

For location biasing on some requests you can do either a box (two corners) or a circle (center + distance). Price levels on text search have a specific internal integer mapping we need to handle correctly. Ping me if anything is unclear.

One small follow-up after the questions that came in: for photo-ish data, if no attribution is provided it must default to an empty string, not null and not 'none', and when width or height are omitted they must each default to integer 0. The output lines there are exactly attributions=<text>, width=<int>, height=<int>. On the address side, if the short name is missing from an address component, we should report the literal string 'none' in lowercase. So the line is short_name=none, not an empty string and not null.

Also wanted to pin down a couple formatting/details things so nobody has to guess. For place output, the order matters: first 'address=<value>', then 'address_component_count=<n>', then for each component at index i we print 'address_component[i].name=<name>', 'address_component[i].short_name=<short_name or none>', and 'address_component[i].types=[<type1>, <type2>, ...]'. For opening hours periods, it is valid for a period to have only a closing boundary and no opening boundary. If the opening boundary is absent, report the single line 'period[i].open=none'. If it exists, then it comes out as 'period[i].open.day=<DAY>' and 'period[i].open.time=HH:MM'. Time formatting should always be zero-padded HH:MM, so hour=0, minute=0 → '00:00' and hour=9, minute=5 → '09:05'.

On fetch-photo requests, if max_width or max_height are not set, they should each come through as the literal string 'none', specifically 'max_width=none' and 'max_height=none'. And on the cancellation side, yes, use the same CancellationTokenSource / CancellationToken pattern from the Background terminology section of start.md: the source owns the token handle, cancelling the source flips isCancellationRequested from false to true, and the request needs to hold the live handle rather than a copy so that flip is visible through the request object.

Last thing, just to make the architecture expectation extra explicit: the JSON parsing, command routing, and stdout formatting all need to live in a physically separate execution adapter layer. The core builder/construction logic should know nothing about JSON or stdout. We still want the split to stay clean between (1) data-model construction helpers and (2) service-request construction helpers, with (3) the adapter layer kept physically separate from both. And adding a new constructible kind should not require modifying existing builder code.
