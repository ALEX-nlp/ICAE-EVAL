## Product Requirement Document

Hey team, we need to build out that HTML content processing library we've been talking about. Basically, users are writing raw text and Markdown in our platform and it's showing up broken, unstyled, or sometimes with weird script injections — had a few complaints this week. We need a pipeline that handles all the usual stuff: wrapping plain text safely, rendering Markdown, making URLs clickable, replacing those colon emoji shortcuts with actual images, fixing image paths so they work across environments, proxying external images through our asset service, upgrading HTTP links to HTTPS for our own domain, making images responsive, and handling @mentions so they link to profiles. There's also a need for extracting mentioned usernames from raw Markdown (remember how we did the mention scanning in that social notifications module last quarter — same idea), sanitizing unsafe tags/attributes, syntax highlighting for code blocks, and generating anchor nav links for headings. The emoji image URLs need special handling for names with weird characters — just follow the same URL-encoding approach we've used elsewhere. The proxy feature uses HMAC with the secret to build the URL — the exact algorithm is the same as what the asset team documented internally. Output format details are a bit fuzzy for some features so check the existing test fixtures. Missing config should be reported gracefully rather than just crashing.

A couple extra specifics from the questions that came in. For plain text, we should treat it as `text` only, wrap it in a `<div>` element, HTML-escape it, and make sure the final output ends with a trailing newline. So input `howdy pahtner` should come out as `<div>howdy pahtner</div>\n`, and characters like `<` and `>` need to become `&lt;` and `&gt;`. Along the same lines, the features that expect a `text` string input, specifically the plain text wrapping and Markdown rendering paths, should reject parsed HTML input with `error=invalid_input_type` and include the usual companion field saying what input category was expected, using the normal key=value error format we use everywhere else.

On Markdown, there’s also a `soft_line_breaks` boolean flag. When `true`, newlines inside paragraph text should turn into `<br>\n` sequences. When `false` or omitted, those newlines stay as literal newlines inside the paragraph element. List structure is unaffected by this flag. For mention collection, same basic behavior we discussed before, but to be explicit: Feature 10 mention collection logic in `rcb_tests/public_test_cases/feature10_mention_collection.json`: extract @username tokens from raw Markdown, deduplicated, ordered by first appearance, excluding emails/domains/teams/inline-code, output as `mentions=name1,name2\n`.

For sanitizing, unsafe elements like `<script>` and `<style>` should be removed entirely, except for `<style>` where the inner text content is preserved and only the tag is removed. Unsafe attributes like inline `style` get stripped from otherwise allowed elements. Invalid table and list structures should be corrected too. The allowlist approach still applies here: allowed elements and attributes stay, everything else gets stripped while preserving safe inner content.

For code highlighting, if it blows up because the language isn’t supported or there’s some processing failure, the response needs to be exactly `error=processing_failed\ncategory=mentoserror\n`. This applies regardless of whether a `default_language` was supplied or the language was set on the `<pre>` tag.

For heading anchors, each heading gets an injected anchor of exactly `<a id="{slug}" class="anchor" href="#{slug}" aria-hidden="true"><span class="octicon octicon-link"></span></a>` right after the opening heading tag, with a preceding newline. Slugs should be lowercased, spaces replaced with hyphens, and special characters removed. If the same heading text shows up more than once, use suffixed slugs like `-1`, `-2`, and so on. Also, the `output` field has to be `annotated_html` to get that format.

For autolink, there’s an optional `enabled` boolean field. If `enabled` is `false`, return the HTML fragment completely unchanged, no URL-to-anchor conversion at all. If `enabled` is absent or `true`, then bare URLs in text nodes should be wrapped in `<a href="...">` anchors.

And one more concrete detail on image proxying since people asked for the exact shape. Feature 6 image proxying in `rcb_tests/public_test_cases/feature6_image_proxying.json`: proxy URL = `{proxy_endpoint}/{HMAC-SHA1-hex(secret, original_url)}/{hex-encode(original_url)}`, and you can verify against the expected output `a5ad43494e343b20d745586282be61ff530e6fa0/687474703a2f2f747769747465722e636f6d2f696d672e706e67`.
