## Product Requirement Document

We need a server-side toolkit that lets developers emit lightweight placeholder elements in a page and have their real content filled in later over a realtime channel, so slow or expensive parts of a page don't block the initial response. A developer should be able to mark a region with one helper call. That helper produces a placeholder element that embeds a tamper-proof, cryptographically signed description of what should eventually be rendered. A realtime channel then reads that signed description, renders the real content through the correct controller, and streams it back to replace the placeholder. The toolkit needs a configurable default controller — when no specific controller is specified, rendering falls back to the application's base controller, similar to how the framework's own controller resolution works. Controllers should be resolved from signed identifiers so a client cannot request an arbitrary controller. The placeholder element builder should handle single templates as well as collections (one element per item), with each item getting its own signed payload carrying the item and its position. There should be an eager-loading option that marks the element for immediate rendering when no placeholder content is provided. Reserved data attributes that carry the signed payload must be stripped if a caller attempts to supply them, to prevent forgery. The toolkit also needs an out-of-band renderer that can evaluate templates in the context of a reconstructed HTTP request (with environment variables, query parameters, and explicit params), and a stream-name builder that computes realtime channel names from a channel identifier and connection-level object IDs.

One small follow-up from the questions that came in: the test bypass setting should default to off, and we’re surfacing that as 'skip_in_test=false'. Also, when something is absent, we should render the missing value as the literal token '_none_' rather than leaving it vague or implied, so think in terms of 'class=_none_', 'data-action=_none_', 'eager=_none_', and the same pattern anywhere that value is missing.

Also worth calling out more plainly on the element attrs side: if someone passes html_options.data, there are two keys we still have to treat as reserved and strip before anything goes out, which are 'sgid' and 'signed_params'. Those are the protected bits and callers don’t get to override them. Any other data attribute should stay on the element as-is, and it should also be folded into the signed payload.
