## Product Requirement Document

Hey team, we need to build that proxy accelerator thing we've been talking about. Basically it sits in front of our app server and handles all the boring stuff so the app doesn't have to. The big things are: forwarding requests to the upstream without mangling them, some kind of response reuse so the origin isn't hammered with the same request over and over, fast file delivery where the app just points at a file path and the proxy handles sending it (there was a nasty bug with compressed files last time, make sure that's handled properly — similar to how we fixed the encoding issue in the asset pipeline), and we need a log line for every request so ops can debug things. There's also a config system that reads from the environment and args, and an in-memory store for the cache backend. Structure this properly please — last time someone built something like this it ended up as one giant file and nobody could maintain it. Split things out by responsibility. The whole thing needs to be driven from stdin/stdout for the test harness but keep that adapter layer separate from the actual logic. Check the existing test scaffolding for the exact input/output shapes — the cases in the rcb_tests folder should tell you what you need. Make sure edge cases like oversized payloads, missing upstream command, and private/no-store directives are handled explicitly.

One extra pass on the cache behavior since there were a few good questions there: something is storable only if both things are true at the same time — the response is explicitly shareable with `public`, and it has a positive freshness lifetime with `max-age` greater than zero. So if it’s `private`, missing `public`, has `no-store` or `no-cache`, has `max-age=0`, or has no `max-age` at all, we should NOT store it. Same deal for statuses: only HTTP 200 OK gets stored. Anything informational, redirects, 304, and any 4xx/5xx should never go in. Also if the body is bigger than the configured size limit, don’t store it either. And for lookup/keying, only GET is even in the running for caching — POST and any other non-GET should always skip cache with disposition `bypass`.

On the cache key itself, please make it from the HTTP method, the URL path, and the full set of query params. Query params need to be order-independent, so `?a=1&b=2` and `?b=2&a=1` land on the same key. If the origin sends a `Vary` header, then fold in the values of exactly those named request headers too, so different values there become different cache entries. Anything not named in `Vary` should be ignored for keying.

Also wanted to be very explicit on the sendfile bug we hit before. When X-Sendfile is on and the origin response has a `Content-Encoding` header, we still need to set `Content-Length` to the real on-disk byte size of the file, not whatever the origin said, since that may be zero or just wrong and causes truncated downloads. In that flow, `Content-Type` comes from sniffing the file, `Content-Length` is the true file size, `X-Sendfile` gets stripped before the client sees the response, and any response body from origin is thrown away. If the feature is off, then `X-Sendfile` should just pass through unchanged. This is the same Content-Encoding + Content-Length fix in feature3_sendfile: if origin sets `Content-Encoding`, override `Content-Length` with the actual file size on disk instead of forwarding the origin value.

For logging, the line needs to include exactly these fields in this order: `path`, `method`, `status`, `remote_addr`, `user_agent`, `req_content_type`, `resp_content_type`, `req_content_length`, `resp_content_length`, `query`, and `cache`. `path` is without the query string, `remote_addr` comes from `X-Forwarded-For` if that request header exists and otherwise from the direct client address, `req_content_length` is the byte length of the request body, `resp_content_length` is the bytes actually written in the response body, and `cache` is whatever X-Cache disposition came back from the downstream side like miss/hit/bypass. No timing fields.

And for anyone wiring against the harness, the stdin JSON action shape and the stdout `key=value` line format are already laid out in the files under `rcb_tests/public_test_cases/`: `feature1_proxy_routing.json`, `feature2_1_caching.json`, `feature2_2_keying.json`, `feature2_3_vary.json`, `feature3_sendfile.json`, `feature4_logging.json`, `feature5_memcache.json`, and `feature6_config.json`.
