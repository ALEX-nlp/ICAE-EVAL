## Product Requirement Document

We need a reusable abstraction that wraps a single persisted value — identified by a string key stored in a backing key-value store — and exposes it as an observable stream. When you write a new value it should persist to the store and push the update to every active observer. When you remove the value it should be deleted from the store and every observer should fall back to the configured default. When a new observer is attached, it should immediately receive whatever value is currently in the store (or the default if nothing is stored), without any write happening. The stream should deduplicate: if you write the same value twice in a row, the observer only fires once for the first change, not again for the repeat. Multiple observers on the same value should all receive every change in attachment order. If the value is constructed with no key at all, both write and remove should be rejected with a clear error — handle that the same way the validation layer handles empty required fields: refuse the operation and report why.

One small follow-up from the questions that came in: for output, let’s keep the wording exact. Observer emissions should come through as 'emit[<i>] <value>\n'. Store writes coming from put should be 'store write key=<k> value=<v>\n', and removes should be 'store remove key=<k>\n'. The operation result lines are 'put ok=<bool>\n' and 'remove ok=<bool>\n'. And for the null-key case, both operations should fail with the exact error line 'error=unsupported_operation op=<put|remove> reason=null_key\n'.

Also, 'preset' is just a side-channel write used to seed the backing store directly WITHOUT notifying existing observers, and it produces no output lines itself. The only thing that matters there is that observers attached after a preset should treat that preset value as their initial value. This is only for test setup, not normal behavior.

One other nuance on deduping: the observer only emits when the re-read value differs from the last value it emitted. So if you write the same value as the current one, it should still go through to the store, which means you still get the 'store write' line and 'put ok=true', but there is NO new observer emission. When the value is actually different, each distinct new value should emit once.
