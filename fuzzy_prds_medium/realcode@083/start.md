## Product Requirement Document

Our package management platform needs a robust version intelligence engine that enables developers to work with software release strings and the rules that govern compatibility between them. The engine must handle the full spectrum of version formats found in the wild — numeric releases, pre-release markers, date-stamped builds, development branch names, and aliased identifiers — and convert any recognized input into a single canonical comparable form that makes ordering and equality checks reliable and consistent.

Beyond normalization, the engine must support filtering a list of candidate releases against a range expression to find what is acceptable, sorting releases from oldest to newest or newest to oldest, and evaluating whether two arbitrary range rules can ever be satisfied by the same release at the same time. Range expressions themselves can be constructed from exact pins, directional comparators, wildcard slots, tilde shorthand, caret shorthand, and inclusive hyphen spans, and multiple clauses can be combined with AND or OR semantics.

Error cases must be surfaced as neutral, language-agnostic category codes — never as raw exception messages — so that the surrounding tooling remains decoupled from implementation details. The system should follow the separation logic described in the internal "stability rules" module and the "branch normalization" pathway. All behavior must be exercised through a thin adapter layer that routes JSON commands to the core engine and prints plain-text results.

A couple follow-ups from the team questions. On normalization, the canonical result always comes out as exactly four dot-separated numeric segments, so think '1.0.0.0', with any missing pieces padded using '.0', and the returned text is that four-segment string plus the newline character '\n'. If something can’t be normalized, the output is exactly 'error=invalid_version\n' and that’s all we expose. No exception type, no stack trace. The cases that should land there include empty string, non-numeric junk like 'a', unknown stability words like 'meh', more than four numeric segments, arbitrary non-dev names, and build metadata with whitespace. Also, the branch aliases 'master', 'trunk', and 'default', whether they show up bare or with a 'dev-' prefix, all normalize to '9999999-dev' in both normalize() and normalizeBranch().

On branch matching, the important default is that without the 'compareBranches' flag, or if it’s false or just absent, anything starting with 'dev-' is NEVER compatible with a numeric bound, so constraintMatch always returns 'false\n'. Two dev branches only line up under '==' if their full names are identical. If compareBranches: true is set, then a dev branch is treated as ordering BELOW any numeric version, which means a '<' upper bound accepts it and a '>' lower bound rejects it.

A couple range details were worth pinning down too. For tilde, if there’s no explicit stability, the lower bound gets a '-dev' suffix, so '~1.2' becomes '[>= 1.2.0.0-dev < 2.0.0.0-dev]'. The upper bound increments the segment one position more significant than the last specified digit. So '~1' and '~1.0' both yield '[>= 1.0.0.0-dev < 2.0.0.0-dev]', and '~1.2.3' yields '[>= 1.2.3.0-dev < 1.3.0.0-dev]'. For a hyphen range 'A - B', if B is partial, meaning 1 or 2 segments like '2' or '2.1', the upper side is exclusive and you increment the last supplied segment. If B is complete, meaning 3 or 4 segments like '2.3.4' or '2.3.4.5', the upper side is inclusive at the fully normalized B. The examples to keep using are '1 - 2' -> '[>= 1.0.0.0-dev < 3.0.0.0-dev]' and '1.2.3 - 2.3.4.5' -> '[>= 1.2.3.0-dev <= 2.3.4.5]'.

And for sorting behavior, in ascending sort with the 'sort' action, a pre-release like '2.4.0-alpha' comes BEFORE the corresponding stable '2.4.0'. In descending 'rsort', that flips so stable comes before pre-release. If two items compare equal, keep their original relative order from input, so this stays a stable sort, and the output should preserve the original input string form of each version.

Last thing, the recognized stability words and shorthand still come from the same place: the parseStability() static method in src/VersionParser.php and the $modifierRegex / $stabilities private static fields that define recognized stability words (stable, RC, beta, alpha, dev) and their shorthands.
