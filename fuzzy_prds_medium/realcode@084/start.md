## Product Requirement Document

Our command-line tool accepts two kinds of repeatable string options that users type at the prompt: HTTP header definitions and URL route mappings. Right now parsing is scattered and inconsistent. We need a small option-parsing library to clean this up. For headers: the user writes something like 'Header-Name: value', and we need that decoded into a map from name to value — the value is everything after the first colon, whitespace trimmed, and names should be handled case-insensitively so duplicates are collapsed. For route mappings: the user writes 'source=destination', and we split on the first unescaped equals sign — if someone needs a literal equals sign in the source or destination path they can escape it with a backslash. The library should accept a list of raw strings and produce a structured map, outputting an entry count followed by one line per entry in input order. Malformed mappings — missing separator, empty source, empty destination, or more than one unescaped separator — should fail with a descriptive error message that tells users exactly what the expected format is. Follow the same error-type convention used in the validation layer for other options in this project.

One small follow-up from the questions that came in on route mappings: when one of those is invalid, it throws an ArgumentException. The output is: 'error=ArgumentException\n' followed by 'message=The format of the key-value pair is invalid. It must contain exactly one \'=\' separator. Make sure non-separator \'=\' characters are escaped (\'\\=\').\n'.

Also, the validity rule here should be read pretty literally: a mapping is valid when, after trimming, it contains exactly one unescaped '=' with non-empty text on both sides. So we should reject the empty string, whitespace-only, no separator, empty left or right side, or more than one unescaped separator. And just to make the escaping behavior explicit, a '\=' sequence in the input is treated as a literal equals sign, not a separator. After splitting on the single unescaped '=', each escaped '\=' on both sides is converted back to a plain '=' in the result.
