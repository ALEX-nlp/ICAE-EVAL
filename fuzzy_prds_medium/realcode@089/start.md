## Product Requirement Document

Hey team, we need a small utility library for container image version tracking. Basically our platform team keeps running into issues where they can't reliably tell if a newer image is available — the tag formats are all over the place and every engineer is hand-rolling their own string parsing which keeps breaking in subtle ways (especially with those debian-style suffixed tags, you know the ones). We also keep getting burned by false positives when detecting which registry a host belongs to — someone's test environment had a domain like `foodocker.io` and it was being picked up as Docker Hub, which caused some fun incidents.

The library should handle parsing tags into something comparable, doing the older/newer check correctly (there's some tricky logic here similar to how we handled pre-release ordering in that semver validation work we did for the auth service — check how that was approached), splitting image paths per registry since each one has different conventions for single-segment paths, and routing a full image URL to the right registry handler.

We need this to be properly structured — not just one big file — with clean separation between the version logic and the registry-specific stuff. Each registry (Docker Hub, GCR, Quay) should live in its own unit. The whole thing needs to be testable via a JSON-in / line-based-out adapter but that adapter should stay decoupled from the core logic. Edge cases like empty strings and unknown registries must not crash anything.

One extra pass on the version behavior since a few people asked where the boundaries are. If a tag does not begin with a digit after stripping an optional leading 'v', we treat the version as 0.0.0 and keep the entire original tag string verbatim in metadata, including that leading 'v' if it was there. So for example, 'hello-1.2.3' yields major=0, minor=0, patch=0, metadata=hello-1.2.3. Also, any missing numeric piece defaults to 0, so '1.2' parses as major=1, minor=2, patch=0, metadata=empty. We only consume up to the first three dot-separated numeric groups as version numbers, and the first non-numeric dot-group stops that numeric parsing, with everything from there on becoming the metadata suffix.

Also worth calling out the empty-tag behavior because that should short-circuit everything else. If either tag in a comparison is empty, ordering is determined purely by string length: the shorter string is 'less than' the longer. If both are the same length, including both empty, less_than is false. That rule wins before any of the regular version ordering logic kicks in.

On release ordering, the clean tag should win over the suffixed one when the numeric version is otherwise the same. So if first has no metadata and second does with the same major/minor/patch, less_than is false. The inverse is also true: a pre-release of a given version is older than the final release of that same version. And when both sides do have metadata on an equal major/minor/patch, each suffix gets tokenised into alternating runs of digits and non-digits. Digit runs are compared as integers, so r9 < r39 because 9 < 39, and non-digit runs are compared lexicographically, so 'alpha' < 'beta'. We compare token by token from left to right, and if one side runs out first, the shorter token list loses on exhaustion. Same metadata suffix comparison logic as described in feature2_version_comparison.json: tokenise each suffix into alternating digit/non-digit runs, compare digit runs numerically and non-digit runs lexicographically, left to right, with the shorter token list losing on exhaustion.

And on structure, yes, this really does need to stay split by responsibility. The version parsing/comparison logic lives in its own module, and each registry (Docker Hub, GCR, Quay) lives in its own sub-unit under a registry/client package. The JSON stdin/stdout adapter is a separate execution layer and should not get mixed into the core domain logic. The router should depend on a registry abstraction (interface), not concrete registry implementations directly, so adding a new registry is adding a new unit instead of having to edit the router every time.
