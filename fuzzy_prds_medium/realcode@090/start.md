## Product Requirement Document

Hey team, we need a CSV conversion library built out — something devs can drop in to handle reading and writing tabular data without rolling their own brittle parsers every time. The big pain points we keep hearing from users: data gets corrupted when fields have commas or newlines in them, spreadsheet files don't open cleanly in Excel because of encoding weirdness, and pipelines that receive data in chunks keep breaking mid-stream because the parser can't stitch things back together. We want both directions covered (writing rows to text AND reading text back into rows), with sensible defaults but enough knobs to handle the weird stuff.

One thing — make sure the quoting behavior works like we did in the formatting module from the analytics pipeline (you'll know it when you see it), specifically around how special characters inside fields get handled. Also need it to work with those spreadsheet hint lines that some exports spit out at the top.

On the decoding side, it'd be great if numbers and booleans came back as actual typed values instead of strings when the caller wants that. Validation upfront would also save a lot of runtime surprises. The output format for the adapter layer needs to be consistent — ask me if you're unsure what the rendering looks like for headers vs plain rows. Multi-file structure please, this isn't a quick script.

Quick follow-up from the questions that came in: defaults should be a comma (`,`) for `fieldDelimiter` and `\r\n` for the line delimiter, and that should be true on both encode and decode unless config says otherwise. Default quoting is `necessary`, meaning we only wrap a field if it contains the field delimiter, a line break, or the quote character itself. The other supported modes are `always` and `strings`, and for `strings` the idea is quote only textual/string-typed fields and leave numbers and booleans unquoted. Also, on the encode side, please match the same quote/escape behavior we already called out before, including the escape-by-doubling behavior.

A couple more specifics on streaming/chunked behavior: `encode_stream` takes `rows` (array of arrays) and emits incrementally, with no leading line delimiter before the first row and the line delimiter prefixed before each later row. If you concatenate all fragments, it should exactly equal the full CSV text, and the output format here is just the raw CSV text, same as `encode`. For decoding, `decode_batches` is basically `decode_chunks` plus grouping. Without `maxRowsPerBatch`, each chunk gives you one batch with all rows decoded from that chunk. With `maxRowsPerBatch: N`, rows from a single chunk are split into batches of at most N rows. Output format is `batches=<n>` then `batch[i] rows=<k>: [ <row> ] [ <row> ] ...`, where each row is rendered as `<tag>=<json> | ...`.

Also, to be explicit about rendered decoded values, each decoded field is rendered as `<tag>=<json>` where tag is `str`, `int`, `double`, or `bool`, and json is the JSON encoding of the value. So strings look like `str="hello"`, numbers are unquoted like `int=123` or `double=1.5`, and booleans are `bool=true` / `bool=false`. Fields inside a row are separated by ` | `. Each row line ends with a newline, and the whole output always ends with a trailing newline after the last row line.

For validation, `validate_decoder` is the upfront config check. `quoteCharacter` and `escapeCharacter` must each be exactly one character. If `quoteCharacter` is multi-character, return `error=invalid_quote_character\n`; if `escapeCharacter` is multi-character, return `error=invalid_escape_character\n`. A valid config returns `valid=true\n`.

One subtle delimiter thing: `fieldDelimiter` can be multi-character, for example `::`. In that case the full sequence has to be there to split fields, and a partial match like a single `:` when the delimiter is `::` is just normal content. In chunked decoding, if that multi-character delimiter gets split across chunk boundaries, it still needs to buffer and rejoin correctly. And if someone is using a multi-character delimiter, `autoDetect: false` needs to be set.

On line endings and BOM handling, if `\r\n` is split across a chunk boundary so one chunk ends with `\r` and the next starts with `\n`, that still counts as one line terminator, not two separate events, so we should not create a fake empty row from that split no matter what `skipEmptyLines` is set to. If there’s a UTF-8 byte-order mark (`\uFEFF`) right at the start of input, consume it and do not let it become part of the first field value. That applies whether the delimiter is explicitly supplied or inferred, and also in chunked/streaming decode modes. On encode, `addBom: true` prepends `\uFEFF` to the output, the `excel` preset should imply that, and by default `addBom` is `false`.

Last thing, yes, we do want the spreadsheet delimiter hint behavior: a leading `sep=;` line should set the delimiter and should not come through as data.
