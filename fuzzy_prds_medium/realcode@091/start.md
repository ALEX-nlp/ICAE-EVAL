## Product Requirement Document

Hey team, we need to build out the recaptcha verification client library. Basically the idea is that right now devs have to hand-roll all the request building, response parsing, and validation logic themselves and it's a mess — people keep missing edge cases around stale challenges or mismatched hostnames and we keep getting bug reports about it. We want a clean client that handles all of that in one place.

The library needs to validate the shared secret upfront (similar to how we handled the auth token checks in the payment gateway module — check that code for reference), build the outbound request payload, parse whatever the verification service sends back, run some optional local checks on the parsed result, and support at least two different ways of actually sending the request over the network.

One thing that's been causing production issues lately is that users are completing the challenge but then the backend is accepting it way too long after the fact — we need some kind of freshness enforcement on that. Also there have been complaints that the library crashes with cryptic internal errors instead of returning something the caller can actually handle gracefully.

The response format needs to be consistent across all operations and the two transport strategies need to be swappable without changing the core logic. Reach out if anything is unclear, but try to check the existing test fixtures first before pinging me.

A couple extra specifics so nobody has to guess at edge behavior. On client creation, we need to reject the bad secret values before any verification attempt happens: empty string, null, numeric zero, an object/array type, and an empty array. Same rule applies anywhere this validation is exercised in Feature 1 (create_client operation): empty string, null, numeric zero, object type, and empty array are all invalid, producing `error=invalid_secret` without exposing runtime exceptions — see rcb_tests/public_test_cases/feature1_secret_validation.json. In general, if we hit one of these invalid secret inputs, the caller should just get `error=invalid_secret` and not any leaked internal exception messages or language runtime errors.

For request building, `build_request` should return exactly two lines. First `fields=<json_object>`, then `query=<form_encoded_string>`. The field object should use compact JSON encoding, with the example staying exactly `fields={"secret":"SECRET","response":"RESPONSE","remoteip":"REMOTEIP","version":"VERSION"}` followed by `query=secret=SECRET&response=RESPONSE&remoteip=REMOTEIP&version=VERSION`. The outbound payload always includes `secret` and `response`. The optional fields `remoteip` (mapped from remote_ip) and `version` (mapped from client_version) only show up when non-null, and if they’re null they should be omitted entirely from both the field object and the query string. Also keep the query string field order as secret, response, remoteip, version.

One other early return case: if the user response token is an empty string, do not contact the verification service at all. Just return `success=false` with error code `missing-input-response` and render all the metadata fields as `null`, specifically `hostname`, `challenge_ts`, `apk_package_name`, `score`, `action`.

On parsing, if the service says `success: false` and the `error-codes` array is missing or empty, synthesize a single error code `unknown-error`. If the JSON response can’t be parsed at all, use `invalid-json`. For the local string checks, `hostname`, `apk_package_name`, and `action` comparisons are case-insensitive. If there’s a mismatch, append the matching code to `error_codes` and flip success to false, while preserving any original service error codes. The mismatch codes are `hostname-mismatch`, `apk_package_name-mismatch`, and `action-mismatch`.

And for freshness, the `challenge_timeout_seconds` behavior in Feature 4.3 should add `challenge-timeout` to `error_codes` when `challenge_ts` from the service reply is older than the configured number of seconds at execution time — see rcb_tests/public_test_cases/feature4_3_challenge_timeout_check.json.
