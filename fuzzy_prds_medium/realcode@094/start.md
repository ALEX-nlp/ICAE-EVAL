## Product Requirement Document

Hey team, we need to get the location library wrapped up. There are a few rough edges that have been causing headaches for the mobile devs. First, the request configuration thing — devs keep complaining that when they dial down the update frequency, the 'floor' value doesn't automatically adjust to stay in sync, which causes weird behavior downstream. Also the priority validation isn't giving them a clean error object they can actually handle in code, it's just blowing up. Second, the 'best fix' selection logic — similar to the tiering rules we figured out for that login module's credential freshness check — we need some concept of a 'recency window' where fresh beats stale, but if everything is stale we still give them something rather than nothing. Third, mock mode is a mess: toggling it on and off while subscriptions are live leaves ghost listeners around, which breaks the listener count assumptions in tests. Finally the GPX replay needs to output a compass heading between consecutive points, but the first point has no previous reference so it should indicate it has no bearing. Please check how the filtering/dispatch layer handles simultaneous fixes from two sources too — only one should get through. I think Rodrigo left some notes on the accuracy comparison logic somewhere in the source, worth grepping for that.

Quick follow-up after the questions that came in: for a brand-new LocationRequest, the built-in defaults should be interval=3600000 ms, fastest_interval=600000 ms, smallest_displacement=0.0 meters, priority=balanced. On the sync behavior between the timing fields, the rule is just that whenever the interval is set to a value strictly below the current fastest_interval, the fastest_interval is automatically lowered to equal the new interval. If interval is set to a value at or above the current fastest_interval, fastest_interval stays as-is.

On the priority issue, unsupported values should not just explode. They need to come back as the normalized two-line output 'error=invalid_priority\npriority=<raw_value>\n', with the offending raw value included exactly in that second line no matter what type it was, and the priority field must NOT be updated.

For the best-fix / last-known behavior, the freshness cutoff is the same idea throughout: a fix is fresh if its timestamp is within 60000 milliseconds of the current time, meaning within 60000 ms of 'now'; otherwise it’s stale. Fresh fixes compete on accuracy, and if all stale, the most recent stale fix wins. The comparison order should be: a null candidate A is never preferred, A is always preferred over a missing/null B, if A is newer than B by more than 60000000000 ns then A wins regardless of accuracy, a fix without accuracy loses to one that has accuracy, and if both have accuracy then the smaller accuracy radius wins. That lines up with the last-known selection logic in FixSelector / BestFixSelector and the notes in feature2_last_known_location.json.

A couple more lifecycle bits too: before connecting, both the location API and geofencing API should be absent/unavailable. Connecting makes both available and sets connected=true. Disconnecting sets connected=false, makes both APIs absent again, and unsubscribes ALL active update listeners so the listener count returns to 0.

And on filtering, a fix should be dropped if the distance from the previously delivered fix is less than smallest_displacement meters. If smallest_displacement is 0.0, which is the default, then there’s no distance filtering at all. The first fix is always delivered.
