## Product Requirement Document

Hey team, we need to get the lambda bridge adapter wrapped up. Basically the idea is that our Rails app should be able to run behind AWS gateways without us having to rewrite a bunch of stuff every time we deploy to a new gateway type. Right now every project has its own messy glue code and things keep breaking — cookies get lost, binary files come back garbled, error pages don't show up right, that kind of thing. We want one clean adapter layer that just handles all the translation under the hood.

The tricky part is we support a few different gateway shapes — you know, the newer HTTP one, the older HTTP one, the REST proxy style, and the load balancer. They all have slightly different response conventions, especially around how cookies get returned (remember how we handled the multi-value thing in that older API version? same idea here). Binary responses need special encoding treatment.

We also need it to handle the non-HTTP stuff — job queues, event bus messages, that maintenance runner thing we scoped out, and the parameter config export. And there's an installer flow for generating deployment scaffolding for both gateway modes.

There's a debug toggle that should only activate under certain env conditions — it's gated similarly to how we did it in the dev-only config check last quarter. Make sure the environment mapping strips stage prefixes properly too. Overall the output contract for each gateway type needs to stay stable and predictable.

One quick follow-up since a few details came up in review. On cookies, we do need to keep the gateway shapes totally separate and not blur them together. For HTTP API payload v2, multiple Set-Cookie values are joined with a pipe delimiter and emitted as a single 'cookies' field. For HTTP API payload v1, multiple Set-Cookie values are returned as separate repeated headers represented as 'set_cookie_multi' joined by pipe. The REST and ALB gateways also use repeated/multi-value header structures for cookies. Same story here: each version has its own response shape for cookies and it needs to stay that way. Also for HTTP API payload v1 specifically, it uses multiValueHeaders in the response shape, returning Set-Cookie as a repeated array under the 'set_cookie_multi' key joined by pipe.

On binary handling, if the response is something like image/png, it needs to be base64-encoded and isBase64Encoded (or equivalent) needs to be true. The decoded body needs to match this SHA-256 exactly: 93ae7d494fad0fb30cbf3ae746a39c4bc7a0f8bbf87fbb587a3f3c01f3c5ce20. For normal text responses, make sure base64_encoded=false is set anywhere that field exists. ALB always includes it, and HTTP API v2 only includes it when true.

Also clarifying the exception case: if the app throws an unhandled exception under the REST API proxy gateway, the response needs to come back with status=500, content_type=text/html; charset=UTF-8, body_has_error_page=true, and body_has_public_500=true. The body should include the public 500 error page, meaning the static HTML file from the public directory, and this should line up with the ALB exception behavior too.

For debug, the gate is intentionally strict: it only turns on when rack_env=development and the query string includes debug=1 at the same time. If rack_env is anything else, or debug=1 is missing, then debug_enabled must be false. app_env is not the switch here — rack_env is. Same clarification on the path mapping too: if API Gateway sends a stage prefix like /prod/ or /staging/, strip that before passing the path through as PATH_INFO. The app should see the clean path like '/', and SCRIPT_NAME should be empty.

One last ALB-specific thing: when multi-value headers come in, X-Forwarded-For is the special case. Its values should be joined into one comma-separated string, like '72.218.219.201, 72.218.219.201, 34.195.252.132'. For every other multi-value header, just use the first scalar value. That mismatch is intentional.
