## Product Requirement Document

We need a request quota enforcement engine for our API gateway that protects backend services from traffic overload. The system should determine, for any incoming HTTP request, whether a usage cap applies to it based on which endpoint it targets. Operators will configure a set of named routes (each identified by a path prefix) and attach quota policies to specific routes. A policy defines how many requests are allowed within a time window and which dimensions (caller IP, URL path, authenticated user identity, etc.) are used to partition callers into independent buckets so they don't share each other's allowance.

When a request arrives, the engine should first resolve which declared route it belongs to by checking path prefixes in order. If the matched route has an associated policy and the feature is globally switched on, the request is subject to quota accounting. Each accounted request should decrement the caller's remaining allowance and report back the current limit and remaining count via standard response metadata headers (look for how the filter sets response headers in the core filter implementation). Once a caller's allowance hits zero, subsequent requests must be rejected with the standard HTTP "too many requests" status code and the response should flag the request as having exceeded the quota.

The system should be cleanly designed so the accounting store can be swapped out (in-memory for development, external stores for production). A test adapter that reads a JSON command describing the configuration, the incoming request, and a replay count from standard input, then drives the engine and writes observable signals to standard output, is required for automated testing. The quota bucket key construction logic — which combines a configurable application prefix with route and caller identity segments — should be inspectable through the test harness.

One extra detail on how the bucket key is put together, since the team asked: the rate-limit key is built using a StringJoiner with ':' as delimiter. It always starts with the keyPrefix (default 'rate-limit-application'), followed by the route id. Then, if the policy's type list is non-empty, the following segments are appended in fixed order: if types contains URL, append the route's path; if types contains ORIGIN, append the remote address (or X-Forwarded-For header when behindProxy=true and the header is present); if types contains USER, append the authenticated user's principal name, or the literal string 'anonymous' if no principal is set. Example key: 'rate-limit-application:serviceA:/serviceA:10.0.0.1:anonymous'.

Also worth being precise about when we actually reject. Rejection happens when the remaining allowance after decrement is strictly less than 0, so remaining < 0, not <= 0. The AbstractRateLimiter.updateRate method sets remaining to Math.max(-1, remaining - 1), so a rate that had remaining=0 becomes remaining=-1 after the next consume call. After that, the filter clamps the reported remaining to Math.max(rate.getRemaining(), 0) = 0, sets the response status to 429 (TOO_MANY_REQUESTS), puts the key 'rateLimitExceeded' = 'true' into the RequestContext, and throws a ZuulRuntimeException wrapping a ZuulException. For the test adapter behavior, the output line for a rejected run is exactly 'run=<n> limit=<L> remaining=0 exceeded=true status=429'.

On the “should this request even be limited” part, shouldFilter() returns true only when BOTH conditions hold: properties.isEnabled() is true AND policy().isPresent() is true. The policy() method resolves the current request's route via the RouteLocator; if the route is non-null, it calls properties.getPolicy(route.getId()) which does policies.getOrDefault(routeId, defaultPolicy) wrapped in Optional.ofNullable. If no route matches the URI, policy() returns Optional.ofNullable(properties.getDefaultPolicy()). So the practical outcomes are: if enabled=false → should_limit=false; if route matched but no policy and no defaultPolicy → should_limit=false; if no route matched and no defaultPolicy → should_limit=false.

And for the response metadata headers, the RateLimitFilter in golden/spring-cloud-zuul-ratelimit-core/src/main/java/com/marcosbarbero/cloud/autoconfigure/zuul/ratelimit/filters/RateLimitFilter.java sets three static final header constants: LIMIT_HEADER = 'X-RateLimit-Limit', REMAINING_HEADER = 'X-RateLimit-Remaining', RESET_HEADER = 'X-RateLimit-Reset'. These are written to the HttpServletResponse in the run() method.
