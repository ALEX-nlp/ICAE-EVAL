## Product Requirement Document

Hey team, we've been getting complaints from a few enterprise customers who are building internal tools on top of our URL parsing utility. The main issue is that their tooling keeps breaking when they hit edge cases — stuff like AWS bucket URLs, blogspot-style subdomains, and some international domains with non-ASCII characters. They're also asking if they can plug in their own list of 'known domains' instead of relying on whatever we ship by default, and one customer specifically mentioned they want to toggle whether those 'platform-managed' domains (you know, the ones in the second section of the public suffix list — like we handled in that reporting pipeline refactor a while back) count as real suffixes or not. There's also a request to run batches of URLs in one shot and get back a compact one-line-per-URL summary instead of the full breakdown. Oh, and one team hit a wall when they disabled all their local caching and network fetching in a locked-down environment — the tool just silently gave wrong results instead of telling them something was misconfigured. We need it to fail loudly in that case. Output format should stay consistent across all these modes. Please make sure the implementation is cleanly structured — last time someone left everything in one file and it was a nightmare to maintain.

One small follow-up from the questions that came in: the `include_private_suffixes` flag is the switch for whether we treat the PRIVATE side of the Mozilla Public Suffix List as real suffixes or not. Default is false. So with the default behavior, things like `blogspot.com`, `amazonaws.com`, and `uk.com` behave like ordinary labels unless the flag is turned on. When it is turned on, those entries can consume more of the host as the suffix. And if that match eats everything so there’s no label left underneath it, then `domain` and `fqdn` should both be empty and `suffix` should carry the full matched value. Also worth calling out that input can still be either just a plain URL string, which uses the default policy, or a JSON object with `url` and `include_private_suffixes`.

Also, on custom suffix data, `suffix_list` and `extra_suffixes` are not the same thing. If a request includes `suffix_list`, that fully replaces the bundled list for that request, so only whatever is in `suffix_list` counts. Wildcard entries like `*.co.jp` are valid there too, and they consume one extra label beyond the wildcard pattern. If the host ends in something whose final label isn’t in that custom list, then it should resolve to an empty suffix, with no fqdn and the last label becoming domain. `extra_suffixes` is additive instead: it appends entries to whatever base list is active, either the bundled data or a `suffix_list`, and it does NOT replace that base. Hosts matching one of those extra entries should resolve normally with that entry as the suffix, and if they match neither the base list nor the extra entries then they still resolve to an empty suffix. It’s also valid for `suffix_list` and `extra_suffixes` to both be present in the same input object.

For batch handling, if the input object uses `urls` instead of a single `url`, that means batch mode. In that mode the output is just one line per input URL, with subdomain, domain, and suffix joined by single spaces in that order, terminated by a newline. Empty pieces stay empty, so you’ll see spacing reflect that, like ` example com\n` when there’s no subdomain. We should not emit the usual six-line key=value output in batch mode.

And on the locked-down environment problem, the check needs to happen up front. If the input includes `data_sources`, it can have `cache`, `remote_list`, and `bundled_snapshot` booleans. If all three are false, we should reject immediately and return exactly `error=no_data_sources\n`. No silent fallback, no best-effort parsing first. If at least one source is enabled, like `bundled_snapshot: true`, then parsing continues normally using what’s available.
