## Product Requirement Document

hey team, we need to wrap up the SQL builder library we've been talking about. the core idea is that devs shouldn't have to hand-concatenate SQL strings anymore — they give us structured input describing what they want and we spit out the SQL + bound args. we need to handle the usual CRUD stuff plus some of the trickier bits like conditional expressions and the runner/cache integration we did for that db abstraction layer a while back.

couple things i want to make sure we nail: the placeholder handling needs to support both the question mark style AND the numbered dollar-sign style (like postgres uses), and switching between them should be seamless. also the argument ordering has to be deterministic — we've had bugs before where args get shuffled and it breaks prod queries silently.

for the runner piece, it should work similarly to how we wired up the recording adapter in the auth service — basically you inject whatever executor you want and the statement just calls through to it. same idea for the cache layer, don't re-prepare the same SQL twice.

one thing i'm fuzzy on is exactly how zero values for limit/offset should behave — make sure we handle that explicitly. error messages need to be normalized machine-readable codes, not free-form strings. reach out if you need the exact formats, i have some notes from the earlier spike somewhere.

quick follow-up after the team questions came in: for limit/offset, zero is definitely a real value and we need to treat it that way everywhere. if someone sets limit to 0, that must render as `LIMIT 0`, and if they set offset to 0, that must render as `OFFSET 0`. same deal across SELECT, UPDATE, and DELETE. basically don't let zero get treated like "not provided" and quietly drop the clause.

on the validation side, let's lock this down so every validation failure comes back in the normalized format `error=<code>\n` and nothing free-form slips through. the codes we need to support are `missing_select_columns`, `missing_insert_target`, `missing_insert_values`, `missing_update_target`, `missing_update_assignments`, `missing_delete_target`, `missing_case_when`, and `empty_list_predicate`. related to that, if a predicate uses `equals` or `not_equals` with an empty list value, that's invalid and we should return `error=empty_list_predicate\n` instead of trying to build SQL from it.

also wanted to be super explicit on placeholders. we support two styles: `question` and `numbered`. in `question`, the `?` markers stay exactly as they are. in `numbered`, each `?` gets rewritten in encounter order to `$1`, `$2`, etc. one important gotcha: `??` is an escape for a literal single `?`, so it must not get counted or rewritten as a bind marker. and we also need the `placeholder_list` command, which returns a comma-separated list of `?` for whatever count was requested.

for the runner behavior, keep following the injected execution interface idea. when a `runner` operation comes in with scenario `select_runner`, the rendered SQL should be passed through all three methods, Exec, Query, and QueryRow, and the output needs to be exactly `exec_sql=<sql>\nquery_sql=<sql>\nqueryrow_sql=<sql>\nscan_error=none\n`. for non-select statements like insert, only `exec_sql=<sql>\n` should come back. if no runner is configured, return a normalized error. and for the runner/cache pieces, this is still the same Feature 8 (runner_execution) and Feature 10 (prepared_statement_cache) behavior from start.md, including the recording runner adapter pattern and the SQL-string-keyed cache deduplication logic.
