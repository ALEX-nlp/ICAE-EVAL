## Product Requirement Document

Hey team, we need to build out the i18n catalog loading runtime we've been talking about. Basically the idea is that developers shouldn't have to hand-roll all that locale file picking, parsing, merging, and key traversal logic every time. We want a single layer that handles all of that.

The system needs to support loading translations from local files in at least two formats (you know, the usual suspects), and also from a remote source abstraction — same output shape either way. When a locale file isn't available, it should gracefully fall back to a default instead of blowing up. We also need that deep-merge behavior we discussed, like what we did with the fallback handling in the login module last time, where the active locale values win but missing keys still come through from the fallback.

For grouped catalog loading, there should be some kind of namespace support, and we need to validate the input before even attempting the load — fail fast, clean error, no stack traces leaking out.

Key resolution needs to handle nested paths using a configurable separator, and plural variants should be selectable by count. There's also a widget rendering scenario where we need to verify the end-to-end visible text output including how object-valued keys behave.

Please make sure the code is well-structured — not everything dumped into one file — and the test harness should be runnable with a single bash command. Cases live under the rcb_tests folder.

One quick follow-up from the questions that came in: for local catalogs we only need to support exactly two formats, JSON (`.json`) and YAML (`.yaml`), and both need to end up in the same output map shape. Which one to use comes from the `format` field in the command input. For locale selection, the default is `en`. If `includeCountry` is true, that becomes `en_US`. If a `locale` string is provided, normalize it by stripping any leading underscore, so `_en` → `en`. That locale identifier is then used directly as the file name when loading catalogs.

On paths, for flat catalogs the asset path is `assets/flutter_i18n/<locale>.<ext>`, like `assets/flutter_i18n/en.json`. For namespaced catalogs it should be `assets/flutter_i18n/<locale>/<namespace>.<ext>`, like `assets/flutter_i18n/en/common.json`. The fallback merge behavior uses a separate asset source named `fallback_merge_asset`, and that resolves to its own subdirectory.

Also to make the fallback behavior extra concrete: if the active locale catalog cannot be loaded, we try the configured fallback catalog. If both fail, return an empty map as `map={}`. If both load, do a deep merge where active leaf values override fallback leaves at the same path, nested maps are merged recursively, and fallback-only leaves stay in the result. That’s the same deep-merge fallback logic called out in Feature 2 (feature2_fallback_deep_merge.json): active locale values override matching fallback keys, nested maps are merged recursively, and fallback-only leaves are preserved in the merged output. In `fallback_merge_asset`, the active `fr` file has `title=flutter_18n_fr` and `block.label2=Bonjour`, while the fallback `en` file has `sub_title=Hello World`, `block.label1=This is my app`, and `block.label2=fallback_label2`. So after merge, `title` comes from fr, `sub_title` comes from en because fr is missing it, `block.label1` comes from en because fr is missing it, and `block.label2` comes from fr because it overrides en.

And just to lock down the test harness details, tests run with `bash rcb_tests/test.sh`. The script accepts `--cases-dir <subdir>` and the default is `test_cases`. For each case in each `*.json` file it writes one output file to `rcb_tests/stdout/<cases-dir>/<filename_stem>@<case_index_zero_padded_3>.txt`. For example, the first case of `feature1_flat_catalog_loading.json` run with `--cases-dir public_test_cases` writes to `rcb_tests/stdout/public_test_cases/feature1_flat_catalog_loading@000.txt`. This also lines up with Feature 1 coverage: JSON (`.json`) and YAML (`.yaml`) are the two decoder formats required by Feature 1 and tested in feature1_flat_catalog_loading.json cases 0 and 1.

A few exact details to be precise about: For flat (non-namespaced) catalogs the asset path is `assets/flutter_i18n/<locale>.<ext>` (e.g. `assets/flutter_i18n/en.json`). For namespaced catalogs the path is `assets/flutter_i18n/<locale>/<namespace>.<ext>` (e.g. `assets/flutter_i18n/en/common.json`). The fallback merge feature uses a separate asset source named `fallback_merge_asset`, which resolves to its own subdirectory.
