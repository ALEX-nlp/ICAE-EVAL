## Product Requirement Document

Hey team, we need a DOM assertion helper tool — basically a small utility that takes a JSON command over stdin and spits out a JSON result to stdout. The idea is that our QA folks are drowning in repetitive DOM inspection boilerplate and we want to give them a clean contract-based interface instead. Think of it like that adapter pattern we used on the login module last time — same separation idea, where the I/O layer just translates commands into calls to the real logic underneath.

The tool needs to handle things like: checking if elements exist (or don't), verifying text content (exact or fuzzy), inspecting attributes and their values, checking CSS classes, looking at form control values, and also disabled/focus states. Oh and there's some whitespace normalization thing we need — users keep complaining that copy-pasted text comparisons break because of invisible formatting differences.

For errors — if someone passes a weird target type that we don't understand, we need a normalized error response rather than a crash. The output format needs to be consistent and pretty-printed.

Also the whole thing should be structured properly, not just one giant file — our new engineer complained last time about god files. Reference the existing test contracts in the rcb_tests folder if you need to understand the expected shapes. Let me know if anything is unclear, I'm not super technical on the DOM side.

Quick follow-up after the team questions: for the whitespace piece, Tabs (\t), carriage returns (\r), newlines (\n), and repeated spaces must all be treated as ordinary whitespace. They should be collapsed into a single space, and the result must be trimmed at both ends. Non-whitespace characters must remain in their original order. See feature1_normalize_whitespace.json for the exact contract.

A little more color on how we describe matched nodes too, since that came up. When describing a collection (NodeList), join descriptions of up to the first five elements with ", ". If the collection is empty, output the string "empty NodeList". If there are more than five elements, append "... (+N more)" where N is the count of remaining elements beyond the first five. Each element in the list is described by its lowercase tag name, followed by #id if present, then .class names, then any non-id/non-class attributes. See feature2_node_description.json.

On presence checks, the default behavior is pretty simple: without a count argument, the presence assertion is satisfied when at least one element matches the selector. When arguments.count is provided, the assertion is satisfied only when the number of matching elements equals that exact count. The actual/expected/message fields must reflect the selector string as provided. See feature3_presence.json.

For focus state, the focused_state operation accepts a top-level 'focus' field in the JSON input, which is a CSS selector string identifying the currently focused element. The actual field in the assertion result should contain the described node (compact selector) of whatever element is currently focused. The expected field contains the target selector string. See feature13_focus_state.json.

Also confirming the bad target handling: when the target's 'kind' is not a recognized type (e.g., not 'selector' or 'all'), the output must be a JSON error object with three fields: error: "unexpected_target", target_type containing the kind string, and target_value containing the string representation of the value. No assertions array should be included. See feature14_unexpected_target_errors.json.

And one formatting thing we should be strict about because it affects every contract check: all JSON output must be pretty-printed with 2-space indentation and must end with a trailing newline character. This applies to all operations including errors. The exact format is JSON.stringify(obj, null, 2) + newline, or equivalent.

Last bit is structure. This should not turn into one giant file. The project must NOT be a single monolithic file. It requires a multi-file directory structure (e.g., src/, tests/) with clear separation: I/O adapter/routing layer, core domain logic, and output formatting must be in distinct modules. The core business logic must be fully decoupled from stdin/stdout and JSON parsing per the architecture constraints in start.md. Same idea as the Dependency Inversion Principle and Separation of Concerns described in start.md Architecture section: the execution adapter translates JSON stdin commands into idiomatic method calls to the core domain, which has no knowledge of I/O. If there’s any doubt on exact outputs, the JSON files under rcb_tests/public_test_cases/ (feature1 through feature14) define the black-box stdin/stdout contract for every operation including exact field names, message phrasing, and error shapes.
