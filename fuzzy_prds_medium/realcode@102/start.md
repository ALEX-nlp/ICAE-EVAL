## Product Requirement Document

We need a transparent relationship-batching layer for a relational data model that groups concurrent requests and avoids repetitive round-trips to the database. The product serves backend developers who frequently load collections of linked records for multiple owner entities at once — for example, loading team memberships for a set of projects in one shot instead of one query per project. The layer must intercept the standard collection-getter calls issued within the same processing tick, fold the owner identifiers into a single batched query, and route each result set back to the correct caller. It must support an optional shared loading context that caches results so that a second wave of identical requests adds zero additional database round-trips. Beyond simple retrieval the layer must handle counting (returning how many linked records each owner has), a flag to treat an empty result as an error rather than an empty list, per-request row caps, and per-request attribute filters. Two requests can share a round-trip only if they are semantically identical in every dimension; any difference in row cap, filter, output mode, or relationship name must force separate queries. The system must also support baked-in relationship scopes — a fixed filter on the relationship itself (applied to either the linked records or the join rows) that narrows results for every load of that relationship. The batch-splitting logic lives in the internal coalescing module, and the observable proof of correct batching is captured by the query-counting instrumentation in the test adapter.

One extra detail on when things do and don’t collapse together: requests need to match on the relationship itself, so 'members' vs 'awesomeMembers' vs 'secretMembers' are separate, and they also need to match on options.limit, options.where, and options.raw. Same idea for anything else that changes the cache key from getCacheKey, since that key is built by stringifying association, attributes, groupedLimit, limit, offset, order, where, through, raw, searchPath, paranoid. So if any of those differ, they should route into different batched queries instead of sharing one.

On the shared loading context piece, when the caller passes context: true the adapter calls createContext(connection) and gets back an object with a 'loaders' map. That context gets attached to the getter options under EXPECTED_OPTIONS_KEY, which is the string 'dataloader_sequelize_context'. In that mode the DataLoader instances use cache: true, which is why a second wave of identical loads can come back with zero additional SELECT queries. If there isn’t a context, we still reuse DataLoader instances through the global LRU cache, meaning GLOBAL_CACHE, max 500 entries, but those loaders still run with cache: false.

Also worth calling out how the empty-result behavior works. If options.rejectOnEmpty is truthy and the result array is empty according to lodash isEmpty, then the rejectOnEmpty function throws a Sequelize.EmptyResultError when rejectOnEmpty is exactly true, or calls the value as a constructor if it is a function, or throws it directly if it is an object. The adapter then catches anything whose name is 'SequelizeEmptyResultError' or matches /EmptyResult/ and turns that into the string 'error=empty_result'. Even with that flag set, those requests still share the same batched query as other requests for the same owner/relationship, because rejectOnEmpty is stripped from findOptions before model.findAll is called.

For limit handling, if options.limit is set and the batch has more than one key, we switch over to groupedLimit instead of doing a plain limit across the whole batch. In the BelongsToMany case that becomes options.groupedLimit = { through: options.through, on: association, limit: findOptions.limit, values: uniq(keys) }. For non-BTM hasMany it becomes { limit: findOptions.limit, on: attributeField, values: uniq(keys) }, and then the limit key is deleted. If there’s only one key in the batch, we just use the normal WHERE + LIMIT path. And since different limit values produce different cache keys, requests with different limits won’t share a DataLoader anyway, so they always split into separate batched queries.

If anyone is looking for the exact implementation surface, this all lives in golden/src/index.js — the core library file that contains the DataLoader-based batching logic, getCacheKey, shimBelongsToMany, and all other shim functions.
