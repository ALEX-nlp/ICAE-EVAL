## Product Requirement Document

hey team, we need a small browser-side helper for our OAuth flows. basically we keep running into issues where different apps are assembling the redirect URLs differently and then can't parse them back properly on return. also the PKCE values sometimes break because of encoding inconsistencies. we had something similar in that config-override pattern we used for the login module — same idea where a platform-specific section should win over the base config, so just follow that same precedence logic.

the helper should handle: building the URL you send to the provider, reading back what comes in on the redirect (could be in the hash or the query string, need to handle both), config merging where the platform block takes priority, and the base64 stuff for PKCE. keep it dependency-free and make sure it's easy to test in isolation.

one thing that keeps biting us is that when the platform override block has a nested object like extra params, teams assume it merges with the base — it doesn't, and that causes subtle bugs. also the URL encoding for spaces in scope strings has been inconsistent across implementations. the test harness should write each case output to its own file (one case per file) so we can diff them cleanly, namespaced so different test runs don't stomp on each other.

quick follow-up since a couple folks asked how literal we want this to be. for the option resolution part, Feature 1 (resolve_option) should treat the nested 'web' block inside the options object as the platform-specific override section, and the winner is decided purely by key presence. if the key exists anywhere in that override block, its value is used, even if that value is an empty string, false, true, or null. truthiness does not matter at all here. if the key is absent from the override section, the base-level value is returned unchanged. that same rule is what we mean in feature1_option_resolution.json.

also, when the resolved value is itself a map like additionalParameters, the override section's map fully replaces the base map. there is no deep merge and no shallow merge. keys that exist only in the base-level map are completely dropped from the result, so the override map stands alone as the final value.

for URL building, the query string order needs to be fixed and consistent: client_id, response_type, then if present redirect_uri, then if present scope, then state (always present), then each entry of the additionalParameters map in iteration order, then if present code_challenge followed immediately by code_challenge_method. if an optional piece is absent, it just contributes nothing. the point is that the same options always produce a byte-identical URL. and on encoding, the fully assembled URL is percent-encoded, so a space in any value like 'files.readwrite offline_access' must come through as %20, never as a plus sign.

on the way back in, the parser should look for '#' first. if there is a fragment marker, everything after it is the parameter section, full stop. only if there is no '#' should it fall back to the query string and take everything after the first '?'. so if both are somehow present, the fragment wins every time.
