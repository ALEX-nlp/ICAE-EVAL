## Product Requirement Document

Hey team, we need to build out the core toolkit piece for our ORM encryption layer. Basically developers should be able to tag sensitive fields in their schema definitions and have the system just handle the crypto stuff automatically — no manual per-field wiring. The key management part needs to support rotation so we don't get stuck when we need to swap keys, similar to how we handled the multi-credential logic in the auth module last time. There's also some annotation parsing work — fields can be marked up in doc comments and those markers need to be interpreted correctly with various option flags. One tricky bit is the hash companion field feature, which lets you do equality searches on otherwise-encrypted data. Beyond that we need a schema analyser that figures out per-model which fields are sensitive, how models relate to each other, and which field to use when paginating through rows for a migration job. Finally there's a tree-walking primitive that visits every node in a query payload so the middleware knows exactly which values to swap out. The whole thing needs to be cleanly separated into modules — not one giant file. We've had bugs before where nested relation payloads were missed entirely, so the visitor logic needs to follow relation connections properly. Output format for the CLI harness should match what the test runner expects. Let me know if anything is unclear.

One extra pass on the details the team asked about: for `configure_keys`, if no encryption key is supplied, the output needs to be exactly the single line `error=no_encryption_key\n` and nothing else. Also, a valid key string is `k1.aesgcm256.<base64-material>` — three dot-separated segments with `k1` as the version token, `aesgcm256` as the algorithm identifier, and the third segment as the base64-encoded key material. Parsing and serializing need to round-trip cleanly, so if we parse a key and then serialize it again, we should get back the identical original string. On the keychain side, the encryption key is always implicitly included as a decryption key, so `encryptionKeyInKeychain` is always `true`. The keychain should contain the encryption key plus any supplied decryption-only keys, with duplicate key strings collapsed if the same key shows up in both places. Same idea as feature1_key_configuration.json test case 3 where we still end up with `keychainSize=3` even though the encryption key is logically present twice.

On traversal, the walk is depth-first and it visits every node, including the root. The root comes first with `key=null` and `path=[]`. For arrays, children are visited by numeric index in source order, and for objects, by key in source order. The total visit count should be the full sum of all nodes, meaning collections plus all nested leaves. The callback state threading is reduce-like, but only downward to direct children of the current node, so if a parent returns some state, each sibling subtree gets that same parent-contributed state independently rather than inheriting changes from one another.

For `visit_input`, when traversal hits a key in the args object that matches a connection field in the current model descriptor, the active model should switch to the connected model for everything nested inside that relation’s create/update block. That’s the bit that makes sure encrypted fields inside nested relation operations like `create` under a `posts` relation are visited with the correct model context. Non-encrypted fields should not emit visits. Each emitted visit needs to include the dotted path to the value within args, the value itself, the active model name, the field name, and the field encryption configuration.

On annotation behavior, if `@encrypted` shows up with no query string at all, that should resolve to `encrypt=true` and `strictDecryption=false`, which is the default mode. Same fallback applies if we get unrecognized query parameters or an unknown `mode` value — just silently use that same default instead of erroring. Also, `@encryption:hash(<sourceField>)` is parsed in Feature 3 (`parse_hash`) and then wired into the schema analyser in Feature 4 (`analyse_schema`), with the hash config block attached to the encrypted source field’s descriptor entry, including a `normalize` list for pre-hash value normalization, as shown in feature3_parse_hash_annotation.json and feature4_analyse_schema_model.json.

Last thing, this still needs to stay split up like a real production codebase. We have distinct jobs here — key management, encrypted annotation parsing, hash annotation parsing, schema analysis, tree traversal, payload visiting, and the I/O adapter — so this should not become one big file. Please keep it in a multi-file layout, like `src/` with separate modules by responsibility plus a `tests/` directory.
