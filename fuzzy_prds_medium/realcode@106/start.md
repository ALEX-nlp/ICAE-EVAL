## Product Requirement Document

Hey team, we need a string transformation utility built out ÔÇö it's been on the backlog forever and devs keep reinventing this wheel. Basically two big buckets of work: (1) renaming/casing stuff, like taking a class name and turning it into a database table name, or doing the reverse for display, and (2) making words grammatically correct in multiple languages ÔÇö English obviously, but also some of the European languages we discussed in the Q3 planning doc, plus Turkish which came up from the Istanbul team's request last sprint.

For the casing side, we need the usual suspects ÔÇö snake case for storage, the two camel variants for code identifiers, and some kind of title-casing thing. There was a note somewhere about how the title one should behave differently depending on what characters you consider word boundaries, kind of like how we handled the locale-aware splitting in the old login module ÔÇö check that logic for reference.

For the language pluralization piece, the tricky parts are the irregular words (English has a TON of those), and languages like Portuguese where the same ending can go multiple ways. Also compound words with hyphens or underscores should still work.

Output should just be a simple key=value line. The whole thing needs to read from JSON on stdin. Needs to be clean, testable, not a giant blob of spaghetti. Should be easy to drop in a new language later without touching existing stuff.

One extra pass on this since a few details came back in questions: the utility reads a single JSON object from stdin, and every result needs to come out as exactly one line in the format `output=<value>` followed by a newline character. If we get an empty string in `text` for case conversion or `word` for number inflection, thatÔÇÖs still a valid no-op, not an error ÔÇö the output in that case must be `output=` plus the newline, with nothing after the equals sign. Same rule for empty input values in general: no extra whitespace, no JSON, no labels other than `output=`.

On the input shape, for case conversion the required fields are `feature` with value `case_conversion`, `style` with one of `table`, `pascal`, `camel`, `title`, and `text`. The `delimiters` field is optional and only matters for `style=title`. For number inflection the required fields are `feature` with value `number_inflection`, `language`, `direction` with either `plural` or `singular`, and `word`. The supported languages for number inflection are exactly `english`, `french`, `spanish`, `portuguese`, `norwegian_bokmal`, and `turkish`, and those come through as the `language` field value.

A small but important clarification on title casing: if `delimiters` is omitted, the default word-boundary characters are hyphen (`-`) and space (` `). If `delimiters` is explicitly provided as a string, then each character in that string is treated as a boundary character, and that supplied set overrides the default set entirely. So underscore is NOT a default delimiter, but it does become one if itÔÇÖs included in the provided `delimiters` string.

For the language-specific bits, Norwegian Bokm├Ñl pluralization appends -er to most nouns. Some nouns are invariant, meaning singular and plural are identical, and singularization of words ending in -er strips the suffix. A lookup list should handle the invariant nouns and any irregular forms. Turkish needs the vowel harmony behavior: use -ler if the last vowel is a front vowel (`e, i, ├Â, ├╝`), and -lar if the last vowel is a back vowel (`a, ─▒, o, u`). Singularization strips -ler or -lar, and personal pronouns plus other irregulars should go through a lookup table before the vowel-harmony rule is applied.

Also, on structure, we do want this to follow the Open/Closed Principle pretty literally: each language inflector should live as its own module/class implementing a shared inflector interface or abstract base. Adding a new language should just mean creating a new implementation and registering it in a registry/factory, with the dispatcher resolving `language` through that registry. In other words, no existing inflector code should need modification when we add another language later.
