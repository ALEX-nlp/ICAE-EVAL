## Product Requirement Document

Hey team, we need a structured logging serializer library that our downstream pipeline can actually consume without us maintaining a bunch of fragile regex rules. Right now every service is hand-rolling their own JSON and the dashboards keep breaking because field names are inconsistent or timestamps are in different formats. We talked about this in the last infra sync — basically the same approach we used for that ECS alignment work, just applied here as a reusable library.

The library needs to take in log event data (time, severity, message, some optional context stuff) and spit out one clean JSON line per event. Timestamps should always look the same. Severity labels should line up visually in the output. We also need to support attaching key/value context to events — most keys should be namespaced to avoid collisions, but some well-known correlation keys (like trace and transaction identifiers) should be promoted to top-level. Tags should be supported too as an array. Errors need their own structured fields.

Also really important: the thing needs to handle bad inputs gracefully — no raw stack traces or internal error objects leaking into stdout. There should be a clean, consistent error reporting format that downstream tooling can parse. The adapter layer that reads commands and writes output must be separate from the actual serialization logic. Ping me if unclear on what 'consistent error format' means here, I can dig up the notes.

Quick follow-up from the questions that came in: for timestamps, we do need this to be totally fixed and boring every single time. It should always be a UTC ISO-8601 string in exactly YYYY-MM-DDThh:mm:ss.SSSZ, with zero-padding on every piece so the width never changes: 4-digit year, 2-digit month, 2-digit day, 2-digit hour, 2-digit minute, 2-digit second, 3-digit millisecond. The timezone is always the literal character Z, never an offset. So for a concrete example, epoch 0 yields 1970-01-01T00:00:00.000Z.

Also on the level formatting, yes, the padding is intentional and it lives inside the JSON string value. log.level should be right-aligned to a width of five characters by left-padding shorter names with spaces. So WARN becomes ' WARN' and INFO becomes ' INFO' with one leading space each, while DEBUG, ERROR, TRACE get no padding.

On the command side, format_timestamp takes a JSON command object with op='format_timestamp' and an integer field 'epoch_millis'. The response for that operation is just the formatted timestamp string plus a single trailing newline character, not wrapped in a JSON object. If epoch_millis is missing or not an integer, use the normalized error contract with field=epoch_millis.

One other formatting detail we should lock down: the first three fields in every output JSON line must always appear in this exact order and with this exact spacing: @timestamp, then log.level, then message. After those three, when present and in this order, come: tags (if any), service.name, process.thread.name, log.logger, then labels fields, then error fields. The spacing after the first three field separators uses a single space after the colon, while subsequent optional fields do not include that extra space.

For labels, the default behavior is still to namespace user context so we don’t stomp on reserved ECS fields. Every key in the 'labels' input map should come out prefixed with 'labels.' by default, so key 'foo' becomes the field 'labels.foo'. If the labels object is absent or empty, we emit no label fields at all. The field names we’re using for the reserved structured stuff are the ECS ones we already discussed: @timestamp, log.level, service.name, process.thread.name, log.logger, error.code, error.message, error.stack_trace.

There is also the allow-list behavior for promoted label keys. A configurable allow-list of key names causes certain labels to be emitted as bare top-level fields without the 'labels.' prefix. The default allow-list is trace.id, transaction.id, span.id, error.id, service.name. If the caller wants to change that, they can pass a 'top_level_labels' array in the command. Keys in the allow-list are emitted without prefix, and everything else keeps the 'labels.' prefix. Both promoted and namespaced keys can coexist in the same input map and preserve insertion order.
