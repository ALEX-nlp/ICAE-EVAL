## Product Requirement Document

Hey team, we need to build out the core scaffolding toolkit primitives — basically the plumbing layer that the rest of the generator tool sits on top of. Think of it like that path-handling util we did for the deploy pipeline last quarter, but more comprehensive. The main pain points right now are: (1) every part of the tool is hand-rolling its own file relocation logic and they all behave slightly differently when there are parent-directory hops involved, (2) nobody agrees on what counts as a 'local' vs 'remote' module source so the fetch logic keeps breaking for certain repo reference styles, (3) manifest files are being parsed ad-hoc in at least three places and the parameter ordering is inconsistent in the output, and (4) parameter values that need to come from a shell command at runtime are being handled inconsistently — sometimes the newlines get stripped, sometimes they don't. We want a single clean core with a CLI adapter on top that reads JSON from stdin and writes results to stdout. There should also be a test harness that can run case files from a directory and save raw output per case — the naming convention for output files should follow the same pattern we settled on for the integration test runner. The whole thing needs to be properly structured, not a single giant file. YAML parsing will be needed for the manifest stuff. Let me know if you have questions.

One extra pass on the details the team asked about. For the path replace behavior, only the first occurrence of the search substring should be replaced. After that substitution, normalize the result so redundant separators are collapsed and . / .. references are resolved where possible. So if the search string shows up twice, only the leading match gets changed.

Also on prefixing paths, the path is normalized first, then we count the entire leading run of consecutive ../ hops and insert the prefix immediately after that run, not all the way at the front. So ../../dir/file.ext with prefix 'prefix' becomes ../../prefix/dir/file.ext. If there are no leading parent hops, the prefix goes at the front, and an empty prefix leaves the normalized path unchanged.

On source classification, a reference is remote if it names a remote repository, including references with or without an explicit URL scheme. So 'github.com/acme/widgets' is remote even though it lacks 'https://'. A local path like 'templates/base' or anything that looks like a filesystem path is local. Remote sources are mapped to a cache directory so relocated=true; local sources are used as-is so relocated=false.

For manifest parameters, keep the YAML declaration order in the output and do not sort. Each parameter line should look exactly like: 'parameter field=<field> label=<label> options=<comma-joined options>'. If a label is absent, the label field is empty. If options are absent, the options field is empty. Do not invent defaults.

And for the test harness output files, the naming convention is rcb_tests/stdout/<cases-dir>/{filename.stem}@{case_index.zfill(3)}.txt, including the cases-dir namespace and the zero-padded 3-digit index, matching what we called for in start.md.
