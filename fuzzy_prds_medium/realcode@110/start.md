## Product Requirement Document

Hey team, we need to ship the browser entropy toolkit we've been talking about. The idea is to give devs a single layer that handles all the messy browser readings — you know, the ones that come back as strings instead of numbers, or just go missing entirely, or randomly collapse to zero in fullscreen. We had a similar defensive-parsing pattern in the old login module, so just follow that spirit but make it a proper standalone thing this time.

The toolkit needs to handle number coercion (both integer and float flavors), some kind of fingerprint-ready hash (the exact algo was agreed on in the architecture call — make sure the output is always the same fixed width), a way to gather multiple signals at once without one bad reading killing the whole batch, and the screen geometry thing where the available area sometimes drops to zero temporarily and we need to remember the last good value.

Please keep the code organized — don't dump everything in one file, we've had complaints about that before. Separate the concerns cleanly. The execution layer that reads commands and writes results should be totally isolated from the actual logic.

One thing I'm not 100% sure was written down anywhere: what exactly happens when availLeft or availTop are missing vs zero — there's a difference there that caused a bug in QA last sprint. Make sure that's handled correctly. Also confirm the rounding behavior for fractional bases, that one always trips people up.

One follow-up after the team questions: for the numeric coercion bits, keep the same defensive parsing vibe we talked about before. Accept both number and string inputs, read leading sign and digits, stop at the first non-numeric character, and return NaN when the input doesn't begin with a parseable number — as specified in Feature 1 (to_int and to_float) in start.md. Also, for fractional parts, we want truncation toward zero, not floor and not normal rounding. So 12.6 becomes 12 and -1.5 becomes -1 (not -2). This is meant to match parseInt/truncation semantics, not Math.floor. And for the base-rounding question, if the base is fractional and below 1, please avoid floating-point drift. 0.1234321 rounded to base 0.0001 must yield exactly 0.1234, not 0.12340000000000001. Use a precision-aware approach such as scaling to integers before rounding. A negative base uses its absolute magnitude.

On the hash side, use the 128-bit x64 variant of MurmurHash3, operating over the UTF-16 code units of the input string. The output must be exactly 32 lowercase hexadecimal characters, zero-padded, with no separators. An empty string must produce 32 zeros. The digest for 'Hello, world' must be ebd28b45027ab97477416103e3fff7b8.

And to close the loop on the screen-frame behavior, the missing-vs-zero distinction is real. When availLeft or availTop are null/unavailable, they are treated as 0 for the arithmetic (right = width - availWidth - availLeft, bottom = height - availHeight - availTop), but they are reported as the null sentinel in their own positions (top and left slots of the frame list). So a null availTop produces frame=[null, right, bottom, null] not frame=[0, right, bottom, 0]. Also, this is the same watch behavior captured in Feature 4.2 (screen_frame_watch) in start.md — the ScreenFrameWatcher that remembers the last non-zero frame and returns it when the live frame collapses to all-zero, as detailed in feature4_2_screen_frame_watch.json. More specifically, the watch facility maintains a remembered non-zero frame. When get is called and the live frame is all-zero but a non-zero frame was previously observed, return the remembered non-zero frame. When the live frame is non-zero, return it and update the remembered value. While the live frame is all-zero and no non-zero frame has ever been seen, return all zeros. The remembered value is updated only when the live frame is non-zero.
