## Product Requirement Document

Hey team, we need to wrap up the WASM runtime loader library. Basically the idea is that devs hand us a raw binary blob and we handle all the ugly plumbing — wiring up their callbacks, letting them read/write module state, and catching every possible way things can go wrong so it surfaces as something meaningful instead of a crash. We had a similar pattern in that import-validation flow we built for the scripting sandbox project — same idea where you pre-check everything before actually running. The big complaints we keep hearing are: (1) when something goes wrong with a binary or a missing hook, people get totally opaque native faults with no idea what happened; (2) numeric values going across the boundary come back mangled or lose precision; (3) stdout from the module just disappears. We also need the memory side to be safe — oversized allocations should fail clean. There's also the question of what happens when a host callback itself throws — that needs to bubble up properly rather than silently swallowing. The execution adapter piece that reads test scenarios and drives all this should stay completely separate from the core library logic. Make sure the overall code is organized by responsibility — don't dump everything in one file. Refer to the existing test scenarios to understand the expected wire format for outputs.

One more pass on the output behavior since a few folks asked for the exact shape. When compilation fails, like a bad magic number case, the adapter must emit exactly two lines: `error=compile_failed` first, then `detail=<runtime validation message>` second. That detail needs to carry the runtime’s raw message as-is, for example `Validation error: Bad magic number (at offset 0)` followed by a null byte `\0`. On function calls, wrong arity and wrong scalar type need to stay separate: wrong number of arguments emits `error=argument_count` and nothing else, and wrong scalar type emits `error=argument_type`, also as a single line.

Also, the export-name check is just a lookup result, not a failure path. If the name exists, emit `lookup[<name>]=present`; if it does not, emit `lookup[<name>]=missing`. No `error=` line for that. Same idea on globals: `read_global` for an exported global emits `global[<name>]=<value>`, `read_import_global` for an imported global emits `global[<ns>::<name>]=<value>`, and a successful `write_global` should just re-report the updated value in the same read format, so `global[<name>]=<new_value>`.

On the formatting side, float output needs to match the actual width crossing the boundary. 32-bit float results must render at single-precision fidelity, so IEEE-754 float32 rounding is what shows up. The concrete example here is 1234.5678 + 8765.4321 as f32 rendering as `9999.9990234375`, not `9999.9999`. For f64, keep full double precision, so those same inputs as f64 render as `9999.9999`. And if the scenario asks for `report_build: true` and instantiation succeeds, the adapter should include `instantiated=true` as a line in the output, but only when `report_build` explicitly asks for it.

On structure, we do want to be pretty strict about separation of responsibility. Binary compilation, import collection/validation, instantiation, typed value marshaling, memory management, globals, and the WASI/stdout layer should live in distinct units, and the execution adapter that does JSON stdin/stdout needs to stay completely separate from the core library. The core side should expose an idiomatic object API around module, instance builder, instance, functions, globals, and memory, with no coupling to JSON or stdio. And for the import pre-check behavior we talked about before, keep the same validation pass before instantiation for all required imports being present, kind mismatches, unknown import names, and duplicate fills, with each one mapping to its own distinct error category.
