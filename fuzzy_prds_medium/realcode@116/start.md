## Product Requirement Document

We need a small, reusable helper library that supports a dependency-version reporting tool used by our engineering teams. The library must handle several common low-level tasks that come up repeatedly when scanning projects for out-of-date third-party packages. First, it should be able to break apart a version identifier string into its individual numeric and label components in a consistent order, so downstream logic can compare versions piece by piece. Second, it must classify how significant a proposed upgrade is — ranging from no change at all, up through incremental fixes, feature additions, and full breaking releases, with a separate category when version information is missing. Third, it needs a safe map-lookup helper that returns a sensible zero-equivalent fallback instead of crashing when a key is not found. Fourth, we want a dedicated error type for invalid user input that is recognizable as a standard error object and can optionally preserve the original underlying cause. Fifth, the library should enumerate the exact allowed values for each of the tool's command-line option categories, in their declared order, so input validation and help text stay in sync with actual behavior. Sixth, it must provide a fixed color assignment for each semantic role in the report output — project identifiers, framework labels, package names, and each upgrade severity level — so the visual rendering is consistent everywhere. All behavior is exercised through a simple command-dispatch protocol over standard input and output.

A couple extra specifics from the team questions: for op='version_parts', NugetVersionExtensions.GetParts() in golden/src/DotNetOutdated/NugetVersionExtensions.cs yields Major, Minor, Patch, Revision, then each ReleaseLabel in order. That means we always output all four numeric parts before any pre-release labels. If a version only has three numeric positions like '1.2.3', the fourth spot still comes out as 0, so the exact output is '1\n2\n3\n0'. And if there is a '-' section, the pieces after it are split on dots and emitted one per line in order. So '1.2.3.4-beta.1' produces exactly '1\n2\n3\n4\nbeta\n1', with 'beta' and '1' as separate lines, not a combined 'beta.1'.

Also on op='upgrade_severity', the DependencyUpgradeSeverity enum in golden/src/DotNetOutdated/Models/DependencyUpgradeSeverity.cs has members None, Patch, Minor, Major, Unknown. One important precedence rule here is that if either 'resolved' or 'latest' is null, the result is always 'severity=Unknown', and that absent-version check happens first before any numeric comparison. After that, if the currently-resolved version has any pre-release label at all, meaning a '-' followed by any text, the severity is always 'Major' regardless of what the latest version number is, even if the latest version appears equal or lower. Example: resolved='1.0.2-al', latest='1.0.2' yields 'severity=Major'. Also, 'Patch' is a little broader than just the third number: if the latest version's tertiary number OR fourth-position number is greater than the resolved version's, the output is 'severity=Patch'.

For op='dict_get_or_default', this is DictionaryExtensions.GetValueOrDefault<TKey, TValue>() in golden/src/DotNetOutdated.Core/Extensions/DictionaryExtensions.cs. If the key is missing, we return the integer default and do not throw, with output exactly 'value=0'. If the key exists, it stays 'value=<number>' using the exact stored integer.

And one small enum detail that came up: the upgrade type enumeration has exactly 2 members: 'Auto', 'Prompt'. Output is: 'Auto\nPrompt'.
