## Product Requirement Document

We need a compact, embeddable messaging client for constrained devices that can talk to a standard broker over any byte-stream transport. The library should hide all the wire-format details: connecting with optional credentials and an optional last-will message, publishing data to named topics (with or without the retained flag, with text or binary payloads), subscribing and unsubscribing to topic filters at configurable quality levels, receiving incoming messages through a user-supplied callback, and sending periodic keep-alive signals so the broker knows the device is still alive. Underneath, the library frames each operation as a binary control packet using the version 3.1.1 protocol. Connection state should be reported back as simple integers so callers can branch on success, specific failures, or timeouts without pattern-matching on string messages. Defaults to assume: a default keepalive interval of fifteen seconds, a default packet buffer of one hundred and twenty eight bytes, and a default broker port of eighteen eighty three when none is given. A publish or subscribe that would overflow the buffer must be refused rather than truncated. Only quality levels zero and one are supported for subscriptions; any higher value must be rejected. The first subscription or unsubscribe request after connecting should use packet identifier two (the connect packet itself consumed identifier one). The password field is ignored when no username is given. Hostname-based endpoints and IP-address-based endpoints work identically. The library should function without depending on any external networking framework, treating the transport as an injectable byte-stream interface the same way the internal packet-identifier counter is managed.

One small follow-up from the questions that came in: the connection state values should be treated as plain integers with fixed meanings. 0 = connected (MQTT_CONNECTED), -1 = disconnected (MQTT_DISCONNECTED), -2 = connect failed / transport refused to open (MQTT_CONNECT_FAILED), and -4 = connection timeout (MQTT_CONNECTION_TIMEOUT). If the broker itself rejects the connection, the state should just be the broker’s CONNACK code directly, so a positive state value (1, 2, 3, 4, 5) mirrors that return code as-is.

On the wire for CONNECT, the fixed header is 0x10, and the protocol name/version bytes are 0x00 0x04 'M' 'Q' 'T' 'T' with protocol level 0x04. Keepalive should go out as 0x00 0x0f, which is the same 15 seconds already called out. If there are no credentials and no will, clean session flag 0x02 is always set in connect-flags. For credentials, username sets 0x80, and password (only when username present) sets 0x40. If there’s a will, that sets 0x04 plus QoS<<3 and retain<<5.

Also, to be extra explicit about packet ids, the counter starts at 1 after construction and lives on the client as a simple incrementing value. The CONNECT packet consumes ID 1 internally, so the first SUBSCRIBE or UNSUBSCRIBE request after connecting sends packet identifier 0x00 0x02 (decimal 2), and each subsequent request increments by one. Same general idea as the transport setup: it’s injected at construction time too, as an object implementing read/write/connect/disconnect operations, rather than something the library reaches out and creates on its own.
