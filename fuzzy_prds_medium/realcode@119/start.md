## Product Requirement Document

Hey team, we need to ship a token utility tool that our backend services can use for auth flows. The basic idea is: given some kind of signing key and a blob of user data, produce a tamper-proof string that other services can validate later. We also need the reverse — take one of those strings, check it's legit, and hand back the original data. Should support all the usual key flavors our infra team uses (shared secrets, RSA certs, elliptic curve keys, and that newer Edwards curve stuff). Oh and we talked about this in the last sprint — remember how we handled the 'time window' logic in the session module? Do the same thing here, tokens should be rejectable if they're stale or not yet valid. Keys also need to travel over the wire, so there's that JSON key format everyone uses — export and import should both work. The tool reads a single JSON blob from stdin and writes a result to stdout, similar to our other CLI utilities. Non-technical folks on ops need readable error labels, not stack traces — so normalize all failures into categories. One last thing: payloads can be either a plain string or a structured object, but if someone passes something weird like a raw number we should call that out explicitly. Reach out if the key format specs are unclear, I can dig up the exact field names from the standards doc.

One extra pass on details since a few folks asked: the JWT header should always be just two fields, no extras — 'alg' with the algorithm identifier like HS256, RS256, ES256, EdDSA, and 'typ' with the string 'JWT'. That header is the first segment of the compact serialization, and it needs to be base64url-encoded without padding. On the payload side, when we sign a map payload we should automatically add an 'iat' claim using the current Unix timestamp, unless the command has 'no_iat' set to true. String payloads stay exactly as-is and do not get modified. The 'no_iat' flag is mainly there so tests can keep output deterministic.

Also, for EC signing, we want RFC 6979 deterministic nonce generation so signatures are reproducible. The supported algorithm identifiers there are ES256 (P-256 curve, SHA-256), ES384 (P-384, SHA-384), ES512 (P-521, SHA-512), and ES256K (secp256k1, SHA-256). The signature format for those should be the raw R||S concatenation, not DER, then base64url, with each component zero-padded to the curve's coordinate byte length.

On key input, every key object in a command needs a 'kind' field. Valid values are: 'secret', 'rsa_private', 'rsa_public', 'ec_private', 'ec_public', 'ed_private', 'ed_public'. For 'secret', it carries a 'secret' string field. For all the others, it carries a 'pem' string field containing the PEM-encoded key. No other fields are required, and the kind is what tells us which key abstraction to spin up.

For verification behavior, if the payload has an 'exp' claim and current time is past it, output exactly 'error=token_expired\n' and nothing extra on another line. That check can be turned off with 'check_exp: false' in the verify command. Same idea for 'nbf': if current time is before it, output exactly 'error=token_not_active\n', and that one can be disabled with 'check_nbf: false'. We should also match the claim validation behavior for 'exp', 'nbf', and 'iat', including the check_exp/check_nbf disable flags and the issue_at threshold parameter.

Last bit: for the JSON key format work, we do need both 'to_jwk' and 'from_jwk', and the JSON output should be canonical with sorted keys.
