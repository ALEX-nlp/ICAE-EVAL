## Product Requirement Document

Hey team, we need to build out that declarative REST state engine we've been talking about. Basically the idea is devs should be able to just describe their API endpoints once (like a map of names to URLs) and get back everything they need to manage loading states, data, and errors automatically — no more copy-pasting fetch boilerplate across every screen. It should handle the full lifecycle from 'request started' to 'done or failed' and expose that through a store so the UI can just read flags.

Some important stuff: URL templates need to support both required and optional path segments (similar to the pattern we used in the routing layer of the auth module last quarter — check how that handled the colon-prefixed segments). Also the engine needs to be smart about not re-fetching data that's already been loaded. Array params in query strings have caused us bugs before so that needs proper handling with configurable formatting.

On the error side, we had that incident where raw exception messages were leaking to users — make sure errors are normalized to neutral categories. There's also a need for pre-request hooks and the ability to chain requests in sequence. Custom helper names on endpoints should fail loudly if they clash with built-in controls rather than silently misbehaving. We should also be able to configure a global root URL so teams don't repeat the base path everywhere. Global options like auth headers should merge cleanly with per-endpoint settings.

One more pass with a few specifics the team asked for. The reducer should listen for these exact action strings: `@@redux-api@<endpointName>` for fetch, `@@redux-api@<endpointName>_success` for success, `@@redux-api@<endpointName>_fail` for failure, and `@@redux-api@<endpointName>_delete` for reset or sync-reset. On state shape, the initial state and a plain reset should go back to exactly whatever is in `initialState`, usually something like `{ loading: false, data: <initial data> }`. After a `fetch`, the slice picks up `error: null`, `syncing: false`, and flips `loading: true`. After `success`, it adds `sync: true` and sets `loading: false`, `syncing: false`. After `fail`, it sets `loading: false`, `syncing: false`, and stores `error`. For a soft sync reset, meaning `reset` with `mutation: 'sync'`, keep the current `data`, set `sync: false`, and keep `loading: false`.

For prefetching, an endpoint can declare `prefetchOf: '<targetEndpointName>'`. That means it becomes a dependency of that target, and when the target is called, the prefetch chain has to run first. The order matters here: the target’s fetch action is emitted immediately, then the dependency endpoint’s action runs and has to finish, and only after that does the target’s own transport call go out, followed by its success or fail. Both endpoints should still end up with their own store slices populated.

On query params, leftover array params need to serialize using `arrayFormat` like `'repeat'`, `'brackets'`, or `'indices'`, plus a `delimiter` option. We also need to keep `qsParseOptions` separate from `qsStringifyOptions`. The parse options are for reading any query string already present in the template correctly, especially if it already has bracket-indexed values, and the stringify options control outgoing array encoding independently of the top-level `arrayFormat`. If there’s no stringify override, default to bracket-indexed encoding with URL-encoded brackets like `%5B0%5D`. Also, if the template already has a query string and new params come in with the same key, concatenate the existing values and the new ones.

For pure requests, they should bypass the store completely. That means substitute path vars into the URL, call the transport directly exactly once, and return the response data as a resolved promise. No fetch, success, or fail dispatches, and no store updates. If there’s a global `responseHandler`, still call it with the result — null error plus data on success, or a normalized error on failure.

For chained async flows, the list of endpoint names runs in order, one at a time, and each endpoint has to finish its full lifecycle before the next begins. The chain resolves with the `data` from the final endpoint’s success. Afterward, each endpoint slice should contain only its own fetched data, not the whole state object, just the raw `data` value like in `state test={...}`. Same bottom line on the return value too: it’s the last step’s data.

A couple smaller utility behaviors that came up also need to stay exact. The single-slot holder only allows one value at once: first `set` is accepted with `accepted=true`, any later `set` while occupied is rejected with `accepted=false`, even if the incoming value is `null`. `empty` tells you whether a value is currently held. `pop` returns the held value and clears the slot, and if nothing is there it returns `null`. After `pop`, the next `set` should be accepted again. The subscriber registry is one-shot as well: it stores callable subscribers only, ignores non-callable values silently, and when it resolves it calls each subscriber with `(null, payload)`. On reject it calls each one with `(payload)`. Calls happen in registration order by index, and once notification is done the registry is emptied so the remaining count is 0.

For merge behavior, if either side is an array, the result is always a concatenated array. Array then scalar means append the scalar. Scalar then array means make a new array with the scalar first and then the array items. Plain objects merge recursively by key, scalar-vs-scalar means later wins, and undefined or absent values should just be skipped. And on URL building, please stick to the placeholder behavior from Feature 2 exactly, including `:name`, `(:name)`, substitution, stripping any unfilled placeholders, and turning leftover params into the query string the same way as in feature2_1 through feature2_6.

Last thing, on errors, same intent as before but spelling it out: failures should surface as neutral category strings only, specifically `request_failed` or `rejected`, along with a `reason` field. We should never pass raw runtime exception types or message strings straight through in the action payloads or stored error state.
