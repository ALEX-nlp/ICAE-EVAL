## Product Requirement Document

Hey team, we've been getting complaints from the platform folks that our deployment configs are a total mess — people keep copy-pasting manifests for every environment and then things drift apart. We need some kind of toolkit that lets you write one config file and have it work across staging, prod, etc. without hand-editing everything each time.

The big thing is there should be a way to pull in values from environment variables and external files inline, kind of like how we did the substitution stuff in the old login module, but smarter — it should catch loops and missing vars before things blow up in prod. Also there needs to be some validation so we know upfront if a resource type even exists on the target cluster or if an API version got removed, instead of finding out at apply time.

For config merging, the rule should be pretty intuitive — shared stuff lives in a common block, and per-environment bits override it. Maps should merge deep, but lists just replace wholesale, no concatenation weirdness.

We also need the template rendering to support some filters like base64 encode/decode and hashing. And there should be tag-based filtering so teams can opt in or out of deploying certain pieces.

Important: this should be a proper multi-file project, not one giant script. Keep the concerns separated cleanly. Oh and the cluster version/deprecation checking should be semantic-version aware.

One extra pass on the merge behavior since a few folks asked: lists are never concatenated or element-merged. When both the base and the override have a value for the same key and either value is a list (or any non-mapping type), the override value replaces the base value entirely. Only when both values are mappings (dicts) is the merge recursive. This applies to all nesting levels. Also, the shared inherited section is named 'common'. Every named section is overlaid on top of 'common' during resolution. The name 'common' is reserved — it exists only to be inherited and must never be used as a deployable section name. Requesting it directly returns error=reserved_section.

On loading and validation, section loading should validate in this order and return a distinct error= line for each case: config_not_found if the file is missing, empty_config if the parsed content is empty/null, invalid_yaml on YAML parse failure, empty_section for a blank section name, reserved_section if the section name is 'common', section_not_found if the requested name is not present in the document, and missing_templates if the merged section has neither a 'templates' list nor a 'kubectl' list. For the placeholder work, this is the same substitution behavior from Feature 3, specifically the {{ env='NAME' }} and {{ file='path' }} inline placeholder syntax implemented in the resolver/substitution module under src/. The two failure cases people called out should come back as neutral error= lines from the adapter before any manifest is applied: error=include_loop for cyclic file include detection and error=env_not_set in strict mode.

A couple more specifics on precedence and normalization too. Namespace precedence from highest to lowest is: (1) k8s_namespace in the configuration context, (2) the kubeconfig default namespace if supplied, (3) the K8S_NAMESPACE environment variable. The first truthy value wins. And 'First truthy' means the first value that is not null, false, zero, empty string, empty mapping, or empty list. For kind naming, a CamelCase kind name is normalized by splitting at each uppercase letter boundary and joining with underscores in lowercase. For example: ConfigMap -> config_map, PodDisruptionBudget -> pod_disruption_budget, Namespace -> namespace.

For the version checks, the deprecation table maps apiVersion -> kind -> {since, until}. A pair is NOT deprecated if the apiVersion or kind is absent from the table, or if the server version is below 'since'. It IS deprecated once the server version reaches or exceeds 'since'. If 'until' is set and the server version reaches or exceeds it, the resource is also considered unsupported/deprecated. An empty 'until' is ignored. All version comparisons are semantic-version aware and boundary-inclusive. Also, K8S_CA and K8S_HOST are the legacy/deprecated environment variables for supplying the certificate authority and cluster master URI respectively. The deprecation check returns true when either variable holds a non-empty value, and false when both are absent or empty.

And just to make the project shape explicit: this needs to be a real multi-file directory tree, like src/ for the core domain modules and tests/ for tests. Each distinct responsibility — configuration merging, placeholder resolution, template rendering, precedence-based settings, version/deprecation logic, resource availability checks — should live in its own cohesive module. A single 'god file' is explicitly prohibited. The adapter should stay thin and just translate JSON test commands into calls to the core and handle stdout formatting; the core itself must never read stdin or write stdout directly.
