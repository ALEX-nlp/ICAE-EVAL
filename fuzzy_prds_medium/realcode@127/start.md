## Product Requirement Document

Hey team, we need to get this serializer toolkit wrapped up. The core idea is that API consumers shouldn't have to get back giant payloads when they only need a couple of fields, and we also want to avoid accidentally leaking internal database IDs in URLs or responses. Think of it like what we discussed back when we built the sparse fields thing for the dashboard — same philosophy but more robust and covering nested relationships too.

Right now the pain points we're hearing from the front-end team are: (1) responses are too fat by default and they have to strip data client-side, (2) when they POST related records they're not sure if the identifier they sent was actually valid or just silently dropped, and (3) the URLs we expose contain raw numeric IDs which the security team flagged.

We also need it to handle cases where someone asks for a field that doesn't exist — right now it just silently ignores it and the front-end has no idea their query param was a typo. Same for when someone asks for both a parent object and one of its children at the same time, that should be an error too.

There's also the request context wiring — query params like `only`, `exclude`, `expand` etc. should flow through automatically, but serializer-level attribute defaults should act as fallbacks when no query params are present (check how we handled the override priority in that earlier PR). Status codes and lookup tokens need to survive through the API response wrapper as well.

One quick follow-up after the team questions: for select_fields and exclude_fields, if the `fields` parameter is omitted or comes through as an empty list, we should just emit the complete nested product representation in full, including all nested objects and collections. No pruning occurs. Also, nested field paths need to consistently use double-underscore (`__`) everywhere, so think `manufacturer__name` and `skus__owners__organization`, and that same separator applies across select_fields, exclude_fields, expand, and expand_id_only.

On the request wiring side, when query parameters like `only`, `exclude`, `expand`, or `expand_id_only` are actually present on the request, they win and the serializer-level defaults `attribute_only`, `attribute_exclude`, `attribute_expand`, and `attribute_expand_id_only` should be ignored. Those attribute defaults are just fallback behavior when the corresponding query parameter is absent. Also worth calling out that query params can arrive either as a JSON array or a comma-separated string, and we should treat them the same — both `"expand": "a,a1"` and `"expand": ["a", "a1"]` need to produce identical results because the context builder normalizes both forms into lists.

For links, relation hyperlinks should use the external opaque ID in the URL, not the internal numeric one, so the shape should be like `http://testserver/models/AMTw/`. If the record hasn’t been saved yet and doesn’t have a persisted identity, the link field should return `null`. And on the response wrapper piece, we need to preserve `status_code` alongside `data`, and for external lookup flows the opaque `lookup` token should be at the top level too next to `data` and `status_code`. If the looked-up record doesn’t exist, the response should be status_code 404 with data exactly `{"detail":"Not found."}`.

One other implementation detail the team asked about: each child serializer should keep track of where it sits in the nested tree as a double-underscore-joined path string like `"skus__owners__organization"`, while the root serializer just has an empty string hierarchy. If we render a calculated child, it should inherit its parent’s context including a `context_label`, and it should reflect its own `hierarchy` path as its key name. And finally, if someone asks for both a whole nested object and one of its descendants in the same selection set, like `manufacturer` and `manufacturer__name`, that needs to come back as `{"error":"invalid_field_selection"}`. That’s separate from the unknown field case and should stay distinct the same way we’ve been handling Features 1–4 and the request-context override behavior in feature 11.
