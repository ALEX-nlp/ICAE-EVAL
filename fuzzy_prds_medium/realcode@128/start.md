## Product Requirement Document

Hey team, we need to wrap up the interop bridge tool we've been talking about. The basic idea is that we want managed-side code to be able to talk to browser script functions and vice versa, and we want to ship a single packaged file developers can drop into their projects without hand-writing a bunch of repetitive glue code every time. Think of it like that publish pipeline we built for the config module last quarter — same kind of idea where the tool does the heavy lifting at build time.

A few things that came up in last week's sync: the JSON number handling needs to be consistent on both sides of the wire (someone on the consumer team complained about mismatched values showing up), and all the different ways you can call a script function (sync, async, with or without a return value) need to route through whatever runtime is configured. Also the published output needs to support optional source maps and type declaration files, and there should be a way to clean up old files before writing new ones. If something can't be resolved during type generation, we should surface a clear error rather than silently skipping it.

The script bindings inside the published file need to handle nested namespaces from assembly names, and the two different annotation types (host-callable vs script-supplied) should behave differently. Type mappings for the declarations file need to follow the same conventions we use elsewhere in the project.

One quick follow-up from the questions that came in: for JSON numbers, if json_number_output is set to numbers_as_strings, that same exact setting has to be used both directions, not just on the way out. So the output should show outbound_number_output=WriteAsString and inbound_number_output=WriteAsString, and there isn’t a separate knob for each side. It’s one shared behavior applied symmetrically to outbound and inbound JSON processing, including the runtime configuration piece where the same JsonNumberHandling option is applied to both outbound (serializer) and inbound (deserializer) pipelines, as specified in feature1_1_runtime_configuration.json.

Also, when the runtime says something just isn’t available yet, we still want the result to include the call form, the forwarded identifier, and which channel it tried to use, and then finish with error=runtime_call_not_implemented. The call forms we care about are sync_without_result, sync_with_result, async_with_result, and async_without_result. sync_without_result and sync_with_result both use runtime_call=sync, while async_with_result and async_without_result both use runtime_call=async.

On the declaration side, the type mapping should stay simple and consistent: numeric types (int, float, double, long, etc.) map to number; bool maps to boolean; char and string map to string; DateTime and DateTimeOffset map to Date; void stays void; Task<T> and ValueTask<T> map to Promise<T> with T mapped recursively; Task and ValueTask map to Promise<void>; all other object types map to any.

A couple formatting specifics too. For a single-segment assembly like foo.dll, the declaration should be export declare const foo: { and then each member on its own indented line ending in a comma, then };. For a multi-segment assembly like bar.nya.dll, the outer segment wraps the inner using inline braces, so it starts export declare const bar: { nya: { and then the members are indented, and it closes with }}; on the same line as the last inner content. And if the input source snippet has no marked partial function declarations at all, the output should be completely empty — literally an empty string with no whitespace, no newlines, nothing.

Last piece is publishing behavior. Publishing always writes the main script library file and always reports library_published=true. Source maps only get written when emit_map is explicitly true; otherwise map_published=false. Type declarations only get written when emit_types is explicitly true; otherwise types_published=false. If assemblies is empty, the warnings count should still be warnings=0. This is all part of the same PublishTask / publisher flow that inspects compiled assemblies, generates the browser script library with bindings, optionally emits source maps and type declaration files, and handles base-directory cleaning — as described in Feature 3 of the PRD.
