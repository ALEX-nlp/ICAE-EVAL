## Product Requirement Document

Hey team, we need to build out that session management thing for our async web framework. Basically devs are complaining they have to manually wire up cookie handling every single time they build a new endpoint — reading cookies, validating them, writing them back, all that boilerplate. It's causing bugs where sessions get saved even when nothing changed, or worse, tampered cookies aren't caught. We need something that just works automatically.

The session itself should behave like a normal Python dict so devs don't have to learn anything new. It needs to track whether it was actually touched so we don't write unnecessary cookies. We also need a few different storage options — a basic plaintext one for dev/testing and at least two secured ones for production (think along the lines of what we did for the auth tokens in the login module, same kind of key-based approach).

There's also the middleware piece — it should wrap the handler, load the session before, and only persist after if something actually changed. Edge cases like streaming responses, redirects, and invalid response types all need to be handled gracefully. If the middleware isn't installed, devs should get a clear error rather than silent weirdness.

One thing I keep hearing from security is we need protection against session replay attacks after logout. Make sure that's covered. Misconfigured keys should fail loudly at setup time, not silently at runtime.

One extra bit the team asked about on how this should present itself: the session state is rendered as a single line in this exact format: `state new=<bool> changed=<bool> empty=<bool> len=<int> identity=<json> created=<int> contents=<json>`. Boolean values are lowercase `true`/`false`. The identity is JSON-encoded (strings get quoted, integers are bare). Contents is a compact JSON object with keys in insertion order.

Also, on when we actually write cookies, the processing layer only emits a Set-Cookie header when the session was actually modified by the handler. If the handler never accessed the session at all, or accessed it read-only without marking it changed, no cookie is written. This applies even if the incoming request carried existing session data. Related to that, mutating a value that is nested inside the session container (e.g. indexing into a dict stored as a session value and adding a subkey) does NOT automatically mark the session as modified. Only top-level assignment, deletion, or pop operations flip the changed flag. Developers must call an explicit 'mark modified' signal to force persistence after nested mutations.

On the secured backends, both authenticated storage backends require a key of a specific valid length at construction time, and passing a bad one should raise `error=invalid_key` immediately instead of waiting until request handling. The exact required length depends on the backend: the HMAC-based one uses a standard symmetric key size; the sealed box one uses a Curve25519 keypair. These are the two authenticated storage backends: 'encrypted' (HMAC/symmetric, feature 5) and 'sealed' (asymmetric/NaCl sealed box, feature 6), both in the storage backends module, each requiring a cryptographic key of specific valid length passed at construction time.

And for the logout/replay concern, after invalidation, the authenticated backends emit a cleared cookie (empty value, not http-only, max_age=0, expires set to the Unix epoch: `Thu, 01 Jan 1970 00:00:00 GMT`). A subsequent login produces a new cookie value that differs from any previously captured cookie, so replaying a captured pre-logout cookie cannot resurrect the old session. The test sequence is: login → capture cookie → logout → login again → verify the new cookie differs and the logout response cleared the old one. For the session fixation test scenarios (feature5_6_session_fixation.json and feature6_6_session_fixation.json), the flow is login → capture cookie value → logout (clears cookie) → login again → verify new cookie differs from captured one, output fields: login_cookie=present, logout_cookie_cleared=true, replay_cookie_differs=true.
