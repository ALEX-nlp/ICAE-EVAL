## Product Requirement Document

Hey team, we need to ship the ordered list positioning engine we've been talking about. The core idea is that any record type should be able to 'join' an ordered list and have all the bookkeeping done automatically — no more manual position juggling when things get inserted, moved around, or deleted. We've had bugs in the past where gaps never closed after deletions, or positions collided when someone inserted in the middle, and that's exactly what this should fix.

A few things I know we need: records should default to going to the end of the list when created, but we need a way to flip that so new stuff shows up at the top instead. There should also be a way to completely opt out of auto-placement. We talked about supporting sub-lists (like, the same table can have multiple independent ordered lists based on some grouping field). Oh, and remember that constraint thing we ran into on the login module — same deal applies here, the shifting logic needs to handle that edge case gracefully.

Also, someone on the data team asked about zero-based ordering for one of their exports, so that config knob needs to be in there. And we definitely need timestamp behavior to be controllable — sometimes you don't want a reorder to look like a content edit. There's also a bulk-operation mode where you want to pause the auto-maintenance temporarily. Error handling should be clean and categorized, no raw exceptions leaking out.

One extra pass on defaults, since a few people asked: if nothing is specified for placement, new records go to the end of their list and get the next sequential position. So the default for add_new_at is 'bottom'. If you switch add_new_at to 'top', new records get prepended instead. And if add_new_at is null, the record is created with no position and just doesn’t participate in ordering.

On numbering, the default start is 1, so top_of_list defaults to 1. If someone wants zero-based behavior, setting top_of_list to 0 makes it act like a zero-indexed array, where the first element is 0, the second is 1, and so on. That same scale should apply consistently to position-based operations like insert_at, move_to_top, etc.

For the independent list behavior, this is driven by scope. You can pass a single field like scope: 'parent_id' or multiple fields like scope: ['org_id', 'board_id']. Records with the same value(s) across those scope key(s) are one ordered list with their own contiguous numbering, and different scope buckets are totally separate lists. If a record changes scope, it should come out of the old list with the gap closed behind it, then get inserted into the new list either at the end or at an explicit position.

Also clarifying the unique index case: if the position column is configured with unique: true, the library should automatically use sequential_updates, which may show up as stepwise_shuffle in query output. The point is to shift rows one at a time instead of bulk-updating them all at once, because the bulk version can briefly create duplicate position values. That automatic behavior kicks in when unique is true, but it can still be explicitly overridden with sequential_updates: false in config. This is the same sequential_updates / stepwise_shuffle behavior tied to unique: true in config, and the related config knobs here are sequential_updates / unique / positive.

For bulk work, suspend_maintenance wraps a nested list of operations and lets us temporarily skip the usual auto-fixups. While that’s active, position changes and deletions are just persisted as raw writes, with no gap-closing or collision-shifting happening automatically. That’s intentional for cases where the caller plans to repair positions manually afterward. It can also take an optional argument for additional record types to suspend, but that argument has to be an array, and every item in it has to be a valid persistable record class. Anything else should be rejected immediately with a categorized error, specifically argument_not_array or argument_not_record_classes.

Last piece is timestamps: touch_on_update is a config boolean and defaults to true. If touch_on_update is false, reorder operations should not refresh modified-timestamps on shifted records. This is covered with the at_time operation wrapper and timestamp_buckets query.
