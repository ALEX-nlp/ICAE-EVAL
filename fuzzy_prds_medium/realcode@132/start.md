## Product Requirement Document

hey team, we need a small bridge utility that takes background job configs and spits out the right payload for the native scheduler side. basically devs keep hand-rolling these wire maps and getting field names wrong or doing the time unit conversions incorrectly (seconds vs milliseconds is a constant source of bugs). we had something similar in the old login module's compatibility layer — same idea, just adapt that pattern here.

the tool should read a JSON blob from stdin and print the result to stdout. there are two main scenarios: one is an app startup handshake thing (involves a handle and a mode flag), the other is the actual work scheduling case which has a bunch of optional stuff like repeat intervals, network/device conditions, collision behavior, and retry settings. oh and user-provided data should get bundled into the payload in a specific way.

if something required is missing we should NOT crash with a stack trace — it needs to output a clean machine-readable error so downstream tooling can parse it reliably. the test harness should dump raw stdout into per-case files under a namespaced folder so we can diff them easily. don't overcomplicate the structure but also don't throw everything into one god file. we'll probably add more policy types later so keep that in mind.

one extra pass on this after the team questions: the routing needs to be dead simple at the adapter/CLI entry point. if the top-level key `callbackHandle` is present, treat it as the startup payload path. if the top-level key `work` is present, treat it as the scheduling payload path. those are distinct cases and they produce structurally different outputs, so please keep that split explicit, basically the same kind of command-dispatcher shape we used in the earlier auth flow bridge.

for the startup case, the output is very strict: exactly two fields, in this order, `isInDebugMode` and `callbackHandle`. `isInDebugMode` is the boolean copied from input `debug`, and `callbackHandle` is the numeric value copied from input `callbackHandle`. nothing else should be included. stdout should just be a single JSON object followed by a newline, with no status text.

for the scheduling case, field order matters too. the output fields should appear exactly as `isInDebugMode`, `uniqueName`, `taskName`, `tag`, `frequency`, `existingWorkPolicy`, `initialDelaySeconds`, `networkType`, `requiresBatteryNotLow`, `requiresCharging`, `requiresDeviceIdle`, `requiresStorageNotLow`, `backoffPolicyType`, `backoffDelayInMilliseconds`, `outOfQuotaPolicy`, `inputData`. also, `initialDelaySeconds` stays as-is and maps straight through to `initialDelaySeconds`, still in seconds. the one that does need conversion is `retryBackoffDelaySeconds`, which becomes `backoffDelayInMilliseconds` by multiplying by 1000, so input value 1 → output 1000 and input value 3 → output 3000.

also confirming the harness file layout since that came up: it writes one file per case to `rcb_tests/stdout/<cases-dir>/{filename_stem}@{case_index_zero_padded_3}.txt`. for example, the first case (index 0) of `feature1_startup_payload.json` run with `--cases-dir public_test_cases` goes to `rcb_tests/stdout/public_test_cases/feature1_startup_payload@000.txt`. each file should contain only the raw stdout of the program under test. if no cases dir is provided, the default is `test_cases`.
