## Product Requirement Document

Hey team, we need to sort out the feature flag system for the web app. Basically right now devs are copy-pasting if/else checks all over the place — in views, templates, URL configs — and every time we want to roll something out to a subset of users or kill a feature during an incident we have to do a full deploy. It's a mess and things keep drifting out of sync between layers.

We need a clean, centralized way to toggle behaviors on/off at runtime without touching code. Flags should be evaluatable against things like the current user, URL path, query params, date/time, and so on — similar to how we handled the conditional login stuff last sprint, but more generalized. The system should work across views, URL routing, and templates consistently.

There should also be a way to pull flag definitions from multiple places — both static config and whatever storage backend we're using — and merge them. Validation, admin forms, caching per request, all of that needs to be covered. If a flag isn't defined at all, that should be distinguishable from a flag that's explicitly turned off — that distinction has burned us before.

Also the code structure matters here, this shouldn't end up as one giant file. Keep things separated by responsibility. Let's make sure edge cases like missing context, bad config, and broken providers are handled gracefully and don't just blow up.

One other important detail on the state behavior: if a flag just doesn’t exist in config anywhere, that needs to come back as its own third state, `state=none`, not get flattened into `state=false`. We really do need callers to be able to tell “explicitly off” from “never defined.” For the convenience helpers, the enabled helper should still treat none as off, so `enabled=false`, and the disabled helper should treat none as on, so `disabled=true`.

On the request-dependent checks, when something in the condition layer needs request context and there isn’t a request available, we should normalize that result instead of blowing up. The shape needs to be exactly three fields: `error=missing_required_argument`, `argument=request`, and `condition=<condition_name>`. Same idea for a class-based view guard with no flag name configured — that should come back as `error=misconfigured` and `reason=flag_name_required`, not an unhandled exception. And if a flag-guarded route gets built with something that’s neither a callable nor a valid route group, return `error=invalid_view` rather than throwing.

A couple notes on caching and call-time inputs too. Once middleware has cached the flag collection on a request, every later check for that same request should use that cached snapshot even if live config changes after that. If a request never went through the middleware, then it should reflect the current live configuration. Also, that cached attribute should only exist after middleware runs. On the template side, both the Django template tags (`flag_enabled`, `flag_disabled`) and the Jinja2 globals should accept an extra keyword argument at call time and pass it through into condition evaluation, which is how the value-matching conditions get the value they need.

For admin/config plumbing, the form’s condition choices need to be calculated when the form instance is created, not once at module import time. That way if a condition gets registered later, a newly created form still sees it as a valid option. Validation should scan all configured flags for conditions that point at unregistered condition types. A clean config should report `warnings=0`, and if a flag references an unknown condition type it should produce exactly one warning entry with the stable diagnostic id `warning id=flags.E001`.

Also, for persisted records coming from the `DatabaseFlagsSource` provider in `flags/sources.py`, each record is still just the persisted flag data `(name, condition, value)` from the database-like store, and it needs to render as a readable sentence in exactly this form: `description=<FLAG_NAME> is enabled when <condition> is <value>`. All three pieces are required. And just to anchor the implementation shape we’re talking about, the condition evaluation subsystem in `flags/conditions.py` is still the named condition class pattern with a `check(value, request)` interface used for user, anonymous, parameter, path, and date conditions.

A few exact details to be precise about: A flag that is not configured at all must resolve to a distinct third state — `state=none` — not the same as `state=false`. This is critical so callers can tell 'explicitly turned off' from 'never defined'. The enabled helper treats none as off (enabled=false), and the disabled helper treats none as on (disabled=true).
