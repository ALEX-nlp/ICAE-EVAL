## Product Requirement Document

We need a shared in-memory name registry and address resolver for a host that runs many isolated virtual networks, each containing multiple workloads that must discover one another by friendly name rather than raw address. A configuration loader reads a simple text-based description for each virtual network — the network's gateway listen points and the membership list of workloads attached to it, including their addresses and the aliases they should be reachable by — and builds lookup tables once at startup. The library must then support six query types: report the listen addresses of all networks broken down by address family; translate a workload name into its addresses from the perspective of a requesting caller; translate a raw address back into the names that own it; retrieve the custom upstream resolvers declared for a specific workload; retrieve the shared upstream resolvers inherited by all workloads on a given network; and report which network or networks a given address belongs to. Name matching must be case-insensitive so that mixed-case registrations do not cause misses. When a name lookup finds no match, the system must signal a non-existent domain rather than returning empty success. A single malformed network description must be silently skipped — it must not abort processing of the remaining well-formed networks. The core domain logic must be cleanly separated from any command-dispatch or input-parsing layer, and the dispatch layer reads a structured operation request and routes it through the internal resolution engine.

One small follow-up from the questions that came in: for name resolution, if the requested name is not registered on any network the caller is part of, the result must output exactly 'nxdomain\n'. Returning an empty success or an empty address list is not acceptable here; the sentinel value 'nxdomain' has to be emitted instead. Name matching is fully case-insensitive end to end, and we normalize names to lowercase internally, so a lookup for 'HELLOWORLD', 'helloworld', or 'HelloWorld' should all hit the same stored entry. The queried value gets lowercased before matching against the stored keys.

Also on the per-container resolver side, when a container line has no fifth field and there are no per-container dns-servers declared, querying that container's address with op='container_dns_servers' must output exactly 'dns_servers=none\n'. The literal sentinel value is 'none' in lowercase, and we should not turn that into an empty list or skip the line.

For op='bind_addresses', please keep the output stable by enumerating networks in lexicographic alphabetical order by network name. For each network, the output is exactly three lines: 'network=<name>\n', 'bind_v4=<comma-separated IPv4 addresses or empty>\n', and 'bind_v6=<comma-separated IPv6 addresses or empty>\n'. Within each family, preserve the address order from the first line of that network’s config file rather than re-sorting.

On malformed input, the intended behavior is still to skip only the bad network and keep going. That includes cases like an unparseable address token on the first line or a container line with fewer than 4 space-separated fields. The overall parse does NOT fail if one network is bad; well-formed networks still load and are returned normally. If the directory only contains malformed networks, that just yields empty output with no error and no networks reported.

And just to anchor where this lives since the team asked: the domain behavior is centered in the DNSBackend struct in src/backend/mod.rs, specifically lookup(), reverse_lookup(), and get_network_scoped_resolvers(), and the config loading piece is parse_configs() in src/config/mod.rs, which reads all files from a directory, each filename being the network name and the file content being the text config format.
