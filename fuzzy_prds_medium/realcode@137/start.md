## Product Requirement Document

Hey team, we need to build out this notification delivery library we've been talking about. The idea is that devs should be able to define a notification once and have it go out through multiple channels without re-writing all the plumbing every time. Think of it like that messaging abstraction we did for the billing service, but more generalized.

At a high level we need: payloads that carry arbitrary data, some kind of validation so we catch bad inputs early, immediate and deferred sending, the ability to skip channels based on conditions, lifecycle hooks so other parts of the system can observe what's happening, and durable storage so we can show users their notification history with read/unread tracking. We also want email, websocket broadcast, HTTP integrations (like Slack), and mobile push support.

One thing that keeps biting us is that when a user deletes something in the app, their related notifications linger in the DB and cause confusion — we need a way to tie notifications to application records and clean them up properly.

Also the ops team mentioned something about needing to store notification payloads in different column formats depending on the database setup, so make sure serialization is flexible.

Finally, it would be great if there was a generator so teams can bootstrap the storage model quickly instead of writing migrations by hand. Localization support would also be a nice-to-have so copy lives in translation files.

One quick follow-up from the questions that came in: when I say payloads are arbitrary, I mean a notification payload is a key-value map of arbitrary developer-supplied fields. The adapter prints `param_count=<N>` followed by one line per entry in the form `param_<key>=<value>`. For example, passing `{"foo": "bar"}` produces `param_count=1\nparam_foo=bar\n`. Also, validation failures should stay normalized and readable, not blow up as raw exceptions. The adapter prints `error=validation\nmessage=missing_param=<field_name>\n`. For example, if `user_id` is required but absent the output is `error=validation\nmessage=missing_param=user_id\n`. The same pattern applies to missing delivery-channel options.

On delivery behavior, email should support both immediate and queued paths. For immediate send the output is `emails_sent=<N>`, `email_kind=mail_message`, `recipient_email=<email>`, and direct channel execution returns a mail-message object. Queued email prints `queued_emails=<N>`. If mailer config is missing, same idea as above: return a normalized error instead of a raw exception. Realtime should broadcast to the recipient-specific channel `noticed:notification:<base64_gid>` where `<base64_gid>` is the base64-encoded Global ID of the recipient. The output includes `broadcast_channel=noticed:notification:Z2lkOi8vZHVtbXkvVXNlci8x`, `broadcast_count_delta=1`, and `last_payload_<key>=<value>`. Channel selectors can come in as a string, object, or callback.

For HTTP integrations, we’re stubbing outbound POST requests. A successful Slack delivery should show `request_host=hooks.slack.com`, `status=200`, `response_body=ok`. Debug logging can be enabled per channel. If the response fails, print a normalized HTTP failure with `status=<code>` and `response_body=<body>` rather than raising a raw HTTP exception. Push credentials also need to be flexible: hash, file path, string path, or callback all work. For a hash credential, output is `credential_keys=<comma_separated_keys>` followed by `<key>=<value>` for each entry. Project IDs and access tokens are extracted from the resolved credential object, and if the callback config is missing that should also come back as a normalized error.

A couple more specifics people asked about: translation lookup supports plain keys, namespace-scoped keys, custom-scope keys, and HTML-safe values. For a plain lookup the adapter prints `lookup_key=<key>` and `message=<translated_text>`. For HTML-safe translations it additionally prints `html_safe=true`. Keys are resolved relative to the notification's translation scope unless a custom scope is provided. On the generator, it creates a notification model file and exactly one matching migration file. Output: `model_file_exists=true\nmigration_file_count=1\n`. The model name is passed as the `name` parameter (e.g., `TestNotification`).

And just to anchor scope, this all sits on the multi-channel delivery engine described in start.md — specifically the shared engine that validates, executes, schedules, persists, serializes, broadcasts, and reports observable delivery effects consistently across all channels. Same goes for payload storage behavior: we need the payload serialization coder selection logic in Feature 10 (param_serialization command) that handles text, json, and json-like column formats with the appropriate coder_kind per format.
