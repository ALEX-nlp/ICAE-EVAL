## Product Requirement Document

Hey team, we need to build out this cross-platform error reporting bridge thing. Basically right now our devs are complaining that they have to write a ton of boilerplate every time they want to get crash reporting working on both web and mobile at the same time — sessions are out of sync, native crash data is missing user info, that sort of thing. We want a single setup flow that just handles everything depending on what platform you're on.

The tricky parts are: (1) starting up the native side correctly without blowing up when config is missing, (2) making sure events actually get packaged up the right way before they go through the native layer (think about how that login module did the payload wrapping last time — same idea here), (3) keeping user/tag/context data in sync on the native side, and (4) there are some old severity names floating around in the wild that we need to quietly remap before they hit the bridge.

Android and iOS behave slightly differently for context data specifically — Android needs a fallback path. Also the envelope format for events has some specific shape requirements around byte length and content type that I don't fully remember, so double-check that.

Please make this a proper multi-file project, not a single blob. Tests should run with a single bash command. Needs to handle bad/missing config gracefully with real warnings, not silent failures.

One extra pass on the platform behavior since folks asked: on the web platform, browser.auto_session_tracking defaults to true when enableAutoSessionTracking is not explicitly set. On iOS and Android (native platforms), browser.auto_session_tracking is always false regardless of the option. For native platforms, native.enable_auto_session_tracking mirrors the enableAutoSessionTracking option (defaults to true if not set). Also for startup, if startNativeBridge is called without a DSN, the bridge does NOT start (native_started=false), no bridge initialize call is made (bridge_call_count=0), and the exact warning emitted is: 'Warning: No DSN was provided. The Sentry SDK will be disabled. Native SDK will also not be initalized.' Note the intentional typo 'initalized' must be preserved exactly.

Also confirming the disabled-native path because that came up too: when enableNative is false and enableNativeNagger is true, the output must include native_started=false, native_enabled=false, and the exact warning: 'Note: Native Sentry SDK is disabled.' No bridge calls are made (bridge_call_count=0). And for the transport handoff path, when a send_event_through_transport scenario runs, the output must be: transport_status=200 and send_event_calls=1. The transport adapter delegates to the native event sender and counts how many times sendEvent was invoked.

On the event packaging piece, the envelope item header length field (e.g. 'length':99) represents the byte length of the serialized payload JSON string. The payload is the full event object with the message field transformed into an object with a 'message' key (e.g. message:'test' becomes message:{message:'test'}). The length is the byte length of that serialized payload. Same deal in Feature 3: the message string field must be wrapped into an object {message: <original_string>} in the payload, and the envelope is split into three parts: header (event_id + sdk), item header (content_type, length, type), and payload.

Last thing, for scope sync on Android, it does not support set_context natively; it must fall back to set_extra with the context value JSON-stringified as a string, exactly as described in the sync_context android test case. And the test runner should be wired exactly through 'bash rcb_tests/test.sh'. It accepts --cases-dir to specify a subdirectory (default: public_test_cases). Output files are written to rcb_tests/stdout/<cases-dir>/ using the naming pattern {filename.stem}@{case_index.zfill(3)}.txt (e.g. feature1_client_initialization@000.txt). Each file contains only raw stdout from the program under test.
