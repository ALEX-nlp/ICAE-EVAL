## Product Requirement Document

Hey team, we need to build out that payload parsing library we've been talking about for the CI bot. The bot runs against multiple hosting environments and right now every automation script has to do its own messy JSON wrangling — it's a nightmare to maintain and keeps breaking when upstream shapes change. We want one clean library that reads in whatever the bot receives and spits out a normalized, typed model that scripts can just use directly without digging into nested JSON themselves.

The library needs to handle all three of our hosting environments (you know the ones — the two cloud ones and the self-hosted server product we support). Each has its own payload shape, different casing conventions, different timestamp formats, etc. We also need it to handle things like diff statistics, file change lists, and platform detection.

For the output side, think about how we did the formatting in that old shell-report module — flat key=value lines, one per field, with dotted paths for nested stuff. Timestamps should come out as numbers not strings. Missing optional fields should come out as null not blow up. The stdin/stdout adapter piece should stay completely separate from the actual parsing logic — same idea as that separation we did on the login module integration. Make sure it's properly split into multiple files, not one giant blob.

A couple extra specifics from the follow-up questions: for the hosting env mapping, this is GitHub (pull requests), GitLab (merge requests), and Bitbucket Server (self-hosted) — corresponding to view prefixes 'github_*', 'gitlab_*', and 'bitbucket_*' respectively. The output format is still the same flat key=value line-per-field output format with dotted paths for nested objects, list index notation for arrays, and literal 'null' for absent optional values, as defined in the adapter output contract in the test cases.

On timestamps, all timestamps (created_at, updated_at, closed_at, merged_at, due_on, authorTimestamp, committerTimestamp, etc.) must be rendered as epoch milliseconds (integer). ISO-8601 strings from the wire payload must be parsed and converted. Timestamps that are absent or null in the payload render as the literal string 'null'. Timestamps already provided as millisecond integers (e.g. Bitbucket server's createdDate/updatedDate/authorTimestamp/committerTimestamp) are passed through directly as integers. Also, the 'state' field must always be rendered as an upper-case symbolic name regardless of how it appears in the wire payload. For example, wire value 'closed' becomes 'CLOSED', 'open' becomes 'OPEN'. This applies to pull request state, merge request state, milestone state, and any other state enum fields. Same deal for the 'type' field on GitHub/cloud user accounts — it must also be upper-cased in output. For example, wire value 'User' becomes 'USER', 'Organization' becomes 'ORGANIZATION'. This applies to the user.type field in github_pull_request and similar views.

A few view-specific notes too. For the 'github_commits' view, the inner git commit object (i.commit.sha) may or may not be present in the wire payload. If absent it renders as 'null'. If present it renders as its string value. This is distinct from the outer commit sha (i.sha) which is always present. For the 'gitlab_merge_request' view, the 'can_be_merged' boolean flag is derived from the wire field 'merge_status': it is true when merge_status equals the string 'can_be_merged', false otherwise. The raw merge_status string is not directly emitted; only the derived boolean 'can_be_merged' is output. And for GitLab milestone, it uses 'due_date' and 'start_date' as wire keys (not 'due_on'). These are exposed as epoch millisecond timestamps in the output. The GitLab milestone also exposes 'project_id', 'iid', and 'web_url' fields not present in GitHub milestones.

One other implementation split to keep clean: the stdin/stdout adapter is solely responsible for reading a JSON command envelope, routing by 'view' key, calling into the core model layer, and formatting output. The core model and parsing layer must have no dependency on stdin/stdout or the JSON command envelope format. Diff statistics computation must depend on an abstraction for running a command, not a concrete process launcher. Also for the 'file_contents' view, the adapter outputs the value of the 'content' field verbatim, with no trailing newline appended beyond what is already in the content string itself. The output is exactly the raw string value of 'data.content'.
