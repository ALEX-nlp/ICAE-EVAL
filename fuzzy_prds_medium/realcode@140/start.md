## Product Requirement Document

Hey team, we need to wrap up the admin panel permissions helper we discussed last sprint. The basic idea is that we want something that can automatically figure out the right 'name' for each admin screen/resource so we don't have to keep manually writing those strings every time someone adds a new screen. Last time we did something similar for the login module — same kind of naming convention applies here, just refer to how we handled that.

The tool also needs to be smart enough to skip certain screens we don't want to expose (like the main dashboard landing), and it should handle the three types of things we have in the panel: the data management screens, the standalone pages, and the little stat cards/charts at the top of the dashboard.

The big thing for the ops team is that whenever a new screen gets added, the super-admin account should automatically get access without anyone having to remember to go in and add it manually. We've had incidents where a new report page went live and the super-admin couldn't even open it for two days.

One more thing — there's been a complaint that some naming outputs are causing weird bugs downstream because of a specific character that conflicts with how our system parses permission strings internally. We need to make sure those are caught and rejected with a clear error. The code should be split into sensible modules, not one big file.

A couple follow-ups from the questions that came in: for the default naming strategy, it should strip everything up to and including the `Resources\` namespace segment, remove the trailing `Resource` suffix, then split the remaining PascalCase on uppercase word boundaries, lowercase it, and join the tokens with `::`. So `App\Filament\Resources\BlogPostResource` becomes `blog::post`, and `App\Filament\Resources\OrderResource` becomes `order`. That same PascalCase-to-double-colon behavior is the intended default permission identifier derivation flow, just to make it explicit since a few people asked how aggressive the splitting should be.

Also, on the bad-character issue, `_` is reserved as the delimiter between an action prefix and a resource identifier inside fully-formed permission names, like `create_post`, so if any naming strategy produces an identifier token containing an underscore, that needs to be rejected instead of returned. In that case the system should emit a neutral error line exactly like `error=identifier_has_underscore class=<FQCN> identifier=<bad_token>`. Same point as before, just spelling out the exact format we want there.

For discovery defaults, `Dashboard` is always excluded from page discovery, even if the caller does not pass any exclusion list at all, which is why `App\Filament\Pages\Dashboard` should never show up in page permission results. Same deal for widgets: `AccountWidget` is always excluded from widget discovery by default. On the resource side, we always inject the built-in role-management resource into the discovered set no matter what is registered, and that one has identifier `role` and model short name `Role`. It should be present in every `discover_resources` result and it cannot be removed through the exclusion list.

And for action prefixes, the global default list, in order, is `view`, `view_any`, `create`, `update`, `restore`, `restore_any`, `replicate`, `reorder`, `delete`, `delete_any`, `force_delete`, `force_delete_any`. If a resource declares its own custom prefix set, that should fully override this default list rather than merge with it, so when that happens we only use the prefixes the resource declared.
