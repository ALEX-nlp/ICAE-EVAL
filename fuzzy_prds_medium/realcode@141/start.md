## Product Requirement Document

We need a reusable documentation toolkit that our content teams can use to convert raw web pages and generated text into clean, ready-to-store knowledge packs. The basic idea is: given a web page (in raw HTML), the tool should produce a clean text version in a standard lightweight markup format so we can store everything uniformly. On top of that, our AI-generated drafts often come back wrapped in extra formatting artifacts or with metadata headers that shouldn't be in the final stored copy — we need something that strips all that away and ensures the output always ends cleanly. We also want to support assembling a knowledge pack from a declared list of sources, where each source's content is gathered into one combined buffer. This step has to be strict: if any source turns out to be insecure or unreachable, the whole operation should fail immediately instead of producing partial results — similar to how the auth module handles credential validation failures. Finally, each knowledge pack is described by a small configuration object that gives it a name, a short summary, an optional extra-guidance field, and an ordered list of content sources; we need a parser that reads this structure and surfaces all fields cleanly. Everything should be organized so the core logic is completely decoupled from command routing and input/output handling, just like the way our data pipeline separates processing stages.

A couple more specifics from the team’s questions. For headings, each h1-h6 maps straight to markdown with the same number of hash marks, then a space, then the heading text. The block shape is exactly '\n# content\n\n', so there’s a leading newline and a trailing double-newline around the rendered heading before the overall result gets trimmed. Because of that, adjacent headings end up looking like '#### H4\n\n\n##### H5\n\n\n###### H6\n' once everything is stitched together and trimmed.

For lists, we’re keeping it intentionally simple: both ol and ul render the same way as flat bullet lists. Every li becomes a line starting with '- ' and then the item text, and the wrapping list container is just '\n<items>\n'. There is no numbering at all, so ordered lists are NOT rendered with numbers.

On embedded media, if a video has no title attribute we use 'Video', and if an iframe has no title attribute we use 'Iframe'. For video rendering, a poster changes the output format: with a poster it becomes '[![title](poster)](videoUrl)', and without a poster it stays a plain link as '[title](videoUrl)'.

Also clarifying the cleanup behavior a bit more. The cleanContent function only removes an outer fenced code block if the entire content is wrapped in a single one. It checks the start with '^\s*```[a-zA-Z]*\s*\n' in a case-insensitive way, and the closing fence with '\n```\s*$'. If there are code fences inside the actual content, those stay exactly as-is. After that outer fence handling, front-matter wrapped by '---' lines is removed, then the result is trimmed, and the final output always ends with exactly one trailing newline ('\n').

For the front-matter part specifically, removal looks for the first '---'. If that first one is at index 0, it looks for the closing '---' starting from index 3 and strips everything through that closing delimiter plus 3 characters. If the first '---' shows up after index 0, meaning there’s noise before it, we still look for the matching closing '---' and strip from that first occurrence through the closing one. After that, the result is trimmed, so yes, leading noise before a front-matter block gets removed too.

And on source aggregation, ResourceFetcherService.fetchAndConvertContent should throw an Exception immediately on the first validation failure, including insecure URL, missing config directory, or missing local file. The point is still fail-fast behavior so we never end up with partial resource aggregation.
