## Product Requirement Document

Hey team, we need to build out that async coordination library we've been talking about. Basically, we're getting a lot of complaints from devs that they keep writing the same boilerplate state-tracking code every time they need to wait on something async — it's messy, buggy, and nobody wants to maintain it. Think of it like what we did with the deferred pattern on the payments flow, but more general-purpose.

Core stuff we need: some kind of result object that can be pending, done, or failed. Once it's settled it shouldn't be changeable (but repeating the same settlement should probably just be fine). Devs should be able to chain work off these results, recover from failures, and cancel things that are still in-flight. Cancellation should ripple upstream through any dependency chain.

We also want batch helpers — wait for all of them, wait for just enough of them, or just get a full report of what happened without blowing up. Plus some iterator-style helpers for processing lists with optional concurrency limits.

There's also a queue thing — tasks should run in the order they were added, and we should be able to schedule a thunk and get a result object back. Oh and generator-style async sequencing would be great, similar to how we wired up the step-runner in the old pipeline module.

Finally some simple state-check helpers so you don't have to poke at internals. Timeline is flexible but let's not gold-plate it.

One small follow-up from the questions that came in: on settlement, we're being strict about outcome changes but relaxed about exact repeats. Repeating the exact same settlement action with the exact same value on an already-settled result is silently accepted and leaves the state unchanged. But if somebody tries to change the outcome after the fact, like rejecting something already fulfilled or fulfilling something already rejected, that needs to raise a `state_change_forbidden` error, and the final state and stored value stay whatever they were from the first settlement. This is all in line with the core `Result` / `Deferred` object lifecycle described in Feature 1 of start.md, specifically the pending → fulfilled/rejected state machine with single-settlement guarantees and producer callback pattern.

Also, a little more color on waiting and errors: when waiting on a rejected result with unwrapping enabled, the output must include `error=result_rejected` and `reason=<original reason>`, and after that the state still remains `rejected`. If unwrapping is disabled, the wait should still drive the state transition, but it should NOT emit those error/reason lines. Separate case here: if the producer callback itself throws, the reported error must be `callback_error`, not `result_rejected`, and it should use the callback's exception message.

On cancellation, cancelling a pending result rejects it and invokes its cancellation hook exactly once. If you cancel an intermediate node in a dependency chain, that cancellation propagates upstream, so all ancestor pending nodes are also rejected. But nodes downstream of the cancelled node don't immediately flip; they remain pending until somebody explicitly waits on them. At that point, the output must include `result_cancelled`, and then that child ends up in the `rejected` state.

For the batch helpers, the main nuance is ordering. The all-required batch can have items settle in whatever order they happen to settle, but the final fulfilled value list MUST preserve the original input order, not the resolution order. And if even one input rejects, the whole batch rejects immediately with that item's reason. For the count-based competitive version, if it becomes impossible to reach the requested number of fulfilled results because too many have already rejected, it must reject with the error key `not_enough_fulfilled` and include a list of all rejection reasons collected so far. Also, `batch_any` should fulfill with the single winning value directly, NOT wrapped in a list, whereas `batch_some` returns the winning values in completion order as a list.
