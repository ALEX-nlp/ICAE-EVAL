## Product Requirement Document

Hey team, we need to build out that auto-loading library thing we talked about last sprint. Basically the idea is that developers shouldn't have to write a bunch of require statements by hand — the system should just figure out what file maps to what constant name based on where the file lives. Like if you have a file called `users_controller.rb` it should just know that maps to `UsersController` without anyone telling it explicitly.

We also need it to handle the nested folder situation — same pattern we used for the namespace resolution in that old config module, you know the one. Directories should act as namespaces automatically.

There's also the reload story — in development people want to be able to change a file and have it pick up the new version without restarting. But we should make sure that only works if someone explicitly opts into it, otherwise it should blow up with a clear error.

Oh and the gem versioning edge case — someone brought this up in the thread — when it's the top-level version file directly under the package namespace it should behave differently than a nested one. Ask around if you're not sure what I mean by that.

Ignored paths, eager loading, duplicate root handling — all the usual stuff. Make sure stale constants get cleaned up properly on unload. Error messages need to be consistent and machine-readable.

One quick follow-up from the questions that came in: the default name conversion is still the simple basename rule, just a little more explicit. Each underscore-separated segment of the basename is capitalized and the segments are concatenated, so `users_controller` becomes `UsersController` and `point_3d_value` becomes `Point3dValue`. Numeric characters stay in the segment they belong to, with no special handling, and acronyms are not guessed unless there’s an explicit override registered. We do support an overrides map of basename strings to constant name strings, and if a basename is in that map we use that mapping verbatim instead of the default capitalization logic. Anything not in the map keeps using the normal conversion, and the override has to be declared before setup/scanning.

A couple more specifics on load/unload behavior too: unloading removes every constant that was loaded from a managed file and also every namespace module that was autovivified by the loader. After an unload those constants are no longer defined on their parent modules, and the next access should lazy load again from scratch. On the require side, a managed file required through the loader returns a truthy loaded result the first time, and a second require of the same path returns a falsy result to show it was already loaded. Same idea if the constant was already autoloaded by constant access first — a later require of that same relative path should still report it as already loaded.

On the error side, if a managed source file gets loaded but does not define the constant its path implies, we should report a normalized `expected_constant_missing` error with the name of the expected constant. That’s a loader contract failure, not a Ruby NameError. And if some path segment turns into something that is not a valid constant name, we should catch that during setup/scanning, not later on access. The example here is `foo-bar.rb`, which would infer `Foo-bar`; that should report a normalized `invalid_constant_name` error including the inferred name and whether the bad entry was a file or directory.

Also, roots need to be real existing directories, and internally we store them as absolute paths. It should be possible to query the aggregate list of all managed roots across multiple loader instances, and both the root count and the absolute path normalization should be visible through the public API.

And just to pin down the version-file thing more clearly: for a packaged library, the version file sitting directly one level below the package namespace directory maps to the all-caps constant `VERSION`. If the version file is nested deeper than that direct child level, it uses the normal basename mapping and becomes `Version`. Everything else that’s just a normal file keeps using the normal mapping regardless of its name. The namespace behavior is the same one we discussed before where directories autovivify Module constants and can share a namespace across multiple roots, matching Feature 3 implicit namespace loading, and the version naming behavior above is the same Feature 1.3 package version name mapping rule.
