## Product Requirement Document

hey team, we need to wrap up the persistent worker integration layer we've been discussing. basically the idea is that instead of spinning up the app on every single request, we keep it alive and reuse it — similar to what we did with that session isolation approach in the login module, but applied to the whole request lifecycle here. the main pain points we keep hearing from ops: the server gets stuck in a broken state after certain errors and keeps serving garbage responses until someone manually restarts it; stale database connections are causing intermittent failures on long-running workers; and context from one user's request is bleeding into the next person's error reports which is a compliance nightmare. we also need the worker to gracefully announce when it's starting and stopping so the load balancer can react properly. there's also a thing where some errors should be tolerated (like known app-level stuff) without triggering a full restart, but truly unexpected errors should force a clean rebuild. the metrics setup needs to happen at boot time, not lazily. and the debug dump tool needs to be wired into the request profiler when running in HTTP mode instead of just spewing to stdout. rough priority: get the recycling logic right first, then lifecycle signals, then the db cleanup middleware, then the rest. let me know if anything is unclear, i'll try to get more specs together soon.

one quick follow-up after the team questions: on the recycling side, the aggregate policy is just a logical OR. if at least one member policy votes yes, we recycle, and if none do, we don’t. if the aggregate has zero members, it never votes to recycle. so the final output there is just one boolean decision, either should_reboot=true or should_reboot=false.

for the exception-based behavior, the allow-list check is hierarchical, not just exact-match. in the OnExceptionRecyclingPolicy, anything whose category matches or falls under an allowed category is tolerated, so if 'app' is allowed then something like 'app_child' is also allowed because it’s a prefix/sub-type case. those do not trigger a recycle vote. only categories not covered by the 'allowed' list, either directly or via hierarchy, should vote to recycle. same idea applies to the earlier note about tolerated app-level errors versus unexpected ones.

also, for force/clear behavior: a force signal includes a human-readable reason string, and when we process it we should record that reason and vote to recycle. the reason needs to be emitted as 'forced_reboot_reason=<reason>' before the 'should_reboot=true' line. if we get a clear/reset signal, that wipes any accumulated error and force state and puts the policy back into abstain mode, meaning should_reboot=false no matter what happened before the reset.

on lifecycle events, the worker always emits 'event=worker_start' before it handles any requests and 'event=worker_stop' after the request loop ends. that still applies even if there are zero queued requests — both events should show up, in that order. if a rebuild happens, we emit 'event=kernel_rebooted' after the request finishes and before the next request starts, or before worker_stop if that was the last one.

for the HTTP runtime bit, when the worker starts there, the global debug-dump facility (var dumper) should be redirected to the worker-provided dumper so dumps get captured by the request profiler instead of going to process stdout. the confirming output is handler_redirected=true and dumped_value=<the value passed>.

one more implementation detail from build time: during container build, services tagged as gRPC services should be scanned and registered once per service-contract interface they implement. so one contract means one registration, multiple contracts means one registration per contract, and any non-contract interfaces on that same service are ignored completely. each registration line should look exactly like 'register service=<id> interface=<contract_alias>' and come out in discovery order.

and just to be explicit on the request isolation piece, this is the same per-request error scope isolation middleware described in Feature 6 (feature6_error_scope_isolation.json), which ensures Sentry/error-reporter breadcrumbs and context do not bleed between consecutive requests handled by the same persistent worker process.
