## Product Requirement Document

We need a reusable data transformation and API output library that converts internal application objects into clean, client-facing API responses. The library should let developers define how individual records map to their public representation — dropping any internal or private fields, normalizing certain field types, and declaring which related entities can optionally be embedded. Callers should be able to request nested relationships using a compact dot-separated syntax in a single string or as a list, and the library should handle de-duplication and implicit parent path expansion automatically. There should be a configurable safeguard against runaway nesting depth, with a default cap that should match the way the include parsing module works today. Relationship directives can carry named modifier parameters using a special colon-and-parentheses syntax, with pipe-delimited values; these parameters must be exposed via a read-only container that behaves like the immutable bag used in the auth module, throwing errors on any mutation attempt. The final output format should be pluggable: one mode wraps everything under a top-level key, another outputs fields directly at the root level for compact responses, and a third side-loads related entities into a separate section to avoid duplication across records. All three modes should support attaching arbitrary metadata and pagination information — either page-based with previous/next link generation or cursor-based. A single runnable adapter should translate JSON commands into library calls and print results to stdout, normalizing any internal errors into a neutral format.

A couple more specifics from the questions that came back. The default recursion limit is 10, and include paths are trimmed to at most 10 dot-separated segments before ancestor expansion, with anything beyond position 10 silently discarded. That ancestor fill-in still happens automatically after the trim, so if someone asks for 'bar.baz' we should end up treating both 'bar' and 'bar.baz' as requested includes. We should also keep de-duping include entries while preserving the first-seen order, so something like ['foo', 'bar', 'bar'] results in ['foo', 'bar']. And if the includes argument is neither a string nor an array, like null or an integer, the system should output exactly 'error=invalid_includes_argument\n' and not produce any normalized list.

On the modifier syntax, the split is pretty literal: values inside one modifier are pipe-delimited with the '|' character, and multiple modifiers on the same include are chained with colons after the include name, like foo:limit(5|1):order(-something). Those modifier params should still be exposed through the same read-only ParamBag class in golden/src/ParamBag.php, which is an ArrayAccess implementation that throws LogicException on any offsetSet or offsetUnset call.

One other transformation detail to make explicit: during transformation, any field whose name begins with an underscore ('_') is dropped from the output, so '_author' should not show up in the visible record, and if 'year' is present it should be normalized to an integer. Also, if a relationship is declared as a default include, it always gets emitted even when the caller did not explicitly ask for it, and those defaults are configured via setDefaultIncludes() on the transformer. How that lands depends on serializer mode: with the array serializer, the relationship data is merged inline at the top level, and with the data serializer, it sits under a 'data' key beside the parent fields.

Last bit on side-loaded output: in the JSON API serializer's 'linked' section, related entities are deduplicated by their 'id' field across the primary records, and if the same id shows up more than once we keep only the first occurrence. If an entity does not have an 'id' field, we include it every time without deduplication.
