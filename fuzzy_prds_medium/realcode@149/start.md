## Product Requirement Document

We need a command-line argument parsing utility that allows developers to declare each program input as a typed field, and have all the tedious work of converting, validating, defaulting, and required-checking happen automatically. Right now, our engineering teams are hand-writing verbose, error-prone setup code every time they build a CLI tool, and there is no single object whose fields describe the whole input surface.

The new utility must support scalar types (integers, floats, strings, booleans, file paths), collection types (ordered lists, unique sets, fixed-length tuples with per-position types, and variable-length homogeneous tuples), optional fields that default to a null-like absent value, and choice-restricted fields. Booleans should work both as bare toggle flags and as explicit true/false value flags, controlled by a constructor option. Another constructor option should allow flag names with underscores to be accepted on the command line using dashes instead.

Arguments can also be loaded from configuration files (one or more, later files take precedence over earlier ones, and command-line values override everything). There should be a way to populate the parsed state from a dictionary instead of from the command line. A partial-parse mode must collect unrecognized tokens into a separate list rather than failing. Subcommands must be supported, each with their own argument declarations.

All error conditions should map to a small set of neutral categories: a standard parse failure, a missing file, an unsupported type configuration, and a lifecycle misuse. The output format should follow the same conventions as the existing error-reporting module so the calling system can process results uniformly. Results should be rendered sorted alphabetically by field name, with each collection type rendered in the same deterministic style we already use for set output in our metrics pipeline.

A couple follow-ups from the questions that came in. On rendering, all resolved fields are rendered sorted alphabetically by field name (ascending lexicographic), and each line is exactly 'name=token\n'. If there’s nothing in the mapping, that just means no output lines. For sets specifically, the elements are rendered sorted lexicographically by their string token representation, which is sorted(render_value(x) for x in v), and the format is '{t0, t1, ...}' with a single space after each comma.

On precedence, config file arguments are prepended before command-line arguments. If there are multiple config files, they’re loaded in list order (left to right), so later files have higher precedence over earlier files, and command-line arguments always override all config file arguments. Also, if a collection-typed argument (List, Set, Tuple) has a non-collection default like null/None or a bare scalar, and the flag is not supplied on the command line, we return that default as-is without trying to coerce it into a collection. So Set[str] with default=null renders as 'none', and Set[int] with default=3.7 renders as 'float:3.7'.

For partial parsing, in known_only mode, unrecognized tokens are collected into extra_args as a list of str-typed values. That extra section renders as 'extra_args=[str:token1, str:token2, ...]'. If someone asks for show=['values','extra'], the values section is printed first, then the extra_args line.

One small naming detail too: when underscores_to_dashes=True, option flags, meaning the ones starting with '-', have their underscores replaced with dashes in the flag name on the command line, but the resolved attribute name keeps its underscore form. Positional arguments, registered without a leading dash, are never rewritten, and their name stays exactly as declared.

And for explicit-bool mode, truthy values are any prefix of 'true' (case-insensitive) or the string '1', and falsy values are any prefix of 'false' (case-insensitive) or the string '0'. So 't', 'tr', 'tru', 'true', 'T', 'TRUE', '1' all parse to True. 'f', 'fa', 'fal', 'fals', 'false', 'F', 'FALSE', '0' all parse to False. Anything else is a usage error.

Last piece is the error shape. The output format mirrors the neutral error contract defined in the execution adapter (rcb_tests/dispatcher/run_case.py), so the emitted line is one of error=usage, error=file_not_found, error=invalid_type_configuration, or error=invalid_request, each terminated by a newline.
