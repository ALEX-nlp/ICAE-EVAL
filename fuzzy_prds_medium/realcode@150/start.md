## Product Requirement Document

Hey team, we need to build out the patch management toolkit we've been talking about. The basic idea is that devs can declare patches for their third-party dependencies and the system handles everything automatically — fetching, verifying, applying, all of it. We want this to feel like a first-class part of the dependency workflow, not a bolt-on.

A few things I know we need: patches should be describable as little records with a source location and some metadata, and those records need to survive being saved and reloaded (think lock file round-trips). We also need to group patches by which package they target and make sure the same patch doesn't get applied twice — remember how we handled the deduplication logic in the login module? Something similar here, but patches can collide in more than one way.

Patches also need to be applied at the right 'level' in the directory tree — some well-known packages have special requirements here, and there's a fallback chain similar to what we discussed in the config resolution spike. The system should support declaring patches either inline in the project config or in a separate file, and that file path situation needs to handle missing or misconfigured files gracefully without blowing up.

We also need the lock file naming to follow the active manifest name. And the download step should verify file integrity and report clearly when something doesn't match. Finally the patching tool itself should check whether it's even usable in the current environment before trying anything.

One small follow-up after the team questions: the patch record shape should stay very simple. It has three required fields, package, description, and url, and three optional ones, sha256, depth, and extra. The intent there is package is the target package name, description is the human-readable label, and url is the source location, whether that’s a local path or a remote URL. For the optional bits, sha256 is the content hash, depth is the apply strip depth as an integer, and extra is a metadata map that defaults to an empty list/array when absent. Also important for save/reload behavior: if an optional field is unset, it should serialize as null in the wire form.

On the dedupe side, there are really two ways patches for the same package can collide, and in both cases we keep the first one we saw and discard the later one. They collide if they share the exact same source URL, or if they share the same sha256 content hash even when the URLs are different, like when one has a fragment suffix such as #something. Same rule in the collection layer too: two patches collide if they share the same URL OR share the same sha256 hash (even with different URLs). And when we list packages, we should preserve insertion order based on when each package was first seen, so patched_packages follows that same insertion order.

Also clarifying clear/remove behavior for the collection: clearing a package removes all of that package’s patches in one shot and does not disturb any other package at all. After that, asking for that package should return a count of 0, and it should no longer show up in patched_packages.

For apply depth, the precedence needs to be strict and predictable. First, an explicit depth on the patch record wins. If that’s not set, we look for a per-package override in project config under 'package-depths'. If that’s not there, we fall back to the built-in defaults for well-known packages, and drupal/core has a built-in default of 2. If the package is not in that built-in list, its default depth lookup returns null. And if none of the earlier options match, we use the global fallback default of 1. In short, it’s patch-explicit depth > project config 'package-depths' override > built-in well-known package defaults (e.g. drupal/core=2) > global default of 1.
