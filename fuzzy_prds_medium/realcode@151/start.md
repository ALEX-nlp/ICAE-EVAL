## Product Requirement Document

Hey team, we need to get the terminal log colorizer shipped for the browser-based job view. Basically right now when you look at build logs in the UI you either see a wall of garbage escape characters or everything is plain unformatted text, and users are complaining it's unreadable especially for error output from tools like lager or webpack that do heavy terminal styling. We need something that takes that raw console output and turns it into proper HTML so it looks right in the browser. There are a few tricky bits — some of our internal log framework embeds special metadata payloads inside concealed text regions (you know, similar to how we handled that marker injection thing in the annotation layer before), and those absolutely cannot get stripped out even though normal hidden text should be. Also we have a couple of named color themes that need to work differently — one of them should wrap the whole output in a container with default colors, and colors that aren't covered by a named theme should fall back to indexed or rgb values. The action marker insertion feature also needs to spit out a specific output format with a normalized placeholder for the action id. Ping me if anything is unclear but let's try to get the core rendering logic separated from the I/O stuff this time, we got burned last time by having everything tangled together.

Quick follow-up from the questions that came in: for concealed regions, the normal rule still applies that ESC[8m...ESC[0m gets suppressed completely, except for the ConsoleNote passthrough case. If the concealed payload starts with the prefix 'ha:' like '\u001b[8mha:CONSOLENOTE\u001b[0m', we need to emit that whole thing verbatim, including the surrounding ESC[8m and ESC[0m escape sequences, and not strip it. Same idea for the action marker path too, so please keep that behavior lined up there as well.

Also on the palette behavior, if the selected palette has both 'default_foreground' and 'default_background' set to non-null index values, the entire rendered HTML output needs to be wrapped in a '<div style="background-color: <bg>;color: <fg>;">...</div>' container. The 'vga' palette is the case where this happens: default_foreground=7 (#AAAAAA) and default_background=0 (#000000), so render_html should produce '<div style="background-color: #000000;color: #AAAAAA;">...</div>' around the output. The 'xterm' palette has no defaults because both are 'none', so it does not wrap. The 'css' palette also has no defaults. That wrapper is for the whole output, not just a span-level color change.

A few rendering specifics too: extended color sequences should support ESC[38;5;<n>m, ESC[48;5;<n>m, ESC[38;2;<r>;<g>;<b>m, and ESC[48;2;<r>;<g>;<b>m. For 256-color values, indexes 0-15 map to the standard palette colors, indexes 16-231 are a 6x6x6 color cube, and indexes 232-255 are a grayscale ramp. Truecolor should render directly as hex '#RRGGBB' regardless of palette. For those extended colors, indexes 16-255 and truecolor are palette-independent.

On inverse video, when ESC[7m is active and there is no explicit foreground set, the foreground should render as '#FFFFFF' and the background should render as 'currentColor'. If explicit fg/bg colors are set, inverse swaps them, so the original fg becomes the background-color and the original bg becomes the color. When ESC[27m turns inverse off, colors go back to their pre-inversion logical values. A full reset with ESC[0m or ESC[m clears inverse mode and all colors.

For faint/dim, SGR code 2 should render as '<span style="font-weight: lighter;">...</span>'. It is separate from bold, where SGR 1 renders as '<b>'. If faint is combined with a foreground color, nest it like '<span style="color: ...;"><span style="font-weight: lighter;">text</span></span>'.

One other parser cleanup item: character-set designation sequences that are ESC followed by '(' and a single character, for example '\u001b(0', are non-rendering control sequences and should be completely filtered out. Visible characters around them should stay exactly as they were.
