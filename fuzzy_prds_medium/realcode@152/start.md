## Product Requirement Document

Hey team, we need a Python library that handles URL/resource reference stuff in a structured way. Basically devs are constantly writing the same brittle string-splitting code to pull apart URLs and it keeps breaking in weird edge cases — especially around passwords leaking into logs, internationalized hostnames, and percent-encoded stuff not being handled consistently. We want something clean that parses a URL into its pieces, lets you navigate around (like going up a directory, resolving relative links), manipulate query params without losing ordering or those weird valueless flag params, and convert between the human-readable Unicode form and the ASCII-safe form. There should also be a way to register custom schemes with their own default ports and authority rules, similar to how we handled custom protocol metadata in that config-registry module from the auth service refactor.

Also important: when something is broken/malformed, it should NOT bubble up a raw Python exception with internal stack details — it needs to output a friendly category name so our API consumers aren't seeing implementation noise. Same deal for the password redaction thing; make sure that's handled consistently across all the output paths. We also want structural equality comparison for URLs so caching layers can deduplicate properly. The output format for all of these operations needs to be consistent and machine-readable. Please align with however we did the encoded vs decoded representation split last time.

One small follow-up from the questions that came in: for text_without_password, if the userinfo has a colon in it like user:pass, we keep the username and the colon and blank out everything after it, so you get 'user:@host'. If there isn’t actually a password component, then text_without_password should match text exactly. That same behavior needs to stay consistent everywhere we render it, so encoded, decoded, and facade should all do it the same way.

Also, the userinfo formatting is intentionally different depending on mode, so please don’t collapse those together. In encoded mode it should render as one quoted string, exactly like userinfo="user:pass". In decoded mode, if there is an authority, it should be a JSON array of decoded pieces, like userinfo=["user","password"]. And if there is no authority but the reference is decoded, we still print userinfo=[""] exactly. That distinction is important.

On scheme registration, there are a couple invalid combinations we do want treated as invalid: if uses_authority=false and a default_port is provided, that’s not allowed, and if uses_authority is null/not a boolean, that’s also not allowed since it needs to be explicitly true or false. On the valid side, if uses_authority=true and there is a default_port, that default should be used as the effective port value and also be suppressed from the rendered text when it matches. This is the same scheme registry abstraction used by the register_scheme_and_render command in rcb_tests/public_test_cases/feature5_1_register_authority_schemes.json and feature5_2_register_path_only_schemes.json, with uses_authority and default_port metadata.

A couple output details too: rooted_path=true when the path starts with '/', and rooted_path=false when it’s a relative path that doesn’t start with '/'. If there’s an authority component, the path is always rooted, so that should come through as rooted_path=true. For ports, when there’s no explicit port, we still derive the effective value from the scheme for normal URL cases, so http=80 and https=443, printed as port=80 or port=443. For custom registered schemes, same idea with the registered default port. The one special case is decoded mode with no authority, where it should print port=null.

For the facade behavior, fragment_value=<value> only shows up if read_fragment=true was explicitly passed in. If read_fragment is missing or false, leave fragment_value out entirely. Same idea with query_values=[...]: only print that when a query_key is provided. The fragment field in the main component block still always prints regardless, so this is just about the extra facade lines.

And for equality/comparison, the rule is normalized structural identity. Two references are equal when their parsed and normalized components match exactly. The output for that should include equal=true/false, distinct_hash_count, and first_text/second_text with each URL’s rendered form. distinct_hash_count is just the number of unique hash values you’d get from a set containing both references, so it’s 1 if they’re equal and 2 if they’re distinct.
