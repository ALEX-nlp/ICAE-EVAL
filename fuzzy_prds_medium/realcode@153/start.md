## Product Requirement Document

We need the core logic for a realtime messaging server that handles channel subscriptions, event broadcasting, and live presence tracking. The server accepts client connections, each with a stable socket identifier, and lets them subscribe to named channels. Channel types are determined purely by name prefix — open channels have no prefix, restricted channels start with a specific private prefix, and presence channels start with a different prefix. For restricted and presence channels, subscribe requests must include a signed token to prove the server authorized the subscription — the signing uses the application's shared secret with a standard message authentication scheme. Presence channels additionally carry member metadata. Broadcasts deliver a payload to all subscribers or to all except one (for echo suppression). When a client disconnects, it must be cleaned up from every channel; empty channels are discarded. Client-originated events (recognized by name prefix) are only relayed when the app configuration explicitly permits it — think of it the same way our webhook-relay module gates untrusted traffic. A keep-alive ping/pong handshake must be supported. A signed HTTP-style inspection API provides three endpoints: list occupied channels, show one channel's subscriber count, and list presence members. Errors from the inspection API report a status code and message. The adapter reads one JSON command from stdin and outputs a line-based transcript.

One extra bit on auth since the team asked for the exact shape: the token comes through as '<key>:<hmac>'. For private channels, the signature is HMAC-SHA256(secret, '<socket_id>:<channel>') in lowercase hex. For presence channels, it is HMAC-SHA256(secret, '<socket_id>:<channel>:<channel_data>') where channel_data is the exact member JSON string. The default app values here are id=1234, key='TestKey', secret='TestSecret'.

Also, the subscribe ack should be the compact JSON form {"event":"pusher_internal:subscription_succeeded","channel":"<name>"}. Presence channels still use that same event, they just also include a 'data' field with hash and member list.

And to be explicit about cleanup behavior: on disconnect, meaning the close event, that client gets removed from ALL channels it subscribed to. If that leaves a channel with zero remaining subscribers, we discard it entirely and stop tracking it, so inspection should report it as '[channel:<name>] absent'.
