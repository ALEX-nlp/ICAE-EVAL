## Product Requirement Document

Hey team, we need a structured logging toolkit built in Go. The core idea is that log events are just flat lists of alternating keys and values, and we want to be able to compose loggers that remember context, tag things by severity, swap out where logs go at runtime, and handle values that need to be freshly computed each time a log line is written. We also need to handle legacy log lines coming in from older parts of the system — kind of like that stdlib adapter pattern we used in the old observability work — and convert them into the same structured format. There should be two output formats: one human-readable and one machine-parseable. The whole thing needs to be split across multiple files, not just dumped into one giant file. There's also a runner/adapter layer that takes JSON commands over stdin and exercises the library, which is how the test harness talks to it. A few things I'm not 100% sure about: how exactly the timestamp parsing works for those legacy lines, what happens when the sink hasn't been set yet, how severity labels are named by default, and the exact ordering behavior when you combine context layers with the JSON format. Can someone clarify those details before we start? The error representation in the wire format is also a bit fuzzy to me.

Quick follow-up after the team’s questions: for severity, the default level key is the string 'level', and the default values are DEBUG, INFO, and ERROR. Those get added as a context pair ahead of the event’s own key/value pairs, so that part should be treated as built-in context rather than something appended at the very end.

Also, on the swappable logger behavior, before any sink is installed, and after the sink is explicitly cleared back to null, it should silently discard all events and return no error. So if anything gets emitted while it’s in that state, it contributes nothing to the output.

For JSON ordering, keys are always emitted in lexicographically sorted order regardless of the order they were supplied. That applies to the fully merged set, meaning all context layers plus the per-event pairs together, not each layer independently.

On the wire format question for errors, an error value comes through in the JSON test input as an object with a single key '$err' whose value is the error message string, for example {"$err": "some message"}. Both encoders should render that as just the plain message text, not a structural dump. In the prefix encoder it shows up as key=message, and in the JSON encoder it shows up as a plain string value.

And for the old log-line adapter, the parsing is a little more specific than we first wrote down. It parses up to three fields from the front of a raw line: an optional date in YYYY/MM/DD form and an optional time in HH:MM:SS form, with optional fractional seconds like .123123, and those two get joined with a single space into one 'ts' field. An optional file reference in path:line form goes into 'file'. Whatever text remains after an optional ': ' separator becomes 'msg'. If any of those pieces aren’t present, they’re omitted entirely. The fixed emission order there is: ts, file, msg. That adapter always uses the prefix encoder for its output. Also confirming Feature 6 is the Standard Log-Line Adapter that parses raw stdlib-style log lines with optional YYYY/MM/DD date, HH:MM:SS time, path:line file reference into structured ts/file/msg key=value pairs, and it should live in a dedicated file separate from the encoders.
