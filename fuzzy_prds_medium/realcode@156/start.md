## Product Requirement Document

We need a small, reusable state container component that holds a single value and lets multiple parts of our application watch that value change over time. Think of it like a reactive data store — any number of listeners can subscribe to receive updates, and when they first subscribe they should immediately get the current value without having to wait for the next change. The system should support reading the current value at any time without subscribing, updating the value through domain operations (both incrementing and decrementing), and cleanly shutting down the container when it is no longer needed. Once the container is shut down, any further attempts to update it should be quietly ignored. The output format for the test adapter should follow the same conventions we use in the notification module — inline key-value pairs on separate lines, with subscriber summaries printed in the order they were attached. An important nuance here is that value delivery to new subscribers happens asynchronously — similar to how our event queue is handled in the session module — so if multiple updates happen right after a subscription in the same synchronous operation window, the subscriber should only see the final settled value, not every intermediate one. Make sure the implementation is clean and decoupled from the input/output wiring.

One small thing the team asked about: if the 'initial' field is absent from the JSON input, we should just start at integer value 0. That is not an error case, and the adapter should treat a missing 'initial' exactly like 'initial: 0' was provided.

Also, this container should behave as a broadcast stream source. The 'isBroadcast' property needs to return exactly 'true', and when the 'broadcast' op runs the adapter should print 'broadcast=true', not 'broadcast=false'. Multiple subscribers can be attached at the same time and they should not interfere with each other.

On the adapter output, please stick to the same inline key=value pairs on separate lines approach from the spec, like 'state=0' and 'broadcast=true'. Subscriber lines need to be printed in the order the subscribers were attached, and each one should follow exactly this shape: 'subscriber=<id> values=[<comma-separated integers with no spaces>] done=<true|false>\n'. Example: 'subscriber=a values=[0,1] done=true\n'. Values are comma-separated with no spaces inside the brackets, and done must be lowercase 'true' or 'false'.

One more nuance on delivery timing since that came up: subscriber delivery happens asynchronously after the current synchronous execution turn. So if a subscribe is followed immediately by synchronous increase/decrease calls in that same turn before delivery actually starts, the subscriber should only get the final settled value and not the intermediate ones. For example: subscribe then increase synchronously results in the subscriber receiving value 1 (not 0 then 1). After all steps, two delayed zero-duration futures are awaited to let deliveries settle.
