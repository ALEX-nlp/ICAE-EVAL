## Product Requirement Document

We need a small utility that helps our configuration-driven framework decide where to look for optional plug-in components. Right now the framework doesn't have a clean way to express which component-discovery strategy should be used, and different host environments have different capabilities — some expose the running process's loaded libraries while others do not. We want a closed, enumerated set of named discovery strategies so that application authors can declare their preferred approach in a config file without writing any custom code. There should be exactly two strategies: the default one that trusts whatever is already loaded in the running process (this is what we should use unless told otherwise), and an alternative that always walks the working directory for library files on disk — handle the fallback the same way as the assembly-loader module does today. Each strategy must have a stable name, a stable numeric position, and we must be able to identify which one is the default. We also need a small command-line resolver: you pass in a strategy name (or its numeric position), and it prints back the canonical name, the numeric position, and whether it is the default. The output format must be exact key=value lines, one per attribute, with no extra whitespace or metadata.

A couple small clarifications from the questions that came back: the resolver output needs to be exactly three lines, in this order: 'name=<canonical identifier>\nvalue=<ordinal as decimal integer>\ndefault=<true or false>\n'. Each line is plain key=value with no spaces around the equals sign, and there does need to be a trailing newline after the last line too.

Also, the only two canonical identifiers are exactly 'UseLoadedAssemblies' and 'AlwaysScanDllFiles', and those names need to stay stable and match exactly in output. 'UseLoadedAssemblies' is ordinal 0 and default=true. 'AlwaysScanDllFiles' is ordinal 1 and default=false. And yes, choosing the mode by its ordinal string should behave the same as choosing it by name, so '"0"' should resolve exactly like 'UseLoadedAssemblies' and '"1"' should resolve exactly like 'AlwaysScanDllFiles'.
