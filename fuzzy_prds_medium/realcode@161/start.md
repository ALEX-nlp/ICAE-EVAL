## Product Requirement Document

We need a two-factor authentication engine that a PHP application can use to add a time-based one-time password second factor. The core capability is taking a shared secret and a point in time and generating a short numeric code; the server and the user's authenticator app must agree on the same code at the same moment. The engine should support verifying a user-submitted code with a configurable tolerance window to absorb small clock differences between the user's device and the server — importantly, the tolerance window only looks backward in time, not forward. Once a code has been accepted, it must be remembered so a second attempt with the same code at the same moment is refused even though the code is still time-valid. The engine should also produce the standard provisioning URI that authenticator apps import by scanning a QR code, and it should be able to present the raw secret as well as a grouped version with spaces every four characters for manual entry. Finally, there should be a recovery-code system: generate a batch of single-use alphanumeric codes of a configurable length, and allow spending them one at a time (a code that is already spent or does not exist reports failure rather than throwing an error). All outputs must be predictable machine-readable strings, not localized messages or exception dumps.

A couple follow-ups from the questions that came in. On the TOTP generation side, the time-step counter = floor(timestamp / seconds) + offset. The HMAC is computed over this counter encoded as an 8-byte big-endian unsigned integer using the Base32-decoded secret as the key. Dynamic truncation extracts a 31-bit integer; it is reduced mod 10^digits and left-zero-padded to exactly 'digits' characters. The output is 'code=<digits-char-string>'.

Also to be extra clear on validation behavior, the window still only looks backward. A code is accepted if it matches the code for the step containing 'at', or any of the 'window' steps immediately preceding it. A code from a future step (offset +1 or later from the current step) is never accepted. Window=0 means only the exact current step.

For the provisioning link, this is the 'otpauth://totp/...' URI format defined by Google Authenticator. The provisioning URI format is: otpauth://totp/<percent-encoded-label>?issuer=<issuer>&label=<percent-encoded-label>&secret=<secret>&algorithm=<UPPER-ALGORITHM>&digits=<digits>. The issuer is the portion of the label before the first colon. Query parameter order is: issuer, label, secret, algorithm, digits. The label path segment and query values are percent-encoded per RFC 3986. The algorithm name is uppercased (e.g., 'sha256' becomes 'SHA256' in the URI). ':' encodes to '%3A', '@' encodes to '%40'.
