## Product Requirement Document

Hey team, we need a small encryption utility — basically something that wraps up sensitive short strings (tokens, passwords, that sort of thing) so we can store or pass them around safely without developers having to deal with all the low-level crypto plumbing themselves. Think of it like that token-vault thing we scoped out a while back but much simpler — just two operations, lock it up and unlock it.

The big thing here is that we want to support two modes of operation. In the first mode everyone shares the same underlying secret, so any caller can read any other caller's data. In the second mode each 'tenant' or caller identity gets its own cryptographic isolation — if you try to read someone else's locked data with the wrong identity it should just fail cleanly, no messy runtime crashes or stack traces leaking out, just a simple machine-readable error signal.

The output of locking something needs to be a plain text string that's safe to put in a database column or JSON field. On the ops side we're worried about two things: first, that the format stays stable so something encrypted today can still be decrypted six months from now; second, that the error messages we surface are neutral and don't expose internals. The adapter that drives this from our test harness reads a JSON blob from stdin and writes results to stdout. Let me know if you need more context on the tenant isolation piece specifically.

One quick follow-up from the questions that came in: the envelope format is fixed and really minimal. It is exactly two base64-encoded segments joined by a single colon character: `<base64(iv)>:<base64(ciphertext)>`. There is no other delimiter, no version prefix, and no additional segments. Splitting on `:` must yield exactly 2 parts; anything else is an invalid envelope. Also, when that first segment is base64-decoded, it must be exactly 16 bytes.

On the crypto details, this is the Symmetric Envelope Encryption Service described in start.md, specifically the encrypt/decrypt operations producing a `<base64 iv>:<base64 ciphertext>` envelope with AES-256-CBC and PKCS7 padding. The initialization vector is always exactly 16 bytes, generated fresh and randomly for every single encryption call. It is never reused or derived. The key must be 32 bytes (256 bits). Plaintext is UTF-8 encoded before encryption, and the ciphertext length is the plaintext length rounded up to the next full 16-byte block boundary.

For the shared secret coming in from the adapter, the master/shared secret key is supplied as a standard base64-encoded string in the `key` field in the JSON request. It is base64-decoded to raw bytes before use, and the raw bytes must be exactly 32 bytes to satisfy AES-256 requirements.

For the tenant-isolated path, the key derivation approach is HKDF-SHA256 using the master key bytes and the UTF-8 context string as info, extracting 32 bytes. More explicitly, in derived-key mode, the per-context encryption key is produced by running HKDF-SHA256 over the master key bytes with the UTF-8 bytes of the context identifier as the info/salt material, extracting exactly 32 bytes. The same master key plus the same context string always yields the same derived key; different context strings yield cryptographically distinct keys.
