## Product Requirement Document

Hey team, spinning up a ticket for the manifest generation work we discussed last sprint. Basically we need to build out the config-to-manifest pipeline for our k8s deployments. The core ask is that devs can pass in a structured config blob and get back deterministic summaries of what would be deployed — stuff like the workload spec, networking, scaling, storage, identity, monitoring hooks, and any custom extensions we bolt on.

The big pain point right now is that every team hand-rolls their own YAML and we keep seeing drift — missing labels, wrong selectors, incomplete port bindings, that kind of thing. We want one canonical way to do this.

A few things that need to work: required fields should blow up loudly when they're absent (similar to how we handled the null-value edge cases in the auth service config loader — you know the one), optional stuff should just silently fall back to defaults. Also the rollout strategy should only kick in when it's explicitly turned on, not default to something that could cause surprise restarts in prod.

We also need the CSI secret volume wiring and the managed cert stuff for GCP environments. The monitoring endpoint config should follow the same pattern we used for the service monitor setup in the platform team's last release.

Output needs to be line-oriented key=value so the deployment pipeline can parse it without a full YAML parser. Happy to clarify anything but let's get this scoped out.

One follow-up from the questions that came in: if a mandatory field like containerImage.repository is explicitly set to null, the output must be exactly two lines: 'error=missing_required_value' followed by 'field=<field.name>', both newline-terminated. Same deal if the JSON adapter uses the 'set' map and containerImage.repository is set to the string 'null' — that still counts as an explicit null/missing value, not the literal string 'null', so it should produce error=missing_required_value and field=containerImage.repository. On the flip side, optional values can still be omitted or set to null without causing an error, including things like image pull policy.

Also, just to be more concrete on the summaries, everything stays line-oriented key=value. For deployment_core, the output reports: kind, name, replicas, container_name, image, command (empty string if none), args (empty string if none), host_aliases count, and then each container port as port_N_name, port_N_number, port_N_protocol. For service, it reports kind=Service, name, type (e.g. ClusterIP), cluster_ip (null if not set), session_affinity (null if not set), external_traffic_policy (null if not set), internal_traffic_policy (null if not set), port_count, and each port as port_N_name, port_N_port, port_N_target. For ingress, it reports kind=Ingress, name, rule_count, tls_count, and for each rule: host, path_count, and per-path entries with path string, backend service name, port_number (integer or null), and port_name (string or null). Numeric ports populate port_number and set port_name=null; named ports do the reverse.

A couple other specific outputs to lock in: managed_certificate reports kind=ManagedCertificate, name (from google.managedCertificate.name), and domains (comma-separated domain names from google.managedCertificate.domainName), and the template key is 'gmc'. hpa reports kind=HorizontalPodAutoscaler, api_version=autoscaling/v2, min_replicas, max_replicas, metric_count, and for each metric: metric_N_resource (cpu or memory) and metric_N_average_utilization as an integer. pdb reports kind=PodDisruptionBudget, name, and min_available, and it should only render when minPodsAvailable is greater than zero.

For monitoring and extensions, service_monitor reports kind=ServiceMonitor, name, endpoint_count, and for each endpoint: interval, timeout, path, port, and scheme. Configuration is typically supplied via a values file, and for Feature 16 service_monitor rendering, configuration is loaded from test/fixtures/service_monitor_values.yaml and the output is a line-oriented summary of kind, name, endpoint_count, and per-endpoint interval/timeout/path/port/scheme fields. custom_resources reports manifest_count and for each manifest: manifest_N_kind and manifest_N_name, with the custom resource data also coming from a values file and rendering alongside the generated resources.

Last small implementation detail since this came up too: the 'set' map in the JSON adapter input accepts dot-notation keys (e.g. 'containerImage.repository') and merges them into the base configuration before rendering. And yes, keeping this in plain key=value output is still the requirement end to end.
