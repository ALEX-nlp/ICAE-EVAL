## Product Requirement Document

Hey team, we need to build out that actor concurrency library we've been talking about. The core idea is that developers should be able to spin up isolated 'systems', put named workers inside them, and have those workers talk to each other via messages without worrying about the plumbing underneath. Think of it like the pattern we used in the notification service refactor last quarter — same isolation idea, but generalized.

A few things I know are important from the stakeholder side: naming has to be strict (we had bugs before from weird characters sneaking into identifiers), parents should know about their kids but not their grandkids, and if something crashes it shouldn't take down the whole world — there should be ways to configure recovery. Also someone mentioned we need a way to 'ask' an actor something and wait for an answer, but it can't wait forever, and the error messages that come back need to be clean, nothing internal leaking out.

We also need lifecycle stuff — actors should be stoppable, stopping a parent should clean up everything below it, and there's some ask about actors that shut themselves down if they've been sitting idle too long. The after-stop situation also needs a callback of some kind so you can inspect what was still pending.

Please make sure the code isn't one giant file — it got messy last time. Reference the structure notes in start.md for how to split things up.

One quick follow-up from the questions that came in. On naming, we do need to be really strict here. A valid actor address segment must be a non-empty string composed solely of letters, digits, and the following URI-safe extra characters: hyphen (-), underscore (_), period (.), dollar sign ($), plus (+), exclamation mark (!), asterisk (*), single quote ('), open parenthesis ((), close parenthesis ()), and comma (,). Any whitespace (leading, trailing, or interior), path separators, reserved query/fragment characters (e.g. ?, #, &, =), and escape characters (e.g. %) make a name invalid. Non-text values and empty strings are also rejected. This needs to be enforced the same way anywhere we build addresses and anywhere we do spawn or routing, same as the Feature 1 segment validation rules in start.md, and Feature 3 (Actor System Identity and Isolation) in start.md still stands too.

Also, on system boundaries, each actor system owns its own isolated namespace. Two systems may each contain an actor with the same local name without any collision. A request-response interaction addressed within one system resolves only against that system's registry and never crosses into another system's namespace. That’s the same isolation expectation we called out before, just spelling it out more directly.

For ask/query behavior, if a query is issued without supplying a timeout value, the system must output the normalized error category 'error=missing_timeout' on its own line. No runtime exception class name, stack trace, or internal message suffix may appear in the output. And if a query times out, the output must be two lines: 'error=query_timeout' followed by 'timeout_ms=<value>' where <value> is the timeout that was supplied. Same deal there: no exception class names or stack traces should leak out.

A couple more runtime details to lock in: mailbox ordering is strictly preserved, so one message is fully processed (including any asynchronous work within its handler) before the next message begins processing. A message delayed by async work inside its handler cannot be overtaken by a later message. The listener's final recorded sequence must always equal the original send order. Also, an actor's initial state can be computed from a template string containing the placeholder '{name}'. At startup, before any message is processed, the runtime replaces '{name}' with the actor's configured name to produce the seed state. Subsequent messages extend from that computed seed.

On failure behavior, the defaults depend on actor kind when no explicit supervision policy is configured. A stateful actor stops automatically after a handler failure, while a stateless actor resumes (continues processing) after a handler failure. Initialization failures can be observed via a crash handler that stops the actor.

And just reinforcing the implementation shape because this came up again: a single-file solution is explicitly prohibited. The codebase must use a multi-file directory structure (e.g. src/, tests/) that separates addressing/validation, mailbox scheduling, lifecycle, supervision, and the JSON I/O adapter into distinct logical modules. The core business logic must be fully decoupled from stdin/stdout and JSON parsing.
