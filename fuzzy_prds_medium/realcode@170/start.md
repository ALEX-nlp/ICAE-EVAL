## Product Requirement Document

Hey team, we need to build out the core backend for our new analytics data collection pipeline. Basically, we're getting hammered with support tickets because our current setup loses visitor identity between page loads, and the ops team can't configure anything without a code deploy. Super frustrating.

The big ask: we need something that tracks visitors with stable IDs that survive cookie round-trips, reads config values safely without blowing up when something's missing (you know how ops always forgets to set every field), maps incoming web hits into clean typed records using a config file instead of hardcoded logic, and writes everything out to files in a way that downstream can trust — i.e., only fully-written files should look 'done'.

For the file writing part, we need two modes: one that just rolls files on a schedule, and one that groups events by 'which time window did this session start in' — similar to how we handled the batching logic in that old session affinity module. If a session started in an earlier window and that file is still open, the event should go there.

One thing I keep forgetting to mention: the config mapping layer needs to loudly reject bad setups early — operators should know immediately if something's misconfigured, not find out hours later from missing data. Please keep all output as plain key=value lines, one per line.

A couple follow-ups from the team’s questions. On the identity piece, the token in Feature 1 is a millisecond timestamp plus random bits encoded into a fixed-width 32-character hex-like opaque string, and it needs to stay exactly 32 characters no matter what the timestamp is. It’s what we use to label visitors, sessions, and page views, and it has to survive serialize→cookie→read-back round trips. Also, if you decode or re-parse a token from its string form, it should be the same token for value equality and same hashcode. But if you independently generate another token for the same timestamp, that one must NOT be equal, since the random bits are different.

On the file side, an in-flight file is not something downstream should treat as real yet. It has to use the '.avro.partial' suffix and show up as '<unreadable>' in the output until it’s finalized. A published file uses the '.avro' suffix and shows its record timestamps as comma-separated values. For the simple/rolling behavior, we write everything into one in-flight file and publish it on clean shutdown. If rolls are forced between batches, each interval gives one published file. The output format here should be exactly plain key=value lines in order: published_files=N, inflight_files=N, then published[i]=.avro|<ts-list> and inflight[i]=.avro.partial|<unreadable>.

For the session-window behavior in Feature 4.2, events are routed to per-session-window files based on when the session started, so the session_start timestamp determines the window. If that window’s file is still open, the event stays with that file. If that window is no longer current, the event goes to the current window’s file. One subtle point here: on shutdown, in-flight binning files are finalized in place but NOT published, so they remain as .avro.partial.

A couple mapper details too. The sessionStart field is a derived boolean for whether this is the first event in the session. It should come out as sessionStart=true or sessionStart=false, and it’s true when the session is new, like when page_view_id matches the session start condition. For regex named-mapping, the value comes from the named capture group in a regex applied to a request signal like location or referer URL. So if the regex runs against 'http://example.com/' and has a named group 'protocol', the populated value would be 'http'. And on the fail-fast validation point, a cookie mapping that leaves out its required cookie name, or an event-parameter mapping that leaves out its required parameter name, needs to be rejected at mapper construction time with a descriptive categorized error.

Last thing, on geo output and formatting. If there’s no geo lookup result at all, all geo fields should be null and rendered as 'null'. If the lookup gives an empty/default result, boolean geo fields should be false and list fields should be empty lists. If there’s a full result, all fields get actual values. All output everywhere stays as plain 'key=value' lines, one per line, in a fixed order. Collections render as '[a, b, c]' with the space after the comma, absent/null values render as 'null', and there shouldn’t be any JSON, headers, or extra whitespace beyond that.
