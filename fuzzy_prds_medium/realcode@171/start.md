## Product Requirement Document

Hey team, we need to get this auth gateway thing wrapped up. Basically it's a reverse proxy that sits in front of our internal apps and handles all the 'who are you and can you come in' stuff uniformly. Right now every team is rolling their own and it's a mess — some check email case-sensitively, some don't sign cookies at all, etc.

The core pieces are: (1) a way to decide if an email address is allowed in, based on domain rules or an explicit list — similar to how we handled the login module last time; (2) checking username/password against a static file using that hashing approach we standardized on; (3) packing session info into a cookie string that can survive a round-trip without leaking tokens; (4) the actual HTTP front door that challenges unauthenticated users and has a couple of fixed public endpoints; and (5) a config validator that catches ALL the problems at once instead of exploding on the first one.

A few things I'm fuzzy on: how exactly the cookie encryption works when someone uses the wrong key to decode — does it blow up or gracefully degrade? Also not sure about the exact rules for when config validation considers the secret 'long enough' for certain features. And what happens with the sign-in page redirect when someone hits the sign-in path directly vs a protected path? Can someone clarify these before we hand this to the dev team?

Quick follow-up after the questions that came in: for the email/domain allow logic, treat it as fully case-insensitive across the board. The configured domains, the allow-list entries, and the queried address all get folded to lower case before any comparison, both for matching the host part against domains and for exact allow-list membership checks. This is the same Feature 1 (Email/Domain Authorization) behavior in start.md. Also, if the configured domain set contains just '*', then every non-empty address is authorized no matter what the host part is and whether it appears in the allow-list or not. But an empty address is never authorized, even with '*'. And if both the domain set and the allow-list are empty, then nothing is authorized at all and every queried address comes back 'denied'. Those empty-address and empty-config outcomes are absolute.

On config validation, yes, we want ALL problems back in one shot, not fail-fast. The four required settings are at least one upstream target, a cookie-secret, a client-id, and a client-secret, and each missing one should show up as its own 'missing_setting' error with the setting name. Same idea for skip_auth_regex: every bad pattern gets its own 'invalid_regex' error, and everything is reported together. For upstreams, parse the URL into scheme, host, path, and if the path is missing normalize it to '/'. In the validated output, report each upstream as 'scheme|host|path'. If there’s an optional redirect URL, report that the same way as 'scheme|host|path'.

Also on the secret-length question: if token pass-through or cookie refresh is turned on, the cookie secret has to be one of the AES key sizes, so 16, 24, or 32 bytes. A random short secret is still fine for basic operation, so something like 'foobar' which is 6 bytes is acceptable unless one of those features is enabled. And cookie refresh has one more rule: the refresh interval must be strictly less than the expiry interval.

For the static credentials piece, each entry is 'username:{SHA}' followed immediately by the Base64 encoding of the SHA-1 digest of the password. A login only passes if the username exists in the store and the Base64-encoded SHA-1 digest of the supplied password exactly matches the stored digest. If the stored secret does not start with '{SHA}', it should never be accepted. That’s the same '{SHA}' prefix marker scheme called out in Feature 2 (Static Credential Verification), and the stored format is '{SHA}' followed by the Base64-encoded SHA-1 digest of the password.
