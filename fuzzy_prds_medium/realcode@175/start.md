## Product Requirement Document

Hey team, we need a maps API wrapper library that handles building request URLs and parsing responses. The idea is developers shouldn't have to manually stitch together query strings or dig through nested JSON — the library takes structured inputs and spits out the right URLs or parsed summaries. We support a bunch of different service types: routing between two points, matrix distances, address lookups, place searches (nearby, text-based, details, photos, autocomplete variants), plus the timezone and geolocation ones which behave a bit differently from the rest (remember how we handled the alternate host situation in that geolocation spike last quarter — same idea here). For place photos specifically there's a size constraint thing we discussed — refer back to how we handled the 'at least one required field' pattern from the nearby search ranking validation. Coordinate pairs should always come out as lat,lng without any encoding. Address strings on the other hand do need proper percent-encoding. For the parsing side, when we get back raw JSON from any of these services we want a flat summary output — status, key counts, first meaningful fields — not a full object model. Also there's a shared time value thing for opening hours that needs to handle day indexing and reject malformed time strings with a neutral error. Calendar time format is exactly 4 digits or it should fail. Make sure the multi-origin/destination matrix endpoint joins entries correctly and that the transit mode list preserves order with pipe separators.

Quick follow-up from the questions that came in: the geolocation one really does live on its own host and should be treated differently from everything else. The URL there is https://www.googleapis.com/geolocation/v1/geolocate?key=<API_KEY>, not anything on maps.googleapis.com. By contrast, all the other endpoints we’re wrapping here — directions, distance_matrix, geocoding, places_*, timezone — stay under https://maps.googleapis.com/maps/api/ . Same separation as before, just calling it out more explicitly so nobody accidentally routes geolocation through the common base.

A couple extra specifics on URL building too. For timezone, the endpoint is https://maps.googleapis.com/maps/api/timezone/json and it requires location and timestamp, with location rendered as lat,lng and timestamp as integer seconds. Language is optional, and the parameter order should be location, timestamp, language (if present), key. On coordinate formatting generally, lat/lng objects should always come through as bare lat,lng strings like -33.8670522,151.1957362 with NO percent-encoding applied. Only free-text address strings get percent-encoded, so spaces as %20, commas as %2C, etc. For distance_matrix specifically, if there are multiple origins or destinations, join them with |, and make sure each address string is percent-encoded on its own before joining. Coordinate entries there still stay as lat,lng with no encoding.

Also tightening up the place photo behavior since that got asked twice. The path is /maps/api/place/photo with no .json extension, and the query names are not the same as the input names: photo_reference → photoreference, max_height → maxheight, max_width → maxwidth. And if neither max_height nor max_width is provided, the output must be exactly: error=invalid_request\nrule=photo_size_required\n. That’s the same basic pattern as the places_nearby rankby=distance validation we already talked about: if the required piece isn’t there, return error=invalid_request plus the named rule.
