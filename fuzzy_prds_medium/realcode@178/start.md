## Product Requirement Document

We need a web request handling toolkit that lets developers build servers by composing small, reusable processing units rather than writing low-level plumbing. Both incoming requests and outgoing replies are treated as immutable data objects you can inspect and transform at any point. Each object carries header information accessible in two ways: a convenient single-value view and a full multi-value view, both sorted alphabetically when output. The system automatically determines body size from actual content, not from any manually set header. When a developer specifies a character set for a body, the content-type field is automatically rewritten to match. Bodies can only be consumed once — a second attempt triggers a well-defined error signal rather than failing silently. Incoming addresses are split into the portion already handled versus the portion yet to be routed, enabling clean nested routing without path arithmetic. Outgoing status codes come with convenience constructors for the most common outcomes — success, redirect, not-found, server error, and cache-unchanged — each populating the right default body and location information automatically. Date-related fields like freshness expiry and last-modification time are surfaced as structured timestamps. Handlers can be arranged in a fallback chain where the platform tries each one in order and skips unacceptable results. Cross-cutting concerns are layered via a wrapping mechanism where outer layers see the request first and the response last. Chunked framing is applied automatically to streaming responses that lack a declared size.

A couple extra details from the follow-up questions. If you construct a message with no body and no explicit headers, it should still report a length of 0, and the output should come through exactly as 'headers[content-length]=0' and 'headersAll[content-length]=["0"]'. More generally, headers are always emitted in ascending alphabetical (sorted) order in both views, so that applies to both headers[...] and headersAll[...]. The two header surfaces are still the same ones already described: the single-value view is singleValues (Map<String, String>) and the multi-value side comes from the base UnmodifiableMapView<String, List<String>>, exposed on Message as headers and headersAll respectively.

On routing, for a request to 'http://localhost/foo/bar', handlerPath defaults to '/' and url defaults to 'foo/bar' with no leading slash and no trailing slash. Request.handlerPath is the root-relative path to current handler and always ends with '/', and Request.url is the remaining path relative to handler plus query string. One small but important rule there: handlerPath always ends with '/' when url's path is non-empty.

A few response constructor specifics too. Response.internalServerError() with no body supplied should default body to 'Internal Server Error' and set content-type to 'text/plain', with output exactly 'status=500', 'contentType=text/plain', 'location=null', 'hasDate=false', 'body=Internal Server Error'. Response.notModified() always adds a 'date' header with the current time, so you'll see 'hasDate=true' in the output; status is 304 and body is empty. Response.found(location) sets status=302, 'location=<value>', 'contentType=null', 'hasDate=false', 'body=' and the location is taken from the argument exactly as-is.

For the date-shaped fields, parsed header values should render in ISO-8601 UTC form with milliseconds, for example '1994-11-06T08:49:37.000Z'. If the value isn't there, the output should be the literal string 'null'.

On cascading behavior, the default cascade skips handlers that return status 404 or 405 and immediately returns any other status. If all handlers return unacceptable statuses, it returns the last handler's response. Also, calling '.handler' on a Cascade with no inner handlers throws a StateError that shows up as 'error=invalid_state' with 'message=Can\'t get a handler for a cascade with no inner handlers.'. And if someone passes both statusCodes and shouldCascade to Cascade(), that should throw ArgumentError with 'error=invalid_argument' and 'message=statusCodes and shouldCascade may not both be passed.'.

For wrapping order, request phases run outermost-first and response phases run innermost-first. With two pass-through middlewares A and B around a trace handler, the trace should be exactly 'request:A', 'request:B', 'handler', 'response:B', 'response:A'. If a middleware returns a response during its request phase, the inner handler is not called and the response phases of all middlewares are skipped, so only the request trace line is emitted before the response lines.

And just to pin down the one-shot body behavior more concretely, Body.read() sets _stream = null after the first call and then throws StateError with message "The 'read' method can only be called once on a shelf.Request/shelf.Response object." on any second attempt.
