## Product Requirement Document

Hey team, we've been getting complaints from a few dev partners who are trying to use our graph query layer on top of their existing relational tables. The core ask is pretty straightforward — they want to declare a graph once (pointing at their existing tables), run pattern-matching queries against it, and get back normal relational rows they can work with in SQL. Think of it like that 'logical view over tables' approach we discussed last sprint.

The big pain points they're hitting: (1) when they write graph traversal queries today they end up with a mess of recursive joins that break every time the schema changes, (2) they can't express 'find me all paths between these two nodes within N hops' without writing custom BFS code, and (3) there's no way to introspect what the graph looks like programmatically — they have to dig through DDL to figure out what vertex/edge tables exist.

Also, some partners mentioned that error messages are completely opaque — they get a generic crash instead of something actionable. We need errors to be categorized so they can handle them in their client code.

One more thing — there was some discussion about that 'CSR structure' approach we used in the adjacency work a while back, make sure that's still exposed correctly. Check how we handled the offset array stuff previously, the partners are depending on that for their graph analytics pipeline.

Basically: define graph, match patterns, project columns, handle paths, expose schema, give good errors. Keep it clean and modular.

Quick follow-up from the questions that came in: label matching in patterns is case-insensitive. The engine normalizes all labels to lower-case internally, so PERSON, Person, and person all resolve to the same label. The declared label is stored in lower-case and all pattern label tokens are lower-cased before lookup. Related to that, if someone repeats the same label inside one property graph definition, we reject it with error category 'constraint'. Each label must be unique within a single graph definition, and that applies to both vertex labels and edge labels across the entire definition.

On the schema introspection side, DESCRIBE PROPERTY GRAPH returns exactly 11 tab-separated fields per row in this order: (1) backing table name (original casing), (2) normalized label (lower-cased), (3) is_vertex boolean (true/false), (4) source vertex table name or NULL, (5) source referenced key list as [col,...] or NULL, (6) edge source key column list as [col,...] or NULL, (7) destination vertex table name or NULL, (8) destination referenced key list as [col,...] or NULL, (9) edge destination key column list as [col,...] or NULL, (10) discriminator column name or NULL, (11) bracketed ordered sublabel list or NULL. Rows are sorted alphabetically by backing table name.

On errors, we should keep this very predictable. The engine uses exactly six error categories: 'invalid' (bad definition, e.g. non-existent base table), 'parser' (syntactically illegal input, e.g. wildcard in property list during definition), 'binder' (semantic resolution failure at query time, e.g. graph not found, unknown variable), 'constraint' (unsafe operation, e.g. unbounded quantifier under WALK mode), 'catalog' (catalog-level failures), and 'not-implemented' (unsupported features such as ALL SHORTEST or non-WALK path modes). Each error is rendered as two lines: 'error=<category>\nmessage=<text>\n'.

One specific pathing edge case the team asked about: an unbounded quantifier (e.g. -[k:Knows]->* or -{0,}) under the default WALK path semantics is rejected at constraint-check time with error category 'constraint' and message: 'ALL unbounded with path mode WALK is not possible as this could lead to infinite results. Consider specifying an upper bound or path mode other than WALK'. Bounded quantifiers like -{1,3} are supported. Path modes other than WALK and the ALL SHORTEST qualifier are rejected with error category 'not-implemented'.

And for the CSR piece, yes, the same shape still needs to be exposed. The CSR (Compressed Sparse Row) structure is built via CREATE_CSR_VERTEX (which accumulates per-vertex out-degree counts into an offset array) and CREATE_CSR_EDGE (which fills the flattened neighbor destination array). get_csr_v(<graph_id>) returns the offset (prefix-sum) array with one entry per vertex plus a terminating total — so for N vertices it returns N+1 rows. get_csr_e(<graph_id>) returns the flattened destination dense-id array in adjacency order. Vertices are identified by their rowid as dense ids. The CREATE_CSR_VERTEX and CREATE_CSR_EDGE SQL functions plus get_csr_v/get_csr_e table-valued functions described in feature6_2_csr_functions.json; these build a Compressed Sparse Row adjacency representation used internally for path search and exposed for external analytics. Also just to close the loop on the label behavior, the case-insensitive label normalization logic is the same one evidenced by feature3_3_case_insensitive_labels.json and the lower-cased label output in feature1_1_describe_graph.json.
