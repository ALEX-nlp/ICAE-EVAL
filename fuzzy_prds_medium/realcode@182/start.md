## Product Requirement Document

hey team, we need a small permissions helper library — something that wraps the raw OS permission result codes into something more usable. right now devs are scattering booleans like isDenied / neverAsk / shouldExplain all over the place and it's a mess, things keep getting conflated especially the 'denied forever' vs 'just denied for now' distinction. we want a typed model with clean predicates so you can just ask 'is this granted?' or 'does any permission in this set still need asking?' without writing loops every time.

also needs a request lifecycle thing — you should be able to attach listeners to a request and get notified when results come in. there's a specific flavor of listener that should fire exactly once and then clean itself up automatically (similar to how we handled the one-time auth callback in the login module, check that for reference). and there should be a builder to assemble a request — it needs to validate its inputs upfront and give back a clear reason when something's missing rather than blowing up weirdly at runtime. the validation order matters here, make sure you check the right thing first.

code organization matters — don't dump everything in one file but don't over-engineer it either. the JSON test harness is just for the adapter layer, keep the core domain totally separate from that. SOLID principles should be visible in the structure.

one extra pass on the status shape since a couple folks asked: there are exactly four mutually exclusive top-level status categories: 'granted', 'permanently_denied', 'should_show_rationale', and 'request_required'. the two denied sub-kinds, 'permanently_denied' and 'should_show_rationale', both roll up under a broader 'denied' grouping, but denied is not itself a top-level category, just a grouping we expose for convenience. for any single status, exactly one of {granted, denied, request_required} is true, and within denied, exactly one of {permanently_denied, should_show_rationale} is true. each single status should report exactly five independent boolean predicates: 'granted', 'denied', 'permanently_denied', 'should_show_rationale', and 'request_required'. when we print it, the format is the permission identifier first as permission=<id>, then one key=boolean line per predicate in that order.

for list aggregation, we want ten quantifier lines total. start with 'count=<n>', then for each of the five categories in this order — granted, denied, permanently_denied, should_show_rationale, request_required — emit the 'all_*' flag, and after that emit the five 'any_*' flags in that same category order. all_* is true only when every element belongs to that category, and any_* is true when at least one element does.

on the one-time listener behavior, the intended pattern is the OneShotResultCallback thing from Feature 3: a listener that calls remove(self) on the request's listener registry immediately after its first invocation, so it cannot fire again and the listener count drops to zero. more concretely, a one-shot callback registers itself as exactly one listener on the request. when the very first result delivery occurs, the callback fires with that delivery's status list and then immediately removes itself from the listener registry. any subsequent deliveries find zero listeners and the callback never fires again. the output needs to show listeners_after_register=1, then for each delivery a 'delivery[i]_listeners_before' line, then callbacks_fired (always 1 after the first delivery), then the content of each fired callback as 'callback[i]=type:permission,type:permission' (or '<empty>'), then listeners_remaining=0.

and on the builder validation, the order is strict: validate permissions first, then the handler provider. if permissions were never supplied and mode is 'unset', the error is 'missing_permissions' regardless of whether a handler was provided. only if permissions are present but no handler provider is attached does the builder return 'missing_runtime_handler'. this part must not be reversed.
