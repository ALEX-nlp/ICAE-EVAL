## Product Requirement Document

Hey team, we need to build out that reactive stream processing library we talked about in the last planning session. Basically developers are tired of hand-wiring callbacks and managing subscriptions by hand every time they need to react to a sequence of values — whether those values come from a static list, a single item, or something that just keeps pushing data in over time. The core idea is that a stream is a first-class thing you can pass around and compose, not a pile of ad-hoc glue code.

We need the usual source factories (you know, the ones we sketched on the whiteboard — static collections, single values, the empty/silent cases), plus a solid set of operators for transforming, filtering, combining, and windowing. There's also the multicast hub thing we discussed — similar to what the notifications team built for their pub-sub module, where consumers can join and leave independently. And we need that promise-style consumer for one-shot last-value resolution.

One thing that keeps coming up in feedback: people get confused when streams cut off early or when a consumer leaves mid-flight — the behavior needs to be consistent and predictable. Also the delay/timing operator has been requested a few times. We want the whole thing to be composable so you can chain operators freely. Please model the output format after what the infra team standardized for the event log renderer — prefixed lines per consumer, value lines, and completion markers.

Quick follow-up from the questions that came in: the rendered output should stay really literal. Each emitted value is a line in the form `value <v>` where `<v>` is the JSON encoding of the value. Completion is `end`. If a stream never completes, render `no-end`. In the multi-consumer Subject cases, every line gets the consumer id in front, so `<id> value <v>` and `<id> end`. That’s the same output contract defined in start.md, and it applies consistently across the library.

A couple source semantics to make explicit: `fromArray` is a cold/pull source, so it emits each element of the given ordered collection exactly once, in order, to a subscribing consumer, then completes, and it re-runs independently for each new consumer. `fromValue` emits exactly one value and then signals completion, regardless of how many further values the consumer requests. For `flatten`, the outer source is emitting inner sources as its values; we subscribe to each one and merge all inner values into one output stream, and it completes once the outer source and all inner sources have completed.

For the combining behavior, `combine` is pairing the most recent values of two live (hot) sources. A pair is emitted whenever either source emits a new value, but only once both have emitted at least one value. The pair is `[latestA, latestB]`, and the combined stream completes after both sources complete. The test adapter input here uses `kind: 'combine'` with an `emissions` array of `{source: 'a'|'b', value}` entries.

On the cutoff-style operators, the main thing is to keep the early-ending behavior predictable. `take` lets at most `n` values pass and then signals completion and ends the upstream source early. If the source completes before `n` values are emitted, the received values and completion are forwarded as-is. `takeLast` buffers all incoming values and only after the upstream source completes emits the final `n` values in their original order, then signals completion, so nothing is emitted before the source ends. `takeWhile` forwards values as long as the predicate holds; the first value that fails the predicate ends the stream, that failing value is NOT emitted, and the upstream source is ended early. In the test adapter, that predicate accepts values up to and including a configured `whileAtMost` bound. `takeUntil` forwards values from its source until a separate `notifier` source emits its first value, at which point the stream ends immediately before any further source values. If the notifier never emits, all values and the source's own completion are forwarded. The `notifier` field in the test adapter is a source descriptor.

Same idea for the skip family: `skip` discards the first `n` values and then forwards every subsequent value, along with the completion signal. `skipWhile` discards leading values as long as the predicate holds, and once the predicate first fails, that value and all subsequent values are forwarded, along with completion. The test adapter predicate is true for values up to and including a configured `whileAtMost` bound. `skipUntil` discards all source values until a separate `notifier` source emits its first value; after that, every subsequent value is forwarded along with completion. If the notifier never emits, all values are discarded and only the source's completion is forwarded.

Also spelling out `switchMap`: each value from the source is mapped to a new inner source and we emit the values of the most recently created inner source, abandoning any previous inner source when a new outer value arrives. In the test adapter, each value `v` is mapped to a `fromValue(v * v)` inner source. The combined stream completes once the outer source and the final inner source have completed.

And on the hub piece, the Subject implementation is a multicast hub that is both a source and an imperative emitter, supporting subscribe/unsubscribe/next/complete operations with per-consumer prefixed output lines, matching the Subject source type in the codebase.
