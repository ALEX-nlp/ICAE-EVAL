## Product Requirement Document

Hey team, we need a library that auto-generates Markdown reference docs from our provider schema definitions. Right now devs are hand-writing all the field documentation and it keeps going out of sync whenever schema changes — it's become a real maintenance headache. The idea is: feed in a schema (with all the attribute/block metadata), get back clean, consistently formatted Markdown. We want the same grouping logic we used in that routing/classification work from the config module — required stuff first, then optional, then read-only. Nested structures should expand into their own sub-sections with anchor links. Each field needs a compact inline label showing the type and any special markers (like if something is sensitive or going away soon). There are also some smaller text utility helpers we need bundled in — stripping formatting from Markdown text, prefixing lines, and wrapping file contents in fenced code blocks (with graceful handling when the file doesn't exist). The whole thing should be modular, not one giant file. Types, labeling, classification, schema rendering, and text helpers should all live separately. The JSON wire format for types supports both bare strings and nested arrays — make sure the rendering handles deep nesting correctly. Can someone take a look at the test cases in the rcb_tests folder and make sure everything lines up with those contracts?

One other pass with a few specifics the team asked for: for type names, the primitive mapping is very literal and limited to just 'bool' → 'Boolean', 'number' → 'Number', 'string' → 'String', and 'dynamic' → 'Dynamic'. Those are the only valid primitives, so anything else should be treated as an unsupported type error. For collections, list/set/map should read as 'List of <element>', 'Set of <element>', and 'Map of <element>' with the element rendered recursively. Object types render as just 'Object' and tuple types as just 'Tuple', without expanding members. Deep nesting should just stack naturally, like ['list', ['map', ['set', ['object', {}]]]] → 'List of Map of Set of Object'.

On the text helper side, plain_markdown should strip Markdown formatting markup while keeping the readable text itself intact. That includes heading markers (#), emphasis markers (* and _), inline code backticks, and list markers, and blank lines between stripped blocks should be collapsed so the result is plain readable text with no Markdown syntax. prefix_lines should add the provided prefix to every line, including the first one, and keep line breaks exactly as-is. The example to match is prefix '  ' on 'line1\nline2' yields '  line1\n  line2'.

A couple classification details were also worth locking down. If an attribute is both optional AND computed, it still lands in 'Optional', not 'Read-only'. 'Read-only' is only for computed when there is no optional flag set. And required beats everything, so if required=true, the label is 'Required' regardless of other flags. When markers are present, they come after the behavior word in a fixed order: 'Sensitive' first, then 'Deprecated'. They’re comma-separated, and only show up when that flag is true, so if the flag is absent or false, nothing extra is added.

Also, the package split should stay clean and explicit: (1) type rendering, (2) attribute/field label rendering, (3) behavior classification, (4) whole-schema markdown rendering, (5) text helpers (plain_markdown, prefix_lines, code_file), and (6) a thin execution adapter for JSON stdin/stdout. Core rendering logic should stay decoupled from I/O and JSON parsing. And for the classification behavior specifically, please make sure we line up with the classify_attribute and classify_block operations defined in feature6_1_classify_attribute.json and feature6_2_classify_block.json, which bucket fields into role=required / role=optional / role=read_only, along with the JSON test case files under rcb_tests/public_test_cases/, specifically feature1 through feature7_3, which define the black-box contract for all ops dispatched through the execution adapter.
