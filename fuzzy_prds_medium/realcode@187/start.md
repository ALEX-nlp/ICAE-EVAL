## Product Requirement Document

Hey team, we need to build out that test-double library we've been talking about. Basically developers are spending way too much time hand-rolling fake implementations every time they want to isolate a unit under test — it's super repetitive and breaks constantly when interfaces change. We need something that lets you spin up a stand-in for any collaborator in one line, tell it what to return (or blow up with an error), and then afterwards confirm it was actually called the right number of times.

A few things that came up in last week's sync: the library needs to handle the async stuff the same way it handles regular calls — no second-class treatment. Also remember how we handled the 'returns self' chaining pattern in the settings module of the auth service refactor? We want something like that here too for default answer strategies. And there was that discussion about the 'stub-only mode' for memory-sensitive paths — make sure that's in scope.

The output format when something goes wrong needs to be completely neutral — no internal runtime class names leaking out. Errors should be categorized in plain language. Named doubles should show up properly in failure messages. Also want spy support so we can wrap real objects and only override specific bits. Please make sure the code is organized properly across multiple files — last time someone dumped everything in one god-file and it was a nightmare to maintain.

Also adding a few specifics that came up while people were reviewing the shape of this. For anything you haven’t explicitly programmed yet, members should just fall back to type-based defaults: numeric members return 0, boolean members return false, and object/string members return null. Those need to show up literally in output as 'value=null', 'value=0', and 'value=false'. If someone uses 'return_consecutively', treat that as an ordered sequence across calls: each invocation gets the next value, and after the sequence runs out, keep returning the final value in the sequence forever. It should not reset and it should not fall back to the type default after exhaustion.

On verification, we do want a couple wildcard matcher options in the 'matcher' field. 'any' should accept any non-null argument, and 'any_or_null' should also accept null arguments. For numeric matching, support 'geq', 'leq', 'gt', 'lt', and 'eq', and let people compose those with 'and', 'or', and 'not' in the obvious way. If an argument match fails, the output should be 'error=arguments_are_different member=<name> times=N'. One other small thing here: when no matcher is provided on a verify step for collection members (varargs), the default should be to match any vararg invocation.

We also need a simple 'capture' script step for pulling back recorded arguments for a member. The 'want' field decides the shape: 'all' returns 'captured=[v1, v2, ...]' in recorded order, 'first' returns 'captured=<first value>', and 'last' returns 'captured=<last value>'. That only works when invocation history exists, so it’s not available in stub-only mode.

On the BDD side, there should be a behavior-style entry point as a straight equivalent to the main stub API, triggered by the 'bdd' input key. 'will_return' sets a fixed value, 'will_throw' sets an error category to raise, and the BDD phrasing should also cover sequences and computed answers. Output stays exactly the same as the primary API.

For async, same story as we said before: suspending members need to behave exactly like synchronous ones, driven by the 'async' input key, with 'asyncNumber' as the member used in these flows. A suspending member should support normal programming like 'returns: 42' and normal verification like 'verify: true, invocations: 2'. Output format is identical to synchronous members.

A couple config details too. If a double is named through 'settings.name', that name should appear in diagnostics. So a missing-call failure should read 'error=wanted_but_not_invoked target=<customName>.<member>()'. If nothing is configured, use 'collaborator' as the default name. For default answer behavior under 'settings.default_answer', we need the three options we discussed: 'returns_self', 'returns_mocks', and 'deep_stubs'. 'returns_self' means unprogrammed members return the double itself and output 'identity=self'. 'returns_mocks' means unprogrammed reference-typed members return a fresh double and output 'presence=present'. 'deep_stubs' means lazy chaining works so a chain can be stubbed declaratively, and when the terminal member is stubbed the output is the normal 'value=N'.

There are also two marker-style settings to carry through. If the double is created with 'settings.serializable: true', then 'report_serializable' should emit 'serializable=true'; otherwise it emits 'serializable=false'. That’s just a capability flag on the double, not actually doing serialization. And if someone creates the double with 'settings.invocation_listener: true', the listener should be notified on every member invocation. Then 'report_listener' should emit 'listener_notified=true' once anything has been invoked since attachment, or 'listener_notified=false' if nothing has happened yet.

Finally, just to pin down stub-only behavior a bit more: with 'settings.stub_only: true', configured responses still work like normal, but we do not record invocation history at all. So verification attempts in that mode should fail with 'error=stub_only member=<name> <mode>' instead of trying to run the check.
