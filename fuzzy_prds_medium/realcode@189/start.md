## Product Requirement Document

Hey team, we need to build that CI config generator tool we've been talking about. The idea is that each package in the repo just has a small config file describing what SDKs to target and what steps to run, and then this tool spits out the full Travis CI setup plus a shell runner script automatically. No more hand-editing that giant YAML every time someone adds a new package.

A few things I know are important: packages should be able to share work when they're doing the same thing (why run the same check twice?), and when packages disagree on step ordering we should catch that early and tell the developer clearly instead of silently generating something broken. There's also some logic around how different task names that end up doing the same underlying thing should be handled — kinda like the deduplication approach we used in that billing pipeline refactor.

The tool also needs to read an optional root-level config file for repo-wide settings, and it should complain loudly if someone references a step name that doesn't actually exist anywhere. Validation of the per-package files should happen before anything is written out.

One more thing — there's a flag for how dependencies get fetched in the generated script, default behavior vs the alternative mode. Make sure warnings about SDK version mismatches and unused SDK declarations still show up in the console output even when we do generate successfully.

Quick follow-up from the questions that came in. For task naming, the important bit is that we only keep the bare task name when every package resolves that name to the exact same shell command. If the same task name like `dartfmt` points at different concrete commands in different packages, then each distinct command needs its own suffixed key like `dartfmt_0`, `dartfmt_1`, etc, and both the job script entries and the runner case branches should use those suffixed keys. That’s the same disambiguation behavior from Feature 2.2 disambiguate task keys logic, as implemented in the task-extraction/de-duplication engine module.

On stage ordering, we’re not picking an order package by package. The order comes from one global topological sort over the union of every package's ordered stage list. So if two packages declare the same pair in opposite order, that creates a cycle and we should stop immediately with a `user_exception` using exactly `message=Not all packages agree on \`stages\` ordering, found a cycle between the following stages: [<sorted stage names>]` and  with an empty `details` field. In that case nothing gets written out.

A couple more specifics on caches and job generation. In `.travis.yml`, cache directories always start with `$HOME/.pub-cache` as the first entry. After that, anything from package `cache.directories` that starts with `/` is treated as an absolute/repo-root path, included once across packages, deduplicated, and sorted. Relative paths get namespaced under the package directory, so `.dart_tool` in `pkg_a` becomes `pkg_a/.dart_tool`. Then the final list is sorted after the pub-cache entry, with absolute paths and namespaced relative paths interleaved in sort order. Also, jobs do not get auto-collapsed just because packages share the same stage, SDK, and task command. Each package still gets its own job entry unless `merge_stages` is configured in `mono_repo.yaml`, which is the only time we combine packages into one job with space-separated PKGS. Cache directories still aggregate the same way either way.

For `tool/travis.sh`, the shape is fixed: shebang + header comment, PKGS guard, argument count guard, EXIT_CODE=0, then a per-package loop with `pushd`, the dependency-resolution block, and then a per-task loop with a `case` statement. The case branches need to be in alphabetical order by task key, and the `*)` branch always goes last. One tiny UX thing the team asked about: in the console output, after we acknowledge the `.travis.yml` write and before we acknowledge the `travis.sh` write, always remind the developer to `chmod +x ./tool/travis.sh`.

On dependency fetching, the `useGet` boolean option default `false` is what controls the verb in the generated runner script, and it comes through as part of the JSON input to the execution adapter. When `false`, use `pub upgrade --no-precompile` and the failure text must be `pub upgrade failed`. When `true`, use `pub get --no-precompile` and the failure text must be `pub get failed`.

And on SDK validation, Feature 4 is specifically the `environment.sdk` check from each pubspec.yaml: parse the semver range there and compare pinned numeric SDK versions against it, then emit the per-package warnings when a version is incompatible. Those warnings should still show up the same way on successful generation too.
