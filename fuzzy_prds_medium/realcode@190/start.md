## Product Requirement Document

Hey team, we need to build out our content moderation pattern engine. Basically the idea is that trust & safety folks can define 'bad phrases' with some lightweight pattern syntax (think optional segments, wildcards, word boundaries), and the engine handles finding those phrases in user text, then blotting them out. We also want to be smart about adversarial obfuscation — l33tspeak, lookalike Unicode chars, stuff like that — so normalization needs to be configurable in a pipeline style. There's also a whitelisting concept so something like 'pen is' doesn't trip on a partial match, similar to how we handled the safe-word carve-outs in that old classifier module. Repeated-character spamming (like 'heeeeello') should be collapsible too. On top of that, phrases should be groupable with metadata so downstream consumers know which category a match belongs to — and datasets should be mergeable/filterable. Output-side, we need a censoring layer that can mask matched spans in a few different ways (full mask, keep first char, keep last char, fixed replacement string). Error feedback should be clean and consistent — no raw stack traces or language-level exception names leaking out to callers. The whole thing should be wired up so JSON commands drive execution from stdin, with results on stdout. Would love this done cleanly so adding new transforms or strategies later doesn't require rewriting half the engine.

One more pass on a few details the team asked about. For match reporting, start_index, end_index, and match_length need to line up with Unicode code-point positions from the original input string, not byte offsets. So if we’re looking at 'cool cool cool cool 🌉', that bridge emoji is one code point and takes one position. A match on 'cool 🌉' that ends on that emoji should come back with end_index=21 and match_length=6 based on the original visible character positions, and we should keep that externally consistent no matter whether anything inside is using UTF-16 or UTF-8.

On the whitelist behavior, the important part is that suppression only happens when the whitelisted span completely covers the prohibited match span after applying the relevant whitelist normalization transforms, including whitelist_transforms where that’s the path being used. Partial overlap is not enough. In the 'penis'/'pen is' example, the whitelist 'pen is' suppresses the partial 'pen is' overlap but NOT the standalone 'penis' at position 29, because that whitelist span does not fully contain that match. Same basic idea in Feature 2 as well: this should be handled in the matcher’s overlap/containment check, not as a simple substring exclusion.

For errors, we need to be really strict about the output shape. All errors should be normalized multi-line key=value strings written to stdout, never raw language exception names and never stack traces. For pattern syntax problems the format is exactly 'error=pattern_syntax\nline=<n>\ncolumn=<n>\nmessage=<human message>'. Unknown pattern ID errors use 'error=unknown_pattern_id\nid=<n>'. Validation errors for invalid remap keys or bad strategy configs follow the same key=value pattern too. The system should catch domain exceptions internally and convert them before anything is written out.

On the adapter side, it reads a single JSON object from stdin for each invocation, and the keys on that object decide what gets called. Presence of 'pattern' alone means Feature 1. 'patterns' or 'terms' with 'text' means Feature 2. 'text' with 'matches' and 'strategy' means Feature 3. 'text' with 'pipeline' means Feature 4. 'default_threshold' with 'sequence' means Feature 5. 'phrases' means Feature 6. Also worth restating that this adapter is the only layer that should know about JSON and stdio at all; the core domain classes need to stay decoupled from I/O.

For censoring, process matches in position order. If a match is fully contained within, or exactly equal to, a range we already processed, just ignore it. If two matches partially overlap and neither fully contains the other, we still replace both, but we do it without duplicating the untouched gap. The example here is matches [0,4] and [0,7] on 'thinking of good test data is hard' with fixed_char '.', which should give 13 dots covering positions 0–7 merged, then ' of good test data is hard'. The supported strategies are deterministic and just these four: 'fixed_char', which repeats a single character across the matched span; 'fixed_phrase', which swaps the whole matched span for a fixed string; 'keep_start', which keeps the first matched character and applies a nested base strategy to the rest; and 'keep_end', which keeps the last matched character and applies the base strategy before it. Whatever strategy we use, the output line always starts with 'text=' and then the censored result.

And for the transform pipeline, the pipeline field is an ordered array of steps applied left-to-right per code point. Each step is either one of the named strings 'ascii_lowercase', 'skip_non_alphabetic', 'leet_speak', or 'confusables', or an object with type='remap_characters' and an inverted map. That’s the shape we should support there, nothing extra.
