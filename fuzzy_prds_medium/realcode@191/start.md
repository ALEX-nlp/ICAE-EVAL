## Product Requirement Document

Hey team, we need a lexical scanner for that Huff-like stack contract language we've been talking about. Basically something that takes raw source text and spits out a clean token stream — similar to what we did for that preprocessor tokenizer a while back, you know the one. The big pain point right now is that every tool that needs to read these contract files is doing its own ad-hoc splitting and keeps getting comments wrong, or misidentifying labels as keywords and vice versa. It's causing inconsistent span reporting downstream and a bunch of brittle one-off parsers.

The scanner should handle the usual suspects: directives, comments (both flavors), include paths, type annotations including the array forms, and the opcode mnemonics that only show up inside macro bodies. One tricky bit is that some words are keywords in one spot but plain names in another — the scanner needs to be context-aware about that. There's also a hex literal normalization thing we discussed that I don't fully remember the details of, just check how the old formatter handled it. Output should be a flat line-per-token format with category, value, and byte positions. Callers need to be able to toggle whether whitespace tokens are included and whether an end-of-input sentinel gets appended. JSON in, formatted token lines out for the test harness layer, but keep the core logic separate from that plumbing.

One extra pass on the scanner details since a few folks asked for specifics. For literals, hex byte literals like `0xa57B` should come out as `literal` tokens, and the value is the raw hex content after the `0x` prefix, left-padded with zeros to exactly 64 hex characters (32 bytes). So `0xa57B` becomes `token=literal:0x6135374200000000000000000000000000000000000000000000000000000000`. Also important for spans: the span only covers the hex digit characters after `0x`, not the prefix, so if `0xa57B` starts at position 0 the span is `2..6`. For non-hex numeric literals, anything without a `0x` prefix is a `number` token, and the value should be the canonical decimal string with leading zeros stripped, so `00` becomes `token=number:0`. In that case the span covers the full original source bytes, so `00` at position 0 has span `0..2`.

On opcodes, inside macro bodies they should be emitted as `opcode` tokens with the compound value format `<decimal_value>:0x<hex_value>`. The examples to stick to are `lt` -> `token=opcode:10:0x10`, `gt` -> `token=opcode:11:0x11`, and `eq` -> `token=opcode:14:0x14`. These only count as opcodes once we’re actually inside the macro body after the opening `{`, not up in declaration headers or other top-level spots.

Also confirming the context-sensitive naming behavior. Words that are usually keywords, like `macro`, `function`, `constant`, `event`, still need to be emitted as `ident` when they’re used in identifier-only positions, specifically right after the declaration-kind keyword, meaning the second token after `#define`. So in `#define function macro(...)`, that `macro` is `token=ident:macro`, not `token=keyword:macro`. Same idea for any other reserved word used as the name in that slot.

For parameter lists, array types should stay together as one token. So `address[3]` and `string[1]` are a single `array` token with the full value included, like `token=array:address[3]`, and the span covers the whole array type text including the brackets. Open-ended forms work the same way, so `address[]` becomes `array:address[]` with empty brackets. We should not split those into separate type and symbol pieces.

And for implementation shape, please mirror the existing Huff lexer/scanner logic in huff-rs from the huff-lexer crate, specifically the Token and TokenKind types and the Lexer::look_ahead / next_token pattern. That’s the architectural model we want this to line up with.
