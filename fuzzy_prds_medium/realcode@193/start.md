## Product Requirement Document

Hey team, following up on the distributed tracing initiative we discussed last sprint. We need to build something that basically wraps all the usual stuff our microservice framework does — handling web requests, talking to downstream services, hitting the database, running things in the background, the scheduler stuff, the message bus, and the circuit breaker thing — and automatically produces tracing data without developers having to touch their business logic at all. Think of it like the logging approach we used for that auth module rollout: plug it in, it just works.

The big pain point from ops right now is that when something goes wrong in production, we have no visibility into which hop in the chain failed or where the latency is coming from. Support tickets are piling up because engineers have to manually grep logs across five services to piece together what happened.

We also need a way to verify this works — some kind of test runner that can go through a directory of test scenarios and tell us pass/fail. The output format needs to be totally deterministic so we can do automated checking, no random IDs or anything environment-specific leaking into the results. There are also some edge cases around skipping certain paths and handling unreachable backends that need to be covered. The rendering for the gateway and message channel scenarios is slightly different from the others — check how we did something similar before for the format details.

One more pass on the verification/output details since the team had a few good questions. For the normal span rendering, each span line must follow this exact shape: `op=<operation name> component=<component|-> span.kind=<kind|-> http.status=<status|-> attrs=[k=v;...sorted] error=<true|false> error_log=<event|-> parent=<parent operation|root|external>`. The full output block starts with `spans=<total>`, then `distinct_traces=<number>`, then one sorted line per span. When there are no spans, distinct_traces should be 0. Parent is `root` when there is no parent span, and `external` when the parent exists but is not part of the recorded set.

A couple feature-specific renderings are intentionally different. The gateway flow does not use the normal per-span line format. There we want total span count, distinct traces count, per-component span counts sorted and formatted as `component.<name>=<count>`, and then, when a gateway span exists, `gateway.op=<method>`, `gateway.http.status=<code>`, `gateway.route.host=<url>`, `gateway.error=<true|false>`. For the gateway scenarios specifically, the successful route (`/test`) resolves to `http://localhost:53751/hello` and the unreachable backend (`/wrong-port`) resolves to `http://localhost:53752/wrong-port`, and those exact values should show up in `gateway.route.host`.

For database work, the span should come through as `op=Execute`, `component=java-jdbc`, `span.kind=client`, with attrs containing `db.statement=<statement text>`, `db.type=h2`, and `db.user=SA` in sorted order. HTTP status is `-` there since it is not an HTTP operation. For background work, async method spans use `component=async` and carry `class=<SimpleClassName>` and `method=<methodName>` in attrs, and scheduled task spans use `component=scheduled` with the same `class` and `method` attrs. The example to stick to is `op=runJob component=async span.kind=- http.status=- attrs=[class=BackgroundJobs;method=runJob]`.

The message channel case also has its own summary-style output instead of the standard span list. That format is `op=<destination>`, `component=websocket`, `span.kind=<server|client>`, `parent_linked=<true|false>`, `same_trace_as_parent=<true|false>`, and for the outbound (client) case also `context_propagated=<true|false>`.

Also clarifying the skip behavior: a configured skip pattern suppresses the server span for matching paths, but outbound client spans are still produced. In the scenarios we talked about, `/notTraced` matches the skip pattern so the server span is omitted, while a `rest_client` call to that path still yields one client span. `/hello` is traced normally with no suppression.

And just to close the loop on activation, this should follow the ambient activation pattern from the PRD architecture constraints: the instrumentation layer activates only when a tracer is available on the classpath and otherwise stays inert, without requiring changes to business code beyond ordinary framework configuration.
