## Product Requirement Document

Hey team, we've got a request from one of our enterprise partners who's been complaining that their mapping integration keeps breaking whenever they try to build requests for places with special characters in addresses, or when they're passing in lists of stops for routing. They also mentioned something about the authentication tokens not being attached consistently — apparently sometimes requests go out without credentials at all, which is a compliance nightmare for them. Can someone look at how we handle the serialization side of things? I think there's some existing logic in the repo similar to what we did for the login module's encoding pass — just follow that same pattern. Also, we need to make sure that when the backend returns something other than a 200, we're not leaking internal exception details to the client — just a clean category label and whatever status number came back. One more thing: the partner also wants the roads and elevation endpoints to properly reject bad input shapes instead of silently doing the wrong thing. I know we've talked about this before but let's make sure it's actually enforced. Please check the existing test structure before adding anything new — I don't want duplicate coverage.

Just adding a few specifics from the questions that came back. For single coordinates, a coordinate value is rendered as 'value=<lat>,<lng>\n'. We should accept structured inputs with 'lat'/'lng' keys, two-item numeric lists, and already-formatted coordinate strings, all through the same coordinate normalization flow we already have in the serialization utilities from Feature 1.1 coordinate_text and Feature 1.2 location_sequence_text. If it’s not actually a location shape, like a plain string that isn’t coordinate-shaped, it should come back as 'error=invalid_type\n'. For multiple locations, they’re joined with a pipe character '|' and prefixed with 'value=', and each one gets normalized first, so the final form is 'value=<loc1>|<loc2>|...\n'.

For routing-style inputs, origins and destinations should also be pipe-joined, with '|' encoded as '%7C' after each location is individually normalized. The URL uses '+' for spaces in address strings and '%7C' for pipe separators, and parameters are sorted alphabetically in the URL. That output also includes 'row_count=0\n'. On the validation side, elevation requests should reject 'locations' when it comes in as a flat two-element numeric list like [40.714728, -73.998672] and return 'error=invalid_type\n' instead. Locations need to be a list of coordinate objects or a list of lists. Same idea for sampled paths: if there are fewer than two points, reject it with 'error=invalid_argument\n'. Roads snap should do the same kind of shape check for 'path' as a flat two-element numeric list and return 'error=invalid_type\n'; the path has to be a list of coordinate pairs or objects. And for speed limits, place_id values need to be a list of strings, with bad shapes rejected before the HTTP request is made.

For timezone, the expected summary is very literal: 'method=GET\nstatus=200\nurl=https://maps.googleapis.com/maps/api/timezone/json?key=<key>&location=<encoded_latlng>&timestamp=<ts>\ntimezone_status=OK\n'. The location is percent-encoded, the timestamp is passed as an integer, and the response summary field is 'timezone_status' with value taken from the mocked API response. Address validation is a POST. The output includes 'method=POST\nstatus=200\nurl=https://addressvalidation.googleapis.com/v1:validateAddress?key=<key>\njson=<body>\nresponse_address_lines=<lines>\n'. The JSON body has an 'address' object with 'addressLines', 'locality', 'regionCode', and a top-level 'enableUspsCass' boolean. Fields are sorted alphabetically in the JSON.

Also, for static map serialization, a marker overlay is rendered as 'marker=size:<s>|color:<c>|label:<l>|<loc1>|<loc2>...\n' and a path overlay is 'path=weight:<w>|color:<c>|<loc1>|<loc2>...\n'. Style attributes come before the locations. Coordinates are formatted as 'lat,lng'. String addresses are passed through as-is. For the actual static map image request, all parameters including markers and path are percent-encoded in the URL, sorted alphabetically, and if there are multiple marker groups they should show up as repeated 'markers=' parameters.

One other mapping detail: place autocomplete maps 'input_text' to 'input', 'session_token' to 'sessiontoken', 'strict_bounds' to 'strictbounds', and 'components' to a colon-separated filter string. Find place maps 'input_type' to 'inputtype' and 'location_bias' to 'locationbias'. Fields are comma-joined and percent-encoded. The expected summaries are 'prediction_count=0\n' for autocomplete and 'candidate_count=0\n' for find place.

And on the auth piece, when I said to follow the existing pattern, I specifically meant the URL signing / HMAC-SHA1 credential logic in the request authentication module (Feature 2.1 / signed_path), which encodes client credentials deterministically using the same base64url + hmac pattern used in prior auth work. That should line up with the coordinate normalization and percent-encoding pipeline applied to location values before they are assembled into query strings, located in the serialization utilities (Feature 1.1 coordinate_text and Feature 1.2 location_sequence_text).
