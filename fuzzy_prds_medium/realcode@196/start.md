## Product Requirement Document

Hey team, we need to build that CLI adapter library we've been talking about. The idea is devs should be able to take a regular Python function and expose it as a command-line command without writing all the argparse boilerplate themselves. Think of it like the decorator pattern we used in that routing module a while back — similar philosophy here.

The library needs to handle the usual stuff: positional args, optional args with defaults, variadic inputs, subcommands, namespacing, aliases, boolean toggles, counting flags, and so on. There's also some output rendering logic we need — by default results print line by line, but there should be a way to get raw output too. Errors need to come out in a normalized neutral format, not as Python tracebacks — users keep complaining about seeing raw exception dumps in the terminal and it looks terrible.

Also important: when parameter names have underscores, the CLI-facing name should use dashes instead (again, refer to how we handled that naming convention last time). We also want help text to surface default values automatically.

There's a confirmation prompt feature that needs to handle three states based on the default — when no default is set, when it's yes by default, and when it's no by default — and the prompt formatting and result mapping should reflect which state we're in.

Please check the existing test scenarios folder to understand the full scope, some edge cases aren't obvious from this description alone.

One extra pass on the behavior details the team asked about: for the decorator side, this should follow the `@command` / `@arg` decorator API in argh where callables are decorated to declare CLI metadata, analogous to how route decorators annotate handler functions. On the naming bit, underscore-to-dash conversion is definitely part of the CLI surface exactly as described in feature19 and the `rcb_tests/public_test_cases/feature19_dashed_option_names.json` test case. So if a parameter is named `baz_baz`, the exposed option is `--baz-baz`, and once it’s parsed it needs to be mapped back to the underscored Python parameter name before the callable is invoked. The concrete example to keep in mind is feature19 test case: `argv: 'abc def --baz-baz 8'` produces correct positional and option values.

Also, for short flags, we should avoid collisions instead of guessing. When two or more parameter names share the same first character, like `foo` and `fox`, we must NOT generate conflicting short options like `-f`. In that situation each one is long-form only, so `--foo` and `--fox`. The expected behavior is the feature17 example: `argv: '--foo 3'` sets foo=3 while fox keeps its default of 2. Separately, counting flags are meant to count, not toggle. A counting option starts at its configured integer default like 0 and increments by 1 every time its short flag like `-v` shows up. No flag means you get the default count, one `-v` means default+1, and repeated `-v` keeps incrementing.

For the confirmation prompt, the three-state behavior is driven by the `default` field in the test input. If default is `"omitted"`, meaning no default, the prompt format is `prompt=<text>? (y/n)\n` and an empty reply maps to `result=unknown\n`. If default is `true`, the format is `prompt=<text>? (Y/n)\n` and an empty reply maps to `result=yes\n`. If default is `false`, the format is `prompt=<text>? (y/N)\n` and an empty reply maps to `result=no\n`. The uppercase letter is what signals the default choice, so that part should stay exactly aligned with the result mapping.

On errors, the normalization needs to be very literal and always on stdout as neutral key=value lines. Missing required arguments should render as `error=missing_required\n<argname>\n`. Unrecognized arguments should render as `error=unrecognized_arguments\n<raw tokens>\n`. Invalid command choice: `error=invalid_choice\n<token>\n`. Domain command errors: `error=command_error\nmessage=<msg>\n`. Wrapped application failures should catch the configured exception classes and render them as `error=wrapped_error\nmessage=<msg>\n` or a similar neutral form. The important part is that no Python tracebacks should ever appear in stdout.
