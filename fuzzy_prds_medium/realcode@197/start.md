## Product Requirement Document

Hey team, we need a small utility that takes a deployment descriptor and spits out a list of parameter sets we can feed into our templating pipeline. Basically, there are three ways someone might describe where deployments come from: they either hand-write a short list, point at a folder structure, or reference some config files sitting in the repo. The tool needs to handle all three cases and produce consistent, ordered output every time — no more each squad inventing their own discovery logic and getting different results.

The output format should be something the CI scripts can parse easily, similar to what the infra team settled on for that config-flattening work we did on the cluster-registration flow a while back. You'll know it when you see it — basically a count line up top, then numbered blocks.

One important thing: when the underlying data source is unavailable or broken, we absolutely cannot silently return empty results. The tool has to loudly signal that something went wrong instead of letting a half-baked deployment sneak through.

Also, the folder-scanning mode has some subtlety around how wildcards work with nested paths — make sure the matching logic doesn't accidentally pick up things two levels deep when you only asked for one level. And for config files, nested JSON keys need to be collapsed in a specific way. Please keep the I/O plumbing totally separate from the core logic so we can unit-test it properly without needing a real repo.

A couple of specifics from the team’s questions: the rendering needs to be exact. The first line is `params=<N>` where N is the total number of maps. Then for each map, output a line `#<i>` using a zero-based index, followed by one `key=value` line per entry sorted alphabetically by key. Every line ends with a newline. If the source-of-record fails, we do not emit a partial payload or an empty one — print exactly one line: `error=source_unavailable` followed by a newline, and nothing else. That’s the output rendering contract from the PRD too: `params=<N>` header line, then per map a `#<i>` index line followed by sorted `key=value` lines, terminating with `error=source_unavailable` on source failure — as specified in the Core Features section of start.md.

On the path matching side, the wildcard behavior is intentionally strict. A single `*` matches any sequence of characters within one path segment only, so it does NOT cross the `/` path separator. In other words, `*` matches `app1` but not `p1/app3`. A pattern like `p1/*/*` matches `p1/p2/app3` but not `p1/p2/p3/app4`. `**` is only for file-scan patterns and does match across segments. Also, matching is done pattern-by-pattern then candidate-by-candidate, preserving that order in output, so please don’t re-sort discovered items afterward.

For the config-file case, nested JSON objects should be flattened by joining property names with dots. So if a value lives at `cluster` → `owner`, the emitted key is `cluster.owner`. Only leaf string values are emitted. Intermediate objects do not produce keys, and non-string leaf values are skipped entirely — numbers, booleans, nulls, arrays all get ignored.

And one extra note for the inline list (`list`) action: each element can include an optional `values` object with extra string-to-string pairs. Each of those entries should show up in the parameter map under a key built by prefixing the entry name with `values.`. So `values: {"foo": "bar"}` produces `values.foo=bar`. The two fixed keys `cluster` and `url` are always present regardless.
