## Product Requirement Document

Hey team, we need to build out that caching utility layer we've been talking about for the web app. Basically devs are super frustrated right now because every time they need to cache something — whether it's a computed value, an API response, or a chunk of rendered HTML — they're rolling their own logic and it's all inconsistent. We keep getting bugs where stale data shows up or the same thing gets computed like 5 times when it shouldn't be.

The core idea: one unified caching adapter that handles storing key/value pairs, wrapping functions so their results get reused, caching HTTP route responses (think query string normalization like we did in that search endpoint refactor last quarter), and template fragment caching. We also need proper lifecycle stuff — like being able to check if something exists, remove things in bulk, and have values automatically go away after a while.

Also important: when you set the thing up, later config should win over earlier config — similar to how the auth middleware init works. And if someone passes bad config (like a network backend with no host), we want a clean, non-framework-specific error, not a raw Python exception blowing up in prod.

Function caching needs to be smart about arguments — if someone calls the same function with args in different orders or using keyword vs positional style, it should still hit the cache. Same deal for HTTP query strings.

Let's keep it clean and modular. More details to come but wanted to get this filed.

One quick follow-up from the questions that came in. For cache reads, if a key is missing we should be very literal about it: render `key=<key> value=none`. That is lowercase `none`, and it should not come back as null, undefined, empty string, or anything else. For bulk reads, keep it to one line as `values=<json_array>`, with the array JSON-serialized with no spaces and in the exact order the keys were requested. The example to follow is `values=[["bar"],"eggs"]`.

On the memoized function side, if a call is missing a required argument, the output for that call should be `error=missing_required_argument`. We do not want a Python exception name showing up there. For the producer in feature 10, the computed value format is `calc-<sum>-<call_count>` where `<sum>` is the sum of all argument values and `<call_count>` is the cumulative producer invocation count. Example: args (1,2) with c=1 default → sum=4, first call → `calc-4-1`. Also, if we delete one argument set for a memoized function, that should only remove that specific argument-keyed cache entry and force recomputation for that argument set only. Other argument sets stay cached, and the recomputed value increments the producer call count. That producer uses the format `total-<sum>-<call_count>`.

For HTTP route caching, the second request to the same URL should return the same body as the first without invoking the handler again. Output is one line per request as `request=<label> status=200 url=<url> body=<body>`, then `handler_calls=1`. The body format is `body-<N>` where N is the handler invocation count. Query params need to normalize for cache lookup, so same params in different orders share one cached response, while different values still get their own cache entry. In feature13_http_query_caching, that means sort query parameters alphabetically to produce a canonical cache key, while preserving the original URL in response output.

For template fragments, the cache key should use `_template_fragment_cache_` plus the fragment name, so `fragment3` becomes `_template_fragment_cache_fragment3`. Output needs to include `fragment_key=_template_fragment_cache_<fragment_name>`. And for deletion, the output should be `before_delete=<body>` showing the fragment value before deletion, `fragment_key=_template_fragment_cache_<name>` showing the key used, then `after_delete=none` confirming the fragment is gone. Again, absent values should use the literal string `none`.

Two more setup details. If a network cache backend is configured without the required host parameter, the output must be exactly two lines: `error=argument_null` and `param=host`. We want that clean language-neutral error representation, not a raw Python exception dumped to stdout. And for Feature 8 configuration initialization, `init_kind` should beat `constructor_kind` as the `active_backend`, and the cache should also report `app_attached=true`.
