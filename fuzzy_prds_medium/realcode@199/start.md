## Product Requirement Document

Hey team, we need a static analysis tool for our CI/CD config files — think GitHub Actions workflows, reusable action metadata, and GitLab pipeline configs. The idea is to parse these files into some normalized structure, figure out what third-party stuff they depend on (actions, images, includes), and flag security problems automatically instead of doing manual reviews.

Basically we keep running into the same issues: people referencing actions without pinning versions properly, workflows with overly broad permissions on triggers that could be hit by fork PRs, jobs running on self-hosted runners that shouldn't be exposed that way, secrets leaking into environment variables, that kind of thing. It's getting hard to catch these consistently as the team grows.

The whole thing should be organized cleanly — not just one giant file — separate concerns properly, keep the business logic away from stdin/stdout stuff. Should handle bad/malformed configs gracefully without crashing.

We also need it to handle that package URL normalization stuff like we talked about in the supply-chain meeting (similar to what the security team uses for the container inventory work). And the runner detection logic should match what we agreed on in the last infra review — basically the same approach as the hosted-vs-self-hosted classification we use elsewhere.

Output should be deterministic and structured so it's easy to consume downstream. Can someone pick this up?

Quick follow-up from the questions that came in: for runner reporting, if runs-on is a structured object, group entries get flattened to 'group:<name>' and label entries to 'label:<name>', and we emit each of those as separate 'runs_on=<target>' lines in order. If it’s just a plain string, emit it as-is, and if it’s a list, emit each element as its own line. On source locations, we track two one-based line numbers: 'line.start' is always emitted for the line where the job mapping begins, and 'line.runs_on' only shows up when there’s actually a runner declaration. When both are present, they should be emitted sorted by key name alphabetically, so line.runs_on before line.start.

Also, anything that comes through as a reference-style value tagged with '!reference [...]' should stay exactly as-is in output, including the trailing newline. So if a script entry is a reference tag, it should come out as 'script=!reference [.vars, variables, SCRIPT]\n' with that newline included. For global variables, emit each one as 'variable <name> value=<value> expand=<bool>'. If expand isn’t specified, it defaults to false, and a variable defined as a plain scalar gets expand=false. If there’s a description, append ' description=<desc>', and if there are options, append ' options=<v1>,<v2>,...'.

On includes, the output format should be 'include <fields>' lines. Remote URLs using https:// should be 'remote=<url>', bare local paths should be 'local=<path>', mapping entries with a 'local' key are also 'local=<path>', and entries with 'project' should be 'project=<p> ref=<r> file=<f>'. If an include has inputs, add them as 'input:<key>=<value>' fields. One important edge case there: includes with CI variable interpolation in the project field should be skipped from the inventory.

For invalid jobs, we should reject a structurally invalid job body as 'error=invalid_job'. That includes wrong node kinds, empty/blank group or label entries, non-mapping permissions/env, malformed nested values, or a job body that is a sequence (list) rather than a mapping. On the rule side, the engine should cover the full catalog we talked about: injection of untrusted input into run steps, missing or overly-broad default permissions on risky triggers (pull_request_target, workflow_run), jobs with self-hosted runners reachable from forks, use of vulnerable/unpinned action versions, and CI_DEBUG_SERVICES/CI_DEBUG_TRACE enabled in GitLab pipelines. Each finding needs the rule id, severity, package URL, file path, one-based line, job/step context, and human-readable detail.

One more implementation-shape note since a few folks asked how strict that is: the physical structure really does need to match the domain complexity. This should be a clear multi-file directory tree with distinct layers: a model/domain layer for parsed structures, a parser layer (separate parsers per dialect), a package-identifier/purl layer, a rule/policy engine layer (open for extension), a dependency inventory/scanner layer, and an I/O adapter layer. A single monolithic file is explicitly prohibited.

And for the two behavior details that need to line up with prior decisions: the self_hosted_runner predicate in feature 6 should treat the recognized GitHub-hosted label set (ubuntu-*, windows-*, macos-* standard variants) as false, and anything else, including an explicit 'self-hosted' label, as true. Also, the purl_normalize and purl_from_uses operations in features 5.1 and 5.2 should follow the githubactions ecosystem lowercasing + subpath extraction behavior, and docker:// prefix yielding pkg:docker/ with percent-encoded image names.
