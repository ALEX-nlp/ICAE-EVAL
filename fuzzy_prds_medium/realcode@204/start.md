## Product Requirement Document

Hey team, we need to wrap up that record-mapping library we've been discussing. The core idea is that devs are sick of writing the same boilerplate row-mapping and relationship-stitching code over and over — it's causing those N+1 read issues we keep seeing in prod and the error messages are all over the place depending on who wrote the query. We basically need something like what we did with the loader abstraction in the billing module, but more general-purpose.

The library should let you pull back lightweight records with just the fields you care about, handle related data (both direct and the multi-hop stuff), and support batching so we're not blowing up memory on big result sets. Raw SQL should still work for the cases where people need it. Projections to scalar arrays would be a nice-to-have too.

One thing that keeps biting us — when a field wasn't selected or a relationship wasn't loaded, the error that comes back is totally inconsistent. We need stable, predictable errors that don't leak internal stack details.

Also there was some discussion about polymorphic targets (where the related record could come from different tables per row) and indirect traversal paths — both need to work. Output should be deterministic JSON on stdout. Ask me if anything's unclear, but I think the test files cover most of the edge cases.

One small follow-up from the questions that came in: for the basic ordered query case, the output shape matters. If it’s just a simple scalar projection like names, it should come back as a JSON object with a `records` key containing the array, like {"records":["Bar","Foo"]}, and that order needs to be deterministic, like alphabetical ascending. If it’s a selected-field query, use `rows` instead, and that should be an array of objects containing only the requested scalar fields.

For first-match behavior, we do want the matched row fields in the output, plus a `sql` array showing the query used, and that SQL trace string needs to specifically include `LIMIT 1`. The trace format is "root: SELECT <table>.* FROM <table> WHERE <condition> LIMIT 1". If the record is required and nothing is found, the output must be exactly {"error":"not_found"}.

On the error side, let’s keep the messages fully consistent. If someone touches a relationship that wasn’t loaded, return {"error":"association_not_loaded","association":"<rel_name>","record":"<ModelName>","trace":"root"} with the trace reflecting depth when it’s nested, like "root.orders.line_items". If they access a column that wasn’t part of a select, return {"error":"column_not_selected","column":"<col_name>","record":"<ModelName>","trace":"<query_path>"} and for nested stuff that trace should be the full dot path, for example "root.orders.line_items". If they try to access a member that doesn’t exist on a related record, return {"error":"undefined_member","member":"<member_name>","trace":"<query_path>"} and again the trace should match the related record path, like "root.category". Also just to reiterate, we want stable, predictable errors and no leaking internals.

A couple more behavior details: value casting needs to cover boolean status, where integer/string DB values map to true/false JSON booleans, coded status where numeric codes map to string labels and unmapped codes return null, and localized timestamps where UTC DB timestamps are converted to a specific timezone offset and formatted as ISO 8601 with offset, like "2017-12-29T10:00:37-05:00". Returned record objects also need to support both string-key and symbol-key access, with the same value either way. Equality should be based on the underlying row identity, so two references to the same row are equal, and records for different rows are not.

On delegated model behavior, records should be able to call through to model-defined methods in the normal way. That needs to work with no extra argument, with an argument, and with an argument plus block/callback value. The example progression there is "Foo - All about Widget A", then "Foo - All about Widget A - foo", then "Foo - All about Widget A - foo - bar".

And for SQL traces, the path labels should use dot notation like "root.through(orders).through(line_items).categories". If there’s an `IN` clause, the values need to be sorted numerically ascending, with IDs comma-space separated and no trailing comma. Also, trace should only be included when the scenario explicitly asks for it. This all applies across the relationship loading subsystem described in features 5, 6, and 7 — specifically the direct/indirect/polymorphic association loading logic that pre-fetches related records in batch rather than per-row. The stdin/stdout contract is still the one defined by the JSON files under rcb_tests/public_test_cases/ (feature1 through feature11).
