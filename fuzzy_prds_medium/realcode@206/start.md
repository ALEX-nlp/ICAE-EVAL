## Product Requirement Document

Hey team, we need to wrap up the geo data library before the sprint ends. The core ask is pretty straightforward — we're building something that lets devs work with geographic shapes and interchange them as standard JSON without hand-rolling all that tedious parsing stuff every time. Think points, paths, areas, collections, the whole family. Each shape needs to be constructable from raw data, and we need proper validation so bad inputs blow up clearly rather than silently producing garbage — similar to how we handled the validation patterns in that coordinate module we did before, just apply the same philosophy here.

The serialization side needs to support both compact and pretty-printed output. We also need support for coordinate reference systems — named ones, linked ones (with URI validation), and the 'no CRS specified' case. Features and feature collections need to wrap geometries with optional IDs and property bags. Equality checks need to work reliably across all types, including hash code consistency so these can safely live in sets or maps.

One thing the business cares about specifically: when a consumer sends us bad data (null where something's required, rings that don't close properly, coordinate lists that are too short), they need to get back a clear, consistent machine-readable error code — not a stack trace or some runtime-specific exception message. The error shape needs to be the same regardless of what went wrong or which layer caught it. We also want a command-driven adapter so QA can black-box test this via JSON commands over stdin/stdout.

One quick follow-up from the questions that came in: when construction fails, we need those failures normalized into the same language-independent block every time. The set we’re using is error=argument_null + param=<name> for required arguments that are null, error=out_of_range + param=<name> for arguments outside allowed range, error=argument_invalid + param=<name> + reason=<must_be_specified|must_be_dereferenceable_uri|not_a_linear_ring|invalid> for other invalid arguments, error=json for unparseable input, and error=not_supported for unsupported conversions. Also worth being explicit here that the core domain must never write to stdout — only the adapter turns those into the output block, following the construction-time validation and error normalization contract defined in the Execution Adapter Protocol section of start.md.

Also, property order is not flexible here. A geometry renders 'type' first, then 'coordinates' (or 'geometries' for GeometryCollection), with any 'crs' member last. A CRS object renders 'properties' first, then 'type'. A Feature renders 'type', then optional 'id', then 'geometry', then 'properties'. A FeatureCollection renders 'type', then 'features', then optional 'crs'. On the command side, the make_crs operation prints first a line 'type=<name|link|unspecified>', then for property-bearing CRS types, one 'prop.<key>=<value>' line per property sorted alphabetically by key. For a named CRS: type=name, prop.name=<value>. For a linked CRS without type: type=link, prop.href=<value>. For a linked CRS with type: type=link, prop.href=<value>, prop.type=<value>. Unspecified CRS prints only the type line.

A couple geometry specifics too: elevation, meaning the third coordinate component, is optional. It only shows up in output if it was supplied in the input. When elevation is absent, only the horizontal and vertical components are emitted. All supplied components are always emitted, including those equal to zero. Axis order is always horizontal first, vertical second, elevation third. For Polygon validation, each ring has to be a closed linear ring, so it must have at least four positions and its first position must equal its last position. A null ring list yields error=argument_null with param=coordinates. Any ring that fails that closed linear ring check yields error=argument_invalid with param=coordinates and reason=not_a_linear_ring.

And for the LineString helper behavior, the is_closed operation should report is_closed=true when the first and last positions coincide. is_linear_ring=true when the line string is closed AND has at least four positions. Both booleans are printed, one per line, in that order.
