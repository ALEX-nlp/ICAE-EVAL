## Product Requirement Document

We need a static HTTP server that makes a directory tree on disk browsable as web pages, so a developer can serve a folder with a single command and let anyone navigate its contents in a browser. Two specific behaviors make the listing genuinely useful. First, every folder page should show a ready-to-paste shell command at the bottom that a visitor can run to mirror the entire sub-folder recursively — this command should use the wget download tool with a fixed set of flags that handle recursion, timestamps, avoiding index files, and not creating host-named subdirectories. The exact form of the command changes depending on how deep the current folder sits below the root: at root level add a destination flag naming the server's authority, one level deep needs no path-adjustment, and two or more levels deep should add a flag that strips the right number of leading path segments. Second, when the listing is requested in a flat/raw mode, the navigational links should be wired so that clicking into a sub-directory keeps the visitor in flat mode, but links to actual files point straight at the file without any mode modifier. The parent-directory link (shown when not at the root) should also stay in flat mode. Handle link construction consistently with how the existing renderer builds other listing links. Directory links always end with a trailing slash on their path.

Quick follow-up from the questions that came in: the mirror command is always built the same way up front, starting with 'wget -rcnHp -R 'index.html*'', then the depth-dependent path piece, then the URL in single quotes ending with '?raw=true'. The intent behind those fixed flags is still the same every time: -r recursive, -c continue partial, -n no clobber, -H span hosts, -p page requisites, -R reject index.html*. For depth, use the number of path segments below root. Depth 0 means add " -P '<authority>'" like -P 'localhost:8080'. Depth 1 means no path-adjustment flag at all. Depth 2+ means add " --cut-dirs=<depth-1>", so for example at depth 2: --cut-dirs=1, at depth 3: --cut-dirs=2.

Also on the flat/raw link behavior, there are really three link roles to think about: 'root' for the parent-directory link that only shows when you’re not at the served root, 'directory' for sub-directory links, and 'file' for regular file links. Each output line is '<role> <url>'. Only directory-type links, so 'root' and 'directory', append '?raw=true'. 'file' links do not. Directory links still always end with a trailing slash on their path.
