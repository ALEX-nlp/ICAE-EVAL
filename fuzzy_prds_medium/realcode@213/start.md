## Product Requirement Document

hey team, we need to build out the mobile analytics tracking client SDK for android. basically devs are complaining that they have to hand-roll all the request stuff themselves and it keeps breaking in weird ways — wrong decimal formats for prices, malformed JSON for cart items, that kind of thing. we need a clean library that handles all of this for them.

the main things we need: some kind of price helper (devs keep getting the decimal formatting wrong), a checksum utility (similar to what we did for the file verification flow in that other module, you know the one), custom metadata support with both the indexed key-value style and the flat dimension fields, a cart/order serializer, a parameter builder with that 'first write wins' mode people keep asking for, URL encoding for the query strings, and a batching wrapper for sending multiple events at once.

one thing that keeps coming up in bug reports — the batching has a page size limit that devs aren't aware of, and the null/empty handling on dimensions and variables is inconsistent across implementations. also the checksum thing should degrade gracefully instead of blowing up.

we should follow the same repo structure pattern as before — nothing too over-engineered but also not one giant file. tests should be runnable with a single bash command and output files should be namespaced properly so different test runs don't stomp on each other. ping me if you need the exact wire format specs, i can dig them up.

quick follow-up after the team questions came in: for prices, we should make this super boring and consistent. the input is always an integer representing minor units (cents), divided by 100, and we always render with exactly two fractional digits using a dot (.) as the decimal separator, regardless of the input value. so 1 cent → 0.01 and 1000 cents → 10.00. no locale weirdness, no dropping trailing zeroes.

on the checksum side, the graceful fallback is important. when the input cannot produce a valid checksum, like null byte arrays, directories, or non-hashable content, we output the literal string 'null' instead of throwing an exception or bubbling an error up through the adapter. only text-mode inputs with a valid string value should produce a 'checksum=<UPPERCASE_MD5>' line. this is Feature 2 (md5_checksum): the checksum utility that computes uppercase MD5 digests for text inputs and returns the literal string 'null' for inputs that cannot be hashed, defined in feature2_checksum.json.

for custom variables, only positive (greater than zero) integer indexes are valid. negative indexes and zero are silently ignored, no error, just skip the entry. if the dev passes array-form entries, they must have exactly two elements; arrays of any other length are also ignored. and if everything gets filtered out and there’s nothing left, the rendered result needs to be 'custom_variables_json=null', not an empty JSON object '{}'.

for custom dimensions, same general spirit: index zero is always ignored. empty string values are ignored and not stored. setting an existing dimension to null removes it from the collection. values longer than the accepted maximum are truncated before output. only positive indexes with non-empty, non-null values produce output fields named 'dimension<index>'.

also, on structure, yes, this really does need to stay split up. we’ve got multiple distinct responsibilities here — formatting, checksums, serialization, parameter building, URL encoding, bulk dispatch — so this must NOT be a single god file. it should use a multi-file directory structure like src/ and tests/, with the core domain logic separated from the I/O adapter. the execution adapter reads JSON from stdin and delegates to the core, and those pieces need to be physically or at minimum logically separated. that matches the Scale-Driven Code Organization constraint in the PRD: a multi-file repository with src/ and tests/ directories separating core domain logic, the execution adapter, and the test harness.
