## Product Requirement Document

Hey team, we need to build out the logging config engine we've been talking about. Basically operators are really frustrated that every time we deploy to a new environment they have to go into the code and manually change logging stuff — it's causing drift between staging and prod and we've had a couple of incidents because of it. The idea is to have an XML-based config system where the logging behavior is driven by the environment at runtime, not hardcoded. Similar to how we handled the conditional logic in that Spring profile integration we did before — you know the one. We also need a way to pull in shared config fragments so we're not copy-pasting the same blocks everywhere.

On top of that, there's a typed settings binding layer — operators need to be able to set things like whether logging is on/off, what port strategy to use, filter includes/excludes etc. via flat config keys and have those map to a proper structured object with sensible defaults.

The whole thing needs a test harness that reads JSON test cases and writes outputs to files for comparison. Output errors should be friendly normalized categories, never raw exceptions. Needs to be modular — don't just dump everything in one file. Let me know if you have questions, I'll try to dig up the old design notes.

One quick follow-up after the questions that came in. For springProperty, we should reject it and count one error when the 'name' attribute is empty/omitted or when the 'source' attribute is empty/omitted, and those are separate checks, so either one independently triggers rejection. If the declaration itself is valid but the environment key just isn’t there, that is not an error. In that case, when a springProperty's source key is absent from the supplied properties map, the 'defaultValue' attribute is used as the resolved value. If neither the source key is present nor a defaultValue is specified, the variable resolves to an empty string (not an error — this is valid). The output error count is only about structurally malformed declarations, not missing environment keys.

Also on variable resolution, use the ${name} syntax inside pattern strings. So if a pattern contains ${console_pattern_prefix}${console_pattern_body}${console_pattern_suffix}, each placeholder should be replaced with the resolved value of that named variable. And if there are XML entity references in pattern text, decode them before output, so &quot; becomes ".

For springProfile, if the name expression is empty or blank, the block is NEVER kept — just drop it and all its contents no matter what the active profiles are. That’s different from negation, but a block like <springProfile name="!"> is also dropped. Only non-empty name expressions, whether positive or prefixed with !, should actually be evaluated. A profile expression prefixed with '!' is negated: the block is kept when the named profile (the part after '!') is NOT in the active profiles set. For example, name="!logback-access-test-disable-default-console" is kept only when 'logback-access-test-disable-default-console' is absent from active profiles. The '!' must be the leading character of the expression. Also, please follow the springProfile element handling in feature3_spring_profile_selection.json, specifically the negation ('!') prefix logic and nested block cascading-drop behavior described in Feature 3 of start.md.

On the harness side, it writes one file per case to rcb_tests/stdout/<cases-dir>/{filename_stem}@{case_index_zero_padded_3}.txt, for example feature1_spring_property_resolution@000.txt. The <cases-dir> part is there so different case directories don’t overwrite each other. Those files should contain only raw stdout from the program under test, no PASS/FAIL metadata. Default cases dir is 'public_test_cases', and that can be overridden with --cases-dir.
