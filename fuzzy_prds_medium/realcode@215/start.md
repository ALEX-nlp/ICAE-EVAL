## Product Requirement Document

Hey team, we need to wrap up the static repo generator tool we've been building. The core stuff is mostly spec'd out but there are a bunch of edge cases and behavioral details that got left fuzzy in the original notes. Basically the tool needs to handle six main workflows: picking packages from repos, scrubbing unsafe download URLs, generating the metadata files clients consume, loading back previously-generated metadata for incremental builds, rendering the browsable HTML catalog pages, and figuring out archive storage/skip logic. A lot of the tricky bits are around things like — when do we actually drop a package entirely vs just null out one of its URLs? How does the hash in the include filename get computed? What counts as a 'dev' package when we're loading existing metadata back? The abandonment reporting also needs to distinguish between 'replaced by something else' and 'just abandoned', which the frontend folks care about for the notice UI. Also there's that pattern we used in the login module for handling the host-matching rules — same idea applies here for the strip-hosts config, just refer back to how we did the CIDR + local path matching there. Output format for every command needs to be exact since it feeds into downstream tooling. Would love to get this fully nailed down before the sprint ends.

One extra pass on the fuzzy bits so nobody has to guess at the last minute: a package only gets removed from output entirely if both its dist URL and its source URL get stripped, meaning both end up null. If one of them survives, we still keep the package and just show the missing one as 'dist=null' or 'source=null' in the output line.

For the include filename, the hash in something like 'include/all$82aa8b329f93e0204b3dc0dd52891a59b81a3196.json' is a SHA-1 hash of the JSON content of that include file, and the filename shape is always 'include/all$<sha1hash>.json'. If a homepage path is present, the providers URL template changes, but the include hash part is still SHA-1 of the file content. Also on package selection output, the selected packages are sorted lexicographically ascending by the full package identifier string, which is the combination of name and version, basically the full name-version string exactly as it appears in the line, like 'vendor/project-alpha-1.2.3.0'.

On the metadata side, providers_url only shows up when 'providers: true' is set in the config. When it does, it uses '/sub-dir/p/%package%$%hash%.json' if a homepage is set, or 'p/%package%$%hash%.json' if there is no homepage. metadata_url always sticks to the 'p2' path pattern. Worth noting too that the include file hash changes when providers mode is enabled, because the root packages.json content is different in that mode.

For strip-hosts, '/local' is its own special matcher and it strips any URL whose host resolves to localhost or 127.0.0.1, so hosts like '127.0.0.1' and 'localhost' should match there. CIDR entries like '192.168.0.0/24' are a separate rule type and strip URLs whose host IP falls inside that subnet. Same basic behavior we talked about before: '/local' means loopback detection, while CIDR strings mean subnet range matching with IP address comparison.

And for loading existing metadata back in, dev=true is just a string-prefix check on the pretty version: if the version starts with 'dev-' like 'dev-master', mark it dev=true, otherwise dev=false. Nothing fancier than that, and it is not based on any normalized version form.
