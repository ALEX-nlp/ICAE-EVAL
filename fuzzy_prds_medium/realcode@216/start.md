## Product Requirement Document

Hey team, we need to build out this security header middleware engine we've been talking about. The basic idea is that you give it a config describing what protections you want on your HTTP responses, and it spits out the right headers automatically. No more hand-rolling these things and getting the formatting wrong.

A few things I know we need: there should be a 'batteries included' mode that just turns on everything sensible at once for teams that don't want to think about it. Individual headers should also be toggleable separately. Some of the headers have dependencies on each other — refer back to how we handled the cross-origin isolation stuff in the embedder policy discussion, the same gating logic applies here. We also need path-based rules so the data-clearing header only fires on specific routes like logout pages.

The engine reads a JSON blob on stdin and writes headers to stdout, one per line, sorted alphabetically. If something is misconfigured it should spit out a neutral error token — no stack traces, no runtime garbage, just clean error codes the frontend team can parse.

Edge cases to handle: what if someone passes in no config at all? What if they want certain paths completely ignored? The CSP stuff has some tricky quoting and deduplication rules too that we need to get right. Output should be totally deterministic.

Can someone scope this out? I think the public test cases folder has examples of expected outputs we can validate against.

One extra pass on details the team asked about: if `with_config` is false, or the engine is invoked with no configuration, the output must be exactly `error=missing_configuration\n`. No headers, no additional lines. Same deal on the cross-origin gating we talked about before: `Cross-Origin-Embedder-Policy` may only be emitted when `cross_origin_resource_policy` is also present in the same configuration. If embedder policy is requested without resource policy, the engine must reject the configuration and output exactly: `error=embedder_policy_requires_resource_policy\n` and nothing else.

Also, when the `preset` field is set to `default`, the `headers` object is completely ignored. The preset applies its own fixed opinionated configuration regardless of what is specified in the headers field, so these are mutually exclusive modes. That default preset emits exactly 11 headers in alphabetical order: `Cache-Control: max-age=0,no-store | Content-Security-Policy: script-src 'self';object-src 'self';block-all-mixed-content;upgrade-insecure-requests; | Cross-Origin-Embedder-Policy: require-corp | Cross-Origin-Opener-Policy: same-origin | Cross-Origin-Resource-Policy: same-origin | Referrer-Policy: no-referrer | Strict-Transport-Security: max-age=31536000;includeSubDomains | X-Content-Type-Options: nosniff | X-Frame-Options: deny | X-Permitted-Cross-Domain-Policies: none | X-XSS-Protection: 0`.

A couple formatting specifics too. For `x_frame_options.directive`, when it is `'allowfrom'`, the emitted header value is exactly `allow-from: (<domain>)`, including the colon-space after `allow-from` and the parentheses around the domain. Example: `X-Frame-Options: allow-from: (https://example.com)`. For `Expect-CT`, the value always starts with `max-age=<n>` with default `86400`. If `enforce` is true, append `, enforce`. If `report_uri` is non-empty, append `, report-uri="<uri>"`. If `report_uri` is an empty string, just ignore it silently. Example: `max-age=86400, enforce, report-uri="https://ex.com/r"`.

For HSTS, defaults are `max_age=31536000, include_subdomains=true`. When `include_subdomains` is true the value is `max-age=<n>;includeSubDomains` with no space around the semicolon. When `include_subdomains` is false the value is just `max-age=<n>`. And for CSP legacy support, when `content_security_policy.legacy_compat` is true, the policy value is mirrored into an additional `X-Content-Security-Policy` header with the same value as the `Content-Security-Policy` header, and both still need to land in the final output in alphabetical order with everything else.

Last thing, the path-based clearing config the team asked about is the `clear_site_data_paths` and `clear_site_data_add_paths` shapes under `headers`, and the matching should use longest-prefix path matching logic. That should line up with the route-specific behavior we wanted for logout-style pages while still keeping output deterministic.
