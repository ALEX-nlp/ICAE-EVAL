## Product Requirement Document

We need a reusable type-ahead text field component. As a user types, a list of matching suggestions should appear below the field and update in real time. The user should be able to navigate the list with arrow keys, pick a suggestion with Enter or by clicking, and dismiss the list by pressing Escape or clicking away. When an item is highlighted by keyboard the field text should mirror that item; when the highlight is cleared (by pressing past the last item) the original typed text should be restored. Mouse hover should also highlight items, and mousing away should clear the highlight. The component needs to support both a flat list and a grouped multi-section layout. The matching logic uses a case-insensitive prefix rule — look at how the existing search bar in the app filters candidates and apply the same trimming convention. The component must emit notifications about focus, blur, value changes (including which interaction method caused the change), selection events, and requests to fetch or clear suggestions. Keyboard focus should stay on the field after clicking a suggestion by default, and an option should exist to change that. Accessibility roles and attributes must be maintained throughout.

One extra detail on the keyboard loop since the team asked: the Down cycle has count+1 steps. Down from item 0 moves to item 1, and so on, and Down from item (count-1) moves to the 'none' state, and one more Down returns to item 0. Up is the mirror here: Up from item 0 goes to 'none', then to item (count-1). That should line up with the earlier note about restoring the originally typed text when the highlight clears.

Also, for suggestion visibility, the default shouldRender ('default') behavior is that suggestions are eligible when the trimmed value is non-empty. 'nonEmptyNoLeadingSpace' means trimmed value non-empty AND first character is not a space. 'alwaysTrue' means always eligible. This is just the render eligibility piece and should still follow the same matching behavior already described.

One last click behavior clarification: when focusInputOnSuggestionClick is false, clicking a suggestion does not return keyboard focus to the input field, and the component emits a blur notification with focusedSuggestion set to the clicked suggestion's name.
