## Product Requirement Document

Hey team, we need to get the Sentry integration for our Laravel app properly wired up. Right now devs are complaining that when something breaks in prod they have no idea what was happening before the crash — no breadcrumbs for cache hits/misses, no DB query trail, nothing about what HTTP calls were going out. It's basically a black box. We talked about this a while back when we did the logging channel work, same idea applies here — we want all the usual framework activity to show up automatically without people having to sprinkle manual instrumentation everywhere.

A few specific things that came up in the last retro: the transaction names on errors are basically useless right now (they show weird internal strings instead of actual route shapes), and the DSN/environment config isn't reliably picking up the right values at runtime. Also someone mentioned that filesystem operations should be observable both as traces AND as the other thing (you know, the lighter-weight diagnostic trail), depending on what's configured.

We also need the SQL stuff to respect the privacy settings — sometimes we want bindings, sometimes we don't. Same pattern as the command breadcrumb feature we scoped out. Oh and for the lightweight router, the path params need to be replaced properly in the transaction name — there was a bug reported where repeated param values only get replaced once instead of twice. Please make sure edge cases like that are handled.

One extra pass on the config/runtime bits since a few people asked how literal this needs to be: for DSN, we need to split the full string into four separate outputs every time. So from something like https://publickey:secretkey@sentry.dev/123 we should produce dsn_base, dsn_project_id, dsn_public_key, and dsn_secret_key, with dsn_base being the scheme+host like https://sentry.dev, dsn_project_id being the path segment like 123, dsn_public_key being the username part, and dsn_secret_key being the password part. If DSN is null or missing, all four should print as the literal string 'null'. On environment, if the 'environment' key is missing or empty, we fall back to the framework runtime value, which is 'testing' in the harness. If a non-empty environment is provided, that wins completely, and the output line should be exactly environment=<resolved_value>.

On the filesystem piece, the two switches are separate: record_spans and record_breadcrumbs. If record_spans=true, each filesystem operation creates a tracing span. If record_breadcrumbs=true, each operation creates a breadcrumb. They can both be on, both off, or just one on. Also worth calling out that there is always one baseline span for the root transaction, so span_count never goes below 1 even if no filesystem spans are added. For the operation payloads, every filesystem span/breadcrumb includes 'disk' and 'driver', for example 'local' and 'local'. For 'put', metadata should include 'path', 'options' as an empty array, 'disk', and 'driver', and the description is '<path> (<size> B)'. For 'get', include 'path', 'disk', and 'driver', and the description is the path. For 'delete' with one path, same shape: 'path', 'disk', 'driver', description is the path. For 'delete' with multiple paths, use 'paths' as the array plus 'disk' and 'driver', and the description is '<count> paths'. For 'files', metadata is 'directory' which is null if not passed, 'recursive' which defaults to false, plus 'disk' and 'driver', and the description is null. Also just to restate the distinction people kept asking about: Breadcrumbs are the Sentry breadcrumb trail recorded via SentrySdk::getCurrentHub()->addBreadcrumb(), separate from tracing Spans in the transaction tree.

For DB spans, the op should be 'db.sql.query' and the description should just be the query text. Metadata needs server_address and server_port from the connection config. For SQLite or in-memory connections, both server_address and server_port should be null. In the harness, mysql is at host-mysql:3306 and mysql_url is at host-mysqlurl:3307. And since the root transaction span is always there, once a query span is recorded the total span_count is at least 2. On the error handling side, when error_types is not provided, the resolved mask must be -1, which is the sentinel for all error types. If a numeric value is supplied, pass it through exactly as-is, like 24575.

A couple more behavior details too. For exception context, we only attach it if the thrown exception has a getContext() method and that method returns an array. If there is no such method, so context_mode=none, then exception_context is null. If it returns a proper associative array, so context_mode=array, serialize that array as JSON and attach it. If getContext() returns something that is not an array, so context_mode=non_array, treat it as absent and set exception_context to null. For log breadcrumbs, the category should be 'log.<level>' like 'log.debug', and breadcrumb_level should be the raw level string like 'debug'. The message is the log message string, and metadata is the log context exactly as passed, including cases where it is an array like ["1"] serialized as a JSON array.

And lastly, the auto-wiring point is broader than just DB and HTTP. The integration should register listeners through Laravel’s event system for HTTP client response events, cache hit/miss/write/forget events, database query events, log message events, console command start events, and filesystem adapter operations, so we get all of that automatically without anyone adding manual per-call instrumentation.
