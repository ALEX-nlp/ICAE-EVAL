## Product Requirement Document

Hey team, we need to build out that test runner thing we've been talking about. The basic idea is devs should be able to write their test steps like prose — setup, do stuff, check stuff, clean up — and it should run them in order without them having to wire everything up manually for every single data row. Right now people are copy-pasting test methods all over the place and it's a nightmare.

A few things that are definitely needed: it should handle multiple data rows elegantly (like that table-driven approach we used in the auth service refactor), steps should be skippable individually or the whole test at once with a reason attached, and if one step blows up the later ones shouldn't just silently vanish — they should show up clearly as blocked. There also needs to be some kind of resource cleanup that runs in reverse order no matter what, even if things fail halfway through.

One tricky thing the team mentioned: sometimes an assertion is "destructive" (it changes state) so it needs its own fresh context to run in, otherwise later steps break. We also need a per-step time cap so slow steps don't hang the whole suite. Oh and someone raised a thread-safety concern — if two test threads register steps at the same time it shouldn't corrupt anything. Make sure the output for each step result is structured and machine-readable. Check how we handled the default value generation in the earlier parameter binding work for reference.

One follow-up from the questions that came in: for the parameter binding/defaulting part, we should stick to the same behavior we talked about before. If a value has to be generated instead of being supplied, integer-like types default to 0 and reference/object-like types default to null. We also need to carry that through in the input representation as BehaviorInputValue, with generated_default=true when we auto-fill and generated_default=false when the caller actually supplied the value. If an example row comes in short and has fewer values than the scenario declares, we bind what it did supply to the leading parameters in order, then fill each remaining trailing parameter with its generated default using the same rules above. That should still run through normally and be reported as completed, not as an error. Also on the example-driven side, each tabular example row runs the scenario body once with bound parameter values, as specified in feature2_1 through feature2_5 test cases, and the Feature 1 InputValue representation is BehaviorInputValue with generated_default flag, integer defaults to 0, object defaults to null — see feature1_1 and feature1_2 test cases.

On time caps, a step can declare a per-step time limit in milliseconds. If it finishes before that, it just passes normally. If it runs long, we should normalize that into a timeout failure and make sure the result carries error='timeout' and the limit_ms value that was exceeded. If a step has no limit, then it’s unconstrained.

A couple more lifecycle details too. Cleanup callbacks should always run after scenario execution in reverse registration order, so last registered runs first, whether the scenario succeeded or failed. If one of those cleanup callbacks throws, we still report that as an error result AFTER the successful step results, and we keep going so the remaining cleanup callbacks still execute in reverse order. Same basic idea for disposable resources: they need to stay usable for all scenario steps, then get disposed exactly once after the scenario completes, in reverse creation order. If a dispose call throws, we still dispose all the other resources in reverse order, and each disposal failure gets surfaced as its own individual error result.

On the thread-safety concern, the registration state needs to be isolated per thread. Two threads should be able to register the same kind of step sequence concurrently without any cross-thread contamination or random errors, and that applies across full sequences, action steps, isolated assertions, observations, and skipped steps. For those concurrent registration checks, the output should show threads=2 and error=none. Also, when multiple example rows run, the 'arguments=' lines in stdout need to come out in a stable sorted order, not whatever arbitrary execution order happened that time, so the output contract stays deterministic even under concurrent runs.

Last thing: if something goes wrong while constructing or defining a scenario, that should turn into an error result instead of escaping the runner. So if the scenario body throws during registration, or a feature class can’t be instantiated, each of those should produce exactly one error result with result_count=1, error_count=1.
