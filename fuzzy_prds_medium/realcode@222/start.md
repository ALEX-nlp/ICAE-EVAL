## Product Requirement Document

Hey team, we need to build out the annotation processing pipeline for the DI wiring system. Basically the idea is that feature modules should be able to "register" themselves without the root module needing to manually list everyone — same pattern we did for that login module auto-discovery thing a while back, you know the one. The processor needs to read annotated Kotlin source files, figure out what's being contributed/merged/wired, and spit out a deterministic text report so we can test it without depending on any specific test framework.

The input comes in as JSON on stdin (list of source strings plus some query fields), and the output is plain text lines — success case lists out what got generated, failure case gives a clean domain-level reason with no stack traces or file paths leaking out.

A few things I'm fuzzy on that someone will need to nail down: what exactly the output lines look like for each feature type, how nested types show up in the report, what happens with visibility modifiers on the various contribution types, and whether there's any special handling needed when you're in that cross-platform common-code mode. Also need to make sure the test harness writes outputs to the right place and does pass/fail comparison properly. The batch loop should stay in-process for speed but exit cleanly even if compiler threads linger.

Quick follow-up since a few people asked what the report needs to look like exactly. On success the very first line is `status=compiled`. After that, for anything we inspect, the shape is fixed: `contribution origin=<fqn>`, then sorted `extends <fqn>` lines, then sorted `provides from=<fqn|none> to=<fqn> kind=<single|multi>` lines. For merged components: `merges <fqn>` lines (sorted). For annotations: `contributing-annotation origin=<fqn>`. On rejection: `status=rejected`, then `error=invalid_usage`, then one `reason=<domain message>` line per diagnostic. If something goes wrong on our side in an unexpected way, that should be `error=internal` and not anything noisy.

Also, nested names in the report should always be normalized to dotted form. So if there’s an inner interface like `SubcomponentInterface.Factory` in package `com.example`, it needs to show up as `com.example.SubcomponentInterface.Factory`, not with `$` or `/` style separators. Same general cleanliness rule for diagnostics too: strip any source-location prefix, and never print host-language exception class names, stack traces, or file-system paths. The only thing that should come out as failures are the processor’s own domain sentences in `reason=` lines.

On the input side, the JSON command schema is `sources` as the required array of Kotlin source strings, plus optional `inspect`, `merged`, `annotation`, and `multiplatform`. `inspect` is the array of FQNs to look up generated stubs for, `merged` is the array of FQN component names to report merged contributions, `annotation` is the array of FQN annotation names to report discovery markers, and `multiplatform` is the boolean switch for KMP common mode.

A couple validation details got called out too. All contributed declarations have to be public. So a private interface using `@ContributesTo` is rejected with `reason=Contributed component interfaces must be public.`. An internal annotation class using `@ContributingAnnotation` is rejected with `reason=Contributing annotations must be public.`. And non-public factory functions for `@CreateComponent` are rejected with `reason=Factory functions for components annotated with \`@CreateComponent\` must be public.`

For `"multiplatform": true`, we’re compiling in common Kotlin-Multiplatform mode, so there are a few extra rules around factory functions annotated with `@MergeComponent.CreateComponent`: they have to be public and use `expect`. The return type has to point at a `@MergeComponent` target, not something hand-wired with `@Component`. If there’s a receiver, it has to be the KClass of the component and the argument/arity has to match.

And just to anchor this back to the original behavior we want, this is still the same contribution + merge flow from start.md: declarations opt in with `@ContributesTo`, `@ContributesBinding`, and `@ContributesSubcomponent`, we emit discoverable stubs into a fixed lookup package, and those are what `@MergeComponent` picks up later.
