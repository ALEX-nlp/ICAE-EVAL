## Product Requirement Document

Hey team, we need to build out a bitmask flag toolkit that our platform teams keep reinventing every quarter. The core idea is simple: developers declare a named set of on/off options backed by an integer, and the library handles all the combining, querying, and printing so nobody has to hand-roll bit math anymore. We've had a few incidents where unknown bits sneaking through caused silent data corruption downstream, so there needs to be a clear story around how 'stray' bits are handled — some callers want them rejected, some want them silently stripped, some want to keep them. Also refer to how we handled the 'external catch-all' pattern in the options registry module, that same concept applies here for sets that need to cover every possible bit. The toolkit should support the usual set algebra people expect (combine, subtract, overlap checks), plus the ability to iterate which named flags are active in a value, serialize/deserialize to a human-readable pipe-separated format, and a classify/dispatch helper for routing values through labeled buckets. There's also a non-ASCII flag name situation that came up from the i18n team — make sure that works. Output formatting for the adapter layer needs to be consistent across all operations. Let's get this scoped out properly.

One quick follow-up from the questions that came in: for unknown bits, there are three construction policies and they should behave pretty literally. 'checked' rejects any integer that contains bits outside the declared flags and returns an absent/none result, 'masked' silently drops anything outside the known flags, and 'raw'/'retain' keeps the integer exactly as-is including unknown bits. The set’s known bits are just the union of all declared flag bit patterns. That matters for the algebra too: union, difference, and intersection should preserve whatever unknown bits are already in the operands, but complement is always masked to known bits and never creates unknown bits.

Also, for the catch-all behavior, the flag is declared with an empty string as its name and 255 as its bit pattern. Since known bits are the union of declared flag bit patterns, any set that includes that catch-all ends up with known bits = 255, so every possible 8-bit value is treated as known. That catch-all should not come back through name lookup though, so lookup by empty string returns none. If it shows up in catalog/decompose_named output, it should appear exactly as 'name= bits=255'. The two catalog entries that use this are 'external' with A=1, B=2, C=4, ABC=7, catch-all=255, and 'external_full' with only the catch-all=255. This same detail should be reflected in the Flag Set Catalog section of start.md for 'external' and 'external_full', where the empty-name flag ('') with bits=255 means all 256 possible 8-bit values are known.

On decomposition, the order is declaration order, and that is what decides how combined flags behave. A flag is yielded when all of its bits are present in the remaining unaccounted bits, and once yielded, those bits get removed from the remaining pool. So if ABC=7 is declared before A=1, B=2, C=4, then ABC comes out first and the individual parts do not. If A=1, B=2, C=4 are declared first, then those come out individually and ABC is skipped. Any leftover bits that do not match a named flag should come out as one trailing element in decompose, but not in decompose_named. Also, ZERO=0 should never be yielded.

For text in and out, the delimiter is exactly ' | '. Parsing should split on '|' and trim whitespace on each token. Tokens can be exact, case-sensitive flag names or '0x'-prefixed hex literals. The empty string means bits=0. The three parse variants are also intentionally different: 'parse' keeps unknown hex bits, 'parse_masked' drops hex bits outside known flags, and 'parse_named' rejects any hex token with error=invalid_hex_flag. If the token is a flag name we do not recognize, that should always be error=unrecognized_named_flag. On the way back out, 'to_text' emits named flags in declaration order and then adds any leftover bits as a hex literal, 'to_text_masked' drops unknown bits and never emits a hex literal, and 'to_text_named' only emits known flag names. Empty value should serialize to an empty string in every variant.
