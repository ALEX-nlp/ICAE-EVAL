## Product Requirement Document

Hey team, we've been getting complaints from platform folks that hand-authoring all these ingress gateway configs is a nightmare — people keep shipping broken setups where traffic just disappears into a black hole, TLS stuff silently fails, or streaming connections drop because someone forgot a flag somewhere. We need a config builder toolkit that lets devs snap together the different pieces (backends, routing rules, host groupings, listeners, the connection handler thingy, etc.) without having to know all the safe defaults by heart.

The outputs need to be consistent — like, if you ask for HTTP/2 it should say so, if you don't it should still say something. Same deal with the websocket situation we always forget about (remember how we fixed that in the login module proxy config? do something like that). TLS material should come back out exactly as it went in, no funny business.

Also need something that reads the operator settings map and turns it into a typed config — there's one boolean setting for logging that has a default, and bad values should give back a clean error rather than blowing up. The whole thing should be split up sensibly, not one giant file. The JSON test harness stuff is just for black-box testing the adapter layer, the actual builder logic should stay decoupled from all that stdin/stdout wiring.

One more pass on the details the team asked about. On the endpoint side, every load-balancer endpoint must always emit `ipv4_compat=true` regardless of the input address. That’s a hardcoded platform default, not something we infer, omit, or make configurable. Same idea on routes: every forwarding route must always include the line `upgrade type=websocket enabled=true`. It’s unconditional and does not depend on any input field, so it should never be missing from a route’s output.

For listeners, keep the naming totally predictable across plain HTTP, HTTPS, and SNI. The name is always derived as `listener_<port>` from the bound port number, so for example port 8080 produces `listener=listener_8080`. On clusters, the `http2` field must always be present. If HTTP/2 is not requested, emit `http2=false` and do not skip the field. If HTTP/2 is requested, emit `http2=true`, and that’s also the case where we include the explicit upstream protocol options at the proxy level.

On the settings parsing bit, if the settings map has a non-boolean string value for `enable-service-access-logging`, the output needs to be exactly two lines: `error=invalid_boolean` and then `key=enable-service-access-logging`. No stack traces, no runtime exceptions leaking through. Same style for bad cluster discovery input: if the cluster builder gets an unrecognized discovery type keyword, the output must be exactly two lines, `error=unknown_discovery_type` followed by a line naming the offending value, and in that error case we should not emit any other cluster fields.

Also confirming the TLS handling is strict: certificate chain and private key values must be embedded and output verbatim, exactly as supplied in the input, byte-for-byte, with no encoding, transformation, or truncation. That applies both to the single-cert HTTPS listener and to each SNI match entry.

And just to remove any ambiguity on structure, this still should not turn into one giant file. Builders should stay split by resource type like endpoint/cluster, route, virtual host, listener/TLS, and connection manager. The settings parser stays independent, and the execution adapter for stdin/stdout/JSON stays physically separate from the core builders. Also repeating the two hardcoded behaviors since they came up more than once: the IPv4-compat flag (`ipv4_compat=true`) that must always be hardcoded as true in the lb_endpoint builder, as specified in Feature 1 of the PRD, and the unconditional `upgrade type=websocket enabled=true` line that every forwarding route must emit, defined in Feature 4 of the PRD.
