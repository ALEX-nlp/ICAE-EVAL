## Product Requirement Document

Hey team, we need to build out that observability SDK thing we've been talking about. Basically devs are complaining that every time something breaks in production they have no idea what was happening beforehand, who the user was, or how bad the issue was — they're hand-rolling all their own error payloads and it's a mess. We want a clean SDK that lets you fire off a one-liner to report a crash or a log message, and have it automatically carry context about the session (user info, severity, tags, that sort of thing). There should be some way to attach context globally so you don't have to repeat yourself every call, but also a way to scope context to just one specific capture. Oh and we need that breadcrumb trail thing — like how Sentry does it — so you can see what happened leading up to the crash. Refer to how we handled the hook/intercept pattern in the login module last time, same idea applies here for letting devs inspect or drop events before they go out. The test runner should work similar to the pipeline validation harness we already use. Output format and field naming conventions should follow whatever we used in the enrichment model spec — just make sure structured data like nested context objects comes out in the right shape. One big thing: the adapter that reads commands and produces output needs to be totally separate from the core logic, same separation-of-concerns principle we always talk about.

One extra pass on details the team asked about: if someone captures a message and doesn’t explicitly set a severity, we should auto-fill INFO, and that should show up exactly as event[i].level=INFO. For exceptions, anything going through capture_exception should always record the type as RuntimeException, with event[i].exception[0].type=RuntimeException and event[i].exception[0].value=<the message string supplied>.

Also, every output needs to start with captured_events=<n>, where n is the total number of captured events, including 0. That line should always be there and it needs to come before any event or breadcrumb lines. On the hook side, yes, we do want the pre-send inspection hook to receive the fully assembled, normalized event object right before dispatch. The execution adapter is what uses that to intercept events for output rendering, and that same hook can also suppress events. Breadcrumbs have their own separate pre-breadcrumb hook just for breadcrumb processing. This is the same before-send and before-breadcrumb callback flow from the PRD, meaning SDK-level callbacks (beforeSend / beforeBreadcrumb) get the assembled event or breadcrumb object and can return it modified or null to suppress it, wired up in the execution adapter to collect events for output rendering.

For scope behavior, a local scope passed directly on one capture call via the 'scope' block only applies to that one event and must not leak into later captures. Global scope set through configure_scope sticks around for all subsequent events. Those two paths are fully independent, so please keep that separation clean. And for breadcrumbs, anything that makes it through the pre-processing hook should be rendered after all captured event lines, using breadcrumb[i].message=<text> with i in zero-based insertion order, and those stay as separate lines rather than being folded into the event output.

Last thing, the runner wiring should match the path we discussed: the script at rcb_tests/test.sh reads *.json case files from a cases directory, feeds the input to the adapter via stdin, and writes raw stdout to rcb_tests/stdout/<cases-dir>/<stem>@<index>.txt so it can be compared directly against expected_output.
