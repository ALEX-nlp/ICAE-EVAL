## Product Requirement Document

We need a small toolkit of HTTP payload helpers. First, a way to serialize a deeply nested mapping or ordered list of key-value pairs into a URL-encoded query string — nested keys should use square-bracket notation to preserve structure, and malformed entries (wrong number of elements) should be rejected with a clear error. Second, a user-agent string builder that takes a primary product name and version plus optional extra products and joins them with spaces — reject malformed extras rather than silently dropping them. Third, a multipart form-data serializer that packs several fields (including file-style fields with their own filename, content type, and extra headers) into a properly framed multipart body using a caller-supplied boundary — the output should report the content type (which embeds the boundary) followed by the encoded bytes. Fourth, a multipart decoder that splits a multipart body back into its constituent parts, preserving headers and content for each part — reject non-multipart content types with a clear error naming the offending media type. Also, a single-segment parser that splits headers from body content at the blank-line separator — handle it the same way the decoder handles missing boundaries: reject with a clear neutral error. Finally, a sized streaming reader that wraps an iterator of byte chunks and satisfies exact sized reads, drawing across chunk boundaries.

A couple small clarifications from the questions that came back. For the query helper, if the input can’t actually be understood as a collection of key/value pairs, like a bare scalar or a tuple with the wrong number of elements, the output should be exactly 'error=invalid_query\n'. Same idea for the user-agent builder: if the extras are malformed, don’t quietly skip them, return exactly 'error=invalid_extras\n'.

For multipart encoding, the first line of multipart_encode output is 'content_type=multipart/form-data; boundary=<boundary>\n' followed immediately by the encoded body bytes. The body uses CRLF (\r\n) for all structural newlines. The shape is '--<boundary>\r\nContent-Disposition: form-data; name="<name>"\r\n\r\n<value>\r\n' per simple field, closing with '--<boundary>--\r\n'.

On the decode side, the check for whether something is multipart is only about the top-level media type being 'multipart', and that check is case-insensitive. If it isn’t multipart, the output is 'error=not_multipart\nmimetype=<media_type>\n'. For example 'image/jpeg' produces 'error=not_multipart\nmimetype=image/jpeg\n'.
