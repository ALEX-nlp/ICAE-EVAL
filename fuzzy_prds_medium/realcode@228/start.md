## Product Requirement Document

We need a small, dependency-free vocabulary library for our package-management tooling. Three things need to be standardized across all backends: which operations a backend supports, how package versions are numbered, and what errors look like when something cannot be found. For backend capabilities, we want a combinable flag set — each operation gets a unique power-of-two integer, they can be OR-ed together, and the text representation lists the contributing names in ascending value order. There's a 'no operations' sentinel value that renders as 'None'. For version schemes, it's a fixed ordered list — each scheme has a canonical name and a position in the list, and that position is what gets transmitted when we exchange scheme values as integers. For errors, we have three 'not found' variants (package, source, provider) plus a base error. Each not-found error has its own default sentence and remembers the offending name. The rendered message appends a parenthesized suffix with the kind and name when a name is provided. A custom message can replace the default sentence while the name suffix is still appended. The base error wraps an optional inner cause whose message surfaces separately. This library should be implemented the same way our existing provider-registry module handles capability lookups — small, pure, no I/O.

One small follow-up from the questions that came in: for the capability flags, the exact values are None=0, Find=1, Get=2, Publish=4, Install=8, Save=16, Uninstall=32, Unpublish=64, Update=128, GetSource=256, SetSource=512. These are meant to stay as the canonical power-of-two values, and when we turn a combined value into text we list the contributing names in ascending value order, comma-and-space separated. Also just to be explicit, None=0 is the sentinel and renders as 'None'.

For version schemes, the ordinals are zero-based ordinals in declaration order, so the exact mapping is AlphaNumeric=0, Integer=1, MultiPartNumeric=2, MultiPartNumericSuffix=3, SemanticVersion=4. That integer position is still the thing we send around when scheme values are exchanged as integers.

On the error message shape, the format should be exactly '<default sentence> (<Kind> \'<name>\')'. Where Kind is 'Package' for PackageNotFoundException, 'Source' for PackageSourceNotFoundException, 'Provider' for PackageProviderNotFoundException. Example: 'Package not found. (Package \'NodeJS\')'. Same idea as before: if there’s a custom message, that replaces the default sentence, and if there’s a name, we still append the same parenthesized suffix.
