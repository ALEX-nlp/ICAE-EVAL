## Product Requirement Document

Hey team, we need to build out that autoscaler config toolkit we've been talking about. Basically the idea is that ops people shouldn't have to write all this boilerplate parsing and merging code every time they stand up a new environment. The system needs to handle loading settings from files or directories (similar to how we handled the multi-file approach in the deploy pipeline thing we did before), merging layered configs where later values win, and translating those scheduler policy documents into something the scaling engine can actually consume. There's also some canonicalization step where we fill in sensible defaults — I think the built-in metric source and target names matter here but I don't remember the exact names off the top of my head, check the existing code. One tricky part: when the built-in metric source is active and the query looks like a short form (no slashes in it), we need to expand it somehow using group and job info. Validation also needs to happen and report multiple errors at once rather than stopping at the first one. The output needs to go to stdout in a consistent line-by-line format. Editor temp files should be ignored when scanning directories. Please make sure the architecture is clean — separate concerns properly, don't dump everything in one file.

A couple follow-ups from the questions that came in. For the default config output, we should emit these exact lines, each newline-terminated and each on its own line: log_json=false, log_level=info, plugin_dir_suffix=/plugins, default_evaluation_interval_ns=10000000000, nomad_address=http://127.0.0.1:4646, http_bind_address=127.0.0.1, http_bind_port=8080, apm_plugins=nomad-apm:nomad-apm, target_plugins=nomad-target:nomad-target. Also confirming the built-in APM plugin name is 'nomad-apm' and the built-in target plugin name is 'nomad-target', and those are the defaults both in the normal config output and in the policy canonicalization path.

On merging plugin declarations, if two layers declare plugins with the same Name field, the later layer replaces the earlier one. If the names are different, both stay. In the rendered output, plugins within each category need to be sorted alphabetically by name, and the line format is exactly category_plugins=name1:driver1,name2:driver2 args=[...] config={key=value,...}. Args only show up as args=[arg1,arg2] when present, and config only shows up as config={key=value} when present, with config keys sorted alphabetically. The example to follow is: apm_plugins=influx-db:influx-db,nomad-apm:nomad-apm,prometheus:prometheus args=[all-the-encryption] config={address=http://prometheus-new.systems:9090}

For canonicalization, if the input is null/nil, the output is exactly: policy=<nil>\n. If it’s a non-null but empty policy object {}, it should receive all defaults: enabled=false, source=nomad-apm, evaluation_interval_ns=10000000000, target_name=nomad-target, target_config={}, strategy_name=, strategy_config={}. And just to make the interval piece explicit, the default evaluation interval is 10 seconds expressed as 10000000000 nanoseconds, and that same value applies in both the default runtime config and policy canonicalization whenever no evaluation interval is provided.

On file loading, Feature 3 is just load a single .hcl file or scan a directory, sort eligible .hcl files by name, merge them in order, ignore editor temp files and non-.hcl extensions — as specified in feature3_configuration_file_loading.json. When loading from a directory, only .hcl files are eligible, so things like files with a ~ suffix should be ignored. If the path does not exist or is marked as missing, the output is exactly: error=path_not_found\n.
