## Product Requirement Document

hey team, we need to get this binary parsing adapter thing wrapped up. the idea is basically that someone hands us a JSON blob describing what kind of binary data to read, and we run the appropriate parsing logic and spit out a flat key=value output. think of it like how we handled the config reader on that older serialization module — same vibe where inputs drive behavior without the caller needing to know internals.

the tricky parts are around the edge cases: like what happens when a marker byte doesn't match what we expect, or when a read fails partway through a variant and we need to decide whether to surface each individual failure or just collapse it into a single 'nothing matched' kind of message. also there's some nuance around reads that are allowed to fail gracefully vs ones that should hard-fail the whole record.

we also need to handle things like text inside binary streams (including the wide character flavor), sequences with separators between items, and cases where an earlier value in the stream controls how many bytes come later without itself showing up in the output.

output format should be stable and consistent — downstream tooling depends on it. errors need to include position info where relevant. please make sure the core parsing logic stays totally separate from the stdin/stdout plumbing, we got burned on that last time. ideally this is structured as a proper multi-file project, not one giant file.

one more pass on the edge cases since a few folks asked for the exact behavior. when a value assertion fails, the output needs to be exactly three lines: 'error=assertion_failed', 'pos=<byte position where the field was read>', and 'message=<the declared condition expression as a string, e.g. a == 1>'. . Custom failures use 'error=custom_failure' instead. the byte position here is where that field read began, not where it ended. for a marker or magic byte mismatch, keep it tighter: output exactly 'error=bad_magic' and 'pos=<byte position of the failing marker byte>'. no message field in that case. also padding bytes are just skipped silently and never show up in output.

for the position-tracking behavior, the 'pos_value' feature is meant to work like this: read a u16 big-endian at offset 0 with value 0xFFFE = 65534, then read a u8 at offset 2 as a separate tracked value with initial_value=253 and tracked_pos=2. that tracked value then gets mutated to 1, so mutated_value=1, and then cloned so clone_value=1 and clone_pos=2. the output for that path is exactly 'status=ok', 'a=65534', 'tracked_pos=2', 'initial_value=253', 'mutated_value=1', 'clone_value=1', 'clone_pos=2'.

also worth being explicit about short reads: if something asks for more bytes than are left in the stream, the cursor reads however many bytes are actually available and fills the rest of the buffer with zeros. 'read_len' is the amount actually read, not the requested amount, 'read_bytes' is the full buffer including the zero-padding, and 'read_pos' is the cursor position after the read. on the endian naming bit, the canonical names are 'Big' and 'Little' exactly, and the output for that feature is just one line: 'name=Big' or 'name=Little'. no 'status=ok' prefix there.

for the zero-field gate record, it literally reads no bytes at all. it only accepts or rejects based on the caller-supplied boolean precondition. if it accepts, output is 'status=ok'. if it rejects, output is 'error=assertion_failed\npos=0\nmessage=<precondition expression>'. there are no payload fields for that record type. similarly, if a field exists only to control the size of a later read, it should NOT appear in the output. the example here is the 4-byte big-endian length prefix that determines how many bytes go into 'values'; only the derived result gets emitted, and values are printed as comma-separated decimal integers.

and just to lock down structure, this really does need to stay multi-file and separated cleanly: a core parsing library module, a domain adapter/router that maps feature names to parse calls, an I/O adapter that handles stdin JSON reading and stdout formatting, and an error type hierarchy. a typical layout uses src/core/, src/adapter/, and a main entry point. the core must not import or reference stdin/stdout/JSON. same idea as the execution adapter pattern from the PRD: a thin JSON-to-method-call translation layer that invokes the core parsing library, keeping I/O concerns fully isolated from parsing logic per the Dependency Inversion Principle. and per the separation-of-concerns note in start.md, the JSON input/output cases are a black-box contract for the adapter only, while the core binary parsing domain stays completely decoupled from stdin/stdout and JSON parsing.
