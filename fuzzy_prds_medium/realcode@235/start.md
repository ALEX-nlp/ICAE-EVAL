## Product Requirement Document

Hey team, we need a small utility library built out — think of it as the Swiss Army knife we keep wishing we had across projects. Basically every new codebase ends up with someone hand-rolling the same boring stuff: splitting up delimited text, poking at JSON blobs, doing math on strings people paste in, collecting results from a chain of operations, dealing with fixed-position records, making tests not flaky because of real-world clocks, and making sure we don't accidentally log someone's credit card number.

The idea is to expose all of this as clean, reusable contracts so we stop copy-pasting fragile one-offs everywhere. For the test harness piece, it should work similarly to how we set up the runner on the payments service last quarter — same kind of namespaced output folders so different test runs don't stomp on each other.

One thing that keeps biting us: the masking feature needs to handle that 'partial reveal' scenario (you know, show a few characters at the start and end, hide the middle), not just the all-or-nothing case. Also the clock stuff — we need both the kind you nudge manually AND the kind that just ticks itself forward on every read, since different test scenarios need different behavior.

Please make sure the adapter layer stays totally separate from the actual logic — last time someone tangled those together it was a nightmare to maintain. Output should go to stdout only, no metadata mixed in.

Quick follow-up since a couple details came up. On the test harness side, the stdout snapshots need to land at `rcb_tests/stdout/<cases-dir>/{filename.stem}@{case_index.zfill(3)}.txt`. For example, the first case in `feature1_csv_records.json` run with `--cases-dir public_test_cases` produces `rcb_tests/stdout/public_test_cases/feature1_csv_records@000.txt`. The index is zero-padded to three digits, and each file should contain only the raw stdout from the program under test, no PASS/FAIL labels or any other metadata. Also, `--cases-dir` defaults to `test_cases` if nobody passes it, and using `--cases-dir public_test_cases` means the runner should read from `rcb_tests/public_test_cases/`. The namespacing by `<cases-dir>` is important so one run doesn’t overwrite another, so `--cases-dir public_test_cases` should write under `rcb_tests/stdout/public_test_cases/` and any other cases dir should get its own subfolder too. See `rcb_tests/test.sh` and the harness spec in start.md.

On the validation bits, supported named string validation rules should include at minimum `min10` (length >= 10), `max10` (length <= 10), exact length rules, bounded length rules, non-empty pattern rules, and composed logical rules like AND/OR combinations. Output format is `valid=<true|false>\nrule=<name>\nlength=<len>\n`. For numeric rules the output is `valid=<true|false>\nrule=<name>\nvalue=<number>\n`. Numeric rules tested include `max10` (value <= 10), `min<N>`, exact value, and inclusive range checks.

Also wanted to be extra clear on the adapter split: the execution adapter reads a JSON command from stdin, routes it to the appropriate core domain logic, and writes only raw stdout output. The core business logic needs to stay fully decoupled from stdin/stdout and JSON parsing, with the adapter being the only thing that translates JSON commands into idiomatic method calls. Core logic must never directly depend on I/O implementation details.

For JSON rendering, parsed values should come back in a normalized textual form followed by a newline. Supported types are null, booleans, numbers, strings with escape sequences decoded before rendering, arrays, and objects. Object keys need to be rendered deterministically, for example sorted, and arrays should render in `[1, 2, 3]` style with comma-space separation.

And on masking, we need at least three policies: `none`, `hidden`, and the prefix/suffix reveal setup where you show a configured number of leading and trailing characters and mask the middle region. The `display` length must always equal the original value length. When the `hidden` mask policy is applied, every character in the value is replaced with the letter `t`. For example, `hellohello` (10 chars) renders as `tttttttttt` with `length=10`.
