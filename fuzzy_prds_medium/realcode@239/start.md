## Product Requirement Document

Hey team, we need to ship the marketplace service client toolkit we've been talking about. The core idea is that devs shouldn't have to hand-roll all the low-level protocol stuff every time they build an integration — things like figuring out which endpoint to hit for a given region, expanding those weird indexed parameter lists the API needs, making sure request bodies have the right integrity checks, and parsing the XML blobs that come back. We also need it to build ready-to-send parameter sets for the main API areas — catalog/product stuff, orders, reports and feeds, inventory/finances, and the inbound fulfillment flows (remember how we handled the address normalization on that old shipment module? same vibe). The architecture should be clean — no giant single files — and the JSON I/O layer should be totally separate from the actual business logic. Credentials should come through as environment-like config (access key, seller account, auth token) and should show up in every outbound parameter map automatically. Timestamps in params need to be URL-encoded ISO-8601. Errors should use neutral category labels, not raw exception names. We'll be running a black-box test harness against stdin/stdout so make sure the adapter layer is solid. Let me know if anything's unclear on the region coverage or which seller ID field name to use for reports vs other areas.

One quick follow-up from the questions that came in: for region coverage, the supported mappings are US -> https://mws.amazonservices.com, CA -> https://mws.amazonservices.ca, UK -> https://mws-eu.amazonservices.com, DE -> https://mws-eu.amazonservices.com, FR -> https://mws-eu.amazonservices.com, JP -> https://mws.amazonservices.jp, IT -> https://mws-eu.amazonservices.com, ES -> https://mws-eu.amazonservices.com, IN -> https://mws.amazonservices.in, CN -> https://mws.amazonservices.com.cn, MX -> https://mws.amazonservices.com.mx, AE -> https://mws.amazonservices.ae, BR -> https://mws.amazonservices.com. Also, the 'marketplace_entry' feature should return a marketplace_id alongside the endpoint anywhere we have a known canonical marketplace ID, and one concrete example there is CA returns A2EUQ1WTGCTBG2.

On the parameter expansion side, there are four modes and we should keep them exactly distinct. 'single' takes a name and a scalar or list of values and produces keys like Name.1, Name.2, and if the incoming name has trailing dots those get stripped before the index is appended. 'multi' accepts a dict of name->value(s) pairs and just applies that same single expansion to each one. 'keyed' takes a name and a list of dicts, producing keys like Name.1.field, Name.2.field for each dict entry indexed from 1. 'dict_keyed' takes a name prefix and a flat dict, then appends each dict key directly to the prefix, and in this one the trailing dot on the prefix is kept, effectively joining prefix+key. One formatting detail that matters for the adapter output: all output keys in the result dict must be sorted alphabetically in the JSON output.

Also confirming the encoding behavior because a few folks asked how literal we need to be there: certain string parameter values are URL-encoded in the built parameter maps. So Query values should encode spaces as %20, like 'hokey religions and ancient weapons' -> 'hokey%20religions%20and%20ancient%20weapons', email addresses encode @ as %40, datetime strings encode colons as %3A, and condition strings with spaces get encoded the same way. Boolean arguments like exclude_me should serialize as the lowercase string 'true' or 'false'. For Timestamp specifically, it needs to be the current UTC time formatted as ISO 8601 to second precision, for example 2017-08-12T19:40:35Z, and then URL-encoded so the colons become %3A. In the validation output, that moving value is represented as '<iso8601-url-encoded-utc-second>'.

And on creds plus the seller id naming split, yes, this should all be automatic in the request builder. Environment variables MARKETPLACE_ACCESS_KEY, MARKETPLACE_ACCOUNT_ID, and MARKETPLACE_AUTH_TOKEN are read at startup and injected as AWSAccessKeyId, SellerId/Merchant, and MWSAuthToken respectively into every request parameter map built by the toolkit. MARKETPLACE_ACCESS_KEY maps to AWSAccessKeyId with test value 'cred_access', MARKETPLACE_ACCOUNT_ID maps to SellerId or Merchant depending on the API area, and MARKETPLACE_AUTH_TOKEN maps to MWSAuthToken with test value 'cred_token'. Reports and feeds use 'Merchant'; all other API areas use 'SellerId'. Along with those, every built map should also include SignatureMethod as 'HmacSHA256', SignatureVersion as '2', and the Timestamp.

Last thing, for the inbound fulfillment piece, feature 9 is feature='inbound_address' and it’s the one that maps snake_case address fields to ShipFromAddress.* prefixed protocol keys, nulls out missing optional fields, and defaults CountryCode to 'US'. That’s the same address-normalization-ish behavior I was referring to before, just calling out the exact shape so nobody has to guess.
