## Product Requirement Document

We need to complete the full interactive console prompt toolkit so it matches the black-box terminal transcripts used by our tests. The scope is the whole prompt surface, not just a few QA fixes: single-line text entry, concealed/password entry, yes/no confirmation, single and multiple choice prompts, search and multi-search prompts, suggested/default entry, multi-line entry, message boxes, table display, progress indicators, spinners that return the underlying task result, pause/acknowledgement prompts, guided multi-step forms with back/submit behavior, and Unicode-aware word wrapping. Each action must preserve the exact transcript rendering, labels, options, key handling, validation messages, defaults, return values, and final stdout expected by the provided public cases. Keep rendering/state logic separated from the JSON adapter; the adapter simply reads an action/scenario payload and emits the deterministic transcript.

One more pass on a few details the team asked about. In the guided form flow, going back should only count steps that actually collect input. If there are display-only steps mixed in, reversion needs to skip right over them when moving backwards. And if the user tries to go back from the very first collectible step, that should raise an error or produce no reversion. That’s the same behavior expected in feature10_guided_form.json, where display steps are basically transparent to the back-navigation logic.

For the progress indicator, there’s a specific hint behavior when `update_hint` is true. The hint value is a 1-based counter of the previous step, i.e. step index before the advance, and it needs to print on the line directly below the closing box border, prefixed with a single space. That hint line starts showing up from the second advance onward. Also, when count equals total, the finished frame should render exactly the same as the last intermediate frame, including the hint, and then after completion we still emit one final duplicate frame. The exact frame sequence should match feature13_progress_indicator.json.

For the spinner, the return behavior is strict: it must return exactly the value supplied by the callback, or the `return_value` field in the test adapter. The output key is `value=<json>` and that value must not be modified, wrapped, or defaulted. Also there is no transcript at all for the spinner, just the value line, same as feature14_spinner_while_working.json.

For the pause prompt, after Enter it should return the boolean `true`, with the output line `value=true`. The transcript should include two rendered frames: the initial prompt frame, then an identical second frame after the key is received, and each of those is followed by a blank line. In non-interactive mode, don’t emit any transcript, but still return `value=true`. That should line up with feature15_pause_for_acknowledgement.json.

On word wrapping, width has to be measured by visible display columns, not bytes and not raw codepoint count. A standard ASCII character is 1 column wide. A multibyte accented character like ê is still 1 column wide. A standard emoji like 📖 is 2 columns wide. A combined emoji sequence with a variation selector like ⬇️ should be treated as 2 columns wide as a unit. Also, if `cut` is false, words longer than the target width should never be split in the middle and should just take their own line. The break string comes from the `break` parameter. The expected wrapping behavior is the one in feature16_unicode_word_wrap.json.

Also calling out the search_choice and multi_search_choice flows in feature7_search_choice.json and feature8_multi_search_choice.json, especially around result-list state management and the extra blank-line frames emitted between result-set transitions.
