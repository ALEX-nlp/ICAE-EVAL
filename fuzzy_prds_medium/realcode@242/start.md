## Product Requirement Document

We need a management layer in PHP that drives the system package manager to install, link, and control packages and their background services, so our local development tool can manage its own runtime dependencies through a clean interface without scattering raw shell strings everywhere. The layer should be able to check whether a named package is installed (by inspecting the metadata the package manager returns), determine whether any supported PHP version is currently installed from a fixed list of recognized version names (both the dotted style and the older compact style), register external package repositories before installing from them, start and stop background services (service restart requires a plain stop then a privileged stop then a privileged start; service stop requires plain stop then privileged stop), install a package with optional repositories and fail loudly with a normalized error if the manager exits non-zero, link and unlink package binaries with similar error handling, and detect which PHP version the package manager has linked by inspecting the symlink on disk. All shell commands issued must be traced in order in the output as 'cmd=...' lines so callers can verify the exact commands used. The invoking user name for user-context commands is a fixed value in the test environment. Errors must be machine-readable category codes, the same approach used by the PHP detection and service-control parts of the system.

One small follow-up after the team questions: the supported PHP names we should treat as valid are exactly 'php', 'php@5.6', 'php@7.0', 'php@7.1', 'php@7.2', 'php@7.3', 'php@7.4', 'php@8.0', 'php@8.1', plus the legacy compact names 'php56', 'php70', 'php71', 'php72', 'php73'. If the package manager reports anything else, like 'php@5.5' or 'php@5.4', that should come back as has_supported_php=false. For linked PHP detection, we’re reading the symlink target path, pulling the version number out of that path, and then only returning it if it matches that supported set, normalized to the canonical 'php@X.Y' form.

Also, on the command wrapping, user-context package-manager calls are always run as sudo -u "testuser" brew <cmd>, and that applies to tap, install, link, unlink, and list. The testuser value is fixed and deterministic in the test environment. Anything privileged should be sudo brew <cmd> without the -u flag. For service control specifically, the fire-and-forget calls stay as the raw brew command strings. Restart should issue exactly 3 supervisor commands in order after the install check: 'brew services stop <service>', then 'sudo brew services stop <service>', then 'sudo brew services start <service>'. Stop should issue exactly 2 in order: 'brew services stop <service>', then 'sudo brew services stop <service>'. And same as the rest of this layer, command tracing still needs to show the exact cmd=... lines in order, and failures should use the same error=<snake_case> style as the other operations.
