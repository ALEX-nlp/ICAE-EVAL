## Product Requirement Document

We need a small library that takes a partially-filled cluster deployment specification and turns it into a complete, validated one, so operators can declare only the fields they actually care about and trust the rest to be filled in consistently. The cluster spec describes two cooperating groups — a primary data store and a monitoring/sentinel component — each with a container image and a number of replicas. The primary group also has a metrics-exporter image, and the sentinel group can carry a list of free-form configuration directives. When any of these fields are missing or have non-positive values, the library should substitute sensible built-in defaults. We also need the library to derive the deterministic names of the infrastructure resources the cluster creates, so that separate tools always agree on what each resource is called. Resource names are built from a fixed product prefix plus per-component tokens. The primary component's shutdown configuration object has an additional override mechanism: a caller can pin an explicit name, otherwise the library falls back to the auto-derived name — handle this the same way the existing operator naming conventions work. We also need a label-map merger: given an ordered list of label maps, merge them left to right where later values override earlier ones, skip any absent entries, and return the combined labels sorted alphabetically. Finally the library must validate the cluster name before doing any defaulting: if the name is too long to be used as a resource identifier, reject it immediately with a clear error.

One extra pass of detail from the team’s questions: the built-in image defaults should be exactly redis image = 'redis:5.0-alpine', redis exporter image = 'oliver006/redis_exporter:v0.33.0', and sentinel image = 'redis:5.0-alpine'. The default replica count is 3 for both redis and sentinel, and that same fallback applies when the field is missing or the provided value is non-positive, just like the original ticket says.

Also, for sentinel config, the default custom_config list is exactly ['down-after-milliseconds 5000', 'failover-timeout 10000']. When this is emitted in the completed spec/output, those entries are joined with '|' as the separator, so the resulting value is exactly 'down-after-milliseconds 5000|failover-timeout 10000'.

And on name validation, the cutoff is 48 characters. If the cluster name is longer than 48 characters, reject it right away with 'error=name_too_long\nmax_length=48\n' and do not perform any defaulting first.
