## Product Requirement Document

hey team, we need to build out that sidecar injection config engine we've been talking about. basically the idea is: given some annotations on a workload, the system figures out whether to inject a helper container, fills in all the missing defaults, validates stuff, and spits out the patch operations for the admission controller. it's a stdin/stdout JSON thing, one request in, one response out. we want it to be totally deterministic so tests can byte-compare the output — make sure keys come out in alphabetical order always.

the tricky parts are: (1) there's a bunch of defaulting logic where blank fields need to fall back to the platform's standard values for things like image version, identity settings, resource sizing, etc. — refer to the same defaults pattern we landed on for the agent rollout last quarter; (2) the secret discovery needs to handle a bunch of annotation sub-types (templates, file overrides, commands, etc.) and the precedence rules between them are subtle; (3) the admission decision needs to block requests from the control-plane namespaces; (4) resource quantity strings have some gotchas with casing — lower-case suffixes that look valid should actually be rejected.

when something goes wrong we want stable category strings in the response, never raw exception text. the full list of ops is in the internal spec doc. the environment var computation for the agent container also needs to handle the config-map suppression case that came up in the last review.

one extra pass on the details the team asked for: the built-in defaults here are fixed and should come out the same every time. that means agent image `vault:1.7.0`, run-as user `100`, run-as group `1000`, secret volume path `/vault/secrets`, cache listener port `8200`, CPU request `250m`, CPU limit `500m`, memory request `64Mi`, memory limit `128Mi`, auth type `kubernetes`, log format `standard`, log level `info`, default template `map`, revoke grace `5`, cache enable `false`, cache exit-on-err `false`, cache use-auto-auth-token `true`, set-security-context `true`, run-as-same-user `false`, revoke-on-shutdown `true`. same story for the init op constants in Feature 4: image `vault:1.7.0`, user `100`, group `1000`, volume path `/vault/secrets`, port `8200`, CPU/memory requests and limits, etc.

also, for admission, requests targeting `kube-system` and the other control-plane/system namespaces should get stopped right away. the `mutate` op checks the incoming `namespace` field against that blocked list before doing anything else, and at minimum that list includes `kube-system`. when it hits one of those, the response should be exactly `{"allowed":false,"error":"system_namespace","status":"denied"}`.

on the quantity validation, we should be strict. valid suffixes are uppercase SI (`M`, `G`) and binary (`Mi`, `Gi`) plus bare `0` and blank (blank means omit the field). lower-case forms that might look close are still wrong here, so `mi`, `gi` for CPU, and `mb`, `gb` for memory, all need to fail with `{"error":"invalid_quantity","status":"error"}`. this check applies to all four fields: `requests_cpu`, `requests_mem`, `limits_cpu`, `limits_mem`.

for template precedence, inline template (annotation `vault.hashicorp.com/agent-inject-template-<name>`) takes precedence over template-file (`vault.hashicorp.com/agent-inject-template-file-<name>`). when the inline template is non-empty, use it and leave the `template_file` field as empty string. only when the inline template is blank does the template-file path populate `template_file`.

and on env vars, the config map behavior is the one we talked through in review: when `config_map_name` is non-empty, `VAULT_CONFIG` is suppressed and should not be included in the output list. when `config_map_name` is empty/absent, `VAULT_CONFIG` is included by default. then add the extras only when their source value is non-empty: `VAULT_MAX_RETRIES` for `client_max_retries`, `VAULT_CLIENT_TIMEOUT` for `client_timeout`, `VAULT_LOG_LEVEL` for `env_log_level`, and `HTTPS_PROXY` for `proxy_address`. as with everything else, the output list is always sorted alphabetically.
