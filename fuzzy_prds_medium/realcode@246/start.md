## Product Requirement Document

Hey team, we need to wrap up the database client library work we've been discussing. The core ask is pretty straightforward — we need something that handles all the messy connection wiring so our app developers don't have to think about protocol-level stuff. Think of it like 'that connection manager abstraction we talked about in the Q2 planning doc' but actually shipped.

Main pain points we keep hearing from devs: (1) They have to manually deal with connection strings in like 3 different formats and it's brittle, especially around SSL stuff and multi-server setups. (2) Error messages sometimes leak sensitive info (passwords) into logs, which compliance is flagging. (3) Running queries — both simple fire-and-forget ones and the parameterized kind — needs to be unified. Devs are copy-pasting protocol code everywhere. (4) Bulk data operations (loading CSVs, dumping tables) are completely un-abstracted right now. (5) Server-side notices and async messages just get silently dropped.

The output format for our test harness reads commands from JSON and prints results as key=value lines — same pattern as the auth module we wired up before. Make sure errors surface clean codes, not raw exception dumps. Environment variable fallbacks (the PG* family) should work the same way our staging config loader does. Need this to cover the full lifecycle: parse → connect → query → copy → notifications.

One small follow-up from the questions that came in: the JSON-in / key=value-out adapter pattern is the same everywhere here. stdin receives a JSON object with an 'operation' field plus parameters, stdout emits newline-terminated key=value lines. This is defined in the test case structure across all feature*.json files under rcb_tests/public_test_cases/. Also, the connection settings shape we’re expecting is the one described in Feature 1 through Feature 4 of start.md, covering URL parsing, DSN key/value parsing, TLS fallback planning, multi-host endpoints, and environment variable defaults (PGHOST, PGPORT, PGDATABASE, PGUSER, PGPASSWORD, PGCONNECT_TIMEOUT, PGSSLMODE, PGAPPNAME).

On the TLS side, the fallback behavior is tied directly to sslmode. 'prefer' means the primary attempt is TLS-enabled with insecure skip verify true, and then there is a single fallback that is plain (tls=disabled). 'allow' means the primary attempt is plain, with a TLS fallback. 'disable' means no TLS at all, no fallbacks. 'require'/'verify-ca'/'verify-full' mean TLS only, no plain fallback. Any time TLS is enabled, the output needs to include tls_insecure_skip_verify and tls_server_name. For DNS hostnames, tls_server_name should be the hostname unless sni is explicitly disabled.

For parse failures, there’s one important redaction detail: for operation: connection_parse_error_message, if the bad input is a connection string URL, the password part in that URL has to be replaced with 'xxxxx' in the rendered message. The exact message format is 'cannot parse `<redacted_url>`: <reason>' and the output lines are error=config_parse_failure and message=<redacted message>. That’s specifically to keep passwords from leaking into logs while still making the error usable.

A couple operation-specific behaviors to lock down too. The operation 'notice_callback_payload' connects to the database, registers a notice handler, executes a statement that raises a notice with the text 'hello, world', and then outputs notice_payload=<notice message text>. The operation 'runtime_parameters_on_connect' connects with runtime params application_name=pgxtest and search_path=myschema set at startup, then checks the server view of those settings and outputs application_name=pgxtest and search_path=myschema.

For literal escaping, the operation 'escape_literal_text' takes a 'text' field and returns escaped=<result> where single quotes are doubled, so hi'there → hi''there. If there are no single quotes, the text comes back unchanged. The output line is just escaped=<escaped_text> with no surrounding quote characters.

There was also a question about copying config before connect. The operation 'connect_and_copy_config' reads live database connection settings from the environment using DATABASE_URL or PG* vars, parses them, makes a copy of the config object, connects using that copy, and outputs connected=true and config_copy_usable=true if both parts succeed. The point there is just proving the config struct is safely copyable/cloneable without reference aliasing issues.

For multi-host connection strings, when hosts are comma-separated like foo:1,bar:2,baz:3, the first host/port is the primary endpoint, so host=foo and port=1, and then the rest become numbered fallbacks like fallback.0.host, fallback.0.port, fallback.0.tls, fallback.1.host, and so on. fallback_count should be the total number of fallback entries, and each fallback inherits the TLS state from the primary config.

And for environment-driven parsing, the operation 'parse_environment_settings' reads from the 'environment' object in the input, not the real process environment. The supported variables are exactly PGHOST, PGPORT, PGDATABASE, PGUSER, PGPASSWORD, PGCONNECT_TIMEOUT, PGSSLMODE, and PGAPPNAME. PGAPPNAME maps to the runtime_params key 'application_name'. Output format is identical to parse_connection_settings, and if PGPORT is missing the default port is 5432.
