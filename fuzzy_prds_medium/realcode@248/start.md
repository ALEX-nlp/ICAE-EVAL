## Product Requirement Document

hey team, we need to wrap up that deployment config toolkit we've been talking about. basically the idea is that devs are spending way too much time hand-rolling the same conversion logic every time someone touches a service definition — things like translating human-readable sizes into what the provider actually wants, figuring out if tags changed between deploys, that kind of stuff. it's causing subtle drift between environments and people only find out something's wrong when the deploy actually fails, which is way too late.

we want a single entrypoint that accepts a JSON blob describing what operation to run and spits out a predictable text result, so it can be wired up to our test harness easily. the core logic should stay cleanly separated from that IO layer though — last time we mixed those together (remember the auth service refactor?) it was a nightmare to test.

some specifics i know we need: the duration stuff should handle both directions (format and parse), the tag diffing needs to show adds/updates/deletes cleanly, resource identifiers from the cloud provider need to be classified, and version constraint checking should fail early with a clear signal. there's also the scheduling strategy edge case that tripped us up before — daemon mode should behave differently than replica. config and definition files need template resolution against env vars too.

check the existing tests folder for the file shapes we're already using, there's a bunch of fixture files in there that show what the loader expects.

one more pass on the details the team asked for so we stay lined up with the fixtures. for duration formatting, the output needs to be exactly three lines in this order: `text=<value>`, `json="<value>"`, and `yaml="<value>"`. the duration text itself should be the normal Go-style canonical form like `10s` or `10m0s`, and both the JSON and YAML forms keep that same canonical text wrapped in double quotes. for duration parsing, the input includes a `format` field with `json` or `yaml` and a `value` field containing the serialized scalar, for example `"10s"` with the quotes included for JSON. the result there should be exactly two lines: `seconds=<integer>` and `text=<canonical_duration>`.

for version checks, the output should be `result=satisfied` when the current version matches the required expression. if the expression is valid but doesn’t match, return `error=constraint_not_satisfied`, and if the expression itself can’t be parsed, return `error=invalid_constraint`. also make sure we handle the `v` prefix in version strings, and this includes the `>=`, `<=`, `~>` semver expression parsing/comparison behavior against the current tool version string with `v` prefix handling.

on desired count, the special cases are important. `DAEMON` always means `desired_count=null` no matter what came in, and we should never send an explicit count to the provider in that mode, regardless of the `requested_desired_count` or `service_desired_count` values. separately, if `requested_desired_count` is missing/null, output `desired_count=null`. if `requested_desired_count` is the sentinel `-1`, preserve the current service count and output that value. any other explicit integer just passes straight through.

for config loading, the output should be five lines: `name=<service_name>`, `region=<aws_region>`, `timeout=<go_duration_string>`, `service_definition=<path>`, `task_definition=<path>`. env vars supplied in the `env` map need to be applied before template resolution, and `region` should resolve from the config itself or from `AWS_REGION`. service definition loading should output `service=<name>`, `desired_count=<int>`, `launch_type=<type>`, `scheduling_strategy=<strategy>`, `target_group=<arn>`, `tag_0=<key>:<value>`, `deploy_max_percent=<int>`, `deploy_min_healthy_percent=<int>`, `alarm=<name>`. tags are `key:value` with a colon, not equals. the fixture paths the team asked about are under `tests/`, including `tests/sv.json` and `tests/test.yaml`.

task definition loading should output `family=<family_name>`, `ephemeral_storage_gib=<int>`, `first_container=<name>`, `docker_label_name=<label_value>`, `log_group=<group>`, `tags=<json_array_or_null>`. if there are no tags, that line should be `tags=null`. template resolution there uses environment variables from the config context. related fixture files are `tests/td.json` and `tests/td-config.yml`.

for parameter lookup, a single string parameter should come back as `value=<string>`. if multiple indexes are supplied for a non-list parameter, or the other way around, return `error=too_many_arguments`. the backing adapter is using test fixtures, and `/string` returns `string value` while `/stringlist` is the list type.

runtime platform normalization should return two lines: `arch=<architecture>` and `os=<operating_system>`. if `runtime_platform` is empty and `is_fargate` is false, default to `arch=amd64` and `os=linux`. also normalize the incoming labels to container-image-friendly values, like `X86_64` → `amd64` and `LINUX` → `linux`.

last thing, CLI option parsing gets an `args` array. for the `status` subcommand, the output should be `subcommand=status` and `options={"Events":10}` with `Events` defaulting to 10, and that options object should be compact JSON. the execution adapter should be the piece that maps the `args` array into the right subcommand handler plus its structured options.
