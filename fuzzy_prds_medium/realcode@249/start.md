## Product Requirement Document

hey team, we need to build out that notification routing core we've been talking about. basically the idea is that when something happens in the platform (deploy goes healthy, sync breaks, whatever) we need a way to turn that into an actual alert that gets sent somewhere — but without hardcoding all the message text, the firing rules, or who gets notified where. three main things: first, there needs to be a way to render human-readable messages using reusable templates with some kind of variable substitution — similar to how we handled the config interpolation in the resource sync module, you know the one. second, we need logic to evaluate whether a given state actually warrants firing a notification, and also handle the case where we only want to notify once per unique 'thing' (like don't spam someone about the same deploy revision over and over). third, we need to be able to figure out who gets notified on which channel based on metadata that's attached to the object — that metadata follows a specific naming convention we already use internally. also need the ability to add/remove people from those notification subscriptions programmatically, and a way to derive the right keys for looking stuff up. none of this should do any actual network calls or delivery — just the pure logic layer. would be great if it's structured cleanly and not just one giant file.

one quick follow-up from the questions that came in: for the template rendering piece, the body uses the `{{.varName}}` double curly-brace placeholder syntax with dotted path traversal for nested maps and a pipe operator `|` for text helper functions like `upper`, as defined in Feature 1 of start.md and the feature1_message_templating.json test cases. So `{{.name}}` substitutes the value of the variable named `name`, `{{.app.status}}` walks nested maps, and `{{.state | upper}}` upper-cases the value. Also important: if a name in the apply list does not correspond to any supplied template, the operation fails and outputs exactly two lines: `error=unsupported_template` followed by `template=<name>` naming the missing template. It does not render a partial message.

On the condition side, each condition produces exactly one output line of the form `triggered=<bool> templates=<comma-separated-template-names> oncePer=<value>`, in the same order as the input conditions list. Boolean values are lowercase: `true` or `false`. A condition fires only when its expression evaluates to boolean `true`. If the expression evaluates to any non-`true` value, or fails to evaluate at all (e.g. because it references a variable that was not supplied), the result is `triggered=false` — it is treated as not-fired rather than producing an error. And every condition line always includes `oncePer=`. If there’s no `oncePer` expression declared, that field is still there but empty as `oncePer=`. If it is declared, evaluate it against the variables and put the resulting string value after `oncePer=`.

For routing/subscription outputs, destination lines are ordered first by trigger name, then by service, then by recipient, and each line has the format `trigger=<t> service=<s> recipient=<r>`. For subscribe and unsubscribe, output all resulting annotation entries as `<key>=<value>` lines sorted alphabetically by key.
