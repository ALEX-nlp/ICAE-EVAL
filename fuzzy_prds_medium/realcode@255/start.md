## Product Requirement Document

Hey team, we need to wrap up the GraphQL subgraph federation transformer library. The core idea is that backend teams hand us their raw subgraph SDL and we give them back everything they need to plug into the supergraph gateway — no manual plumbing. Basically same pattern we followed for that schema stitching utility we built last quarter, just applied to the federation spec here.

There are a few operations we need to support: one that returns the SDL the gateway actually fetches during service discovery (the 'trimmed' view that hides internal plumbing), one that returns the full executable schema with all the injected bits visible, one that just tells you which types are resolvable entities, and one that handles bad input gracefully with meaningful error categories instead of blowing up.

The tricky part is we need to support both the old-style subgraphs (no explicit import, directives just assumed available) and the new-style ones that use the link import mechanism. The way fields and types get sorted and rendered matters a lot because the gateway does string comparisons on the SDL. Also we need to be careful about which federation version is being referenced — some features only exist in certain versions and we should reject anything we can't safely handle. Error cases should produce simple machine-readable output, not stack traces. Let me know if anything is unclear, I'll try to dig up the old spec notes.

One quick follow-up after the questions that came in: the transformer supports exactly four operations keyed off the 'op' field. 'service_sdl' returns the trimmed gateway-facing SDL string. 'federated_schema_sdl' returns the full executable federated schema SDL and also needs the boolean 'federationV2' flag. 'entities' returns a list of resolvable entity type names with a count. And if the schema is invalid or malformed, we return an error response with a category string rather than doing anything noisy.

Also, the output ordering is not best-effort, it needs to be totally deterministic. All types, fields, directive declarations, and directive arguments in the output SDL are emitted in alphabetically sorted order. That includes object type fields, union members, directive argument lists, and top-level type definitions. The sorting is strict lexicographic order on the name string.

On the trimmed view specifically, the service_sdl output intentionally omits the _Entity union, the _Service wrapper type, the _Any and _FieldSet (or federation__FieldSet / link__Import) scalars, and the _entities and _service root fields from the Query type. Those are still present in the executable schema, they’re just hidden from the service-discovery view. The service_sdl view does include a schema block with a query entry, and if no query fields remain after stripping, an empty 'type Query' definition is rendered.

For error behavior, keep it very plain and machine-friendly. Errors are reported as a single line in the format 'error=<category>\n'. The defined error categories are: 'unsupported_federation_version', 'multiple_federation_links', 'feature_not_in_version', 'rename_not_allowed', 'missing_interface_key', and 'schema_error'. More specifically, those map to the federation spec URL pointing at a version the library doesn’t support, declaring more than one @link to the federation spec, importing a directive or feature that doesn’t exist in the declared federation spec version, giving an 'as' rename to a directive that is not allowed to be aliased, an entity type implementing an interface that requires a @key without declaring it, or just a generic schema construction or validation failure.

And on the adapter piece, yes, still the same JSON-based stdin/stdout adapter pattern we talked about before: a thin I/O adapter translates JSON 'op' commands into method calls on the core transformer domain object, with the business logic kept completely separate from I/O.
