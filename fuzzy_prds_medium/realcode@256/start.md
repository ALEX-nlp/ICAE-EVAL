## Product Requirement Document

We need a reusable polymorphic container library for C++ that allows any object meeting a declared set of behavioral contracts to be stored and operated on uniformly, without requiring a shared base class or inheritance hierarchy. The library must support four main scenarios: (1) A container that holds any of several unrelated shape-like objects and can invoke both a text-rendering operation and a numeric measurement — results use fixed-point notation with five decimal places, and unrecognized or mis-parameterized object kinds are rejected with a structured error signal; (2) A single named operation that carries multiple overloads dispatched by argument count, so callers with different argument shapes reach the correct implementation automatically; (3) An operation declared as optional, allowing the container to hold collections that may or may not implement it — when unavailable, a well-defined not-supported signal is raised instead of crashing; (4) Run-time type inspection and value recovery: the container reports a tag for what it holds, and a stored value can be retrieved by naming its expected type — a mismatch produces a neutral type-mismatch error. A thin adapter program reads a JSON command from standard input, routes it by action name to the appropriate handler, and writes results to standard output. The automated test runner script under rcb_tests saves raw per-case output for comparison. The core type-erasure machinery — concentrated in the main header file of the library — must remain completely decoupled from any I/O concerns.

A couple extra specifics from the team’s questions. For the shape rendering path, a Rectangle with width W and height H is rendered as: {Rectangle: width = W, height = H}. A Circle with radius R is rendered as: {Circle: radius = R}. A Point is rendered as: {Point}. All numeric values, both inside the rendered text and in the area field, use fixed-point notation with exactly 5 fractional digits (std::fixed, std::setprecision(5)). The full output line format is: shape = <rendered text>, area = <measure value>\n

Also, if the command names an unrecognized kind like Triangle, or even a recognized kind but with the wrong number of parameters, the adapter should emit exactly two lines: first 'error=invalid_command\n', then 'command=<the raw command string>\n'. So for example, input command 'Triangle 2 3' yields: error=invalid_command\ncommand=Triangle 2 3\n

On the overloaded log operation, a log entry with only a 'message' field goes to the one-argument overload and renders as '[INFO] <message>\n'. A log entry that also carries an 'error' field goes to the two-argument overload and renders as '[ERROR] <message> (exception info: <error>)\n'. Lines should stay in the order supplied, and the full accumulated buffer is emitted as a single output block.

For the optional keyed lookup behavior, when that operation is invoked on a backing that does not support it, like a sequential/list-backed collection, the library raises pro::not_implemented. The adapter catches this and emits exactly two lines: 'error=not_supported\n' then 'operation=lookup\n'. No crash or silent wrong result occurs. The indexed backing (std::vector) supports at() and returns the stored element; the sequential backing (std::list) does not implement at() so the fallback fires.

And just to anchor the file references people asked about: golden/proxy.h is the single-header implementation of the microsoft/proxy library (pro:: namespace), containing pro::proxy<F>, pro::facade_builder, pro::make_proxy, pro::allocate_proxy, pro::weak_dispatch, pro::skills::rtti, pro::not_implemented, pro::bad_proxy_cast, PRO_DEF_MEM_DISPATCH, etc. The test harness is rcb_tests/test.sh, the bash script that reads *.json case files from a cases directory (default: rcb_tests/public_test_cases/), runs each case through the dispatcher binary, and writes raw stdout to rcb_tests/stdout/<cases-dir>/{filename.stem}@{case_index.zfill(3)}.txt
