## Product Requirement Document

Hey team, we need to build out that HTTP client toolkit we've been talking about. The goal is basically to let our app devs stop writing all that repetitive glue code every time they spin up a new service — you know, the stuff where everyone hand-rolls their own URL building, param encoding, header juggling, etc. It always ends up slightly broken in some edge case.

The toolkit should handle building requests declaratively, deal with the auth token situation automatically when the server pushes back (similar to how we handled the refresh flow in the old login module, check that for reference), and support attaching middleware so teams can inject stuff like tracing headers without touching the core logic.

We also need decent logging — both for outbound requests and what comes back — and a way to render requests as copy-pasteable shell commands for debugging. On the data side, it needs to handle nested structures and collections in query params, not just flat key=value stuff. There's also a file upload scenario we need to support, but only byte-array payloads — if someone tries to pass something weird there it should fail cleanly without leaking internal error details.

The whole thing should be organized properly across multiple files, not one giant blob. Output from the adapter layer should always be deterministic and machine-readable. Let's get a proper breakdown of what needs to be built.

One extra pass on the behavior details since a few folks asked. For query strings, by default if a key’s value is null we just leave it out completely, so that key produces nothing. If includeNullQueryVars is set to true, then null comes through as a bare key= with an empty value. Empty string values always emit key= no matter what. For lists, each item becomes its own repeated key=value pair with the same key, like foo=bar&foo=baz. Empty-string and null elements inside a list are skipped entirely, but whitespace-only elements are kept and percent-encoded. And if useBrackets is enabled, list keys get the percent-encoded [] suffix, so you’d see foo%5B%5D=bar.

For nested data, maps flatten with dot notation as deep as needed, like parent.child=value and grandparent.parent.child=value. If useBrackets is enabled, those nested map keys switch to bracket form instead, so parent%5Bchild%5D=value. If there’s a nested list under one of those paths, it keeps repeating that flattened key. Also on scalar rendering, numbers and booleans should just come out in their textual form. Trailing zeros in a fractional number are dropped, so 123.450 becomes 123.45, and booleans are always true or false. Same rule for top-level scalar values and list elements.

On deterministic output, all map-like outputs like headers and fields need to be sorted by key. Structured decoded values should be rendered as canonical JSON, while plain strings stay verbatim. Every output line is key=value, one entry per line. For multipart specifically, the output should list the resulting fields first, sorted by name, in the form field.<name>=<value>, then file_count=<n>, then each file part as file.<name>.bytes=<comma-separated byte values>. Fields are sorted, but file parts should stay in the order they were provided.

For debugging output, the curl format should be exactly curl -v -X <METHOD> followed by each header as -H '<name>: <value>', then -d '<body>' if there is a body, and then the quoted URL as "<url>". If the body encoder adds headers like content-type, those should show up there too. The request log format starts with --> <METHOD> <URL>, then one line per header as <name>: <value>, then the body, then --> END <METHOD> (<n>-byte body).

On responses, any 2xx status is treated as success and should emit response_status=<n>, response_successful=true, and response_body=<body>. Anything outside 2xx is unsuccessful and should emit response_status=<n>, response_successful=false, and response_error=<error_payload> instead of response_body.

And for the auth retry piece, when a 401 comes back, the Authenticator interceptor should attach an authorization header with the provided token and replay the request once, using a pluggable authenticator abstraction in the client dispatch layer. Also worth being really strict on error output: all errors should come out as a single neutral line in the form error=<category>, and the only categories we should emit are base_uri_has_query or unsupported_part_type. No exception class names, no stack traces, and no runtime-generated message text in stdout output.
