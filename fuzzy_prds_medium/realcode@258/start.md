## Product Requirement Document

We need a small utility library that helps our tooling work reliably with hosted code repositories and download links. The core problem is that the same repository can be referenced in multiple equivalent ways — some use a shorthand connection format while others use a standard web address format with optional embedded credentials — and we need a single consistent identifier regardless of which form was used. Additionally, download links often include volatile session tokens and analytics markers that change on every request, which makes it impossible to compare or cache links reliably; we need those stripped out while preserving any parameters that are genuinely part of the resource address. The library should expose two clean, focused functions: one that converts any recognized remote address format into a stable owner-slash-repository string, and one that returns a sanitized version of a URL with ephemeral tracking noise removed. Handle edge cases gracefully — follow the same error contract we use in the authentication module so callers get a meaningful exception rather than a silent failure. The output format for the identifier function should mirror how our existing pipeline identifies repository slugs. We also need a thin command-line adapter that accepts a JSON request on standard input and routes to the appropriate function, printing the result to standard output, so automated test harnesses can drive the library without importing Python code directly.

A couple specifics the team asked about: for the SSH-style form, we only accept remotes that match `^git@github.com:(?P<org_repo>[^/]+/[^/]+?)(?:\.git)?$`. . Only the `org_repo` capture group is returned, so that means just owner/name in the output. If there’s a `.git` suffix, it gets stripped, and if there isn’t one that’s fine too. Anything that does not start with `https:` and also does not match that pattern should raise `ValueError`, and the `git@` prefix plus host name should never show up in the returned value.

On the HTTPS side, if the remote starts with `https:`, the function should call `urllib.parse.urlparse(url)` and return `o.path[1:]`. That keeps this very literal: just the path with the leading slash removed. If the URL includes `user:password@`, that lives in `netloc` rather than `path`, so it gets excluded automatically and never appears in the output. Also, the repo name should be preserved verbatim, including dots, so something like `com.dropbox.Client` stays exactly that.

For `strip_query(url)`, the rule is simply to remove every query parameter whose name starts with an underscore (`_`). Parse with `urllib.parse.parse_qsl`, then encode again with `urllib.parse.urlencode`. Everything else stays in its original order. If there is no query string and `parts.query` is empty/falsy, just return the original URL unchanged without any modification. Also important: percent-encoded values in retained parameters are preserved exactly as-is.

And on errors, `parse_github_url` should raise `ValueError` for anything unrecognized, in the same spirit as the specific exception types we already use from `src/lib/errors.py`, like `CheckerRemoteError` and `CheckerQueryError`. The CLI dispatcher should catch `ValueError` and print structured `error=invalid_git_url` lines so this behaves consistently with the rest of the tooling.
