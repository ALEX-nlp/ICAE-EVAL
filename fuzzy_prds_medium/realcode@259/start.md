## Product Requirement Document

hey team, filing this based on some feedback we got from a few backend devs who are trying to wire up our log shipping pipeline. basically the complaint is that every team is rolling their own version of the same plumbing — someone manually builds the JSON body, someone else wrote their own retry loop, another person has their own timestamp hack to avoid duplicate entries. it's a mess and nothing is consistent.

we need a library/adapter thing that handles all of this in one place. it should be able to take log events and pack them into the right wire format for the ingest endpoint (we use that same push API as before, similar to what the metrics team did with their protobuf stuff a few months back). it also needs to group events by their label identity — some events come in with extra labels attached at emit time that should affect grouping. timestamps need to be deduplicated somehow within the same millisecond bucket.

on the delivery side, when something fails it should retry automatically — but there's a special case for rate-limit responses that the PM from infra mentioned should be configurable. also there's a buffering layer that should respect a byte ceiling, and reuse allocated buffers when possible.

label text can come in different formats with different separators — including regex-based ones — and bad label data should surface as a clean error, not a stack trace. there's also some string utility stuff we apparently need for encoding correctness. let's get this scoped and built.

one extra pass on the details people asked about: for the JSON push payloads, the timestamp we send is the nanosecond timestamp string computed as (timestamp_ms * 1_000_000) + nanos. so for example, timestamp_ms=3000 and nanos=1 produces the string "3000000001". that’s the exact shape we should be emitting there. also on the duplicate timestamp issue, for repeated events within the same millisecond, the nanosecond suffix is incremented by 1 for each successive event in that millisecond. the first event at timestamp_ms=X gets suffix (X % 1000) * 1000, the second gets that plus 1, and so on. if a timestamp older than the current maximum is seen for a given millisecond, it is clamped to the last used suffix for that millisecond. if more than 1000 events occur within the same millisecond, the suffix is capped at the maximum (999 offset) and subsequent events reuse it.

also to be really clear on grouping: records are grouped into streams based on their full label set. two records are placed in the same stream only if their label keys and values are identical. per-event marker labels are appended to the base label identity, so any difference in marker labels causes those records to be placed in separate streams. same idea for Per-event marker_labels maps (Feature 2.3): additional label key/value pairs supplied per log event that are appended to the stream identity, causing records with differing marker labels to be placed in separate streams, as defined in feature2_3_marker_labels.json.

for message content, the actual text should stay the same, but when it gets embedded in JSON push payloads it must be JSON-escaped. carriage returns become \r, line feeds become \n, tabs become \t, and other control characters are similarly escaped. the original content must not be altered — only the JSON encoding representation changes. that’s just so the ingest API always gets valid JSON no matter what characters showed up in the log message. and for the rendered line format we’ve been using in batch and HTTP delivery checks, it is exactly "l=<LEVEL> c=<logger> t=<thread> | <message> " with the trailing space. for example, an event with level=INFO, logger=test.TestApp, thread=thread-1, message="Test message 1" renders as "l=INFO c=test.TestApp t=thread-1 | Test message 1 ".

one more batching edge case: when a single event's encoded size exceeds the configured batch_max_bytes limit, that individual event is silently dropped. the remaining valid events in the input sequence continue to be batched and sent normally. the oversized event does not cause the batch to be rejected or the adapter to halt — it is simply excluded from the output payload.

on retries, the rate-limit behavior is driven by the configurable flag called drop_rate_limited. when drop_rate_limited=false, rate-limited failures are retried the same as connection failures up to the configured retry count. the test contract shows 3 send attempts for both connection failures and rate-limited failures when drop_rate_limited=false. when drop_rate_limited=true, the batch should be dropped without retrying.

and yes, this scope also includes Feature 1.2 protobuf_batch operation: serializing log records into a Snappy-compressed Protobuf binary push payload matching the Loki push API binary wire format, as defined in feature1_2_protobuf_push_payload.json.
