## Product Requirement Document

Hey team, we need to build out the core execution engine primitives for the blockchain node project. This is similar to what we did with the value encoding layer on the wallet service — same idea, just applied here to block/tx domain objects. The main ask is that we have a clean way to handle all the different 'block selector' strings that RPC clients throw at us (you know how messy those can get), plus we need solid conversion logic for those database decimal strings into the actual integer types the engine uses internally. There's also the block header derivation piece — we need deterministic hashes with proper parent linkage, and genesis should handle the zero-case correctly. On top of that we need to be able to ingest external block and receipt documents and decode them into our internal model. Finally there's a transaction execution path that runs against some pre-seeded in-memory state — the gas reporting needs to correctly distinguish between a regular transfer and the creation case. One thing that keeps biting us: errors surfaced to the output layer need to be clean category strings, not raw exception dumps. The output format is the flat key=value style we agreed on. Please make sure the code is properly split across files — last time someone just dumped everything in one file and it was a nightmare to review.

Quick follow-up from the questions that came in: for block headers, the hash should be a deterministic 32-byte digest derived solely from the block height, so the timestamp does NOT affect it. Parent linkage is just based on height too, meaning the parent hash for height N is simply the derived hash of height N-1. For genesis at height 0, use the all-zero 32-byte hash as the parent hash, rendered as 0x0000...0000, 64 hex zeros. In general, hashes in output should be rendered as 0x-prefixed 64-hex-digit strings.

On the block selector side, keyword matching is case-insensitive for latest, pending, earliest, and 'pending' is just an alias for 'latest'. There is no separate pending state here, so both should resolve the same way and output selection=latest. Also, if the selector is a hex string that is exactly 64 hex digits long without a prefix, OR exactly 66 characters including the '0x' prefix, treat it as a hash selection and echo the canonical hash. Any other hex-prefixed value should be treated as a numeric height selection and should echo the decimal height. If it’s not a recognized keyword, valid hash, or valid hex number, return a neutral error and echo the raw selector. This matches the Block Selector Parsing section of the PRD and what’s defined in feature2_block_selector.json.

For execution, the state should be a freshly initialized in-memory state seeded with funded well-known accounts indexed by integer position. Gas is the simple split we discussed: a plain value transfer between two accounts where 'to_account' is provided uses 21000 gas, and a transaction with no recipient field, treated as an empty contract creation where only 'from_account' is provided, uses 53000 gas.

Also want to be extra explicit on output hygiene: every error surfaced to stdout MUST be normalized to a neutral category line in the form error=<category>, with any offending parameter on its own field line like selector=<raw>, field=<name>, or value=<raw>. We should not leak host-language exception class names, runtime message suffixes, object-repr renderings, stack traces, or internal type names.

Last thing, for feature 7 in the PRD, encode_value needs round-trip handling for address (20-byte, 0x-prefixed 40-hex-digit), hash (32-byte, 0x-prefixed 64-hex-digit), bytes (0x-prefixed hex), block_number (minimal hex, no leading zeros), gas (minimal hex), and execution_result (bare string or raw JSON object). Those are the cases covered across feature7_1 through feature7_6 test case files.
