## Product Requirement Document

Hey team, so we need to get the gallery scaffolding generator wrapped up. The idea is that devs on any project can point this tool at their component list and get back a ready-to-run Dart gallery app without writing any of the boring boilerplate themselves. It basically needs to spit out all the different pieces — the navigation wiring, the import block at the top, the theme hookup, the type registries for enums and models, and the root app widget. We talked about this in the last arch sync and everyone agreed the output has to be 100% deterministic, like two runs on the same input must produce byte-identical output, because downstream tooling diffs the files.

One thing that keeps coming up from the mobile team is that the theme integration is fragile — sometimes the accessor is a plain function, sometimes it lives on a class, sometimes it's a getter not a method, and the current hand-rolled approach keeps getting it wrong. We need that handled cleanly without people having to edit the generated file afterwards.

Also there's a known pain point where multiple independently generated JSON chunks get concatenated and the boundary between them breaks downstream parsing — same kind of fix we did for the export merge issue in the settings module, just apply that same pattern here.

The whole thing should come in as a simple stdin command and write back to stdout so it fits into the existing CI pipe. Reach out if the enum seeding behaviour or the provider dependency detection logic is unclear.

A couple extra specifics from the questions that came in. For the enum registry, `framyEnumMap` needs to be seeded with exactly one framework entry before anything user-defined, and that entry is `'MaterialTapTargetSize': MaterialTapTargetSize.values,`. It should show up on its own indented line with two spaces immediately after the opening brace. After that, any user-defined enum entries come next with no leading indentation, and then there’s a trailing blank line before the closing `};`.

On imports, please keep them in insertion order while still deduping. The fixed framework imports always come first in this order: `package:flutter/foundation.dart` and then `package:flutter/material.dart`. After that, append each object’s own `import` path. The provider package import only gets appended last if at least one object has `category` equal to `widget` and that widget object has at least one provider-style dependency in its dependency list. If that never happens, don’t emit the provider import at all. Same deal for the provider-dependency check we talked about for the import aggregation behavior in Feature 2: it’s just checking whether any object has `category == 'widget'` and at least one provider-style dependency before appending the provider package import, as defined in `rcb_tests/public_test_cases/feature2_import_aggregation.json`.

For the route map formatting, when there are no widget descriptors, the body should end with the four built-in entries and then a four-space-indented blank line, exactly `    \n`, before the closing `  };`. If there is at least one widget, the custom route entries should come immediately after the built-in ones with no blank line gap, and the last custom entry should still end with a comma before the closing `  };` on the next line. In both cases, those four built-in routes use four-space indentation.

One more precision point on model factories since a few folks asked about constructor arg emission. Each factory writes one line per declared parameter using `dep.subDependencies.singleWhere((d) => d.name == '<param>').value`. . When a parameter's `named` flag is `true` the line is prefixed with `<paramName>: `, and when `named` is `false` there’s no prefix. The closing `)` for the constructor call should be on its own line with no indentation relative to the opening of the factory lambda, and it’s followed immediately by a comma and newline. Just to mirror the wording exactly because this is where the confusion was: the line is prefixed with `<paramName>: ` when the `flag is `true`.

And for the JSON fragment issue, the `fix_json` / `feature6_json_fragment_merge` behavior is specifically about detecting the `]\n\n[` boundary between two concatenated JSON array fragments and collapsing that into one merged array, matching what’s defined in `rcb_tests/public_test_cases/feature6_json_fragment_merge.json`.
