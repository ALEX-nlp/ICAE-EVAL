## Product Requirement Document

Hey team, we need to get the new service wiring toolkit shipped. Basically the idea is that right now every dev on the team is manually newing up objects, passing deps through constructors by hand, and copy-pasting cache logic everywhere — it's a mess and we keep having bugs where the wrong instance gets reused (or doesn't get reused when it should). We need a single container thing where you register stuff once and just ask for it later. Should handle the usual cases: pre-built objects, factory callables, closure-based factories, and also that abstract factory pattern we used in the config module last quarter (you know the one). It also needs to support aliasing so you can ask for an interface name and get the concrete impl back, and aliases should chain. There should be a way to wrap or decorate objects after they're built — similar to what the initializer hooks do in that login flow we shipped — so downstream teams can bolt on behavior without touching core. Oh and if someone tries to reconfigure something that's already registered it should blow up with a clean error, not a silent overwrite. Also need that child-container / plugin-registry pattern for type-validated extension points. Finally, there's a PSR-compatible wrapper layer so third-party containers can be plugged in. Make sure cycle detection is in there for aliases. Rough deadline is end of sprint.

One extra pass on details the team asked about: by default, every registered entry should be treated as shared, so if you call get() twice for the same identifier you get back the exact same cached object instance unless that specific entry says otherwise or the caller uses the explicit build path. And yes, each registration can carry its own sharing flag. If that flag is present, it wins over the container-wide default, so we can have a globally non-shared container with a few entries still cached, or the other way around.

On the auto-wiring side, if constructor inspection hits a required typed parameter and that type is not registered, the output needs to be exactly error=service_not_found on one line and detail=unresolvable_typed_dependency on the next. Same idea for required scalar params with no default value — they should normalize to the equivalent error behavior too.

For the plugin manager / constrained child container bit, normal lookups should cache the same way the parent container does. But if a get() call comes in with options, it needs to skip the cache, build a fresh object, and pass those options through to the factory. So two optioned lookups should produce two distinct instances even if sharing is turned on. Same general rule for the explicit build path too: build() with an options array always makes a fresh object no matter what the sharing rules say, and two back-to-back build() calls for the same identifier should return two distinct instances. The options map goes straight into the factory so the object can actually see them.

Also confirming the wrapper layer behavior: the PSR-compatible decorator should just forward has() and get() straight through to the wrapped container, and it also needs a dedicated accessor for the wrapped container itself so callers can confirm they’re looking at the same underlying instance. For a known identifier, the output should be exactly has=yes, get_id=<the identifier>, same_wrapped_container=yes.

And just to be explicit about supported registration shapes, we still need all four: a prebuilt service registered directly, a direct factory using a factory class instance, a closure factory using an anonymous callable/closure, and the abstract factory fallback when there’s no specific match. All four should go through the same sharing behavior and the same lifecycle hook flow. That includes the ConfigAbstractFactory (Feature 2) which reads a dependency_map configuration key to recursively resolve and inject constructor dependencies, as described in feature2_1_configured_dependency_factory.json and feature2_2_configured_dependency_errors.json, plus the delegator and initializer lifecycle hooks described in Feature 1.5 (feature1_5_lifecycle_hooks.json): delegators wrap the factory call, initializers run post-creation, both mutate the final resolved object and are tested via the delegator_initializer operation.
