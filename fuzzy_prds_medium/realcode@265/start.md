## Product Requirement Document

We need a small toolkit of stateless helper routines that a model-and-collection layer can use to talk to a backend API consistently on both server and client. The toolkit handles four concerns: (1) replacing named placeholders in a URL template with values from a record's attributes, and optionally cleaning up the placeholder keys from a parameter map so they don't get appended as redundant query parameters; (2) deeply comparing two objects or arrays to detect whether they differ in structure or content, checking both orderings; (3) building client-facing URLs by prepending a routing prefix to a resource path — with an option to pass through already-absolute URLs unchanged even when prefixing is requested; (4) translating a high-level persistence operation (read, create, update, delete) into either a transport-agnostic descriptor handed to a data adapter on the server side, or a call to the underlying HTTP transport on the client side. For the client side, when the caller supplies an error callback it should be wrapped so failures arrive as a normalized two-field object instead of raw XHR state — handle it the same way the server-side adapter normalizes errors before forwarding to downstream services.

One other detail the team asked about on the client-facing URL side: the default base segment is '/api' and the separator marker is '/-'. So if you hand it a URL '/path/to/resource' with no config, it should come out as '/api/-/path/to/resource'. If apiPath is configured, like '/foo/bar', then that same path becomes '/foo/bar/-/path/to/resource'. And if an optional api segment is provided, that gets inserted between the base and the separator, so the shape is '/api/<segment>/-/path'.

Also, for the pass-through behavior, a URL counts as absolute if it contains '://'. In those cases it should be returned unchanged regardless of whether clientPrefix is true. That includes things with ports like 'http://host:8080/path' — the ':8080' is just part of the absolute URL and must not be treated like some kind of path placeholder.

On the persistence mapping, the operation to HTTP method is read->GET, create->POST, update->PUT, delete->DELETE. The resource path is root joined with id, for example '/listings/0'. For read/delete, body is {}. For create/update, body contains the full record attributes including id.
