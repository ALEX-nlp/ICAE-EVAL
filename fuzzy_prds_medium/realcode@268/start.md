## Product Requirement Document

hey team, we need to build out this scoping/isolation layer for our redux-style store setup. basically the idea is that individual feature teams are complaining that everything is tangled together — one team's async handler is seeing state it shouldn't and reacting to actions meant for other teams. we talked about this in the Q3 planning session and agreed we'd do something similar to what the auth team did with their slice isolation approach (you can look at how they handled that if you need a reference point). the core of it is: each handler should only see its own slice of state, only care about actions meant for it, and when it fires off new actions they should automatically get routed back to the right place without the developer having to manually tag everything. there's also the dependency injection side — services and config need to flow into these handlers cleanly. a big thing for us is that misconfiguration should fail loudly with clean error messages, not silently do weird things. we don't want raw exception stack traces showing up in the output, just normalized category codes. oh and handlers should be composable/nestable so teams can build hierarchies. the middleware wiring piece is important too — if someone wires it up wrong it should blow up immediately with a clear signal. can someone take a pass at this and make sure the error handling and scoping rules are solid?

one quick follow-up from the questions that came in: for dependencies, the only valid actual value is a plain key/value object. no options, an empty options object, an absent dependencies key, an explicit undefined, or an explicit null are all fine and should just be treated as no dependencies. a plain object like {foo: 'bar'} is accepted. booleans, numbers, strings, and arrays are all rejected, and yes that includes arrays even though js technically treats them as objects. when that happens, just emit the normalized line `error=invalid_dependencies` and stop there — no exception class name, no extra message detail leaking out.

also, if a scoped epic gets invoked without the subspace-aware epic middleware putting the parent store into the dependency context, it needs to fail immediately with `error=missing_store`. it should not silently limp along, produce partial output, or run with no store at all. that’s specifically to catch cases where somebody used generic middleware instead of the scope-aware wiring.

on the scoping behavior itself, a namespaced scoped handler should only see actions tagged for its namespace, meaning `namespace/TYPE`, plus anything explicitly marked `globalAction: true`. if an action is tagged for some other namespace, or it’s an untagged root-level action without the global flag, that handler should act like it doesn’t exist. if the scope has no namespace, then it sees everything with no filtering. on the way back out, every action emitted by a namespaced scoped epic should automatically get `<namespace>/` added unless the emitted action is explicitly marked `globalAction: true`. in that global case, it should escape prefixing completely and go out with its original plain type and the global marker intact so the whole store can observe it.

and yes, we do want the shorthand where a scope can be created from a single string argument. that one string is doing both jobs at once: it picks the state property via `state[key]` and it also becomes the action namespace. in the test DSL, `selector: {key: 'subState'}` is representing that pattern. so the scoped epic reads from `state.subState` and anything it emits gets prefixed as `subState/`. this is all in the `subspace(selector, namespace?)` shape from Feature 3 of start.md — wrapping an epic with a selector and optional namespace to isolate its state view and action routing. and just to restate the error behavior one more time, the only normalized outputs here are `error=invalid_dependencies` and `error=missing_store`, with no exception identity leaked.
