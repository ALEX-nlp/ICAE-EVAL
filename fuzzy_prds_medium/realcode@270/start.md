## Product Requirement Document

Hey team, we need to get that UML diagram toolkit shipped. The basic idea is that devs are wasting a ton of time hand-translating those XML diagram files into entity configs and then re-typing the same stuff in like three different formats. Super error-prone and nobody wants to do it. We want a tool that can read those diagram files and spit out something usable downstream — basically the same pattern we used for the config export pipeline in the auth module last quarter, you know the one.

The tool needs to handle a few things: reading field/type info from diagrams, checking whether field types are actually valid for whichever storage backend someone is using, and then producing both a JSON config format and a plain-text model definition. There should also be some way to put together the command you'd actually run to kick off generation for a given entity, and we need it to understand the CLI flags people typically pass.

One thing that keeps biting us is that some relationship types just don't work with certain storage setups — we need that to come back as a proper structured error, not just blow up. Same goes for reserved names or anything malformed. Oh also the editor detection thing — when we open a diagram we should know which tool produced it. Should work for at least the main editor we use internally (you know which one). Make sure the whole thing is tested end-to-end.

One quick follow-up from the questions that came in: for the SQL profile, the supported types are exactly String, Integer, Long, BigDecimal, LocalDate, ZonedDateTime, Boolean, Enum, Blob, AnyBlob, ImageBlob, TextBlob, Float, Double, and that order matters. Also for String on SQL, the validations we support are exactly required, minlength, maxlength, pattern, in that order.

On the entity output side, the build-entities result should be keyed by class name. Each entity payload includes dto, entityTableName, fields, fluentMethods, pagination, relationships, and service. The fields value is an array of {fieldName, fieldType} objects. fluentMethods should always default to true. relationships is an array, and each relationship item includes otherEntityName, relationshipName, relationshipType, and then when applicable ownerSide, otherEntityField, otherEntityRelationshipName, and relationshipValidateRules. Also worth calling out: changelog timestamps are excluded from output. For the default entity data object, use comment as empty string, dto as "no", fields as empty array, name as empty string, pagination as "no", service as "no", and tableName as empty string, and all keys should be output in alphabetical order.

For command building, build-command should always return a JSON object with a "command" key set to "yo" and an "args" array. The args always start with "jhipster:entity" and the entity name, and always end with "--regenerate". If people pass optional flags, those get inserted between the entity name and --regenerate as --force, --skip-client, --skip-server, --skip-install, --skip-user-management. If angular_suffix is provided, --angular-suffix and its value go in before --regenerate too.

On editor detection, what we’re checking is whether the root object's eAnnotations array contains an entry whose "$".source value equals "Objing". If yes, identify the editor as "modelio" and include the banner line "Parser detected: MODELIO." followed by a blank line before the JSON. If root is null, return the error object {"detail": null, "error": "null_input", "input": "detect-editor"} and do not include any banner.

And for the export-jdl piece, the export-jdl action in feature5_jdl_export.json parses diagram then emits a structured textual representation of entities and relationships, analogous to how a config export serializes structured data to a human-readable format.
