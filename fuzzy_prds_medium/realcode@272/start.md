## Product Requirement Document

We need a reusable backend library that lets service teams declare their traffic allowances as simple configuration files instead of embedding that logic in code. The idea is straightforward: someone writes down 'for domain X, when we see a request matching these categories, allow at most N per time window' and the library figures out which rule applies and returns the answer. Different teams may send multiple config files for the same domain, and we want to support optionally merging those rather than treating them as conflicts. Individual requests should also be able to carry their own temporary override of the allowance number and window size, which should take precedence over whatever is in the config files. The system should also support a special 'always allow' flag per rule so we can whitelist certain traffic patterns without removing the accounting entry, and a dry-run mode flag that indicates a limit is being tested but not enforced — handle this the same way as the shadow_mode flag in the auth module. Additionally, we need a utility that generates the storage key used to count hits in a time window, and another utility that sanitizes connection strings before they appear in logs — the masking approach should follow the same convention used in the credential-scrubbing helper. The library should validate configuration on load and report categorized errors when the config is malformed.

One extra pass on the matching behavior since a few people asked what the library should actually print back. When a limit is matched, output exactly these lines in this order: 'matched=true\n', 'full_key=<value>\n', 'requests_per_unit=<number>\n', 'unit=<UNIT_STRING>\n', 'unlimited=<bool>\n', 'shadow_mode=<bool>\n'. When no limit matches, output only 'matched=false\n'. Every line ends with a newline. Unit strings are uppercase: SECOND, MINUTE, HOUR, DAY, UNKNOWN.

A couple specifics on special cases. When a descriptor is configured as unlimited (unlimited: true), the resolved result reports: matched=true, requests_per_unit=0, unit=UNKNOWN, unlimited=true. The full_key is still derived from the domain and descriptor path. Also, the full_key is formatted as: domain + '.' + descriptor_path, where descriptor_path is the matched descriptor entries joined by '.', and each entry segment is 'key' (for key-only default matches) or 'key_value' (for key+value matches). Example: 'test-domain.key1_value1.subkey1' for a two-level match where the second level matched by key-only default.

On wildcard values, a configured value ending with the character '*' acts as a prefix wildcard: any request value that begins with the characters before the '*' matches that descriptor. Wildcard matching is only triggered when the trailing character is '*'; a '*' in the middle of a configured value is not treated as a wildcard. The full_key in the match result uses the configured wildcard string (e.g., 'foo*'), not the actual request value.

For runtime overrides, when a descriptor carries a runtime override (requests_per_unit + unit), the override values replace the configured limit. The full_key is derived from the descriptor's key/value entries using the descriptorKey function (domain + '.' + each 'key_value' joined by '.'), not from any configured FullKey. shadow_mode and unlimited are both false when an override is used. An override on an unknown domain still returns no match (matched=false).

And just to be explicit about the dry-run piece, the shadow_mode boolean field in the RateLimit struct (src/config/config.go and config_impl.go). When a descriptor has shadow_mode: true in the YAML config, the resolved RateLimit carries ShadowMode=true, and this is reported as 'shadow_mode=true' in the output. It means the limit is being monitored but not actively enforced.
