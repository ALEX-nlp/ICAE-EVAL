## Product Requirement Document

Hey team, we've been getting complaints from a few enterprise CAD integrators who are trying to use our DXF toolkit in their pipelines. The main pain points are around compatibility — some of their older drawings just don't parse right, especially when they're coming from machines set to non-English regional settings (the numbers come out wrong). Also, a couple of partners are asking about support for that compact binary format some legacy tools spit out, plus the regular binary encoding. The BOM issue keeps coming up too — some pre-processing tools add that invisible prefix and everything breaks downstream.

On the authoring side, people want to target specific product versions when they export, and apparently some features (like certain table types and entity kinds) shouldn't appear in older-format outputs. The guy from that one integration project mentioned it's similar to the version-gating logic we did for the classes section — just extend that pattern.

We also need clean error handling — no raw stack traces leaking out, just stable category codes so downstream tools can parse failures programmatically. Can someone make sure the whole thing is properly split across files too? Last time we shipped a god-file and it was a nightmare to maintain. Thumbnail embedding came up as well — stripping the bitmap file header before storing should be handled automatically.

Quick follow-up from the questions that came back: for the locale problem, the reader needs to give the same numeric result no matter what the host machine is set to. So even under something like 'de-DE', where commas are used for decimals, we still need to read period-decimal DXF group-code values exactly the same as under 'en-US'. The concrete check here is that reading Julian day number 2456478.590142998 under both 'de-DE' and 'en-US' has to land on the same value, specifically creation_date=2013-07-04T14:09:48.355.

Also on the date fields, the header vars like $TDCREATE and $TDUPDATE are floating-point Julian day numbers, with the integer part as the Julian day and the fractional part as the fraction of a 24-hour day, and we need to turn that into an ISO 8601 calendar datetime. The examples to hold onto are 2451544.91568287 -> '1999-12-31T21:58:35.000' and 2456478.590142998 -> '2013-07-04T14:09:48.355'. Same story there: locale-independent, always.

For stream detection, we need to auto-detect three cases just from the leading bytes: normal text DXF, AutoCAD Binary DXF with the sentinel 'AutoCAD Binary DXF\r\n\x1A\x00', and DXB compact binary with the sentinel 'AutoCAD DXB 1.0\r\n\x1A\x00'. On the DXB side, the stream is a current-color record followed by geometry primitives and then ends with a null record type byte. If we write then read a DXB entity, we should get the original geometry back.

One more parser detail on text input: if a text DXF stream starts with a UTF-8 BOM, meaning 0xEF 0xBB 0xBF, we should quietly strip it before parsing. The tiny sanity check for that is the BOM-prefixed minimal document with base64 '77u/MA0KRU9G' — that should parse as an empty drawing with entity_count=0 and layer_count=0, no error.

On CLASSES, there are two real on-disk layouts we need to honor. The older one, R12-class, AC1012, uses the record name itself as group 0, then group 1 = C++ class name, group 2 = application name, group 90 = class version integer. The newer one, R14-class and later, uses 'CLASS' as group 0, then group 1 = record name, group 2 = C++ class name, group 3 = application name, group 90 = proxy capabilities integer. For writing, R13-class should still use the old form with class_version, and R14-class and later should use the CLASS marker with proxy_capabilities.

And for version gating, there are two specific ones to make sure we don’t miss: the CLASSES section only gets emitted for R13-class and later, so section.CLASSES=absent for R12 and section.CLASSES=present for R13. Same idea for the BLOCK_RECORD table: only emit it for R13-class and later, so table.BLOCK_RECORD=absent for R12 and table.BLOCK_RECORD=present for R13, even if a block record was added. This should follow the same target-version enum pattern at the R12-class vs R13-class boundary that we already talked about for including or excluding whole sections and tables, and then that same pattern should also carry over to individual entities and entity properties.
