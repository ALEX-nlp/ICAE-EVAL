## Product Requirement Document

Hey team, we need to build out that saga testing toolkit we've been talking about. The core idea is that devs should be able to test their workflow logic without having to spin up real stores or actual network calls — basically a way to verify what a workflow *does* without caring about *how* it does it internally. There are two modes we talked about: one where you go step by step and check each instruction one at a time (kind of like that debugger flow we did in the auth module a while back), and another where you just let the whole thing run and check the results after the fact without caring about order.

The step-by-step mode needs to support rewinding, bookmarking spots, restarting with different inputs, cutting it short early, and throwing errors in to test recovery paths. If someone tries to go back further than they've gone or jump to a bookmark that doesn't exist, we should surface a clean error category — not just blow up.

The run-to-completion mode should support mocking out function calls (either returning a value or throwing), intercepting state reads, wiring up a real reducer so state updates propagate, and dispatching actions the workflow is waiting on. If a workflow is stuck waiting forever, it should still time out gracefully.

Also there's a small utility for serializing action-matching patterns to strings that needs to be included. Multi-file structure is a must — no giant single files please.

One extra pass on the behavior details the team asked about: in the step-by-step flow, the assertion counter only moves on steps that actually have an `assert` key. Things like `next`, `nextWith`, `back`, `save`, `restore`, `restart`, `finish`, `finishWith`, and `throw` are just navigation/control moves and do NOT increment anything. The numbering is strictly per `assert` step in declaration order, starting at 1. Also, if an assertion is met or unmet, that by itself does NOT stop the walk — we still keep going through the remaining steps. The only thing that should halt a unit-mode run early is a navigation error. In that case we emit `error=<category>` and stop immediately. That applies to the debugger-style navigation pieces too, especially the `back`/`save`/`restore`/`restart` behavior in the step-by-step walker.

On the navigation error specifics, the output is a single line `error=<category>` followed by a newline, and then the run stops. If somebody rewinds further than they've actually gone, that should emit `error=cannot_step_back`. Same deal for the back case already discussed. If they try to restore a bookmark that was never saved, that also needs to come back as an error, and for the unknown bookmark case the exact category string needs to match what’s in `feature2_6_navigation_errors.json`, so please follow that exactly.

A couple serialization and runtime details too. In `serialize_pattern` mode, a plain string pattern should be written verbatim and then a newline. If it’s an array of strings, output it as `[A, B, C]` with comma-space separation and square brackets, then a newline. An empty array is `[]` followed by a newline. No quotes around the individual strings inside the brackets. For integration runs, each item in the `expect` array should produce exactly one `expectation <n> met` or `expectation <n> unmet` line, numbered sequentially from 1 in declaration order, and the report lines come after all of those expectation lines.

Last couple edge cases: if a workflow is sitting on a `take` effect and the matching action never shows up, the run still needs to terminate because there’s an internal timeout. The test author does not configure that timeout. When it fires, the run completes with whatever state/effects were recorded up to that point. And in the saga body, `{"put": <action>}` without a `yield` wrapper should be treated as shorthand for `{"yield": {"put": <action>}}`. It still emits an effect and advances the generator by one step, and that shorthand is valid in both the main `body` and the `catch` recovery body.
