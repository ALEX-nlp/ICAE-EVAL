## Product Requirement Document

hey team, we need to wrap up that async toolkit thing we've been discussing. basically we want something that lets devs coordinate groups of async operations without writing the same boilerplate every time. there are a few different 'flavors' of aggregation we need — like sometimes you want everything to succeed, sometimes you're okay with partial failure, sometimes you just want whoever finishes first. we also talked about that cancellation pattern (similar to what we did in the login module, check how we handled the subscription/unsubscription flow there) and some kind of guarded config object that catches typos and gives helpful hints. oh and there's the streaming thing for iterables too. the output format needs to be consistent — i remember we agreed on some rule about when to use arrays vs objects for the position stuff but i can't find where we wrote that down. also the error categories need to be generic/neutral, no stack traces or runtime internals leaking out. the settle feature should work with both plain strings and nested objects/arrays. for the struct/typo-guard thing, private-ish fields (ones starting with underscore) should never show up as suggestions. can someone make sure all these are wired up properly? the test harness reads one JSON blob from stdin and writes key=value lines to stdout.

quick follow-up since a couple details came up in questions. for the position rendering rule, let’s stick to the thing we agreed on: if the positions are contiguous and start at zero, render as a compact JSON array like [1,2,3]. if they’re non-contiguous, meaning there are gaps in the index sequence, render as a compact JSON object keyed by string position like {"0":1,"2":3}. and if it’s empty, it should always come out as [].

on the some behavior, required defaults to 1 when it’s not provided, so the baseline is that at least one operation has to succeed. also, if required is greater than the total number of operations in the promises list, it should immediately report status=error with error=too_few_promises and not attempt to run any of the operations. same general idea for validating inputs up front: a valid operand spec in the promises array is only {"resolve": <value>} for a succeeding operation or {"fail": <message>} for a failing one. if any element uses {"value": <x>} or any other key that is not 'resolve' or 'fail', treat it as a raw non-operation value and immediately return status=error with error=invalid_operand without processing any other operands.

for iterator/streaming, if the source isn’t actually a valid iterable — for example if the elements key is absent or replaced by a non-array field — then it should immediately report status=error with error=invalid_iterable and emit nothing before that. for settle in resolve mode, the value field should always be rendered as compact JSON no matter what shape it is, so even a plain string needs to be quoted exactly like value="Resolution value", and arrays/objects should also be compact JSON with no extra whitespace.

also just reinforcing the error output expectation: keep everything at the neutral domain-category level like invalid_operand, negative_required, no_promises, multiple_failures, unknown_property, invalid_iterable. we should not leak host-language runtime details, exception class names, stack traces, or interpreter-specific messages into output. and on cancellation, use the same pattern we referenced before: a CancellationSource with a token that supports subscribe/unsubscribe, fires handlers on cancel, is idempotent across multiple cancel calls, and notifies late subscribers immediately. that’s the same behavior we want carried through here.
