## Product Requirement Document

Hey team, we need to build out this load rate engine thing that the platform team has been asking about for a while. Basically, developers are frustrated because every time they write a load test they end up copy-pasting the same ugly timer loop boilerplate and it never quite works right — the request counts drift off target, ramps feel janky, and nobody wants to debug fractional accumulation math at 2am. The ask is to give them a clean way to describe traffic shapes (like 'ramp up, hold, ramp down', or that bell-curve pattern we use for the daily peak simulation) and have the engine just figure out exactly how many requests to fire each tick.

The config format should be readable enough that non-engineers can tweak it without help — similar to how we approached the scheduling layer on that campaign delivery system last quarter, just adapted for load testing. Make sure bad inputs don't blow up silently; they should get clear, categorized error messages.

Also important: the code shouldn't be one giant file. We've had reviews bounce back before for that. Split it up sensibly. The JSON test harness we already have should be treated as a black-box contract for the outer adapter layer only — the core logic shouldn't know anything about JSON or stdin. Ping me if the gaussian volume math needs clarification, that one always trips people up.

One extra pass on details the team asked about: for any durations we print back out, please keep them in the compact Go-style form and always include the unit down to seconds. So seconds-only should look like `0s`, `5s`, `10s`; if minutes are involved use forms like `1m0s`, `2m30s`; and if hours are involved use forms like `1h0m0s`. Same rule everywhere: `1m0s` not `1m`, and `1h0m0s` not `1h`.

On the even spread behavior, the intent is the standard accumulator approach so totals stay exact over the whole interval. For regular distribution across N sub-ticks, where N = interval / 100ms, maintain a floating-point accumulator initialised to 0. On each sub-tick, add `rate / N` to the accumulator. The integer part of the accumulator is emitted as that tick's count, and the fractional remainder is carried forward. That way the emitted integers over a full interval sum exactly to the requested rate. The concrete example to match is: rate=7, N=10 → add 0.7 each tick, emit floor each time: ticks produce 0,1,1,0,1,1,0,1,1,1.

Also, if `iteration_duration` is at or below 100ms, don't do the sub-tick smoothing at all. In that case the effective sub-tick interval is the original interval, not 100ms, and the full rate comes back as a single tick count. Only when the interval is larger than 100ms should the effective sub-tick interval be 100ms.

For random distribution mode, the total still has to land exactly on the requested rate. On each sub-tick except the last, emit the supplied draw value clamped to the remaining budget and subtract it. The final sub-tick always emits whatever budget remains, regardless of any draw value. The `random_values` array is just one draw per non-final sub-tick, so it does not need a value for the last tick.

On structure, yes, this absolutely must NOT be a single file. Please keep a multi-file directory structure with core domain modules separated out so rate-string parsing, stage interpolation, distribution smoothing, gaussian volume calculation, and config parsing/validation live in distinct units, plus a thin JSON adapter layer that only reads JSON commands from stdin, calls core APIs, and writes text output, plus tests. The core engine should have zero dependency on stdin/stdout or JSON, and the JSON cases remain a black-box contract for the adapter only.

And on the gaussian piece, this applies both to Feature 4 (`peak_to_volume`) and the gaussian stage mode in Feature 5. It's a Gaussian/normal distribution profile over a 24-hour window, parameterised by peak_rate, peak_time, and stddev, integrating to a total volume. The engine integrates a normal distribution over a 24-hour window (86400 seconds). The peak rate, converted to requests-per-second, is the amplitude at the peak. The integral is computed as: `volume = round(peak_rate_per_second * stddev_seconds * sqrt(2*pi))` scaled by the fraction of the normal distribution that falls within the 24h window centred on the peak_time, and the final result is rounded to the nearest integer. stddev and peak_time are parsed as duration strings like `4h` and `14h`.

Last bit: for the config-file schedule in Feature 5, this is specifically the shape covered by the config modes and config errors examples — a YAML config with a `limits` block, a `stages` list, `default` inheritance, and stage modes `constant`, `ramp`, `staged`, `gaussian`, and `users`, all parsed and validated by the engine.
