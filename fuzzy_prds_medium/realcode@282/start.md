## Product Requirement Document

Hey team, we need to build out the core data layer for our HTTP client library. This is basically the same kind of thing we did for the serialization layer in the old networking module, so use that as a reference for how we handled edge cases there. The main pain points we're hearing from devs: (1) every call site is doing its own parameter encoding and they keep getting inconsistent results — especially around nested objects and lists, some endpoints expect comma-separated values, others expect repeated keys, it's a mess; (2) headers are being compared case-sensitively in some places and case-insensitively in others, causing bugs where the same header gets duplicated; (3) when you layer a per-request config on top of global defaults, nobody agrees on which one wins, especially for content-type; (4) after a response comes back, parsing the body is inconsistent — JSON responses sometimes come back as raw strings instead of parsed objects. We need a clean, unified toolkit that handles all of this deterministically. One thing I keep forgetting to mention — there's a special wrapping format for per-parameter list overrides, ask the backend team if you need the marker key details. Also make sure the JSON media type detection covers vendor subtypes, not just the obvious ones. Output should go to stdout, one value per line in labeled format.

One quick follow-up since a few folks asked for more detail on the exact shaping here. For nested maps, keys should flatten with bracket notation, so if `x` lives under `c` then `e`, that key becomes `c[e][x]`, and those bracket characters are percent-encoded in the actual output, so `[` becomes `%5B` and `]` becomes `%5D`. On the list side, we do need to support all six formats: `csv`, `ssv`, `tsv`, `pipes`, `multi`, and `multiCompatible`. That means comma-joined for `csv`, space-joined for `ssv` with the space encoded as `+`, tab-joined for `tsv` with the tab encoded as `%5C%74` or `\t` percent-encoded, pipe-joined for `pipes` with the pipe encoded as `%7C`, repeated bare keys for `multi` like `b=5&b=6`, and repeated bracketed keys for `multiCompatible` like `b[]=5&b[]=6` with the brackets percent-encoded. The separator characters are part of the value encoding here where they’re reserved, so please keep that behavior consistent.

Also, on request composition, the final request should build the URL by joining the base URL and the path, and the `uri` field in the output should reflect that assembled value. If the base URL is empty or absent, the uri can just be the path. The concrete example here is `baseUrl=https://www.example.com` with `path=/test` producing `uri=https://www.example.com/test`. And for `url_encode`, the output is just the encoded query string followed by exactly one trailing newline character. No extra blank line, just the encoded string with `\n` appended.

The special per-parameter list override is the wrapper I mentioned before, and this is the exact shape: an object with `@list` holding the array of values, and `format` holding the desired format name like `csv`, `pipes`, `multi`, etc. Example: `{"@list": [1, 2], "format": "pipes"}`. This can show up at any depth in the data map, and when it does, that wrapped object should trigger per-value format selection instead of the global format for that one entry.

Last thing, for JSON media type detection, the classifier returns true in three cases only: when the essence is exactly `application/json`, when the essence is `text/json`, or when the subtype ends with the `+json` structured-syntax suffix like `application/vnd.api+json`. If there are media-type parameters such as `; charset=utf-8`, ignore those and just check the essence.
