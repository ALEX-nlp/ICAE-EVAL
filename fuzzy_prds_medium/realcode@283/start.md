## Product Requirement Document

Hey team, we've been getting complaints from a few enterprise customers who say their backend services are running out of connections under load and things just hang or crash in weird ways. We need to build some kind of connection/resource pooling layer that handles this gracefully. The basic idea is: you have a limited number of expensive things (think DB connections, HTTP clients, whatever), and you need to lend them out, get them back, and reuse them without leaking or deadlocking.

We talked about this in the last architecture sync — basically the same pattern we used in that retry/backpressure module but applied to object lifecycle. Key things I remember being discussed: there should be a way to cap how many things exist at once, some warmup behavior so we're not cold-starting on the first real request, a queue for when everything is busy (but also a way to say 'don't queue at all'), and clean shutdown. Oh and someone mentioned we need to handle the case where a caller tries to return a resource twice or both return and destroy it — that was causing double-free bugs in the old code.

Also the reuse order matters apparently — FIFO vs LIFO style — and there's some eviction logic for stale resources. I don't remember all the exact thresholds but it was something like 'if it's been sitting idle too long, toss it'. Context passing per-request was also brought up but I'm fuzzy on how that interacts with reuse. Can someone just build this out properly?

One extra pass on this since a few folks asked for edge-case specifics: when the pool is already at maximum allocation and there are no idle resources left, what happens next depends on max_pending. If max_pending is 0, the borrow request should fail right away with error=pending_acquire_limit and reason=no_pending_allowed. If max_pending is 1 and there’s already one waiter in the queue, then the next request should fail with reason=queue_limit_reached. Also worth calling out that the pending count is observable before the failure occurs.

Also, despite the earlier warmup talk, the pool does NOT allocate any resources at construction time even if a minimum is configured. The constructor.allocated count must be 0. The first borrow is what triggers warmup, and that allocates up to the configured minimum. After warmup, one resource is borrowed and the remaining (minimum - 1) newly allocated resources become idle. After that, subsequent borrows should consume idle resources without triggering new allocations.

On reuse order, this is a real config and not just an implementation detail. The reuse_order field should accept 'lru' or 'mru'. 'lru' means least-recently-used, so the oldest idle resource comes back first. 'mru' means most-recently-used, so the newest idle resource comes back first. Concrete example: with mru and 3 idle resources numbered 1, 2, 3 released in that order, the first reacquire returns 3 and the second returns 1 (skipping 2 because it was released second, making it middle-aged). This is the same idle-pool selection behavior we talked about around Feature 3.3 idle_reuse_order scenario.

Another important safety thing: a borrowed resource can only be completed once, and the first action wins. If release happens first, the resource becomes idle, release_count increments by 1, destroy_count stays 0, and any later invalidate is silently ignored with no metric changes. If invalidate happens first, allocated count drops to 0, destroy_count increments by 1, release_count stays 0, and any later release is silently ignored. That should close off the double-complete/double-free weirdness from the old code.

We also need the disposed state to be explicit. Before dispose() is called it is false, after it becomes true. Any borrow attempt on a disposed pool must fail immediately with error=pool_shutdown and must not allocate a new resource or return any resource value.

Last thing, the permit side of this should follow the CapacityPermit abstraction in the pool, specifically BoundedPermitStrategy and UnboundedPermitStrategy, which handle permit accounting analogous to backpressure semaphores; see Feature 1 in start.md and the size_strategy_sequence / unbounded_strategy_sequence scenarios. That part should line up with the general capacity behavior rather than being a separate one-off path.
