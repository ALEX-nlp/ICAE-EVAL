## Product Requirement Document

Hey team, we need to build out the time-tracking plain-text engine we've been talking about. The core idea is that people write their work days in a simple text file — date at the top, some notes, then a list of time entries underneath — and the system can parse all of that, validate it, compute totals, and spit it back out in a clean format. Think of it like how we handled the structured-input parsing in that invoicing module we shipped last quarter, but for time logs.

The tricky parts are around the clock time handling — people write times in all sorts of ways (12h, 24h, crossing midnight, etc.) and we need to support all of that gracefully. There's also the question of what happens when someone leaves a timer running and never closes it — we need to handle that 'open' state sensibly depending on the current time. Date grouping (weeks, months, quarters, years) needs to work correctly even at year boundaries. We also want to flag likely mistakes — duplicate open timers, overlapping time blocks, future-dated entries, that kind of thing.

Outputs should include parsed structure, totals, diffs against targets, filtered views by date range or tag, and sorted listings. The whole thing should be well-organized code, not one giant file. We'll have a thin JSON adapter for testing but the core logic should be totally separate from that.

One more pass on the edge cases the team asked about. For duration formatting, zero minutes needs to come out as '0m' everywhere, including the signed version. So even in signed form we do not stick anything on it — it stays '0m', not '+0m'. On parsing, if a duration includes both hours and minutes, that minute piece has to stay in the 0–59 range. So '1h60m' is rejected with error 'unrepresentable_duration'. But if there’s no hour part, minutes can go past 59 just fine, so '120m' parses to 120 minutes without error.

On the open-range behavior in hypothetical_total, let’s keep it narrow. An open range only gets closed at the reference timestamp if the record date is the same day as the reference timestamp or the day before. If it’s from an earlier record, it does not count. The 'ongoing' flag is true only when an open range was actually closed and contributed time to the total. The reference value for this is the 'until' ISO-8601 timestamp parameter, and yes, the reference timestamp is provided as an ISO-8601 string with UTC offset. This is the same rule for the hypothetical_total feature (feature9_3): open ranges are closed at a reference timestamp only when the record date is the same day or the previous day relative to the reference; controlled by the 'until' ISO-8601 timestamp parameter.

For filtering, date ranges are inclusive, so records on the boundary dates stay in. Tag filtering is a little more specific than just “record matches or not”: a record is included if its summary OR any of its entries contains the required tags, but the total_minutes in a tag-filtered result should count only the minutes from entries that match the tag, not the full record total. If a record ends up with no tag-matching entries, exclude it entirely.

On week grouping, we’re using ISO weeks, so Monday through Sunday. Given any date, period_week returns the Monday and Sunday of the week containing that date as since/until, plus the Monday and Sunday of the immediately preceding week as previous_since/previous_until. This needs to behave correctly across month and year edges, so for example 2009-01-02 belongs to the week starting 2008-12-29.

Warnings are just four types, and each one reports its kind and the date of the offending record. Those are (1) 'unclosed_open_range' — a record in the past (before the reference date) that contains an open range entry; (2) 'future_entry' — any record dated after the reference timestamp's date; (3) 'overlapping_ranges' — two or more time ranges within a single record whose intervals overlap; (4) 'record_exceeds_day' — a record whose total tracked time exceeds 1440 minutes (24 hours). Also, keeping the exact phrasing here because someone asked, s date; (3) is part of that list exactly as written above.

For the plain-text output, canonical serialisation should use 4-space indentation for entries beneath a record date, and continuation summary lines for an entry use 8-space indentation. Range formatting has a single space on each side of the dash, exactly like '11:25 - 17:10'. And the should-total annotation format is '(Xh Ym!)' using the compact duration form inside the parentheses and then the !.

On parse errors, each one should report 'code', 'line', 'position', and 'length'. 'line' is 1-based. 'position' is a 0-based character offset within the line excluding leading indentation/whitespace. 'length' is the number of erroneous characters. The example to follow is: in '2020-01-01 (asdf!)' the malformed should-total 'asdf' starts at position 12 (after '2020-01-01 (') and has length 4.

And just to restate the architecture bit since it came up again, the JSON stdin/stdout adapter pattern described in the Architecture section still stands: a thin execution adapter translates JSON commands into domain calls without coupling core logic to I/O — see the 'Strict Separation of Concerns' NFR in start.md.
