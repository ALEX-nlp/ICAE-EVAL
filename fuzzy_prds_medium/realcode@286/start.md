## Product Requirement Document

Hey team, we need a codec library for the real-time messaging layer we're building. Basically the idea is that our app developers shouldn't have to worry about the low-level wire format stuff — they just say 'send this event to this namespace with these args' and the library handles everything underneath. We had some complaints recently that people are getting subtly wrong frames when they hand-assemble these strings, especially around binary data and the multi-version support (remember how we handled that versioning split in the auth module last time? something similar applies here). The library needs to handle building connection URLs, encoding outbound messages of various kinds, and decoding incoming frames back into structured data. There's also something around binary attachments that needs to be tracked across multiple incoming frames before a message is considered complete. One thing that keeps coming up in complaints is that ports in URLs sometimes show up when they shouldn't, or get dropped when they should be kept — we need that to be airtight. Output should be line-oriented key=value pairs. Please keep the code organized across multiple files, not one big blob. The exact rules for how the v3 vs v4 protocol differences work for connect frames are a bit fuzzy to me, so make sure whoever picks this up digs into the spec carefully.

Quick follow-up from the questions that came in: the port behavior needs to be decided by the scheme and the actual port together, not just “is there a port.” For http/ws, port 80 is the default and must be dropped. For https/wss, port 443 is the default and must be dropped. If the port does not match the default for that scheme, we always keep it, so https on port 80 keeps `:80`, and http on port 443 keeps `:443`. Same rule no matter whether the transport is websocket or polling.

Also wanted to tighten up the connect-frame split between protocol versions because that was one of the fuzzier parts. Under protocol v3, the default namespace, meaning null/empty, produces no frame at all, so output `frame=null`. If it’s a named namespace, it should be `40<namespace>` and it can optionally have `?key=value` query params inline, then it ends with a trailing comma, like `40/test?key=value,`. Auth does not get embedded there. Under protocol v4, the default namespace always sends a connect frame, which is just `40`. A named namespace is `40<namespace>,`. If there’s an auth object, serialize it as compact JSON and append it directly after the prefix or after `<namespace>,`, so `40{"userId":1}` or `40/test,{"userId":1}`. Query params do not go in the v4 connect frame at all; those belong only in the handshake URL.

On the binary side, if any argument inside an event or ack contains binary data shown as `{"$bytes": [...]}`, each one gets pulled out as its own ordered attachment. In the original payload it gets replaced with `{"_placeholder":true,"num":N}` where N is the zero-based attachment index in argument order. That also changes the frame type: event `42` becomes `45`, ack `43` becomes `46`, and the text frame gets `<count>-` inserted right after the type digits, like `451-` when there’s one attachment. After the `frame=` line, emit each extracted payload as its own `attachment=<comma-separated byte values>` line.

And for decode, a binary message is not complete off the first text frame. A `45` or `46` comes in as one text frame that declares the attachment count, then that many raw byte-array frames after it. We should not emit anything until all declared attachment frames have arrived. Until then, nothing is output. Once they’re all in, emit `type=binary_event` or `type=binary_ack`, include the event name for `binary_event`, include namespace if present, include id if present, include `attachment_count=<N>`, and set `data=<JSON array with placeholders still in place, minus the event name element for binary_event>`.

So the short version is: v3 encode_connect omits default-namespace frames and puts query params inline, while v4 always sends the default-namespace connect frame and uses auth JSON instead. And on URLs, the default-port drop rule is scheme-relative: port 80 is dropped for http/ws, port 443 is dropped for https/wss, but a non-default port for that scheme must be preserved exactly.
