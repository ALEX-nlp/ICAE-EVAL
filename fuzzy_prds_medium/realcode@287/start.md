## Product Requirement Document

Hey team, we need to ship the interweave rich text renderer. Basically it needs to handle a bunch of stuff — taking user content (could be plain text, could be HTML snippets) and spitting out safe rendered HTML. Core things: wrap content in the right container tag (or just a fragment if needed), handle when people paste in HTML with dangerous stuff like scripts or onclick handlers (strip those out), normalize attributes so things like datetime and aria- and data- attributes survive but sketchy ones don't, and auto-convert line breaks in plain text to break tags (but not when disabled). Also needs to linkify content — emails, hashtags, regular URLs, and IP-based URLs should all become clickable anchors, but don't nest anchors inside existing anchors. There's also emoji stuff — matching unicode, shortcodes, emoticons and packaging emoji records with canonical shortcode/primary shortcode/skin metadata. For validation, non-string content and full HTML documents should come back with the right neutral error codes — similar to how we handled the rejection logic in that auth-input module we did before. Pattern matching for emails, hashtags, URLs, and IPs needs to expose the individual parts too. Make sure the core logic is totally decoupled from the stdin/stdout adapter layer. Refer to the existing test cases for exact output shapes.

One quick follow-up from the questions that came in: if no tagName is provided, the default outer container is a <div> element. If tagName is 'fragment', no wrapper element is emitted and only the inner rendered content is output. On the line break side, newline → <br/> only applies to plain text input. If the content already contains HTML markup, like existing <br> tags or other HTML elements, it is not treated as plain multiline text, so newlines stay as literal characters instead of being converted. That conversion can also be turned off with options.disableLineBreaks=true.

For autolinking, a few edge behaviors matter. Anchors we generate should always include rel="noopener noreferrer". If the input already has an <a> element, we leave it alone as-is, including its rel or href, and anything already inside that existing <a> should not get wrapped again. Also, surrounding punctuation stays outside the generated anchor, so '(http://domain.com/path)' should come out as '(<a href="...">http://domain.com/path</a>)' with the parens outside the anchor element. For URL matching, the hostname matcher with kind=url rejects localhost, raw IPv4 addresses, unsupported protocols, and malformed hosts. Only http/https or schemeless are supported there, and raw IPv4 addresses need to be matched with kind=ip instead. The IP matcher also rejects IPv6 bracket notation and out-of-range octets (>255).

On sanitizing styles, if a style attribute contains CSS functions that pull in external resources or code, like url(), image(), expression(), those declarations get removed. Safe declarations like color or font-size should stay. If everything in style is unsafe after filtering, then the style attribute should be omitted entirely.

For emoji, emoji_match returns a single-line JSON object followed by a newline. With convertUnicode=true, the fields are hexcode, match, unicode. With convertShortcode=true, the fields are hexcode, match, shortcode. In both cases, fields are output in alphabetical key order. And package_emoji should take the raw emoji record and return the full merged object sorted by key, followed by a newline, with the added/transformed fields canonical_shortcodes, primary_shortcode, unicode, and skins. canonical_shortcodes is the array form with colons, like [':cat:'], primary_shortcode is the first canonical shortcode, like ':cat:', and skins is an empty array if there are no skin variants.

Also confirming the validation behavior for feature 5: parse_content returns 'error=invalid_content_type' for non-string content, and validate_document_fragment returns 'error=unsupported_document' for full HTML documents containing <!DOCTYPE>, <html>, <head>, or <body> tags. And for the adapter contract, the JSON files under rcb_tests/public_test_cases/ (feature1 through feature12) are the source of truth for exact stdin/stdout shape, including field ordering, newline placement, attribute casing like dateTime, and alphabetical JSON key ordering in the emoji responses.
