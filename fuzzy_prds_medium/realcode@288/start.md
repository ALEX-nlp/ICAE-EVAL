## Product Requirement Document

We're building a small set of helper utilities for tools that work with cluster configuration records. The main pain point is that we keep re-implementing the same fiddly low-level operations in every tool: making bulk edits to nested configuration documents, deciding whether a resource matches a selection filter, figuring out when a workload was actually running versus just defined, and cleaning up the machine-managed bookkeeping fields before re-applying a record. Without a shared place for this logic, different tools give different answers for edge cases — especially around selector rules and the tricky question of 'is this collection entry still running?' We want these handled the same way as our existing resource-lifecycle module. Each utility should be a pure function: same input always gives the same output, nothing external. The bulk-edit helper needs a way to target every item in a list at once using a wildcard, and it must handle the case where the target key already exists — sometimes we want to overwrite it, sometimes leave it alone. The filter matcher needs to support both 'match one of these values' and 'does this key exist at all' style checks. The running-interval logic needs to collapse all the per-container start and end times into one clean interval, and only consider the job finished once every main container is done. Deliverable is a clean library plus a CLI adapter that reads commands from stdin and writes results to stdout.

One small follow-up from the team’s questions: the CLI should emit compact JSON with no whitespace, object keys sorted lexicographically, and every output line should end with a newline character. For the bulk edit behavior, overwrite=true: means the key is written even if it already exists. overwrite=false: means existing keys are left unchanged, but missing keys are still added. And for removals, patch_remove leaves locations that do not contain the key unchanged.

Also, on the running window piece, the end is only reported once EVERY main container has finished. If any main container is still running or waiting, the result is state=Running with no end. Only when all main containers are finished does the state become Finished with start and end timestamps. Init containers do not block the end.
