## Product Requirement Document

Hey team, we need to get this web-native bridge toolkit wrapped up. The core idea is that our mobile app embeds web content and right now every team that uses it ends up writing the same boilerplate glue code to talk between the web layer and the native side — parsing messages coming in, building reply messages going out, figuring out why a page failed to load, that kind of thing. It's a mess and everyone's doing it slightly differently.

We need a clean library that handles all of this consistently. Things like reading structured data out of web messages, sending replies back, dealing with those annoying JSON converter situations (you know the two cases we always hit), summarizing page load responses without dumping the whole HTML, categorizing HTTP failures and browser-level errors and SSL cert problems, and then the path config + routing logic — basically the same approach we used for the route matching in the login flow, just applied here.

One thing that keeps biting us: when HTML is really long we don't want to log the whole thing, just enough context from both ends. And the routing needs to handle those sneaky credential-stuffed URLs properly — we got burned by that before.

The whole thing needs to be structured properly, not one giant file. Tests should drive it. Can someone pick this up and make sure all the edge cases from the existing test suite are covered?

Quick follow-up from the questions that came in: for the HTML summary bit, the preview should be based on normalized HTML, meaning whitespace is collapsed and newlines are stripped before we decide whether to shorten it. From the test behavior, a 26-character string is not truncated, but a 75-character string is. When it does get shortened, we keep roughly the first 26 characters and the last 26 characters of that normalized HTML and join them with ' [...] '. Separately, response_length should still report the original byte/character length before normalization, accurately.

Also on bridge payload encoding failures, there are only two buckets we should emit and they need to stay stable. If there is no converter at all, that is 'no_json_converter'. If a converter is present but not actually a valid/functional converter object, that is 'invalid_json_converter'. In those cases we should not leak runtime exception names or generated messages — the output should just be 'error=no_json_converter' or 'error=invalid_json_converter'. Same thing reflected in feature3_bridge_payload_encoding_errors.json: converter='none' producing error=no_json_converter, and converter='invalid' producing error=invalid_json_converter.

For the routing question, if the URL has userinfo before the host, like 'https://app.my.com@fake.domain', we need to compare against the real authority host, which is 'fake.domain', not 'app.my.com'. So start_host matching should always use the correctly parsed target host. And just to make this explicit: decision and matches are separate. route_kind drives decision every time — 'app' means decision=navigate no matter what, and 'external_browser' plus 'external_browser_tab' mean decision=cancel no matter what. Then the 'matches' field is just whether the parsed target host equals the parsed host of the start location, true or false. That’s the same behavior called out in feature9_navigation_route_classification.json: comparing the parsed host of the target URL against the parsed host of the start_location URL, with decision determined by route_kind (app=navigate, external_browser/external_browser_tab=cancel) and matches determined by host equality.

Last thing on payload decoding: the only built-in schema we support right now is 'title_subtitle'. That expects a JSON object with 'title' and 'subtitle' string fields and should output exactly 'title=<value>\nsubtitle=<value>\n'. If the schema name is unknown or unsupported, like 'unsupported_empty_shape', we should return 'data=null\n' instead of throwing. The idea is the core decoder stays narrow and future schemas can be added via extension without changing that core path.
