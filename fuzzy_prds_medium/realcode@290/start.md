## Product Requirement Document

hey team, we need to get this email toolkit wrapped up for the transactional messaging service. basically devs are complaining they have to hand-wire everything themselves every time a new service needs to send emails — it's a mess and keeps breaking in prod when someone forgets a header or messes up an attachment. we want a clean library that handles building email objects, sticking files onto them, and rendering the body from templates without everyone having to learn a new engine from scratch each time.

the value objects need to blow up early if someone forgets something important — similar to how we handled the validation in that user-profile module, just make sure errors are specific enough that devs know exactly what they forgot. for images embedded in the body, there's a fixed set of supported kinds (check the existing catalog logic) and each one needs its mime type resolved automatically.

for templates we want to support a few different engines — the ftl one, a couple of html-based ones (there are at least three of those, one is the logic-less one, one does inheritance, and there's that natural-templating one). wrong extensions or blank names should fail clearly, and broken templates should also have their own distinct failure mode.

attachments should auto-detect content type from actual bytes when no type is given explicitly. the whole thing should be modular — no god files. prioritize getting the test harness wired up so we can validate everything end to end.

one quick follow-up from the questions that came in: for the email value object, the required stuff is just 'from', 'to', 'subject', and 'body'. if any one is missing, fail right away at construction time and keep the output exact and simple as two lines only: 'error=missing_required_field' and 'field=<fieldname>' where fieldname is one of: from, to, subject, body. same validation pattern applies in Email, InlinePicture, and Attachment value objects as well, so each mandatory field gets checked up front and the failure shape stays exactly 'error=missing_required_field' plus 'field=<fieldname>'.

also on recipients and headers, let’s keep the output predictable. if there are multiple recipients in the same bucket, join them with a semicolon (';'), so two primary recipients come out as 'to=roberto@gmail.com;titus@de.natura'. if a recipient bucket isn’t provided, we still print it empty, so 'cc=' and 'bcc=' should still be there. for custom headers, output the count first as 'custom_header_count=<n>', then each one as 'custom_header.<key>=<value>', sorted alphabetically by key. if there are no custom headers, the count is 0 and there shouldn’t be any individual header lines after that.

for template behavior, Pebble and Thymeleaf both expect the 'html' file extension. for Pebble, a blank reference or wrong extension should return error=invalid_template_name with engine=pebble, and a syntactically broken template like wrong_email_template.html should return error=template_processing_failed with engine=pebble. for Thymeleaf, a blank reference or wrong extension should return error=invalid_template_name with engine=thymeleaf. it also needs to support attribute-level expressions for both text content and hyperlink targets, like th:text and th:href, and the model can include an 'activationLink' variable that gets substituted into anchor href attributes. rendered output needs to match verbatim, including the resolved href value.

for the standard template check, FreeMarker, Mustache, and Pebble should all render email_template with model {name: 'Titus'} to exactly '<!doctype html>\n<html>\n<body>\n<p>\n    Ciao <em>Titus</em>!\n</p>\n</body>\n</html>\n'. Thymeleaf is intentionally a little different there: no doctype line, and it includes an additional activation link paragraph.

last thing, just to make the boundaries clear, the thin execution adapter is the piece that reads JSON commands from stdin line by line, one JSON object per line, looks at the 'op' field, routes to the right core library method, and writes results to stdout. the core library itself should know nothing about JSON, stdin, or stdout. and for the image catalog, the supported kinds in order are GIF (extension=gif, content_type=image/gif), JPG (extension=jpg, content_type=image/jpeg), JPEG (extension=jpeg, content_type=image/jpeg), PNG (extension=png, content_type=image/png). that list needs to be available in exactly that order, each kind needs to be lookable up by name, and InlinePicture uses that same ImageType catalog to resolve content_type from an imageType name.
