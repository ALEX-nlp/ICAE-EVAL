## Product Requirement Document

Hey team, we need to put together that shared plumbing library we keep talking about for the annotation-driven framework. Right now every app team is doing their own thing for basic stuff — figuring out where their classes live, running startup steps in the right order, turning handler responses into something the client can actually read, and dealing with messy incoming request values. It's causing inconsistencies and a bunch of copy-paste bugs.

Basically we need four or five small building blocks. One for reading a project config and knowing which packages to scan (the convention stuff we agreed on, like we did for the login module layout — check how that was structured). One for running async startup tasks one-at-a-time so nothing fires out of order. Something to track what kind of value a handler actually returned so downstream logic knows what to do with it. Then serialisation — both the happy-path response and error/status outcomes — into whatever format the client asked for (we're supporting at least two formats). And finally a converter for raw incoming text params so handlers get proper typed values and bad input doesn't blow up with internal stack traces leaking out.

Errors should never expose internal runtime details to the client, ever. Keep output consistent across all the formatters. The async piece especially needs to be rock-solid — order must be guaranteed. Let me know if anything's unclear, I'm around.

One quick follow-up since a few details came back from the team chat. For the package layout bit, the convention is very literal: we append '.controllers' for the controller role, '.verticles' for the verticle/component role, '.fixtures' for the fixture role, and '.domains' for the domain role directly onto the value of the 'src-package' key in the config descriptor. Also worth calling out that the explicit package keys win if they’re present, so 'controller-packages', 'verticle-package', 'fixture-packages', and 'domain-package' should override the derived values. The controller and fixture ones are multi-valued lists of package strings and when we write them out they’re joined with commas, while 'verticle-package' and 'domain-package' are just single-valued strings. This is the same package layout resolution logic in Feature 1, and it should line up with feature1_config_layout.json and the config descriptor handling in the core library.

For the async startup chain, the output format needs to stay consistent and simple: three lines total. First is 'order=<accumulated string>' where each task appends its zero-based index followed by a semicolon, like '0;1;2;'. Second is 'length=<count>' for however many tasks were scheduled. Third is 'completed=yes' if the chain finished successfully. If there are no tasks, that first line is still there and should just be 'order=' with an empty string. That should make it really obvious that ordering was preserved and nothing ran out of sequence.

On the payload holder, a brand new one should report the sentinel type token 'Void' with a capital V. After a value gets set, it should report the concrete type instead: 'Dog' if it’s an object value with name and breed attributes, or 'String' if it’s a scalar/text value. In both cases we print the initial type and the post-set type one per line as 'type=...'.

And for serialisation, the Dog entity needs to emit all bean-style properties in this exact order: 'age' first, and that is null when not assigned, then 'name', then 'breed', then 'puppy', which defaults to false. JSON should stay compact with no extra whitespace, for example {"age":null,"name":"Snoopy","breed":"Beagle","puppy":false}. That same consistency rule still applies across the rest of the formatters too.
