## Product Requirement Document

Hey team, we need a fixed-point math library for embedded targets that can't rely on hardware floating-point. The idea is that every number has a scaling factor baked into its type so the compiler tracks precision automatically instead of us doing it by hand everywhere. We had something like this in the signal processing module a while back but it was all ad-hoc — please don't repeat that mess.

Basically we need: storing real numbers as scaled integers, printing them back as readable decimals, the usual math operators that behave like normal machine integers (fast, predictable), but also 'safer' named versions that don't silently lose bits when results get big. We'll also need sin/cos/sqrt for some geometry work, and the ability to pick what happens when a number doesn't fit — sometimes we want it to wrap, sometimes clamp, sometimes just blow up loudly. Oh, and some users are working with astronomically large numbers so we need a big number mode too. Finally there was that leading-bit counting thing from the DSP pipeline — include that.

The code should NOT be one giant file, please keep things separated sensibly. The JSON test harness is just for validation, the actual library should be usable without any of that JSON stuff. Reach out if anything is unclear, there are a few details I probably glossed over.

One extra pass on the details people asked about: when we store a real number, the integer under the hood is computed as data = truncate(real * 2^frac_bits), and that truncation is always toward zero, so floor for positives and ceiling for negatives. The represented value is then data * 2^-frac_bits. So for example real=3.5 with frac_bits=1 gives data=7 and value=3.5. Also on the signedness bit, use the signed/unsigned representation flag and truncation-toward-zero rounding used in feature2_construction — see store_exact op with signed parameter defaulting to true.

For printing, the decimal output needs to reflect the exact stored value, meaning the exact data * 2^-frac_bits value at full precision. We do not want unnecessary trailing zeros, but we do want enough digits to uniquely show the stored approximation. The easy sanity checks here are data=843314856 with frac_bits=28 renders as 3.141592652, and data=25 with frac_bits=8 renders as 0.09765625.

On the compound-op bit, it starts from a real value and then applies the fixed in-place compound assignment sequence from the contract, printing result=<value> after each step. For real=22.75, the six lines should be exactly result=35.25, result=-0.25, result=-0.0625, result=9.9375, result=-29.8125, result=-30. The actual operators in that sequence are +=, -=, /=, or *= with the specific operands coming from the test contract.

Also just to be explicit about arithmetic behavior: the plain operators (*, +, -, /) stay in the operand’s fixed-width representation and overflow wraps modulo the representable range, exactly like normal machine integer arithmetic. The example to keep in mind is squaring 15.9375 in an 8-bit unsigned 4-frac-bit format, which overflows and wraps to result=14. If somebody wants the safe path, that’s what the named widening ops are for. Named multiply doubles the bit width so it can hold the exact product, so squaring 15.9375 yields exactly 254.00390625. Likewise divide_widen widens the result format so you keep the fractional quotient, so 15/2=7.5, while divide_truncate stays in the operand representation and truncates toward zero, so 1/2=0 and 3/2=1.

And for overflow policy, there are three modes and they apply to both conversion and addition operations. 'wrap' reduces modulo the target range, so 259 into unsigned 8-bit → 3. 'saturate' clamps to the nearest representable bound, so 259 → 255. 'throw' does not produce a value at all, and the output line must be exactly 'error=overflow'.
