## Product Requirement Document

Hey team, we need a serialization utility that can look at our data types and automatically figure out what the wire format should look like — basically no more hand-writing schemas every time someone adds a field. It should spit out something machine-readable (JSON-based) that describes the shape of the data, and also be able to turn actual values into a compact byte sequence and read them back. Think of it like what we did with the nullable handling in the profile service last quarter — same kind of 'wrap it in a union' pattern.

The tricky bits: integers have some collapsing behavior (the narrower ones fold into a single wire type), characters are special-cased with a marker, and text needs a length prefix baked in. For enums specifically, we want the schema to carry documentation, aliases, and a designated fallback symbol — and if someone misconfigures the fallback (more than one), it should fail loudly with a recognizable error token, not a stack trace. We also need the decode side to be tolerant of schema drift, so if the reader's enum is a subset of the writer's, unknown symbols resolve gracefully. Output bytes should be lowercase hex. Please keep the I/O wiring totally separate from the core logic — last time everything got tangled and it was a nightmare to test.

One quick follow-up since a few details came up in review: for the integer collapsing, byte, short, and int all produce the schema string "int", and only the 64-bit width maps to 'long'. Same deal stated another way so there’s no ambiguity: int8 (byte), int16 (short), and int32 (int) all map to the Avro wire type 'int'; only int64 maps to 'long'. When a primitive resolves to just a single type name, we print it as a bare quoted string like "int" or "string". We only switch to a pretty multi-line JSON object when the schema has extra structure, like a record, an enum, or the char case.

On char specifically, it is not a bare primitive string. A character is represented as a 32-bit integer type carrying a logicalType marker, and the schema should be exactly {"type": "int", "logicalType": "char"}, printed as a pretty multi-line JSON object. Also, optional values should always use the two-branch nullable union with null first, so the output is a JSON array [ "null", <underlying-type-schema> ]. That ordering is fixed, and the same rule applies for nullable primitive schema derivation too: [ "null", <primitive-schema> ].

For encoding, a couple specifics to lock down: booleans are exactly one byte, 0x01 for true and 0x00 for false, and they do not use zig-zag encoding. Signed integers do use zig-zag variable-length integer encoding across all the signed widths, including character code points. So all signed integer widths (8, 16, 32, 64-bit) and character code points use zig-zag variable-length integer encoding, and for a concrete example, the integer 44 encodes as the single byte 0x58 (zig-zag of 44 = 88 = 0x58).

And just to reinforce the separation-of-concerns part because that one matters a lot: the core business logic for schema derivation, encoding, and decoding needs to stay completely decoupled from stdin/stdout and JSON parsing. A separate execution adapter should handle reading a JSON request from stdin, calling into the core domain, and formatting the result to stdout. The core must never directly reference I/O.
