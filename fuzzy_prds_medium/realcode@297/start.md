## Product Requirement Document

Hey team, we've been getting complaints from our backend devs that they have zero visibility into what's happening with our Kafka messaging layer. Right now if something goes wrong between a producer and a consumer, there's no way to trace the flow end-to-end without manually adding logging everywhere — which is a nightmare to maintain and people keep forgetting to do it consistently. We need to build some kind of auto-instrumentation wrapper around our Kafka client so that spans get created automatically whenever someone publishes or consumes a message, without the dev having to write any tracing code themselves.

The context propagation piece is really important — similar to what we did with the HTTP middleware tracing last quarter, we need trace IDs to flow through message headers so a consumer can pick up where the producer left off. There's also a question about batches — the PM from the data pipeline team mentioned their batch consumers work differently from per-message ones and the parent/child relationships should reflect that.

Also a few edge cases we've been burned by before: hooks that crash shouldn't break the whole trace, failed sends should still emit spans, and error messages in our observability tooling should be normalized category labels rather than raw exception strings. We also want to support some kind of version tagging on spans for future compatibility auditing. The adapter needs to read scenarios from stdin and write results to stdout for our test harness.

One extra pass on the batch behavior since a few folks asked: for a batch publish, each message gets exactly one PRODUCER span, and that span is named after that specific message's own topic, not the batch as a whole. Spans are emitted in declaration order across all topic groups, so if you have 2 messages to topic-A and 1 message to topic-B, that means 3 spans total: two named topic-A, one named topic-B. On the consume side, consume_batch and consume_message really do behave differently. For consume_batch, we create one CONSUMER span with operation=receive as the root of a new trace, then one CONSUMER span with operation=process per message, and each of those process spans is a child of the receive span, not a child of any producer span. If those messages already had upstream trace context in their headers, that upstream context is not used as the parent in the batch case. Instead, the batch receive span starts a brand-new trace and the per-message processing span carries a link back to the original producer span. For consume_message, there is exactly one CONSUMER span with operation=process per delivery, and it uses the extracted context directly as parent when that context is present, so it becomes a child of the original producer span and shares the same trace ID. If no upstream context is present in the message headers, it just starts its own root trace.

Also confirming the propagation model is the W3C TraceContext + Baggage propagation pattern: injecting trace context into outgoing message headers at produce time and extracting it from headers at consume time, mirroring how HTTP instrumentation propagates context via request/response headers. If a consume_message step has observeBaggage=true and the message headers carry propagated baggage, we print a 'baggage key=value ...' line after the inject lines but before the spans block. All baggage key-value pairs go space-separated on that single line. That line only shows up for consume steps that explicitly set observeBaggage=true and only when baggage was actually present.

One more important implementation note from the team review: this must NOT turn into a single god file. Client patching, span lifecycle management, context propagation, output rendering, and the execution adapter for the stdin/stdout JSON bridge all need to live in separate modules. The instrumentation core has to stay fully decoupled from stdin/stdout and JSON parsing, and only the execution adapter should handle that translation.

And lastly, on the enrichment hooks, if producerHook or consumerHook throws, we still create the span and it still finishes with status=unset, not error. A bad hook must never corrupt the trace or change the span's status. The only effect is that the custom attribute the hook was supposed to add just won't appear on the span if that hook blew up.
