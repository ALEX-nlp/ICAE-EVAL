## Product Requirement Document

Hey team, we need to wrap up the auth toolkit we've been building. The core pieces are mostly specced out but there are a few gaps I want to flag before we hand this off to the dev team.

First, the password hashing — we need to support multiple strategies and one of them is that silly 'reverse' trick we used in the demo last quarter to prove pluggability works. Make sure the verification output format is consistent across all strategies.

For the permission system, there's a nuance around how denials behave depending on which 'mode' you're in — the strict vs the other one. We had a whole discussion about this in the Figma meeting but basically: one mode lets the person's own settings win, the other lets any denial from anywhere kill the access. Also there's that thing with the star-based key matching and the grouped class keys (you know, the @ syntax thing) — make sure exact keys beat the pattern ones.

The login protection stuff needs three separate sources checked in a specific order, and the delay math has two flavors — a simple cutoff and a tiered one. The entity permission mutations need to respect insertion order when printing.

One thing I'm unclear on is how many sources the throttle gate actually checks and what order it checks them — can someone confirm the priority chain? Also need to nail down the 'create on update' behavior for permission mutations.

Quick follow-up after the questions that came in. On the hashing side, the 'reverse' strategy really is just that trivial custom callback strategy we kept around to prove the strategies are pluggable via a common interface. It reverses the plaintext string to produce the digest, and verification is just checking whether reversing the candidate produces the same digest. Even though it’s intentionally silly, the output still needs to look like the others: include 'algorithm=reverse', include 'hash_differs_from_input=true' because the reversed string must differ from the original, and include a 'verify[candidate]=true|false' line per candidate. More generally, for 'sha256' and 'whirlpool', those are salted digest schemes with a random salt, so the digest is non-deterministic and must not be printed in output. The 'native' strategy uses the platform default hashing scheme, and the 'bcrypt' strategy is an adaptive salted scheme. In all cases, only verification outcomes are printed, never the digest itself, and all strategies must produce a digest that differs from the plaintext input (hash_differs_from_input=true).

For the permission behavior, the naming in output should be 'policy=<lenient|strict>' for the 'permissible_policy' op, not 'mode='. After that, the query lines stay in the same shape as the permissions op, so 'all[key1,key2,...]=true|false' or 'any[key1,...]=true|false'. For 'lenient', the rule is that secondary sets are accumulated first, merged in order, and then the actor’s primary set is merged on top. So the primary set’s explicit grants and denials override whatever accumulated result came from the secondary sets, and if a key is absent from all sets it defaults to denied (false). For 'strict', every contributing set gets folded into one prepared map with the secondary sets first and then the primary set, and an explicit denial (false) for a key always overrides any earlier grant for that same key. So if ANY contributing set denies a key, that key stays denied no matter what the primary set says. Also, the Class@method1,method2 grouped permission key syntax expands into the individual Class@method entries during permission resolution, and exact keys should still beat the pattern ones.

On the auth gate and mutation bits, when the activation gate blocks someone because activation is not completed, the output should be exactly 'error=not_activated' and 'user_attached=true'. That user_attached bit is there because callers need the structured user reference to identify the blocked account. If activation is completed, the output is just 'outcome=allowed'. For permission mutations, after all operations are applied, if no keys remain in the permission map, the output is the single token 'permissions=empty'. Otherwise, print keys as 'key=true' or 'key=false' lines in insertion order.

And for the throttle config question, the config op reads back all six configured values and the order is fixed: 'global_interval=N', 'global_thresholds=N', 'ip_interval=N', 'ip_thresholds=N', 'user_interval=N', 'user_thresholds=N'. Each source — global, ip, user — has its own independent interval and threshold settings.
