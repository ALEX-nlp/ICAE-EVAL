## Product Requirement Document

Hey team, we need a small utility library for the pooling system we've been building. Basically it needs to handle a few things: expressing time-based wait periods in a way that doesn't break when different parts of the code use different units (you know how that's been causing those subtle comparison bugs), some logic for deciding when a pooled resource is 'too old' to reuse, and a config object for the pool itself. For the age stuff, we need at least two flavors — one simple cutoff, and one that spreads retirements across a window (similar to how we handled the jitter logic in that old connection module). There should also be a way to stack two of those age rules together so both can be enforced at once. The config thing is pretty straightforward, just needs a size and two on/off switches — check the existing defaults we agreed on internally, I don't have them in front of me. One thing that keeps biting us: bad inputs (like missing units or out-of-range thresholds) are currently bubbling up as raw exceptions which looks terrible in logs. We need those to come out as clean, categorized error messages instead. The boundary behavior on the age checks has also been inconsistent — make sure it's pinned down precisely so we stop getting off-by-one complaints from the team.

One extra pass on the details the team asked for: the system’s base unit is always NANOSECONDS, and every duration value needs to be convertible to nanoseconds so comparisons and storage all land on the same common denominator. So if something comes in as 13 MILLISECONDS, that becomes base_value=13000000 in nanoseconds. Also, zero and negative magnitudes are explicitly accepted for durations — they conventionally mean 'no waiting at all' — and they need to round-trip faithfully with no conversion artifact. There is no lower-bound validation on the duration magnitude itself.

On equality, two durations are equal if and only if they represent the same absolute span of time when both are converted to the base unit (nanoseconds). The units they were expressed in are irrelevant. Equal durations must also produce the same hash value. Durations of different absolute lengths are not equal and must not share a hash. Also, if the time unit is null/absent, construction must be rejected with the neutral two-line output exactly as 'error=null_argument' on the first line and 'param=unit' on the second line. That same behavior applies to duration creation, max_age policy, and spread policy alike, and no host-language exception identity should be exposed.

For the pool config, there are four operations via 'op': 'defaults' reads the freshly-initialized state; 'set_size' sets the integer target size; 'set_precise_leak_detection' sets the precise-leak-detection boolean; and 'set_background_expiration' sets the background-expiration boolean. After any operation, the full snapshot (size, precise_leak_detection, background_expiration) is reported.

And on the error formatting in general, all validation errors should come out as exactly two neutral category lines with no exception class names or stack traces: first line is 'error=<category>' and it can only be either 'error=null_argument' or 'error=illegal_argument', then second line is 'param=<parameter_name>'. Each line ends with a newline. No host-language runtime exception identity may appear in output.

Last thing, for the windowed retirement behavior, the spread/windowed validity policy (Feature 4) is the one with lower and upper age bounds where retirements are distributed across a window rather than at a single cutoff instant, and that’s the one implemented in the spread action handler using lower_millis and upper_millis.
