## Product Requirement Document

Hey team, we need to build out this HTTP-to-backend transcoding layer we've been talking about. Basically the idea is: clients send normal HTTP requests with JSON bodies, and we need to route them to the right handler and convert everything into our internal message format — and do the reverse for responses. Think of it like that API gateway translation work we did on the billing service, but generalized.

The big pain points we're hearing from developers: they keep writing the same URL parsing and JSON mapping glue code over and over, it's inconsistent across services, and whenever someone adds a new endpoint they have to touch like five different files. We want one declarative place to define routes and schemas.

The system needs to handle path patterns with variables (including nested ones), action-style suffixes at the end of paths (you know, the colon thing), query parameter extraction, and both single-message and streaming flows in both directions. There's also some nuance around how percent-encoded characters in URLs get decoded — single-segment vs multi-segment captures behave differently, which has been a bug source.

For the message format side, field names come in camelCase from JSON but need to map to the snake_case internal names. Unknown fields should just be dropped. We also need to support placing a JSON body under a specific sub-field rather than at the top level.

More detailed specs are in the usual place. Please make this production-grade, not a quick hack.

Quick follow-up from the questions that came in: on the route side, a path pattern is invalid if it does not start with '/', contains empty path segments like double slashes, or tries to stick a variable expression inside a literal segment, so '/a{x}' is invalid because '{x}' is embedded in the literal 'a'. Other than that, the usual shapes are fine, including wildcards, double-star wildcards, named variables with sub-patterns, and the optional colon suffix behavior at the end. That terminal action suffix should work the way we described in Feature 4, meaning a colon-separated suffix at the end of the final path segment like ':verb', and it only acts as a route discriminator when that suffix is actually configured in a registered route pattern.

Also, for the route debug output, in the 'segments=' line we should expand each variable into the pattern it matches instead of printing '{field}'. So '{field}' and '{field=*}' show up as '*', '{field=literal/*}' expands to its literal segments followed by '*', and '{field=**}' or '{field=literal/**}' expands to the equivalent wildcard segments. The variable name itself does not appear there. For variable reporting, each one should come back as 'field.path[start:end:single|multi]' where start and end are zero-based segment indices, start inclusive and end exclusive, counting from the first segment after the leading slash. 'single' means it captured exactly one segment, and 'multi' means it used '**'. If there are several, separate them with semicolons.

One more route-selection detail: if multiple routes match, we want the most specific one to win. An exact literal segment beats a single-segment wildcard (*), which beats a multi-segment wildcard (**), and routes with more literal segments rank higher. The route_index we return is the zero-based index of the winning route in the original routes array. This all sits in the HTTP/JSON transcoding architecture described in start.md, specifically the route matching (Features 2-4) and payload translation (Features 5-8) subsystems that together form the transcoding pipeline.

On the payload side, the built-in schemas for testing are fixed. 'shelf' has string fields 'name' and 'theme'. 'book' has string fields 'name', 'author', 'title', and a nested message field 'author_info' containing string fields 'first_name' and 'last_name'. 'create_book_request' has a nested 'book' field of type 'book'. These schemas drive both json_to_proto and proto_to_json translation.

For JSON output, by default we omit fields that are sitting at their default/zero values, so empty string, 0, and false do not get printed. If 'always_print_primitive_fields' is set to true, then all primitive fields defined in the schema should be included in the JSON output even when they still have the default value, for example an empty string field should appear exactly as '"fieldName":""'. And for streaming proto_to_json mode, collect all response messages and write them as one JSON array on a single line, like 'json=[{...},{...}]'. In non-streaming mode, it stays a single plain object like 'json={...}'. If we hit frame errors such as 'incomplete_frame_header', report them as 'error=<category>' and stop processing.
