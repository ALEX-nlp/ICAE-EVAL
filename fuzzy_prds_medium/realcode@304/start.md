## Product Requirement Document

Hey team, we need to get the XML toolkit shipped. The core idea is that devs can hand us a chunk of XML text and we do the right thing with it — parse it, poke around inside it, spit it back out nicely, that kind of thing. We also need the builder side so people can construct documents in code instead of concatenating strings like animals.

A few things came up in the last review that I want to make sure we nail: the serializer needs to be 'canonical' in the same way we did the JSON normalizer last quarter — you know, clean up the unnecessary fluff in tags without breaking the actual content. The pretty-printer should handle mixed content at the top level gracefully, not just nested elements. And the entity/escaping stuff needs to support at least the basic XML names plus the HTML ones people always ask about — including the case-insensitive hex thing that tripped us up before.

For errors, don't just throw a generic crash — we need position info so devs can actually debug their files. The query feature should let people filter by both name and namespace, and there's a wildcard mode that has to work correctly. Finally the builder ops need to cover declarations, processing instructions, comments, CDATA, text, elements, and attributes. Attribute quote style should be preserved throughout. Please make sure you check how the old escaping module handled the double-encoding edge case — same logic should apply here.

One more pass on the XML bit since a few specifics came up. On serialization, we only want to normalize the optional space before a self-closing slash, so `<root key="value" />` should come back as `<root key="value"/>` and `<foo />` → `<foo/>`. Other than that, leave the inside of the tag alone. Also, if something was written as `<foo></foo>`, it needs to stay `<foo></foo>` and must NOT be collapsed to `<foo/>`. More broadly, we should preserve the rest of the structure exactly, including attribute quoting, CDATA, comments, PIs, and DOCTYPE declarations exactly.

On inspection, the `self_closing=true` flag is about how the source was written, not whether the element happens to be empty. So `<foo/>` reports `self_closing=true`, while `<foo></foo>` reports `self_closing=false`. Weird but important wrinkle: `<ns:data key="value">text</ns:data>` also reports `self_closing=true` in the inspection output because the source form was not an empty open/close pair — the field reflects whether a separate closing tag was present in source, not whether the element is empty. Also for counts, `descendants` includes ALL descendant nodes: direct children, their attributes, and all deeper descendants. Attributes are counted as nodes in the descendant total. So `<ns:data key="value">text</ns:data>` has 1 child (the text node) plus 1 attribute = 2 descendants.

On attributes, we do need to preserve quote style all the way through parsing and serialization. The `attributes` action should report `quote=single` or `quote=double` based on the original source, and the `serialized` field for each attribute should re-use that same style. So if we got `ns:attr='Am I or are the other crazy?'`, it should serialize back exactly with single quotes.

For entity handling, if a named reference is not in the selected table, like `&invalid;` under the `xml` mapping, we should leave it completely untouched — pass the literal string `&invalid;` through verbatim. Same deal for references with no closing semicolon or empty references. And if the mapping is `none`, return the entire input unchanged with zero processing. Also please make sure numeric hex references work in both forms: `&#x41;` and `&#X41;` should both resolve to the same code point, and the hex digits themselves need to be case-insensitive too.
