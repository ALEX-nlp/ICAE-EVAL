## Product Requirement Document

Hey team, we need to get this GrAMPS-style GraphQL composition layer wrapped up. The idea is basically what we scoped out in Q3 — one HTTP endpoint that works in both dev and prod, with the dev mode returning placeholder data and prod hitting real resolvers. We also need that interactive docs page (you know, the browser-based query explorer thing) available during development. There's also the middleware piece where we attach context and data sources per request — similar to how we handled the auth tenant stuff in the old login module, so check that for reference on how context keys get merged.

On the schema side, sources should contribute their own query fields and the built-in root ops (ping and version) should always be present. Dev sources loaded from external paths should be able to swap out local ones when the context key matches, and we should log something when that happens. For outbound calls, connectors need consistent request config (JSON mode, full response, etc.), and GETs should play nicely with Redis caching — serve cached when available, refresh in background, skip cache on errors. Mutations go out as POST or PUT with JSON body. Errors need to be normalized so clients never see raw stack traces or internal exception names, and the format should differ slightly between dev and prod (some fields hidden in prod). Make sure the model base classes blow up properly if someone tries to instantiate them directly.

One quick follow-up from the team questions: for the built-in root ops, the contract is fixed. In development/mock mode, `grampsVersion` returns `"Hello World"`, and `grampsPing` returns `"GET OFF MY LAWN"` in both mock and live modes. Those built-ins are always part of the composed schema regardless of what sources are loaded, so the root operations are `grampsVersion` (Query) returning `"Hello World"` and `grampsPing` (Mutation) returning `"GET OFF MY LAWN"`. Also, the `/graphiql` endpoint should be there for Feature 2 and return `text/html` with `containsGraphiql: true`, HTTP 200, basically the in-browser GraphQL IDE behavior we talked about.

On the middleware/context bit, Feature 3 should stay as a flat merge. Extra context keys like `tenant` sit alongside data source model keys like `Catalog` and `Inventory` in the execution context object, same general shape as the old auth tenant pattern, just merged together at the top level.

A couple specifics on connector behavior too. For batch loading with `connector_load`, call each URI separately and return a `calls` array listing each URI in order, plus a `result` array with one entry per input URI in the same order, where each entry is `{ "loaded": true, "uri": "<the uri>" }`. For `connector_mutation`, the `request` object needs `body` as the caller-supplied body, `headers` as `{}` if none are supplied, `json: true`, `method` as POST or PUT, `resolveWithFullResponse: true`, and `uri` as baseUri + endpoint concatenated. A successful simulated write returns `result` as `{ "accepted": true }`.

For GET caching, on a fresh successful request with no cached value, Redis does exactly two ops in order: first a `get`, then a `setex`, and that `setex` uses TTL 300 with the response body as the value. The `redisOps` array needs to reflect that exact sequence. If the Redis lookup itself fails, the output is exactly `{ "error": "cache_error", "value": <the error value string> }`, where `value` is the raw error string from the Redis stub, like `"redis unavailable"`. In that case we stop there — no upstream request and no domain error wrapper.

On error formatting, if we wrap a domain error like from `wrap_query_error`, it should contain `code` like `"GRAPHQL_ERROR"`, `component` as null if not set, `description` as the original error message, `docsLink` as null if not set, `error: "domain_error"`, `hasGuid: true`, `message: "An internal server error occurred"` (fixed public message), `statusCode: 500`, and `targetEndpoint` as null if not set. Stack traces and exception class names should never show up. Then for `format_client_error`, development mode includes all of `statusCode`, `error`, `message`, `description`, `errorCode`, `graphqlModel`, `targetEndpoint`, `docsLink`, and `guid`. In production mode, `targetEndpoint` and `docsLink` are omitted from the client-facing payload. Also, `description` falls back to the `message` value if it wasn’t explicitly provided.

Last thing, for the abstract base pieces, trying to directly instantiate a connector, model base, or similar abstract component should throw a normalized error, not a raw language-level exception, and it should follow that same domain error shape without leaking host-language class names or any stack trace strings. Concrete model instances created with a valid connector should keep the connector reference and return it in their output.
