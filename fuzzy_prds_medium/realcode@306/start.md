## Product Requirement Document

We need a batching primitive for our in-memory key-value store client that lets developers group many commands together and dispatch them in a single round-trip instead of paying network latency once per command. Commands issued inside the batch are collected and sent together; the batch itself returns the ordered list of replies, and each individual command issued inside the batch also gives back a deferred handle whose value becomes available once the batch executes. Batches can be nested inside each other and can contain database-switch commands that take effect immediately for the rest of that batch and persist afterward. When a store command fails, the batch should surface the failure as a normalized error carrying the store's own message — not a raw Ruby exception. Handles for commands that were queued after a failed command should be clearly unresolved rather than silently returning stale values. Reply types should be preserved: a field/value reply comes back as a map, a multi-element reply as a list — handle that the same way non-batched typed replies are handled, just inside the pipeline context.

One extra bit that came up while we were talking through expected output: for reply rendering inside the batch, we want the same typed behavior to stay visible in the emitted values. A nil reply should come through as 'nil'; integer as 'int:<n>'; boolean true as 'bool:true'; boolean false as 'bool:false'; string as 'str:<text>'; and float as 'float:<n>'. If the reply is a map, render it as 'map:<size>\n' and then '<key>=<type-tagged-value>\n' for each entry, sorted by key. If it’s a list, render it as 'list:<size>\n' and then '<index>=<type-tagged-value>\n' for each element. That’s just to make sure batched typed replies line up with what people already expect, only now in the pipeline flow.

Also, on deferred handles, if somebody tries to read one before the batch has actually executed, that raises Redis::FutureNotReady. The adapter should catch that and emit 'error:value_not_ready' exactly like that, using the colon separator, not equals. And if a batch has already failed and there are handles that never got resolved because they were queued after the failing command, those should come back as 'h<index>:value_not_ready' so it’s obvious they never became valid.

For batch failure formatting, when the cause is a rejected command (Redis::CommandError), the output should be 'error:command_error\ndetail:<store error message>\n'. The detail line carries the store's own error text verbatim. The detail line carries the store's own error text verbatim.
