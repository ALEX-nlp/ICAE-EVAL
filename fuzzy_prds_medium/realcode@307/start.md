## Product Requirement Document

Hey team, we need to build out that argument parsing utility we've been talking about. Basically devs are tired of writing the same boilerplate token-walking code every time they spin up a new CLI tool — it's brittle, inconsistent, and nobody wants to maintain it. The idea is you declare your args once and the library handles the rest: figuring out what's optional vs positional, dealing with those weird concatenated short flag cases like we handled in the auth CLI last quarter, converting values to the right types, yelling at users when they do something wrong, and spitting out decent help text automatically.

The JSON-in/JSON-out adapter layer needs to stay totally separate from the core logic — same principle we used on the config service, keep the I/O plumbing away from the business rules. Output should be sorted bindings, one per line, and errors should be plain English with no runtime type names leaking through.

Also needs to handle subcommands (think git-style), mutual exclusion groups, and the double-dash end-of-options separator. The help/usage rendering has to be auto-generated but also overridable. There's some nuance around how greedy positionals yield tokens to later required ones — make sure that's solid. Reach out if anything's unclear, there are test cases in the repo that cover the edge cases pretty thoroughly.

A couple follow-ups from the questions that came in: if an argument has default='::suppress::', then that binding should be omitted from the parse result completely. The name should not show up at all in the output, not even as null. That only applies when we're talking about the default suppression sentinel though — if the user actually provides the argument on the command line, we bind it normally. Same deal for help text: help='::suppress::' means hide that argument from the help output entirely.

On the positional side, the nargs='*' case should be careful, not just vacuum up everything that's left. It needs to look ahead and leave enough operands for any later fixed-count or required positionals, then only take the surplus for the star positional. If there’s nothing available for that positional, it binds an empty list, or its configured default if one is set and no tokens were supplied at all.

For printing results, keep it exactly one line per binding, sorted alphabetically by name, in the format <name>=<value>. A null value renders as 'null'. A boolean renders as lowercase 'true' or 'false'. A list renders as '[a, b, c]' with elements separated by ', ' and nested lists nest as '[[a], [b]]'; an empty list renders as '[]'. An enumerated value renders as its display string. And for parse failures, the whole output is just a single line in the form 'error: <message>', with no host-language runtime type names sneaking into the message.

One more parsing nuance: tokens that look like negative numbers, like '-3' or '-1.5', should be treated as positional values, not option flags, unless there's an actual declared option whose flag matches that token exactly. If there is, parse it as that option; otherwise leave it available to positionals.

Also, on short options, we do want the fused form handled the same way we talked about before: if the parser sees a multi-character short token, expand it left to right. Each character that maps to a no-value action like store_true, store_false, count, append_const gets consumed as a flag, and the first character that needs a value takes the rest of that same token as its embedded value if there is one, otherwise it consumes the next token. So something like '-vvvv' with count action increments four times. Mutual exclusion checks still need to fire even if the conflicting options showed up inside one concatenated token. That short-option lexing behavior from Feature 4 needs to stay intact too: expanding fused short-option tokens left-to-right, handling embedded values, and applying count/append_const actions across repeated characters in a single token.

For enums, there are two separate converter behaviors. 'enum' matches against symbolic member names from {RED, GREEN, BLUE}. 'enumstr' matches against display strings from {SMALL, MEDIUM, LARGE}, with the special case that LARGE displays as 'X-LARGE', so the user has to pass 'X-LARGE' and the output should also render as 'X-LARGE'. If conversion fails for either one, the error should be in the form argument <flag>: could not convert '<value>' (choose from {members}).

And just reaffirming the separation piece since that came up again: the JSON stdin/stdout execution adapter should stay completely decoupled from the core parser domain logic. The adapter’s job is only translating JSON requests into idiomatic core API calls.
