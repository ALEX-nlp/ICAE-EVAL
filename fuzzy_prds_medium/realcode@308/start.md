## Product Requirement Document

Hey team, we need to get the coverage toolkit's core plumbing sorted out. Basically the whole thing around tracking which lines got hit during tests, being able to save and reload that data, and the config/CLI side of things. There are a bunch of small helpers we need too — like figuring out the right file names when we back up a binary before we mess with it, and also reading those runtime config docs to find where extra assemblies live (you know, the same kind of thing we did for the loader in the old probing work). Oh and the graph trimming piece — when we only care about a subset of nodes, we need to be able to cut the rest out without breaking the connections between the ones we keep. For the CLI options, things like output paths, glob patterns for source/test/assembly matching, log verbosity, and the coverage percentage cutoff all need sensible fallbacks when nothing is provided, and bad input should come back as a clean error category rather than just blowing up. The serialization round-trip for hit records needs to be lossless — line IDs and counts both — and when we load a coverage manifest that doesn't exist it should fail gracefully with enough info to know what path it was looking for. Make sure the code is split up properly across files, not all dumped in one place. Check the existing test scaffolding for the exact edge cases around cycles and duplicate contexts.

One extra pass on the details the team asked about. On the graph trimming behavior, when a node is not in the allowed set it is removed and each of its children is re-attached directly to the removed node's parent position, preserving reachability through the removed node. If the root itself is removed, its surviving descendants become the new top-level set and are rendered as a comma-separated list. Also for cycle handling, a node that has already appeared on the current path is rendered as *v (with an asterisk prefix) rather than expanding its children again. This prevents infinite recursion on cycles and self-references. Children at each level are sorted in ascending value order so the output is deterministic.

On the manifest side, when the resolved manifest file does not exist the operation fails with a normalized error: error=coverage_file_not_found on one line, path=<the resolved absolute path> on the next. The path is resolved against the working directory using the same rules as resolve_file_path for the coverage role (defaulting to coverage.json). Related to that, the working directory option defaults to './' when no value is supplied, which resolves to the current working directory itself, so nothing changes. When a relative value such as 'folder' is given it is resolved against the current working directory, that directory is created if needed, and it becomes the new working directory. The output reports both the resolved path and the resulting current working directory. Default directory paths per role are hits->coverage-hits (placed in workdir), html->coverage-html (placed in workdir), parent->the working directory itself. A relative value is resolved against the working directory to an absolute path.

Also want to be explicit about the structure of the code since that came up: the project must be split into multiple files separating at minimum core measurement primitives (graph filtering, hit aggregation, serialization, file utilities, probing path extraction), the configuration/option layer (path resolution, glob resolution, threshold parsing, verbosity parsing, manifest loading), and an execution adapter responsible only for translating JSON commands to core calls and formatting stdout output. A single god file is explicitly prohibited.

And for duplicate contexts, contexts that share the same (assembly, class, method) identity triple are merged into a single context whose per-line counts are the element-wise sums of all contributing contexts. The merged context occupies the position of the first occurrence in the input (stable first-seen order). The contexts count in the summary line reflects the number of distinct identity triples, not the number of raw input entries.

Last bit: Feature 5 (Runtime Probing Path Extraction) is specifically reading runtimeOptions.additionalProbingPaths from a .runtimeconfig.json document and filtering out entries containing |...| macro placeholders, as specified in feature5_probing_paths.json.
