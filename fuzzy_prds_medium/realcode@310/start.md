## Product Requirement Document

Hey team, we need to ship that expression reconstruction library we've been talking about. The idea is pretty straightforward — devs are sick of maintaining two copies of the same business logic, one for running and one for query stuff. We want a single place where you write the logic once and it just 'knows' how to represent itself symbolically too.

We need it to handle the usual suspects: basic value stuff, nullable/fallback cases, collection building, and then the more interesting case where computed properties on records get 'unfolded' into what they're actually doing under the hood so query providers don't choke on them. Also the method body reconstruction thing — similar to what we did with the output-slot pattern in the auth module, where it parses and falls back cleanly.

There's also a config piece — the engine should have a sensible startup story, like a default kicks in if nobody sets one up, but if you try to reconfigure it after the fact it should fail in a clean, framework-neutral way (no raw exception names leaking out). And we need it to stay fast even when predicates get really large — someone mentioned 27 items as the benchmark case.

The test harness needs to write per-case output files into a namespaced folder so different test runs don't stomp each other. Keep the adapter layer totally separate from the core logic please, last time things got messy.

One quick follow-up since a few details came up while folks were wiring the harness. Each case writes its raw stdout to `rcb_tests/stdout/<cases-dir>/{filename.stem}@{case_index.zfill(3)}.txt`, and that `<cases-dir>` piece is there specifically so runs against different case directories never overwrite each other. For example, the first case in `feature1_scalar_expression_reconstruction.json` run with the default `--cases-dir public_test_cases` goes to `rcb_tests/stdout/public_test_cases/feature1_scalar_expression_reconstruction@000.txt`. The index is zero-padded to 3 digits. Also, `rcb_tests/test.sh` reads every `*.json` file from its cases directory, with the default cases directory being `public_test_cases` relative to `rcb_tests/`, and you can override it with `--cases-dir <subdir>`. It only reads `*.json` files at the top level of the specified cases directory and does not recurse into subdirectories. Each JSON file may contain multiple cases under a `cases` array, and each case is indexed from 0 with that zero-padded 3 digit output filename convention.

A couple exact output expectations to lock down too. For the `identity` scalar case with value `"alpha"`, the expected output is exactly: `expression=o\ndebug_hash=62bc3bd4\nresult=alpha\n`. The expression symbol is the single letter `o`, the debug hash is `62bc3bd4`, and the result just echoes the input. For the `nullable fallback` expression with a null input value, the expected output is: `expression=(x ?? 100)\ndebug_hash=61b09b26\nresult=100\n`. The symbolic form uses `??` operator notation, the fallback literal is `100`, the debug hash is `61b09b26`, and the result is `100` when the input is null.

For `decompile_collection` with `expression: "array with parameter"` and `value: 8`, the expected output is: `expression=new [] {x, 2, 3, 4}\ndebug_hash=0651bbc0\nresult=[8,2,3,4]\n`. Collection results use bracketed comma-separated values with no spaces between items, and the parameter `x` substitutes to `8` in the result. For `inline_query` with `expression: "computed full text property"` and match `"Ada Lovelace"`, the expected output is: `query_expression=records.Where(p => (((p.First + " ") + p.Last) == target))\nmatched=Ada Lovelace\n`. The computed property is expanded to field-level string concatenation with an explicit space literal, and matched records are shown as their full-text value one per line.

On the method body reconstruction side, the `parse with output slot` case in Feature 5 (`feature5_method_body_reconstruction.json`) is the one where the method body calls `int.TryParse` with an output parameter (`var Param_0`), returns the parsed int on success, and returns `-1` as a fallback on failure. The symbolic form is `s => {var Param_0; ... }` with debug hash `8cbf20fc`.

And for Feature 6 config lifecycle, if `Configure` gets called after configuration already exists, the adapter should output `error=already_configured\n` and keep it at that fixed snake_case identifier. We do not want raw framework-specific exception naming, runtime-generated message text, or stack trace bits showing up in stdout.
