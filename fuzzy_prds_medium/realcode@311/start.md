## Product Requirement Document

Hey team, we need to get the lambda-api serverless adapter project wrapped up. Basically the idea is that right now our backend devs are writing the same boilerplate over and over every time they spin up a new Lambda function — manually unpacking gateway events, building response objects by hand, dealing with cookies, headers, all that stuff. It's a mess and we keep seeing bugs pop up around edge cases like special characters in cookies or headers not being lowercased consistently.

We need a proper routing layer that works the way our express-style stuff does internally — you know, similar to the pattern we used on that API gateway normalization project a while back. It should handle path params, wildcards, middleware chaining, and be able to short-circuit when something goes wrong. Error handling is a big one — right now exceptions bubble up in weird ways and clients see stack traces sometimes which is obviously not great.

Also the cloud event normalization piece is important — we need it to work cleanly with both the gateway v1/v2 formats and the load balancer format since we use all three depending on the service. Cookie encoding has been a pain point specifically; there were complaints about URLs and special chars not surviving the round-trip correctly.

Can someone make sure the response envelope format is locked down and consistent? The downstream consumers are parsing that output programmatically so any variance breaks things. Let's keep it tight.

One extra thing the team asked about: the response envelope needs to come out on stdout in one very specific shape, as exactly four newline-terminated lines in this order: `status=<code>`, `base64=<true|false>`, `headers=<JSON object of lowercase-keyed string arrays>`, `body=<raw serialized body>`. The last line also needs to end with a newline. We really can’t drift on order or formatting here because the downstream readers are expecting that exact layout.

On headers, let’s normalize every response header key to lowercase, and every header value should be an array of strings even if there’s only one value. So the shape should look like `{"content-type":["application/json"]}` and `set-cookie` works the same way, with each cookie as its own array item. For cookies specifically, values need to be percent-encoded using encodeURIComponent semantics, so characters like `:`, `/`, `[`, `]`, `;`, and spaces need to stay encoded correctly, like `http://` becoming `http%3A%2F%2F`. If the cookie value starts as an object, JSON-serialize it first and then percent-encode it. The default format is `name=encodedValue; Path=/`, and if we have extras like Domain, Expires, HttpOnly, and SameSite, append them in that order when they’re present.

Also confirming the route param behavior people asked about: params are defined with the usual colon style like `/test/:id`, and the matched value should be available on the request object. In some cases the same matched value may show up under multiple keys depending on how the handler suite is wiring request access, so a path like `/test/:id` could end up exposing the same thing as both `param` and `param2` in the response body.

For middleware, if something sends a response by calling the response object’s send/json/error method, that should short-circuit the rest of the chain completely and nothing later should be able to modify the response. Path-scoped middleware should only run when the request path matches that scope, whether that’s exact, wildcard, or parameterized, and global middleware should always run before route matching on every request. The API surface here is the normal `app.get()`, `app.post()`, `app.use()` style with path scoping and the usual `(req, res, next)` signature consistent with Express.js conventions, same as what’s being exercised in features 1 and 5.

And on the event normalization side, yes, this is the same cloud event handling called out in Feature 7. We need to cover API Gateway v1, API Gateway v2, and ALB event shapes exactly as defined in `rcb_tests/public_test_cases/feature7_cloud_events.json` and line that up with the corresponding adapter normalization module in the lambda-api source.
