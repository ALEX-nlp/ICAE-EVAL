## Product Requirement Document

Hey team, we need to build out that RPC server wrapper thing we discussed last sprint. The basic idea is: clients send requests in that standard JSON-based RPC format (you know the one, 2.0 spec), and our server needs to route them to the right backend operation and send back a properly shaped response. We also need it to work over plain byte streams AND HTTP — the HTTP side needs to return the right status codes depending on what happened (the usual suspects: success, not found, bad input, server error, etc.).

One thing that tripped us up with the old login module compatibility logic — make sure whatever dispatch mechanism we use handles methods that have the same name but different argument signatures. Clients might also send a whole bundle of requests at once in one shot, and we need to handle that gracefully.

Also critical: when backend services blow up, we absolutely cannot leak any internal Java class names or stack traces back to the client. Map errors to clean numeric codes only.

There's also a testing shim/adapter we need — something that reads a single JSON blob from stdin describing the scenario, runs the engine, and prints out a normalized line-by-line view of the result to stdout so our CI pipeline can diff it. The adapter needs to support a flag that lets callers say 'I know I'm sending extra/fewer args than expected, that's fine.' Refer to the existing parameter flexibility logic we discussed for details on how permissive mode should work.

A couple extra specifics from the questions that came in: if the incoming JSON is malformed or just empty, that should come back as error code -32700 with category 'parse_error' and message 'JSON parse error', and in that case the id needs to be rendered as the string "null", so id="null". On the HTTP side, the status mapping should be straightforward: ok/success → 200, method_not_found → 404, parse_error/invalid_request → 400, and method_params_invalid/internal_error/error_not_handled/bulk_error/custom server codes → 500. Also, if it’s a batch and even one sub-request fails, the overall HTTP response should still be 500.

For the normalized adapter output, for HTTP transports (http_post, http_get), start with 'http_status=<int>' and 'content_type=application/json-rpc' before any body fields. If there isn’t actually a body, like a notification over HTTP, render that as the single line 'body=empty'. Also on the error formatting side, error_data only gets printed if it exists. If the value is just a plain string, print it as 'error_data=<text>'. If it’s a structured/object value that carries a message field, print it as 'error_data_message=<text>' instead. If no data was set, neither one should appear.

Also wanted to be explicit about the arg-count flexibility knobs since that came up a few times. The two boolean flags are 'allowExtraParams' and 'allowLessParams', both default false. 'allowExtraParams' permits and silently ignores surplus positional arguments. 'allowLessParams' permits missing trailing arguments, which are bound to their type default/null. If those flags are not enabled, any argument count mismatch should return error code -32602 (method_params_invalid) with message 'method parameters invalid'.

And on the same-name-methods piece, the selection rule is by both parameter count and parameter types, not just the name. For positional params, types are inferred from JSON value types (string vs integer). For named params (object), matching is by parameter name set and types. The result string should reflect which variant was actually picked, like 'overloaded:string:x' vs 'overloaded:int:7'. Same deal in the dispatch engine generally: when multiple operations share a method name, choose the one that matches parameter count and JSON value types (string vs integer), following the service's overloadedMethod variants and the engine's argument binding policy.
