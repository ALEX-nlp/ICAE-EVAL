## Product Requirement Document

Hey team, we need to get the workflow manager CLI tool properly handling a bunch of the core interactions that keep coming up as pain points. Users are complaining that when they pick an agent from the list it sometimes does weird things with their input — spaces, out of range numbers, that kind of stuff. Also the color theming is broken for some people who paste in hex codes from their design tools. The first-launch experience is inconsistent depending on whether someone is a fresh install vs migrated vs returning user — we keep getting support tickets about this. Config merging between global and project level isn't respecting the right priority order either, which is causing teams to wonder why their project-level settings get ignored sometimes. The board navigation feels off when columns have different numbers of tasks — refer to how we handled the clamping logic in the selection module from the earlier sprint. Workspace paths need to follow the hidden directory convention we settled on (check the existing path resolution stuff). The git worktree create/remove cycle needs to actually work against a real repo, not just simulate it. Skill mapping per agent family needs to follow the command convention each CLI expects — claude and opencode behave differently here. Also the terminal capture trimming is leaving too much blank space. Please make sure all the edge cases around blank input, invalid text, missing optional fields, and empty columns are covered properly.

Quick follow-up on a few specifics the team asked about. On agent picking, if selection_text is blank or whitespace-only, we should treat that as choosing the first agent, so the output is selection=valid, selected_index=0, display_number=1. That’s the default/blank-input convention. If the entered number is less than 1 or greater than agent_count, or the text is non-numeric, then it should come back as selection=invalid and we should not emit selected_index or display_number at all.

For the workspace location, the path should always resolve to <project_path>/.agtx/worktrees/<task_slug>. The .agtx directory is hidden, dot-prefixed, and sits directly under the project root. Same deal anywhere this comes up: the .agtx/worktrees/<task_slug> convention is the one to follow, with .agtx as the hidden dot-directory under the project root. Also on the git side, this needs to run against a real git repository, not a mock. After creation, exists:<slug>=true. After removal, exists_after_remove:<slug>=false. The created path should be reported as the relative path .agtx/worktrees/<slug>.

On terminal capture trimming, the intent is to remove trailing blank lines beyond one blank line following the last non-empty line. More specifically, keep content lines plus up to 2 trailing empty lines as the blank buffer tolerance. So in the case with 2 content lines and 4 trailing blanks, the result keeps 4 lines total, meaning 2 content + 2 blank.

For command handling, claude and opencode need to stay distinct. For claude, use native_dir=.claude/commands|agtx, filename=plan.md, invocation=/<namespace>:<command> format like /agtx:plan, and reference is empty string. The pipe in native_dir is just separating the base commands dir from the subdirectory namespace. For opencode, canonical plugin commands in slash-colon form like /gsd:plan-phase need to be transformed by replacing the colon ':' with a dash '-', so that becomes /gsd-plan-phase. Anything after the command name should stay exactly as-is.

On skill discovery, we’re reading markdown files from the agent’s native command directory, parsing YAML frontmatter for 'name' and 'description' fields, and then outputting each discovered item as skill=/<namespace>:<command> | <description>. skill_count should report the total number of discovered skills.

One other setup detail: workspace initialization should copy every file listed in copy_files into the worktree directory, then run init_script as a shell command inside that worktree. warning_count=0 means nothing went wrong there. And if we list paths in check_paths, those need to be verified for presence and have their content reported.

Last thing, for the board behavior, when moving horizontally into a shorter column, selected_row should clamp the same way we discussed before: max(0, len(column)-1). That should cover the weirdness people are feeling when columns have uneven task counts.
