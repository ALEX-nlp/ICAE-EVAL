## Product Requirement Document

hey team, we need to build out that server-side HTML manipulation thing we discussed last sprint. basically it should work like the browser DOM stuff we already use on the frontend but headless — parse markup, let you poke around and change things, then spit it back out as a string. the input/output should be over stdin/stdout as JSON so it's easy to wire up from any language.

the big use cases are scraping pipelines and our post-processing templates. users have been complaining that the current regex-based approach keeps breaking on nested tables and auto-closed tags — the swiss chocolate page was the one that broke last week btw, something about data attributes with camelCase keys. also someone from the forms team flagged that serialization needs to handle disabled fields and multi-selects correctly, which the old thing never did.

for error cases, we want clean predictable output — no stack traces leaking into the response, just a simple category label. there's also the XML vs HTML thing, refer to how we handled strict mode in the config parser module from the auth team, similar idea. oh and fragment loading should skip the full document wrapper scaffolding. if there's time, tree traversal (parents, siblings, find) would be great too. let me know if anything is unclear, happy to jump on a call.

one more pass on the details the team asked about. for failures, let’s keep it super boring and consistent: every error should come back as a single line in the format `error=<category>` and nothing else, so no stack trace and no runtime-specific text mixed in. the only categories we want to expose are `invalid_input` and `invalid_selector`. `invalid_input` covers stuff like the document field not being a string, and `invalid_selector` is for when the CSS selector can’t be parsed. if something else blows up and we can’t classify it any better, it should still follow that same `error=<category>` shape.

also on the HTML/XML split, when the `options` field contains `{ "xml": true }`, we want strict XML parsing and serialization. that means tag name case is preserved, empty elements are self-closed like `<foo/>` instead of `<foo></foo>`, and attribute case is preserved too. without that option, it should stay in normal HTML mode, which lowercases tag names and repairs malformed structure like implicit `<tbody>` insertion. same deal on fragment loading: when `fragment` is set to `true`, parse the markup as a document fragment and do not inject the usual `<html>`, `<head>`, `<body>` wrapper bits. if `fragment: true` is not set, then it’s a full document parse and the parser can add that implicit scaffolding.

for the `data` operation, it should read a `data-*` attribute with camelCase-to-kebab-case conversion, so `bestCollection` maps to `data-best-collection`. the output there is always two lines, `type=<type>` and `value=<value>`. if the attribute value is a valid integer in the strict sense, like `42`, then `type` should be `number` and the value should be unquoted. if it looks like scientific notation, like `1E10`, do not coerce it; that stays a string and string output should look like `value="<string>"`.

and for forms, both `serialize` and `serialize_array` need to skip disabled controls, anything without a `name` attribute, and `input[type=submit]`. `serialize` should return the normal URL-encoded query string form, including spaces as `+`. `serialize_array` should return newline-separated `name=value` pairs, one per control, and multi-select should expand to multiple lines, one entry per selected option. last thing, the XML strict mode parsing/serialization behavior described in feature9_2_xml_mode.json and controlled by `options: { xml: true }` in the command object should mirror the pattern used in the `options` field of the load step.
