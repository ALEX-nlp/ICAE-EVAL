## Product Requirement Document

hey team, we need to wrap up the binary protocol toolkit for the collaboration server project. most of the core stuff is done but we still need to finish wiring up a few pieces that the client team is blocked on. specifically we need the tracker registration payload builder (similar to how we did the news category encoding last sprint, just follow that pattern), the chat delivery routing logic, the folder creation handler, and the file download transfer writer. the chat thing is particularly tricky because there are two modes — normal and the other one we discussed — and we need to make sure permission checks gate the whole thing properly. for folder creation same deal, permissions matter and path safety needs to work the same way as that safe path resolution feature. the file download side needs to actually write the full FILP formatted payload including the name section and data fork, i think it's 165 bytes total in the happy path. the tracker payload has a specific byte layout that the tracker server expects, and the name and description need to be length-prefixed. if anyone needs the exact wire formats just check the existing test fixtures, they've got the byte sequences. please make sure the output format for all these matches what the test runner expects, same key=value line style as everything else.

one quick follow-up since a few details came up in chat. for the tracker registration piece, the byte order is fixed and should be exactly: 2-byte magic value 0x0001, then the 2-byte port, 2-byte user count, 4-byte pass_id, then a 1-byte length-prefixed name string, then a 1-byte length-prefixed description string. same length-prefix pattern we used for string fields in feature9_news_category_encoding.json, and specifically the 1-byte length prefix used for name and description in tracker registration, as seen in feature10_tracker_registration_payload.json. for the example with port=0x0010, user_count=2, pass_id=0x00000001, name='Test Serv', description='Fooz', the expected hex is 0001001000020000000000010954657374205365727604466f6f7a.

also, on chat delivery, there really are just the two modes: the default mode and 'emote' mode. the mode field in the input will be 'emote' for emote mode and absent or any other value for normal mode. in default mode the message is formatted as: 0x0D followed by 10 spaces, then the 2-byte user_name bytes, then ': ' (colon space space), then the text bytes. in emote mode the format is: 0x0D followed by '*** ' then the user_name bytes as ASCII, then a space, then the text. this is the same 'emote' chat mode in feature11_chat_delivery.json, where message format changes to '0x0D *** <username> <text>' instead of the default '0x0D <spaces><username>: <text>' format.

for routing, each entry in client_read_access corresponds to a client numbered starting from 1 (recipient 0001, 0002, etc.). a transaction is emitted for each client whose read access is true. the recipient field is formatted as a 4-hex-digit zero-padded client number. all emitted transactions have reply=0, type=106, error_code=00000000, field_count=1, and field id=101 containing the formatted chat message bytes.

and just to close the loop on the permission failures, when has_permission is false for chat, return exactly one transaction with no recipient, reply=1, type=0, error_code=00000001, and a single field with id=100 containing the UTF-8 bytes of 'You are not allowed to participate in chat.' The hex for that string is 596f7520617265206e6f7420616c6c6f77656420746f20706172746963697061746520696e20636861742e. for folder creation, when has_permission is false, return one transaction with recipient=0001, reply=1, type=0, error_code=00000001, and a single field id=100 containing the UTF-8 bytes of 'You are not allowed to create folders.' The hex is 596f7520617265206e6f7420616c6c6f77656420746f2063726561746520666f6c646572732e.
