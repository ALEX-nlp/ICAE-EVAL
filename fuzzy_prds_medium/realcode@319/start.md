## Product Requirement Document

We need a small state-machine engine that can be attached to any plain object. The developer declares a set of named states, designates a starting state, and declares transitions as trigger-name + source-state + destination-state rules. Calling a trigger by name moves the object between states according to those rules. Each trigger should report whether it actually moved and what state is now active. A per-state predicate lets callers ask 'am I in state X right now?' Transitions can be guarded by boolean checks — some always pass, some always fail — and multiple transitions can share the same trigger name with the first eligible one winning. States can have enter and exit callbacks that run as side effects and mutate observable data on the object. Triggers can carry keyword arguments and the machine has a payload-delivery mode that controls whether callbacks receive those arguments directly or wrapped in an event object. The engine should also auto-generate a to_<state> convenience trigger for every state by default, with an option to disable that. Handle it the way similar ordered-transition helpers are handled internally — a bulk generator that wires up a linear chain of transitions under one trigger name.

One extra bit of color from the team’s questions: when a trigger actually goes through, it should emit exactly 'trigger=<name> result=true state=<new_state>'. If the trigger exists but doesn’t get to move because the guards didn’t pass, it should emit exactly 'trigger=<name> result=false state=<unchanged_state>'. An invalid transition emits: 'error=invalid_transition trigger=<name> state=<current_state>'. A missing trigger emits: 'error=no_trigger trigger=<name>'.

Also on the callback side, the examples are meant to be literal. greet() records the text 'Hello World!' in model.message. farewell() records 'So long, suckers!' in model.message. store_kwarg_message(message=...) records the 'message' kwarg value directly. store_event_message(event_data) reads message from event_data.kwargs['message'] when send_event=True.

And one more clarification around the generated helpers: when auto_transitions=False, the to_<state> triggers simply do not exist. So if somebody tries to fire one anyway, the result should be 'error=no_trigger trigger=to_C' (or whichever state name was used). No move happens and no exception leaks to stdout.
