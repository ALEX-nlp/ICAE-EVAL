## Product Requirement Document

We need a client library that wraps a remote infrastructure management service's REST interface. The library should allow operators to inspect and manage the host daemon — retrieving a summary of the daemon's environment (including its trust status, underlying storage backend, runtime versions, and kernel information) as well as reading individual fields from that environment. When the daemon is unavailable, field lookups should return a null value rather than raising an error. Network management should follow the same general pattern: list all networks by their short names, show details for a single network, or read individual fields. Async operations tracked by the daemon should be listable, inspectable, cancellable, and waitable. Containers form the core resource: operators should be able to list them, check whether they exist or are actively running, start/stop them with a force flag, create/destroy them, read state and logs, migrate them to other hosts, publish them as catalog entries, manage their snapshots, push or pull files, and execute commands inside them. The catalog of images should support listing, searching, existence checks, detailed summaries with human-readable sizes and dates, per-field access, export, upload, and deletion. The system must also manage named aliases for catalog entries, trusted client certificates, and configuration profiles — all following the same list/show/create/update/delete pattern as the network module. Errors from the backend should be surfaced as typed exceptions, and boolean helpers should indicate success or existence cleanly.

A couple extra specifics from the questions that came back. For the host summary, the returned dict is just these keys, in alphabetical order: kernel_version, lxc_version, lxd_api_compat_level, lxd_backing_fs, lxd_driver, lxd_trusted_host, lxd_version. The types matter here too: kernel_version (string), lxc_version (string), lxd_api_compat_level (int), lxd_backing_fs (string), lxd_driver (string), lxd_trusted_host (bool), lxd_version (float). lxd_trusted_host is only true when the auth field equals 'trusted'. Also lxd_version should come back as a floating-point number, so 0.12 and not '0.12'. And lxd_api_compat_level comes from the metadata-level api_compat field.

Also worth being explicit about the host field accessor behavior: if the daemon returns a daemon_error while looking up one specific host field, that accessor should swallow it and return null / Python None. It does NOT raise there. That exception handling is only for the single-field accessor though; the full host.info call still raises on errors.

For operations, operations.list should return the full operation URLs exactly as the daemon gives them, like '/1.0/operations/1234', rather than trimming down to just the last segment. That is intentionally different from containers.list and images.list, which do return short names.

On file push for containers, containers.file.put sends uid, gid, and mode as HTTP headers named X-LXD-uid, X-LXD-gid, X-LXD-mode. The destination path is passed as a query param, so the shape is POST /1.0/containers/<name>/files?path=<dst>. The file content itself is just the raw request body.

One lower-level thing the team asked about: conn.object returns a [status_code, document] pair for HTTP 200 and 202. For anything else, including something like 500, it raises or reports an api_error with status_code=null, not the real HTTP code. If the body is empty, that should be treated as a daemon_error for transport failure.

For image existence checks, images.defined should return false, not raise, specifically when the API comes back with an api_error where status_code 404. If it succeeds, return true. If it is any other kind of error, or a non-404 status, then it should still raise.

For command execution inside containers, the request is POST /1.0/containers/<name>/exec with a JSON body containing exactly {'command': [<args>], 'environment': {<env dict>}, 'wait-for-websocket': <bool>, 'interactive': <bool>}. The important bit people kept asking about is that the key is 'wait-for-websocket' with the hyphens, not 'web_sockets'.

For the running check, containers.running should look at GET /1.0/containers/<name>/state and return true when the container status, case-insensitive, is one of running, starting, freezing, frozen, thawed. Anything else should be false, including 'stopped' and 'STOPPED'.

And on the other resource classes, the pattern is meant to stay consistent. pylxd/network.py uses an LXDNetwork class with the same list/show/field style and the same error handling conventions used elsewhere, where api_error raises APIError, daemon_error raises PyLXDException, and KeyError becomes missing_field. In the same spirit, pylxd/certificate.py, pylxd/profiles.py, and pylxd/image.py for aliases should expose LXDCertificate, LXDProfile, and LXDAliases classes with list(), show(), create/update/delete methods mirroring the pylxd/network.py LXDNetwork structure.
