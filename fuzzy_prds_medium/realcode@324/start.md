## Product Requirement Document

hey team, we need to build out that logging engine thing we talked about last sprint. basically the idea is devs should be able to fire off a log entry with some context attached and have it show up in one or more places without them having to do a bunch of wiring themselves. there's a bunch of stuff we need to handle — like, sometimes the message bytes are garbage and we don't want the whole thing to blow up, so we need to quietly fix that before it hits storage. also we talked about the thread thing, like how last time on the jobs service one noisy thread was drowning everything out — we need to be able to turn down the noise on just that thread without touching the rest. we also want the same entry to go to multiple destinations at once, kind of like how we set up the audit trail fan-out in the payments module. oh and there's a mode where you want the entry to go downstream but NOT get written to the main store — that needs to still fan out properly. entries should be able to carry extra context (stack info, environment stuff) without us manually passing it everywhere. the adapter layer should read commands from stdin as JSON and write results to stdout, same pattern as the notification dispatcher we built. deferred message producing should also be supported. check the existing test cases for the exact output format, i don't have time to spell it all out here.

one more pass on this since a few details came up. for the encoding cleanup part, we’re treating everything as UTF-8, and if a message has byte runs that aren’t valid in that encoding, each bad run gets replaced with U+FFFD before we record it. the important bit is the repaired version is what gets stored, and `valid_encoding=true` should reflect the stored message being valid after cleanup, not whether the original bytes were clean. we’re not surfacing the original invalidity anywhere beyond that flag behavior. the exact byte-level example is bytes [97, 32, 228, 32, 116, 101, 115, 116], where 0xE4 (228) is not a valid standalone UTF-8 byte, and that should come out as `message=a � test` with `valid_encoding=true`.

on the thread noise control, each thread can have its own override threshold. so if we set `thread_threshold: 2`, only entries from that thread whose numeric level is >= that threshold get recorded. other threads with no override still use the default behavior, which accepts all levels, basically threshold of 0 / no threshold. if we set `thread_threshold: null`, that clears the override and puts that thread back to accepting everything. in the adapter JSON protocol, the `thread_threshold` action applies to the 'main' thread, and log actions can include an optional `thread` field for which thread emitted the entry.

also wanted to be really explicit about the skip-store behavior in the fan-out case. when `skip_store` is active, the primary sink must NOT record the entry, so its count does not move, but any attached downstream loggers still need to receive that entry. and while forwarding it, the `skip_store` flag has to go along too, so the downstream sees `skip_store=true` on that forwarded call. the exact output there is `chain_probe skip_store=true message=testing` followed by `primary total=0`. that `primary total=0` line still shows up in stdout even though nothing got written to the primary sink.

for rendering, each entry the primary sink records is one line exactly like `primary level=<L> source=<tag> message=<text> backtrace=<bt> env=<json> valid_encoding=<flag>`. if a field is absent or empty, it still renders with nothing after the `=`, like `source=`. `env` is a compact JSON object string when present, otherwise empty. after all entries, there’s a trailing line `primary total=<n>` with the recorded count. for a chained recording sink the prefix is `chained` instead of `primary`. for the standard chain it is `chain_standard severity=<letter> message=<text>`. For the probe chain it is `chain_probe skip_store=<bool> message=<text>`. each sink/downstream type keeps its own count.

and just to close the loop on the fan-out setup itself, the downstream loggers get attached via a `chain` action, and then every accepted entry gets broadcast to all of them before reaching the primary sink.
