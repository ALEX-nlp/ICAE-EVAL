## Product Requirement Document

Hey team, we need to build out that test linting library we've been talking about. Basically the idea is that devs paste in their test source code and a config of which checks they want enabled, and the tool spits out a structured report of problems. Think of it like the same pattern we used for the import validator we shipped last quarter — same kind of adapter wrapping the core logic.

The output needs to be consistent and machine-readable so CI pipelines can consume it. Each problem should say which rule fired, a human-readable message, where in the file it is (line and column), and how bad it is. Some checks also need to support auto-fixing, where the tool emits the corrected source alongside the diagnostics.

We also need to handle that weird snapshot preprocessing flow — someone on the test infra team said there's a two-phase thing where you preprocess a snapshot file and then postprocess a message array to filter it down. That part is a bit fuzzy to me honestly, just make sure it works like the existing snapshot tooling expects.

The checks we need cover things like accidentally committed skip/focus flags, tests with no real assertions, broken async assertion patterns, bad test titles, messy hook setup, wrong import patterns, and a few more. The rules should be individually toggleable. Severity should map to numbers the way our other linting tools do it. Please keep the core logic totally separate from how input/output is handled.

A couple more specifics from the questions that came back. On the output shape, we need stdout to be newline-delimited key=value pairs in this exact order: first an optional `output=<fixed source>` line only when a fix was applied, then `messages=<count>`, then for each diagnostic in zero-based index order: `messageN.rule`, `messageN.text`, `messageN.line`, `messageN.column`, `messageN.severity`. No extra blank lines, no JSON, no other formatting. For severity, emit integers only: 'error' maps to severity=2 and 'warn' maps to severity=1, so the line needs to be exactly `messageN.severity=2` or `messageN.severity=1`, not the words themselves.

Also, all rule identifiers in the output need the `jest/` prefix. So even if the incoming rules config uses the short name like `"no-disabled-tests": "error"`, the emitted diagnostic should look like `message0.rule=jest/no-disabled-tests`. Same idea for the rest of the rules.

A few rule-specific details too. For `no-duplicate-hooks`, flag the second occurrence, not the first, and report the column where that duplicate hook call starts. In `describe("suite", () => { beforeEach(() => {}); beforeEach(() => {}) })`, the second `beforeEach` starts at column=49, so that is what gets reported. For `lowercase-name`, the column should point to the opening quote of the title string argument, not the start of the `describe` or `it` call. In `describe("FOO", () => {})`, the string `"FOO"` starts at column=10, so the diagnostic reports column=10. And for `no-mocks-import`, the message needs to be exactly `Mocks should not be manually imported from a __mocks__ directory. Instead use jest.mock and import from the original module path.` and it should fire whenever an import path contains `__mocks__`.

On the architecture side, yes, this should follow the adapter pattern I mentioned before: a thin I/O adapter (stdin JSON → core API calls → stdout key=value) wrapping a pure domain engine, as seen in feature9_import_export_restrictions.json and the overall adapter design in start.md. The execution adapter is responsible for handling stdin/stdout and JSON parsing, and it needs to stay strictly separated from the core linting engine. The core domain module should expose an idiomatic programmatic API, like a `lint()` function or `Linter` class, and it should not import or reference any I/O modules. The adapter just translates JSON commands into calls to that core API and formats the results into the key=value output format.
