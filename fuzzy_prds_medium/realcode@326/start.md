## Product Requirement Document

Hey team, we need to build out that schema migration safety toolkit we've been talking about. Basically devs keep shooting themselves in the foot when they push db changes to prod — things like adding columns that cause full table rewrites, or foreign keys that grab locks at the wrong time. We had a similar pattern in the background job resolution module we built a while back, so maybe look at how that handles the 'is this a valid task' check for inspiration.

The tool needs to cover a few areas: checking whether a column addition is safe (this depends on the postgres version — there was a big change around version 11 that changed how defaults are stored), checking foreign key patterns, actually applying defaulted columns and reporting back what the db sees, managing check constraints (including the case where old bad rows exist), index operations (concurrent ones have that transaction restriction we always forget about), and background batch data repair jobs.

For the retry stuff, there's a meaningful difference between how transactional vs non-transactional migrations behave when they hit a lock — the retry count should reflect that correctly. We want 2 retries configured so total attempts for transactional should be 3.

Output should be plain key=value lines. Please look at how we handled the constraint-not-found and association-not-found error cases in similar modules — those need clean normalized error keys, not raw exception messages.

One extra pass on the version-specific column behavior since that came up: the threshold is PostgreSQL major version 11. On version 10 and below, adding a column with a constant default triggers a full table rewrite, so that’s unsafe. On version 11 and above, the database can apply the default without rewriting existing rows, so that’s safe. Also, a volatile expression default is never in the safe bucket here. A volatile expression default (e.g., NOW(), random()) always requires a per-row evaluation and therefore causes a table rewrite regardless of the database version. It is always reported as result=unsafe, error=unsafe_migration, reason=column_default_rewrite.

On the defaulted column reporting side, please keep the exact db-facing metadata straight. On version 11+ with nullable=true: column metadata is nullable=true and rejects_null=false. On version 10 with nullable=false: the column metadata remains nullable=true (the column itself is still nullable at the db level) but a check constraint is added to enforce non-null, so rejects_null=true. The column name is 'status', default is 0, and existing_record_value is 0 in both cases. Calling out those two cases exactly matters because people were mixing up column nullability with the extra enforcement we add around it.

For foreign keys, the intended split is: adding a validated foreign key to existing tables is unsafe with reason=foreign_key_validation_lock. Adding it without immediate validation, so deferred, is safe. Then later validation is only safe when it happens outside the kind of heavy-lock transaction we’ve been warning about; validating inside one is unsafe with reason=foreign_key_validation_in_transaction, while validating outside any such transaction is safe.

On the background task resolution bit, this should match the BackgroundTask resolver module described in Feature 7, and specifically the two-tier error taxonomy plus the snake_case-to-PascalCase label transformation used to produce the 'name' field in error output. Task labels use snake_case and are converted to PascalCase for the resolved class name, so does_not_exist -> DoesNotExist and plain_object -> PlainObject. background_task_not_found means the label cannot be resolved to any constant at all, with resolved=false and name as that PascalCase form, for example DoesNotExist. not_background_task means the label resolves to a real constant/class but it does not implement the background-task contract, so resolved=false and name=PlainObject. A valid task returns resolved=true and base_type=background_task. For the specific good case, make_records_inactive resolves successfully and does not output a name field — only task, resolved=true, and base_type=background_task. This is the same distinction the team asked about in Feature 7 (resolve_background_task) in start.md and feature7_background_task_resolution.json.

Last thing on the batch repair flow: the orphan deletion batch must remove rows missing required parent links including rows that would normally be hidden by default query filters, like soft-delete scopes. The expected output is relation_ids=1,2 for the orphans removed, remaining_ids=3 for the parented row that stays, and kept_parented=true to confirm the valid parent case was preserved.
