## Product Requirement Document

Hey team, we need to build out that auth handoff flow we talked about in the last sprint — the one where a third-party app can get a short-lived code on behalf of a user without us ever handing them the user's password. Basically it's the redirect-based handshake thing. The tricky part (and where we got burned last time on the login module) is knowing WHEN to throw an error directly vs. when to bounce the user back to the app with the error in the URL — we need to get that split exactly right or clients will have no idea what went wrong. There are also some config toggles the ops team needs to be able to flip — things like whether we force that anti-forgery value, whether we're strict about the callback URL matching, and optionally enabling that identity assertion feature for our SSO partners. Scope handling needs to work too — if someone sets up the deployment without a fallback scope, the request should fail gracefully. We also need the issued codes to be one-time-use and redeemable at a separate token endpoint. The output format for the runner needs to be totally deterministic — no raw secrets in logs ever. Please separate the concerns properly, don't just dump everything into one file like we did on the billing service. Refer to how we structured the redirect reconstruction piece for the URL component handling.

One more pass with the details the team asked for, since the main thing here is still getting the error split right. If the client cannot be identified at all, meaning missing or unregistered client_id, or if we cannot establish any safe redirect target, we do not bounce anything back to an app. That direct-response bucket includes no URI stored and none supplied, fragment in supplied URI, URI mismatch against registered, and multiple registered but none selected. In those cases the endpoint returns a direct HTTP 400 body response with the error. For all the other failures, once we already have a validated redirect target, we do a 302 back to that target and put error and error_description in the query string. That redirect bucket covers bad/missing response_type, missing state when enforced, invalid/missing scope, and user denied access.

For the client lookup edge cases specifically, both of the no-client cases are direct responses and never redirects. The response is HTTP 400, type=response, body.error=invalid_client. The only difference is body.error_description: use 'No client id supplied' when the client_id parameter is absent entirely, and use 'The client id supplied is invalid' when a client_id is present but not found in the registered catalog.

Also, the fixed client catalog should be treated as exactly four entries: (1) 'Test Client ID' — secret 'TestSecret', no registered redirect URI; (2) 'Test Client ID with Redirect Uri' — secret 'TestSecret2', one registered target 'http://brentertainment.com'; (3) 'Test Client ID with Multiple Redirect Uris' — secret 'TestSecret3', two targets 'http://brentertainment.com' and 'http://morehazards.com'; (4) 'Test Client ID with Redirect Uri Parts' — secret 'TestSecret4', one target 'http://user:pass@brentertainment.com:2222/authorize/cb?auth_type=oauth&test=true'. Any other client_id is treated as unregistered.

On config defaults, if the config object is omitted entirely from input, assume enforce_state=true, require_exact_redirect_uri=true, allow_implicit=false, use_openid_connect=false, and issuer is unset. That means state is required by default, exact redirect matching is on by default, implicit flow is not allowed by default, and the identity-connect stuff is off unless someone explicitly turns it on.

For redirect matching, there are two modes controlled by require_exact_redirect_uri (default: true). In exact mode, the supplied redirect_uri must match the registered one character-for-character including query string. In relaxed mode with require_exact_redirect_uri: false, the supplied URI only needs to share the registered URI as a prefix, so extra query parameters in the supplied URI are accepted. If the supplied URI fails matching in exact mode, return a direct HTTP 400 error with body.error=redirect_uri_mismatch, body.error_description='The redirect URI provided is missing or does not match', and body.error_uri='http://tools.ietf.org/html/rfc6749#section-3.1.2'. This is the same redirect-URI validation behavior we called out for feature 2, especially features 2.3, 2.4, 2.5, including the missing-selection cases when there are multiple registered URIs and none was selected.

For the denied-user path, if is_authorized=false on an otherwise-valid authorization request — valid client, valid redirect, valid response_type, and state and scope checks all passed — then we still redirect 302 to the validated target, but with query.error=access_denied, query.error_description='The user denied access to your application', and query.state=<echoed state value if present>. No grant code is issued in that case.

On the rendering/output side, keep it totally deterministic. Each step produces one block in stdout. Blocks are separated by exactly one blank line, there is no blank line before the first block, exactly one blank line between consecutive blocks, and the entire output stream ends with a trailing newline. Each block starts with response=<step_index> using 0-based indexing, then status=<http_status>, then either type=redirect or type=response. For type=response, body lines are sorted by key. For type=redirect, always emit all six URL components even if they are empty: redirect.scheme=, redirect.user=, redirect.host=, redirect.port=, redirect.path=, redirect.fragment=. Pre-existing query parameters from the redirect_uri need to be preserved and merged with protocol-appended parameters like code, state, error, etc. In the final redirect output, all query parameters are written as query.<key>=<value> lines sorted alphabetically by key, and multi-valued parameters are comma-joined. That should follow the same redirect rendering behavior already shown in the Execution Contract and in feature6_1_redirect_reconstruction.json.
