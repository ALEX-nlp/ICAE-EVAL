## Product Requirement Document

hey team, we need to wrap up the state estimation toolkit for the navigation crew. basically they want a single entry point that can handle a bunch of different operations — running the two flavors of recursive filter (you know, the 'extended' and the 'sigma-point' one), checking some shared bookkeeping stuff like how measurements are ordered and what filter params look like after a round-trip, doing the coordinate projection thing we talked about (lat/lon ↔ grid), and querying that ring-buffer history thingy. the driver reads JSON from stdin, routes to the right handler, and prints results to stdout. the tricky part is the output formatting — there was some agreement about how numbers get printed (check how we handled the covariance diagonal in that other nav module) and the filters need identical external contracts so they're swappable. the first measurement in a filter session is special — it doesn't predict, it just seeds the state. also the geo conversion should round-trip to a fixed number of decimal places that the QA team signed off on somewhere. the buffer needs to handle capacity overflow by dropping the oldest entry, and queries outside the stored range should label themselves differently from interpolations. make sure errors are typed properly and the core logic stays totally decoupled from stdin/stdout plumbing.

one quick follow-up from the questions that came in: for printing, state-vector and scalar parameter values must be printed with exactly 6 digits after the decimal point, like 0.150000, and covariance diagonal values must be printed in scientific notation with exactly 6 digits after the decimal point, like 1.000000e-09. Negative zero must be normalized to positive zero. Values on the same line are space-separated and come right after their label, like 'state: ' or 'covariance_diagonal: '. same rule as called out in the 'Numeric output formatting' section of Domain Conventions in start.md.

also, on the filter lifecycle, a freshly constructed filter reports initialized=false, and after the first 'process' step has been handled it becomes initialized, so the filter_parameters output needs to report that as exactly 'initialized=false' or 'initialized=true'. that first 'process' step is the special one we mentioned before: it initializes the estimate directly, so each observed element where update mask = 1 is set to the measured value, and the corresponding block of the estimate-error covariance is set to the measurement covariance. No prediction step is performed for the first measurement. Every subsequent measurement first predicts forward over the elapsed time, then corrects.

for shared data shape stuff, the state vector is exactly 15 elements in this fixed order: x, y, z (positions in meters), roll, pitch, yaw (angles in radians), vx, vy, vz (linear velocities), vroll, vpitch, vyaw (angular velocities), ax, ay, az (linear accelerations). this order is shared by all features. and if a covariance matrix comes in, it may be supplied in one of three compact forms: {"identity_scale": s} meaning the 15×15 identity matrix scaled by s; {"diagonal": [...15 values...]} meaning a diagonal matrix with those 15 values; or {"matrix": [[...], ...]} meaning the full explicit 15×15 matrix.

last couple bits: the measurement ordering predicate compares two records by time and reports whether the first argument is strictly newer, meaning first.time > second.time, than the second. if the timestamps are equal, it evaluates to false in both directions, so it is a strict ordering, not ≥. and yes, the two filter paths still need to line up externally, including the Unscented Kalman Filter (UKF) described in Feature 2 (feature2_ukf_fusion.json), which propagates uncertainty through sigma points rather than linearization, parameterized by alpha/beta/kappa coefficients.
