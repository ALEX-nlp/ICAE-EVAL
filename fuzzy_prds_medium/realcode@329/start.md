## Product Requirement Document

Hey team, we need to build out that proxy config generation engine we talked about in the last planning session. Basically the idea is that each service instance sends us some metadata about itself (what it talks to, how it wants to receive traffic, health check stuff, etc.) and we need to spit out all the right config artifacts for it deterministically. Think of it like what we did for the login module's compatibility layer but for sidecar proxies.

Right now operators are hand-writing all these route tables and cluster references and it's a total mess — every small policy change means touching dozens of files and there's no consistent way to tell proxies their config changed. We want a clean engine that takes instance metadata + server policy and derives everything: the routing labels, dependency descriptors, which traffic group an instance belongs to, the ingress/egress route configs, and some kind of versioning so connected proxies know when to reload.

A few things I'm not 100% sure on: there's apparently some special handling needed when services request access to everything vs a specific list, and some tag combinations are allowed per-service but not globally. Also the versioning piece — I know we discussed stable labels so tests aren't brittle but I don't remember the exact scheme. The code should be organized sensibly across multiple files, not one giant blob. Errors should come out as clean category strings, never raw exceptions. Let's keep it extensible for new policy fields down the road.

One extra pass on the edge cases the team asked about: tag filtering should use full-match regular expressions from excludedTags, so a tag gets dropped if it matches ANY pattern in that blacklist, and this is full match, not substring. After that filtering, if nothing is left, emit exactly one line: `tags=none`. If there’s just a surviving single tag, we always keep it. Any combination expansion is only from per-service rules that match exactly the service name in the request, not anything broader.

For the combination output itself, for a service with two-tag combination rules, emit a joined value `a,b` for every surviving pair of tags where one matches the first pattern and one matches the second. Inside that joined value, the two members are ordered ascending (lexicographic). For three-tag combination rules, also emit the joined two-tag sub-pairs for every matching sub-pair, plus the joined three-tag value for each matching triple. Final output is sorted ascending, and it’s still one `tag=<value>` per line.

On the outgoing permission side, when that feature is enabled and a service asks for the `*` wildcard, the service has to be in `servicesAllowedToUseWildcard`. If it isn’t, emit `error=wildcard_not_allowed` and then `service=<serviceName>`. If it is allowed, emit `result=accepted`.

For versioning, the thing we wanted is per-group tracking, not a raw hash or timestamp. Cluster versions are `c0`, `c1`, etc. and endpoint versions are `e0`, `e1`, etc. Each one only increments when that specific content actually changes for that group. So an endpoint-only change bumps the endpoint label and leaves the cluster label alone. If the same group gets the same content again, both labels stay as they were. Per step the output should be `step=<N> clusters=c<X> endpoints=e<Y>`. This is the same snapshot behavior we talked about in feature8_snapshot_versions.json, where cluster versions use labels c0/c1/... and endpoint versions use e0/e1/... incremented independently per group only on actual content change, rather than using raw hash strings or timestamps.

Also on header injection, these are separate switches. The `x-service-name` request header with the caller’s service name as value is added ONLY when incoming permissions are enabled for that instance. The `x-envoy-upstream-remote-address` response header is added ONLY when `addUpstreamAddressHeader` is explicitly true. Output format stays `requestHeaders=[...]` and `responseHeaders=[...]` using bracket-enclosed comma-separated `key=value` pairs.

Ingress route ordering should be fixed so we don’t get churn: first token-gated secured path routes, including the 401 fallback per secured path+method, then public admin routes with POST before the catch-all method, then the redirect route for the status path without trailing slash, then the per-HTTP-method retry routes for enabled methods, and finally the catch-all local route. If `createVirtualCluster` is true, list virtual clusters as `status,endpoints`. We should also report the total as `routes=N`.

For mode support, if the instance requests ADS mode with `ads: true` but the server has `adsSupported: false`, emit `error=mode_not_supported` and then `mode=ADS`. If the instance requests XDS mode with `ads: false` but `xdsSupported: false`, emit `error=mode_not_supported` and then `mode=XDS`. The mode string in the error is the uppercase label of the requested mode.

For the incoming summary shape, endpoints should render as `path|matchingType|methods=[...]|clients=[...]`, with methods and clients both as bracket-enclosed comma-separated lists. Endpoints are sorted ascending, and the full line is `incoming.endpoints=[ep1,ep2,...]` using those rendered endpoint strings.

One formatting detail to keep consistent: duration strings in output use exactly 3 decimal places when the input had a fractional component, so `13.33s` becomes `13.330s`. Whole seconds stay as-is, so `777s` stays `777s`. If there’s no timeout configured, output `none`.

And one parser behavior to mirror from earlier auth path handling: in feature3_incoming_endpoint_path.json, if a JSON field is present with an explicit null value, treat it as absent. That should match the prior optional-field behavior. The team specifically pointed at the `includeNullFields` test case in public_test_cases/feature3_incoming_endpoint_path.json.
