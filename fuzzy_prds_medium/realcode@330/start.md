## Product Requirement Document

We need to finish the black-box surface for our transparent build accelerator, not just the wire serialization layer. The tool should expose a deterministic adapter for the same scenarios our tests cover: command-line/help and invalid option handling, direct command execution with stdout/stderr/exit status passthrough, descendant process interception across common execution methods, environment-fixup behavior for missing or hostile environment variables, diagnostic stream passthrough for warnings/errors, cache-hit acceleration behavior across repeated builds, and the typed wire-message serialization format used internally by the accelerator. Keep the core accelerator/process/serialization logic separate from the JSON adapter, and preserve the exact stdout contracts in the public cases.

One more thing the team asked about on the typed wire-message side: the fixed schema is exactly 10 fields, and the names need to stay canonical everywhere, including the numeric suffix in both wire encoding and decoded output. That set is ri1 (required int), oi2 (optional int), ri3 (required int), oi4 (optional int), rs5 (required string), os6 (optional string), rs7 (required string), os8 (optional string), sa9 (string array), sa10 (string array). This is the compact, allocation-free typed wire-message layer from feature 7 in the PRD, specifically features 7_1 (serialize all fields), 7_2 (optionals absent), and 7_3 (missing required fields), with test cases in rcb_tests/public_test_cases/feature7_*.json.

Also, for the wire frame itself, the payload length has to be padded to an 8-byte boundary, and that has to be true in both places, not just one. The declared payload length in the frame header needs to include the padding, and the actual bytes written need to reflect that same padded size. In decoded output we should see both `payload_padded_to_8=true` and `declared_length_matches_payload=true`, and both need to pass at the same time, as called out in feature7_1_serialize_all_fields.json.

On required vs optional behavior, if we’re building a message and any required field is missing, we should fail before producing any wire bytes at all. The error format there needs to be exactly `error=missing_required_field` plus a `field=<name>` line for the first missing required field. The required fields are: ri1 (required int), ri3 (required int), rs5 (required string), rs7 (required string). Optional fields are oi2, oi4, os6, os8, sa9, sa10.

For decoded output, each optional field needs an explicit presence signal. For optional integers (oi2, oi4) and optional strings (os6, os8), if the field is absent we should emit `has_<fieldname>=false` and not emit a value line. If it’s present, emit `has_<fieldname>=true` and then `<fieldname>=<value>`. For the string arrays, sa9 and sa10, if they’re unset or explicitly empty they should decode as `<name>_size=0` and there should be no element lines.
