## Product Requirement Document

hey team, we need to ship the distributed tracing toolkit we've been talking about. basically the pain point is that every service team is rolling their own header parsing and sampling logic and they keep breaking each other's traces because nothing lines up end-to-end. we had that incident last quarter where the checkout service dropped 40% of traces because their hex encoding was subtly wrong compared to what the auth service expected — that kind of thing.

the core pieces we need: some way to pack and unpack the trace context into a single header string (remember how we handled the sign/unsigned thing in that payments codec? do something consistent like that), a few different sampling strategies including one that gives a guaranteed floor so low-traffic operations still show up, a token bucket thing for rate controlling the volume (needs to work with a fake clock for testing), some small helpers for IP addresses and metadata keys, support for the B3 headers that zipkin folks use, the text-map propagation config, a metrics backend for counting spans, and a baggage policy checker.

please keep things modular — last time someone jammed everything into one file and it was a nightmare. errors should be clean categories not raw exception dumps. ping me if you have questions on the sampling math or the wire format details.

a couple follow-ups from the questions that came in. for the trace header wire format, the four fields are joined by colons with no zero-padding on the ids — only their natural hex length is used. all three 64-bit ids in context_encode need to render as unsigned lowercase hex, using full unsigned two's-complement for negatives, so for example -10 encodes as 'fffffffffffffff6'. the flags byte is rendered as its unsigned hex value (0 through ff). on the way back in, context_decode parses each of those three 64-bit id fields as signed 64-bit integers, so 'fffffffffffffff6' round-trips back to -10, and the flags field is parsed as a signed byte, so hex '81' (decimal 129 unsigned) decodes to -127 as a signed byte. the decode output should be the four lines trace_id=, span_id=, parent_id=, flags= with the signed decimal values.

also on bad input, let's keep the errors normalized and boring. an empty input string yields exactly 'error=empty_tracer_state_string'. if it's non-empty but the colon-separated field count is not exactly four, return 'error=malformed_tracer_state_string' and then 'value=<raw input>'. those are the categories we want, not exception text.

for text-map propagation, the defaults are context_key='uber-trace-id', baggage_prefix='uberctx-', and url_encoding as configured, and all three are overridable. the config output emits exactly three lines: context_key=, baggage_prefix=, url_encoding= with true or false for that last one.

on metrics, they are keyed by both name and their exact tag set, so exact-match behavior matters. querying a name/tag combination that was never created returns -1. a created counter or timer starts at 0 and accumulates increments via inc or recorded durations via duration. a gauge starts at 0 and each update replaces the current value entirely instead of accumulating. each query result line in tracer_metrics output should look like '<metric_name>|<tags_string>=<value>', where the tags string is exactly the comma-separated key=value string from the query, like 'sampled=y,state=started'. as one concrete sanity check, starting one sampled span increments jaeger:started_spans with tags 'sampled=y' and jaeger:traces with tags 'sampled=y,state=started' to 1. if the query tag string does not exactly match a created series, return -1.

for baggage restrictions, the default policy allows every key regardless of the service name or key name queried, and every allowed key has a maximum value length of 2048 characters. output there is just two lines: 'key_allowed=true' and 'max_value_length=2048'.

and for rate limiting, sampler_type is 'ratelimiting' and sampler_param should be reported as a decimal with one decimal place, so '123.0' not '123'. with a fresh bucket the first request is always admitted, meaning sampled=true. one important implementation detail here: the limiter must use an explicit virtual clock driven by advance_micros and advance_millis fields in each step, not wall-clock time, so the behavior stays fully deterministic and reproducible.
