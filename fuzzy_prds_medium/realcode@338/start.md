## Product Requirement Document

Hey team, we need to ship the TypeScript/JS code formatter plugin. Quick summary of what we need: it should take messy source code and spit out clean, consistently styled output. Users have been complaining that their code looks different across machines and PRs are full of noisy whitespace diffs — super frustrating for the team. The formatter needs to handle all the usual stuff: spacing, indentation, string quotes, semicolons, trailing commas, that kind of thing. Should also support JSX and the CJS module variant (check how we handled dialect detection in the file-type logic we already spec'd out). One thing that came up in the last retro — people want to be able to drop a special comment to opt out of formatting for a specific node or the whole file, similar to what we did for the ignore patterns in that other plugin. Also needs to be robust: bad input or weird config values shouldn't crash anything, they should come back as a clean error message. The whole thing needs to be split into proper modules — no giant single-file dumps please, the last time someone did that it was a nightmare to maintain. Oh and make sure running the formatter twice doesn't change anything the second time around. Ping me if anything's unclear but I think the test cases cover most of the edge cases we care about.

A couple follow-ups from the questions that came in. On errors, let’s keep the output super exact and predictable: it should always come back as two lines in key=value form, with the first line `error=<error_type>` and the second line `detail=<message>`. For source we can’t parse, that means `error=parse_error` and `detail=could not parse source text`. For bad config values, that means `error=invalid_config` and `detail=Found invalid value '<value>'.` exactly like that, including the period and the single quotes around the value.

Also wanted to pin down the ignore behavior a bit more. We have two directives. Whole-file ignore is `// dprint-ignore-file`, and that should work with or without a space after `//`, as long as it shows up as the first comment in the file. In that case we return the entire file verbatim, no reformatting at all, and no trailing newline gets added. The per-node version is `// dprint-ignore`, and that only applies when it immediately precedes a statement. In that case just that single node stays verbatim and the rest of the file still formats normally.

For file type handling, the parser dialect is chosen from the `file_name` field on the request. If no file name is provided, default to TypeScript with `.ts`. JSX should be accepted for plain JavaScript `.js` and CommonJS `.cjs` files too. The important nuance here is that `file_name` only decides which parser dialect we use to accept or reject syntax — it does not change the actual formatting rules. This is the same thing called out in Feature 13 in the PRD and `rcb_tests/public_test_cases/feature13_file_type_detection.json`.

One other small but important input cleanup item: if the source starts with a leading UTF-8 byte-order mark, U+FEFF, strip it before formatting. The formatter output should never include a leading BOM. After that, just format the rest of the source normally.

And on import sorting, this is only driven by `module.sortImportDeclarations`. The only supported value is `caseSensitive`. When that’s set, sort import declarations by their module specifier string using raw code-point (byte-level) ordering, so uppercase letters sort before lowercase. We’re only reordering whole import declaration lines here, not reordering the bindings within an import.

Last thing, just so there’s no ambiguity, the ignore directive behavior above is the same scope we called out under Feature 10 in the PRD for `// dprint-ignore-file` and `// dprint-ignore`, including the coverage in `rcb_tests/public_test_cases/feature10_1_ignore_file.json` and `feature10_2_ignore_node.json`.
