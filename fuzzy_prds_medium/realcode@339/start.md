## Product Requirement Document

Hey team, we need to build out that compression/archive toolkit we've been talking about. The core idea is that users should be able to take raw bytes, run them through various compression formats, and get them back out intact — all in memory, no file system touching. We also need to handle the reading/writing side of things for binary data (think header parsing, integer decoding, that kind of thing). Checksumming needs to work both as a one-shot call and as a streaming/chunked approach, similar to how we handled the incremental digest stuff in the old hash utility module.

For the archive side, we need ZIP and TAR support. ZIP should handle password-protected files (same crypto approach the old zip utility used), non-ASCII filenames, and the big-file extension. TAR needs to cover the various historical flavors and long filename extensions. Both need round-trip guarantees.

One thing that came up in review: the collection management piece needs to handle duplicate names gracefully — ask Sarah about the exact behavior we agreed on for ordering. Also the error handling story needs to be clean; external callers can't be seeing internal exceptions bubble up.

Compression-wise we need DEFLATE, zlib, gzip, bzip2, and xz. The whole thing should be organized properly — no giant single files please, we got burned on that with the old codec monolith. I/O contract goes through JSON stdin/stdout for the adapter layer.

Quick follow-up from the questions that came in. On checksums, we should be really literal about the output format: CRC-32 values must be printed as exactly 8 lowercase hexadecimal digits, zero-padded. For example, a value of 0 is printed as 00000000. The one-shot op prints 'crc32=<value>' and the streaming digest op prints 'digest=<value>' using the same 8-digit format. Also, the CRC-32 of an empty byte sequence is zero, reported as crc32=00000000. The Adler-32 of an empty sequence is the initial seed value 1, reported as adler32=00000001. And on the streaming side, yes, this is meant to be a stateful object fed data in chunks and then finalized, mirroring both crc32_digest and adler32_digest ops as described in features 1.2 and 2.2 of start.md.

For ZIP, decoding needs to support ZipCrypto password-protected entries. The 'password' field in the input provides the decryption key as a string. After decryption, the entry's CRC-32 must be verified. Output follows the same format as unencrypted zip_decode: files count and per-entry name, size, isfile flag, and crc32. zip_decode also needs to handle zip64 size extensions for large entries and non-ASCII (UTF-8) entry names. If there’s a 'comment' field from the end-of-central-directory record, print it as 'comment=<text>' before the file count when present. Directory entries appear in the file list but isfile=false. Also just to be explicit, this is ZipCrypto (traditional PKZIP encryption) decryption using the 'password' field from the JSON input.

For TAR, please make sure tar_decode covers old GNU, star, and classic V7 dialects. For each entry output: files count (once), then name, mode (decimal), uid, gid, size, modtime (Unix timestamp decimal), type (single char, '0' for regular file), uname, gname, link. GNU long-name (@LongLink) entries must reconstruct the full filename for the following entry. For symbolic links, the type flag is '2' and the 'link' field contains the target path, so the output for a symlink entry includes 'type=2' and 'link=<target>'. Regular file entries have type '0' and link is empty string.

On the collection behavior, the duplicate-name case is now nailed down: in archive_build, adding an entry whose name already exists replaces the existing entry's content in place without changing the order or count. The collection preserves insertion order. Output is 'length=<count>' then per entry 'entry=<name> b0=<hex first byte>'. If 'clear' is true, all entries are removed after processing and length=0 with no entry lines.

One more thing on failures since folks asked how strict we want to be: when an operation cannot complete due to invalid input (e.g. bytes that are not a valid archive), the adapter prints exactly one line: 'error=invalid_archive\n'. No stack trace, exception class name, or additional lines may appear. This applies to tar_invalid and any parse failure in tar_decode or zip_decode.

And for xz_decode, it needs to handle streams with no integrity check and streams with CRC-32, CRC-64, or SHA-256 checks. An empty xz stream decodes to zero bytes, output as 'length=0\ncrc32=00000000\n'. All xz decode outputs follow 'length=<n>\ncrc32=<8hex>\n'.
