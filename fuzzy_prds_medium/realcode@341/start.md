## Product Requirement Document

Hey team, we need to build out this spell-checking tool for source code that we talked about in the last planning session. The big complaint from devs is that existing spell checkers are totally useless on code — they flag every identifier, URL, hex code, whatever, as a misspelling and people just turn them off. We need something smarter.

Basically it should be able to break apart compound names (like the camelCase stuff, underscored names, etc.) into real words before checking, and it should know to skip over technical garbage like URLs and email addresses and file paths. It also needs to actually understand code structure for different languages — so it only checks the parts humans wrote as words, not keywords and operators and all that.

There should be a way to get spelling suggestions for bad words, and the end-to-end check should support some kind of override list so teams can whitelist their own domain terms. Also teams sometimes have naming conventions that look like misspellings — there should be a way to define patterns to skip those too.

For the architecture, we want this structured properly — not one giant file. Think about how we did the config layering on the auth module last quarter, similar vibe for extensibility. The I/O adapter should be totally separate from the core logic. Output should be byte-accurate positions so editors can use it directly.

One detail the team asked about on the word-splitting side: splits happen at a lowercase letter immediately followed by an uppercase letter, and also inside a run of uppercase letters where the last capital of the run sticks with the next lowercase word, so XML before Http comes out as XML|Http. We also split at underscores, periods, colons, and hyphens, and those separators are consumed and never emitted as words. Same idea for every transition between letters and digits. If a segment is only digits, we discard it entirely. For anything that survives, we report the word with its UTF-8 byte start offset, end offset, and text.

On the stuff we should ignore, the built-in skip patterns cover web URLs with http/https schemes, email addresses, hex color codes introduced by a leading '#', absolute filesystem paths, and long hexadecimal hashes. If a word is fully or partially covered by one of those masked spans, it gets dropped from results, but normal words around those spans should still come through.

For dictionary lookups, a query word counts as known when its lowercased form equals the lowercased form of any dictionary entry, so the original casing on either side does not matter. The output for each query is just the word exactly as given, then a space, then 'true' or 'false', and it should stay in the same order the queries were supplied. For suggestions, candidate casing should adapt to the query word’s casing: an all-uppercase query uppercases every candidate, an all-lowercase query lowercases every candidate, and a mixed- or title-cased query leaves the candidate text unchanged. Return at most five candidates, one per line, in ranked suggestion order.

Also, to make the syntax-aware part more concrete, we only want to extract declared names like function names, parameter names, local bindings, type names, and field names, plus comment text and the contents of string literals. Language keywords, operators, punctuation, and numeric literals are excluded. If the language identifier is unrecognized, don’t fail the whole thing — just fall back to treating the input as plain prose.

And on the extensibility point from the original note, the intent is still that adding a new source language or a new skip pattern must not require modifying the core checking engine. The way we get there is by depending on a dictionary abstraction and a configuration abstraction (DIP), and by registering language parsers and skip patterns as pluggable extensions instead of baking more hardcoded branches into the engine.
