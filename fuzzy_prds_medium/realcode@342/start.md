## Product Requirement Document

Hey team, we've been getting complaints from devs using the local sidecar tooling that they keep hitting subtle issues when working with the cluster status and pod inspection features. Specifically, some folks are confused about how the tool decides whether a deployment is 'healthy' vs not, and what gets shown in the status table (like how age is formatted, or what happens when multiple replicas of the same service are running). There's also some confusion around how the tool figures out which pods match a given selector — apparently it's not obvious how the matching logic works when labels only partially overlap. On a separate note, a few people hit a weird edge case where the tool crashes or gives garbage output when the client isn't initialized yet, which should just be a clean error. We want to make sure the status and pod-listing outputs are consistent and human-readable. I think the logic for how we format ages and aggregate replicas should follow what we did in the dashboard command stuff, but I can't find the exact reference — just make sure it's consistent with how the rest of the CLI handles those. Please make sure the output format is stable and matches what downstream tooling expects.

Quick follow-up after the team’s questions: for age, we should keep it really simple and derive it straight from age_minutes. If the age is 0 minutes, render it as '0s'. Otherwise render it as '<N>m', so 20 minutes shows up as '20m'. No hours/days rollup in the basic output. Also, for the healthy value in the table, the output needs to use title-case boolean strings, so 'True' when the pod is ready and 'False' when it is not, not lowercase 'true'/'false' even if the underlying value people are thinking about is true or false.

On the grouping side, pods should be grouped by their 'app' label value. Anything with the same app identity gets collapsed into a single status row, and the replica count should reflect the total number of pods in that group. For that rolled-up row, the name, namespace, healthy, status, version, and age fields all come from the first pod in the group. One special case here: all control plane status rows should report the namespace as 'dapr-system', regardless of what namespace the pods were actually listed under in the input.

Also confirming that the version field in the status output is fixed at '0.0.1' for all pods in the test contract. It should not be derived from a container image tag or any other dynamic field in the pod input. And on the earlier reference point, Feature 10 (dashboard_cmd) and Feature 11 (metadata_endpoint) in start.md and public_test_cases/feature10_dashboard_command.json are only meant as the pattern for fixed binary names and simple arg rendering, not the age formatting piece. The actual age formatting contract is defined in public_test_cases/feature20_control_plane_status.json.
