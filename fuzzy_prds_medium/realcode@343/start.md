## Product Requirement Document

We need a geometry calculation engine for grid-based list layouts. The product allows developers to place visual separators between items in any kind of list — single-column, multi-column, or staggered. The engine should be able to answer questions about separators: where they sit relative to the overall grid boundary, whether a particular separator falls on an outer edge of the layout, and how the visual thickness of a separator is split between the cells on each side so items look balanced regardless of how many columns the grid has.

Specifically, we want the engine to support two types of grid models: one where we know exactly how all items are arranged (full layout data), and a simpler one where we only know the column count and direction. For the simpler model, we just need to know which outer sides of the grid a particular cell touches.

The offset splitting logic should work the same way as our existing padding distribution module — split the divider size proportionally across the cells in a row, but handle odd pixel sizes gracefully so complementary sides always add up to the full size. When the outer side separators are hidden, the distribution math changes significantly.

All output should be expressed as plain text key-value pairs, one per line, using lowercase keys. Coordinates and counts are zero-based integers. Error conditions should produce a normalized error token rather than exceptions, similar to how the authentication module standardizes error codes.

One more thing to make explicit since the team asked about the rounding and edge checks. For offset splitting, when the raw offset lands exactly on a .5 decimal like 5.5, the leading sides (TOP or START) take the floor like 5 and the trailing sides (BOTTOM or END) take the ceiling like 6. That’s the bit that keeps the paired offsets adding back up to the full divider size. For anything that is not exactly .5, we just use standard roundToInt(). This is the same behavior in normalizedOffsetFromSize in golden/recycler-view-divider/src/main/kotlin/com/fondesa/recyclerviewdivider/offset/NormalizedOffsetFromSize.kt, which uses a floor/ceiling normalization when rawOffset decimal part is exactly .5.

Also spelling out the offset formulas exactly so there’s no guesswork. When areSideDividersVisible=true: dividersInLine = spanCount + 1, cellSideOffset = multiplier * dividersInLine, rawOffsetBefore = multiplier * (spanCount - spanIndex), rawOffsetAfter = cellSideOffset - rawOffsetBefore. When areSideDividersVisible=false: dividersInLine = spanCount - 1, cellSideOffset = multiplier * dividersInLine, rawOffsetBefore = multiplier * spanIndex, rawOffsetAfter = cellSideOffset - rawOffsetBefore. The multiplier is always size / spanCount.

On the boundary flags, a horizontal divider is the bottom boundary only in two cases: the grid is vertical AND originY == grid.linesCount, or the grid is horizontal AND originY == line.cellsCount AND the line is filled with sum of all cell spanSizes == spanCount. That’s when isBottomDivider=true. A vertical divider is never a bottom boundary. Same idea for the end side: a vertical divider is the end boundary only if the grid is horizontal AND originX == grid.linesCount, or the grid is vertical AND originX == line.cellsCount AND the line is filled with sum of all cell spanSizes == spanCount. That’s when isEndDivider=true. A horizontal divider is never an end boundary.

And for accumulated_span, if the divider's orientation differs from the grid's orientation, the output is exactly: 'error=orientation_mismatch\n'. No numeric value should be returned in that case. The wording people asked me to confirm was basically when the divider's orientation differs from the grid's orientation, and yes, that includes the case someone described as s orientation differs from the grid.
