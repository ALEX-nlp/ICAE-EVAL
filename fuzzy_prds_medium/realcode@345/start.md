## Product Requirement Document

hey team, we need to wrap up the zonal extraction tool we've been building. the core engine is mostly done but we're getting complaints from a few data science folks that the output tables are inconsistent depending on how they pass in polygons and rasters. basically the ask is: given a set of polygon features and one or more raster grids, we should be able to spit out a clean CSV (or shapefile in some cases) with whatever summary numbers they need per polygon. things like averages, most common value, spread of categories, that kind of stuff. there's also a bunch of edge cases around empty intersections and missing data that keep biting people — same kind of handling we did for the nodata sentinel logic in the reprojection module, just make sure that's consistent here too. one tricky bit: when someone only has a single unnamed raster, the column headers in the output should just use the operation name directly instead of inventing some raster prefix. also cell-level outputs need to work where each intersecting cell gets its own row. and there's some metadata about scaling/offsets on raster bands that needs to be applied before values go out. the identifier column in output should support renaming and type-casting. geometry passthrough to shapefiles should be supported when requested. please make sure fractional overlap is exact and not just a nearest-cell approximation — that was a big complaint last time.

one small follow-up from the questions that came in: the unnamed-raster column naming rule is intentionally narrow. when exactly one raster is supplied and that raster has no name field, stats that don’t explicitly reference a raster should use just the operation name, like 'mean' or 'variety', instead of building a 'raster_operation' style name. that only applies when the raster is unnamed AND there is exactly one raster in the input. if the raster has a name, the compound prefix form is always used.

also, on overlap math, we do mean exact fractional area of overlap for every intersecting cell, not a simple intersects flag and not centroid containment. so if a cell is 25% covered, it contributes 0.25 to weighted aggregates like mean. and if someone asks for 'coverage' at cell level, that stat should expose that exact fraction directly on each row.

for nodata / empty intersections, please keep the behavior aligned with the existing logic: integer rasters use the configured 'nodata' field value as the sentinel mask, float rasters treat NaN as missing, and when all intersecting cells are missing the mode behavior depends on dtype. for an integer raster with all cells masked by the sentinel, mode returns the nodata sentinel itself, like -999, NOT nan. for float rasters where NaN is the missing value, mode should return nan. count and mean always return 0 and nan respectively when there are no valid observations, regardless of dtype.

one more detail on 'frac': the schema expands across the whole batch, not feature by feature. we emit one column per category observed across ALL features in the batch, named 'frac_<category_value>', and sort those columns by category value ascending. if a feature doesn’t contain a given category, that spot in the CSV should be an empty field, not zero and not nan, literally empty. and if a category appears in no feature, it should not be in the schema at all.

last thing, the cell-level stats are 'coverage', 'values', 'cell_id', and 'centroid_x'/'centroid_y'. if any of those are requested, output switches to one row per intersecting raster cell per feature instead of one summary row per feature, and all requested cell-level columns need to line up on the same per-cell rows. 'cell_id' is the linear raster index of the cell in row-major order, zero-based, and it should be output as a quoted string. rows should be ordered by cell_id ascending.
