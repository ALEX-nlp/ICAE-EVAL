## Product Requirement Document

Hey team, we've been getting complaints from a few internal devs that our numeric safety toolkit is hard to use consistently — people keep accidentally doing things like mixing up index types with plain integers, or forgetting to check if an optional value is actually there before using it. We also had a bug last sprint where someone did a cast without realizing data was getting silently truncated (you know, the same kind of thing we ran into with that old unit conversion module). There's also been confusion around the flag management stuff — devs aren't sure whether toggle returns the old state or the new one, and whether the conditional set operations actually report back what happened. Oh and the interval boundary behavior tripped someone up — they weren't sure if the endpoints counted or not. We want a single adapter binary that reads a JSON command from stdin and writes a clean key=value report to stdout, covering all these scenarios. Output needs to be totally deterministic and byte-comparable. Boolean values must render as lowercase true/false in the report lines. Numbers should just print naturally. Errors should report a normalized category, not raw exception text. Can we get this specced out properly and implemented? Should cover integers, floats, booleans, flags, index math, constraints, optionals, variants, and the compact storage trick where the empty state lives inside the value domain itself.

A couple follow-ups from the questions that came in. For increment behavior, we should spell it out directly: post-increment returns the value BEFORE the increment, and the stored state becomes value+1. Pre-increment returns the UPDATED value. In both cases we should report both pieces, so result= is what the operator returned and value= is the state after the operation. Concretely, post_increment on 7 yields result=7 and value=8.

On the flag side, toggle returns the PRIOR state, not the new one. The output lines there are returned= for what toggle gave back and state= for the resulting state. So if a flag starts false, toggling it yields returned=false and state=true. Also for the conditional flag ops, try_set returns true only if the flag was not already set, and try_reset returns true only if the flag was set and actually got cleared. Same reporting shape there too: returned= for whether a change happened, and state= for the resulting flag state. The flag set itself has exactly three named flags: a, b, and c. For summaries, all= is true only when all three are set, none= is true only when none are set, and any= is true when at least one is set. Individual membership should print as a=, b=, c= in that order.

For deferred construction, the slot starts empty. We should report initial_has_value=false before assignment, then after assignment report has_value=true and value=<n>. It can be filled either by direct assignment or in-place construction, but only once. Related to that, for the output parameter behavior, assignment writes through to the backing storage immediately. The assign_value op sets the backing store to the given value, and the report line is backing=<value>. If it’s a string, the backing stores the string as-is.

On the strong typedef, the intent is that it opts into arithmetic, comparison, and bitwise operator sets. Supported arithmetic ops include +=, -=, *=, /=, %=, and their binary forms (+, -, *, /, %). But it still stays a distinct type and cannot be implicitly mixed with the underlying primitive.

For the variant lifecycle, after emplacing an int alternative the type_index reported is 1. The expected flow is initial_has_value=false, then type_index=<n> and value=<v> after emplace, then after_destroy_has_value=false after destroy. The type index maps to the position in the union definition, so int=1, double=2, or similar.

And one more note on the historical truncation issue we mentioned: Feature 7 (narrow_cast) is specifically about explicit narrowing conversion between integer widths or floating-point widths. That old footgun was lossy silent truncation, and this path is handled via the narrow_cast fn with kind=integer or kind=float.
