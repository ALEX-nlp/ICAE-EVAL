## Product Requirement Document

Hey team, we need to build out that message transport adapter thing we've been talking about. Basically developers are copying the same boilerplate over and over every time they wire up a new queue or topic, and it's causing all sorts of inconsistencies when messages move between services — metadata gets dropped, retries are handled differently in every repo, that kind of mess. We need a clean library that handles the whole lifecycle: turning typed objects into something you can actually put on a wire, wrapping and unwrapping those standard event containers we agreed on (you know, the same envelope shape the platform team standardised last quarter), dealing with the three different outer wrapper formats that come in from our infra layer, and making sensible decisions about whether to back off and wait when the queue goes sideways. Settings should load from JSON config so teams can plug in their publishers, subscribers, and pollers without touching code. The retry stuff should support at least a no-op mode, a fixed-wait mode, and that exponential thing with a ceiling — similar to what we did for the login service polling loop, just adapted here. Output from the test harness should be flat key=value lines so our CI scripts can parse it easily. Please make sure the field naming conventions on the serialised objects are consistent — there were complaints last time about casing mismatches.

One extra pass on the shape here since a few folks asked for specifics. For the application message, the serialised Person JSON needs to use PascalCase keys exactly as FirstName, LastName, Age, Gender, plus a nested Address object with Unit, Street, ZipCode. On the way back out, we read those same PascalCase keys and print the parsed values as flat snake_case lines: first_name, last_name, age, gender, address_unit, address_street, address_zip.

For the event container, please stick to the exact field names id, source, specversion, type, time, and data. That’s specversion specifically, not version, although when we print parsed envelope fields the output key for it should be version. Also, specversion must always be the string '1.0'. When creating a new envelope the version is fixed at '1.0', and the type identifier for an address payload is 'addressInfo'. The data field is a little special: when we JSON-encode the envelope it needs to be serialised as a JSON string, not a nested object. And for the compact JSON form, the inner data string should escape double quotes as \u0022 while the outer envelope still uses normal JSON quotes, so the final data value looks like "{\u0022Unit\u0022:123,...}".

A couple metadata details too. If there’s extension metadata on the envelope, print it with the prefix metadata_ using the part of the key before the first hyphen, so 'some-metadata' becomes 'metadata_some'. SQS metadata lives under the 'SQSMetadata' key and the sub-fields there are MessageGroupId, MessageDeduplicationId, and MessageAttributes. On the SNS side, attributes come from the SNS envelope's MessageAttributes field and should be printed as two lines per attribute in order: sns_attr_{name}_type={DataType} and sns_attr_{name}_value={Value}. For the test case that means attr1 with type=String and value=val1, then attr2 with type=Number and value=3.

On wrapper handling, we do need all three modes behind the 'wrapper' discriminator. 'none' means the body is already an event envelope, and in that case we expose SQS receipt, group, dedup, and message attributes. 'sns' means the body is a notification envelope, and we expose SNS message id, topic ARN, subject, unsubscribe URL, timestamp, and SNS attributes, plus empty SQS fields. 'eventbridge' means the body is an event-routing envelope, and we expose event id, outer source, detail type, account, region, time, and the comma-joined resources list, plus empty SQS fields. This is the CloudEvents-inspired envelope with fields id, source, specversion='1.0', type, time, and data as defined in feature 2 (cloud_event_parse / cloud_event_create / cloud_event_json_encode operations).

Last bit on retries: the capped_exponential policy is delay = 2^retryCount seconds, capped at capBackoffTime (default 60s), and it only kicks in when errorCategory is 'generic_queue_error'.
