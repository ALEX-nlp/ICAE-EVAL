## Product Requirement Document

Hey team, we need to get this graph library wrapped up and working end-to-end. The core idea is that developers should be able to throw any kind of graph at it — the undirected kind, directed, the ones keyed by actual values, the stable-index ones where holes are okay, and also that union-find thing — and run all the usual algorithms without writing their own adjacency bookkeeping. Think shortest paths, traversals, cycle checks, spanning trees, component grouping, and even that structural equivalence check we talked about (similar to what we did for the equality logic in the mapping module, you know the one). The tricky bits are around mutation: when you remove a node everything touching it has to disappear cleanly, but unrelated stuff must survive untouched. Also the edge-update behavior needs to be consistent — no surprise duplicates when you update the same connection twice. The stable-index graphs are a bit special because outside code might be holding onto old IDs so we can't just renumber things after a removal. Error cases need to surface clearly rather than silently doing the wrong thing, especially when someone references an index that no longer exists. Output for the DOT format export also needs to be exactly right. We're using the petgraph crate as the backbone so make sure the adapter layer lines up with how that API actually works. Can someone own the full implementation and make sure all the public test cases pass?

Quick follow-up after the questions came in. On the undirected mutation case in feature1, we do want true parallel edges here, so same endpoints can exist more than once and they stay distinct. That’s why before_edges=7 is correct for this exact sequence: add_edge(a,b,0), add_edge(b,a,0), add_edge(a,a,0), add_edge(a,b,0), add_edge(d,a,0), add_edge(b,c,2), add_edge(d,a,0). That includes the reciprocal A-B adds, the self-loop, the extra A-B, the two D-A adds, and the B-C edge. Also, the neighbor list of b before removal should come out as [0, 2, 0].

For feature2, the replacement behavior needs to follow petgraph’s update_edge semantics, not add_edge. If an edge for that endpoint pair already exists, update_edge returns the old edge index and reuses it. So for directed, if we add A->B as e0 and C->D as e1, then update_edge on A->B should give back e0, set weight=2, and leave total directed_edges=2, with directed_reused_forward_id=yes. For undirected, add A-B as e0 and then update_edge B->A should still return e0, set weight=2, and total undirected_edges=1, with undirected_reused_reverse_id=yes.

For feature4_3, please build a DiGraphMap<i32,i32> with 9 nodes (0..8) and edges arranged so the SCCs are [0,3,6], [1,4,7], [2,5,8]. Use petgraph::algo::kosaraju_scc, and make sure the output is normalized the same way every time: each component sorted ascending, and then the list of components sorted by first element. When converting to indexed form, use petgraph::graph::Graph::from(graphmap) or into_graph(), and the result should be converted_nodes=9 and converted_edges=11. That 11 is the full directed edge total across those three cycles plus the inter-component links.

On feature5_1, the right type is petgraph::stable_graph::StableGraph. Start with 10 nodes (0..9), remove 1,3,4,5,6,7,8,9, and you should be left with remaining_node_indices=[0,2]. The bound behavior is important here: node_bound_full=10 and node_bound_after_removals=10 because graph.node_bound() does not shrink just from removals. Separately, if you do a real graph.clear(), that wipes nodes and edges and you should get node_bound_after_clear=0. But the reused_after_clear_edges=[1,9] expectation is about clear_edges(), not clear(), so the actual sequence there is add 10 nodes, remove 8, clear_edges(), then add new nodes into the vacant slots, with edge_count_after_clear_edges=0 and the new nodes reusing 1 and 9.

For feature5_2, also use StableGraph but directed. Build the 0..9 graph, form the SCCs, remove node 4, then add a replacement and StableGraph should reuse that vacancy so the replacement is node 9 in the expected output grouping. The SCCs after that should be [0,3,6], [1,7,9], [2,5,8]. Both petgraph::algo::kosaraju_scc and tarjan_scc need to agree, and again the components should be sorted with each component ascending and the full list sorted by first element.

For feature5_3, the DOT export needs to match exactly. Use petgraph::dot::Dot on a StableGraph with node 0 labeled 'x' and node 1 labeled 'y', plus edges 0->0 with weight 10 and 0->1 with weight 20. Dot::with_config or Dot::new is fine, and formatting can be with {:?} or display, as long as the exact output is 'digraph {\n    0 [label="x"]\n    1 [label="y"]\n    0 -> 0 [label="10"]\n    0 -> 1 [label="20"]\n}\n'. The node label should come from the node weight and the edge label from the edge weight, with no extra attributes getting injected.

For feature5_4, please guard add_edge on StableGraph yourself. If the index is vacant because that node used to exist and was removed, petgraph may not give you a clean failure, so we need to check first. Same idea for an out-of-bounds index. The helper should use graph.contains_node(idx) before calling add_edge, with indices built via NodeIndex::new(n), and if the node is missing just return 'error=invalid_graph_reference\n'. That same normalized string should be used in both cases.

For feature6, this is petgraph::unionfind::UnionFind<usize>. Start with 8 elements (0..7), and initially find(i)==i for all of them. Then union(0,1), union(0,3), same(1,3)=yes, different(0,2)=yes, union(7,0), union(6,5), different(6,7)=yes. We do still need to preserve set_count=3 exactly as written, and the separate labeling check is specifically that labeling_single_component=yes when using into_labeling() on a fully connected structure. So treat the set-count expectation and the fully connected labeling check as two distinct moments in the flow.

And on feature7_1 / feature7_2, this should use petgraph::algo::is_isomorphic and is_isomorphic_matching for the structural equivalence checks. For the incremental directed Graph case, the expected progression is empty graphs yes, one node each yes, two nodes each yes, left gets extra node no, three nodes each yes, left gets one directed edge no, then both get matching directed edges yes. The renamed-shape version is still yes because insertion order and names don’t matter for plain structure. For the matching version, use node matcher |a,b| a==b and edge matcher |a,b| a==b, and after changing an edge weight on one side the result should be no in both checks, so edge0_weight_changed=no and edge1_weight_changed=no.
