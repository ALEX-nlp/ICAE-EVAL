## Product Requirement Document

Hey team, we need to wrap up the auth framework work we've been discussing. The basic idea is that app developers shouldn't have to hand-roll all the account flow stuff — sign in, sign out, registration, the whole nine yards. We talked about this being similar to what we did on that other project with the scope-based session handling, so just follow that same pattern for the multi-scope parts.

A few things I want to make sure are covered: users have been complaining that when they type their email in a different case it sometimes doesn't log them in — that needs to be handled consistently. Also we got a bug report that after a password reset the user isn't being signed in automatically, which is frustrating UX. The locked account flow and the confirmation email flow should both support both the regular HTML path and also the JSON API path since some of our consumers are mobile clients.

The remember-me cookie behavior needs to work properly — right now it's not clear if a bad/expired remember token just silently fails or actually redirects. And the session expiry stuff needs to respect whether we're kicking out just one scope or all scopes at once, same logic as the sign-out scope policy. Tracking sign-ins (count, timestamps, IP) should also be suppressible. Can someone make sure all of this is specced out properly before we start? Thanks

One extra detail from the questions that came back: email handling needs to be configurable per account scope, and that applies to the identifier field, including whitespace stripping. The case normalization toggle and the whitespace stripping toggle are independent. So when normalization is enabled, having 'Foo@Bar.com' stored and submitting 'foo@bar.com' should still authenticate successfully and come back status=200, authenticated. When normalization is disabled, mismatched case needs to fail authentication and return the invalid-credentials message.

For scope behavior, the sign-out policy is the configurable boolean 'sign_out_all_account_scopes'. When false, signing out the 'user' scope should leave the 'admin' scope still authenticated with authenticated_admin=true. When true, signing out any single scope terminates all active account scopes. Either way the response is always a 302 redirect, typically to '/'. That same cross-scope policy logic from feature3_sign_out_scope_policy, specifically the 'sign_out_all_account_scopes' boolean flag, also needs to be mirrored in the session timeout scope policy in feature 11 and in the multi-scope sign-in/sign-out flows.

Also, account updates should require the current password to be correct, and if the current password is wrong the persisted account data for email and password must remain unchanged. On a successful update with a new email, the result should be status=200, path=/, authenticated_user=true, message='Your account has been updated successfully.', and stored_email should reflect the new value. If a new password is submitted too, include a password_valid signal in the output.

On the password reset side, once a valid reset token is consumed with matching password and password_confirmation fields, the framework should automatically sign the user in. That response should be status=200, path=/, authenticated_user=true, with message='Your password has been changed successfully. You are now signed in.' and the stored password updated with password_changed=true. If the token is invalid or the confirmation fields do not match, the password must stay unchanged.

For the confirmation and unlock flows, both the email confirmation flow in feature 8 and the account unlock flow in feature 9 need JSON format support in addition to the normal HTML path. The JSON response needs to include a 'json_has_errors' signal. For a valid instruction request in HTML mode, the expected result is status=200, path=/users/sign_in, mail_count=1, and json_has_errors=false. If token consumption happens in JSON mode with an expired or invalid token, that should return JSON error signals instead of an HTML redirect.

And for remember-me, if a request comes in with an invalid or expired remembered-session token, we should redirect to sign-in and leave the user unauthenticated with authenticated_user=false, authenticated_admin=false. A valid remembered token should authenticate the user without an explicit sign-in. Also please make sure remember=false at sign-in means no remember cookie gets created, so remember_cookie_present=false, and remember=true means one does get created, so remember_cookie_present=true.
