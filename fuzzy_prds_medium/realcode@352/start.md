## Product Requirement Document

Hey team, we need to get the embedded device build pipeline sorted out properly. Right now devs are manually copying engine binaries and plugin libraries around and it keeps going wrong on-device because things end up in the wrong place or the wrong variant gets picked. We talked about this in the last sprint retro and everyone agreed it's a time sink.

Basically we need a pipeline that takes a compiled app and its native plugins and produces whatever the device runtime actually needs — the right layout, the right files, signed and ready to install. There are a few different flavors depending on whether you're doing a managed or native app, and whether you want a final installable thing or a flat module you can embed into a host app. We also need it to handle that whole situation with prebuilt libraries where only some of them are valid for your target — like what we did for the compatibility filtering in that login SDK integration a while back, same idea but for arch and api version.

Also important: if someone tries to package without a signing cert set up, it should fail gracefully with a clear message rather than just blowing up. Each stage should report back exactly which files were produced so you can inspect what happened. Can someone pick this up? The JSON contract for the adapter is already roughed out somewhere in the test cases folder.

One quick follow-up from the questions that came in: for the line-based reporting, every probed output should come back as exactly two consecutive lines, artifact=<relative path> and then present=true or present=false. Linkage metadata should come back as user_libs=<space-separated names>, and if we hit an error it should be a single line in the form error=<category>. There isn’t any other output shape around that, so no JSON, no headers, and no blank lines between pairs. This ties back to the stdin/stdout JSON adapter contract defined across rcb_tests/public_test_cases/ — each file specifies input {stage, scenario} and expected line-oriented output strings.

On the artifact specifics, the debug asset bundle stage needs to produce exactly three artifacts in this order: vm_snapshot_data, isolate_snapshot_data, and kernel_blob.bin. Each one is reported as an artifact= line followed by a present=true|false line. For the native C++ embedding stage, it reports two artifacts in this order: include/flutter.h and libembedding_cpp.a, and both have present=true.

For native plugins, the user_libs= value is a space-separated list of bare library base names, so the lib prefix and file extension are stripped, like libshared_arm.so becomes shared_arm. The toolchain default pthread is always prepended as the first entry. For the user_libraries scenario, the full order is exactly pthread some_native_plugin static shared shared_arm shared_40. Also, the Feature 3.4 user library selection logic in the native_plugins stage is the filtering of prebuilt libraries by target architecture tag and api version level, as covered in rcb_tests/public_test_cases/feature3_4_plugin_user_libraries.json.

One other packaging behavior to keep straight: if a plugin is declared as a static library, it gets folded into the combined shared object lib/libflutter_plugins.so. If a plugin is declared as a shared library, it is NOT folded in, so the combined object is absent and should be reported with present=false, and instead the plugin’s own shared object lib/libsome_native_plugin.so and any side-car shared objects like lib/libshared.so are reported as present.

And for plugin resources, we do want a straight recursive copy from the plugin’s platform resource directory into a per-plugin sub-directory under res/ in the build output, keeping nested paths intact. So if there’s a file at a/b.txt in a plugin named some_native_plugin, it should end up at res/some_native_plugin/a/b.txt.
