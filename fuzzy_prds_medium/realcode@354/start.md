## Product Requirement Document

Hey team, we need to wrap up the tree ensemble deployment toolkit. The core idea is that data scientists train models externally (XGBoost-style gradient boosted forests, including ones with those category-based branching conditions) and then hand them off to us — we need to load them, let engineers poke at the structure a bit, compile them down to something fast, and serve predictions. We've been burning too much time on every new model deployment because there's no standard interface, people keep shipping the full training stack into prod which is a nightmare, and scoring code keeps drifting out of sync with the actual model.

The interface should accept a JSON command on stdin and print key=value lines back — one action per invocation. We need metadata inspection, the ability to chop off trees from the back (like for ablation studies), compiled predictor info, batch scoring, single-row scoring with a few different ways to express missing values (refer to how we handled the null encoding in the feature store pipeline last time), and handling for when input data is the wrong width. Also need a serialization round-trip so models can be stored and reloaded without losing numeric fidelity. Oh and there are some compiler tuning knobs — quantization and that folding thing — those should never change the actual numbers output. Numeric outputs should be formatted consistently. Errors should be clean categories, not stack traces.

A couple of specifics to lock down from the questions that came in. For outputs, every numeric value we print needs to use exactly 6 decimal places in fixed-point form, like `0.110274` or `0.500000`, and if something would come out as negative zero we normalize it to `0.000000` and never print `-0.000000`. On model IDs, there are exactly three reference ensembles: `binary_mushroom`, `multiclass_dermatology`, and `categorical_toy`. The loader infers the format from that identifier, so the caller does not pass a separate format flag. `binary_mushroom` is a binary classifier with sigmoid output, 2 trees, 127 features, 1 output group. `multiclass_dermatology` is a 6-class softmax classifier, 60 trees, 33 features, 6 output groups. `categorical_toy` is a single-output regressor using categorical/set-membership splits rather than numeric thresholds.

On errors, keep them neutral and adapter-owned. The runtime/core can throw whatever idiomatic internal exceptions it wants, but the only thing that reaches stdout is key=value output. That means never host-language exception types, stack traces, or Python/Java/C++ error messages. Every error starts with `error=<category>` on its own line, then any supporting context fields after that, like `requested=`, `model_tree_count=`, `input_features=`, or `model_features=`. Related to that, tree limiting is only valid when the requested limit is an integer in the range [1, current_tree_count]. If it is 0, negative, or larger than the current model tree count, print exactly three lines: `error=invalid_tree_limit`, `requested=<N>`, `model_tree_count=<M>`, and leave the ensemble unchanged.

A bit more detail on metadata and prediction behavior too. The `predictor_metadata` action should compile the ensemble and then report exactly five fields in this order: `num_feature=<int>`, `num_output_group=<int>`, `pred_transform=<name>`, `global_bias=<float6>`, `sigmoid_alpha=<float6>`. The transform name is `sigmoid` for binary classifiers and `softmax` for multiclass. For `binary_mushroom`, use global_bias=0.000000 and sigmoid_alpha=1.000000. For `multiclass_dermatology`, use global_bias=0.500000 and sigmoid_alpha=1.000000. For single-instance prediction, the three missing-value encodings we support are `sparse`, `dense_zero_missing`, and `dense_nan_missing`, matching Feature 5 of the PRD and the behavior covered in `rcb_tests/public_test_cases/feature5_single_instance.json`. Also, the `code_folding` compiler tuning option is passed as a float in `predict_batch`, for example `code_folding: 2.0`, and like the other compiler knobs it must not change the actual prediction numbers; this is the same invariance covered in `rcb_tests/public_test_cases/feature8_compile_option_invariance.json`.

Last thing is structure. This needs to live as a multi-module repository with real separation between the model representation module, compiler module, runtime predictor module, serializer/deserializer module, and I/O adapter module. The compiler should be open for extension and closed for modification so we can add output formats or tuning later without rewriting the core. The adapter alone should deal with JSON parsing and stdout rendering, and the compiler, predictor, and model representation need to stay fully decoupled from stdin/stdout and JSON. Please do not flatten this into one file. And for `categorical_toy`, just to be explicit, it is only exercised in margin mode with `pred_margin: true`, it returns one regression value per row, and the expected reference numbers are num_row=180, num_col=1, row0=-20.738508, mean=-10.065133.
