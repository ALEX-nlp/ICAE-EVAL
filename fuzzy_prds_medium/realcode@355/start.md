## Product Requirement Document

We need a binary encoding and decoding library for communicating with a relational database over its binary wire protocol. On the encoding side: given a typed value — integer, float, text, boolean, or null — produce the exact byte sequence the protocol expects. Integers go out in network byte order at a fixed width based on their type. Floating-point uses the standard 32-bit representation. Null produces zero bytes. For arrays, there's a structured header followed by each element with its size framed before it. Some values need to be self-describing frames that include both the type identifier and the size before the payload — handle framing the same way the parameter-binding layer handles typed nulls, where the type identifier travels alongside the value. On the decoding side: given wire bytes and the expected type, check that the incoming type code is compatible, then extract the value. Report three distinct error categories: wrong type identifier, unexpected null when the field cannot be null, and malformed payload when the byte count doesn't match the type. For sending parameterized queries, the library should assemble the per-parameter metadata: type identifier, format code (always binary for this driver), byte length, and the bytes themselves. Typed nulls send length=-1 and no payload. Finally, the system also needs a bidirectional conversion for compact alphanumeric status codes that encode error numbers in a base-36 scheme.

One small follow-up from the team questions: for the PostgreSQL type ids, please use the exact OID values bool=16, bytea=17, name=19, int8=20, int2=21, int4=23, text=25, text-array=1009, int4-array=1007, and float4 OID=700. Those are the ones we expect to flow through serialize, frame, deserialize, and bind_meta, so the same exact mapping should be used end to end rather than having different assumptions in different paths.

Also, on the array shape, the binary header should be treated as 4 x 32-bit big-endian words: (1) dimension count, (2) data-offset/flags word (0), (3) element type OID, then per dimension: (4) dimension length, (5) lower-bound word (0). After that, each element is just 32-bit big-endian size + payload bytes. That’s the structure we want for both text-array=1009 and int4-array=1007 cases.

And just to make the parameter side completely explicit: parameters are ALWAYS sent in binary format. The format code is always 1 (binary). This applies to all parameter types including text and nulls. So even when the value is text, and even for typed nulls with length=-1 and no payload, bind_meta should still mark them as binary.
