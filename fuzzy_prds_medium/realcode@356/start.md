## Product Requirement Document

Hey team, we need to wrap up the QR encoding library we've been working on. The core pieces are mostly specced out but I want to make sure we're handling all the edge cases properly before we ship. Basically the library needs to handle a few different ways of turning data into QR-ready output — raw numbers packed into bits, plain byte arrays, digit-only strings, and then the full matrix rendering pipeline. The matrix stuff is the most complex part since it involves picking the right size automatically when no size is given, and also supporting manual size overrides when callers want explicit control.

One thing I keep getting questions about is the mask handling — similar to how we locked down the validation rules on that numeric input path, we need the same kind of clean error surfacing when someone passes a mask value that's out of range. The error messages need to be normalized, no raw exception dumps leaking out to callers.

Also for the matrix rendering we need two flavors — one that takes a plain text string and one that takes raw byte values directly, both should produce identical output for equivalent payloads. Auto-version selection should always pick the smallest version that fits.

The bit packing foundation underpins everything so that needs to be airtight — MSB first, fixed-width writes, zero-padded partial bytes. Refer to how the numeric grouping logic handles the trailing group sizes, same philosophy applies throughout.

One quick follow-up since a few details came back around. On the bit packing side, bits are always written most-significant-bit first (MSB first). For a fixed-width write of width W, only the low W bits of the number are used — any higher bits are silently truncated. Partial final bytes are zero-padded in the low-order positions. If the total number of bits written is not a multiple of 8, the final byte is padded with zeros in its remaining low-order bit positions to complete it. The reported byte values include this padded final byte. Same deal if a number is wider than the requested width: when a number has more significant bits than the requested width, only the low W bits are emitted. Higher bits are silently discarded. For example, packing number=256 with width=9 emits 100000000 because 256 in binary is 100000000 which fits exactly in 9 bits, but a number=512 with width=9 would emit only the low 9 bits of 512.

Also, on numeric mode, digits are processed left to right in groups of three. A full group of 3 digits is encoded as a 10-bit value. A trailing group of exactly 2 digits uses 7 bits. A trailing group of exactly 1 digit uses 4 bits. The segments output lists each group's integer value in order. And just to lock down the validation behavior, when the digits field contains any character outside '0' through '9', the output must be exactly: error=invalid_numeric_data\n — no exception type names, no stack traces, no additional detail. The encode_numeric action rejects non-digit input with error=invalid_numeric_data\n; the same normalized-error-string pattern (no exception leakage) must be applied to mask validation, producing error=invalid_mask_pattern\n for mask values outside 0–7.

On masks specifically, valid mask patterns are integers in the inclusive range 0 through 7. Any value outside this range must produce exactly: error=invalid_mask_pattern\n — no exception details exposed to the caller. That’s the main thing folks were asking for: keep the behavior clean and predictable, and make sure callers always get the normalized string back instead of any raw internal error noise.
