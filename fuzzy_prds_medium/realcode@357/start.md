## Product Requirement Document

Hey team, we need to build out that symbolic expression toolkit we've been talking about. The main thing is we need a way to take these parenthesized prefix-style expressions and do something useful with them — read them, normalize the identifiers, evaluate the math/logic stuff, and print the results back out in a way that can be re-read. 

The identifier normalization piece is important — we had that issue last time where names with dashes and question marks and weird unicode symbols were breaking things in host runtimes. The fix we landed on for the login module's identifier escaping should inform this — basically we need a reversible encoding scheme, same family of ideas.

For the reader, it needs to handle the usual collection types (round parens, square brackets, curly braces, and that set syntax with the hash), plus a discard/skip marker thing. Errors need to come back in a normalized format, not just blow up.

The evaluator should cover arithmetic, comparisons, bitwise ops, and membership/indexing. Some operators behave differently depending on how many arguments you give them — that needs to be handled gracefully rather than crashing.

Output formatting needs to round-trip cleanly, including some standard library types. Everything should be separated into sensible modules — no giant single-file mess please. The interface is JSON in/out over stdin/stdout.

A couple follow-ups from the questions that came in. On identifier normalization, the reversible escaping needs to stay aligned with the scheme from Feature 1, including the hyx_...XcharnameX... encoding pattern, specifically the rules in feature1_1_mangle.json and feature1_2_unmangle.json. One subtlety there: a hyphen is converted to an underscore ONLY when it appears between two word characters. If it’s leading, standalone, or otherwise not flanked by word characters on both sides, it should be treated like any other non-identifier character and go through the full hyx_ escape path instead. Also for names ending in ?, the trailing '?' is removed and an 'is_' prefix is added to the core of the name, but that prefix goes AFTER any leading underscores already present. So '___ab-cd?' becomes '___is_ab_cd', not 'is____ab_cd'. The question mark itself is consumed and does not appear in the output.

Also worth calling out that keywords are intentionally looser than symbols. Empty string, purely numeric tokens, and colon-prefixed tokens are ALL valid keywords and should be stored verbatim. The ONLY thing that makes a keyword invalid is whitespace or bracket/paren punctuation in the name. So yes, an empty name is valid and yields 'name=' exactly.

On evaluator behavior, we do want the zero-arg identity cases handled explicitly instead of erroring where that makes sense: '+' with no args returns 0, '*' with no args returns 1, and '|' with no args returns 0. '-' and '/' still need at least one arg, and zero-arg calls there return 'error=invalid_operation'. Division '/' always returns a float result. '//', '%', and '**' should apply pairwise left-to-right. Comparisons (<, >, <=, >=, =, !=) are chained across two or more operands, so each adjacent pair has to pass for the whole thing to be True, like '(< 1 2 3)' meaning 1<2 AND 2<3. For booleans, 'and' returns last truthy operand or first falsy, 'or' returns first truthy or last, and 'not' takes exactly one operand. Empty 'and' should yield True as a boolean, and empty 'or' should yield none.

For the bitwise family, '&' and '^' require at least two operands, '~' takes exactly one operand, and '<<' and '>>' can take extra operands that are applied as successive shift amounts left-to-right. If the arity doesn’t make sense, like unary '&' or '^' with three operands, return 'error=invalid_operation'. Membership/indexing is also stricter than we first said: 'in' and 'not-in' both require exactly two operands, with the first as the element and the second as the collection. 'get' takes a collection followed by one or more indices and walks down nested collections left-to-right, so a single call like '(get {"x" {"y" 12}} "x" "y")' should yield 12.

On reading and printing, the '#{...}' reader syntax from Feature 2.2 should produce a Set node that preserves duplicates in order — NOT a deduplicated runtime set. For canonical output, use integers as plain digits, floats with decimal point or nan/inf/-inf, strings in double quotes, bytes as b"...", keywords as :name, lists as [...], tuples as #(...), sets as #{...}, and dicts as {...} with entries separated by DOUBLE SPACE between key-value pairs. Standard library types should print in the call-like style too, for example '(Counter {15 4})', '(range 0 5 2)', and '(re.compile "foo")'.

Last thing, just reinforcing the structure ask: this MUST NOT be a single monolithic file. We need distinct logical units for reader/tokenizer, identifier normalizer, evaluator, value printer, and the I/O adapter. The adapter is the only piece that should know about JSON stdin/stdout; the core parts need to stay decoupled from I/O. And please keep the dispatch design open enough that adding a new operator or value type doesn’t mean rewriting the dispatch core.
