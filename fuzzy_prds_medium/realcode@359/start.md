## Product Requirement Document

We need a stream pipeline engine that lets developers compose a finite sequence of fallible values through declarative transformation stages and collect the result. Each element in a source sequence is either a successful integer item or an error carrying an integer code. Stages can map values, filter them by predicate, combine filter and map in one step, take or skip leading elements (counting only successful items, not errors), stop or skip while a predicate holds, rewrite error codes, chain fallible transforms on success or on any outcome, recover from errors by substituting a value, and combine two sequences by pairing their elements or by racing to whichever produces first. There is also a batching stage that groups items into fixed-size chunks. The terminal collectors either gather all items into a list, fold items into a single accumulator, or simply drain the sequence. A separate flatten operation concatenates groups together. Handle errors the way the standard transform chain handles them — when an error arrives it stops the collecting sink and that error code is reported, unless a recovery stage intercepts it.

A couple small clarifications from the questions that came in. On the output side, successful items are reported as 'item=<value>\n' lines. If an error makes it through and isn’t recovered, collection stops and that is reported as 'error=<code>\n'. If the collect completes normally, it produces 'end\n'. The drain sink follows that same basic reporting shape for termination, so it reports errors as 'error=<code>\n' and normal completion as 'end\n'. For fold_sum, success comes back as 'value=<result>\n' and failure still comes back as 'error=<code>\n'.

Also, for take(n) and skip(n), the count only advances on SUCCESSFUL items. Error outcomes should still be forwarded, but they do NOT count against the limit. So the intended behavior is that take(2) on [1,err(5),2,3] yields items 1 and 2, the error is forwarded and does not count.

And just to pin down the collector formats for the later features: the zip sink (feature5_1) emits paired results as 'pair=<a>,<b>\n' for each pair, then 'end\n'. The select/race sink (feature5_2) reports winners as 'item=<value>\n' and then 'end\n'. The chunks sink (feature6) reports each batch as 'chunk=<v1>,<v2>,...\n' and then 'end\n'.
