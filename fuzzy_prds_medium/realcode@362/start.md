## Product Requirement Document

hey team, we need a small DOM querying + assertion utility built out. basically the idea is that QA and devs can throw an HTML snippet at it and ask things like 'find the element the user sees as X' or 'is this thing visible' or 'does this have the right styles' without writing fragile CSS selector chains every time. it should work like that accessibility-friendly locator pattern we talked about in the last sprint — you know, find by what the user actually reads, not by div > span:nth-child(3) nonsense. also needs to handle the case where text is split across child nodes (think deeply nested spans) and still match what a human would read. for the text matching part, we want some flexibility — sometimes you care about exact casing and spacing, sometimes you just want 'does it contain this somewhere'. there's also a plain object comparison use case, like asserting a response payload has at least these fields with these values without caring about the rest. the whole thing needs to be driven from a simple JSON command over stdin so our test harness can shell out to it. refer to how we handled the normalization flags in the content-query module from the previous spike for the whitespace rules. output should be line-oriented, human-readable key=value pairs. make sure hidden inputs and ancestor-inherited hiding are both covered in the visibility check.

one more pass on the details the team asked for. the stdin side is just one JSON object per run, and the `op` picks the behavior: `locate`, `locate-content`, `visibility`, `class`, `style`, or `partial`. each one should route straight to its own core method, nothing fancy there.

for `locate`, the `by` field supports exactly six strategies and only these six: `text`, `label`, `placeholder`, `alttext`, `title`, and `displayvalue`. `text` means exact visible text of the element, `label` means text of an associated `<label>` and it should return the control, not the label, `placeholder` means a control's placeholder attribute, `alttext` means an image's alt attribute, `title` means the element's title attribute, and `displayvalue` means a control's current display value. matching for these is always exact against the relevant accessible text. also, when `by` is `label`, the flow is: find the `<label>` whose text matches the query string, then resolve the associated form control either through the label's `for` pointing at the control's `id`, or through a wrapping label that contains the control. again, the returned result is the control element, not the label element. if a `selector` is included on a `locate` request, use that as a pre-filter on candidate elements before doing the accessibility-style match, which is just for narrowing things down when the same text shows up more than once.

on the output shape for found elements, keep it consistent: first `count=<n>`, then one line per result in document order as `match=<tag>#<id>`. `<tag>` is the lowercased tag name, and if there is no `id` attribute then use the literal `(no-id)`, so you can get `match=span#(no-id)`.

for `locate-content`, the normalization flags are `trim` and `collapseWhitespace`, both default `true`, same whitespace behavior we referenced before. those control whether leading/trailing whitespace and internal whitespace runs are normalized before matching.

for `partial`, the check is only that every key in `expected` exists in `actual` with a strictly equal value using `===`. this is not deep or recursive for nested objects, just top-level property values. missing keys fail, different values fail. output starts with `pass=<bool>`, then for each expected key in declaration order emit either `<key>=match` or `<key>=mismatch`.

and for `visibility`, the output is exactly three lines and in this order: `selector=<the CSS selector as given>`, then `visible=<true|false>`, then `hidden=<true|false>`. `visible` and `hidden` should always be complementary booleans, and the selector should be echoed back verbatim from the input.
