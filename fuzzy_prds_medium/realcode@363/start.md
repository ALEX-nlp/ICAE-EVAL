## Product Requirement Document

Hey team, we need to build out the random forest engine we've been talking about. The core idea is that a developer should be able to point it at a dataset, say which column is the outcome they care about, and just get a trained model back without doing a bunch of manual setup work. We want to support the usual stuff — classifying categories, predicting numbers, and also that survival analysis thing we discussed (like what we did in the clinical data project last quarter, similar Surv() syntax). There should be a way to describe the problem either with a formula string or by just naming the column directly, both should work the same way. Data can come in as a typed table or as the all-numbers version of the same data, and the engine shouldn't care which one it gets. We also need proper error handling — not just crashes, but meaningful category labels for things like bad sampling settings, wrong split rules for the task type, missing values showing up, that kind of thing. Reproducibility is important so make sure seeding works correctly. We want OOB error, variable importance when asked for, confusion matrix for classification, probability outputs, per-tree prediction shapes, and survival timepoint tracking. Oh and warnings when something like unused categories get dropped during training. The adapter layer should be thin and separate from the actual engine logic — keep I/O concerns out of the core.

One extra note on the formula side since folks asked: if the left-hand side wraps the response in a user function, like transform(Species) ~ ., that still needs to resolve the same way as the plain version. We should peel out the inner column name, look that up in the data, and infer the task from that column’s type. For predictors, same behavior as the unwrapped form: every other column in table order. The wrapper function name itself is ignored for task inference and predictor resolution. For survival, the left-hand side is Surv(time_col, status_col) ~ . and both named columns are part of the response, not predictors. So the dot means every remaining column after removing both of those, and the inferred task is 'survival'. If removing time and status leaves us with nothing to train on, reject it before training with 'no_covariates', whether the user came in through a formula or named columns.

A couple behavior clarifications around classes and outputs too. If the training subset is missing one or more response levels that do exist in the full dataset, that should be a warning, not a failure, with category 'dropped_unused_response_levels' and a 'levels' field listing the dropped class names. The model still trains on the classes it has, probability estimation uses the surviving class count, and predictions still cover the full original class set, so prediction_levels should list all original classes. Also, if someone asks for importance on a model that was not trained with importance turned on, return error category 'importance_not_available'. That has to be explicitly requested up front, like importance='impurity'. When it is enabled, return one numeric importance value per predictor, keyed by predictor name, in predictor order.

On a few error cases, let’s be strict early. If the user’s mtry function blows up or returns something unusable while we evaluate it, stop training and emit 'mtry_function_error' with no partial model. If someone requests a prediction output type we do not recognize, like 'class' when that is not a supported type string, reject it with 'invalid_prediction_type' before doing any prediction work. And for anything survival-only like timepoints, cumulative hazard, or survival function, if the model’s inferred task is not survival, return 'not_a_survival_forest' for those accessors.

Last thing on data sources and the survival reference case. A matrix-backed source can’t tell classification from column types alone because everything is numeric, so for classification or probability on that kind of source the task has to be declared explicitly, like a classification_matrix flag or an explicit task parameter. Regression and survival can still be inferred from numeric types. Predictor resolution should still just take every non-response column in table order, same as a typed table source. For the survival reference, we’re talking right-censored survival analysis on the veteran dataset with columns trt, celltype, time, status, karno, diagtime, age, prior. It has 137 rows, status uses 1=event, 0=censored, and using the unique sorted event times where status=1 gives 101 distinct timepoints.
