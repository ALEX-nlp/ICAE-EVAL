## Product Requirement Document

Hey team, we need to build out the error monitoring adapter library for our Ruby web apps. Basically the idea is that right now every team is manually wiring exception handling into their endpoints and it's a mess — different teams produce different payloads, some are leaking user IPs and cookies when they shouldn't be, and we're missing traces on a bunch of routes. We want a single library that wraps both our generic rack-style apps and our routed Rails-style apps and handles all this automatically.

The library needs to parse our monitoring DSN format (similar to what we discussed in the infra meeting last quarter), figure out the real client IP from proxy headers the same way the login module does it, pull request correlation IDs from the environment, sanitize request snapshots based on whether we're allowed to send personal data, and attach all that context to outgoing events. It also needs to wrap rack apps to catch exceptions and do transaction tracing, and integrate with the Rails request cycle for route-aware error reporting including manual captures inside controllers.

One thing that's been biting us is the privacy rules around cookies, IP headers, and request bodies — we need those to follow the pii flag strictly. Also the transaction naming for Rails errors needs to use the controller#action format. Can someone spec out the implementation? We should follow the same pattern we used for the last integration adapter project.

Quick follow-up from the questions that came in: on the DSN side, we should break it out into project_id, public_key, secret_key, scheme, host, port, path, canonical, and envelope_endpoint. project_id is the last path segment, public_key is the username, secret_key is the password, path is everything before the project_id segment, and canonical is just the original DSN string untouched. The DSN format is http://{public_key}:{secret_key}@{host}:{port}/{path}/{project_id}. For envelope_endpoint, use {base_path}/api/{project_id}/envelope/ and keep the trailing slash. base_path is just the DSN path minus the project_id segment. So for /ingest/42, base_path=/ingest, so envelope_endpoint=/ingest/api/42/envelope/. Same example written end to end: http://12345:67890@telemetry.example.com:3000/ingest/42 yields envelope_endpoint=/ingest/api/42/envelope/

Also, on client IP selection, the intended behavior is to prefer an explicit real/client address field if present and valid. If that’s not there, parse the X-Forwarded-For chain and select the oldest valid non-local address, which is the rightmost-trusted / leftmost-original one. We should skip private/loopback addresses while doing that. If every candidate is local/private, then fall back to the direct remote_addr value. That’s the same basic expectation here too: select oldest valid non-local address from X-Forwarded-For chain, falling back to remote_addr.

For request IDs, read HTTP_X_REQUEST_ID from the environment map first, then fall back to a framework-provided request-id field. If neither exists, use an empty string. The output there should stay a single line in the form request_id={value}.

One other detail on exception output formatting: all host-language exception class names like RuntimeError and StandardError should be normalized to application_exception anywhere we print them in stdout fields, specifically error= and exception_category=. We still keep the actual exception text exactly as-is in exception_message=.

And just to tighten up the privacy behavior: when send_default_pii is false, Cookie headers stay out of the headers map, cookies field is empty, request body/data is omitted, IP-related headers like X-Forwarded-For and REMOTE_ADDR are excluded, and user_ip is empty. When send_default_pii is true, cookies are included as a parsed object, form data is parsed into a structured data object, X-Forwarded-For header is included, REMOTE_ADDR is added to env, and user_ip is set to the selected client IP.
