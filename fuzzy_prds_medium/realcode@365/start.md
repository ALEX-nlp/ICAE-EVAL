## Product Requirement Document

We need a lightweight in-memory data filtering and transformation service that our testing teams can use without standing up any external infrastructure. The core idea is simple: developers should be able to ask the service to scan a list of records and return only those that match a given set of criteria, apply changes to a single record and get the updated version back, or reorder a mixed list of values according to a specified direction. Matching needs to support the usual range checks, set membership tests, structural checks (like whether a field exists at all, or an array has a certain length), pattern-based string matching, nested field access using dotted paths, and combinations of all of the above using logical connectives. Modifications should handle assigning and removing fields, moving values between paths, numeric increments, adding and removing items from lists, and bitwise adjustments to integer fields; for targeting a specific position inside a list, the service should look at the filter that was used to find the record in the first place, similar to how the data-access layer in the auth module figures out which row to lock. When sorting a mixed bag of values that includes numbers, text, booleans, timestamps, identifiers, and the special boundary markers, the service must follow the same cross-type ranking that our document storage layer uses. Invalid or conflicting instructions should produce a concise machine-readable error token rather than an exception trace so downstream tools can handle them cleanly.

A couple extra details from the follow-up questions. For sorting, the fixed ascending order is MinKey < null < numbers (Integer/Long/Double/Float compared numerically) < String < embedded document (BasicDBObject) < array (BasicDBList) < binary/byte[] < ObjectId < Boolean (false before true) < Date < Pattern/regex < MaxKey. Numbers of different kinds (int, long, double, float) are compared by their numeric value using BigDecimal. Also, embedded documents that contain the sort field are placed after all non-document values and ordered by that field's value. If direction is reversed with -1, reverse the entire resulting order, including where those sort-field-bearing documents land. For the boundary markers, MinKey is represented in the JSON wire format as {"$minKey": 1} and sorts before all other types (weight Integer.MIN_VALUE), and MaxKey is represented as {"$maxKey": 1} and sorts after all other types (weight Integer.MAX_VALUE). In sort output they are rendered as {"$minKey": 1} and {"$maxKey": 1} respectively.

On matching, there were a few edge cases worth spelling out. The $ne operator keeps documents where the field is absent (not present in the document at all), but with the same null behavior we use elsewhere: when the stored field path resolves to an empty list (field missing), $ne returns true if and only if the query value is non-null. If the query value is null, an absent field does NOT satisfy $ne. Also, if the stored field is an array and it has at least one element equal to the query value, then $ne returns false. For direct equality when the query value itself is an array, like {"a": [1,2,3]}, a document matches if either the field 'a' is an array equal element-by-element to [1,2,3], or the field 'a' is a non-array value equal to one of the array elements. If the query value is an embedded document, then the stored field has to be an equal document. Equality for typed values like ObjectIds uses value identity.

One other thing the team asked about is how null behaves inside $in and $nin. When a null entry appears in the $in list, documents where the queried field is absent are also matched. Similarly, $nin with a null entry in the list will exclude documents where the field is absent, because absent matches null. That null-matches-absent rule applies to $in and $nin but not to comparison operators like $gte/$lte.

And on updates, if the update spec has no $-prefixed operator keys at all, treat it as a full document replacement. In that case the engine removes all fields from the existing document except _id, then adds all fields from the update specification. The _id field is always preserved from the original document and cannot be replaced.
