## Product Requirement Document

Hey team, we need to build out that static analysis tool we've been talking about for the Angular codebase. Basically devs keep shipping components and directives with inconsistent naming, sketchy event bindings, and gnarly template logic that nobody catches in review. It's causing real pain in prod — we had that incident last sprint where a custom event shadowed a browser event and broke the checkout flow on Safari.

The tool should read source code, run one of the named checks against it, and spit out a structured report so CI can fail on violations. A few of the checks also need to be able to auto-fix the source when asked — similar to how we handled the auto-correct flow in that i18n annotation pass we did a while back, just apply the same sentinel-block pattern.

Input comes in over stdin as a JSON blob, output goes to stdout. Line/column positions in findings need to be consistent so golden-file diffing works. The whole thing should be modular — each check lives independently so we can add more later without touching existing ones. Need to handle bad input gracefully too, no raw stack traces leaking out. We're thinking Python but open to discussion. Can someone scope this out and get it running against the test cases in the repo? The test files lay out the exact scenarios we care about.

One small follow-up from the questions that came in: all line and column positions in findings are 0-based, so the first line is line 0 and the first character on a line is character 0. Also for the component/directive selector checks, the span should point at the selector value only, not the whole `selector: '...'` bit. So if you’ve got `selector: 'foo-bar'`, the range starts on the `f` and ends right before the closing quote. The concrete example to follow is `start_character=12, end_character=21` for `selector: 'foo-bar'`.

On fixes, when `apply_fix` is true and there’s at least one finding with a mechanical fix, we still print all the failure blocks first and then append the rewritten source using the sentinel block format. The exact shape is `fixed_source<<EOF\n<rewritten source>\nEOF\n`, so literally a line `fixed_source<<EOF`, then the rewritten source text, then a line `EOF`. Same pattern as the earlier sentinel-block flow, and that exact contract is what’s shown in the feature7_async_boolean_expression.json test cases.

A couple config details too: `max_cost` defaults to 5 if nothing is set, which is why the first test case with a 3-operator expression passes by default. For that complexity check, the cost is based on logical operators (`||`, `&&`) and comparison operators (`===`, `!==`, `==`, `!=`) inside the `*ngIf` expression. If the cost goes over the limit, the message needs to be exactly `The condition complexity (cost '<N>') exceeded the defined limit (cost '<max>'). The conditional expression should be moved into the component.`

For the modularity piece, each check should hang off the same abstract interface, like `run(source, config) -> list[Finding]`, and the engine just routes by check name to the right implementation. Adding a new check should just mean registering a new class, without changing existing checks or the engine core, and the adapter layer should only know about the abstract `Finding` type, not concrete check classes.

Last couple bits on input/output behavior: `prefixes` can be either a single string like `"sg"` or a list like `["fo", "mg", "sg"]`, and if it’s a list then matching any one of them counts as a pass. For `pipe_name` messages, if there are multiple prefixes they should be joined with commas and no spaces. And for command handling, bad JSON must return exactly `error=invalid_json`, a non-`lint` action must return exactly `error=unsupported_action`, and any internal failure must return exactly `error=execution_error`. We should never leak exception names or stack traces to stdout.
