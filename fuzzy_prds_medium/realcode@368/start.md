## Product Requirement Document

Hey team, we've been getting complaints from a few integrators who are hand-rolling their own InfluxDB write logic and keep running into data corruption bugs — mostly around special characters in measurement names and tags, and weirdly formatted timestamps. We want to build out a proper client adapter library that handles all of this cleanly. It should cover the serialization side (you know, that line protocol thing InfluxDB uses), plus a query builder so people stop concatenating strings directly into queries, and an HTTP write layer that handles the actual sending. Oh, and wire up logging so we can observe what's happening internally — similar to how we did it in that transport abstraction layer from the last sprint, where we just delegated calls through to a caller-provided handler rather than owning the log output ourselves.

A few things I'm not 100% sure about: for the write layer, we had a discussion about compression but I don't remember what threshold we landed on, or whether retries were supposed to be configurable or fixed. Also for the query builder, there was something about how undefined values should behave differently from null — I just can't remember the exact distinction we agreed on. And for timestamps, I think there were some edge cases around non-standard formats that someone on the backend team had opinions about. Can we make sure all that's nailed down before we ship? The integration tests should tell us if we got it right.

Quick follow-up after the team’s questions: on the line protocol side, we do need to be strict about escaping. Spaces and commas in measurement names get escaped with a backslash, so 'meas, 2' becomes 'meas\,\ 2', and tags follow that same rule. For field string values, backslashes and double-quote characters both need escaping, with backslash becoming double-backslash and double-quote becoming backslash-double-quote, and then the whole string value stays wrapped in double quotes. Also, both tag keys and field keys need to be sorted alphabetically in the final line protocol output. Tags sit between the measurement name and the field set, sorted ascending by key, and fields are sorted ascending by key too. That still applies after merging default tags with explicit tags.

For writes, the route is '/api/v2/write?org=<org>&bucket=<bucket>&precision=<precision>'. The authorization header defaults to 'Token <token>'. If a headers map is provided in the write configuration and includes an 'authorization' key, that value overrides the default 'Token <token>' header derived from the server token. The route and body format are unaffected by custom headers. On success handling, only HTTP status 204 (No Content) counts as a successful write. Anything else is treated as a write rejection and should produce output starting with 'error=write_rejected\nstatus_code=<code>\n'. In those cases, failed_lines is the number of lines in the rejected batch and success_lines remains 0. Also, request_count in output should reflect total HTTP requests made including retries.

On compression, the behavior we wanted was tied directly to gzip_threshold in write settings. If that is configured, request bodies at or above that byte threshold are compressed with gzip and the content_encoding header reflects 'gzip'. If no threshold is set or the body is below the threshold, content_encoding is 'identity'. Important detail there: the observable output body is always the decompressed line protocol text.

One more write-layer thing: once a writer is closed, any attempt to enqueue records or points needs to fail immediately with 'error=writer_closed\noperation=<operation_name>\n'. The operation names are exactly: 'single_record', 'multiple_records', 'single_point'. In that state, no data should be written and no HTTP request should be made.

For logging, this is still the same pattern from the earlier transport work: the logger must delegate to a caller-supplied handler without modifying the message or error payload. The logger feature (Feature 4) where warn and error calls are forwarded unchanged to a caller-supplied logger callback, as described in feature4_logger_delegation.json — the library never owns log output, it only delegates with the original message and optional error argument. The output format is: 'level=<level>\nmessage=<message>\nerror=<error_or_empty>\nargument_count=2\n'. For error-level calls with both a message and an error object, argument_count is 2. For warn-level calls with only a message (no error), argument_count is still 2 — the second argument is passed as an empty/undefined value.

And on the query side, the null vs undefined distinction we were trying to remember is this: in automatic query value conversion (feature2_2_query_value_conversion.json), encoded null produces literal 'null' fragment while encoded undefined produces an empty fragment string; additionally in query templates (feature2_4_query_template_errors.json) an undefined-encoded interpolation value triggers error=missing_interpolation_value. For regex values, use the constructor syntax 'regexp.compile(v: "/pattern/")'. With explicit type 'regular_expression', the value is wrapped directly. In automatic conversion mode, an object with encoded type 'regexp' and a 'pattern' field is converted using the same constructor syntax with the pattern wrapped in forward slashes.
