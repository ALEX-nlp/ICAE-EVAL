## Product Requirement Document

Hey team, we need to wire up the recurring billing adapter properly. Right now devs are spending way too much time hand-rolling payment state glue code every time we add a new billing flow. The big ask is to get the adapter reading JSON from stdin and spitting clean key=value lines to stdout so our black-box test harness can validate everything end-to-end.

A few things that came up in the last sprint retro: money handling keeps breaking when people pass amounts in different formats — same problem we had with that currency rounding thing in the checkout module a while back, so make sure that's consistent. Also the coupon discount behaviour for first payments needs to reflect as a negative line item, not just a reduced total. The webhook side is particularly fragile right now — both the order-payment path and the first-payment path need to go through the proper HTTP controller/route and return the right HTTP status alongside all the state changes.

For the subscription checkout payload, trial logic needs to be baked in — someone on the gateway team mentioned the sequence type and locale fields keep getting dropped. Collection-level stuff (currency grouping, totals, error handling for mixed currencies) also needs to be solid.

Basically: clean separation between the core domain and the I/O adapter layer, no god files, and make sure edge cases like lookup failures on webhooks behave differently depending on the debug flag. Ping me if anything is unclear.

One extra pass on the billing details since a few people asked the same follow-ups. For the initial payment coupon case, the fixed discount is always 5% of the unit_price_minor_units. So if the unit price is 10000 minor units, the discount line should be -500 minor units. That first-payment discount line item always has zero tax, full stop: tax_percentage=0 and tax_minor_units=0. The tax is always zero there, there is no follow-up payload, and the execute_count is 0.

Also, for line item processing state, keep it really literal: a line item is processed if and only if its order_id is not null. processed_when_true reflects is_processed(), processed_when_false reflects is_unprocessed() (the logical inverse), and nothing else should influence that. Same idea for the counts: processed_count = items where order_id is not null. unprocessed_count = items where order_id is null. should_process_count = unprocessed items where process_at is in the past (past timestamp). due_count = all items where process_at is in the past regardless of processed state.

On collection currency behaviour, list_currencies returns unique currencies in collection order, meaning first-seen order, as a JSON array string, e.g., ["USD","EUR"]. single_currency returns currency=<VALUE> only when all items share the same currency. If somebody asks for single_currency on a mixed-currency collection, print error=multiple_currencies and currencies=[...] with the unique currencies in collection order. Don’t throw anything to stdout. While we’re here, collection.grouping with operation=where_currency should print count= for the number of matching items and descriptions= as a JSON array of descriptions of matching items in order, and only include items matching the specified currency.

For totals and amount math, collection.total is the sum of the total_minor_units from each line item, with each line already rounded individually. Example stays: two items of unit_price=12150 at qty=1 and qty=2 with 10% tax gives item1=12150+1215=13365, item2=24300+2430=26730, total=40095. The rounding rule in feature2_order_item_amounts is tax = round(subtotal * tax_percentage / 100) using standard PHP round(), as exercised in rcb_tests/public_test_cases/feature2_order_item_amounts.json and the OrderItemAmounts domain class.

One more payment creation detail from the gateway path: payment.create_from_gateway always initialises refunded_minor_units=0 and charged_back_minor_units=0, and owner_linked=true always. The amount comes from the gateway amount object, meaning currency+value decimal string, and mandate_id should be preserved from the gateway payload.

And on the webhook lookup path, the normalisation needs to match feature12_webhook_lookup.json exactly, especially the split between debug=true and debug=false in the WebhookController/PaymentLookup domain service. In debug=true, lookup failure comes back as gateway_lookup_failed. In debug=false, that same kind of miss should come back as payment_found=false.
