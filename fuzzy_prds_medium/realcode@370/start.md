## Product Requirement Document

Hey team, we need to build out a math utility library for working with rotations and rigid body transforms — think the kind of stuff robotics and computer vision folks need constantly. The core idea is that developers should be able to plug in a transformation (like a rotation angle or a set of angles for 3D), do stuff with it (combine two transforms, undo one, find the 'difference' between two poses), and also nudge a transform by a small local correction from either side. There should also be a way to move a point using a transform, get the linear map that converts between left/right tangent representations, and do smooth interpolation between two poses.

We also need all the derivative/Jacobian stuff baked in — basically every operation should optionally return its analytic gradients, since downstream optimization code depends on this heavily.

The tricky part is the input/output format — I remember we discussed something similar to how the login module handled its coordinate conventions, but I don't recall the exact agreed-upon format for quaternions or how SE3 parameters are ordered. Also the number formatting had some special rule about near-zero values that I think got documented somewhere. The system should cover four groups: planar rotation, 3D rotation, planar rigid motion, and spatial rigid motion. JSON in, human-readable text out. Please make sure the architecture is clean and not just one giant file.

One quick follow-up since a few details came up in questions. For the coordinate conventions, please use the rules documented in start.md under 'Coordinate conventions (shared by all features)', specifically the input construction parameter ordering and canonical output vector format for each of the four groups (SO2, SO3, SE2, SE3). The main gotcha is SO3: the quaternion is stored with the three vector (imaginary) components first, then the scalar (real) component last, so the canonical output for SO3 is [qx, qy, qz, qw], not the [qw, qx, qy, qz] convention used in some other libraries. Along the same lines, SO2 emits (cos θ, sin θ). SO3 emits the unit quaternion as (qx, qy, qz, qw). SE2 emits (tx, ty, cos θ, sin θ). SE3 emits (tx, ty, tz, qx, qy, qz, qw). The rotation part of SE2/SE3 output follows the same pattern as SO2/SO3 respectively.

Also confirming the “natural” construction inputs. For SO3, the natural construction parameters are three successive axis-rotation angles: first a rotation about the X axis, then Y, then Z. The resulting rotation matrix is Rz(third) · Ry(second) · Rx(first). The same convention applies to the rotation part of SE3 inputs. For SE2 the natural construction parameters are: translation_x, translation_y, angle (in that order). For SE3 they are: translation_x, translation_y, translation_z, rotation_x_angle, rotation_y_angle, rotation_z_angle. Translation components always come before rotation components in the input.

On jac_act, when those Jacobians are returned, jacobian_group has shape (point_dimension × dof): e.g. for SO2 acting on 2D it is 2×1, for SE3 acting on 3D it is 3×6. jacobian_point has shape (point_dimension × point_dimension): e.g. 2×2 for planar groups, 3×3 for spatial groups. Both follow the standard matrix output format.

And just to make the implementation expectation explicit: this has multiple distinct responsibilities now — I/O routing, group math, output formatting, Jacobian computation — so it must NOT be a single monolithic file. The solution must use a multi-file structure (e.g., src/ with separate modules for each group, an adapter for JSON I/O, and a formatter) following SOLID principles. A single-file 'god file' solution is explicitly prohibited.
