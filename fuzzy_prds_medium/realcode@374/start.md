## Product Requirement Document

Hey team, we need to build out that file transfer task modeling library we've been talking about. The core idea is that any transfer — whether it's a simple download, an upload with multiple files, or one of those parallel mirror downloads — should be representable as a single self-contained object that can be saved, sent over the wire, and reconstructed perfectly on the other side without losing anything. Think of it like what we did with the session state objects in the login module last quarter, same round-trip guarantee idea.

The tricky parts are: (1) when someone builds a task, we need to catch bad inputs early rather than letting garbage propagate — things like putting folder info in the wrong field are a classic support ticket generator, (2) the progress display stuff needs to handle the speed and time-left formatting consistently because right now every team does it differently and users see wildly inconsistent UIs, and (3) when a big file gets split across multiple download sources we need a clean way to figure out which piece covers which byte range and then roll all the child statuses back up to a single parent status. Users keep reporting that large downloads show 'stuck' progress because the aggregation logic is broken in some clients. Please make sure the serialization is tolerant of slight variations in how numbers come in from different platforms — that's been a recurring pain point.

A couple follow-ups from the questions that came in: for the display formatting, the network speed string should use MB/s when the speed value is >= 1.0 MB/s, rendered as a whole number like '5 MB/s'. If the speed is between 0 and 1 MB/s, switch to kB/s, multiply by 1000, and round to a whole number like '200 kB/s'. If no speed value is available because it was not supplied or is negative, show the fixed placeholder '-- MB/s'. Same idea for remaining time: use zero-padded MM:SS when fewer than 3600 seconds remain, like '04:30', and switch to H:MM:SS once the value is >= 3600 seconds, with hours not zero-padded and minutes and seconds zero-padded, like '1:02:03'. If no time value is available because it was not supplied or is negative, show '--:--'.

On the multi-file upload side, each file entry in a MultiUploadTask can come in one of three shapes. It can be a bare filename string, in which case the field name is derived as the filename without its extension and the mime type is inferred from the extension. It can be a two-element array [fieldName, filename], where the mime type is still inferred from the extension. Or it can be a three-element array [fieldName, filename, mimeType], where mime type is taken verbatim. If the extension is unknown or absent, default the mime type to 'application/octet-stream'.

For deserialization, we do need to stay forgiving about wire differences. Numeric fields may arrive as either integers or floating-point numbers, so something like retries may be 5 or 5.0 and should behave the same. Extra fields that show up in some formats, including 'chunks', 'urls', 'fields', 'fileField', 'mimeType' on a plain DownloadTask, should be silently ignored. Also, task fields may either be nested under a 'task' key or flattened into the top-level map, and both forms should deserialize identically. This is part of the same lossless round-trip expectation from Feature 3: every domain object needs to deserialize from its wire map and re-serialize to a byte-identical canonical JSON string while still tolerating integer/float variance and extra fields.

One other subtlety on copying: when a task is copied with no overrides, meaning copyWith=true with all original values passed, retriesRemaining must equal the retries value supplied and must not reset to zero. Also, all fields need to be preserved exactly as provided, including taskId, creationTime, headers, metaData, group, updates, requiresWiFi, allowPause, baseDirectory, directory, httpRequestMethod, post.

And just to make the validation behavior explicit since this came up twice: a filename must be rejected at construction time if it contains a path separator, meaning a '/' character. The error should surface as the single-line string 'error=filename_has_path_separator'. The directory must be supplied separately via the directory field. Likewise, a directory must be rejected if it is an absolute path, which for us means it begins with '/'. That should surface as 'error=directory_not_relative'. An empty string, which is the default when omitted, and any relative path are both accepted as-is. This matches Feature 2.3 and Feature 2.4 exactly.
