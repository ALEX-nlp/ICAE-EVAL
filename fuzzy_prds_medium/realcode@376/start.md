## Product Requirement Document

Hey team, we need a telemetry runtime support library for our mobile SDK. Basically the problem is that right now every team that wants to add telemetry has to reinvent the wheel — they manually track when the app goes to background/foreground, they lose signals that were emitted before the exporter is even ready, and retry logic is all over the place. It's a mess and we keep getting bug reports about missing session data and crashes on cold start.

The library needs to handle session lifecycle (you know, like that rotation logic we discussed, similar to what we did for the login token refresh), buffering signals before the real exporter connects, writing batches to disk so we can retry them, managing storage paths cleanly, and some kind of periodic background worker. Oh also we need log enrichment — attaching context like which screen the user is on or what session they're in. And there's something about a frame-rate monitoring config that the mobile performance team asked for, they mentioned a polling interval but I don't have the exact spec for what counts as invalid.

Please make sure the code is well structured, not one giant file. We'll want tests too. Reach out if anything is unclear but ideally just look at how we handled similar buffering/retry patterns in the existing codebase.

One quick follow-up after the team’s questions. On the session side, while the app is backgrounded, a session times out when elapsed time since the latest bump reaches the configured timeout_ms, and the check operation should report timed_out=true right at that threshold. If the user bumps activity while still backgrounded, that resets the timeout counter, so a later check should come back timed_out=false. Also, if the app is already in foreground state and has not gone through a background/foreground transition, check operations must never report timed_out=true no matter how much time has elapsed or advanced. This is also meant to follow the session timeout evaluation behavior we talked about in Feature 1, including the background/foreground transition rules and the first-foreground-check-still-observes-background-window behavior. And for the rotation behavior in Feature 2, rotate when lifetime_ms is reached or when the timeout_sequence source returns true, producing a new 32-char lowercase hex ID.

For storage, signals_dir should return a custom signal directory when one is explicitly supplied. Otherwise use the default path <cache_dir>/opentelemetry/signals, and create that directory if it does not already exist. Also, the per-signal folder quota is just integer division of the total configured cache size by 3, so for example 10MB becomes floor(10MB / 3) per signal type.

On the periodic worker behavior, an attempt only actually runs if the current time is at least the previous execution time plus the minimum_delay_ms. If that delay has not been met, skip the run, times_run does not increment, but still record an enqueue. The first attempt is always eligible immediately. Also, if stop_after_run is true on an attempt that really does execute, it should not enqueue itself for another loop, so enqueued count does not increment for that run. If stop_after_run is false after execution, then it enqueues itself again as usual.

For enrichment, global enrichment should read from the supplier fresh on every emit, not cache some earlier snapshot. The output should report emit_index and the complete attribute set in key-sorted key=value form for each set provided. And just to be explicit on the keys, the session enrichment processor uses 'session.id' and the screen enrichment processor uses 'screen.name'.

Last thing, the slow-rendering poll interval config is stricter than I originally wrote down: only positive values (> 0) are accepted and replace the current interval. Zero and negative values are rejected and leave the current interval unchanged. The default interval is 1000 milliseconds before any updates are applied.
