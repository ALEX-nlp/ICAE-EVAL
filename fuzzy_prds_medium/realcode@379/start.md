## Product Requirement Document

We need a text rendering engine that accepts wiki-style markup and produces the fully resolved output string. The engine takes a piece of source text, an optional page title (which defaults to a standard placeholder name), and an optional list of supporting pages, each with a title and a body. Pages whose titles use a designated template namespace prefix can be transcluded by name inside the source text. The engine must substitute positional and named arguments in template bodies, support nested template calls, and handle special punctuation injection helpers. Conditional branches must evaluate based on emptiness, equality, multi-way matching, and numeric expression results. Inline error detection in already-expanded sub-values must be supported. Arithmetic expressions including trigonometric functions, constants, scientific notation, comparison operators, and integer division must be computed. String operations including substring, search, pad, replace, and split are required, along with letter-case conversion and number formatting with grouping separators. URL percent-encoding and decoding, page title path slicing, and namespace identifier resolution must be implemented. Page-title introspection and date formatting with format codes are also required. Error conditions must surface as inline HTML-style markers in the output, never as runtime exceptions. The input-output adapter layer that reads requests and writes results must follow the same separation-of-concerns pattern used in the authentication module, remaining strictly decoupled from the core engine logic.

A couple of extra specifics from the follow-up questions. If the request doesn’t include a title, the default page title is "Tt". For transclusion, a page is considered a template when its title starts with "Template:" and that maps to namespace_id 10. On the registration side, the dispatcher should call ctx.add_page(title, namespace_id, body, model=model) for normal pages and ctx.add_page(title, namespace_id, redirect_to=body) for redirect model pages, and then ctx.analyze_templates() must be called after all pages are registered before expansion begins.

Also, if someone calls a template that was never registered, we don’t blow up or try to keep its args around. We replace it inline with the exact marker <strong class="error">Template:NAME</strong>, where NAME is the requested template name, and any arguments passed to that unknown template are discarded.

A few parser behavior details too: for a triple-brace arg like {{{1}}}, if no argument was passed and there’s no default, it should stay literal in output as "{{{1}}}". But {{{1|}}} is different: with an empty default, it becomes the empty string when that argument is missing. For #if, a whitespace-only test like " " counts as empty, so it should go to the else side. For #ifeq, trim surrounding whitespace on both operands before comparing, and if both end up empty they count as equal. For the inline error check, it only counts as an error if the first argument’s expanded value actually contains an HTML tag with class="error". Just mentioning the word error somewhere, or having unrelated HTML, does not count. Plain text like 'foo<div>bar</div>' does not trigger the error branch.

On #pad, if direction is omitted, default to left padding. The accepted direction values are "left", "right", or "center", and the padding string is repeated and truncated as needed to fill the gap. And for formatnum, the normal behavior is grouping digits to the left of the decimal into threes with commas, while leaving the fractional part alone, so 987654321.654321 -> 987,654,321.654321. The NOSEP flag means don’t insert separators, and the R flag means reverse it by stripping existing grouping separators back out.
