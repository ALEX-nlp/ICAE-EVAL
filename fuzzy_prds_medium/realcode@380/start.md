## Product Requirement Document

Hey team, we need to build out that physics toolkit we've been talking about — the one that lets devs work with collision shapes, bodies, and joints without writing all the low-level math themselves. Think of it like what we did with the rendering abstraction layer last quarter, but for 2D rigid-body simulation stuff.

Basically devs should be able to create circular and rectangular collision shapes and ask them geometry questions (size, boundaries, how far a point is from the edge, that kind of thing). Bodies should have some flags and gravity behavior — there was a complaint from the mobile team that fast-moving objects were clipping through walls, so we need a way to handle that. The world itself should track gravity, and bodies should be able to opt out or scale down how much they're affected.

Contacts between fixtures need a simple on/off toggle. Constraints between bodies need a full lifecycle — create, query, destroy — and when a body goes away its connections should clean up automatically. Some constraint types have motor settings that should wake up the attached bodies when changed. Each constraint type also needs to emit some kind of debug lines so devs can visualize what's connected. There's also a coupled-constraint type that combines two joints and should gracefully reject invalid combinations.

Output formatting should follow the same pattern we used in the last simulation module — two decimal places throughout. Let me know if anything is unclear.

Quick follow-up from the questions that came in: for the circle shape, we do want it to expose both its radius and center directly, and if someone copies it the copy should keep the exact same values for both. The formatting expectation is still the same everywhere, so examples should come out like radius=5.00, center=1.00,2.00, copy_radius=5.00, copy_center=1.00,2.00. More generally, all numeric values in output are formatted to exactly two decimal places, like 1.00, -3.00, 0.87, and if floating-point math naturally gives negative zero we should keep that as -0.00. Vectors should be X.XX,Y.XX with no spaces, and if we output lists of vectors like vertices or normals those should be semicolon-separated.

Also, on the circle point-distance behavior, the ask is a signed separation distance from the transformed boundary to the query point, plus an outward unit normal. Negative distance means the point is inside. The transform has both a translation vector and a rotation angle, and the order matters: rotate the local center by the transform angle using the usual 2D rotation matrix, then add the translation to get the world-space circle center. From there, distance is computed from that world center to the query point, minus the radius, and the normal is the normalized vector from world center to query point. Output should look like distance=X.XX, normal=X.XX,X.XX, and yes, the normal can include -0.00.

For the box shape, an axis-aligned box created from half-extents (hw, hh) should produce exactly four vertices in counter-clockwise order: (-hw,-hh), (hw,-hh), (hw,hh), (-hw,hh). The matching outward edge normals should be (0,-1), (1,0), (0,1), (-1,0). When copied, the box should also expose a skin radius of 0.01. So the output there is vertices and normals semicolon-separated, then copy_radius=0.01, then copy_vertices and copy_normals.

On the joint/debug rendering side, each family has a fixed segment count and points and circles stay at 0 across the board. The counts are distance_link=1 segment, surface_friction=1 segment, linear_motor=3 segments, pointer_target=0 segments, slider=3 segments, pulley=3 segments, hinge=3 segments, rope_limit=3 segments, weld=3 segments, wheel=3 segments, constant_area=0 segments. The output format should be segments=N, points=0, circles=0. One extra detail for the rope-style limiter in the rope_limit / rope_defaults scenario: both local endpoint anchors start at the origin, so that should come out as anchor_a=0.00,0.00, anchor_b=0.00,0.00.

For the coupled joint type, the only valid source pairs are hinge+hinge, hinge+slider, slider+hinge, slider+slider. For valid combinations, return created=yes and then either segments=3 for the render check or velocity_setup=completed and position_solution=true for the solve_setup check. If the pair is not supported, like unsupported+unsupported, the output should be error=unsupported_constraint_pair, first=<first>, second=<second>. No other source pairings should be accepted.

And one last thing since a couple folks asked about structure: we still want the separation of concerns approach here, with the core domain logic decoupled from the JSON I/O adapter, same as described in start.md. Also keeping the two-decimal formatting rule consistent across everything, matching what we’ve been doing in rcb_tests/public_test_cases/.
