## Product Requirement Document

Hey team, we need to ship the lifecycle container runtime we've been talking about. Basically the idea is that every 'region' of the app (a screen, a session, a subsystem) should be a named thing that lives in a tree, and when you kill a parent everything underneath it cleans up automatically — no more manually wiring teardown in 50 call sites. We've had a bunch of production incidents where background jobs keep running after the screen they belong to is gone, and also cases where cleanup callbacks blow up because they try to use something that's already been released.

The container should let you plug in observers that get notified when it starts up and shuts down. Registration should be safe to call multiple times without blowing things up. There's also that builder pattern we used in the auth flow last time — same idea here where observers attached during construction shouldn't fire until the thing is actually ready.

Background work needs to be tied to the container lifetime in a specific order — the work stuff has to stop BEFORE the regular cleanup callbacks run. Also need ancestor-walking for service and dependency lookups, and a neutral error story (no raw exceptions leaking out) for destroyed containers, missing deps, unnamed work scopes, etc. Check the existing test scaffolding for the exact shape of the ops.

One quick follow-up from the questions that came in: for anything touching a container after it’s already gone, we want one consistent outcome. If someone tries build_child, register observer, or get_service on a destroyed container, that should raise the neutral category 'scope_destroyed'. The one exception is destroy itself — calling destroy again is just a no-op and must NOT raise any error. Same general rule as before on failures too: raw host-language exception types must never leak into output.

Also, on teardown ordering, the async side needs to go first every time. If a container has bound asynchronous-work, that scope MUST be cancelled before any other lifecycle observer's exit callback fires. The expected reaction order in output is: background work first, then remaining registered observers in registration order. For Feature 5.7 (feature5_7_work_canceled_first.json), that means the async work scope bound to a container must be cancelled as the very first step of teardown, before any registered lifecycle observer's exit callback fires, producing output order: background,first,third. And if that container is hosting an asynchronous-work scope, both the hosted (parent) work scope and any child work scope derived from it must be cancelled. Both must report active=false after teardown.

A couple safety details too: teardown must be re-entrant safe. If an exit callback calls destroy on a container that is already mid-teardown (e.g., an ancestor), the second destroy call must be a no-op and must not recurse or raise an error. The entire tree must end up destroyed without infinite loops. Observer registration is idempotent as well, so registering the same observer N times results in exactly one entry notification and exactly one exit notification, regardless of how many times register or destroy is called.

And on the builder behavior, same intent as the auth flow pattern but spelling it out more explicitly: an observer registered while a container is still being assembled (inside the builder phase) must NOT receive its entry notification immediately. Entry is deferred until assembly completes. A mid-assembly probe of the observer's entry tally should report 0; only after the container is fully built should it report 1. That’s the bit covered by Feature 3.4 (feature3_4_builder_deferred_enter.json): the 'builder_register' and 'builder_emit_enter' ops in create_root that defer observer entry notification until after assembly completes.
