## Product Requirement Document

Hey team, we need to build out the core client-side library for the interactive code playground project. This is the glue layer between the editor UI and the backend execution service — think of it as all the plumbing that everyone keeps rewriting from scratch on every project. The main pain points we've heard from devs: loading code from different remote places is a nightmare because each source fails differently and nobody knows what URL was actually hit or what HTTP code came back; breaking a multi-file exercise blob apart is fragile and crashes when the file markers are weird; and keeping track of whether someone has unsaved edits is scattered all over the place. We also need the reactivity stuff — you know, the binding system we sketched out similar to what we did in the notifications module last quarter — and a way to scope shared services so they don't leak between sessions. There's also some token-parsing logic for the embed activation markers (check how we handled the CSS class prefix stripping before), and we need to pull body content out of raw HTML responses cleanly. The whole thing needs to be split into sensible modules, not one giant file. Errors should come back as stable category strings, never raw exception messages. Remote loading must work against a stubbed HTTP client for testing — no real network calls.

A couple extra specifics from the questions that came in. For the offset/line mapping, an offset belongs to whichever line has the half-open [start, nextStart) range that contains it, so in 'one\ntwo\nthree' offset=3 resolves to line=0 and offset=4 resolves to line=1. The newline character itself belongs to line 0. If the offset is at or beyond the end of the buffer, it resolves to the last line, and for an empty buffer every offset resolves to line 0. Going the other direction, if someone asks for a line number beyond the last line, we clamp to the start offset of the last line. In 'one\ntwo\nthree' (3 lines, indices 0-2), requesting line=3 returns offset=8. An empty buffer maps every line number to offset=0.

On the dynamically scoped registry behavior, outside any execution region like that, the ambient registry instance is absent. While we're inside the region, that registry becomes the ambient instance and its registered values are visible, and once the region exits the ambient instance is absent again. The output people should expect there is exactly 'outside_instance=absent', 'before_zone_instance=absent', 'inside_instance=present', 'inside_value=<v>', 'after_instance=absent'.

For remote loading, we need to surface the exact request URL and the HTTP status code we observed every time, because the tests are going through an injected HTTP client abstraction with an in-process HTTP stub keyed by request URL. No real network calls, but also no shortcuts that skip the HTTP boundary since then we lose the observable request URL and status behavior. For gists, the URL is 'https://api.github.com/gists/<id>'. A 200 returns the primary file content, a 404 gives 'error=content_not_found', and a 403 gives 'error=rate_limit_exceeded'. For Flutter documentation samples, the URL is 'https://api.flutter.dev/snippets/<id>.dart', and the output includes the request URL, HTTP status, and either 'main.dart.length=<n>' on success or 'error=content_not_found' / 'error=rate_limit_exceeded' on failure.

For repo exercises, the first fetch is always 'dartpad_metadata.yaml' from 'https://api.github.com/repos/<owner>/<repo>/contents/<path>/dartpad_metadata.yaml', and if there’s a ref it gets appended as '?ref=<branch>'. That descriptor is what provides the exercise name and file list, and if those fields are missing or invalid the result should be 'error=invalid_exercise_metadata'. Also, a file entry can declare an alternate path for where we fetch the content from, while still keeping its logical name in output. So the output key stays the logical file name, but the content comes from the alternate URL path.

On the editable overlay, it starts clean with dirty=false. It flips dirty after any field changes away from its original value, goes back clean if that field is restored to its original value, and also resets clean when a new backing document is adopted via set_backing. Output should report the dirty flag after each step.

One other thing to keep tight: failures need to stay normalized as stable category tokens only. We should not leak host-language exception type names, runtime message suffixes, or object representations into output. The adapter render layer is what does that cleanup. The valid tokens here include 'content_not_found', 'rate_limit_exceeded', 'invalid_exercise_metadata', and 'snippet_parse' with a 'reason=' field.

And for the remaining feature callouts, Feature 5 is the reactive value bindings piece: cancellable bindings between a source (stream or observable property) and a target (observable property or callback), with LSP-compliant interchangeable source/target types as specified in the SOLID constraints and test cases feature5_1 through feature5_4. Feature 3 is the embed-token interpretation piece: a token is valid when it contains the activation keyword 'run-dartpad', optionally preceded by a leading prefix such as 'language-'. The colon-delimited option parsing with first-hyphen key-value splitting is the logic to replicate.
