## Product Requirement Document

hey team, we need to ship the binary header codec thing for the encrypted message format. basically we need something that can take a header description and turn it into bytes and back, similar to what we did for the session serialization layer a while back (check how we handled the partial-read logic there, it should apply here too). the main pain points we've been hearing from devs are: 1) they keep screwing up the byte math when fields arrive in chunks over a socket and the whole thing crashes or silently corrupts, 2) nobody agrees on what an 'empty' or 'unbuilt' header should look like before it's been populated, 3) we've had a couple incidents where someone sent a header with no integrity protection attached and downstream just accepted it — that needs to be a hard stop. also there were some complaints about oversized metadata blobs blowing up in weird ways instead of giving a clean error. the codec needs to be split into sensible pieces, not one giant file, and the JSON/stdin stuff should be totally separate from the actual header logic. i'd say look at how the login module handled the compatibility validation last time for reference on the error reporting style. we need solid round-trip guarantees and the streaming/incremental decode is a must-have for the network team.

one extra pass on the sizing and streaming bits since a few folks asked. the serialized length is fully determined by the fixed-size fields plus the variable sections. using the example we talked through: a 12-byte nonce, 16-byte tag, one key blob whose provider id is 'None' (4 bytes), provider info 'TestKeyID' (9 bytes), and a 32-byte encrypted key, an empty/null encryption context yields serialized_length=113, a context serializing to 30 bytes yields 143, and a context serializing to 40 bytes yields 153. the formula we should stick to is: fixed_header_bytes + encryption_context_length + key_blob_section_length + nonce_length(12) + tag_length(16).

also, for the incremental path, the decoder really does need to work cleanly when bytes show up a little at a time in growing chunks. on each call it should consume whatever complete fields it can from the bytes provided so far, report exactly how many bytes it consumed, and if there still isn’t enough data yet it should just pause without error. partial input must never corrupt internal state or throw an exception — it simply returns 0 bytes consumed. after all bytes have been fed piecemeal, re-encoding the decoded header must reproduce the original byte sequence exactly (incremental_decode_stable=true). this is the same streaming/incremental decode behavior called out in Feature 2 of start.md: the decoder consumes bytes in growing chunks, reports bytes_consumed per call, and suspends without error on partial input, as specified in feature2_streaming_decode.json.

on the “what does an unbuilt header look like” question, let’s make that explicit: a newly created, unpopulated header must report message_id=absent, header_nonce=absent, and header_tag=absent. those three optional identity and authentication fields need to be explicitly absent, not null, not zero, not empty bytes, until the header is either fully constructed or decoded from bytes.

and two guardrails we should be strict about. first, the encryption context byte length is stored in an unsigned 16-bit field, so the maximum serialized context size is 65535 bytes. if the context exceeds that, the encoder must fail with error=encryption_context_too_large and max=65535 on the next line. it must not silently truncate the length field. second, if someone tries to serialize a header without the auth pieces in place, that should fail immediately: serializing a header that is missing either its header nonce or its header authentication tag must fail with error=missing_header_authentication followed by missing=nonce or missing=tag, whichever is absent. both have to be present before serialization is allowed, and we should never emit an unprotected header.
