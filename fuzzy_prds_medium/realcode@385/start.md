## Product Requirement Document

We need a rules engine that evaluates declarative matching rules against collections of discovered system attributes and produces derived outputs, so platform tooling can describe in configuration what labels and variables should be attached to a machine based on its hardware or software capabilities. Attributes are organized by domain and can be presented in three shapes: a bare set of named flags (no values), a dictionary of name-to-string-value pairs, or a list of instances each with their own attribute map. Matching expressions should support a range of operators: checking whether an attribute exists or does not exist, whether its value is in or not in a list, whether it matches a regular expression, whether it is numerically greater-than, less-than, or within a numeric range, and whether it equals the literal strings for true or false. An empty operator should be treated as an unconditional match. Each operator should be validated up front — wrong arity or non-integer values for numeric operators should produce a clear reason code rather than a runtime error. A compound match groups multiple named expressions that must all hold simultaneously (logical AND); an empty group matches anything. Rules combine a required-all group with an any-of list of groups (logical OR at the outer level). When a rule fires, it emits its static labels and variables plus any templated outputs where attribute values are interpolated into the template strings.

One small follow-up from the questions that came in: the presence-style operators are stricter than I implied. Any, which is the empty string '', along with Exists, DoesNotExist, IsTrue, and IsFalse, all require an empty value list. If anything is provided in values for those, that should validate as 'valid=false' with 'reason=value_must_be_empty'. Also, the empty string '' is treated as the 'Any' operator which always matches unconditionally, so that one is basically our unconditional case and still expects no values.

For validation, the reason codes we should return are exactly 'value_must_be_empty', 'value_must_be_nonempty', 'value_count_invalid', 'value_not_integer', 'value_out_of_order', 'value_invalid_regexp', and 'unknown_operator'. More specifically, 'value_must_be_nonempty' is for membership op with empty values, 'value_count_invalid' is for the wrong number of values like Gt/Lt needing exactly one, 'value_not_integer' is for non-integer values on Gt/Lt/GtLt, 'value_out_of_order' is for GtLt where first >= second, 'value_invalid_regexp' is for InRegexp with invalid regex, and 'unknown_operator' is for an op string we don’t recognize.

Also clarifying the match output formatting for the three feature shapes. For mes_keys, on match the output is 'match=true\nmatched=<names>' where names is the comma-separated matched attribute names sorted alphabetically. For mes_values, it is 'match=true\nmatched=<pairs>' where the pairs are 'name=value' sorted by name. For mes_instances, it should be 'matched=<N>\n' followed by N lines, each line containing comma-separated 'name=value' pairs sorted by name for that instance, and if an instance has empty attributes that line is just empty.

Last bit is the structure of the input we evaluate. The 'features' field is a map keyed by domain name, and each domain can provide up to three feature set types: 'keys' (NameSet — bare presence flags), 'values' (KeyValueSet — name→string map), and 'instances' (InstanceSet — list of attribute maps). The reference format is '<domain>.<featureset>' e.g., 'cpu.topology'.
