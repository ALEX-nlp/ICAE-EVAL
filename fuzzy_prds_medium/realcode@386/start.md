## Product Requirement Document

Hey team, we've been getting complaints from devs using our CLI toolkit that they keep having to write the same boilerplate argument scanning code over and over again. Every project reinvents the wheel — someone parses --long flags, someone else handles -s short forms, and nobody handles the edge cases the same way (like what happens when you pass a number that's too big, or forget to give a value after a flag). We need a proper library that lets you just declare what options your program accepts and get back a clean result object you can query.

Remember how we handled the 'compatibility logic' for grouped short flags in that config-loader module last quarter? Something like that approach would work here. Users should be able to bundle short flags together, mix positional arguments with named ones, and get sensible errors back when something goes wrong — not a crash or a silent wrong value.

We also want support for things like 'if the flag is missing, use this fallback' and 'if the flag is present but has no value attached, use this other fallback.' Numeric stuff needs to handle hex, negatives, overflow — the usual. Oh and there should be some kind of help/synopsis output. The JSON adapter contract we discussed internally should stay completely separate from the core engine. Please refer to how we scoped the iteration and optional-value patterns in the last sprint.

One extra pass on the details people asked about: short flags can absolutely be packed into one token like `-abc`, and the scanner should treat that the same general way we did before. The important nuance is that only the last short option in the bundle can be value-bearing. If that last one takes a value, it absorbs the rest of the token as-is, so `-bcfoo=something` sets `b` as a boolean flag and `c` to the value `foo=something`. Any non-final short flags in a bundle need to be boolean/non-value-bearing. This belongs in the tokenisation/scanning layer of the parsing engine, and the exact behavior we want is the same one shown by `-bcfoo=something` sets `b` as boolean and `c` to value `foo=something`.

Also, on defaults versus implicit behavior, these are meant to stay distinct. A default value only applies if the option never appears on the command line at all. In that case the rendered value is the default text, but the count is 0 because it was never actually seen. That is different from an implicit value, which is what we use when a value-bearing option is present with an empty equals-assignment like `--name=`. In that case you should treat it as present, with the empty string as the value and a count of 1. Separately, when `no_implicit` is set to true on an option, giving the bare flag with no argument should fail with `missing_argument` naming the option.

For options whose value type is itself optional, like `opt_string` or `opt_bool`, we want to keep the parsed value when one is supplied and also expose whether the optional currently holds a value. Querying the option should return the contained value when present. One specific case that came up: a boolean optional with a default of false should still record true when it is explicitly set to true on the command line.

On naming in ordered argument output, each recognised argument should use the option’s first long name from the `opts` specification as its canonical key. If there is no long name and the option only has a short single-character name, then use that short name as the canonical key instead. Aliases are not separate entries here; all aliases for the same option share one count and one value.

For list-valued options, repeats should accumulate in command-line order rather than overwrite. So `-j 9 --job 7 --job=10 -j5` produces `[9,7,10,5]` and a count of 4. The value can be attached directly like `--job=10` or `-j5`, or it can come from the next token.

And just to restate the boundary on architecture, the core parsing engine needs to stay completely decoupled from stdin/stdout and JSON parsing. The JSON input/output contract is only a black-box testing interface for the execution adapter. The adapter’s job is to translate JSON commands into idiomatic calls to the engine and then render results as output lines. The engine itself should just expose the typed API we talked about: register options, parse, query, with no awareness of JSON or console I/O.
