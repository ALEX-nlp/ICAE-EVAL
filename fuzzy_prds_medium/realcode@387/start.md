## Product Requirement Document

We need a small collection of reusable utility helpers that our desktop application can use across many different screens and components. These helpers should cover a few everyday needs: cleaning up version label strings so they all look consistent, producing grammatically correct text when showing counts of things, turning a flat list of release records into a fast lookup table, ordering those releases so the newest ones always appear first (while still handling oddly-formatted labels gracefully), navigating and updating deeply nested settings objects using a simple path notation, building a visual split-panel arrangement from a list of panels and also reading that arrangement back into an ordered list, computing where a floating tip or detail card should appear on screen without going off-screen edges, and reordering action buttons in dialogs to match what the operating system expects. The version-cleaning logic should work the same way it already does in our existing string normalization module, and the button ordering behavior should mirror the platform-aware sorting we use elsewhere in our dialog management code. All helpers must be pure and dependency-light so they can be tested independently and composed freely. Outputs should be plain text or compact serialized data with no extra whitespace, and each result should end with a newline.

A couple small clarifications from the team chat so nobody has to guess. For the version cleanup piece, normalizeVersion behaves exactly like the one in golden/src/utils/normalize-version.ts: it strips only the lowercase letter 'v' when it appears as the very first character of the version string. If the first character is anything else, including uppercase 'V', it stays as-is. If the version argument is missing or undefined, it should default to ''. The function just returns that modified string, and output should be followed by a trailing newline.

On pluralizing, this is intentionally very literal. The helper appends 's' only when the items array has length strictly greater than 1. So length 0 returns the word unchanged, length 1 also returns the word unchanged, and only arrays with 2 or more elements get the suffix. The suffix is exactly 's', with no 'es' handling or irregular forms.

For index_versions, the key should be the normalized version string, but the value should stay the original record untouched. So if the input record has version 'v1.0.0', the key becomes '1.0.0' but the stored value is still the original {"version":"v1.0.0"} object. Keys should be inserted in the same order as the input array. This reuses the exact normalizeVersion behavior from golden/src/utils/normalize-version.ts to produce the normalized keys. Output here should be compact JSON with no extra whitespace, followed by a trailing newline.

For sorting release keys, valid semver values are ordered highest-to-lowest using semver comparison, but anything that is not valid semver is treated as LOWER than any valid version and placed FIRST before all valid versions. If there are multiple invalid keys, those should be sorted alphabetically ascending with localeCompare. The example to keep in mind is ['garbage', 'moreGarbage', '3.0.0', '1.0.0'] — 'garbage' and 'moreGarbage' come first in alphabetical order, then the valid versions newest-first. Output is one key per line, each terminated by a newline.

Also, one detail on the layout tree builder: the topmost node always uses the 'row' direction by default, and every nested node under that always uses the 'column' direction regardless of the top-level direction. There can be an optional 'direction' parameter in the input, but that can override only the topmost node's direction. The split rule is firstHalf = first Math.floor(n/2) panes (rounding down), secondHalf = remaining panes. And if there is only a single pane, the result should just be that pane identifier as a bare JSON string, not a node object.
