## Product Requirement Document

Hey team, we need a CLI tool for working with IPv6 network blocks — basically a dev utility that our infrastructure folks can pipe JSON into and get structured text back. Think of it like that subnet calculator we built for the IPv4 stuff last quarter, but IPv6 only this time. The tool needs to handle the usual stuff: parsing network blocks, figuring out range boundaries, converting between the mask and prefix-length forms, checking if an address falls inside a network, splitting blocks into smaller chunks, and merging adjacent blocks back together. Also needs to gracefully handle bad input — our old tool would just crash and spit out Python tracebacks which broke the pipeline scripts downstream, so we need normalized error output this time regardless of what language we pick. One thing that kept coming up in retro: the numeric representation thing — apparently some of our allocation scripts need the raw integer value of an address for database storage. Oh and there was a request from the routing team about being able to "guess" a reasonable prefix for a bare address input. The whole thing should read a JSON blob from stdin and write plain key=value lines to stdout. Keep it clean, no extra logging noise on stdout. Architecture-wise, follow the same separation pattern we used on the last network utility — don't cram everything into one file.

Quick follow-up from the questions that came in: for IPv6, there is no broadcast concept, so the broadcast field should always come back as the literal string `null`. Not omitted, not blank, just exactly `broadcast=null`. Same general idea for counts: on IPv6 networks, usable equals total, so we are not doing the old IPv4-style subtract 2 behavior here. A /128 has total=1 and usable=1, and a /127 has total=2 and usable=2.

On dispatching, the adapter should decide what to do based on which keys are present in the JSON object from stdin. `notation` key → Feature 1/2 (CIDR string parse); `address`+`netmask` → Feature 1 (network summary); `prefix` integer → Feature 3.1; `netmask` alone → Feature 3.2/3.3; `address` alone → Feature 4 or Feature 8; `network`+`address` → Feature 5 containment; `network`+`other_network` → Feature 5 overlap; `network`+`new_prefix` → Feature 6 subnetting; `networks` array → Feature 7 supernetting; `prefix_text` → Feature 8.

Also, error output needs to stay normalized and always be exactly two key=value lines: `error=<category>` and `param=<value or null>`. The only valid categories are `invalid_argument` (malformed/unparseable input), `argument_null` (null/missing required value), and `argument_out_of_range` (value outside valid range, e.g. prefix > 128). If there is no specific param name to point at, use `param=null`.

For the integer form, this needs to be an unsigned decimal integer only — no hex, no `0x` prefix, and no language object repr. Since IPv6 is 128-bit, the number can be bigger than 64-bit, so please use arbitrary-precision integers. The output format here is `parsed=true\ninteger=<decimal>\n`.

One more edge case we wanted to lock down: in Feature 5, if somebody sends an IPv4 address into an IPv6 network containment check, that should return contains=false and not an error, same as described in feature5_ipv6_containment_and_overlap.json case 3.

And on structure, let’s keep the same split we talked about already: multi-file layout with strict separation between core domain logic (src/), the I/O adapter (adapter or cli entry point), and tests (rcb_tests/), mirroring the architecture described in start.md sections on Scale-Driven Code Organization and Separation of Concerns.

Last bit is just for how the team runs checks locally: the harness is invoked as `bash rcb_tests/test.sh [--cases-dir <subdir>]`. Default cases-dir is `test_cases`. For each case in `feature<N>_<name>.json`, stdout is written to `rcb_tests/stdout/<cases-dir>/feature<N>_<name>@<ZZZ>.txt` where ZZZ is zero-padded 3-digit case index. Those output files should contain only raw stdout, with no PASS/FAIL metadata mixed in.
