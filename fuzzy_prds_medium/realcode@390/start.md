## Product Requirement Document

Hey team, we need to build out a holiday/business-day calendar library — think of it like that date-utility thing we prototyped for the payroll team a while back, but much more complete and multi-country. The main pain points we're hearing: scheduling systems are breaking because nobody knows which dates are actual working days vs. public holidays, especially around Christmas/New Year when weekends shift things around. We need it to handle a bunch of countries (US, UK, France, Italy, Japan, Canada, NZ, Austria, Australia at minimum), and it should also support regional stuff like Canadian provinces or NZ districts because right now people are getting the wrong dates for BC vs Saskatchewan. Also need the ability to work with totally custom holiday lists for clients who have their own calendars. The date-shifting logic when a holiday lands on a weekend needs to follow proper substitution rules — similar to how we handled the observed-date logic in that other project. Business-day addition and counting should work correctly too. One more thing: the easter calculation needs to be baked in since a lot of the European holidays depend on it. Architecture-wise, please keep concerns properly separated — I/O and business logic should not be tangled together. Make it clean and extensible so we can bolt on new countries later without rewriting the core.

A couple extra details from the team’s questions. For the holiday list output, keep it exactly two lines: first line is `count=<n>` where n is the number of observed holiday dates, second line is `dates=<comma-separated YYYY-MM-DD list>` in ascending order. These are observed dates with substitutions already applied, not the raw holiday dates. Example format is `count=10\ndates=1999-01-01,1999-01-18,...\n`.

On the weekend shift behavior, the default rule is `next_monday`, and under that setup if a holiday falls on a Saturday, the observed date moves to the preceding Friday, and if it falls on a Sunday, the observed date moves to the following Monday. There is also a `saturday_before_sunday_after` rule where Saturday holidays move to Friday and Sunday holidays move to Monday. The `weekend_observed_date` feature exposes these rules directly, including feature10 with rules `next_monday` and `saturday_before_sunday_after`, with Saturday moves to preceding Friday, Sunday moves to following Monday under next_monday rule.

Also, when the calendar code is `CUSTOM`, we should not load any predefined country holidays at all. In that mode the caller sends a `holidays` array in the input, and each item has a `date` (YYYY-MM-DD string) and optionally a `name`. That custom calendar should treat only those dates, plus normal Saturday/Sunday weekends, as non-working days. This is specifically what we need for previous_working_day, next_working_day, and business_days_add.

For factory lookup, the `factory_resolution` feature is just checking that a two-letter country code resolves to a calendar and then returning the holiday count for the requested year. The output format there is exactly four lines: `country_code=<CC>\nresolved=true\nyear=<YYYY>\nholiday_count=<n>\n`. For reference, AU for 2017 yields holiday_count=7 and that is national holidays only, no state. AT for 2017 yields holiday_count=13.

Japan needs a little extra care. The JP calendar should include fixed holidays with Monday substitution when falling on Sunday, the 'Citizen's Holiday' (kokumin no kyujitsu) sandwich rule where a weekday between two public holidays also becomes a holiday, Vernal Equinox Day and Autumnal Equinox Day which are astronomically calculated and vary by year, and the Happy Monday rules for Coming-of-Age Day (second Monday of January), Marine Day (third Monday of July), Respect for the Aged Day (third Monday of September), and Health-Sports Day (second Monday of October). For 2025, 16 observed dates are expected.

One other behavioral point on working-day search defaults, since that came up. By default, if `same_day` is not specified, next_working_day does NOT return the input date itself, so the search starts AFTER the input date. For example, if 1999-11-25 (Thursday, Thanksgiving) is a holiday, the next working day returned is 1999-11-26. The `same_day=true` flag must be explicitly set to allow the input date to be returned as-is when it is already a working day with zero offset. Similarly, previous_working_day by default starts the search BEFORE the input date. For example, 2007-01-01 (New Year, Monday) results in 2006-12-29. When `same_day=true` is explicitly provided, the input date itself can be returned if it is a working day. Also, negative open_days values must be rejected with `error=argument_out_of_range`, `param=open_days`, and `value=<the supplied value>`.
