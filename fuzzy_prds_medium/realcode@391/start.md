## Product Requirement Document

We need a small scripting language interpreter that developers can embed in tools without pulling in a large external runtime. The language should feel familiar: integers, floats, booleans, strings, arrays, and objects are all first-class values; the usual arithmetic, comparison, and logical operators work as expected; variables are declared with a keyword and can be reassigned; there's a standard for-loop with an early-exit construct and a skip-to-next construct; functions are first-class and support recursion; objects have named fields and a method-call syntax that passes the object as the first argument; and there's a try/catch mechanism for structured error handling. The only built-in output operation is a print call that writes a value followed by a newline. Float literals are written with a trailing marker to distinguish them from integers. A float that has no fractional part should render without a decimal point. Bitwise operators work on integers using the same two's-complement semantics as the type-inspection layer does for fixed-width integers. When source text cannot be parsed, the run should stop immediately with a normalized error before any code executes. An assertion built-in should fail loudly with a normalized error when its condition is false. Uncaught exceptions at the top level should produce a two-line normalized report with the category and the rendered thrown value, never leaking the host runtime's exception type.

A couple quick clarifications from the questions that came in: float literals should be written with digits, a decimal point, and a trailing 'f' marker, like '3.5f' and '10.0f'. Without the 'f' a number is an integer. And just to be explicit on rendering, a float with no fractional part still prints without a decimal point, so 2.0f prints as '2'.

Also, the typeof operator should return the normalized names we've been using: 'int32' for 32-bit integers, 'string' for text, 'bool' for booleans, and 'float32' for floating-point numbers.

On methods, the call style is the colon form, 'obj:method(args)'. That passes the receiver object as the implicit first parameter. So if somebody stores a function in a field with 'obj.method = function(self) {...}' and then calls 'obj:method()', that should pass obj as 'self'.
