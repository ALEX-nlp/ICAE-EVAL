## Product Requirement Document

We need a small, reusable numeric engine that takes a list of closing prices and a look-back window size and produces a full trend or momentum curve — one smoothed value per input price, in order. The engine must support at least four classic signal types: a plain running average, a recency-weighted average that reacts faster to new prices, a linearly tapered average that puts the most emphasis on the latest price, and a bounded momentum oscillator that compares upward pressure to downward pressure on a zero-to-one-hundred scale. All output values must be formatted as fixed four-decimal strings using standard rounding; a mathematically indeterminate result should be represented by a special not-a-number marker rather than crashing. Before a full window of historical data exists, each indicator should degrade gracefully to a shorter window rather than returning nothing. The system must expose a clean command-line adapter that accepts a JSON request on standard input and writes the curve to standard output; if the requested signal type is not recognised, the adapter should emit a two-line neutral error message. The core calculation logic must be fully decoupled from the I/O layer. Agents implementing this should look for the existing indicator hierarchy in the tracker sub-package and the decimal wrapper used throughout the library for exact rounding semantics.

One quick follow-up since a few people asked how the early points should behave. If the indicator name is not recognised, emit exactly two lines: first line is 'error=unknown_indicator', second line is 'indicator=<name>' where <name> is the lowercased indicator string from the request. On the averaging side, SMA should use the shorter history that exists instead of waiting for a full window. So at position i (0-based) the average is taken over positions 0..i using divisor (i+1), and the effective window is Math.min(timeFrame, index+1). Also worth calling out that a timeFrame of 1 reproduces the input series exactly.

For EMA, the startup behavior should also lean on the growing-window average before the normal smoothing recurrence kicks in. Positions where (index+1 < timeFrame) use the SMA value over the same growing window as the seed. At index==0 when timeFrame > series length, the value equals the first closing price directly. Once (index+1 >= timeFrame) the recurrence applies: prevEMA + multiplier * (currentPrice - prevEMA) where multiplier = 2/(timeFrame+1).

For WMA, same idea on the front edge: when index < timeFrame, weights run from (index+1) down to 1 for positions (0..index), and the divisor is ((index+1)*(index+2))/2. At index==0 the value is the first closing price. When a full window is available, weights run from timeFrame down to 1 and divisor is (timeFrame*(timeFrame+1))/2.

And for anyone wiring this up in the existing codebase, the package to look at is eu.verdelhan.ta4j.indicators.trackers, which contains SMAIndicator.java, EMAIndicator.java, WMAIndicator.java, and RSIIndicator.java — the four core indicator implementations.
