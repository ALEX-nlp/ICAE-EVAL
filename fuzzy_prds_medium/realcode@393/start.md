## Product Requirement Document

Hey team, we need to build out that Bitcoin primitives toolkit we've been talking about. Basically a library that handles all the low-level wallet stuff — encoding, hashing, key management, address building, that kind of thing. The main pain point we keep hearing from devs is that they're hand-rolling all this binary format logic themselves and then losing funds because of tiny mistakes (wrong byte order, missing checksums, etc). We need something solid and well-organized — not just one giant file dumped together, please. Think about how we structured that encoding/crypto separation we did on the last project.

The library needs to handle the full stack: the Bitcoin-style base encoding with and without checksums, all the standard digest functions including the keyed variants, the compact integer format used in wire messages, big number math with the script-specific buffer format, opcode name mapping, the 12/24-word seed phrase stuff for wallet recovery, private and public key operations, and address generation for both mainnet and testnet. Error cases matter a lot here — bad checksums, invalid curve points, unknown opcodes — these should come back as clean typed errors, not stack traces. There should be a thin JSON adapter layer on top so we can test it via stdin/stdout without the core knowing anything about JSON. Follow that SOLID breakdown the architecture doc mentions.

One more thing to make super explicit since the team asked: for Base58, the alphabet is exactly 123456789ABCDEFGHJKLMNPQRSTUVWXYZabcdefghijkmnopqrstuvwxyz, and it’s 58 characters because it deliberately leaves out 0 (zero), O (uppercase o), I (uppercase i), and l (lowercase L) so people don’t mix them up. Also, if the payload starts with zero bytes, those need to survive as leading '1' characters in the encoded output. For Base58Check, the checksum is the first 4 bytes of the double-SHA256 of the full payload, and those 4 bytes get appended before Base58 encoding. On decode, strip the last 4 bytes, recompute the double-SHA256 of the remaining payload, and the first 4 bytes have to match. If the decoded buffer is shorter than 4 bytes, return error=input_too_short. If the bytes don't match, return error=checksum_mismatch.

Also clarifying the CLI/output shape since that came up a few times: every operation writes one or more key=value lines to stdout, each terminated with a newline (\n). No trailing whitespace, no blank lines between outputs. The keys are operation-specific and should stay exactly as follows: encode ops use base58=, base58check=, or hex=; hash ops use digest=; arithmetic uses result=; comparison uses gt=, lt=, eq= (three separate lines); boolean checks use valid=; address ops use address=, network=, type=, hash=; key ops use public_key=, wif=, xprv=, xpub=, serialized=, compressed=, x=, y=; mnemonic ops use phrase=, seed=, count=, first=; opcodes use name= or number=; errors use error=<category>.

For the number conversion helpers, bignumber.to_buffer takes dec (base-10 string), size (byte width), and optional endian ('big' default or 'little'). The integer is converted to a big-endian byte array, zero-padded on the left to exactly 'size' bytes. If endian is 'little', the padded array is reversed. bignumber.from_buffer takes hex and endian; if little, reverse the bytes before interpreting as unsigned big-endian integer. Output is hex= for to_buffer and result= (base-10) for from_buffer. For varint, Bitcoin varint encoding is the usual one: values 0x00–0xFC encode as a single byte, values 0xFD–0xFFFF encode as 0xFD followed by 2 little-endian bytes, values 0x10000–0xFFFFFFFF encode as 0xFE followed by 4 little-endian bytes, and values above that encode as 0xFF followed by 8 little-endian bytes. Output for encode is hex= and for decode is number=.

On script numbers, use sign-magnitude little-endian encoding. The magnitude is written in little-endian byte order, and the sign lives in the most significant bit of the last byte: if negative, set the high bit of the most significant byte. If the magnitude's MSB is already set, append an extra byte to hold the sign bit, using 0x80 for negative or 0x00 for positive. Zero encodes to an empty buffer. Example: 1000 decimal = 0x03E8, little-endian = e803; -1000 = e883.

And on structure, yes, we do want the repo split the way we described before: separate directories for encoding/ (base58, base58check, varint), crypto/ (sha256, ripemd160, hmac variants), and key/address domains (privatekey, publickey, hdkey, address, mnemonic, script, opcode, bignumber). Same deal with the SOLID expectations from the architecture notes: SRP keeps parsing/codec/error/formatting in distinct units; OCP means adding a hash doesn't touch existing primitives; LSP means xprv and xpub share a substitutable key interface; ISP means per-primitive interfaces are small; DIP means the JSON adapter depends on core abstractions, never the reverse.
