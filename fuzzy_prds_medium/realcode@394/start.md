## Product Requirement Document

Hey team, we need a text layout engine library built in Rust. We've been getting complaints from users that rendering mixed-direction text, handling cursor positions, and doing paragraph wrapping all require tons of custom glue code that keeps breaking. The idea is to wrap the parley/fontique ecosystem into something clean that apps can just call into.

Basically it needs to handle the usual stuff — laying out lines, measuring how wide content is before committing to a layout, embedding little boxes inside text (like icons or images inline), controlling how lines wrap when text overflows, aligning text left/right/center, and inspecting individual character clusters for hit testing and cursor logic. Also line height control — we had a bug last quarter where the absolute spacing mode wasn't being respected, similar to how we fixed the metric scaling in that other rendering module.

The adapter layer that reads commands and spits out results should be totally separate from the core logic — we don't want the business rules tangled up with the JSON parsing like what happened last time. There should be a test harness that runs all the case files and saves raw output per case so we can diff them. Cases live under rcb_tests/public_test_cases/ and output should go under rcb_tests/stdout/ namespaced by the cases directory name. Needs to match the expected output exactly.

One quick follow-up since a couple details came up while people were wiring the harness and checking outputs. For the saved stdout artifacts, the filename format needs to be `{filename_stem}@{case_index_zero_padded_3}.txt`. So for example, the first case (index 0) in `feature1_multiline_layout.json` run with `--cases-dir public_test_cases` should land at `rcb_tests/stdout/public_test_cases/feature1_multiline_layout@000.txt`. The case index is zero-padded to 3 digits. Also, `rcb_tests/test.sh` takes an optional `--cases-dir <subdir>` flag, and if you don’t pass it the default is `test_cases` (not `public_test_cases`). If you want the public suite specifically, you do need to pass `--cases-dir public_test_cases`.

Also want to be extra explicit on the separation line here: the core layout logic needs to stay completely decoupled from stdin/stdout and JSON parsing. The adapter should be a dedicated execution adapter in a physically separate file/module whose only job is deserializing JSON input into typed structs and calling idiomatic methods on the core domain. The core must never import JSON or I/O libraries. That’s the Dependency Inversion Principle version of this: high-level layout rules depend on abstractions, not I/O details. Along the same lines, each `.txt` file in `rcb_tests/stdout/<cases-dir>/` should contain ONLY the raw stdout from the program under test — no PASS/FAIL labels, no metadata, no headers — so we can do direct byte-for-byte comparison against the `expected_output` field in the JSON case file.

For the line range question, yes: when text contains explicit newline characters (`\n`), the line range for that line includes the newline byte. For example, `"Hello world!\nLine 2\nLine 4"` produces line[0] range 0..13 (includes the `\n` at index 12), and line[1] range 13..20 (includes the `\n` at index 19). The final line has no trailing newline so its range ends exactly at the last character byte.

And for the text dump format, the `layout_summary` action outputs: `direction=<ltr|rtl>`, `line_count=<n>`, `width=<f>`, `full_width=<f>`, `height=<f>`, then for each line: `line[i]=range:<s>..<e> advance:<f> trailing:<f> offset:<f> baseline:<f> break:<reason> items:<n>`, followed by each run: `run[i.j]=range:<s>..<e> dir:<ltr|rtl> clusters:<n> advance:<f>`. All floats are formatted to exactly 2 decimal places.

Last thing, on the line height bug we called out before: the `line_height` feature (Feature 7) with `mode: absolute` must set baselines at `line_height_value * (line_index + 0.75)` effectively. So for absolute 20.0 the baselines are 15.00 and 35.00, matching the parley LineHeight::Absolute configuration path in the layout engine core.
