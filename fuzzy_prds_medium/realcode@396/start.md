## Product Requirement Document

Hey team, we need a small utility library that handles all the annoying low-level plumbing stuff our CLI tool keeps reinventing. Basically things like expanding those dollar-sign placeholder tokens in config strings, parsing those semicolon-separated key=value blobs users pass as single arguments (remember how messy that got last time with the quoted connection strings?), figuring out relative paths between project directories, reading properties out of .csproj files, picking the right SDK version from the installed list, and sorting out the Linux/macOS user identity stuff. The user-identity piece especially needs to be testable without actually running OS commands — similar to how we handled the injectable abstraction in that other auth probe module we did. For the key=value parsing, there was a specific edge case that caused a support ticket a while back around empty keys — make sure that's handled gracefully, not just a crash. The path logic also needs to handle Windows-style backslash paths coming in even when running on Linux. Output should just be simple line-based text. The whole thing should be split into sensible groupings of related functionality rather than one giant file. We want to be able to drop this in and have the CLI call these helpers without them knowing anything about JSON or stdin/stdout.

One quick follow-up from the questions that came in: for the env substitution behavior, if we see a $(NAME) token and there is no matching environment entry for that name, we leave it exactly as written, including the surrounding $( and ) characters. It should not turn into an empty string and it should not be treated as an error. The output line format there is exactly result=<expanded string>.

On the key=value parsing side, the empty-key case needs to be treated as a real domain error, not something that bubbles up as an unhandled exception. So if the input is '=aaa', the output must be exactly the single line error=command_line_parse_error\n and nothing else — no count line, no partial results. Also, either or both sides of a key=value pair may be wrapped in double quotes, and those quotes get stripped in the parsed result. That quoting is what allows literal '=' and ';' inside the value, so "ConnectionString"="User=foo;Password=test" should parse as one pair with key ConnectionString and value User=foo;Password=test. Also worth calling out that a trailing semicolon is fine and should not create a bogus extra empty pair, so 'ShouldCreateTable=true;BlogTableName=;' is the same as 'ShouldCreateTable=true;BlogTableName='.

For the POSIX user-resolution piece, please keep it behind an injectable process-probe abstraction plus an injectable logger abstraction instead of directly running OS processes. Tests need to be able to feed in scripted results with executed flag, exit_code, output string, error string and verify behavior without spawning anything real. The interface here should stay small and focused per ISP. Same general architecture note as before: related primitives should be grouped into cohesive units rather than one giant file or one file per tiny helper. The suggested split still makes sense: (1) string/option-parsing unit (env substitution, key-value parsing), (2) path/manifest unit (relative path, XML project manifest inspection), (3) process/user unit (SDK version selection, POSIX user resolution), (4) argument-parsing unit.

Also just to make the boundary extra clear, the thin execution adapter is the only layer that should decode the JSON input payload, call the right primitive with native values, and format the result as plain line-based text output. The core primitives should just take and return ordinary native values like strings, maps, booleans, numeric versions, and small structs, and they should never reference JSON, stdin, or stdout directly. For the team members wiring tests, Feature 7 (POSIX user resolution) requires a dependency-injected process-probe interface and logger interface as described in the DIP constraint in the Architecture section of start.md, enabling test doubles via the id_results scripted input in feature7_posix_user.json, and Feature 2 key/value parsing: a pair with an empty key (e.g. '=aaa') must output exactly 'error=command_line_parse_error\n' as specified in feature2_key_value_parsing.json and the Resilience NFR in start.md.
