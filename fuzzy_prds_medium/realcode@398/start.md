## Product Requirement Document

We need a small collection of general-purpose helper routines that our teams keep reimplementing from scratch on every project. The pain points are: (1) handling list positions that count from the end — negative indices should work the same way they do in the "standard utilities" we already use elsewhere; (2) merging one list into another with optional start/end bounds; (3) running a risky operation and capturing the outcome as a two-slot result without scattering try/catch blocks; (4) deciding whether two decimal values are 'close enough' to treat as equal; and (5) copying selected properties from one object into a fresh object, with the option to rename them, and correctly handling zero and empty-string values (not just truthy ones). We also need an in-place way to add extra properties to an object or to every object in a list, and a safe way to ask whether a subject actually has a given property before touching it. Please treat undefined/absent properties consistently with how the auth module handles absent values — look at how that convention is established in the codebase and follow the same sentinel pattern throughout.

A couple small clarifications from the team’s questions. For the index helper, toIndex does NOT clamp. A non-negative offset is returned unchanged even when it exceeds array.length. A negative offset resolves to array.length + offset, which itself may be negative when |offset| > array.length. That’s the behavior we want to preserve rather than trying to force it back into range.

On the list merge behavior, when 'from' is undefined or empty, the 'to' value is returned unchanged (which may itself be undefined). When 'to' is undefined and 'from' is non-empty, a new array (slice of from) is returned. When both are undefined, undefined is returned, printed as 'result=undefined'. For the risky-operation wrapper, the second slot (result[1]) contains only the error's .message string — never a runtime type name, stack trace, or object rendering. If the value thrown has no .message, its string form is used.

Also, for the property-picking / has-property side of this, the TypeScript 'undefined' value is the sentinel for 'not present'. JSON null maps to the language null (a defined value that IS picked). The absent sentinel is the TypeScript undefined, rendered as the literal string 'undefined' in output.
