## Product Requirement Document

Hey team, we need a new CLI tool for the cargo-vet workflow. Basically devs are spending way too much time manually cross-referencing dependency review records, exemptions, and changelogs before a release — it's error-prone and nobody's doing it consistently. The tool should wrap the existing cargo-vet commands and give us a unified interface for checking audit coverage, getting review suggestions, and doing source diffs between package versions. Think of it kind of like what we built for the login module's compatibility layer — same idea where we normalize errors so automation doesn't explode on unexpected output.

The interface should accept JSON commands on stdin and return structured results. We need it to handle at least: reporting its own version, showing audit status in both human and machine-readable formats, listing review suggestions ranked by effort (suppress the live registry noise so CI runs are stable), exporting the full dep graph with edge metadata, normalizing bad certification requests into clean domain errors instead of stack traces, diffing two package revisions including git-pinned ones, and validating revision strings upfront. Error cases especially need to be clean — ops was complaining last sprint that they can't tell a domain failure from a Python crash.

Format options matter: some consumers want plain text, others want JSON for downstream tooling. Output needs to be deterministic so we can regression-test it. Fixture data is already in the repo somewhere. Please keep the architecture clean — last time someone dumped everything in one file and it was a nightmare to maintain.

One more pass on the specifics the team asked about. For the version command, please keep the contract really tight: it must print exactly one line to stdout in the format `<name> <version>`, where version is exactly three dot-separated numeric components and each component is 0–255. stderr must be empty (0 bytes) and the exit code must be 0. The checks we care about there are exit_code=0, stderr_bytes=0, has_name=true, version_component_count=3, version_components_all_numeric=true.

On audit status for the bundled fixture project, the expected numbers are exactly fully_audited=7, partially_audited=2, exempted=97. In text format the conclusion word should be 'succeeded', and in JSON format it should be 'success'. stderr_lines must be 0 on success. Also for the JSON shape, include the first and last entries in each category, with the lists sorted alphabetically: vetted_fully_first='atty 0.2.14', vetted_fully_last='tracing 0.1.33', vetted_partially_first='cfg-if 1.0.0', vetted_partially_last='unicode-bidi 0.3.7', vetted_with_exemptions_first='bumpalo 3.9.1', vetted_with_exemptions_last='winreg 0.10.1'.

For revision handling, a git-pinned revision is formatted as '<semver>@git:<40-hex-digit-sha1>', for example '1.0.37@git:4445659b0f753a928059244c875a58bb12f791e9'. That hash must be exactly 40 lowercase hexadecimal characters. This same form is used in graph exports, diff commands, suggestion actions, and as a valid 'to' revision in source comparison. Plain semver like '1.0.90' is also valid. The fixture project itself lives under the repository at the path referenced by the cargo-vet working directory used in tests, so look for the 'test-project' workspace member with Cargo.toml and the supply/audit files. The cached source tarballs for syn and proc-macro2 used in diff tests are in the cargo registry cache that the test harness can see, and if the path is annoying to find just use grep or a directory listing from the repo root.

For review suggestions, they should be ranked by estimated review size ascending, so smallest first. The top 5 actions for the fixture project are action_0=diff proc-macro2 1.0.37 1.0.37@git:4445659b0f753a928059244c875a58bb12f791e9, action_1=inspect tinyvec_macros 0.1.0 Soveu tinyvec 81 lines, action_2=inspect matches 0.1.9 SimonSapin url/idna/form_urlencoded 210 lines, action_3=inspect foreign-types-shared 0.1.1 UNKNOWN foreign-types 302 lines, action_4=inspect try-lock 0.2.3 seanmonstar want 380 lines. Total recommended_action_count=97 and note_count=5. Registry suggestions still need to stay suppressed so the output is deterministic.

Also confirming the certification error normalization case: Feature 5 should return exit_code=255 and error=not_a_foreign_package, with no stack traces or exception class names in output, and that should behave the same in both JSON and text format modes. And on the architecture side, keep using the Dependency Inversion Principle adapter pattern described in Architecture constraint #2 and #3, meaning the execution adapter translates JSON stdin commands into idiomatic core domain method calls, with I/O kept completely decoupled from business logic.
