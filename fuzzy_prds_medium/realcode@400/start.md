## Product Requirement Document

Hey team, we need to build out the secret state management thing we discussed last sprint. Basically, clients need isolated storage for sensitive stuff — like secrets, keys, signing material — and we need to make sure none of the raw secret bytes are ever exposed in any output. Think of it as similar to what we did with the session isolation logic in the old auth module, but more structured this time.

The system should support writing indexed records into named vaults per client, removing records, and also doing a full garbage collection pass. There's also a need for a separate non-secret key/value store that works independently of the vault. Persistence is important — we should be able to save all state into a named encrypted snapshot and reload it later after wiping the in-memory state.

On the crypto side, we need hashing, signing with hierarchical key derivation, key agreement, authenticated encryption, and BIP39 recovery flows. All crypto outputs should be in lowercase hex, and the operations need to be deterministic given fixed inputs. The hint listing order for vault records is deterministic but not sorted in any obvious way, so devs are going to need to match the internal storage order exactly.

Multiple clients need to be fully isolated even when targeting the same vault path. Client lifecycle stuff (spawn, switch, remove, retarget) should produce proper error categories, not raw exceptions. Let's make this clean and production-grade.

A couple follow-ups from the questions that came in: for vault hints, the order is the internal record-storage order and we need to match that exactly as-is. It’s deterministic, but it is NOT sorted by index or insertion order. For the case with indices [0, 5, 15, 19], the expected hint order is h19, h5, h0, h15. Same idea called out another way: indices [0,5,15,19] produce hint order [h19, h5, h0, h15]. Please don’t sort by index, don’t sort alphabetically, and don’t assume insertion order.

Also, garbage collection does not delete the vault itself. After gc, all records can be revoked and hints_after_gc=0, but the vault still needs to show up as present. The output needs to include both 'garbage_collected_vault=present' and 'vault_after_gc=present'. So we’re clearing record contents, not destroying the vault container.

On the multi-client side, Feature 5 (multi_client) and Feature 6 (client_targeting) in start.md are the important reference points: per-client vault isolation means each client has independent record storage even when sharing the same vault path string, plus explicit spawn/switch/kill lifecycle with error on retargeting a removed client. If multiple clients write to the same vault path, they should still be totally isolated from each other. Each client’s record_count must be 1, and the hint shown for that client must only be that client’s own hint. The per-client output line format is exactly 'client=<name> record_count=<n> hint=<hint>' all on one line with spaces between fields.

For errors, keep everything language-neutral and single-line. The format is 'error=<category>'. Valid categories include 'error=invalid_input', 'error=procedure_rejected', and 'error=target_not_found'. For client retargeting specifically, if someone tries to re-target a client that has already been removed/killed, the output must be exactly 'reswitch_error=target_not_found'. Don’t emit exception class names, stack traces, error codes, or any host-language-specific detail.

One more snapshot detail: the snapshot key is just a plain string, for example 'abcdefghijklmnopqrstuvwxyz012345', and that exact kind of string is used as the encryption key when persisting client state. The flow should be write records, persist all client state under the named snapshot, fully clear the in-memory client cache, then reload from the named snapshot using the same key string. After restore, hint counts and record existence should match the pre-snapshot state exactly.
