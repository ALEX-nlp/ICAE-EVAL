## Product Requirement Document

We need a bridge layer that makes compiled native functions callable from a dynamic scripting environment. When a developer exposes a native function through this bridge, callers in the dynamic environment should be able to pass arguments by position or by name, and any parameters not provided should fall back to their declared defaults. The bridge should also support exposing multiple implementations under the same name that differ by argument type — the runtime picks the right one automatically. If no implementation matches the argument, the bridge reports a clean type-mismatch error rather than crashing. Fixed-width integer types need explicit range checking: passing a value that doesn't fit the target size should fail with a type-mismatch, not silently wrap or truncate. Text and raw byte buffers both need to round-trip correctly across the bridge. Certain parameters can be declared as keyword-only following the same convention as the argument-binding layer uses for positional-only params, meaning the caller must name them explicitly and cannot accidentally pass them by position. Functions that handle open-ended argument lists should be able to capture any extra positional and keyword arguments beyond the declared parameters. When a native function throws an exception, that exception should propagate across the bridge as a runtime error carrying the original message, never exposing the host language's internal error type.

One small follow-up from the questions that came in: for the subtract example, the defaults are first parameter default=8 and second parameter default=1, so if it’s called with no args the result should be 8-1=7. The named parameters there are first='i' and second='k', and mixed binding should work the same way you’d expect from the rest of the argument handling, so args=[3], kwargs={'k':5} gives 3-5=-2.

Also on the overload example, 'classify' should return integer tag 1 for integer overload and 2 for real/float overload. The dispatch there is based on the genuine value type, not just whether something looks coercible, and a non-numeric type like a string should come back as 'error=type_mismatch'.

And for count_extras, the first two positional args fill the declared parameters and are NOT counted as extras. Only positional args after those first two should be included in 'positional=<n>', and any keyword arguments should be included in 'keyword=<n>'.
