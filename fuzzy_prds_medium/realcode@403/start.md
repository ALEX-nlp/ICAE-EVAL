## Product Requirement Document

Hey team, we need a BSON toolkit library that lets devs work with binary JSON stuff without having to hand-roll all the encoding/decoding themselves. Basically the pain point is that every team that touches MongoDB data ends up writing their own brittle conversion code and then things break at the edges — weird number widths, dates that lose precision, identifiers that don't round-trip cleanly, that kind of thing.

The tool should be runnable from the command line where you pipe in a JSON command and get back predictable text output. Think of it like that encoding helper we built for the auth service — same idea but for BSON types. It needs to handle the two flavors of Extended JSON (you know, the strict one and the looser one), plus wire-format encode/decode for whole documents, looking up nested fields by path, object IDs, dates, the high-precision decimal type, binary blobs with subtype metadata, and regex+timestamp combos.

Errors should be consistent — don't just throw, normalize them into something the downstream consumer can pattern-match on. Output format needs to be deterministic line-by-line, not just a blob.

Also the codebase structure matters here — don't just dump everything in one file, keep the CLI adapter separate from the actual domain logic. We've had bugs before where test output formatting leaked into core business logic.

Let me know if the scope is unclear on any of the type-specific behaviors.

One quick follow-up after the questions that came in: for Feature 1.1 (json_to_canonical_extended) and Feature 1.2 (json_to_relaxed_extended), canonical mode really does need to stay explicit about types, so plain JSON integers in the 32-bit signed range (-2147483648 to 2147483647) should come out as {"$numberInt":"<value>"} and anything supplied through $numberLong should come out as {"$numberLong":"<value>"}. In both cases the numeric value inside the wrapper stays quoted. Relaxed mode is the opposite direction stylistically: use plain JSON scalars where safe and ISO date strings. So plain JSON integers and $numberLong values that are safely representable as a JavaScript number can render as bare JSON numbers with no wrapper, and values like 9007199254740991 are still considered safe there. Also, even values outside the safe range, such as -9223372036854775808, should still be rendered as bare JSON numbers in relaxed mode because that mode drops the $numberLong wrapper entirely. The one thing that does keep its wrapper in relaxed mode is non-finite doubles, so NaN, Infinity, and -Infinity should still show up as {"$numberDouble":"NaN"} / {"$numberDouble":"Infinity"}.

Dates need to be handled differently by mode too. In relaxed Extended JSON mode, BSON dates should render as {"$date":"<ISO 8601 string>"} using UTC with second precision, for example "1970-01-01T00:00:00Z". In canonical mode, dates should render as {"$date":{"$numberLong":"<milliseconds>"}} and that milliseconds value is a quoted string.

On the output side, the regex_timestamp feature should emit exactly one line in the form canonical=<JSON document>. That document has exactly two keys: 'regex' with {"$regularExpression":{"pattern":"<pattern>","options":"<options>"}} and 'timestamp' with {"$timestamp":{"t":<unsigned int>,"i":<unsigned int>}}. The t and i values are bare JSON integers, not strings, and the options field must always be present even if it's an empty string.

For bson_roundtrip, the output should be exactly two lines: byte_length=<integer> and then canonical=<canonical Extended JSON document string>. That canonical document should come from decoding the BSON and then running it back through the canonical Extended JSON renderer, so typed values keep their proper wrappers. Also, byte_length is the full BSON wire size, including the 4-byte document length prefix and the 1-byte null terminator at the end. As a concrete example, {"name":"Ada","age":36,"active":true,"scores":[10,20]} is exactly 64 bytes. That count needs to reflect the real encoded bytes, not a guess.

Last thing, just reinforcing the structure expectation since a few people asked: this needs to use a multi-file structure. The stdin-reading CLI entry point should parse a JSON command object with a 'task' field, dispatch into core domain functions, and print deterministic line-by-line text to stdout. The CLI/execution adapter part stays physically separate from the core domain modules. The domain side owns BSON value types, Extended JSON rendering, wire encoding, and path lookup with no dependency on I/O, and the adapter is just the layer that translates command objects into those calls and formats stdout. Please don’t put all of it in one file.
