## Product Requirement Document

We need a stateless filtering engine that, given a declarative rule and a batch of incoming blockchain blocks, decides which of those blocks contain transactions we care about. Each block carries a list of transactions; each transaction has an identifier hash, may optionally have a kind (contract deployment, contract method call, or a plain token transfer), and may carry receipt events (token movements, smart-contract log messages). The caller describes what they want with a single predicate — for example 'give me every block where a transaction minted or transferred a specific fungible token', or 'give me deployments from a particular account address', or 'give me any transaction that emitted a log message from a given contract containing a certain substring'. The engine should evaluate the predicate across all provided blocks and report back: whether anything matched at all, how many blocks had at least one matching transaction, and the identifiers of every matched transaction in block-then-in-block order. The report format must be consistent across all predicate types. For the smart-contract log predicate, the contract identifier field and the payload substring both support a wildcard that matches anything. For the deployment predicate with a standard-trait filter, we explicitly do not detect which trait a deployment implements, so that form never selects any transaction. The whole thing should behave like a pure function with no side effects — given the same predicate and the same blocks, always returns the same result.

One small follow-up from the questions that came in: the adapter is really strict about what it prints. It always prints exactly three lines in this order: 'triggered=<true|false>', 'apply_block_count=<n>', 'matched_transactions=<csv>'. The csv is comma-separated txids in block order then in-block order. When nothing matched, matched_transactions= has an empty value after the equals sign. Each line is newline-terminated.

Also, for the print_event predicate, the contract_identifier field accepts the literal string '*' as a wildcard that matches any contract identifier, and the contains field also accepts '*' as a wildcard that matches any payload. Both wildcards can be combined. Any other value is an exact-match string. On the payload side, the print event raw_value is a hex-encoded Clarity value. For substring matching the engine decodes it to a human-readable string representation first, then checks whether that string contains the predicate's 'contains' value. For example hex '0x0d00000010616263736f6d652d76616c7565616263' decodes to a string containing 'some-value'.

And just to make the runtime shape explicit, the engine is stateless: no database, no file I/O, no network. The adapter reads one JSON request from stdin containing both the predicate and the complete blocks array, evaluates, prints three lines, and exits.
