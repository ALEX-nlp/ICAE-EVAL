Quick follow-up after the questions that came in. On the period string side, the collapsing behavior is narrower than I remembered. A period becomes just a plain year string like '2014' in exactly two cases: when the unit is 'year', the length is 1, and the start is January 1st of that year, and also when the unit is 'month', the length is 12, and the start is January 1st of some year. A single month collapses to 'YYYY-MM'. Anything else that doesn’t collapse uses the long form '<unit>:<start>:<length>', and the trailing ':1' is dropped when length is 1 (e.g. 'year:2014-03' not 'year:2014-03:1'). This is the same idea we want in the periods domain module too, specifically for the month-unit length 12 case that starts on January 1st and canonicalizes to a plain year string, mirroring the year-unit single-length January case. On the bracket math question, if an optional rounding instruction is present for marginal-rate bracket application, the base value gets rounded first using banker's rounding (half-to-even, i.e. Python's built-in round()) to the requested number of decimal places, and then the brackets are applied. That rounding applies only to the base, not to the resulting charge.

For threshold-driven choices, the contract is that choices must have exactly len(thresholds) + 1 entries, including the band for the below first threshold case. If choices has fewer entries than bands, the missing bands return 0. If the cardinality is fundamentally invalid and we can’t interpret it, emit error="invalid_input". We also do support per-position vector thresholds, so thresholds may be a list of lists where each inner list provides per-element threshold values.

On output and errors, everything still goes to stdout as key=value lines, and every error needs to stay neutral, like error="invalid_input" or error="not_found". We should never leak a Python exception class name, traceback, or runtime message to stdout. If there’s a rejected selector or path involved, echo it alongside the error where it makes sense, for example error="invalid_input"
selector="<rejected_value>".

One more formatting thing: numbers should render compactly. Integers print without a decimal point, like 0, 1, 300. Non-integers are rounded to at most 10 decimal places. Positive infinity should render as the string "Infinity" quoted inside array literals, and negative infinity as "-Infinity", also quoted inside array literals. Array literals use JSON-style square brackets with no spaces after commas.

And for household ordering, within a household the order is first all parents in declaration order, then all children in declaration order. The role key is 'first_parent' for the first parent, 'second_parent' for the second parent, and every child gets 'child'. The legacy role number is 0 for first_parent, 1 for second_parent, 2 for the first child, 3 for the second child, and so on incrementing per child. Position within the household is just the 0-based index across the combined parents+children order.
