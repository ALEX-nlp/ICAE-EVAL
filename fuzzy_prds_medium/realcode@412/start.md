## Product Requirement Document

Hey team, we've been getting complaints from a few partner integrations that our URL handling is inconsistent — basically they're building things on top of our parsing output and keep hitting weird edge cases that break their pipelines. The core ask is: we need a proper URL parsing library (Python) that actually follows the living web standard the browser folks use, not just a regex slapped on a string. It should handle the full lifecycle: parse a string (maybe against a base URL), read individual pieces back, mutate those pieces, and re-serialize cleanly. Think of it like what we did for the config validation module — same idea where setting an invalid value should never corrupt the existing state, it just silently rejects. Also needs a lightweight 'would this even parse' probe so callers don't have to do a full parse just to validate. Query string manipulation is a big one too — partners are encoding things wrong constantly, the space/plus encoding thing keeps coming up. The multi-file structure thing is non-negotiable per the platform team's last architecture review. Oh and the I/O adapter layer needs to stay completely separate from the actual parsing logic — we got burned on that coupling before. Should read commands from stdin as JSON and write results to stdout. Reach out if you need the old browser compat reference doc.

One extra detail that came up on output formatting: each result line is `key=value` plus a newline. After a parse_url action, the very first line is always `url=valid` or `url=invalid`. After that, every get/set step just appends its own line in order — gets are `fieldname=value`, sets are `set_fieldname=true/false`, except search and hash which use `set_search=ok` and `set_hash=ok`. For search params, `entries` prints `entries` on its own line and then each `key=value` pair on its own line. `get_all` prints `get_all=N` and then each value on its own line. And the lightweight probe should just output `can_parse=true` or `can_parse=false`.

Also wanted to be very explicit on mutation behavior since that was a big source of confusion: if a set fails, the URL has to stay completely unchanged, no partial update, no half-applied state. The set returns false or the relevant failure indicator, and serializing afterward should give exactly what it gave before the attempt. That applies to host, hostname, protocol, port, username, password, and href assignments. The only exception is search and hash setters, which always succeed and return `ok`. Related to that, if a scheme change makes the current port equal to the new scheme’s default, that port should disappear from serialization — port getter becomes empty string and host getter stops including the port. So changing `http://host:443` to `https` drops the `:443` because 443 is the default port for https. And if somebody tries to switch between incompatible URL shapes, like opaque-path to special, that should be rejected.

For `search_params`, when an action comes in with an `init` string, strip one leading `?` before parsing. Then split on `&`, and split each segment on the first `=`. If a segment has no `=`, treat it as a key with an empty-string value. Those pairs then stick around for whatever later ops show up in the same steps array. On encoding, this needs to follow application/x-www-form-urlencoded rules: spaces become `+`, not `%20`, literal `+` signs become `%2B`, and `&` and `=` get percent-encoded when they appear inside keys or values. Non-ASCII text should be UTF-8 percent-encoded. Important bit here is that `get` and `get_all` return the decoded original string, not the encoded version, while `to_string` returns the encoded serialization.

Last thing, `sort` on search params should be a stable sort by key using Unicode code unit comparison, basically lexicographic by UTF-16 code units. If multiple pairs have the same key, keep their original relative insertion order. Empty keys and empty values don’t get any special treatment, they sort like any other string.
