## Product Requirement Document

Our team needs a self-contained utility library that helps application developers safely assemble and inspect the two main text-based query languages used by centralized identity services. The first concern is neutralizing special characters inside arbitrary user-supplied strings before those strings are embedded in queries or address paths — the escaping mode should be selectable so only the characters relevant to the target context are touched, and specific characters can optionally be passed through unchanged. The second concern is building search queries from a declarative list of conditions rather than hand-writing fragile text: conditions can require an exact match, a range, set membership, or substring presence, they can be combined with AND or OR logic, nested into explicit logical groups, or supplemented with verbatim pre-built fragments. Every built query must expose both a wire-safe form and a human-readable form. The third concern is working with hierarchical address strings that identify individual entries in the directory tree: the library must decompose such an address into its constituent parts and expose named scalar views of the first component and the parent path, validate whether a string is a well-formed address, compare two addresses to determine ancestor or child relationships, and support an ordered sequence of non-destructive edits such as reversing, trimming from either end, or attaching new components whose reserved characters are automatically neutralized. Finally, a parser must convert an existing query string back into a structured node tree — following the internal node representation used by the group and condition types — and reassemble it into canonical form; malformed input must surface as a dedicated error signal rather than a silent failure. The team has flagged the escaping-mode selection logic and the DN hierarchy comparison as the two areas most likely to cause subtle bugs.

One extra detail on escaping since that came up: if the 'mode' field is missing, it should default to 'all', and in that mode we escape every byte of the value, not just reserved characters. Each byte is rendered as a backslash plus its two lowercase hexadecimal digits, so 'abcdefghijklmnopqrstuvwxyz' becomes '\61\62\63\64\65\66\67\68\69\6a\6b\6c\6d\6e\6f\70\71\72\73\74\75\76\77\78\79\7a'. Also, the optional 'ignore' field is just a list of characters that should pass through untouched even if they would normally be escaped. With value 'ab' and ignore 'a', the output is 'a\62'. The ignore string is passed directly to the underlying ldap_escape function as its second argument. For the implementation touchpoints people asked me to call out, this is the EscapedValue class in src/Models/Attributes/EscapedValue.php and the mode dispatch in rcb_tests/dispatcher/Dispatcher.php (the 'escape' case calling ->dn(), ->filter(), ->both(), or leaving flags=0 for 'all').

On the query builder side, a term with a 'raw' key should be inserted verbatim into the filter with no escaping at all. Both query= and unescaped= need to show that raw string unchanged. The concrete example here is raw '(field=value)' producing 'query=(field=value)\nunescaped=(field=value)\n'. Raw fragments still join into AND/OR groupings the same way normal conditions do, so they’re not a separate path, just an unescaped one.

For dn_parts, the output format needs to stay very literal. It returns exactly nine lines in this order: 'components=<json_array>', 'rdns=<json_array>', 'values=<json_array>', 'multi=<json_array_of_pairs>', 'assoc=<json_object>', 'name=<string>', 'head=<string>', 'relative=<string>', 'parent=<string>'. The assoc object maps each lowercased attribute to an array of its values, and the 'multi' array contains [attribute, value] pairs. All JSON is encoded with JSON_UNESCAPED_SLASHES | JSON_UNESCAPED_UNICODE flags. If parent is null or empty, return an empty string, not 'null'. And if the input is malformed or not a distinguished name at all, we still return the full shape, just empty: 'components=[]\nrdns=[]\nvalues=[]\nmulti=[]\nassoc={}\nname=\nhead=\nrelative=\nparent=\n'.

For dn_validate, it reports exactly two booleans, 'valid=' and 'empty='. Important nuance there: anything invalid is also treated as empty, so if valid=false then empty=true. A bare word like 'invalid' gives 'valid=false\nempty=true\n'. A well-formed pair like 'cn=john doe' gives 'valid=true\nempty=false\n'. We should tolerate whitespace around the attribute, the equals sign, and the whole string when deciding validity.

For dn_relate, the output is exactly four lines in this fixed order: 'ancestor_of=<bool>', 'descendant_of=<bool>', 'parent_of=<bool>', 'child_of=<bool>'. Booleans are lowercase 'true' and 'false'. Identical names are neither ancestor nor descendant. Empty or non-DN inputs should return all false. Comparisons are case-insensitive for both attribute names and values. The relationship semantics are also pretty specific: parent_of=true means the second name is exactly one level below the first, meaning the first name’s components are a suffix of the second with exactly one extra component at the front of the second. ancestor_of=true is the broader any-depth version. So for 'dc=bar' vs 'cn=foo,dc=bar': ancestor_of=true, parent_of=true. For 'dc=bar' vs 'cn=foo,ou=bar,dc=bar': ancestor_of=true, parent_of=false. The code areas tied to this are the isChildOf(), isParentOf(), isAncestorOf(), isDescendantOf() methods in src/Models/Attributes/DistinguishedName.php, which compare lowercased component arrays after stripping the leading component for child/parent checks.

One last DN builder clarification: the 'reverse' step flips the order of all components, but it’s applied at serialization time rather than mutating the stored component order in place. So with 'cn=john,dc=local,dc=com' plus a reverse step, the output is 'dn=dc=com,dc=local,cn=john\n', and any later steps still operate on the original order. Also, when appending or prepending a component, reserved characters in the value should be auto-escaped using the DN escaping mode. Example: appending attribute 'dc' with value ',' should produce 'dc=\2c' in the final DN, and for 'cn=john,dc=local,dc=com' with append dc=',' the full output is 'dn=cn=john,dc=local,dc=com,dc=\2c\n'.
