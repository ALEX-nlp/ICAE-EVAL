## Product Requirement Document

Hey team, we need to wrap up that credential helper utility library we've been talking about. The core idea is that right now every time someone builds a new credential helper they're re-writing the same fiddly bits from scratch — parsing Git's wire format, figuring out config values, building auth headers, etc. We keep getting bug reports that boil down to someone getting the casing rules wrong or messing up the encoding.

Basically we need a clean multi-file library (not one giant file please, we had that problem last time with the login module) that handles all these building blocks in one place. Things like interpreting config strings the way Git does, chopping strings around delimiters, encoding credentials for headers, reading and writing those key=value blocks Git sends over stdin/stdout, and reconstructing URLs from the query fields Git passes in.

The tricky part is the library itself shouldn't know anything about JSON or how we're testing it — there should be a separate thin adapter layer that handles that translation. Each feature should have proper error signals, not just generic crashes, and the errors should be clean category names, nothing with stack traces or internal type info leaking out.

I know Priya had some notes on the case-sensitivity rules for config keys that we should follow — can someone track those down? Also need to make sure the URL scope-walking logic matches what Git actually does when it walks remotes. Timeline is pretty tight so let's make sure we nail the edge cases on the boolean parsing and the base64 encoding stuff first.

One more thing from the questions that came in: for Git-style booleans, we should treat the accepted spellings as a fixed case-insensitive set. Truthy is exactly true, yes, on, 1 and falsey is exactly false, no, off, 0. Anything else should come back as 'none' — that includes empty string, whitespace-only strings, and null/absent values. So the library should report the three-state result as 'true', 'false', or 'none'.

Also found the extra detail on config key matching. A configuration key has the form section[.scope].property. The section segment before the first dot and the property segment after the last dot are compared case-insensitively, but the scope segment in the middle is compared case-sensitively. So 'foo.example.com.bar' and 'FOO.example.com.BAR' are equal, but 'foo.example.com.bar' and 'foo.EXAMPLE.COM.bar' are not. Same rule everywhere we do Feature 4 key comparisons: section and property are case-insensitive, scope is case-sensitive.

On error handling, let's keep it really strict and boring: errors come back as one normalized line only, in the format 'error=<category>' and nothing else. The categories we need are 'argument_null' for a required string input that is absent or null, 'out_of_range' for an integer that fails a range constraint, 'unknown_op' for an unrecognized operation, and 'invalid_input' for other invalid input conditions. We should never leak host-language type names, stack traces, or runtime-generated message text into that output.

For the URL scope walking, the order is most specific to least specific. Start by unfolding the path right-to-left one component at a time, and make sure query strings and fragments are dropped and trailing slashes are trimmed. After that, unfold the host left-to-right by removing one leading sub-domain at a time, and stop before emitting a bare top-level domain, meaning a host with no dots. For example, 'http://buzz.foo.example.com/bar/baz' produces five scopes: the full URL with path, then with just /bar, then host-only, then foo.example.com, then example.com.

And for the URL-safe base64 bit, byte value 62 maps to '-' and byte value 63 maps to '_' instead of the standard '+' and '/'. If padding is enabled, keep trailing '=' so the output length is a multiple of four. If padding is disabled, strip all trailing '='. An empty input produces an empty output string.
