## Product Requirement Document

Hey team, we need to build out this scenario runner tool we've been talking about. Basically it should let devs write plain English test specs and have the system run them and tell you what passed, what broke, what's not done yet, etc. Think of it like that BDD thing we were experimenting with last quarter — same general vibe.

A few things I know we need: it should handle shared setup blocks that run before each test, support table-driven tests where one template runs multiple times with different data, and let people filter which tests to run by labels or by name or even by pointing at a specific line in a file. Oh and there's some kind of safety valve where if a label is overused (more than a set number of times) the whole run should stop with a clear error before anything even executes.

There's also a mode where nothing actually runs but you still get the report (useful for sanity checking), a strict mode where incomplete steps count as failures, and a command that suggests code stubs for steps that don't have implementations yet. We also need the keyword lookup thing for different languages — the team mentioned something about reusing the approach from the i18n module we did before.

Please make sure the codebase is split into sensible modules, not one giant file. The JSON test harness should be a thin wrapper, not mixed into the core logic.

One extra pass on the behavior details since a few good questions came up. A scenario’s status should come from the worst step outcome in this order exactly: failed > undefined > pending > skipped > passed. So if any step fails, that scenario is failed. If nothing failed but at least one step is undefined, the scenario is undefined. Same idea the rest of the way down.

At the run level, the line should be 'result=ok' when no scenario failed, and undefined/pending are still acceptable there when strict mode is off. It should be 'result=failed' when at least one scenario failed, or when strict mode is on and any undefined or pending step exists. Use 'result=error' for config-type problems, like the features source not being there or the tag cap getting exceeded before execution starts. Related to that, if there are no feature source files at all, whether that’s an empty files list or no features directory, the engine should not run anything and should report error=missing_features_directory followed by result=error.

A little more color on selection too: the --name filter is just a substring match, and it should check feature names, scenario names, scenario outline names, and examples block names. If a scenario sits under anything that matches at any of those levels, include it. Only the matching scenarios should show up in the report and tallies. Also for path:line selection, the line can point anywhere inside the scenario body, not only the header. The engine should figure out which scenario covers that line and run only that one, and sibling scenarios from that same file should stay out of the report and tallies.

For the snippets side, the 'snippets' command should emit one line per undefined step phrase. The line format is 'snippet expression=<parameterised pattern>' optionally followed by 'args=<arg names>'. Literal integers in the phrase become {int}, quoted substrings become {string}. If the step has a data table argument, 'args=table' is appended. Steps with no parameters or table have no args field.

On the language lookup bit, the 'i18n_keywords' command takes a 'lang' code and returns the natural-language keywords for that language covering feature, background, scenario, scenario_outline, examples, given, when, then, and, but, plus bare code forms for each step keyword. If the language code is unknown, return error=invalid_language followed by language=<code>. This is the same dedicated lookup module idea we talked about for the i18n_keywords command in feature11_i18n_keywords.json — basically a clean mapping from codes like 'pt' and 'en' to the structured keyword lists for feature/background/scenario/given/when/then/and/but. And just to be explicit on the source format we’re parsing, it’s the Gherkin-style plain-language scenario format with Feature/Background/Scenario/Scenario Outline/Examples blocks and Given/When/Then/And/But steps, as defined in the PRD and parsed by the feature parser module.

Last thing, I want to be really firm on the shape of the code. The implementation should be split into distinct modules with one job each: source parsing, scenario compilation/expansion, selection/filtering, step matching, execution, result classification, report formatting, snippet suggestion, and i18n keyword lookup. Then a thin adapter handles JSON I/O. The core engine needs to work as a library on its own, with no awareness of JSON, stdin/stdout, or any wire format.
