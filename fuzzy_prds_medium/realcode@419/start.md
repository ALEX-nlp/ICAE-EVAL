## Product Requirement Document

Hey team, we need to wrap up the numerical toolkit library we've been building. Most of the core math pieces are done but we need to make sure the whole thing is wired up properly end-to-end so someone can actually run it. The execution layer needs to read JSON from stdin, route each command to the right piece of logic, and write the result back out — basically the same pattern we used on the recommendation service adapter last quarter, so follow that same separation philosophy.

A few things I keep hearing from devs who tried early builds: error messages are leaking internal stack traces which is a no-go for any external-facing tool, and some of the numeric output has inconsistent decimal formatting (trailing zeros showing up in some places, missing in others). Also the matrix printing for multi-output operations needs to be consistent — comma-separated on one line per row.

One thing that came up in last week's sync: the KD-tree query feature needs to support at least three distance strategies and two splitting approaches — make sure those are all hooked up. Similarly the learning rate schedules each have their own formula so don't accidentally collapse them into one generic thing.

The whole thing should be organized into separate files by concern — no monoliths. Each feature family gets its own module. Adapter stays separate from math logic. If something is invalid input, return a clean category-based error string, nothing else.

Quick follow-up from the questions that came in. On formatting, let’s keep it really strict and boring: integers print without a decimal point, so score=0 and score=1296, and fractional values print with trailing zeros trimmed, so score=0.888889, not score=0.8888890. Each logical record ends with a newline, and matrix cells stay comma-separated with one row per line. Same cleanup rule applies anywhere we’re printing numeric matrix output, including the intercept shaping case where scale=2.0 prints as 2 and scale=0.0 prints as 0.

On errors, every failure path should collapse to one neutral line only, in the form error=<category>. That means things like error=invalid_matrix_columns, error=out_of_range, or error=invalid_argument are fine, but no host-language exception types, no stack traces, and no runtime messages ever showing up in output. Also, an empty sequence or zero-length universe is rejected with error=invalid_argument, and any index or targetId outside the column range returns error=out_of_range.

A couple output-specific details too. For vector-type values, the distribution key is built by joining the numeric components with a pipe character '|', like 0|0|1=0.4. For number and string types the value is printed as-is, and output entries are sorted in ascending key order. Tree impurity output uses error_value= rather than score=. If we’re given a stump as a list of child nodes, the result is the row-count-weighted average of the per-node impurities using sum(node_size * node_impurity) / total_rows, and empty child nodes are ignored.

For the KD-tree piece, yes, all three distance metrics need to be wired: euclidean, manhattan, and cosine. Both split strategies need to work too: inOrder and largestVariance. inOrder cycles through dimensions in order, and largestVariance splits on the dimension with the highest variance among current node points. The leafSize parameter is what controls when a node becomes a leaf. Output there is the zero-based indices of the k nearest points, ordered nearest-first, comma-separated.

For the target column split behavior, duplicate target indices should be de-duplicated before processing, so only unique indices count as target columns. The output format is a 'features:' header line, then the feature rows using all columns not in targetIndices, then a 'target:' header line, then the target rows using columns at targetIndices in original index order. If fitIntercept is false, return the matrix unchanged. If fitIntercept is true, prepend a new first column filled with the given scale value to every row.

And on the gradient call, the formula is -2 * x^T * (y - x*w). The output is one component per line, meaning a column vector with one entry per feature, not per observation. No prefix label there, just the raw numeric values one per line with trimmed formatting.

Last thing, just reinforcing the structure expectation since a few people asked: this absolutely should not turn into one god file. Each family of primitives — metrics, link functions, kernels, optimization, schedules, cross-validation, impurity, retrieval, data shaping — should live in its own cohesive module. The JSON adapter that reads stdin and writes stdout stays separate and should contain no numerical logic. Core numerical logic needs to stay completely decoupled from I/O and JSON parsing, with the adapter only translating each JSON command into calls into the core library.
