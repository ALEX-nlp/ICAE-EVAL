## Product Requirement Document

Hey team, we need to wire up the OGNL expression evaluation runner for the upcoming demo. Basically we want a single entry point that reads a JSON blob from stdin, figures out what fixture to load (we have a handful of pre-baked ones — simple, sample, the array ones, list, etc.), evaluates the expression against it, and prints the result in that normalized two-line format we settled on. If there's a set_to field we should do the before/after thing like we discussed in the retro. Also make sure variables get seeded into the eval context before we run. For error cases just print the category label, no stack traces or anything noisy. One thing I keep forgetting to mention: the output format for numbers needs to match whatever we agreed on for the int-vs-float distinction — I think it's the same rule we used in the last formatter we shipped, so just check how that handles trailing zeros and the dot. Also the map result ordering was a pain point in QA last sprint so double-check that. The fixtures themselves need to match the sample graph from the existing test expectations, specifically the nested map structure, the disabled flag, the array contents, etc. We want the whole thing runnable as a standalone Java program fed JSON on stdin.

One small follow-up from the questions that came in: for number formatting, we do need to be pretty literal about the int-vs-float split. Whole numbers print without a decimal point, so result=11 and result=256, while floating-point values always keep a decimal point and at least one fractional digit, so result=2.5 and result=1.0. If the source literal is 1. then the output still needs to be 1.0. And we should never print extra trailing zeros beyond one fractional digit unless they’re actually significant.

Also, for map output, please keep keys sorted in ascending lexicographic order every time we serialize a map result, including map construction results. The example to anchor on is #{"foo", "bar" : "baz"} rendering as {"bar":"baz","foo":null}, so 'bar' comes before 'foo'.

A couple fixture specifics to pin down too. The supported fixture names are none, simple, sample, string_array, int_array, and list. none is a null root. simple is a SimpleObject with isDisabled()=true, isThisVarArgsWorking()=true, getValueIsTrue(boolean)=passthrough. sample is the root object with a 'map' property containing key 'size'=5000, key 'array'=int[]{1,2,3,4}, key 'list'=List containing elements [someObject, anotherObject, int[]{1,2,3,4}, 99, null], and [^] gives first=99... wait — map[^] gives 99 meaning the first key resolves to 99, map[$] gives null as the last entry; that same root also has 'disabled'=true. string_array is String[]{"hello","world"}, int_array is int[]{10,20}, and list is a java.util.List with two elements, iterable.

And just to make the set flow totally explicit: when set_to is present, we first evaluate the expression as-is and print before_kind=<kind> and before=<value>, then parse the set_to string as an OGNL expression for the new value and assign it, then evaluate the original expression again and print after_kind=<kind> and after=<value>. In that mode we do not use kind= or result= at all.

For errors, we’re keeping it super strict and quiet: on any parse/syntax failure or evaluation exception that indicates a malformed expression, print exactly error=syntax\n and nothing else. No stack trace, no message, no extra line. This is the same behavior we want to match from the expected outputs in feature1_literals.json, feature2_arithmetic_logic.json, and the map ordering example in feature6_composite_construction.json.
