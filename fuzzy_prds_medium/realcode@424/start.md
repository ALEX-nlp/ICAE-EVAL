## Product Requirement Document

hey team, we need to wire up the observability + i18n layer for the todo app we've been building. basically three things: (1) there's no consistent way right now to see what's happening inside the state machines — every dev is just slapping print statements everywhere and the logs look completely different, we need one unified thing that handles both the 'something changed' case and the 'something broke' case; (2) the app title and language stuff is all over the place, different screens are doing their own checks and some of them are accepting languages they shouldn't — remember how we handled the 'supported locale' logic in that flutter bloc example project we referenced a while back? just do it the same way, static strings, no dynamic reloading needed; (3) we keep copy-pasting the spinner widget, just make it a proper reusable component that centers itself and can be identified in the widget tree. the output formats for all of these need to be exact because we're running automated checks against them. i don't have the exact string formats in front of me but the test runner will validate everything. keep the code organized — not one giant file please. adapter layer should be separate from the actual logic. let me know if anything is unclear

one quick follow-up since a couple details came up. for the state-machine observer piece, the transition line needs to be exact: `Transition { currentState: <current>, event: <event>, nextState: <next> }` and then exactly one trailing newline. the `<current>`, `<event>`, and `<next>` parts are just the string representation of those values, and there shouldn’t be any extra prefix, suffix, timestamp, label, or anything else wrapped around it. same idea for errors: when the observer gets an error, emit the error’s own string representation verbatim as a single line with no prefix, suffix, or wrapping, just the raw message followed by exactly one trailing newline.

on the localization side, please lock the display title to `Bloc Library Example`, and the output for that should be exactly `app_title=Bloc Library Example` with a trailing newline. also, since these strings are static and never change at runtime, the delegate should always say no reload is needed, unconditionally, with output exactly `should_reload=false` followed by a trailing newline. for supported locales, use the same core check as the bloc library example app’s `AppLocalizationsDelegate` in the core localization module: `isSupported` should check `locale.languageCode == 'en'`. that means support is granted if and only if the language code is `en`, regardless of region/country code, and anything else should end up as `supported=false`. when echoing that result, print the locale label as `language=<language>` if there’s no country code, or `language=<language>_<country>` if there is one, then on the next line print `supported=true` or `supported=false`, with each line followed by a newline. and just to be explicit, the delegate’s `shouldReload` method always returns `false` because all localized strings, including `Bloc Library Example`, are compile-time constants.
