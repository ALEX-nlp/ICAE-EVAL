## Product Requirement Document

Hey team, we need to build out that source quality analysis toolkit we've been talking about. The idea is basically a library + command adapter that devs can use to run various checks on Dart source files and get back consistent, machine-readable output. Think of it like that pipeline we wired up for the login module's validation layer — same kind of separation between the transport layer and the actual business logic.

High level: the adapter reads JSON commands from stdin, dispatches to the right core behavior, and prints results to stdout. We need support for a handful of things: converting identifier strings into different casing formats, normalizing file paths and URI inputs, parsing YAML config blobs, mapping severity labels, detecting suppression comments in source, picking the right report renderer by name, and running several source analysis rules (empty blocks, magic numbers, duplicate arguments, null assertion abuse, boolean literal comparisons, first-element access patterns, trailing commas, and throw-inside-catch).

The suppression comment thing is a bit tricky — we need both file-wide and line-local flavors, and I recall there was some case-sensitivity nuance we handled differently in the old linter prototype, so please double-check how identifiers in comments are matched.

Output format needs to be totally deterministic so the test harness can do exact string comparison. Multi-file structure is expected given the complexity here. Please make sure the core logic is not tangled up with the I/O layer.

A couple extra specifics from the follow-up thread so nobody has to guess. For the casing helpers, camel_text only splits camelCase tokens on uppercase letter boundaries and inserts spaces. If the input is already snake_case or kebab-case and there are no uppercase letters mid-word, return it unchanged. So snake_case → camel_text=snake_case, kebab-case → camel_text=kebab-case, and only something like camelCaseString turns into split output like camel case string. Also, kebab is much narrower than camel_text: it is strictly a snake_case-to-kebab-case transform, so it just replaces underscores with hyphens. It does NOT split camelCase. That means camelCaseString → kebab=camelCaseString, and snake_case_string → kebab=snake-case-string. The capitalized field is its own separate operation and only uppercases the first character of the raw input.

On the magic number behavior, 0, 1, and -1 are always exempt without any config since they’re common sentinel values. We also ignore numbers in named constant declarations like const x = 5, variable initializer declarations like final/var x = 500, DateTime and DateTime.utc constructor arguments, Duration constructor arguments, const collection literals like const [32, 12] and const {'key': 3}, const constructor arguments like const Bar(5), and list literal initializers like [100, 200, 300]. The only things that should get flagged are numbers directly embedded in executable expressions, so arithmetic, comparisons, and function-call arguments outside those exempt cases.

For duplicate arguments, the rule output name is no-equal-arguments and severity is warning. Named parameter duplicates should use the issue text in the format 'paramName: value', for example lastName: user.firstName. Positional duplicates should just use the argument expression text by itself. Also, literal arguments are not reported here, so if the duplicate is just an identical string or number literal, skip it — this one is only for non-literal expression arguments.

For null assertions, each ! in a chain is its own issue, but the text grows with the chain. So object! produces one issue with text=object!, and object!.field! produces a second issue with text=object!.field!. Both of those should be reported at the same starting line and column where the chain expression begins. One special case to keep excluded: map['key']! is NOT reported.

On the boolean literal comparisons, the replacements are fixed. expr == true → replacement=expr. expr != true → replacement=!expr. true == expr → replacement=expr. false == expr → replacement=!expr. false != expr → replacement=expr. expr != false → replacement=expr. The replacement_comment is always either 'This expression is unnecessarily compared to a boolean. Just use it directly.' or '...Just negate it.' depending on whether negation is needed. And if the boolean expression is nullable, like anything using ?., don’t report it.

For trailing commas, this only applies to multi-line constructs: function parameter lists, argument lists, enum value lists, and collection literals that span more than one line. Single-line declarations are never flagged. The issue location should be the last item, not the closing bracket, and the replacement is that item text with a comma appended. The rule output name is prefer-trailing-comma and severity is warning.

A few more rule-specific details: for throw-in-catch, the output rule name is avoid-throw-in-catch-block and the severity is warning. Only throw statements directly inside a catch or on-catch block body should be reported. A bare throw, meaning rethrow behavior, is not the same thing as throw SomeException(), and the rule is only about explicit throw expressions inside catch handlers that lose the original stack trace context.

For first-element access, both elementAt(0) and [0] should replace the whole access with .first. The issue text is elementAt(0) for that call form and [0] for index access. The rule output name is prefer-first and severity is style. This applies to Iterable, List, Set, and Queue, including DoubleLinkedQueue, HashSet, LinkedHashSet, and LinkedList.

And last thing, on the architecture bit from the PRD, the JSON adapter on stdin/stdout needs to stay completely decoupled from the core domain logic. Same separation we talked about before, with DIP and SRP followed cleanly so the transport layer doesn’t leak into the actual rule behavior.
