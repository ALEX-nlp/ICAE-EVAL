## Product Requirement Document

Hey team, we need to build out this Bitcoin chain-state and transaction indexing toolkit we've been talking about. The core idea is that developers should be able to plug in their block/transaction observations and get back clean, queryable results without having to write all that painful reconciliation code themselves. Think of it like the wallet-sync plumbing we did for the Lightning stuff last quarter — same philosophy, just applied to UTXO tracking and chain state.

The main things we need covered: handling chain tip updates including when blocks get reorganized out, inserting individual checkpoints, figuring out how much value is flowing in/out for scripts we care about, tracking whether a script has been 'used' yet, computing fees, resolving conflicts when we have both a loose output record and a full transaction for the same coin, tracking which tx spent which output, deriving scripts in a lookahead window, and finally filtering down to just the canonical transaction history when there are conflicting spends — confirmed vs mempool, newer vs older, that kind of thing.

One important thing: the I/O adapter layer that reads commands and writes results should stay completely separate from the actual business logic — we burned ourselves last time by mixing those together. Also please keep the file structure sensible, not everything crammed into one giant file. Language is Rust. Tests are in the rcb_tests folder if you need to cross-reference expected behavior.

A couple follow-ups from the questions that came in. On chain updates, if an update can’t be connected because we’re missing the checkpoints needed in between, the output needs to be exactly two lines: `error=cannot_connect` and then `try_include_height=<N>`, where N is the lowest height that has to exist to make that update connectable. So if somebody sends a disjoint update starting at height 3 while the chain is empty, that should come back as `try_include_height=0`.

For reorgs, the changeset needs to show both sides of what happened, not just what we ended up with. Removed checkpoints should be emitted as `<height>:<removed>` and that literal `<removed>` string is the hash placeholder. Anything new or replacing something shows up as `<height>:<hash>`. That whole list is comma-separated inside square brackets and ordered by height ascending. Then the final `chain=` line should only show the checkpoints that actually survive after the removal. Also related here, if an update happens to agree with the existing chain at two different points, like both chains sharing genesis and tip, the changeset should only include the actually newly added checkpoints. The final chain still includes ALL checkpoints, meaning the old ones that were already there plus the new ones, sorted by height, and matching checkpoints that already existed should not get repeated in the changeset.

On single checkpoint insert, if we try to insert at a height that already exists but the hash is different, we should not mutate anything at all. The output there needs to be `error=conflicting_block`, `height=<N>`, `original_hash=<existing_hash>`, `update_hash=<attempted_hash>`, and `chain=[<unchanged>]`. Chain state stays exactly as it was.

For canonical conflict handling, the priority order is important. A confirmed transaction on the active chain always beats any conflicting unconfirmed transaction. If both conflicting transactions are unconfirmed, the more recently-seen one wins. The one extra wrinkle is that if a transaction has unconfirmed descendants, that whole branch is treated as canonical even if that parent is older than the conflicting transaction that has no descendants. Anything non-canonical, plus its descendants, should be left out of `chain_txs`, `chain_txouts`, `unspents`, and the balance math.

Balance reporting also needs to stay split into the four buckets we discussed: `balance_immature` for coinbase outputs not yet matured, `balance_trusted_pending` for unconfirmed outputs from transactions we sent ourselves, `balance_untrusted_pending` for unconfirmed outputs from external unconfirmed transactions, and `balance_confirmed` for outputs in confirmed blocks. Only canonical unspent outputs should count toward any of those buckets.

On next unused script selection, we should prefer filling the lowest unused gap before deriving anything new. So if 0-15 are used, 16 is unused, and 17-26 are used, `next_unused` should return `index=16` with `is_new=false` and `changed=0`. We only derive a brand new script once every previously revealed index is already used, and that’s when it should return `is_new=true` and `changed=1`.

And just to restate the architecture part because a few people asked what “separate” really means here: the execution adapter in `src/adapter/` (or equivalent) should be the layer that turns JSON/string commands into normal Rust calls into the core domain, with stdin/stdout parsing kept completely out of the business logic in `src/domain/` or `src/core/`. Same idea for dependencies in general: the high-level chain/tx pieces should depend on abstractions (traits), not concrete I/O implementations, in the same spirit as BDK’s existing chain-source abstraction.
