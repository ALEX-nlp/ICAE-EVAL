## Product Requirement Document

Hey team, we need a numeric utility library built out — basically a toolkit for doing low-level integer and fractional math in a portable, predictable way. Think of it like that bit-manipulation helper we shipped for the embedded project last quarter, but way more comprehensive. The big ask is: developers shouldn't have to hand-roll their own code every time they need to do things like inspect bit patterns, shift bits around cyclically, work with fractions exactly (no floats mucking things up), or deal with fixed-precision decimal-ish values. There are also some edge cases around what happens when numbers go out of range — we want that behavior to be explicit and configurable rather than just silently wrong. The library needs to be structured cleanly so it's not just one giant file, but also don't go overboard if something is genuinely simple. It should be testable via a CLI adapter that reads from stdin and writes to stdout. The test harness should be runnable with a single bash command and output per-case result files in a specific directory structure so CI can compare them easily. For the fraction stuff specifically, make sure you look at how we handled the normalization vs. canonicalization distinction in the older math module — same philosophy applies here. Word count and output formatting must match the spec exactly or the automated comparisons will fail.

One other thing the team asked about on the rotate behavior: negative shift distances should just mean rotate the opposite way from whatever direction was asked for. So if direction=left and shift=-5, that’s effectively 5 positions to the right. Shift distances are interpreted modulo the bit width, so on an 8-bit value a shift of -5 is equivalent to a right rotation of 5, or left rotation of 3. Same deal for large positive values: a shift of 4090 on an 8-bit value means 4090 mod 8 = 2, so it rotates by 2 positions in the specified direction. In both cases, please keep echoing the original input exactly in the output, so shift=-5 stays shift=-5 and shift=4090 stays shift=4090 rather than swapping in the normalized version.

Also, for the bit boundary helpers, if the input is already an exact power of two, bit_ceil should return that same value. So bit_ceil of 4 is 4, bit_ceil of 128 is 128. That’s because bit_ceil is the LEAST power of two not less than the value. Same idea for bit_floor: for an exact power of two it also returns the same value. And whenever we print hex results, keep them as lowercase with a lowercase 0x prefix, like result_hex=0x2 or result_hex=0xa0, never uppercase.

For fraction normalization, the distinction we wanted is that reduce divides numerator and denominator by their GCD but keeps the sign placement exactly where it already is. So if both are negative, both stay negative after reduction: -6/-3 becomes -2/-1, not 2/1. Canonicalize is the one that also moves the sign into a stable canonical position, typically the numerator. Absolute magnitude should return a non-negative numerator while still preserving the denominator shape used by that operation.

And just to make the arithmetic expectation explicit, fraction arithmetic results are NOT implicitly simplified or canonicalized. We return the raw arithmetic result as-is. For add, use numerator = (left_num * right_den) + (right_num * left_den), denominator = left_den * right_den. So 1/3 + 2/3 = (1*3 + 2*3) / (3*3) = 9/9. If something happens to come out reduced already, fine, but don’t do extra cleanup unless the caller asked for it. Use the fraction_normalize command explicitly to reduce. This is the same Feature 5 (fraction_normalize) behavior from the PRD, including the key example that under reduce, -6/-3 → -2/-1 (not 2/1).
