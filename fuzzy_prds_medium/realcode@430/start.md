## Product Requirement Document

Hey team, we need to wrap up the Markdown processing toolkit we've been scoping out. The core idea is that devs should be able to throw Markdown text at it and get back HTML, plain text, or structured tree info without wiring up their own parsers or regex nightmares. We talked about this in the last planning sync — basically the same philosophy as what we did with the config sanitizer, where the adapter layer keeps all the messy I/O stuff away from the actual business logic.

Some specific things that came up: people need to be able to turn on/off individual syntax extensions (tables, strikethrough, that kind of thing), handle fancy quote/apostrophe conversion for publication-style output, and also do things like walk the document tree or poke at individual node properties. There was also a request from the content team about being able to edit the tree before rendering — like inserting or moving chunks around programmatically.

One thing I keep hearing from the dev side is that error messages right now are too implementation-specific and break when we switch runtimes. We need errors to come back in a normalized, language-neutral format so the frontend can handle them consistently regardless of what blows up underneath.

The input/output contract needs to be stable and JSON-based for the adapter layer. Make sure the round-trip serialization story is solid too — that one came up in QA last sprint. Roughly 13 features total per the scope doc.

One quick follow-up from the questions that came in: the adapter should read a single JSON value from stdin, figure out where to send it based on the input shape, and write the result to stdout. If that incoming value is just a plain JSON string and not an object, we treat it as raw Markdown text and render straight to HTML with default options, no extensions, no flags. If it is a JSON object, it needs at minimum a 'markdown' key. If the input type is neither a string nor a valid object with a markdown key, return error=invalid_input_type. Same general split as the config sanitizer adapter from the previous sprint: all JSON parsing and output formatting stays in the adapter layer, the core exposes idiomatic Ruby methods, and the core Markdown engine should know nothing about JSON or IO. Same Dependency Inversion Principle pattern too, with the adapter layer in src/adapter/ or equivalent delegating to pure domain objects in src/core/, and core business logic must never import or reference JSON parsing or IO modules directly.

Also clarifying a couple option details. Hard line break rendering is controlled by the HARDBREAKS render option, which maps to :hardbreaks in the Ruby layer. When that is on, a newline inside a paragraph renders as a self-closing <br /> tag followed by a newline instead of getting collapsed into a space. The expected shape is exactly what feature2 shows: <p>foo<br />
baz</p>
. Smart punctuation is different: that is a PARSE-time option, enabled with the SMART parse flag and `:smart` in Ruby. It converts straight double quotes to curly “”, straight single quotes/apostrophes to curly ‘’, and nested quote patterns accordingly. It must NOT be passed as a render flag. If somebody does that, that is invalid input and should come back as error=invalid_input_type with selector=html_parse_flag_not_allowed.

For tree inspection and edits, here are the supported node attribute kinds and fields the team asked about. 'link' exposes url and title (new_url to update); 'image' exposes url and title (new_title to update); 'heading' exposes level (new_level to update); 'list' exposes list_type (bullet or ordered), start (ordered start number), and tight (boolean); 'code_block' exposes fence_info string. The output format there is 'key=value' lines, followed by 'updated_key=new_value' lines for any updated fields. On the mutation side, the supported values for the 'edit' key are 'insert_empty_paragraph_before', 'insert_empty_paragraph_after', 'prepend_text', 'append_text', 'replace_emph_text', and 'unwrap_emph'. More specifically, 'insert_empty_paragraph_before' inserts a new empty paragraph node before the first paragraph in the document; 'insert_empty_paragraph_after' appends a new empty paragraph node after the first paragraph; 'prepend_text' prepends an inline text node to the first paragraph's children; 'append_text' appends an inline text node; 'replace_emph_text' replaces the text inside the first emph node; 'unwrap_emph' removes the emph node and hoists its text child into the parent. After mutation, the full document is rendered to HTML.

On errors, the adapter should always normalize them into key=value lines and never throw exceptions or send back JSON error objects. It is always exactly three lines: 'error=<category>', 'context=normalized_errors', 'selector=<input_selector_value>'. Right now the known category is 'invalid_input_type' for wrong input type, a flag used in the wrong position, or a disallowed option. The selector line should echo the 'attempt' value from the input object that triggered the error. No stack traces, no language-specific error text.

And just to lock the scope wording down, this still covers Features 1 through 13 as defined in start.md: basic HTML, hard breaks, smart punctuation, extensions, fenced code metadata, footnotes, task lists, plain text, round-trip serialization, tree traversal, node attributes, tree mutation, and normalized errors — each corresponding to a public test case file in rcb_tests/public_test_cases/.

A few exact details to be precise about: The execution adapter reads a single JSON value from stdin, routes it to the appropriate core domain method based on input shape (string vs object, presence of 'view', 'edit', 'kind', 'extensions', 'attempt' keys), and writes the result to stdout. The core Markdown engine must have zero knowledge of JSON or IO. All JSON parsing and output formatting lives in the adapter layer. The core exposes idiomatic Ruby methods. This is the same separation pattern used in the config sanitizer adapter from the previous sprint.
