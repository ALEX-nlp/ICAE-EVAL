## Product Requirement Document

Hey team, we need to wrap up the markdown rendering library integration. The core engine is mostly there but we need a proper adapter that reads JSON from stdin and writes results to stdout — basically the same pattern we used on that content pipeline project last quarter, you know the one. The adapter needs to handle all the rendering modes we've talked about: the default HTML output, that normalized re-serialization mode, and the XML tree dump. Extensions should be toggleable per-request too — things like tables, checkboxes on list items, the strikethrough/superscript stuff, auto-detected links, footnotes, that anchor-on-headings thing, and definition lists. There's also some safety stuff around raw HTML passthrough that the security team flagged — by default it should be suppressed, but callers can opt in. Oh and there's a tag filtering layer on top of that when unsafe mode is on. Smart punctuation (the curly quotes / em-dash thing) lives on the parse side not the render side, just FYI. Bullet marker style and line-wrap width are configurable for the normalized output mode. If anything goes wrong parsing the request or running the render, we should still return a well-formed JSON error rather than crashing. The whole thing needs to stay decoupled — the business logic shouldn't know anything about stdin/stdout.

One small follow-up from the questions that came in: on the stdin/stdout side, the adapter should read a JSON object from stdin with fields `markdown`, `format`, `extensions`, `parse`, and `render`, then write a JSON object to stdout with an `output` field (or an error field on failure). Also, if `format` is absent, we should just treat that as normal HTML rendering. Only the explicit values `"commonmark"` and `"xml"` switch to those modes.

On the raw HTML piece, the default is still to suppress both raw HTML blocks and inline HTML. If someone wants that content to pass through, they need to explicitly set `render.unsafe_html` to `true` in the JSON input. Separate from that, `render.tag_filter` is its own opt-in. If that is enabled alongside `render.unsafe_html: true`, we still escape this specific dangerous set of tag names: script, plaintext, iframe, textarea, style, xmp, title, noembed, noframes, noscript, listing.

For XML mode, when `format` is `"xml"`, the output starts with `<?xml version="1.0" encoding="UTF-8"?>` and `<!DOCTYPE document SYSTEM "CommonMark.dtd">`, then a `<document xmlns="http://commonmark.org/xml/1.0">` root. Nodes are indented with 2 spaces. Source position attributes are not automatic there; they only show up when explicitly requested with `render.sourcepos: true`.

A couple rendering specifics too: with `description_lists`, terms followed by colon-prefixed definition lines should render as `<dl>` with `<dt>` and `<dd>`, and the opening `<dl>` tag plus the first `<dt>` should be on the same line, so no newline between them. Inline markup inside both terms and definitions should still parse normally. For tables, cells with no alignment marker should have no `align` attribute at all. If the delimiter is `:---:` that means `align="center"`, `:---` means `align="left"`, and `---:` means `align="right"`, and that applies to both `<th>` and `<td>` cells.

Also confirming the heading anchor behavior: the `header_id_anchors` extension should prepend each heading with an `<a>` anchor element, using `user-content-` prefixed ids and bare-slug hrefs, and it should dedupe collisions with numeric suffixes.

And just to underline the failure behavior, if the JSON input is malformed or a required field is missing or invalid, the adapter should return a well-formed JSON error object rather than panicking or writing partial output. In other words, surface it as a structured response, not a raw process crash.
