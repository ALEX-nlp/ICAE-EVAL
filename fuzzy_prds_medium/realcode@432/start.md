## Product Requirement Document

Hey team, we need to build out that cloud storage adapter library we've been talking about. Basically devs are tired of writing the same glue code every time they wire up a new storage backend — parsing connection strings, dealing with encrypted credentials, figuring out what capabilities a driver exposes, validating config maps before they blow up at runtime, etc. It's a mess and every team does it differently.

The library should cover the main integration points: parsing those listener address strings (you know the ones, like the format we standardized after the infra incident last quarter), doing encrypt/decrypt round trips for secrets (similar to how we handled it in the auth module, ask Priya if you need context), checking driver identity and what operations it supports, validating option maps with all the conditional/dependency logic, building volume sources from shared filesystem responses, handling access grants over HTTP, and parsing those user agent flags from CLI args.

For errors, we want consistent normalized category names — no raw exception messages leaking out. Output should be stable and deterministic so it's testable. The library needs to be structured properly, not just one giant file — separate concerns, keep the I/O adapter layer away from the business logic. Think production-grade repo layout.

We'll need this to pass the integration test suite in rcb_tests. Ping me if anything's unclear but please check the existing test files first before asking obvious stuff.

One small follow-up after the team questions came in. On the endpoint parsing piece, we’re only supporting `unix://` and `tcp://`, and that check is case-insensitive on the scheme. The separator has to be exactly `://`, so a thing like `unix:/` is invalid, and the address after `://` has to be non-empty. If the separator is wrong like `:/`, the scheme is missing, the address is empty, or the scheme just isn’t supported, return `error=invalid_endpoint` and then `endpoint=<raw_input>` on the next line. On successful parses, the protocol field should preserve whatever original casing came in, so if somebody passes `UNIX`, it stays `UNIX`.

On the secret round-trip behavior, this is AES-CBC with a 32-byte (256-bit) key, PKCS7 padding, and a random IV prepended to ciphertext. The important contract here is not the raw ciphertext bytes, because with IV/padding that output is intentionally non-deterministic. What we care about in stable output is decrypting the generated ciphertext and printing `plaintext=<recovered>` and `ciphertext_len=<length>`. For a 10-character input like `mypassword`, the ciphertext length is 32. If `corrupt_ciphertext` is true, the adapter should corrupt the generated ciphertext before decrypting it, and in that case the output should be only `error=invalid_ciphertext`.

For plugin/driver info, the driver name is `cinder.csi.openstack.org` and the vendor version is `1.1.0`. For plugin info, print `name=` first and then `vendor_version=`. For capabilities, all six named ones should return `supported=true`: `unknown`, `publish_unpublish_volume`, `create_delete_volume`, `list_volumes`, `create_delete_snapshot`, and `list_snapshots`. The capability output order is `supported=true` first and then `capability=<name>`.

Also spelling out the validation rules a little more since there were questions there. There are eight named rule sets: `required`, where field `a` is required; `optional`, where field `a` is optional and outputs empty string if absent; `required_when_a_is_FOO`, where `b` is required only when `a` equals `FOO`, and both `a` and `b` should always appear in output; `optional_when_a_is_FOO`, where `a` is always required and `b` is optional only when `a` equals `FOO`, which means `b` is required when `a` is not `FOO`; `default_FOO`, where `a` defaults to `FOO` when absent; `depends_on_exactly_one_of_a_or_b_and_c`, where if `d` is present then `c` must be present and exactly one of `a` or `b` must be present, with both present being an error; `precludes_a_or_b_when_c_present`, where if `c` is present then neither `a` nor `b` may be present; and `boolean_text_pattern`, where `a` must match a boolean text pattern and accepts `true`/`false` but rejects other strings. For successful validation output, print `valid=true` and then all fields in stable alphabetical key order with their values, using empty string for absent optional fields. For failures, print `valid=false` and then `error=<category>`.
