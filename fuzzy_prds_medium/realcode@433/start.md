## Product Requirement Document

Hey team, we need to build out that container image catalog toolkit we've been talking about. The basic idea is that release engineers should be able to describe their image families in a declarative way and have all the derived strings computed for them automatically — things like how we label architectures for different contexts (tags vs package filenames vs display), how we show OS names in docs, how we stamp build IDs so patch noise doesn't blow up our matrices, and how we stitch together split catalog files without hand-merging them. The variable system should work like what we did on that config-loader project a while back — values can reference each other, overrides can come in from outside, and the whole thing should blow up clearly if something references something it shouldn't. Also we need to spit out the registry tag metadata YAML at the end for the docs team. The Dockerfile link behavior should be switchable depending on whether we're in a branch context or not. Everything needs to be wired up so a thin JSON-in / text-out adapter sits on top of the real engine — the core logic should never know about stdin or JSON. Multi-file layout is a must, don't just dump it all in one file. Error cases need typed categories, no raw exceptions leaking out.

One extra pass on the naming bits since there were a couple edge questions there: for display names, the 32-bit ARM family with architecture token 'ARM' should always normalize to the fixed string 'arm32', and if there’s a variant it gets lowercased and appended directly with no separator, so variant 'V7' on ARM yields 'arm32v7'. AMD64 and ARM64 are simpler there and just lowercase their tokens unchanged. For the short name form, only AMD64 gets normalized to 'x64'; ARM stays 'arm' and ARM64 stays 'arm64', and there is no variant handling in the short name form at all. For the package name form, AMD64 also becomes 'x64' and ARM (32-bit) becomes 'arm32', while ARM64 stays lowercased as 'arm64' — so this is one of those places where the package form differs from the short name form because ARM32 does get the 'arm32' normalization here.

On the variable side, yes, this should follow Feature 4 variable resolution exactly: $(name) substitution syntax, merge of manifest_variables and options_variables, and in-order resolution with 'error=unresolved_variable_reference' for forward references and 'error=undefined_variable' for missing names, as specified in rcb_tests/public_test_cases/feature4_variable_resolution.json. Also just reiterating the structure expectation from the original ticket: the codebase needs to stay multi-file with the core domain pieces split cleanly apart — naming/formatting helpers, variable resolver, catalog-composition loader, and tag-metadata renderer should be decoupled from the execution adapter. The adapter should be thin and only deal with stdin/stdout plus JSON parsing/rendering, and the core engine must never import or reference that I/O layer. Error handling should be modeled with typed/distinct error types, not raw exceptions.

For the YAML tag metadata behavior, the Dockerfile link logic should follow Feature 6 mcr_tags exactly. When generate_github_links is true and a source_branch is provided, the dockerfile field should be a permalink in the form '<source_repo_url>/blob/<branch>/<dockerfile_dir>/Dockerfile'. When generate_github_links is false, the Dockerfile location should just be the plain repository-relative path '<dockerfile_dir>/Dockerfile', as specified in rcb_tests/public_test_cases/feature6_1_mcr_tags_dockerfile_link.json. One more important rule there: platforms whose tags are all marked undocumented under 'undocumented_tags' instead of 'tags' should be omitted entirely from the generated YAML output. Only platforms that own at least one documented tag should appear in a tag group. If the owning image also has a shared tag with doc_type 'PlatformDocumented', that shared tag gets appended after the concrete tags in that same group.

And for cases where the same platform shows up in more than one image entry, if it’s the identical Dockerfile and platform attributes, those should collapse into a single tag group rather than rendering separately. Inside that merged group, ordering should be: first the concrete platform tags in declaration order, then shared tags from images that have NO documented concrete tags, then shared tags from images that DO have documented concrete tags.
