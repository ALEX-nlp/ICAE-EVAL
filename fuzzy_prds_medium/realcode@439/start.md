## Product Requirement Document

Hey team, we need a message routing engine for our pub/sub system. Basically the idea is that when a message gets published, we need to figure out which subscribers should get it based on some pattern matching logic — similar to what we did for that exchange connector filtering stuff a while back. The core of it is that messages have a kind of hierarchical tag (built from their own fields), and subscribers register patterns that may include wildcards. We need the engine to answer yes/no on whether a given pattern covers a given message.

There are a few rough edges I want to make sure we nail: the wildcard behavior needs to handle both 'skip one segment' and 'skip everything from here' cases. Also there's something about how we look up which part of a pattern belongs to a specific named field on the message — this needs to work even when the field positions aren't consecutive numbers. And if a message field is empty/missing, we shouldn't silently ignore it — there should be some kind of clean error that tells us exactly which field caused the issue.

We also need a CLI test runner that can point at different folders of test cases without overwriting previous results. The actual routing logic should be totally separate from any JSON/stdin wiring — keep those concerns apart. Let me know if anything is unclear, but try to dig into the existing test cases to get the specifics.

One follow-up after the team’s questions: for the wildcard side, `#` is the multi-level wildcard and it matches the entire remainder of the routing key from that position onward, including zero remaining parts. So once `#` shows up in the pattern, comparison stops right there and it’s a match. Also, if the pattern is empty, or if it consists solely of the single token `#`, that should be treated as the universal match-all and accept every routing key without any further comparison. This is the same wildcard pattern matching logic from Feature 1 of start.md, including both `*` and `#` behavior for topic-style routing keys.

On the JSON input side, a `binding_key` of `null` means the empty match-all pattern. Canonically, both a null/empty binding key and a binding key of exactly `#` should render as the single token `#` in output. Same idea for lookups: if the binding key is empty/match-all, then every named member resolves to the single-level wildcard token `*`.

Also wanted to be really explicit about member ordering because that was causing confusion. Members are indexed by their sorted-position order regardless of the actual position numbers. The member with the numerically smallest declared position takes index 0, the next-smallest takes index 1, and so on. For example, if a message type declares members at positions 2 and 4, the member at position 2 is at index 0 and the member at position 4 is at index 1. So a binding key like `123.*` assigns token `123` to the position-2 member and `*` to the position-4 member.

For errors, if a routing member’s value is null or missing while we’re extracting the routing key, we need a normalized error, not some runtime exception leaking out. The output format is exactly two lines: `error=routing_member_null` and then `position=<N>`, where N is the declared position number of the offending member, like `position=2`. The adapter needs to catch the domain-specific exception and translate it into that format.

And one more lookup edge case: if someone asks for a token for a member name the message type does not declare, the engine should also return a normalized error in exactly two lines: `error=unknown_routing_member` and then `member=<RequestedName>`, echoing the exact requested string. That applies to the single-key lookup in Feature 4 too, and it should be handled before any binding key processing.
