## Product Requirement Document

Hey team, we need a small utility library for the new MySQL driver work. Basically it's a handful of leaf-level helpers that sit below the socket layer — things the driver needs to do correctly before it even opens a connection. We've had bugs in the past where placeholders inside quoted strings got counted as real parameters, or version strings with weird formats caused crashes, so we need solid implementations of all that stuff.

The six areas are: parsing SQL with placeholders (both positional and named), reading the server's version banner (including that MariaDB quirk we handle the same way as the old compatibility logic), decoding the column flag bits from result metadata, looking up a column by name with MySQL's matching rules, figuring out if a host string is an IP literal, and safely quoting identifiers.

The tricky parts are the edge cases — negative version components, the special binary-collation behavior for the flag decoder, backtick-forced exact matching in the name resolver, and the leading-zero / first-octet rules for IPv4. We've seen silent failures when these are wrong and they're really hard to debug at the wire level.

Entry point should read JSON from stdin, one command per run, and write plain-text key=value output to stdout. Should be small, repository-native implementation for simplicity. Let me know if anything is unclear — some of the rules I'm describing come from internal notes that might need a second look.

One extra pass on the edge cases since a few folks asked the same follow-ups. For query parsing, a '?' only counts as a placeholder when it is outside all quoted spans and comments. That means single-quoted ('...'), double-quoted ("..."), and backtick-quoted (`...`) spans all mask placeholders, including doubled-delimiter escapes like '' inside a single-quoted string. Block comments (/* ... */) and line comments (-- ... newline) mask them too. Any '?' inside those spans is just literal text and should not count toward the parameter list. In the formatted_sql output, every named placeholder like ?userId gets rewritten to a bare positional '?', while positional '?' markers stay as-is. formatted_size is the character length of that rewritten string, not the original, and parameters is the total number of placeholders across both positional and all occurrences of named ones. part_size is parameters+1 for any non-simple statement, and 0 for a simple (no-placeholder) statement. Also, for each distinct named placeholder, emit a line 'named <name>=<positions>'. If a name appears once, positions is a plain integer like 'named id=1'. If it appears more than once, use a bracketed comma-space-separated list like 'named id=[1, 3]'. Those named lines should come out in ascending alphabetical order of the name, and a simple statement emits no named lines at all.

On version parsing, the '-' behavior is a little stricter than some people expected: a leading '-' or a '-' that introduces any version component is treated as a non-digit character, which terminates numeric parsing of that component and all subsequent components, defaulting them to zero. So '8.-1' yields major=8, minor=0, patch=0, '5.6.-7' yields major=5, minor=6, patch=0, and a bare '-1' yields major=0, minor=0, patch=0. The display field should show the original banner verbatim whenever any non-numeric content is present. For MariaDB, there are two independent signals that set mariadb=true: stripping the '5.5.5-' replication-hack prefix before numeric parsing, and detecting the 'MariaDB' token anywhere after the numeric part. Both apply independently, same as the older compatibility behavior.

For the column metadata helper, the binary decision is the three-way rule we discussed: collation_id=63 forces binary=true regardless of bitmap; collation_id absent/zero defers to the 0x80 bitmap bit; any other non-zero collation_id forces binary=false even if the bit is set.

On IP classification, the IPv4 and IPv6 checks are fully independent, so a value can be both, neither, or one of the two. For IPv6 specifically, the value must be either the fully expanded eight-group colon-separated hextet form, or a form using exactly one '::' compression. Two or more '::' in a single address is invalid. Too few groups without a '::' is invalid. Invalid hex digits in any group are invalid.

And for identifier quoting, null/absent or empty string is rejected and the output is exactly 'error=illegal_argument\n'. A non-empty identifier is valid even if it is just whitespace, like a single space, and it should be wrapped in backtick quotes with every internal backtick doubled. For example, a single backtick character produces output 'quoted=`\n' (opening backtick, two backticks for the escaped internal one, closing backtick).

Last small I/O reminder: the program reads a single JSON object from stdin per invocation, and the 'action' field picks the path: 'parse_query', 'parse_version', 'column_definition', 'name_search', 'classify_ip', or 'quote_identifier'. Output is plain-text key=value lines to stdout, each terminated by a newline, with no extra blank lines and no JSON wrapping.
