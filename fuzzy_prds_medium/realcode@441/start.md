## Product Requirement Document

Hey team, we need to build out the linked data processing library we discussed in last quarter's planning. Basically we want something that can take JSON-LD documents and do all the standard transformations on them — you know, the usual graph stuff. Think of it like that RDF utility we prototyped for the semantic web client last year, but productionized properly.

The big pain point right now is that every team that touches linked data ends up writing their own one-off scripts to handle IRI resolution, graph flattening, context application, etc. and they always end up with subtle data loss or serialization inconsistencies when exchanging data with external RDF systems. We need one canonical place for all of this.

It should handle the full lifecycle: expanding compact documents, squishing them back down with a context, making the graph structure flat and comparable, reshaping data around a template, converting back and forth to the raw triple format, and producing a stable canonical form. There's also some namespace shortening behavior we need — refer to how we handled prefix compaction in the graph client module — and it needs to report problems cleanly without exposing internal exception noise to callers.

The output format needs to be consistent and pretty-printed in a way that downstream tools can parse reliably. Architecture should be clean, multi-file, separated concerns. Don't make it a god file.

Quick follow-up from the questions that came in. For expanded output, this needs to come back as a pretty-printed JSON array every time, and each node object should use full IRIs as keys. Plain strings turn into `{"@value": "..."}`, typed literals include `@type`, language-tagged literals include `@language`, and native numbers (integers) stay numeric as `@value` entries. If a property has multiple values, that stays as an array of individual `@value` objects. Also, the `@type` field on a node is always an array of full IRI strings. Please keep the output ending with a newline character, and keep the example key ordering behavior consistent within each value object: `@type` before `@value` for typed literals, `@language` before `@value` for language-tagged literals, and `@id` alongside other properties for nodes.

For compacted output, we want a single pretty-printed JSON object, not an array. The `@context` from the supplied context document should be appended as the last key in the output object. IRIs that match context terms should be shortened to their term alias. A single-element `@type` array should compact down to a plain string, while multi-element `@type` arrays stay arrays of compacted strings. If the context says a property has `@type: @id`, then its values should render as compact IRI strings instead of node objects. Single-value property arrays can be unwrapped to scalars where the context allows it, multi-value arrays stay JSON arrays, and native scalar values (numbers) should stay native without `@value` wrapping when there isn’t a type coercion conflict.

On the adapter side, it reads a single JSON object from stdin. The `algorithm` field chooses the operation and needs to support `expand`, `compact`, `flatten`, `frame`, `to_rdf`, `from_rdf`, `normalize`, or `roundtrip_namespaces`. The `document` field carries the JSON-LD input for most operations. The `nquads` field (string) carries N-Quads text for `from_rdf`. The `context` field carries the compaction context for `compact`. The `frame` field carries the frame document for `frame`. The `base` field provides the base IRI. The `options` object carries processing options. The `namespaces` field (boolean or object) and `triples` array are used for namespace-related operations. The `contains` array is used for `roundtrip_namespaces` containment checks.

For flattening, the flow is to expand first, including all context aliases and IRI resolution, and then collect all graph nodes into a single top-level JSON array. For simple single-node documents, the result will look structurally the same as expansion. Also, native boolean and integer values should be preserved as `{"@value": true}` and `{"@value": 123}` with no type annotation when the original value has no explicit datatype. And prefixed terms like `e:bool` need to be resolved to their full IRI, for example `http://example.org/vocab#bool`, before they show up as keys.

For framing, the output should be a JSON object with a `@context` taken from the frame’s context and a `@graph` array containing the matched top-level nodes. Nodes matched by the frame are embedded with their related nodes nested inside them according to the frame s shape. The `@context` is the frame's context compacted form. Linked nodes used as property values should come through as full embedded objects, not just `@id` references, when the frame specifies a sub-frame for that property. Type values should be compacted using the frame context. The output ends with a newline, and single-value arrays for `@type` plus single-valued object properties should be unwrapped to plain strings or objects in the compacted framed output.

For `to_rdf`, the output is RDF statements in N-Quads format, one statement per line, and each line ends with ` .
`. IRI terms are wrapped in angle brackets like `<...>`. Plain string literals are double-quoted. Language-tagged literals use `"value"@lang`, typed literals use `"value"^^<datatype-iri>`, and blank nodes render as `_:bN`. The full output also ends with a newline. If the source document has no explicit `@id`, the subject should be the blank node `_:b0`. RDF type triples use the predicate `<http://www.w3.org/1999/02/22-rdf-syntax-ns#type>`.

Also, we do need the namespace/prefix behavior from feature8_namespaces.json kept intact, specifically the `to_rdf` path with `useNamespaces: true` that emits `namespace.<prefix>=<iri>` lines, and the `roundtrip_namespaces` path that emits both namespace declarations and `contains.<compactIRI>=true` lines.

And just to close the loop on scope, this should cover the full JsonLdProcessor API for all 7 algorithms: expand, compact, flatten, frame, toRDF, fromRDF, normalize, matching the JSON-LD 1.0 spec.
