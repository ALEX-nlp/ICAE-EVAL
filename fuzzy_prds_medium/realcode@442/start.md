## Product Requirement Document

We need a general-purpose HTTP client utilities library for our service integrations. Developers should be able to build HTTP requests by providing a verb (get, post, put, delete), a URL, optional query parameters, headers, and a body payload. Query parameter names and values with special characters need to be percent-encoded automatically. When a URL has no scheme or host, the client should recognize it as relative and apply a placeholder internally. Payloads can be raw text, a JSON object, or a set of form fields — each should set the right content-type automatically, and if the content type doesn't already specify a charset, UTF-8 should be appended. We also need a small immutable key/value pair object, a parameter-map utility that can either drop or keep a listed set of keys without mutating the original, and some basic guard functions that reject null values, empty sequences, or false conditions with uniform error messages. There's also a need to serialize a few specific data records — timing pairs, confidence scores, and date lists — into compact positional arrays, following the same layout convention the audio-processing module uses for wire data. Finally, we need a way to resolve service credentials from a deployment configuration map: given a service name and optionally a plan name, find the right instance and return a standard authorization token.

One small follow-up from the questions that came in: if the URL itself is null, we should reject it with 'error=null_url'. Also, if someone passes form fields as a flat list and it has an odd number of elements so we can’t make proper name/value pairs out of it, reject that with 'error=odd_argument_count'.

On the body handling side, the intent is still the same but a little more explicit: when a raw string body is sent with a media type that carries no charset, append '; charset=utf-8'. A JSON body always gets 'application/json; charset=utf-8'. A form body uses 'application/x-www-form-urlencoded' and that one stays as-is, with no charset appended.

For relative URLs, a path-only URL with no scheme/host should be treated as relative, and we apply a placeholder endpoint internally so the request is well-formed. The important part is that the 'relative' facet reports 'true' and the caller’s path is retained exactly.

And for the little name/value pair object, creating one with a null name should emit 'error=null_name'. If the value is absent, meaning null, it should report as '(null)' in output. The string form when there is a value is 'name=value', and when the value is absent it is just 'name'.
