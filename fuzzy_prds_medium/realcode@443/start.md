## Product Requirement Document

Hey team, we need to wrap up the contract testing adapter work. The core pieces are mostly spec'd out but there are a bunch of edge cases and default behaviors that aren't fully nailed down in the ticket. A few things that have been biting users lately: people keep getting confused about what happens when they pass in a minimum array size of zero or a negative number — we need a clear error there. Also some folks on Windows are hitting issues because we're not picking the right executable name for their platform (remember how we handled that in the platform detection module we did for the other CLI tool? do something similar here). The mock server startup args need to be exactly right — the log file path construction especially trips people up. For the verifier CLI, if someone only gives one of the two provider-state config options instead of both, that should be caught early. There's also the question of what the default working directory is for logs and pact files — I think we agreed it should be the testbed path but I don't see it written down anywhere. Oh and for HTTPS connections, cert and key paths should show up in the process command. Make sure the broker password gets pulled from the environment, not passed directly. Timeout default for the verifier should also be something sensible. Let me know if you have questions, I'll be around.

A couple more specifics to lock in since folks asked. For array_items_similar, any minimum value less than 1 needs to fail with exactly error=minimum_too_small. Also in relationship creation, if the provider field is missing, null, or an empty string, that should come back as error=invalid_provider.

On defaults, we should be consistent everywhere: log_dir and pact_dir both default to /testbed, and that applies across relationship creation, interaction definition, and mock server startup. Same idea for connection defaults: host is localhost, port is 1234, and protocol is HTTP, which means the default endpoint is http://localhost:1234 unless something overrides it. That same default behavior should carry across interaction definitions, relationship creation, and mock server commands. The default pact specification version is 2.0.0, and we want it to show up as --pact-specification-version=2.0.0 in process commands and as "version":"2.0.0" in JSON outputs.

For mock service HTTP API calls, if we get any non-200 back, the output needs to be error=remote_interaction_failed, and then a detail= line that includes the response body content when available, the operation name, and the calls array observed up to that point. For stop, the command is ["stop", "--port=<port>"], with port defaulting to 1234 if it wasn’t specified.

And just to remove any ambiguity on executable naming: on posix, the mock server executable is pact-mock-service and the verifier is pact-provider-verifier. On Windows (nt), both get .bat appended, so pact-mock-service.bat and pact-provider-verifier.bat. Same rule as feature10: posix platforms use bare binary names (pact-mock-service, pact-provider-verifier), Windows (os.name == 'nt') appends .bat to each name.
