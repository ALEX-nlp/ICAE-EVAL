## Product Requirement Document

Hey team, we need to build out the browser-side auth helper lib for the SPA. This is the thing that handles all the handshake stuff when users log in via the external identity provider — no backend involved, purely client-side. We've got about 10 distinct operations that need to work, covering everything from building the login redirect URL, handling what comes back after the redirect, dealing with the secret/challenge stuff for the secure flow, parsing the user identity token, and keeping a smart cache so we're not hammering the provider unnecessarily.

One thing I want to flag — remember how we did that compatibility encoding logic in the login module before? Make sure the byte mapping and alphabet stuff is consistent with that approach. The cache piece is important too: we had a bug last quarter where users were getting tokens back that had the wrong access level because the cache lookup was too loose. That needs to be airtight.

Everything should be deterministic — same inputs, same outputs, always. The adapter layer that reads commands and writes results needs to stay totally separate from the core logic. Error cases should produce clean neutral category labels rather than raw exceptions bubbling out. Please check the existing test layout under rcb_tests for the exact shapes expected.

A couple extra details from the team questions. For the PKCE side, keep the same compatibility mindset all the way through feature5 (op: base64url) and feature6 (op: code_verifier). The URL-safe base64 behavior is the usual swap where '+' becomes '-' and '/' becomes '_', and any '=' padding is removed completely. So bytes [255, 255, 255] must encode to '____', not '////' or '////'. Decoding also needs to be forgiving when padding is missing.

For the verifier itself, the code verifier uses a fixed unreserved character set, and each input byte maps deterministically to exactly one character in that set by modulo indexing. The set is A-Z, a-z, 0-9, '-', '.', '_', '~', which is the 66-character RFC 3986 unreserved set. Also important: this generator always returns a fixed 43-character value, regardless of how long the input byte array is, so if needed it pads/repeats the byte sequence to get there. Given bytes [1, 5, 10, 15, 100] the output must be 'verifier=15AFa15AFa15AFa15AFa15AFa15AFa15AFa15AFa15A'. The main thing is that the unreserved character alphabet and the byte-to-character modulo mapping stay used consistently across both PKCE features.

On the callback parsing, if 'expires_in' is missing from the query string, we still include it and report it as JSON null. So it should be 'expires_in': null, not omitted, not zero, and not a string. If it is present, parse it as an integer. The emitted output stays a single line in the form result=<json>, with keys in parsed order and expires_in always last.

On token validation, the checks need to happen in one strict sequence and we only report the first thing that fails. The order is issuer match → error=invalid_issuer, then audience match → error=invalid_audience, then signing algorithm must be RS256 → error=invalid_algorithm, then nonce match → error=invalid_nonce, then expiry (exp + leeway >= now) → error=expired, then issued-at not in the future (iat - leeway <= now) → error=future_issued, then not-before (nbf - leeway <= now) → error=not_yet_valid. If everything passes, output is 'valid=true\nclaims=<json>\n' and claims needs to include the '__raw' field with the original token string.

And on the cache bug we talked about, the keying rule is exactly both 'audience' AND 'scope' together in feature10 (op: cache_key). A lookup is only a hit if BOTH match the stored entry exactly with string equality. If either one is different, the result is 'miss'. The output lines are zero-indexed like lookup0=, lookup1=, and while the in-memory cache keeps the full entry object, on a hit we only return the access_token value. This is the piece that prevents a token for one audience or permission set from getting reused for another.
