## Product Requirement Document

Hey team, we need to build out that gateway routing support library we've been talking about. Basically devs keep complaining they have to rewrite the same boilerplate every time they wire up a new service — stuff like figuring out whether to use secure connections, handling those weird international characters in URLs that break things downstream, and making sure monitoring dashboards can actually group metrics together properly. It's a mess right now and every team is doing it differently.

The library should handle the full lifecycle of a proxied request — from taking a discovered service name and turning it into a clean route path, all the way through to how headers get copied and filtered. There's some existing compatibility logic we used in the header filtering piece from that security hardening sprint a while back — make sure the new implementation stays consistent with whatever defaults we landed on there.

Also the metrics naming has been a recurring complaint from the SRE team — they can't aggregate anything reliably because the tag formatting is inconsistent across services. Fix that too.

The adapter layer that wires up the JSON test harness should stay totally separate from the actual business logic — we got burned last time when someone mixed those together and it made the core untestable. The test runner should output files per case so we can diff them individually. Ping me if anything's unclear.

One other pass on the details the team asked about: for route generation, please normalize the final path after substitution by collapsing repeated '/' into a single separator and trimming any leading or trailing '/' characters. This matters a lot when an optional capture group comes back empty. If the pattern is '/${version}/${nevermatch}/${domain}/${name}/' and 'nevermatch' is unmatched, the empty substitution still needs to cleanly disappear so the result is 'v1/domain/service', not '/v1//domain/service/'. Also, if the service identifier does not match the configured capture pattern at all, just use the original service identifier unchanged as the route value. No error, no special handling, just a silent fallback to the raw service ID.

On headers, the built-in default ignored headers are exactly: Cache-Control, Expires, Pragma, X-Content-Type-Options, X-Frame-Options, X-XSS-Protection. The built-in default sensitive request headers are exactly: Authorization, Cookie, Set-Cookie. These are always the baseline, and any extra ignoredHeaders from config get appended to that list rather than replacing it. Same defaults again as defined in Feature 7: ignored headers [Cache-Control, Expires, Pragma, X-Content-Type-Options, X-Frame-Options, X-XSS-Protection] and sensitive headers [Authorization, Cookie, Set-Cookie], as specified in feature7_gateway_header_defaults.json. Also important: route-specific sensitive headers stay totally separate from the global sensitive headers and do not inherit from them. If routeSensitiveHeaders and sensitiveHeaders are both present, print route_sensitive_headers separately first, then the global ignored_headers and sensitive_headers lines. That route-specific list is its own isolated set.

For query strings, when a parameter has an empty string value, it must be rendered as a bare parameter name with no '=' sign appended. For example, a parameter named 'wsdl' with value '' renders as '?wsdl', not '?wsdl='. Non-empty values render normally as 'name=value'. Since this got missed before, keeping the exact phrasing here: (e.g., 'wsdl' with value '' renders as '?wsdl', not '?wsdl='. Non-empty values render normally as 'name=value'.

And just to lock down the harness/setup side: the test harness writes one output file per test case, using the filename format '{feature_file_stem}@{zero_padded_3_digit_index}.txt' — for example, 'feature1_route_mapping@000.txt' for the first case. Files go under 'rcb_tests/stdout/{cases-dir}/', where cases-dir defaults to 'test_cases' but can be overridden with --cases-dir. Each file should contain only the raw stdout from the program, with no PASS/FAIL annotations or extra metadata. The core business logic still needs zero dependency on stdin/stdout or JSON parsing. The Execution/Test Adapter described in the Deliverables section — a physically separate file/module from src/ that reads JSON from stdin, calls core domain methods, and writes formatted output to stdout, per the Separation of Concerns NFR in the Architecture section — is the only place that should deal with that plumbing. Core domain code should stay fully testable with no I/O involvement at all.

A few exact details to be precise about: Generated route paths must be normalized by collapsing repeated '/' separators into a single one and trimming any leading or trailing '/' characters. This is especially important when optional capture groups in the pattern produce no value — the substituted empty string must not leave orphaned separators in the final route. For example, '/${version}/${nevermatch}/${domain}/${name}/' where 'nevermatch' is unmatched should yield 'v1/domain/service', not '/v1//domain/service/'.
