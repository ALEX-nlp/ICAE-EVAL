## Product Requirement Document

Hey team, we need to wire up a new batch of utility features for the data table toolkit — mostly around the credential/token stuff and the table interactions. The license side needs to handle encoding an order ref plus an expiry into a portable signed string (same pattern as the auth module we built for the checkout flow last quarter, you know the one). It also needs to verify those tokens against some encoded release metadata and surface the right status back to the caller — including a special error path when the release metadata isn't decodable as a proper timestamp. On the table side, we need the usual suspects: initial render order preserved, clicking column headers should cycle sort direction (ascending first, then descending), server-driven sort should just trust whatever order the server sends back, string/number/date filtering with sensible comparison ops, page slicing plus page-change callbacks in controlled mode, single-row selection that doesn't leak into multi-select even with modifier keys, checkbox disabling for non-selectable rows, custom row identity fields with conditional CSS class application, cell and header class injection, toolbar-driven column visibility toggling, and density-based row/cell height scaling. Numbers in filter results need locale-style formatting and dates need m/d/yyyy formatting. The Base64 and MD5 primitives should also be exposed cleanly. Please make sure Unicode round-trips correctly through the encoding path.

One quick follow-up on the license bit since a few folks asked how exact the token shape needs to be. The token should be built by first creating the payload string exactly as 'ORDER=<order_number>,EXPIRY=<expiry_timestamp>,KEYVERSION=1', then Base64-encoding that payload string as UTF-8, then taking the MD5 digest of that Base64-encoded string, and finally concatenating the MD5 hex digest directly followed by the Base64-encoded payload, with no delimiter in between. For the concrete example, order 'MUI-123' with expiry 1591723879062 should produce 'e34253b37166e7a4a85189b91b653e63T1JERVI6TVVJLTEyMyxFWFBJUlk9MTU5MTcyMzg3OTA2MixLRVlWRVJTSU9OPTE='. Same deal for the Feature 4 flow: ORDER=<order_number>,EXPIRY=<expiry_timestamp>,KEYVERSION=1 payload → UTF-8 Base64 encode → MD5 digest → concatenate digest + Base64 payload.

On verification, the important nuance is that 'release_info' gets Base64-decoded first, and that decoded value has to be parseable as a numeric timestamp in milliseconds epoch. If the Base64 decode works but the decoded string is not a number, we should return 'error=release_info_invalid' specifically, not throw and not fall back to some generic status. Only once the release timestamp is valid do we continue with the normal license signature and expiry checks and return 'status=<result>'. The example that came up was release_info 'X19SRUxFQVNFX0lORk9fXw==' which decodes to '__RELEASE_INFO__', and that should go down the 'error=release_info_invalid' path.

Also confirming the Unicode/Base64 behavior since that feeds into the token path too: strings should be turned into UTF-8 bytes first, then Base64-encoded with the standard Base64 alphabet, and decoding is just Base64 → bytes → UTF-8 string. That’s what makes things like '✓' and 'à' survive the round-trip correctly. The example here is '✓ à la mode' encoding to '4pyTIMOgIGxhIG1vZGU='.

On the table side, for header-click sorting on text columns, the initial render stays in input order, first click sorts ascending, second click sorts descending, and then it keeps repeating that cycle. With the default brand data [Nike, Adidas, Puma], that means initial=Nike|Adidas|Puma, after 1 click=Adidas|Nike|Puma, after 2 clicks=Puma|Nike|Adidas.

For custom row identity, the caller can pass a field name instead of relying on a default row key. In the example the identity field is 'clientId', so visible_values should report c1|c2|c3. There’s also a row class predicate in play there: rows where age < age_threshold (20) get the supplied class name like 'under-age', and that gets reported as row0_under_age=yes, row1_under_age=yes, row2_under_age=no.

And for the toolbar column toggle, the reported header text is just the pipe-separated column field names that are currently visible. If a column is hidden, it disappears from that header output. Using rows [{id:0,brand:'Nike'},{id:1,brand:'Adidas'}], hiding 'id' should give initial_headers=id|brand and after_hiding_id=brand.
