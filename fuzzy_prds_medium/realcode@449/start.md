## Product Requirement Document

We need a small, self-contained retry scheduling engine for our background job system. When a job fails, we need two things: first, calculate when the next attempt should happen — the wait time should grow exponentially with each failure so we don't hammer a struggling downstream service; second, decide whether the job is still allowed to retry at all given its current state. The exponential back-off formula needs to be consistent everywhere — no more each team picking their own multiplier. All durations in the policy are specified in seconds but need to be converted to milliseconds for the internal time representation. The eligibility check is the tricky part: it needs to handle three lifecycle states — a job that failed outright, a job that was prepared and dispatched but timed out waiting, and a job that succeeded. For the timed-out prepared state, the check needs to look at a deadline timestamp and compare it against wall clock time, similar to how the payment-expiry module handles this. There's also a 'should retry right now' signal that combines eligibility with timing — only fire if the next-retry time has already passed. Output a named snapshot bundling attempt count, timeout deadline, and next-retry time point together so schedulers have everything in one place.

One small detail to lock down from the questions that came back: the next retry timestamp should be calculated exactly as nextRetryAt = currentRetryAt + (minBackoff * 2^retries * 1000). minBackoff is in seconds, converted to milliseconds by multiplying by 1000, and retries is zero-based (first attempt is retries=0). That gives us one consistent rule everywhere instead of different flavors of backoff depending on who touched the code last.

Also, the field name for the deadline keeps the original typo on purpose: it's 'timoutAt' (not 'timeoutAt'). So the output line should be timoutAt=<epoch_ms>. For retry eligibility, PREPARED means canRetry is true only when isRetryable=true AND timoutAt has elapsed (current wall-clock time > timoutAt). FAILED means canRetry=true whenever isRetryable=true. SUCCEEDED means canRetry is always false. That same logic should flow into the "retry right now" signal too, so it only goes true when the job is eligible and the computed next-retry time has already passed.
