## Product Requirement Document

We need a pluggable metrics collection component that acts as a central hub for gathering health and usage signals from different parts of the application. The component should support registering multiple data sources, each capable of contributing a list of named numeric measurements. During startup, the component should go through a two-stage initialization process: first verifying that all registered sources are healthy, then pulling in their measurements. If any source reports a problem during the first stage, we stop right there and report which stage failed. Same deal in the second stage. Once initialized, the component should check its own configuration for obvious problems — like missing required identifiers, measurements without names, measurements missing their sampling logic, or duplicate measurement names. The system needs clear on/off control: when turned off entirely, it should be completely inert and skip all the above checks. When running, it periodically samples every measurement on a configurable schedule. On startup it should print a human-readable message reflecting the current mode, similar to how the auth module announces its status. It should also expose a single HTTP endpoint so external monitoring tools can pull all the collected values. When the feature is disabled, that HTTP route should not be registered at all. The component must integrate cleanly with the broader application framework, following the same lifecycle conventions that other subsystems use — provision, validate, start, stop, and route registration.

A couple follow-ups from the questions that came in. For defaults, the namespace should fall back to 'gotenberg'. The flag for that is --prometheus-namespace and its default value is 'gotenberg', so if nobody provides a namespace we use 'gotenberg', not empty string. The collect cadence also has a real default: 1 second (time.Duration(1)*time.Second). The flag is --prometheus-collect-interval with default value of 1s.

On the HTTP side, the scrape route is exactly '/prometheus/metrics' and it is GET. If collection is off, Routes() returns nil, so you end up with routes=0 and nothing gets registered. If collection is on, exactly one route is returned. Route logging is controlled separately by --prometheus-disable-route-logging, which is a boolean with default false. If that flag is true, the route's DisableLogging field is true, and in output that shows up as 'disable_logging=true' or 'disable_logging=false'.

Also to be extra clear about the off switch, collection is disabled via the --prometheus-disable-collect boolean flag, default false. When that is true, Provision exits early without consulting any provider, Validate always passes, Start is a no-op, StartupMessage returns 'collect disabled', and Routes returns nil. Stop is intentionally boring too: it does nothing and always returns nil. The method signature is Stop(_ context.Context) error, and the output is always 'stopped=true' since no error is raised.

One other small detail on metric definitions: Description is optional on a Metric. The only required pieces are Name being non-empty and Read being non-nil. Description gets passed through to the Prometheus gauge Help field and it can be an empty string.

And on provider failures during setup, the way we recognize that case is by checking whether the error message starts with 'get metrics providers:'. Provision wraps it as fmt.Errorf("get metrics providers: %w", err), and the neutral category to report there is 'provider_validation'.

Last thing, the startup line is coming from StartupMessage() in the App interface (pkg/gotenberg/modules.go). Each module implementing the App interface provides its own StartupMessage() string method to print a custom message at startup, and for this module the return is 'collect disabled' or 'collecting metrics'.
