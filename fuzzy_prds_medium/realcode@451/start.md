## Product Requirement Document

We need to complete the black-box command surface for the entity data-modeling toolkit, not only namespace signing. The package should preserve the behavior covered by the public cases: typed value normalization and validation for emails, phones, dates, names, identifiers, URLs, IPs, MIME types, JSON and countries; numeric/date casting and best-value selection; country inference; RDF term rendering; schema specificity and similarity; registry metadata; entity proxy add/set/remove/pop/first/merge/inverted behavior; deterministic entity and node IDs; graph adjacency; record mapping; JSON unpacking; provenance/name helper transformations; and namespace signing/verification. The adapter reads a command payload and emits deterministic key=value or line-based stdout. Namespace signing remains one feature, but the implementation must support the full command set.

One small follow-up from the questions that came back: for the namespace signing piece, the signing uses HMAC with SHA-1. The namespace string is the key and the value string is the message. The resulting hex digest is the signature. For example: namespace='banana', value='split' → signature='18375e11927c10b772d57d7535888545782bd7cf'. The signed form is the original value, a literal dot '.', then the hex signature: e.g., 'split.18375e11927c10b772d57d7535888545782bd7cf'. The output fields are 'signed' (the full dotted form) and 'signature' (the bare hex digest alone).

Also, just to be explicit about the pass-through cases: when the namespace is null (or not provided), the value is passed through unchanged as 'signed' and 'signature' is '<none>'. No HMAC is computed. Output: signed=<value>, signature=<none>. When the value to sign is null, both output fields are the null token. Output: signed=<none>, signature=<none>. This applies regardless of whether the namespace is set.

That behavior should match the HMAC-SHA1 signing as implemented in the namespace/signing module; see feature10_1_sign.json and feature10_2_verify.json test cases for the exact expected inputs and outputs.
