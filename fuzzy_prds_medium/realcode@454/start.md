## Product Requirement Document

Hey team, we need to get the decompiler support toolkit wrapped up. There are a bunch of small helper utilities that keep getting reimplemented in slightly different ways across the codebase and it's causing inconsistencies in the output — users are complaining that decompiled code sometimes doesn't round-trip properly or looks weird. We basically need a clean collection of building blocks that handles all the fiddly low-level stuff: rendering constants back to source literals, name checking, figuring out types from those compact compiler formats, comparing version strings, some string utility stuff, the navigation history thing we talked about, the flag parsing (similar to how we handled it in the config module last time), figuring out where output files go, a little flag-set thing, the class relationship queries, and cert inspection. Each one needs to be its own thing — no giant single files please, we got burned by that before. The adapter layer that hooks into the JSON test harness should be kept totally separate from the actual logic. Edge cases need to be handled gracefully, no crashes on bad input. Make sure the unicode and escape handling matches what we agreed on — there were some bugs there before with certain character ranges. The generic type descriptor parsing is probably the trickiest bit, especially with wildcards and nested stuff. Just make sure it all lines up with the existing test contracts.

One quick follow-up from the questions that came in: on the literal rendering side, when escape_unicode is true, every character whose code point is 127 (0x7F) or greater must be rendered as a four-hex-digit \uXXXX sequence. Characters below 127 that are not special control characters are kept verbatim. For char literals, characters that are not printable ASCII and not one of the named escapes (\n, \t, \r, \b, \f) are also rendered as \uXXXX. Also, a single-quote character inside a double-quoted string literal must be left unchanged — it is NOT escaped. Only the double-quote character gets a backslash escape (as \") inside double-quoted literals. Conversely, inside a single-quoted char literal, the single-quote IS escaped as \' and the double-quote is left verbatim.

For the navigation history behavior, if an 'add' operation is called with the same id that is already the current position, the operation is ignored (no duplicate is appended and the history state does not change). And if you move back and then add a new id, that should truncate any forward history beyond the current position before appending the new entry.

On the little attribute set helper, every mutating operation needs to echo what happened: 'add' prints 'add=<FLAG>', 'remove' prints 'remove=<FLAG>', and 'clear' prints just 'clear' (no flag name). A 'contains' query prints 'contains[<FLAG>]=true' or 'contains[<FLAG>]=false'. Each line ends with a newline. The set is fresh (empty) at the start of each command.

For the cert inspection output, for the 'signature' field, output two lines: 'Signature type: <algorithm name>\n' and 'Signature OID: <OID string>\n'. For the 'public_key' field with RSA, output lines for the key algorithm, exponent, modulus bit-size, and full modulus value, each as 'label: value\n'. The DER bytes are provided as a base64-encoded string in the 'der_base64' field.

On the class relationship query, the 'implements' relation is strictly directional: if type A is a subtype/implementor of ancestor B, then implements=true; but the reverse (B implements A) is false unless B actually is a subtype of A. The query uses a bundled static Java class hierarchy — it is not computed dynamically. The result is a single 'implements=true' or 'implements=false' line.

Also confirming the structure expectation here: the implementation must NOT place all features in a single file. Related responsibilities must be grouped into cohesive modules under a src/ directory tree (e.g., text/literals for escaping, type descriptors module, CLI/config module, GUI-support utilities, crypto inspection). The execution adapter that translates JSON commands must be kept separate from core logic. Core helpers must not know about JSON, stdin/stdout, or the wire format.

For identifier validation, each dot-separated segment of a fully-qualified identifier must start with a letter, underscore, or dollar sign, and all subsequent characters must be letters, digits, underscores, or dollar signs. All characters must be printable ASCII. An empty segment (from a leading dot or consecutive dots), a segment starting with a digit, or an empty overall string are all invalid.

And one last thing on the flag parsing bit: this includes the CLI flag parser described in Feature 8 (feature8_cli_flags.json), specifically the 'override' mode where a base settings map is provided and only explicitly-present tokens modify it, leaving other base values untouched. Same deal on the escape behavior too — the unicode escape threshold at code point 127 (0x7F) for string literals (Feature 1) and the \uXXXX fallback for non-printable, non-named-escape characters in char literals (Feature 2) need to match exactly.
