## Product Requirement Document

Hey team, we need a small utility built for the framework wiring layer. Basically we keep running into silent bugs where values get swapped when injected into methods — devs aren't noticing until runtime and it's causing real pain in production. The fix is a validation + resolution tool that checks whether a method's parameters can be safely bound before anything runs, and also lets you look up exactly what name a given parameter will be bound under.

For validation: we need to catch the case where devs only half-annotate their parameters — like the auth module approach we discussed before where some params get the name-flag and others don't. That inconsistency should be a hard reject with a structured error that tells you exactly which group is broken and who's in it.

For resolution: there's a precedence rule for figuring out the final binding name — explicit marker value wins, then falls back to the param's own name, then a default. Need to handle the edge cases (blank values, missing names, etc.) cleanly.

Should be runnable from stdin/stdout with JSON commands, and there needs to be a test harness that runs all the case files automatically. Keep the core logic clean and separate from the I/O plumbing — don't want a repeat of the last module where everything was tangled together. Output format needs to be exact, cases are in the test folder.

One extra note after the team questions came in: for the type-group rejection behavior, this is narrower than it may have sounded. A type group is only rejected when it has MORE THAN ONE parameter AND those parameters are INCONSISTENTLY marked — meaning some but not all have name_match set to true. If ALL members have name_match=true, that's accepted. If NO members have name_match=true, that's also accepted. The only conflict is the mixed case where some are true and some are false, and single-member type groups are always fine regardless of their flag. This is the same inconsistent name_match flagging scenario called out earlier where some but not all parameters in a same-typed group have name_match=true, and that exact case is what should reject with error=duplicate_parameter_type.

Also, when we do reject, the output needs to be exact and in order: 'result=rejected', then 'error=duplicate_parameter_type', then 'conflict_count=<N>' where N is the number of offending type groups, then one 'conflict=' line per offending group. Each of those lines must look like 'conflict=type:<typeLabel> positions:<pos0>,<pos1>,... names:<name0>,<name1>,...' and the positions are zero-based indices in declaration order. Important detail there: list ALL members of the ambiguous group on that line, including any that individually had name_match=true.

And on the binding-name lookup, the precedence is strict. (1) If there is NO marker at all (name_match key absent from input), the binding name is always the default 'it', regardless of what parameter_name is. (2) If a marker IS present and its value is non-blank, that explicit value is the binding name. (3) If a marker is present but its value is blank (empty string), fall back to the parameter's declared name. (4) If the marker is present with blank value AND parameter_name is null/absent, the result is null, rendered as the literal string '<null>'.
