## Product Requirement Document

Hey team, we need to build out a media URL toolkit for our CDN integration. Basically devs are constantly complaining that building image/video delivery URLs by hand is super error-prone — wrong codes, broken overlays, messed up signatures, all that fun stuff. The goal is a library that takes declarative intent and spits out correct wire-format strings. Think transformation chains, overlay/layer descriptors, full delivery URLs, auth tokens, signature checking, some string/array/path utilities, and a config URL parser. 

For the execution layer it should read a JSON command from stdin and write line-oriented key=value output to stdout — errors should be normalized, no raw exceptions leaking out. The multi-file layout matters here; don't dump everything in one file, this is supposed to be maintainable. 

For the conditional expression stuff, just follow the same operator-mapping approach we used in that filter/expression module from the previous SDK iteration — you know the one. Similarly the text escaping rules for overlays are a bit special (double-escaping for certain chars), refer to the existing layer-encoding logic we settled on before. 

Signing tokens need HMAC-SHA256, config parsing handles the cloudinary:// scheme. Missing lifetime or empty secret should be proper error states, not just nulls. Oh and the sort-by-array fallback behavior when the order list is empty or irrelevant needs to be sensible. Make sure all edge cases are handled cleanly.

One extra pass on the transformation formatting since folks asked what “canonical” really means here: inside a single transformation segment, qualifiers should come out in this order: if (conditional) → crop → width → height → aspect_ratio → density → angle → effect → flags → border → audio_codec → bit_rate → offset (expanded to so_ and eo_) → custom_function. The big gotchas are that the if condition always goes first, and offset does not stay as one thing once rendered. It expands into two parts, so_ for start and eo_ for end, and those two should be sorted among the other qualifiers alphabetically by their short code, which means eo_ before so_.

Also on overlay text encoding, the escaping is a little more particular than normal URL encoding. Text in a layer/overlay qualifier is URL-encoded, but commas and forward slashes get double-escaped, so ',' becomes '%252C' and '/' becomes '%252F'. The '?' character is just single-escaped to '%3F', and spaces become '%20'. Everything else follows standard URL percent-encoding. Related to layers, if a layer has a 'fetch' key, that value is the thing we use, not public_id. The remote URL gets base64-encoded with the standard Base64 alphabet, not URL-safe base64, and then appended after 'l_fetch:'. Example stays exactly 'https://cloudinary.com/images/old_logo.png' → 'aHR0cHM6Ly9jbG91ZGluYXJ5LmNvbS9pbWFnZXMvb2xkX2xvZ28ucG5n'.

For custom functions, the remote flavor should render as 'fn_remote:<base64-encoded-source>'. The source URL uses standard (non-URL-safe) Base64 with padding. Example: 'https://df34ra4a.execute-api.us-west-2.amazonaws.com/default/cloudinaryFn' encodes to 'aHR0cHM6Ly9kZjM0cmE0YS5leGVjdXRlLWFwaS51cy13ZXN0LTIuYW1hem9uYXdzLmNvbS9kZWZhdWx0L2Nsb3VkaW5hcnlGbg=='. For the if qualifier, we do want the compact short-code translation everywhere, same pattern as before: 'aspect_ratio' → 'ar', 'width' → 'w', 'height' → 'h', '>' → 'gt', '<' → 'lt', '>=' → 'gte', '<=' → 'lte', '==' → 'eq', '!=' → 'ne', '&&' → 'and', '||' → 'or'. Spaces around operators become underscores, and the whole thing is prefixed with 'if_'. So 'aspect_ratio > 0.3 && aspect_ratio < 0.5' becomes 'if_ar_gt_0.3_and_ar_lt_0.5'. Same note applies to the conditional expression operator-mapping table in Feature 1 transformation qualifiers: translating human-readable expressions like 'aspect_ratio > 0.3 && aspect_ratio < 0.5' into short-code form 'if_ar_gt_0.3_and_ar_lt_0.5' using the operator and variable substitution rules defined in the feature1 test cases.

One more behavior note on truncate_middle because there was a question about tiny max_length values. If the string already fits within max_length, return it unchanged. If max_length is smaller than or equal to the glue length, the prefix goes to zero characters and the glue itself is truncated from the left to fit, while always keeping the rightmost characters of the original string to fill remaining space. In other words, the tail of the original string takes priority — the output is always the last (max_length) characters where the glue (possibly truncated) leads and the original tail fills the rest.

And on signature verification, the exact input to SHA1 is 'public_id=<public_id>&version=<version><api_secret>' with no separator between the version value and api_secret. The result should be a lowercase hex digest. If api_secret is null or empty string, reject it with 'error=invalid_api_secret'. Even then, the output format should consistently include both 'expected_signature=<hex>' and 'signature_match=true|false'.

A few exact details to be precise about: Text in a layer/overlay qualifier is URL-encoded, but commas and forward slashes receive double-escaping: ',' becomes '%252C' (not just '%2C') and '/' becomes '%252F' (not just '%2F'). The '?' character is single-escaped to '%3F'. Spaces become '%20'. All other characters follow standard URL percent-encoding.
