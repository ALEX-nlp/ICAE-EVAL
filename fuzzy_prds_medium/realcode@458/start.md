## Product Requirement Document

We need a reusable report formatter for a code quality tool. The tool scans source files and produces a list of rule violations; each violation records which file it is in, the one-based line and column, the rule that was broken, a human-readable message, and whether it was already auto-corrected. The formatter's job is to turn that list into a clean human-readable output, handling three scenarios: a flat list where each uncorrected violation gets its own line with its location and message; a colorized variant where the file path parts and location separators are wrapped in terminal color codes so they visually stand out; and a grouped-by-file variant where violations are collected under a file header. We also need a way to build the formatter from a plain string map the way a command line would pass options, and reject invalid color names immediately. Handle color validation the same way the auth module rejects bad credentials — fail fast with a clear error message, not silently produce wrong output.

One small follow-up from the questions that came in: for the plain ungrouped output, each uncorrected violation is emitted as: '<file>:<line>:<col>: <message>' followed by a newline. Corrected violations with corrected=true are just skipped with no placeholder or note, and if every violation in the input is corrected the output is empty.

On the color side, the specific values matter here: DARK_GRAY has numeric code 90 and RED has numeric code 31. The activating escape sequence is ESC[<code>m (byte 0x1B followed by '[', the numeric code, and 'm'), and the reset sequence is ESC[0m. In colorized mode there are exactly three colored segments: the directory portion of the file path (everything up to and including the final '/'), the colon after the file's base name, and the combined ':<col>:' segment.

Also, for group_by_file, the behavior is a little stricter than the first writeup may have implied. Each file with at least one uncorrected violation gets a header line containing just the file path, followed by one indented violation line per uncorrected violation. Each indented line is two leading spaces, then '<line>:<col> <message>'. Files where all violations are corrected produce no output at all, and the groups should come out in first-seen order.
