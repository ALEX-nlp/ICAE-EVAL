## Product Requirement Document

We need a lightweight, reusable toolkit of pure helper functions for cleaning and interpreting web content without requiring network access. The library should cover several categories of operations: normalizing messy page text (collapsing extra whitespace, stripping trailing site-name suffixes from titles), classifying web addresses by the type of content they likely point to (video, audio, downloadable documents, archives, executables, or readable pages), and cleaning up URLs by following well-known link-forwarding interstitials from major platforms and stripping campaign analytics tags from query strings. The toolkit should also support extracting metadata from page markup fragments — specifically parsing icon size declarations to find the best high-resolution favicon, and pulling the single most representative image or link reference out of an arbitrary HTML subtree. All operations must be pure and deterministic given the same input, with no side effects or external calls. The I/O contract is a simple line-oriented key-value protocol where each result is emitted as one or more lines in the form "key=value", with booleans written as lowercase words. URL results must be returned in normalized form (a bare host URL gains a trailing slash). When classifying a URL, the system must output exactly seven boolean lines in a fixed order. The image extractor searches for content in a specific priority order — checking direct attributes on the root element before descending into the subtree and finally inspecting inline style declarations for background references.

A couple extra specifics from the team questions: for URL classification, classify_url emits exactly 7 boolean lines in this fixed order: video, image, audio, binary_document, archive, executable, article. Each is 'true' or 'false' lowercase. The 'article' line is true only when all six media-type flags are false. Also, isLikelyBinaryDocument() returns true for: doc, pdf, ppt, rtf, swf, xls. One slightly weird but intentional case here is that 'pdf' appears in BOTH the image and binary_document extension sets, so a .pdf URL will have both image=true and binary_document=true, and article=false.

For whitespace cleanup, clean_whitespace emits a single line: result=<normalized text>. All maximal runs of whitespace (spaces, tabs, newlines) are replaced by a single space and leading/trailing whitespace is trimmed. An all-whitespace input yields 'result=' (empty value). For titles, clean_title checks: if the LAST position of '|' is greater than half the string length (lastIndexOf('|') > length / 2), it keeps only the text before the FIRST '|', trimmed of surrounding whitespace. Otherwise the title is returned with whitespace normalized (removeWhiteSpace). The midpoint comparison uses the total string length divided by 2.

On URL cleanup, remove_tracking removes any query parameter whose name is in the TRACKING_PARAMETERS set: _openstat, action_object_map, action_ref_map, action_type_map, fb_action_ids, fb_action_types, fb_ref, fb_source, ga_campaign, ga_content, ga_medium, ga_place, ga_source, ga_term, gs_l, utm_campaign, utm_cid, utm_content, utm_id, utm_medium, utm_name, utm_place, utm_pubreferrer, utm_reader, utm_source, utm_swu, utm_term, utm_userid, utm_viz_id, yclid. All other parameters are preserved. The platform redirect cleanup the team asked about is the same FacebookUrlRewriter (host ends with .facebook.com, path /l.php, param 'u') and GoogleUrlRewriter (host ends with .google.com, path /url, param 'q' then 'url') behavior that lives in golden/src/main/kotlin/com/chimbori/crux/plugins/, and the tracking-strip piece is TrackingParameterRemover in golden/src/main/kotlin/com/chimbori/crux/plugins/TrackingParameterRemover.kt — removes the 29-entry TRACKING_PARAMETERS array (utm_*, ga_*, fb_*, gs_l, yclid, _openstat, etc.)

For icons, parseSize() and findLargestIcon() in golden/src/main/kotlin/com/chimbori/crux/extractors/MetadataHelpers.kt — parses 'WxH' size strings and selects the icon element with the largest dimension. More specifically, findLargestIcon() selects the icon element whose 'sizes' attribute yields the largest value from parseSize(). If multiple icons tie on size, maxByOrNull picks the first in document order. The result is returned as an absolute URL via 'abs:href'. If the winning element has no href, null is returned and the output is 'url='.
