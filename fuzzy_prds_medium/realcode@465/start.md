## Product Requirement Document

Hey team, we need to build out that document database client adapter we've been talking about. Basically devs on our side keep complaining that they're hand-rolling all these weird nested path strings and type envelopes every time they touch the database layer, and it's causing a ton of subtle bugs in production — wrong paths, broken type conversions, that kind of thing. We want a clean library that handles all of this for them.

The library needs to cover the usual stuff: building and parsing those long resource name strings (you know, the wire prefix format we use), handling nested field paths safely (including the escaping logic — refer to how we handled special characters in the query builder from the last sprint), converting between our internal value types and what the wire actually expects, reading document snapshots without accidentally mutating them, navigating between collections and documents, putting together structured queries with filters/ordering/limits/cursors, and batching up write operations like creates, sets, updates, deletes, and transforms.

The whole thing should be driven by a JSON command interface so we can black-box test it easily. Output needs to be pretty-printed JSON. Error cases should be handled gracefully, not just crash. Multi-file structure please, this is getting complex enough that a single god-file would be a nightmare to maintain. Python is fine.

One quick follow-up from the questions that came in. Absolute document paths need to use the format `projects/<project>/databases/<database>/documents/<relative_path>`. Database names use `projects/<project>/databases/<database>`. The `append_to_database` operation takes an existing absolute database name and appends `/documents/<relative_path>` to it, and relative path extraction strips the `projects/<project>/databases/<database>/documents/` prefix.

On field paths, the escaping rule is a little narrower than some folks assumed: field path segments must be backtick-escaped in wire mask strings only when they contain characters outside `[a-zA-Z0-9_]` or when they start with a digit. Simple alphanumeric segments are written as-is with dot separators. Also, for the JSON interface shape, the `segments_to_path` operation returns both `segments` and `path` as arrays of the segment strings (not a dotted string), while the `string_to_segments` operation returns the original dotted string under `path` and the split list under `segments`. Same escaping behavior should be reused anywhere we render wire field paths, including query `select.fields[].fieldPath` and write `updateMask.fieldPaths[]`.

A couple decoding details too. A `referenceValue` wire field decodes to an object with keys `type` (always `"document_reference"`), `name` (the full absolute path), `path` (the relative path after the documents prefix), `id` (the last segment of the relative path), and `parent` (the absolute path of the parent collection, i.e., everything up to but not including the last segment). Timestamps decode to `{"kind": "timestamp", "seconds": <int>, "nanos": <int>}`. GeoPoints decode to `{"type": "geo_point", "latitude": <float>, "longitude": <float>}`. Bytes decode to `{"type": "bytes", "value": <base64string>}`.

For snapshot reads, the `identity` operation returns exactly four fields: `name` (the full absolute document name), `path` (the relative path below the documents root), `id` (the last segment of the relative path), and `exists` (boolean). The `timestamps` operation returns `readTime`, `updateTime`, and `createTime` each as timestamp descriptor objects with `kind`, `seconds`, and `nanos`. The `fields` operation returns the document's field data nested under a `fields` key.

And on the navigation helpers, `collection_identity` should return `type` (`"collection_reference"`), `name`, `path`, and `id` and it does NOT include a `parent` field. `document_identity` should return `type` (`"document_reference"`), `name`, `path`, `id`, AND `parent` (the absolute path of the containing collection, i.e., all segments except the last).
