## Product Requirement Document

We need a backend-agnostic file storage toolkit combined with a small library of standalone data-shaping helpers. The storage side lets the application save, retrieve, and manage files through a single consistent interface regardless of where the bytes physically live — a local machine today, a remote object store tomorrow. A central registry component reads a configuration that names one or more storage destinations, lazily creates each destination on first use, caches it, and allows new storage backend types to be registered or new destinations added at runtime. Two safety concerns are non-negotiable: file access must always stay within the configured storage area so no caller can escape it, and failures must surface as well-structured, machine-readable errors carrying a stable category label, a domain error code, an HTTP-style status number, and a human-readable explanation — never raw system fault messages. Separately, a collection of small transformation utilities handles common recurring needs: abbreviating large counts for display, coercing scalar-or-list values into lists, converting words between singular and plural forms, generating short non-cryptographic fingerprints, reordering list elements, recursively transforming nested data objects, classifying data collection definitions, resolving the endpoint path for a named collection (including the system-reserved prefix logic), determining the relationship direction for a given viewpoint, recognizing special placeholder variables that begin with reserved tokens like the current-time marker, and parsing structured text safely to strip prototype-pollution attacks. All of this lives in the drive package and the shared utilities module.

A couple follow-ups from the questions that came in. On the storage error side, the unsupported-operation response needs to come out exactly as 'error=method_not_supported\ncode=E_METHOD_NOT_SUPPORTED\nstatus=500\ndetail=Method <operation> is not supported for the driver <backendName>\n'. The detail line includes both the operation name and the backend class name. Status is always 500. Code is always E_METHOD_NOT_SUPPORTED. And if there is no default disk name configured, that response is 'error=invalid_config\ncode=E_INVALID_CONFIG\nstatus=500\ndetail=Make sure to define a default disk name inside config file\n'. The exact detail message is 'Make sure to define a default disk name inside config file'.

Also worth being explicit about the registry output. listDisks emits 'disks=<comma-separated-sorted-list>', and that list is only the disks that have actually been instantiated so far, not every configured disk. So before any disk() call it starts empty as 'disks='. Once there are entries, they are sorted alphabetically and joined with commas, no spaces. listDrivers works the same general way and emits 'drivers=<sorted-comma-separated-list>' for the registered driver names, including the built-in 'local'.

For relation detection, getRelationType returns 'm2a' when relation.collection === collection AND relation.field === field AND relation.meta?.one_collection_field is truthy AND relation.meta?.one_allowed_collections is truthy. It returns 'm2o' when relation.collection === collection AND relation.field === field, as long as those polymorphic meta markers are not what made it match. It returns 'o2m' when relation.related_collection === collection AND relation.meta?.one_field === field. Returns null otherwise.

A small behavior detail on deletes and file writes: deleting a missing file does not raise anything, it just returns { wasDeleted: false } and the dispatcher emits 'was_deleted=false'. If the file existed and was removed, it returns { wasDeleted: true } and emits 'was_deleted=true'. That missing-file case on delete is intentionally swallowed. For prepend, if the target path does not exist yet, the FileNotFound is caught internally and the new content gets written as a fresh file, basically the same outcome as put. Append already behaves that way too since it uses fse.appendFile, which creates the file if missing.

On the safe JSON helper, parseJSON strips any literal '__proto__' key from the parsed object. Input '{"a":1,"__proto__":{"polluted":true}}' produces '{"a":1}' and all other keys parse normally. The safe parser re-serializes the result as standard JSON without the stripped key.

One more error shape to keep exact: when a disk points at a driver that is not registered, emit 'error=driver_not_supported\nstatus=400\ndriver=<driverName>\ndetail=Driver <driverName> is not supported\n'. Note: no 'code=' line is emitted. Status is 400, not 500, and the driver name is echoed on its own 'driver=' line.

And just to pin the package split down clearly, packages/drive contains Storage abstract class, StorageManager, LocalFileSystemStorage, and the normalized exception hierarchy like MethodNotSupported, InvalidConfig, DriverNotSupported, FileNotFound, etc. packages/shared/src/utils contains abbreviateNumber, toArray, pluralize, depluralize, getSimpleHash, moveInArray, deepMap, getCollectionType, getEndpoint, getRelationType, isDynamicVariable, parseJSON.
