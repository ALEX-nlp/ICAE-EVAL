## Product Requirement Document

Hey team, we need to build out that content utilities library we've been talking about. Basically a collection of small helper functions for the file management front end — things like parsing user-pasted lists, handling collaborator emails (including that external-check logic we discussed, similar to what the sharing module does), formatting file sizes in different languages, abbreviating big numbers on dashboards, fuzzy search matching, network identifier validators, hex encoding, and sorting items in collections plus activity feeds.

The tricky parts are: the size formatting needs to work for at least Russian, French, and German in addition to English — each with their own decimal separators and unit labels. The number abbreviation should handle edge cases like booleans, null, objects, and strings being passed in, and there's some locale-specific behavior for German, Russian, and Japanese that we need to get right (including Russian pluralization rules). The item sorting must group folders before files before links, and there's some error handling that should output clean category strings rather than anything internal.

For the adapter layer, refer to the same pattern we used on that previous parsing pipeline — stdin/stdout JSON dispatch, no business logic in the adapter itself. Each utility should be its own small module. The output format is line-oriented labeled key=value pairs. Let me know if anything is unclear, but most of the edge case behavior should be self-evident from the existing test fixtures.

A couple of extra specifics from the questions that came in. For the pasted-list parsing, a field wrapped in double quotes may contain commas and carriage returns, and the outer quote pair is removed. Trailing commas or trailing line breaks must not produce an extra empty field. Null or empty input yields an empty list. The output there should be 'count=<n>' header then one 'item=<value>' line per field.

On the external collaborator check, the result is true ONLY when all three conditions hold simultaneously: isCurrentUserOwner is true, ownerEmailDomain is non-null/non-empty, AND the domain portion of emailToCheck differs from ownerEmailDomain. Any one missing — non-owning viewer, null ownerEmailDomain, null emailToCheck — forces false. Output echoes all three inputs plus is_external as four labeled lines, and null values render as empty string, for example 'email_to_check='. Same idea as noted for Feature 2.2 isExternalCollaborator logic: true only when isCurrentUserOwner=true AND ownerEmailDomain is non-null AND emailToCheck's domain differs from ownerEmailDomain, as specified in feature2_2_external_collaborator.json.

For sorting, the grouping is folders first, then files, then links. Within each kind, sort by chosen field (name, modified_at, interacted_at, size). Missing attributes fall back to neutral defaults, and ties are stable. If the current order already matches requested field+direction, return items untouched. Error categories are rendered as 'error=<category>'): unknown sort field, unknown item kind, malformed collection — and we should never expose host-language exceptions. Normal output is 'count=<n>' then one 'key=<entry_key>' line per item.

For the activity-feed-style merge, null containers in the list are skipped entirely. All entries from non-null containers are merged and sorted ascending by created_at ISO timestamp. Output is 'count=<n>' then one 'id=<entry_id>' line per entry in chronological order.

Also just to be extra clear on the adapter split: core utility functions must NOT read stdin, write stdout, or know about JSON. A single thin adapter/dispatcher is the only layer that reads a JSON command from stdin, calls the appropriate core function with idiomatic arguments, and writes the line-oriented output to stdout. No host-language type names, exception classes, stack traces, or object reprs ever reach stdout. Same adapter/dispatcher pattern as the thin adapter/dispatcher pattern described in Architecture Constraint #2: a single module reads a JSON command object from stdin, routes to the appropriate core utility function, and writes line-oriented key=value output to stdout. Core functions have no I/O knowledge.

For the hex helper, convert hex string to standard Base64 by treating each pair of hex digits as one byte. Carriage returns and line breaks in the input are ignored/stripped before encoding. Example: '12AB34' → 'Eqs0'.

For email extraction, it needs to handle bare addresses, angle-bracket-wrapped addresses with display names ('Foo Bar <foo@x.com>'), and multiple addresses delimited by spaces, commas, or semicolons. Output is an ordered list of bare addresses, whitespace-trimmed. Null or empty or address-free input → count=0. Otherwise it's 'count=<n>' then one 'email=<address>' line per address.

And one small formatting gotcha on file sizes: any falsy or absent input (null, 0, missing argument) renders as '0 Byte' (singular, no 's'). The value 1 renders as '1 Bytes'. Powers of 1024 produce KB, MB, etc. with up to two decimal places.
