## Product Requirement Document

Hey team, we need to build out that filename parsing library we've been talking about for the Kavita cataloguing backend. Basically the idea is that users are constantly complaining that their manga and comic collections show up in completely random order, and titles look like garbage because of all the bracket noise and underscores from various scanlation groups. We want a clean library that can look at a raw filename and spit out structured info — stuff like the title, which volume or chapter it is, whether it's some kind of special release, what format the file is, etc. It should work for both manga-style naming (the chapter-heavy stuff with all the weird group tags) and western comic naming (the Asterix/Batman style with year parentheticals and publisher info). Also needs to handle sorting — you know how file explorers sort x10 before x3, we need the human-friendly version of that. Should be pure functions, no database, no network. I want this modular, not one giant file. Refer to the same sentinel pattern we used in the chapter-numbering edge cases from that old sorting module — same idea applies here for volumes too. Also make sure the special-release detection covers the usual suspects that the ops team flagged last quarter. Language should be Python. One entry point script that reads JSON from stdin and routes to the right function.

Quick follow-up from the questions that came in: for the missing-number behavior, we should be totally consistent here. When no volume marker is present the function must return the string '-100000' and that applies to both manga volume parsing and comic volume parsing. Same deal for chapter/issue parsing: when no chapter or issue marker is present the function must return the string '-100000' for both manga chapter and comic issue extractors. Don’t swap in None, an empty string, or 0. This is the same sentinel return value used in both parse_volume and parse_chapter when the marker is absent, and it needs to stay aligned across src/parsers/manga_parser.py and src/parsers/comic_parser.py.

A couple more specifics on file handling too. A file counts as a cover image when its base name without the extension matches things like 'cover', 'folder', '!cover', or a volume cover pattern, and only if it’s a supported image extension. Back-cover filenames should not match, and dot-prefixed files should be excluded. For junk path filtering, the blacklist check is not just for the last path piece — it applies to every segment in the path, including stuff like __MACOSX, recycle bin variants, snapshot folders, and similar system/junk directories.

For normalization, the key should come from lower-casing the input and then stripping spaces, underscores, hyphens and dashes, plus most punctuation, while still preserving meaningful characters and non-Latin scripts like Cyrillic, CJK, and Korean. End result should just be one continuous lowercase string with no separators. For the zero-padding helper, it should left-pad a numeric string to width 3 using '0', but leave anything already at 3 or more chars alone. So input '1' → '001', input '10' → '010', input '100' → '100'.

On the little utility side, the font-src separator needs to split a CSS font-source declaration into exactly three parts, each printed on its own line: first the opening directive through the opening quote or parenthesis, then the file path, then the closing quote or parenthesis. Example stays exactly 'src: url(fonts/file.ttf)' → 'src: url(\n' + 'fonts/file.ttf\n' + ')\n'. The range helpers should accept either a plain number string or a hyphen-separated pair. min_from_range returns the lower bound as a number, max_from_range returns the upper bound, and if it’s just one number then both return that same number. If the input can’t be parsed, return 0 rather than raising anything.

Last piece is the adapter shape. The execution adapter should read a JSON object from stdin with at least a 'feature' key, like 'feature1_1', and an 'input' key, route that to the right domain function, and print the result plus a trailing newline to stdout. The core parsing/domain functions themselves should stay clean and not depend directly on stdin/stdout.
