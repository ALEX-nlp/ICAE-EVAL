## Product Requirement Document

We need a small utility that can convert date and time related values to and from a standard exchange format so different parts of our system can share them reliably. Right now every team handles this differently and it causes bugs when values are passed between services. The utility should handle all the common kinds of time-related data our business uses: plain calendar dates, times of day, combined date-and-time values, elapsed durations, calendar-offset periods, standalone years, year-and-month combinations, and recurring anniversary-style month-and-day values. For each kind, there should be two ways to represent the value: a compact numeric form for machine-to-machine efficiency, and a human-readable text form for logging and display. The numeric versus text choice should be driven by a simple flag on the request, and the flag should default to the compact form when not specified. The utility should gracefully handle bad input by returning a clean machine-readable error signal rather than dumping internal stack traces or implementation details — similar to how our payment validation module handles its own error responses. Errors should identify what went wrong using a stable code, not runtime internals. The interface should accept a direction flag (encoding vs. decoding), a type identifier, the value to process, and optionally the format preference. The whole thing should be runnable from a command line that reads a single request and writes a single result.

A couple small details to make sure we’re all aligned on. If the request JSON does not include the 'as_timestamp' field, it should default to true, so we use the numeric/compact wire form by default. We should not treat missing 'as_timestamp' as false.

On the text side, when we serialize with as_timestamp=false, the result should be a JSON-quoted string, like '"2013-08-21"'. That same quoted-string behavior also applies to period and month_day, since those only have one wire form anyway. The one exception here is year, which always emits a bare integer, so serialize output for year is never quoted. Also for LocalTime in text form, it should be a quoted ISO-8601 local time string and it should always be normalized to include seconds, even if the input didn’t have them, so if it only came in with hour and minute we append ':00' for the seconds field.

On the numeric side, durations with as_timestamp=true should serialize as a decimal number of seconds. If it’s a whole-second duration, we still always render exactly nine fractional digits, so PT1M becomes 60.000000000. That fractional part is carrying nanosecond-level precision. For LocalTime with as_timestamp=true, the output is a numeric array. It always includes hour and minute. Second only shows up if it actually carries information, meaning non-zero, and nanosecond is only included if it carries information, also non-zero. Example: 22:31:05.000829837 becomes [22,31,5,829837], and that last value is nanoseconds not microseconds.

One other leniency we do want: when deserializing a date, besides accepting the usual [year,month,day] array and ISO-8601 string forms, we should also accept a single integer meaning count of days since the epoch, where day 0 = 1970-01-01. That looser input is only for the 'date' type, not the others.
