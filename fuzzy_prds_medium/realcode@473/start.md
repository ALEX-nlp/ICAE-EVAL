## Product Requirement Document

hey team, we need to ship that foundational toolkit we've been talking about — the one that handles all those low-level building blocks so our messaging and scheduler teams stop reimplementing the same stuff over and over. you know the pain points: someone needs to grab a chunk of items from a list, or loop through things round-robin style, and everyone just rolls their own with slightly different edge case behavior and then we get bugs in prod.

basically we need: ways to slice/filter sequences (think 'give me everything starting from X' or 'stop before Y'), a looping iterator that wraps around, a one-pass cursor thing, a map that can hold multiple values per key with clean removal behavior, some kind of ordering queue that doesn't scramble things when priorities match, a simple on/off gate that handles failed transitions safely (similar to how we did the auth toggle in that login flow last quarter), a rate limiter with a pluggable clock so tests aren't flaky, raw byte buffer utilities with encoding support, and a path parser that handles quoted segments.

the whole thing needs to be split into separate modules — no monolithic files please, we got burned by that before. the JSON adapter approach from the previous spike should work fine for wiring it all up. edge cases matter a lot here, especially around empty inputs, wrap-around, and bad arguments. let's make sure errors come back in a clean normalized format rather than raw exceptions.

one more pass on the edge cases the team asked about so we stay aligned. for the sequence slicing bit, if `start` is at or beyond the end of the sequence, the result is just an empty list — no error, no padding. if fewer than `count` elements remain after `start`, we only return what’s actually there. and the output needs to stay exactly two lines: `length=<n>` and then `items=<comma-separated>`. same idea for the `from` operation: if the target value doesn’t show up at all, the result is empty as `length=0` and `items=`. if it’s the very first element, we return the entire sequence. and the match is the first occurrence only.

for the rate limiter, invalid construction args need to come back as normalized error lines, not anything noisy. negative capacity means `error=invalid_argument` with `param=capacity`, and a non-positive rate means `error=invalid_argument` with `param=rate`. also if someone passes a negative cost on `offer`, reject it as `error=invalid_argument` with `param=cost`. one subtle timing thing too: a step object may carry both `set_time` and `offer` fields, and when both are present we advance the clock to `set_time` BEFORE evaluating the `offer`. the delay calc uses the updated token count after that clock advance. output line there is still `offer cost=<n> delay=<d>`.

on the byte utility side, `unicode` encoding means UTF-16, so it produces 2 bytes per BMP character, which means `hello` encodes to 10 bytes. `utf8` gives 1 byte per ASCII character and 2+ bytes for non-ASCII. decode has to roundtrip cleanly back to the exact original text, and the output format should be `byte_count=<n>`, `decoded=<text>`, and `roundtrip=<true|false>`.

for the path parser, dots inside double-quoted spans are literal characters, not separators, and the quotes themselves get stripped from the returned segment. output should be `count=<n>` followed by one `segment=<text>` line per segment. so a fully quoted single segment like `"ab.cd"` should come out as `count=1` and `segment=ab.cd`. related to that, the `quote` operation just wraps text in leading and trailing double-quote characters. an empty string becomes `""`, and a null/absent `text` field also becomes `""`. the line format is `quoted="<text>"`, with those outer quotes being literal output.

also just to be extra clear on structure, each primitive needs to live in its own cohesive module: sequence utilities, circular enumerator, forward iterator, multimap, priority queue, latch, rate limiter, byte sequence, path parser. then there’s a separate thin execution adapter that handles JSON parsing and output formatting. that adapter should read one JSON object from stdin via the `op` field, route to the right primitive, and write newline-terminated key=value or key -> value lines to stdout. it should not contain business logic, and the primitives should know nothing about stdin/stdout or JSON.

last thing, error rendering stays normalized everywhere: `error=<category>` and optionally `param=<neutral-name>`. never leak host-language type names, stack traces, or raw exception text. examples are `error=action_failed` and `error=invalid_argument` + `param=capacity`. and for feature 6 atomic on/off latch, please preserve the revert-on-failure behavior for switch_on_throw/switch_off_throw, along with the `-> error=action_failed` output format.
