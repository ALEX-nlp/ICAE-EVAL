## Product Requirement Document

We need a small cross-platform image asset metadata utility that helps build tooling answer a handful of recurring questions when packaging visual resources for multiple deployment targets. The tool should accept a single request at a time via a structured input channel and return a plain-text answer on the output channel.

The utility must support five capabilities. First, it should tell a developer whether a proposed image file name forms a legal identifier for the target packaging system — the validation rules are embedded in the naming checker module. Second, it should convert a textual color annotation (either a hex code or a recognizable English color name) into a normalized eight-digit color value suitable for tinting; the color resolution logic lives in the color lookup subsystem. Third, it should parse a dimension annotation expressed as one or two numbers separated by a standard delimiter into explicit width and height values. Fourth, given a short deployment-target label, it should return the folder path and the base scaling multiplier where the unscaled copy of the image should be placed; the per-target mapping is maintained in the destination-path registry. Fifth, it should determine from the file extension alone whether an image is a scalable vector or a pixel-based raster.

The program must remain robust against empty or missing inputs, returning a clearly marked absence value instead of crashing. The solution should follow clean separation between the command-routing layer and the core domain rules.

A couple small clarifications from the questions that came in. For the file-name check, we’re validating the base name only, with directory stripped and extension stripped, using Path.GetFileNameWithoutExtension(filename), and the actual pattern is ^[a-z]+[a-z0-9_]{0,}[^_]$. In plain terms that means it has to start with one or more lowercase ASCII letters a-z, anything in the middle can only be lowercase letters, digits 0-9, or underscores, and it cannot end with an underscore because the last character is matched by [^_]. One side effect of that exact regex is that names have to be at least 2 characters long, so single-character names always fail because [a-z]+ consumes the only character and [^_] requires one more. Uppercase letters, leading digits, leading underscores, trailing underscores, hyphens, spaces, and all other punctuation are rejected. The answer for this one should be exactly 'valid=True' or 'valid=False'. This behavior comes from Utils.IsValidResourceFilename() in golden/src/SingleProject/Resizetizer/src/Utils.cs.

On colors, the output format is always 'argb=AARRGGBB', with AARRGGBB as exactly 8 uppercase hexadecimal digits in alpha-red-green-blue order. If the input is a 6-digit hex like '#abcdef', treat it as fully opaque and force alpha to FF, so that becomes 'argb=FFABCDEF'. If it’s already 8-digit hex like '#12345678', keep that explicit alpha and return 'argb=12345678'. If we can’t resolve the value, the output is exactly 'argb=null'. Named colors are case-insensitive off SKColors fields, and both '-gray' and '-grey' spellings should work the same. Examples the team asked for: 'Red' -> 'argb=FFFF0000'; 'Green' -> 'argb=FF008000'; 'Blue' -> 'argb=FF0000FF'; 'Gray' and 'Grey' both -> 'argb=FF808080'; 'LightSlateGray' and 'LightSlateGrey' both -> 'argb=FF778899'. The grey aliases registered are: DarkGrey, DarkSlateGrey, DimGrey, Grey, LightGrey, LightSlateGrey, SlateGrey. This is coming from Utils.ParseColorString() combined with ColorTable.TryGetNamedColor() in golden/src/SingleProject/Resizetizer/src/ColorTable.cs — SKColor.TryParse handles hex, ColorTable handles named colors including grey/gray aliases.

For sizes, the response format is two lines exactly: 'width=<w>\nheight=<h>' where <w> and <h> are plain integers, so no decimal point when the values are whole numbers. We split on comma ',' or semicolon ';', taking at most 2 parts, and the delimiter can be either ',' or ';' interchangeably. If there’s just one number, use it for both sides, so width=height=that number. If the input is null or an empty string, return exactly 'size=null' as a single line instead of width/height lines.
