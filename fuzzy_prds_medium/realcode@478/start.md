## Product Requirement Document

Our build system needs to work with software release identifiers — the kind that have a major number, optional minor and patch, optional pre-release label, and optional build stamp. Right now every tool team re-derives the same comparison and range logic. We need a shared library that can: (1) parse and normalize a version string so we know it's valid and have a canonical form; (2) compare two versions and tell us which comes first; (3) take a version range expression — including the shorthand formats like 'at least 1.2', 'compatible with 1.2.3', 'between these two versions', and the version shorthand used in our dependency manifests — and normalize it to a canonical comparable form; and (4) test whether a concrete version falls inside a given range. The tricky cases are pre-release labels (they should rank below the corresponding release), wildcards (both explicit and implicit), and the two shorthand range styles that follow the conventions in standard package-manager ecosystems. Edge case to handle: a range expressed as just a major version like '1' should expand the same way our package manifest tooling does — look at how that expansion is done in the existing version utilities to stay consistent.

One small follow-up from the questions that came in: if the input range is an empty string or just a bare wildcard like '*', 'x', or 'X', that should canonicalize to the single token '*', and that means it matches every version. Also, for partial versions, please keep the manifest-style expansion behavior exact. So '1' (major only) expands to '>=1.0.0-0 <2.0.0-0'. The important bit there is that the synthesized bounds include a '-0' pre-release suffix so that pre-release versions at the boundary are handled correctly.

Also worth being explicit about the pre-release matching rule, since that’s where people usually get tripped up: a pre-release version satisfies a comparator only when the comparator itself targets a pre-release at the same major.minor.patch triple. For example, '>4.8' expands to '>=4.9.0-0', which has a pre-release '-0' at 4.9.0, so '4.9.0-beta' (which is > 4.9.0-0) is accepted. But '<4.9' expands to '<4.9.0-0', so '4.9.0-beta' is rejected.

And for consistency, please mirror the behavior in the semver package at internal/semver/ in the TypeScript-Go repository, particularly TryParseVersionRange which performs the canonical expansion of partial/wildcard/tilde/caret ranges.
