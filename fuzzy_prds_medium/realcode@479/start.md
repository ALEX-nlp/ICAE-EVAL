## Product Requirement Document

hey team, we need a new internal tool for the data platform squad ã basically a mini query engine that can take columnar data (think arrays of values per column) and run expressions over them in batch. the idea is you describe what computation you want as a tree of nodes (like conditions, math ops, membership checks, pattern matching etc), and the system compiles that once and evaluates it over a whole batch at a time. we need two modes: one that produces a new computed column (projection), and one that gives back the row positions that pass some condition (filtering). these two should also be chainable ã filter first, then project only the surviving rows.

we talked about this being similar to what the old ETL team did with their expression evaluator last quarter, so maybe reuse that pattern where you separate the tree-building logic from the actual evaluation runtime. also there was that thing we did in the login module around input validation ã same vibe, reject bad inputs early with a clear error category rather than letting garbage through.

null/missing values in input columns need to be handled gracefully ã they should flow through and appear as nulls in output. also make sure the output format is consistent and machine-readable since downstream tools will parse it. the whole thing should be structured across multiple files, not one big blob. let me know if you have questions, i'll try to get more details from the data team.

quick follow-up from the questions that came back: the top-level operation values the adapter should recognize are `project`, `filter`, `filter_project`, and `literal_check`. `project` means evaluate an expression over all rows and produce an output column, `filter` means evaluate a boolean condition and produce a uint32 selection-index vector of surviving row positions in ascending order, and `filter_project` means filter first and then project only those surviving rows. `literal_check` is just for validating typed literal construction, and it reports per-literal `type`, `form`, and `status=ok` or `status=error error=type_mismatch`.

On the output shape, we do need this to be very strict. For a projection operation the output must be exactly four lines: `op=project`, `result_type=<type>`, `length=<n>`, `values=[<v1>, <v2>, ...]` and each line ends with `
`. For a filter operation itãs also exactly four lines: `op=filter`, `selection_type=uint32`, `length=<n>`, `indices=[<i1>, <i2>, ...]` and again each line ends with `
`. For the filter-then-project pipeline, the format matches the projection format. In the value lists, boolean values must be lowercase `true`/`false`, and missing values must be rendered as `null`.

Also spelling out the expression tree a bit more so thereãs no guessing: it needs to support `field`, `literal`, `function`, `if`, `in`, `and`, and `or`. `field` is a column reference by name. `literal` is a typed constant value. `function` is a named function with typed args and a declared return type, covering `add`, `greater_than`, `less_than`, and `like`. `if` has `cond`, `then`, and `else` children plus a declared `type`. `in` has an `expr` child, a `values` list, and a declared `type`. `and` and `or` both take an `args` list of child nodes.

Supported scalar types are `int32`, `int64`, `float64`, `string`, and `bool`, and the selection/index vector from a filter always has type `uint32`. One other small thing the team asked about: type names can come in either as a plain string, the `name` form, or as a structured type object, the `datatype` form, and both forms need to be accepted when constructing literals.

For null behavior, the rule is consistent all the way through: a null input value has to propagate through every expression node that touches it ã arithmetic, comparison, conditional, membership, pattern-match ã and produce a null in the output column for that row. In the filter-then-project path, if the projected value is null it should show up in the output values list as the literal token `null` in lowercase with no quotes. For filtering, if a boolean condition comes out null for a row, that row should be treated as non-matching and not appear in the selection vector.

And on structure, same direction as before but a little more explicit: the core compiler/evaluation logic needs to stay decoupled from stdin/stdout and JSON parsing, and the execution adapter is the only piece that translates JSON requests into calls on the core engine. So the multi-file split should cleanly separate the expression node module, projection engine, filter engine, and adapter layer.
